# ADR-038. Toll — cluster-wide detector массового оттока Souls.

**Контекст.** При массовом оттоке Soul-ов (network split, DC outage, broken upgrade) Keeper-кластер сейчас принимает disconnection-события молча: метрики падают, но никакого alert / circuit-breaker. Оператор узнаёт о проблеме через Grafana или жалобы. Дополнительно — если в этот момент оператор пытается запустить scenario, он получит partial-failed (часть Soul-ов недоступна) вместо явного отказа от API «кластер в проблеме, retry позже».

Имя `soul-shedding` коллидирует с уже зафиксированной сущностью Watchman ([ADR-027](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim) amend (k), [naming-rules.md → Watchman](../naming-rules.md)) — там это **активное** закрытие стримов изолированным Keeper-ом. Новая сущность — **пассивный** наблюдатель.

**Решение.**

1. **Сущность `Toll`** (звон колокола / death toll — souls-метафора, propose-and-wait). Detector массового оттока Soul-ов на cluster-level. Фиксирует rate-of-disconnect через окно, выставляет cluster-degraded флаг при превышении threshold.

2. **Архитектура — per-instance + Redis-leader агрегация.**
   - **Per-Keeper goroutine `tollwatcher`** (по аналогии с Watchman): подписывается на gRPC disconnect-events EventStream-а, фильтрует не-graceful-shutdown closures, считает локальный rate в sliding-окне 60s.
   - **Cluster-leader через Redis-lease** (по аналогии с Reaper): один Keeper-инстанс выигрывает lease `cluster:toll:leader` (TTL 30s + renew 10s), агрегирует rate-ы из всех инстансов (через Redis sorted-set per-second buckets), сравнивает с baseline `souls.status='connected'`-counter (snapshot в начале окна) и взводит флаг `cluster:degraded` (Redis key, TTL 60s) при превышении threshold.
   - Per-instance детекторы публикуют свой rate в общий Redis sorted-set; leader агрегирует и принимает решение. Per-instance НЕ выставляют degraded-флаг сами — только наблюдают.

3. **Источник «потеряно» — gRPC disconnection events, НЕ heartbeat.**
   - `mark_disconnected` Reaper-правило измеряет heartbeat-выпадение (~90s до перевода soul в `disconnected`) — это слишком медленно для 1-минутного окна Toll.
   - `keeper_grpc_streams_closed_total` (extends existing metric) — counter инкрементируется при EventStream `cleanup`-вызове (когда `gRPC.Recv` возвращает err / клиент закрыл).
   - **Фильтрация graceful-shutdown:** Toll учитывает **только закрытия по причинам, отличным от собственного graceful-shutdown** (отделяет `cleanup` под `ctx.Done()` от `Recv-error`). Дополнительно — фильтр «не считать первые 60s после старта инстанса» (warmup-immunity против race на cluster startup).

4. **Threshold и окно — cluster-wide, не per-coven для MVP.**
   - **Sliding window 60s** (per-second buckets в Redis sorted-set).
   - **Threshold 20%** — `disconnect_rate / baseline_connected_souls > 0.20` за окно.
   - **Baseline** — snapshot `COUNT(*) FROM souls WHERE status='connected'` в начале окна, refreshes каждые 60s.
   - Per-coven cardinality отложена (cardinality risk + сейчас не нужно для MVP).

5. **Soft-degraded mode mechanics.**
   - Middleware на Operator API + MCP-handler-ах.
   - **Блокируются (HTTP 503 + Retry-After):** `POST /v1/incarnations/{name}/scenarios/{scenario}` (запуск scenario), `POST /v1/push/apply` (push-orchestrator).
   - **НЕ блокируются:** read-API (GET *), `POST /v1/incarnations/{name}/unlock`, `POST /v1/operators/{aid}/...` (RBAC), destroy-операции (это recovery actions).
   - Errand-endpoint (`POST /v1/souls/{sid}/exec`) — **не блокируется** (точечная диагностика нужна в degraded-режиме).
   - Audit-events `cluster.degraded_set` (при взведении) + `cluster.degraded_cleared` (при снятии) в новую область `cluster.*`.

6. **Re-armed — asymmetric hysteresis.**
   - Сработать: rate > threshold за 60s окно (single tick).
   - Снять: rate ≤ threshold (восстановилось) **+ grace 60s** до автоматического снятия (требует устойчивого normality, не флапа).
   - Audit-event `cluster.degraded_cleared` пишется только после grace.

7. **Alert-channel MVP — audit-event + Prometheus-metric.**
   - Prometheus gauge `keeper_cluster_degraded` (0/1, set leader-ом).
   - Prometheus counter `keeper_toll_disconnects_total{coven}` (per-coven cardinality безопасна на counter — не fanout-флага).
   - Audit-event `cluster.degraded_set` + `cluster.degraded_cleared`.
   - OTel-span на сам fire-event (дёшево).
   - Webhook / PagerDuty / Slack — отложено отдельным slice-ом (alert-fan-out — отдельная подсистема).

**Контракт.**

| Поле | Значение |
|---|---|
| Имя сущности | Toll |
| Position | Keeper-cluster level, не per-instance |
| Источник данных | gRPC disconnect events (filtered) |
| Окно | sliding 60s, per-second buckets |
| Threshold | 20% от baseline `souls.status='connected'` |
| Leader-election | Redis lease `cluster:toll:leader` (TTL 30s/renew 10s) |
| Degraded-flag | Redis key `cluster:degraded` (TTL 60s) |
| Blocked endpoints | POST /v1/incarnations/.../scenarios/..., POST /v1/push/apply |
| Hysteresis | asymmetric, 60s grace для clearing |
| Warmup | 60s после старта инстанса |
| Audit-events | cluster.degraded_set, cluster.degraded_cleared |
| Metrics | keeper_cluster_degraded (gauge), keeper_toll_disconnects_total (counter) |

**Инварианты.**

- **Toll НЕ принимает решений по recovery** — только наблюдает и блокирует write-API.
- **Toll НЕ закрывает стримы** (это работа Watchman). Toll — passive observer.
- **Soft-degraded ≠ полный лock**: read-API + destroy + Errand остаются доступны для recovery-действий оператора.
- **Single-leader агрегация** — два cluster-degraded флага одновременно не выставляются (Redis-lease гарантирует exclusive leader).
- **False-positive defense:** graceful-shutdown excluded (rolling cluster restart НЕ триггерит Toll), warmup-immunity первые 60s (cluster cold start safety).

**Отвергнутые альтернативы.**

- (а) **Per-instance independent decisions** (каждый Keeper сам решает degraded?) — отвергнуто: thrashing-риск (два Keeper-а одновременно сбрасывают флаг), inconsistent UX (одни 503-ят, другие нет).
- (б) **Heartbeat-выпадение как источник** — слишком медленно для 1-минутного окна (90s до перевода в `disconnected`).
- (в) **Per-coven threshold** — отложено (cardinality + MVP-простота).
- (г) **Webhook/PagerDuty alert-channel в MVP** — отложено (alert-fan-out = отдельная подсистема).
- (д) **Имя `soul-shedding`** — отвергнуто, коллидирует с Watchman (активным шединг-actor-ом).

**Amendment 2026-05-27. Toll extensions: webhook + per-coven thresholds.**

Закрывает два отложенных пункта без новых ADR (additive-extension к ADR-038, без breaking-changes для существующих keeper.yml).

1. **Webhook alert-channel** (`toll.webhook.*`) — best-effort POST на `cluster.degraded_set` / `cluster.degraded_cleared`. Три формата:
   - `generic` — плоский JSON-payload (`event_type` / `leader_kid` / `rate` / `baseline_connected` / `threshold` / `window_seconds` / `timestamp` / опц. `coven_name`);
   - `pagerduty_v2` — PagerDuty Events API v2 (`routing_key` из Vault KV под полем `routing_key`, `dedup_key: soul-stack/cluster:degraded` для trigger+resolve в одну incident, `event_action: trigger/resolve` для set/cleared);
   - `slack` — Slack incoming webhook (attachment с `color: danger/good`, поля Rate/Threshold/Baseline/Window/Coven).
   `url_ref` — vault-ref (рекомендуется: интеграционные ключи — секреты) либо inline URL (для dev). Default-off (`enabled: false`) — добавление поля не меняет поведение существующих конфигов. Best-effort: ошибки POST-а логируются, но НЕ блокируют Set/Clear (cluster degraded — primary goal). Поднимается ТОЛЬКО на leader-инстансе (single-winner) — alert не дублируется N инстансов.

2. **Per-coven thresholds** (`toll.per_coven_thresholds: {coven: threshold}`) — OR-семантика поверх global threshold-а:
   - Если задан и непустой, leader дополнительно тикает `ZRANGEBYSCORE` (extra round-trip только при настроенных per-coven), парсит member-value `<sid>|<kid>|<coven>|<nano>` и группирует по coven (3-й `|`-сегмент).
   - При превышении ЛЮБОГО per-coven threshold-а (rate per coven > threshold) ИЛИ global threshold-а — `cluster:degraded` взводится.
   - При per-coven trigger-е audit-payload + webhook-payload содержат `coven_name`; при global — без него (cluster-wide отток).
   - При двойном превышении (global И per-coven) global перевешивает в diag/audit/webhook (для оператора cluster-wide отток приоритетнее локального).
   - Cardinality-риск (изначальная причина отказа в MVP) митигирован: ключи `per_coven_thresholds` — закрытый список оператора в keeper.yml, не динамическая cardinality. Prometheus counter `keeper_toll_disconnects_total{coven}` уже несёт ту же cardinality.

Контракт middleware / blocked endpoints / asymmetric hysteresis / leader-election — БЕЗ ИЗМЕНЕНИЙ. Audit-event `EventClusterDegradedSet` остаётся тот же (forward-compat: `coven_name` — опц. поле, отсутствие = global). Метрики — без новых дескрипторов.

3. **Hot-reload `toll.*`** ([ADR-021](0021-hot-reload-config.md#adr-021-hot-reload-конфига-с-write-back-yaml)). `threshold` / `window_size` / `degraded_ttl` / `clear_grace` / `per_coven_thresholds` / `webhook.*` применяются на лету через `Leader.UpdateConfig` (atomic swap полей под RWMutex; tick читает snapshot в начале aggregation-тика). Webhook-notifier пересоздаётся только при изменении `webhook.url_ref` / `format` / `timeout` / `enabled` (diff-проверка в daemon-callback-е) — частые reload-ы с неизменным webhook не дёргают Vault-резолв. Restart-required: `enabled` (вся Toll-инфраструктура), `lease_ttl` (захвачен в renew-loop), `warmup_delay` (применяется per-instance Watcher-ом на старте). Полная таблица — [`docs/keeper/config.md → toll`](../keeper/config.md#toll).
