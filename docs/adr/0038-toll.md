# ADR-038. Toll — a cluster-wide detector of mass Souls attrition.

**Context.** During a mass attrition of Souls (network split, DC outage, broken upgrade) the Keeper cluster currently accepts disconnection events silently: metrics drop, but there is no alert / circuit-breaker. The operator learns about the problem through Grafana or complaints. Additionally — if at that moment the operator tries to run a scenario, they get a partial-failed (some Souls are unreachable) instead of an explicit API rejection "the cluster is in trouble, retry later."

The name `soul-shedding` collides with the already-fixed entity Watchman ([ADR-027](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim) amend (k), [naming-rules.md → Watchman](../naming-rules.md)) — there it is the **active** closing of streams by an isolated Keeper. The new entity is a **passive** observer.

**Decision.**

1. **Entity `Toll`** (the tolling of a bell / death toll — a souls metaphor, propose-and-wait). A detector of mass Souls attrition at cluster level. It records the rate-of-disconnect over a window and sets a cluster-degraded flag when the threshold is exceeded.

2. **Architecture — per-instance + Redis-leader aggregation.**
   - **Per-Keeper goroutine `tollwatcher`** (by analogy with Watchman): subscribes to the EventStream's gRPC disconnect events, filters out non-graceful-shutdown closures, counts the local rate in a 60s sliding window.
   - **Cluster leader via a Redis lease** (by analogy with Reaper): one Keeper instance wins the lease `cluster:toll:leader` (TTL 30s + renew 10s), aggregates the rates from all instances (via a Redis sorted-set of per-second buckets), compares against the baseline `souls.status='connected'` counter (a snapshot at the start of the window), and raises the `cluster:degraded` flag (a Redis key, TTL 60s) when the threshold is exceeded.
   - Per-instance detectors publish their rate into a shared Redis sorted-set; the leader aggregates and makes the decision. Per-instance detectors do NOT set the degraded flag themselves — they only observe.

3. **Source of "lost" — gRPC disconnection events, NOT heartbeat.**
   - The `mark_disconnected` Reaper rule measures heartbeat loss (~90s before a soul is moved to `disconnected`) — that is too slow for Toll's 1-minute window.
   - `keeper_grpc_streams_closed_total` (extends existing metric) — the counter is incremented on the EventStream `cleanup` call (when `gRPC.Recv` returns an err / the client closed).
   - **Graceful-shutdown filtering:** Toll counts **only closures for reasons other than its own graceful shutdown** (it separates `cleanup` under `ctx.Done()` from a `Recv error`). Additionally — a filter "do not count the first 60s after instance start" (warmup immunity against a race at cluster startup).

4. **Threshold and window — cluster-wide, not per-coven for MVP.**
   - **Sliding window 60s** (per-second buckets in a Redis sorted-set).
   - **Threshold 20%** — `disconnect_rate / baseline_connected_souls > 0.20` over the window.
   - **Baseline** — a snapshot `COUNT(*) FROM souls WHERE status='connected'` at the start of the window, refreshes every 60s.
   - Per-coven cardinality is deferred (cardinality risk + not needed for MVP right now).

5. **Soft-degraded mode mechanics.**
   - Middleware on the Operator API + MCP handlers.
   - **Blocked (HTTP 503 + Retry-After):** `POST /v1/incarnations/{name}/scenarios/{scenario}` (running a scenario), `POST /v1/push/apply` (push orchestrator).
   - **NOT blocked:** the read API (GET *), `POST /v1/incarnations/{name}/unlock`, `POST /v1/operators/{aid}/...` (RBAC), destroy operations (these are recovery actions).
   - The Errand endpoint (`POST /v1/souls/{sid}/exec`) is **not blocked** (targeted diagnostics are needed in degraded mode).
   - Audit events `cluster.degraded_set` (on set) + `cluster.degraded_cleared` (on clear) in the new area `cluster.*`.

6. **Re-armed — asymmetric hysteresis.**
   - Trigger: rate > threshold over a 60s window (single tick).
   - Clear: rate ≤ threshold (recovered) **+ a 60s grace** before automatic clearing (requires sustained normality, not a flap).
   - The audit event `cluster.degraded_cleared` is written only after the grace period.

7. **Alert channel MVP — audit event + Prometheus metric.**
   - Prometheus gauge `keeper_cluster_degraded` (0/1, set by the leader).
   - Prometheus counter `keeper_toll_disconnects_total{coven}` (per-coven cardinality is safe on a counter — not on a fan-out flag).
   - Audit events `cluster.degraded_set` + `cluster.degraded_cleared`.
   - An OTel span on the fire event itself (cheap).
   - Webhook / PagerDuty / Slack — deferred as a separate slice (alert fan-out is a separate subsystem).

**Contract.**

| Field | Value |
|---|---|
| Entity name | Toll |
| Position | Keeper-cluster level, not per-instance |
| Data source | gRPC disconnect events (filtered) |
| Window | sliding 60s, per-second buckets |
| Threshold | 20% of the baseline `souls.status='connected'` |
| Leader election | Redis lease `cluster:toll:leader` (TTL 30s/renew 10s) |
| Degraded flag | Redis key `cluster:degraded` (TTL 60s) |
| Blocked endpoints | POST /v1/incarnations/.../scenarios/..., POST /v1/push/apply |
| Hysteresis | asymmetric, 60s grace for clearing |
| Warmup | 60s after instance start |
| Audit events | cluster.degraded_set, cluster.degraded_cleared |
| Metrics | keeper_cluster_degraded (gauge), keeper_toll_disconnects_total (counter) |

**Invariants.**

- **Toll does NOT make recovery decisions** — it only observes and blocks the write API.
- **Toll does NOT close streams** (that is Watchman's job). Toll is a passive observer.
- **Soft-degraded ≠ full lock**: the read API + destroy + Errand remain available for the operator's recovery actions.
- **Single-leader aggregation** — two cluster-degraded flags are never set at the same time (the Redis lease guarantees an exclusive leader).
- **False-positive defense:** graceful shutdown is excluded (a rolling cluster restart does NOT trigger Toll), warmup immunity for the first 60s (cluster cold-start safety).

**Rejected alternatives.**

- (a) **Per-instance independent decisions** (each Keeper decides degraded on its own?) — rejected: thrashing risk (two Keepers clearing the flag at the same time), inconsistent UX (some return 503, others do not).
- (b) **Heartbeat loss as the source** — too slow for the 1-minute window (90s before the move to `disconnected`).
- (c) **Per-coven threshold** — deferred (cardinality + MVP simplicity).
- (d) **Webhook/PagerDuty alert channel in MVP** — deferred (alert fan-out = a separate subsystem).
- (e) **The name `soul-shedding`** — rejected, it collides with Watchman (the active shedding actor).

**Amendment 2026-05-27. Toll extensions: webhook + per-coven thresholds.**

Closes two deferred items without new ADRs (an additive extension to ADR-038, without breaking changes for existing keeper.yml).

1. **Webhook alert channel** (`toll.webhook.*`) — a best-effort POST on `cluster.degraded_set` / `cluster.degraded_cleared`. Three formats:
   - `generic` — a flat JSON payload (`event_type` / `leader_kid` / `rate` / `baseline_connected` / `threshold` / `window_seconds` / `timestamp` / opt. `coven_name`);
   - `pagerduty_v2` — PagerDuty Events API v2 (`routing_key` from Vault KV under the `routing_key` field, `dedup_key: soul-stack/cluster:degraded` to trigger+resolve into a single incident, `event_action: trigger/resolve` for set/cleared);
   - `slack` — a Slack incoming webhook (an attachment with `color: danger/good`, fields Rate/Threshold/Baseline/Window/Coven).
   `url_ref` — a vault-ref (recommended: integration keys are secrets) or an inline URL (for dev). Default-off (`enabled: false`) — adding the field does not change the behavior of existing configs. Best-effort: POST errors are logged but do NOT block Set/Clear (cluster degraded is the primary goal). It runs ONLY on the leader instance (single-winner) — the alert is not duplicated by N instances.

2. **Per-coven thresholds** (`toll.per_coven_thresholds: {coven: threshold}`) — OR semantics on top of the global threshold:
   - If set and non-empty, the leader additionally ticks `ZRANGEBYSCORE` (an extra round-trip only when per-coven is configured), parses the member value `<sid>|<kid>|<coven>|<nano>`, and groups by coven (the 3rd `|`-segment).
   - When ANY per-coven threshold is exceeded (rate per coven > threshold) OR the global threshold — `cluster:degraded` is raised.
   - On a per-coven trigger the audit payload + webhook payload contain `coven_name`; on a global one — without it (cluster-wide attrition).
   - On a double breach (global AND per-coven) the global one wins in diag/audit/webhook (for the operator a cluster-wide attrition takes priority over a local one).
   - The cardinality risk (the original reason for the MVP rejection) is mitigated: the `per_coven_thresholds` keys are a closed operator-defined list in keeper.yml, not dynamic cardinality. The Prometheus counter `keeper_toll_disconnects_total{coven}` already carries the same cardinality.

The middleware / blocked endpoints / asymmetric hysteresis / leader-election contract is UNCHANGED. The audit event `EventClusterDegradedSet` stays the same (forward-compat: `coven_name` is an optional field, its absence = global). Metrics — no new descriptors.

3. **Hot-reload `toll.*`** ([ADR-021](0021-hot-reload-config.md#adr-021-hot-reload-конфига-с-write-back-yaml)). `threshold` / `window_size` / `degraded_ttl` / `clear_grace` / `per_coven_thresholds` / `webhook.*` are applied on the fly via `Leader.UpdateConfig` (an atomic swap of fields under an RWMutex; the tick reads a snapshot at the start of the aggregation tick). The webhook notifier is recreated only when `webhook.url_ref` / `format` / `timeout` / `enabled` change (a diff check in the daemon callback) — frequent reloads with an unchanged webhook do not trigger a Vault resolve. Restart-required: `enabled` (the whole Toll infrastructure), `lease_ttl` (captured in the renew loop), `warmup_delay` (applied per-instance by the Watcher at start).
