# ADR-040. Tide — invocation-time scope chunking + target-override

> **Статус: Superseded by [ADR-043](0043-voyage.md#adr-043-voyage--унифицированный-батчевый-прогон) (Voyage), 2026-05-29.**
>
> **Реализация Tide УДАЛЕНА (Wave 5, 2026-05-30).** Таблица `tides` дропнута migration `061`; пакеты `internal/tideorch` + `internal/tide` (вкл. TideWorker / claim / surge / audit), эндпоинты `/v1/tides`, proto-сообщения `EventTide*` и audit-семейство `tide.*` удалены из кода. Функциональность invocation-time chunking scenario-батчей теперь живёт в Voyage `kind=scenario` ([ADR-043](0043-voyage.md#adr-043-voyage--унифицированный-батчевый-прогон)): Surge → Leg, per-Surge state-commit → per-incarnation state-commit (B1), claim+lease failover и snapshot-scope унаследованы. Изменения при поглощении: **(1) invocation-time target-override УДАЛЁН** — AND-merge `target.coven[]`/`target.where` заменён выбором цели из уже-разрешённого RBAC-скоупа (security-смысл «нельзя расширить scope» сохранён, см. [ADR-043](0043-voyage.md#adr-043-voyage--унифицированный-батчевый-прогон) пункт 5). **(2) Сохраняются** инварианты: per-incarnation state-commit (parity [ADR-009 §7](0009-scenario-dsl.md#adr-009-scenario--полная-dsl-задач-destiny-граница-с-destiny--рекомендация)), claim+lease failover (Acolyte-style PG-lease), snapshot-scope набора единиц. **(3) Surge → Leg** (единица батча Voyage).
>
> **Реализация удалена — см. [ADR-043](0043-voyage.md#adr-043-voyage--унифицированный-батчевый-прогон).** Ниже — исходная фиксация решения (запись истории, не действующий контракт).

**Контекст.** Scenario-DSL ([ADR-009](0009-scenario-dsl.md#adr-009-scenario--полная-dsl-задач-destiny-граница-с-destiny--рекомендация)) фиксирует `on:`/`where:`/`serial:` на уровне **декларации scenario** в git-репо сервиса. Для массовых выкаток на 100k+ souls в одной incarnation этого недостаточно — оператор хочет при invocation:
- разбить total scope на последовательные **waves** (10k × 10 прогонов с барьером и аудитом между ними);
- сузить target по labels/coven + CEL-predicate при запуске (но НЕ расширить — security);
- override concurrency-knob (runtime-параметр исполнения, не declared invariant).

Аналоги: Salt `salt -b <size>` + `-E/-G` targeting; Ansible `serial:` per-play + `--limit`; Bolt `--concurrency` + `--targets`. Все три — invocation-level, не declaration-level.

**Решение.**

1. **Новая сущность Tide** (top-level entity) и **Surge** (одна волна). Имя выбрано propose-and-wait, architect-вердикт `ae5494e6cbb491804` + PM-decisions 2026-05-27. Tide = invocation-instance массового scenario-прогона, разбитого на последовательные **Surges**. 100k souls + wave_size=10k = 1 Tide × 10 Surges.

2. **Surge = одна волна = один `apply_run`.** Surges выполняются sequentially (failure-policy управляет переходом к следующему). Каждый Surge — полноценный scenario-прогон со своим barrier + state-commit, **парность с [ADR-009 §7](0009-scenario-dsl.md#adr-009-scenario--полная-dsl-задач-destiny-граница-с-destiny--рекомендация)** (barrier/state-commit per-Surge, не per-Tide).

3. **PG-table `tides`** (детали колонок — W-1 после pre-W-2 failover-architect-consult):
   - `tide_id` ULID PK;
   - `incarnation_name` FK на `incarnation(name)`;
   - `scenario_name`;
   - `target_coven_override` (text[]) + `target_where_override` (CEL-строка) — invocation-time labels/coven и per-host predicate;
   - `wave_size` int + `on_surge_failure` enum (`abort`|`continue`) + `concurrency_override` int (nullable, REPLACE scenario `serial:`);
   - `claimed_by_kid` (nullable) + lease-поля (Acolyte-style — для failover, см. ниже инварианты);
   - `status` enum (`pending`|`running`|`succeeded`|`partial_failed`|`failed`|`cancelled`|`orphaned`);
   - `started_by_aid` FK `operators(aid)`;
   - `summary` jsonb (surge-results: per-Surge apply_id + status + souls_count);
   - timestamps (`started_at`, `finished_at`, `last_renewed_at`).

4. **`apply_runs.tide_id` (nullable FK) + `surge_index` (int).** Single-run apply_runs (без Tide, classic `POST /v1/incarnations/.../scenarios/.../run` без `wave`-поля) продолжают работать backward-compat — `tide_id IS NULL`. Композиция `(tide_id, surge_index)` идентифицирует Surge в рамках Tide.

5. **API — opt-расширение существующего endpoint-а.** `POST /v1/incarnations/{name}/scenarios/{scenario}` (см. [operator-api.md](../keeper/operator-api.md)) принимает дополнительные opt-поля:

   ```json
   {
     "input":  { ... },
     "target":      { "coven": ["prod-eu"], "where": "soulprint.self.os.family == \"debian\"" },
     "wave":        { "size": 10000, "on_failure": "abort" },
     "concurrency": 100
   }
   ```

   Без `wave` — старое поведение (один `apply_run`, classic single-run). С `wave` — Tide-orchestrator (W-2). Backward-compat — only-add полей, существующие клиенты не ломаются.

6. **CLI (`soulctl`):** флаги `--wave-size`, `--wave-on-failure {abort|continue}`, `--where <CEL>`, `--target-coven <name>` (повторяемый), `--concurrency <N>`, `--invocation-file <yaml>` (полный invocation-payload файлом).

7. **MCP:** существующий tool `keeper.incarnation.run` расширяется теми же opt-полями (`target`/`wave`/`concurrency`). **НЕ** новый tool — forward-compat.

**Инварианты.**

- **AND-merge target-override (security).** Invocation-override `target.coven[]`/`target.where` только **сужает** scope (пересечение со scenario `on:`/`where:`), не расширяет. Оператор не может через invocation выйти за рамки декларированного scenario-таргета. **REPLACE отвергнут** как нарушение [ADR-009 §3](0009-scenario-dsl.md#adr-009-scenario--полная-dsl-задач-destiny-граница-с-destiny--рекомендация) — иначе scenario-author теряет контроль над максимальным blast-radius-ом.
- **Glob/regex matching через CEL (parity SaltStack `-G`/`-E`).** `target.where: 'sid.glob("prod-*")'` (shell-glob) или `target.where: 'sid.matches("^db-[0-9]+$")'` (regex stdlib) — обычные CEL-предикаты; единая граматика без отдельного DSL. Композиция с фактами: `target.where: 'os.family == "debian" && sid.glob("web-*")'`. См. [templating.md §2.3](../templating.md#23-зарегистрированные-cel-функции-стартовый-минимум).
- **Multi-label семантика внутри одного списка (resolved 2026-05-27).** Перечисление меток как в scenario `on: [a, b]`, так и в invocation `target.coven: [a, b]` — **AND-пересечение**. Хост попадает в роутинг, только если у него присутствуют **ВСЕ** перечисленные метки. Композиция scenario-on ↔ invocation-target — также AND. Это обеспечивает fail-closed-инвариант на двух уровнях: ни декларация (scenario), ни invocation не могут расширить scope через перечисление меток. Для OR-кейса оператор использует `target.where: CEL` с явным predicate. Симметрично [orchestration.md §3](../scenario/orchestration.md) (single source of truth текста DSL).
- **REPLACE concurrency-override.** `concurrency` — runtime-knob исполнения, не declared invariant; REPLACE — естественный contract (оператор знает специфику текущего железа кластера). Без override — scenario `serial:` сохраняется.
- **Per-Surge state-commit, parity с [ADR-009 §7](0009-scenario-dsl.md#adr-009-scenario--полная-dsl-задач-destiny-граница-с-destiny--рекомендация).** Каждый Surge = отдельный scenario-прогон со своим barrier; `state_changes` коммитятся после успешного Surge, не после всего Tide. Иначе нарушился бы barrier-инвариант DSL-ядра.
- **Tide-orchestrator stateless относительно конкретного Keeper-instance** (Acolyte-style lease в PG для failover). Pre-W-2 architect-consult фиксирует mechanics (lease-renew period, reclaim-rule в Reaper, fencing-epoch).
- **`run_once:` + Tide — abort с error на валидации.** Сценарий с `run_once:`-шагом несовместим с разбиением на waves (логически run_once = ровно один хост таргета; разбивать на waves нечего). Валидация при `POST` отвергает запрос (`422 validation-failed`, error code `tide_run_once_conflict`).
- **Cross-incarnation Tide запрещён грамматикой** — symmetry с [ADR-008](0008-coven-stable-tags.md#adr-008-coven--только-стабильные-логические-теги). Tide привязан к одной incarnation.
- **Surges sequential.** Следующий Surge стартует только после **terminal** статуса предыдущего (`success`/`failed`/`cancelled`/`orphaned`/`no_match`, см. [ADR-027](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim) FINDING-01). Параллельные surges — отдельная сущность, в MVP не рассматриваются.
- **RBAC reuse — `incarnation.run`.** Тот же permission, что и classic single-run; новый perm не вводится — Tide это та же операция, просто разбитая на waves. Granular Tide-perm (`tide.create`/`tide.cancel`) — отложено до запроса оператора.

**Отвергнутые альтернативы.**

- (а) **Client-side N requests** (оператор сам в цикле зовёт `POST .../scenarios/.../run` с разными `target`-фильтрами). Нет атомарности Tide, нет audit-связности, нет cluster-side progress-view, нет cancel-возможности. Решение перекладывает оркестрацию на клиента — это работа Keeper-а.
- (б) **REPLACE target-override.** Security regression — оператор мог бы расширить scope за рамки декларированного scenario `on:`. Нарушает [ADR-009 §3](0009-scenario-dsl.md#adr-009-scenario--полная-dsl-задач-destiny-граница-с-destiny--рекомендация).
- (в) **Per-task wave-size** (разные задачи scenario бьются разными ширинами). Требует переделки Acolyte-dispatch (per-task `RenderedTask`, см. open Q «per-task волны» в [ADR-009](0009-scenario-dsl.md#adr-009-scenario--полная-dsl-задач-destiny-граница-с-destiny--рекомендация)). Tide работает на уровне scope, не на уровне задач.
- (г) **Aggregate state-commit** (один commit на весь Tide). Ломал бы [ADR-009 §7](0009-scenario-dsl.md#adr-009-scenario--полная-dsl-задач-destiny-граница-с-destiny--рекомендация) barrier-инвариант — Surge №3 падает, а state-commit Surges №1–2 уже произошёл? наоборот, не произошёл, но первые две волны уже изменили хосты? Per-Surge commit — единственно корректная семантика.
- (д) **`stop-and-wait` failure-policy в MVP** (после failed Surge ждать ручного решения оператора). Операционно сложно (нужен polling-эндпоинт состояния «жду решения», cancel-эндпоинт для отмены ожидания, TTL для protection от forgotten Tide). MVP — `abort` + `continue`; `stop-and-wait` — отложено post-MVP.
- (е) **Новый MCP-tool `keeper.tide.run`.** Удваивает tool-каталог, forward-compat не нужна (existing-tool only-add полей не ломает старых клиентов).

**Что отложено пост-MVP.**

- **Tide cancellation mid-flight.** `DELETE /v1/tides/{tide_id}` + cancel-policy (как обрабатывается currently-running Surge — abort немедленно / дать дойти / abort+wait). Требует решений по cancel-семантике Acolyte-stream-ов. Аудит-событие `tide.cancelled` зарезервировано.
- **`stop-and-wait` failure-policy** — manual operator retry/abort flag после failed Surge.
- **Tide-progress SSE для UI** (live-feed surge-completions). Отдельный slice после стабилизации core-orchestrator-а.
- **Dynamic resize wave-size mid-run** (оператор увеличивает/уменьшает wave_size в процессе). Operational nicety, не блокирующая для MVP.
- **Mid-scenario pause/resume.** Отдельная подсистема (нужны state-snapshots scenario-runner-а), не относится к Tide-scope-у.

Failover-resilience mechanics.** Architect-consult `a112874e417338ff5` зафиксировал mechanics, позволяющие любому Keeper-инстансу подобрать Tide через PG-claim+lease (Tide stateless относительно конкретного инстанса). Parity с Acolyte/Ward ([ADR-027](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim)) для `apply_runs`, но на один уровень выше — над work-queue. Этот amendment ЗАКРЫВАЕТ детализацию PG-схемы и mechanics, отложенную в исходном пункте 3 ADR-040; конкретные имена колонок и SQL — ниже.

**Архитектурные инварианты.**

1. **TideWorker pool — отдельный worker-pool**, **не** общий с Acolyte (deadlock-prevention: при `acolytes: 1` + 1 Tide занял Acolyte-слот, Surge-у внутри Tide некуда было бы попасть). Конфиг `keeper.tide_workers: N` (default `1`, рекомендация `max ≤ acolytes-1`). Цикл воркера — простой: `claim → orchestrate → poll/sleep`, без сложной wake-логики (Tide-операции редкие крупные, не hot-path).

2. **Claim-pattern (parity [ADR-027(d)](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim))**:
   - `SELECT … FOR UPDATE SKIP LOCKED` для атомарного pickup из `tides` со `status='pending'`.
   - `attempt++` на каждый claim для fencing-epoch (если orchestrator-логика когда-нибудь породит дубль, поздний `RunResult`-обработчик отсечёт его по epoch).
   - **PG-based lease.** Redis-lease отвергнут — single-leader bottleneck; PG-advisory-lock отвергнут — не самоочищается по TTL (висит до Postgres-разрыва коннекта, ненадёжно при OOM Keeper-а с зависшим коннектом в pgbouncer).
   - **TTL 60s + renewal 20s** через CAS-UPDATE с guard `WHERE tide_id=$1 AND claimed_by_kid=$me AND claim_expires_at > NOW()`. 0 rows affected → `ErrLeaseLost`, TideWorker немедленно бросает работу (Surge-spawn-у не дано продолжать без подтверждённого владения).

3. **reclaim_tides Reaper-rule** ([keeper/reaper.md](../keeper/reaper.md)):
   ```sql
   UPDATE tides
      SET status='pending',
          claimed_by_kid=NULL,
          last_renewed_at=NULL,
          attempt=attempt+1
    WHERE status='running'
      AND claim_expires_at < NOW()
   ```
   Default `enabled: false` (operational gate — включается после стабилизации fencing-Soul флота, parity с [ADR-027(i)](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim) GATE-1). Audit-event `tide.reclaimed { tide_id, last_renewed_at, attempt_after }` пишется per-row.

4. **Same-transaction invariant (parity [ADR-027(j)](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim) single-winner)**: spawn `apply_run` + `UPDATE tides SET current_apply_id=Y, current_surge_index=N` в **одной PG-транзакции**. Crash посередине → PG откатит, `apply_run` не существует, Keeper-Б при подборе видит `current_surge_index=N-1`, стартует surge N заново. Без same-tx два Keeper-а могли бы породить два `apply_run` на один Surge.

5. **AwaitApplyTerminal — гибридный режим**:
   - **Redis-subscribe** `apply:terminal:<apply_id>` для responsiveness (terminal-handler публикует при переводе `apply_run.status` в терминал).
   - **Periodic poll** `apply_runs.status WHERE apply_id=$current` каждые 5s как fallback на потерю pub/sub-сигнала (HA-cluster N=3 Keeper-инстансов, `RunResult` может прилететь на любой инстанс, sub-к-локальному-pub может пропустить).
   - Tide-orchestrator на Keeper-Б при pickup НЕ трогает existing `apply_run` — Acolyte-recovery работает своим путём через [`reclaim_apply_runs`](../keeper/reaper.md) (если включено), либо Surge висит до прихода `RunResult`. Tide-orchestrator лишь читает `current_apply_id` и ждёт terminal.

**Финальная PG-schema `tides`.**

```sql
CREATE TABLE tides (
    tide_id                TEXT PRIMARY KEY,                              -- ULID
    incarnation_name       TEXT NOT NULL REFERENCES incarnation(name),
    scenario_name          TEXT NOT NULL,
    total_surges           INT  NOT NULL,
    current_surge_index    INT  NOT NULL DEFAULT 0,                       -- 0 = ничего не запущено
    surge_size             INT  NOT NULL,
    target_resolved_souls  JSONB NOT NULL,                                -- snapshot SID[] от старта Tide
    target_coven_override  JSONB,                                          -- coven[] AND-merge со scenario on:
    target_where_override  TEXT,                                           -- CEL-строка AND-merge со scenario where:
    concurrency_override   INT,                                            -- REPLACE scenario serial:
    on_surge_failure       TEXT NOT NULL CHECK (on_surge_failure IN ('abort','continue')),
    status                 TEXT NOT NULL CHECK (status IN (
                              'pending','running','succeeded','failed','partial_failed','cancelled')),
    current_apply_id       TEXT,                                          -- nullable FK apply_runs(apply_id), same-tx invariant
    claimed_by_kid         TEXT,                                          -- nullable; KID-claimer (NULL когда pending)
    last_renewed_at        TIMESTAMPTZ,
    claim_expires_at       TIMESTAMPTZ,
    attempt                INT  NOT NULL DEFAULT 0,                       -- fencing-epoch (parity ADR-027(d))
    started_by_aid         TEXT NOT NULL REFERENCES operators(aid),
    started_at             TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    finished_at            TIMESTAMPTZ,
    summary                JSONB,                                         -- {surges:[{surge_index,apply_id,terminal,...}]}
    CONSTRAINT tides_running_claim_consistency CHECK (
        (status='running' AND claimed_by_kid IS NOT NULL AND claim_expires_at IS NOT NULL)
        OR status <> 'running'
    )
);

-- Recovery-скан (reclaim_tides) — только активные running с истёкшим lease:
CREATE INDEX tides_claim_scan_idx
    ON tides (claim_expires_at)
    WHERE status='running' AND claim_expires_at IS NOT NULL;

-- TideWorker claim — FOR UPDATE SKIP LOCKED по pending:
CREATE INDEX tides_pending_pickup_idx
    ON tides (started_at)
    WHERE status='pending';

-- Apply_runs back-link (один Tide → N apply_runs):
ALTER TABLE apply_runs
    ADD COLUMN tide_id     TEXT REFERENCES tides(tide_id),
    ADD COLUMN surge_index INT;
```

`tides_running_claim_consistency` CHECK гарантирует, что `running` без `claimed_by_kid`/`claim_expires_at` невозможна — Reaper-`reclaim_tides` обязан сбросить оба поля одновременно с переводом в `pending`.

**target_resolved_souls — snapshot, не reference.** JSONB-массив SID[], зафиксированный на старте Tide (после резолва AND-merge target). На 100k SID-ов — 3–5 МБ, терпимо для MVP single-row INSERT. Альтернатива (отдельная таблица `tide_souls` с FK) — отложена в отдельный slice; bench `target_resolved_souls` 100k под нагрузкой PG-write latency должен подтвердить, нужна ли миграция. Snapshot-семантика обязательна: scope Tide не должен «дрожать» от добавления/удаления Soul-ов между Surge-ами (новые Soul-ы появятся в следующем Tide, старые исчезнут из набора — но текущий Tide работает с фиксированным набором, как Salt `-G grain` фиксирует matched-minions перед запуском).

**Финальное множество `tides.status`: 6 значений** — `pending` / `running` / `succeeded` / `failed` / `partial_failed` / `cancelled`. **`orphaned` УБРАН** из enum: Reaper-`reclaim_tides` возвращает протухший Tide в `pending` (parity [ADR-027](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim) — `claimed → planned`), а не в новый отдельный терминал. Visible-side-effect: оператор, опрашивающий `GET /v1/tides/{id}` во время reclaim, увидит `pending` с `attempt > 0` (не путать с свежепринятым `pending` где `attempt=0`); audit-trail `tide.reclaimed` фиксирует факт.

**Failover-flow worked example.**

```
T+0s   Keeper-А TideWorker.ClaimNextTide → tide=X claimed,
       claim_expires_at = NOW() + 60s
T+0s   Tide-orchestrator: surge 4/10 → spawn apply_run Y
       (single TX: INSERT apply_runs + UPDATE tides.current_apply_id=Y, current_surge_index=4)
T+20s  Keeper-А lease renewal: CAS-UPDATE WHERE claimed_by_kid=A AND expires > NOW()
       → 1 row affected, claim_expires_at = NOW() + 60s
T+25s  Keeper-А OOM (in-flight Surge 4 на хостах продолжается, Acolyte ведёт его)
T+60s  Reaper-leader (Keeper-Б): reclaim_tides Reaper-rule срабатывает
       → tide=X status=pending, claimed_by_kid=NULL, attempt++
       → audit tide.reclaimed { tide_id=X, last_renewed_at=T+20s, attempt_after=2 }
T+62s  Keeper-Б TideWorker.ClaimNextTide → tide=X claimed (attempt=2, claim_expires_at=T+122s)
T+62s  Keeper-Б Tide-orchestrator читает строку:
       - current_surge_index=4
       - current_apply_id=Y (если crash был ПОСЛЕ same-tx commit)
                   ИЛИ
       - current_apply_id=NULL, current_surge_index=3 (если crash был ДО commit) → стартует Surge 4 заново
T+62s  AwaitApplyTerminal(Y): subscribe apply:terminal:Y + poll 5s
       Acolyte-recovery (reclaim_apply_runs) подбирает Y если нужно — независимый путь
T+...  apply_run Y terminal → Tide-orchestrator продолжает Surge 5
```

**Что отложено (внутри amendment).**

- **Bench `target_resolved_souls` 100k SID** — проверка PG-write latency на одну `INSERT INTO tides` строку (если > 50ms — переход на отдельную `tide_souls` table со streaming-insert).
- **Tide cancellation (`DELETE /v1/tides/{id}`)** — остаётся в общем «Что отложено пост-MVP» выше. После cancellation invariant: `status='cancelled'` + cancel-policy для current_apply_id.
- **`stop-and-wait` failure-policy** — остаётся в общем «Что отложено».
- **Tide-progress SSE** — остаётся в общем «Что отложено».
