# ADR-040. Tide — invocation-time scope chunking + target-override

> **Status: Superseded by [ADR-043](0043-voyage.md#adr-043-voyage--unified-batch-run) (Voyage), 2026-05-29.**
>
> **The Tide implementation has been REMOVED (Wave 5, 2026-05-30).** The `tides` table was dropped by migration `061`; the `internal/tideorch` + `internal/tide` packages (incl. TideWorker / claim / surge / audit), the `/v1/tides` endpoints, the proto messages `EventTide*`, and the `tide.*` audit family were removed from the code. The invocation-time chunking of scenario batches now lives in Voyage `kind=scenario` ([ADR-043](0043-voyage.md#adr-043-voyage--unified-batch-run)): Surge → Leg, per-Surge state-commit → per-incarnation state-commit (B1), claim+lease failover and snapshot-scope are inherited. Changes on absorption: **(1) invocation-time target-override is REMOVED** — the AND-merge of `target.coven[]`/`target.where` is replaced by choosing a target from an already-resolved RBAC scope (the security intent "cannot widen the scope" is preserved, see [ADR-043](0043-voyage.md#adr-043-voyage--unified-batch-run) point 5). **(2) Preserved** invariants: per-incarnation state-commit (parity with [ADR-009 §7](0009-scenario-dsl.md#adr-009-scenario--the-full-destiny-task-dsl-the-boundary-with-destiny-is-a-recommendation)), claim+lease failover (Acolyte-style PG lease), snapshot-scope of the unit set. **(3) Surge → Leg** (the Voyage batch unit).
>
> **The implementation has been removed — see [ADR-043](0043-voyage.md#adr-043-voyage--unified-batch-run).** Below is the original decision record (history, not an active contract).

**Context.** Scenario-DSL ([ADR-009](0009-scenario-dsl.md#adr-009-scenario--the-full-destiny-task-dsl-the-boundary-with-destiny-is-a-recommendation)) fixes `on:`/`where:`/`serial:` at the **scenario declaration** level in the service's git repo. For mass rollouts to 100k+ Souls in one incarnation this isn't enough — at invocation the operator wants to:
- split the total scope into sequential **waves** (10k × 10 runs with a barrier and audit between them);
- narrow the target by labels/coven + a CEL predicate at launch time (but NOT widen it — security);
- override the concurrency knob (a runtime execution parameter, not a declared invariant).

Comparable batch/targeting controls in other tools are **invocation-level**, not declaration-level.

**Decision.**

1. **A new entity Tide** (top-level entity) and **Surge** (one wave). The name was chosen via propose-and-wait, architect verdict `ae5494e6cbb491804` + PM decisions 2026-05-27. Tide = an invocation-instance of a mass scenario run, split into sequential **Surges**. 100k Souls + wave_size=10k = 1 Tide × 10 Surges.

2. **Surge = one wave = one `apply_run`.** Surges run sequentially (a failure-policy governs moving to the next one). Each Surge is a full-fledged scenario run with its own barrier + state-commit, **paired with [ADR-009 §7](0009-scenario-dsl.md#adr-009-scenario--the-full-destiny-task-dsl-the-boundary-with-destiny-is-a-recommendation)** (barrier/state-commit per-Surge, not per-Tide).

3. **PG table `tides`** (column details — W-1 after the pre-W-2 failover architect-consult):
   - `tide_id` ULID PK;
   - `incarnation_name` FK on `incarnation(name)`;
   - `scenario_name`;
   - `target_coven_override` (text[]) + `target_where_override` (a CEL string) — invocation-time labels/coven and a per-host predicate;
   - `wave_size` int + `on_surge_failure` enum (`abort`|`continue`) + `concurrency_override` int (nullable, REPLACE the scenario `serial:`);
   - `claimed_by_kid` (nullable) + lease fields (Acolyte-style — for failover, see the invariants below);
   - `status` enum (`pending`|`running`|`succeeded`|`partial_failed`|`failed`|`cancelled`|`orphaned`);
   - `started_by_aid` FK `operators(aid)`;
   - `summary` jsonb (surge-results: per-Surge apply_id + status + souls_count);
   - timestamps (`started_at`, `finished_at`, `last_renewed_at`).

4. **`apply_runs.tide_id` (nullable FK) + `surge_index` (int).** Single-run apply_runs (without a Tide, the classic `POST /v1/incarnations/.../scenarios/.../run` without a `wave` field) keep working backward-compat — `tide_id IS NULL`. The composition `(tide_id, surge_index)` identifies a Surge within a Tide.

5. **API — an opt extension of the existing endpoint.** `POST /v1/incarnations/{name}/scenarios/{scenario}` (see [operator-api.md](../keeper/operator-api.md)) accepts additional opt fields:

   ```json
   {
     "input":  { ... },
     "target":      { "coven": ["prod-eu"], "where": "soulprint.self.os.family == \"debian\"" },
     "wave":        { "size": 10000, "on_failure": "abort" },
     "concurrency": 100
   }
   ```

   Without `wave` — the old behavior (one `apply_run`, classic single-run). With `wave` — the Tide-orchestrator (W-2). Backward-compat — only-add fields, existing clients don't break.

6. **CLI (`soulctl`):** flags `--wave-size`, `--wave-on-failure {abort|continue}`, `--where <CEL>`, `--target-coven <name>` (repeatable), `--concurrency <N>`, `--invocation-file <yaml>` (the full invocation payload as a file).

7. **MCP:** the existing tool `keeper.incarnation.run` is extended with the same opt fields (`target`/`wave`/`concurrency`). **NOT** a new tool — forward-compat.

**Invariants.**

- **AND-merge target-override (security).** The invocation-override `target.coven[]`/`target.where` only **narrows** the scope (an intersection with the scenario's `on:`/`where:`), never widens it. The operator cannot use invocation to step outside the declared scenario target. **REPLACE is rejected** as a violation of [ADR-009 §3](0009-scenario-dsl.md#adr-009-scenario--the-full-destiny-task-dsl-the-boundary-with-destiny-is-a-recommendation) — otherwise the scenario author loses control over the maximum blast radius.
- **Glob/regex matching via CEL.** `target.where: 'sid.glob("prod-*")'` (shell-glob) or `target.where: 'sid.matches("^db-[0-9]+$")'` (stdlib regex) — ordinary CEL predicates; a single grammar, no separate DSL. Composition with facts: `target.where: 'os.family == "debian" && sid.glob("web-*")'`. See [templating.md §2.3](../templating.md#23-registered-cel-functions-starting-minimum).
- **Multi-label semantics within one list (resolved 2026-05-27).** Listing labels, both in scenario `on: [a, b]` and in invocation `target.coven: [a, b]` — is an **AND-intersection**. A host is routed to only if it has **ALL** the listed labels. Composition of scenario-on ↔ invocation-target — is also AND. This provides a fail-closed invariant on two levels: neither the declaration (scenario) nor invocation can widen the scope by listing labels. For an OR case, the operator uses `target.where: CEL` with an explicit predicate. Symmetric to [orchestration.md §3](../scenario/orchestration.md) (the single source of truth for the DSL text).
- **REPLACE for the concurrency-override.** `concurrency` is a runtime execution knob, not a declared invariant; REPLACE is the natural contract (the operator knows the specifics of the current cluster hardware). Without an override — the scenario `serial:` is preserved.
- **Per-Surge state-commit, parity with [ADR-009 §7](0009-scenario-dsl.md#adr-009-scenario--the-full-destiny-task-dsl-the-boundary-with-destiny-is-a-recommendation).** Each Surge is a separate scenario run with its own barrier; `state_changes` are committed after a successful Surge, not after the whole Tide. Otherwise the barrier invariant of the DSL core would be violated.
- **The Tide-orchestrator is stateless with respect to a specific Keeper instance** (an Acolyte-style lease in PG for failover). The pre-W-2 architect-consult fixes the mechanics (lease-renew period, the reclaim rule in Reaper, the fencing-epoch).
- **`run_once:` + Tide — abort with an error at validation.** A scenario with a `run_once:` step is incompatible with splitting into waves (logically run_once = exactly one target host; there's nothing to split into waves). Validation on `POST` rejects the request (`422 validation-failed`, error code `tide_run_once_conflict`).
- **Cross-incarnation Tide is forbidden by the grammar** — symmetric to [ADR-008](0008-coven-stable-tags.md#adr-008-coven--stable-logical-tags-only). A Tide is bound to a single incarnation.
- **Surges are sequential.** The next Surge starts only after the previous one reaches a **terminal** status (`success`/`failed`/`cancelled`/`orphaned`/`no_match`, see [ADR-027](0027-apply-work-queue.md#adr-027-apply-execution-model--work-queue--claim-acolyte-pool-ward-claim) FINDING-01). Parallel surges are a separate entity, not considered in the MVP.
- **RBAC reuse — `incarnation.run`.** The same permission as the classic single-run; no new permission is introduced — a Tide is the same operation, just split into waves. A granular Tide permission (`tide.create`/`tide.cancel`) — deferred until an operator request.

**Rejected alternatives.**

- (a) **Client-side N requests** (the operator itself loops calling `POST .../scenarios/.../run` with different `target` filters). No Tide atomicity, no audit cohesion, no cluster-side progress view, no cancel capability. The decision shifts orchestration onto the client — that's Keeper's job.
- (b) **REPLACE target-override.** A security regression — the operator could widen the scope beyond the declared scenario `on:`. Violates [ADR-009 §3](0009-scenario-dsl.md#adr-009-scenario--the-full-destiny-task-dsl-the-boundary-with-destiny-is-a-recommendation).
- (c) **Per-task wave-size** (different scenario tasks split by different widths). Requires reworking Acolyte dispatch (a per-task `RenderedTask`, see the open Q "per-task waves" in [ADR-009](0009-scenario-dsl.md#adr-009-scenario--the-full-destiny-task-dsl-the-boundary-with-destiny-is-a-recommendation)). Tide operates at the scope level, not the task level.
- (d) **Aggregate state-commit** (one commit for the whole Tide). Would break the [ADR-009 §7](0009-scenario-dsl.md#adr-009-scenario--the-full-destiny-task-dsl-the-boundary-with-destiny-is-a-recommendation) barrier invariant — Surge #3 fails, but the state-commit of Surges #1–2 already happened? actually it wouldn't have happened, but the first two waves have already changed hosts. Per-Surge commit is the only correct semantics.
- (e) **`stop-and-wait` failure-policy in the MVP** (wait for a manual operator decision after a failed Surge). Operationally complex (needs a polling endpoint for the "awaiting decision" state, a cancel endpoint to cancel the wait, a TTL to protect against a forgotten Tide). MVP — `abort` + `continue`; `stop-and-wait` — deferred post-MVP.
- (f) **A new MCP tool `keeper.tide.run`.** Doubles the tool catalog, forward-compat isn't needed (an existing-tool only-add of fields doesn't break old clients).

**What is deferred post-MVP.**

- **Tide cancellation mid-flight.** `DELETE /v1/tides/{tide_id}` + a cancel-policy (how a currently-running Surge is handled — abort immediately / let it finish / abort+wait). Requires decisions on Acolyte-stream cancel semantics. The audit event `tide.cancelled` is reserved.
- **`stop-and-wait` failure-policy** — a manual operator retry/abort flag after a failed Surge.
- **Tide-progress SSE for the UI** (a live feed of surge completions). A separate slice after the core orchestrator stabilizes.
- **Dynamic resize of wave-size mid-run** (the operator increases/decreases wave_size in flight). An operational nicety, not blocking for the MVP.
- **Mid-scenario pause/resume.** A separate subsystem (needs scenario-runner state snapshots), out of Tide's scope.

**Amendment 2026-05-27. Failover-resilience mechanics.** Architect-consult `a112874e417338ff5` fixed the mechanics allowing any Keeper instance to pick up a Tide via a PG claim+lease (a Tide is stateless with respect to a specific instance). Parity with Acolyte/Ward ([ADR-027](0027-apply-work-queue.md#adr-027-apply-execution-model--work-queue--claim-acolyte-pool-ward-claim)) for `apply_runs`, but one level higher — above the work-queue. This amendment CLOSES the PG-schema detailing and mechanics deferred in the original point 3 of ADR-040; the specific column names and SQL are below.

**Architectural invariants.**

1. **A TideWorker pool — a separate worker pool**, **not** shared with Acolyte (deadlock prevention: with `acolytes: 1` + 1 Tide holding an Acolyte slot, a Surge inside the Tide would have nowhere to land). Config `keeper.tide_workers: N` (default `1`, recommendation `max ≤ acolytes-1`). The worker loop is simple: `claim → orchestrate → poll/sleep`, without complex wake logic (Tide operations are rare and large, not hot-path).

2. **The claim pattern (parity with [ADR-027(d)](0027-apply-work-queue.md#adr-027-apply-execution-model--work-queue--claim-acolyte-pool-ward-claim))**:
   - `SELECT … FOR UPDATE SKIP LOCKED` for an atomic pickup from `tides` with `status='pending'`.
   - `attempt++` on every claim for the fencing-epoch (if the orchestrator logic ever spawns a duplicate, a late `RunResult` handler cuts it off by epoch).
   - **A PG-based lease.** A Redis lease is rejected — a single-leader bottleneck; a PG advisory lock is rejected — doesn't self-clean by TTL (hangs until a Postgres connection drop, unreliable during a Keeper OOM with a stuck connection in pgbouncer).
   - **TTL 60s + renewal every 20s** via a CAS-UPDATE with the guard `WHERE tide_id=$1 AND claimed_by_kid=$me AND claim_expires_at > NOW()`. 0 rows affected → `ErrLeaseLost`, the TideWorker immediately abandons the work (a Surge-spawn is not allowed to continue without confirmed ownership).

3. **The `reclaim_tides` Reaper rule** ([keeper/reaper.md](../keeper/reaper.md)):
   ```sql
   UPDATE tides
      SET status='pending',
          claimed_by_kid=NULL,
          last_renewed_at=NULL,
          attempt=attempt+1
    WHERE status='running'
      AND claim_expires_at < NOW()
   ```
   Default `enabled: false` (an operational gate — turned on after the fencing of the Souls fleet stabilizes, parity with [ADR-027(i)](0027-apply-work-queue.md#adr-027-apply-execution-model--work-queue--claim-acolyte-pool-ward-claim) GATE-1). The audit event `tide.reclaimed { tide_id, last_renewed_at, attempt_after }` is written per-row.

4. **The same-transaction invariant (parity with [ADR-027(j)](0027-apply-work-queue.md#adr-027-apply-execution-model--work-queue--claim-acolyte-pool-ward-claim) single-winner)**: spawning `apply_run` + `UPDATE tides SET current_apply_id=Y, current_surge_index=N` in **one PG transaction**. A crash in the middle → PG rolls it back, the `apply_run` doesn't exist, Keeper-B on pickup sees `current_surge_index=N-1`, restarts surge N. Without a same-tx, two Keepers could spawn two `apply_run`s for one Surge.

5. **AwaitApplyTerminal — a hybrid mode**:
   - **Redis-subscribe** `apply:terminal:<apply_id>` for responsiveness (the terminal handler publishes when `apply_run.status` transitions to terminal).
   - **Periodic poll** of `apply_runs.status WHERE apply_id=$current` every 5s as a fallback for a lost pub/sub signal (an HA cluster of N=3 Keeper instances, a `RunResult` can arrive at any instance, a subscription to a local pub can miss it).
   - The Tide-orchestrator on Keeper-B, on pickup, does NOT touch the existing `apply_run` — Acolyte recovery works its own way through [`reclaim_apply_runs`](../keeper/reaper.md) (if enabled), or the Surge hangs until a `RunResult` arrives. The Tide-orchestrator only reads `current_apply_id` and waits for terminal.

**Final PG schema `tides`.**

```sql
CREATE TABLE tides (
    tide_id                TEXT PRIMARY KEY,                              -- ULID
    incarnation_name       TEXT NOT NULL REFERENCES incarnation(name),
    scenario_name          TEXT NOT NULL,
    total_surges           INT  NOT NULL,
    current_surge_index    INT  NOT NULL DEFAULT 0,                       -- 0 = nothing started yet
    surge_size             INT  NOT NULL,
    target_resolved_souls  JSONB NOT NULL,                                -- SID[] snapshot from Tide start
    target_coven_override  JSONB,                                          -- coven[] AND-merge with scenario on:
    target_where_override  TEXT,                                           -- CEL string AND-merge with scenario where:
    concurrency_override   INT,                                            -- REPLACE the scenario serial:
    on_surge_failure       TEXT NOT NULL CHECK (on_surge_failure IN ('abort','continue')),
    status                 TEXT NOT NULL CHECK (status IN (
                              'pending','running','succeeded','failed','partial_failed','cancelled')),
    current_apply_id       TEXT,                                          -- nullable FK apply_runs(apply_id), same-tx invariant
    claimed_by_kid         TEXT,                                          -- nullable; the claiming KID (NULL when pending)
    last_renewed_at        TIMESTAMPTZ,
    claim_expires_at       TIMESTAMPTZ,
    attempt                INT  NOT NULL DEFAULT 0,                       -- fencing-epoch (parity ADR-027(d))
    started_by_aid         TEXT NOT NULL REFERENCES operators(aid),
    started_at             TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    finished_at            TIMESTAMPTZ,
    summary                JSONB,                                         -- {surges:[{surge_index,apply_id,terminal,...}]}
    CONSTRAINT tides_running_claim_consistency CHECK (
        (status='running' AND claimed_by_kid IS NOT NULL AND claim_expires_at IS NOT NULL)
        OR status <> 'running'
    )
);

-- Recovery scan (reclaim_tides) — only active running rows with an expired lease:
CREATE INDEX tides_claim_scan_idx
    ON tides (claim_expires_at)
    WHERE status='running' AND claim_expires_at IS NOT NULL;

-- TideWorker claim — FOR UPDATE SKIP LOCKED over pending:
CREATE INDEX tides_pending_pickup_idx
    ON tides (started_at)
    WHERE status='pending';

-- apply_runs back-link (one Tide → N apply_runs):
ALTER TABLE apply_runs
    ADD COLUMN tide_id     TEXT REFERENCES tides(tide_id),
    ADD COLUMN surge_index INT;
```

The `tides_running_claim_consistency` CHECK guarantees that `running` without `claimed_by_kid`/`claim_expires_at` is impossible — the Reaper's `reclaim_tides` must reset both fields at the same time as transitioning to `pending`.

**`target_resolved_souls` — a snapshot, not a reference.** A JSONB array of SID[], fixed at Tide start (after resolving the AND-merge target). At 100k SIDs — 3–5 MB, tolerable for a single-row MVP INSERT. The alternative (a separate `tide_souls` table with an FK) — deferred to a separate slice; a bench of `target_resolved_souls` at 100k under load must confirm whether a migration is needed for PG-write latency. Snapshot semantics are mandatory: a Tide's scope must not "jitter" from Souls being added/removed between Surges (new Souls will appear in the next Tide, old ones will disappear from the set — but the current Tide works with a fixed set, frozen before launch).

**The final `tides.status` set: 6 values** — `pending` / `running` / `succeeded` / `failed` / `partial_failed` / `cancelled`. **`orphaned` is REMOVED** from the enum: the Reaper's `reclaim_tides` returns a stale Tide to `pending` (parity with [ADR-027](0027-apply-work-queue.md#adr-027-apply-execution-model--work-queue--claim-acolyte-pool-ward-claim) — `claimed → planned`), not to a new separate terminal. Visible side effect: an operator polling `GET /v1/tides/{id}` during a reclaim will see `pending` with `attempt > 0` (don't confuse with a freshly-accepted `pending` where `attempt=0`); the `tide.reclaimed` audit trail records the fact.

**Failover-flow worked example.**

```
T+0s   Keeper-A TideWorker.ClaimNextTide → tide=X claimed,
       claim_expires_at = NOW() + 60s
T+0s   Tide-orchestrator: surge 4/10 → spawn apply_run Y
       (single TX: INSERT apply_runs + UPDATE tides.current_apply_id=Y, current_surge_index=4)
T+20s  Keeper-A lease renewal: CAS-UPDATE WHERE claimed_by_kid=A AND expires > NOW()
       → 1 row affected, claim_expires_at = NOW() + 60s
T+25s  Keeper-A OOMs (the in-flight Surge 4 continues on the hosts, Acolyte is driving it)
T+60s  Reaper-leader (Keeper-B): the reclaim_tides Reaper rule fires
       → tide=X status=pending, claimed_by_kid=NULL, attempt++
       → audit tide.reclaimed { tide_id=X, last_renewed_at=T+20s, attempt_after=2 }
T+62s  Keeper-B TideWorker.ClaimNextTide → tide=X claimed (attempt=2, claim_expires_at=T+122s)
T+62s  Keeper-B Tide-orchestrator reads the row:
       - current_surge_index=4
       - current_apply_id=Y (if the crash was AFTER the same-tx commit)
                   OR
       - current_apply_id=NULL, current_surge_index=3 (if the crash was BEFORE the commit) → restarts Surge 4
T+62s  AwaitApplyTerminal(Y): subscribe apply:terminal:Y + poll 5s
       Acolyte recovery (reclaim_apply_runs) picks up Y if needed — an independent path
T+...  apply_run Y goes terminal → the Tide-orchestrator continues with Surge 5
```

**What is deferred (within this amendment).**

- **Bench `target_resolved_souls` at 100k SIDs** — checking PG-write latency for a single `INSERT INTO tides` row (if > 50ms — move to a separate `tide_souls` table with streaming-insert).
- **Tide cancellation (`DELETE /v1/tides/{id}`)** — remains in the general "What is deferred post-MVP" above. Post-cancellation invariant: `status='cancelled'` + a cancel-policy for `current_apply_id`.
- **`stop-and-wait` failure-policy** — remains in the general "What is deferred."
- **Tide-progress SSE** — remains in the general "What is deferred."

**Post-MVP scope (an explicit deferred list).**

- **Tide cancellation + stop-and-wait.** `DELETE /v1/tides/{tide_id}` + a cancel-policy (abort immediately / let it finish / abort+wait). A `cancel_requested` field already exists on `apply_runs` (a cluster-wide apply_run cancel); for Tide-level cancel — a separate slice. - **Failover-recovery of an in-flight Surge.** When a Tide is reclaimed onto another Keeper instance, the previous in-flight Surge **does not get** a per-Surge state-commit from the new orchestrator (the normative file `tide-state-commit.md §4` was never created; Tide was abolished — see [ADR-043](0043-voyage.md#adr-043-voyage--unified-batch-run)) and **will not make it** into the final Summary. The Surge's apply_runs continue to terminal independently via the Acolyte+RunResult pipeline; incarnation.state stays pre-Surge. Full per-Surge failover semantics — requires a `surge_runs` journal table or extra columns on `tides`, deferred.
- **Per-SID state-commit within a failed Surge** (failure_policy=continue with a partial-fail surge — a commit for the subset of successfully completed hosts of one wave). Requires per-host classification in `SurgeOutcome` + an extension of the `SurgeStateCommitter` interface. The normative file `tide-state-commit.md §2.1` was never created; Tide was abolished — see [ADR-043](0043-voyage.md#adr-043-voyage--unified-batch-run).
- **Audit-log scaling.** The volume of `tide.*` events in a mass Tide run (thousands of surge-completed events per Tide × N parallel Tides) — is written to the shared [`audit_log`](0022-audit-pipeline.md#adr-022-audit-pipeline-storage-schema-retention) without a separate partition. Backlog §26 (Audit-pipeline scaling) applies symmetrically to the other `keeper_internal` sources.
- **Bench `target_resolved_souls` at 100k SID.** The PG-write latency of `INSERT INTO tides` with a 3–5 MB jsonb array — if > 50ms on a steadily loaded PG, move to a separate `tide_souls` table with streaming-insert.
