# ADR-009. Scenario — полная DSL задач destiny; граница с destiny — рекомендация

- **Контекст.** Изначально действовал инвариант изоляции: scenario мог только `apply: { destiny: … }` (плюс `target`/оркестрация), но **не** `module:` напрямую — вся «работа» обязана была жить в изолированной, версионируемой, molecule-тестируемой destiny. На практике это вынуждало заводить отдельный destiny-репо ради одно-двух-шаговой инлайн-операции (probe роли, разовая команда, диагностика), которая никогда не переиспользуется. Инвариант мешал даже read-only-шагам (probe), без которых не реализуется волатильная роль из [ADR-008](0008-coven-stable-tags.md#adr-008-coven--только-стабильные-логические-теги).
- **Решение.**
  - **Инвариант «scenario без `module:`» снят полностью** (не только для read-only). Scenario получает **всё DSL-ядро задач destiny** из [`docs/destiny/tasks.md`](../destiny/tasks.md): `module:` (включая изменяющие модули), `templates`, task-level `vars:`, `register:`, `loop:`, `block:`, `parallel:`, `onchanges:`/`onfail:`/`require:`, `changed_when:`/`failed_when:`, `retry:`, `timeout:` — плюс оркестрационный слой сверху (`on:`/`where:`-таргетинг, `apply: { destiny: … }`, `state_changes`).
  - **DSL-ядро не дублируется.** Источник правды по ядру — `docs/destiny/tasks.md`; scenario-дельта — отдельный нормативный документ [`docs/scenario/orchestration.md`](../scenario/orchestration.md).
  - **destiny остаётся** самостоятельной сущностью: переиспользуемый, независимо-версионируемый (git ref, [ADR-007](0007-versioning-git-ref.md#adr-007-версионирование-артефактов--через-git-ref-а-не-через-поле-в-манифесте)), изолированный, molecule-тестируемый кирпич.
  - **Граница destiny / scenario — рекомендация, не стена.** Критерий выноса в destiny — три «да»: (1) переиспользование, (2) нужна molecule-идемпотентность, (3) изолируемо. Все три → выноси в destiny через `apply:`; иначе допустима инлайн-`module:`-реализация в scenario.
  - **Инвариант barrier / state-commit (не опция).** Коммит `incarnation.state` (`state_changes`) — это **cross-host final-barrier ПОСЛЕ безусловного ожидания всех parallel-задач всех хостов прогона**. Если хоть одна finally-failed → state **не** коммитится → `error_locked`. Это расширение final-barrier из [destiny/tasks.md §6](../destiny/tasks.md#6-параллелизм-parallel-true) с одного хоста на cross-host scenario-уровень. Подробнее — [`docs/scenario/orchestration.md`](../scenario/orchestration.md#7-инвариант-barrier--state-commit).
  - **`state_changes` несёт источник значения, не только allow-list.** `sets` — это map `<поле> → <CEL-выражение>`: декларация задаёт **и** какое поле `incarnation.state` обновить, **и** откуда взять значение (рендерится Keeper-side в scenario-CEL-окружении после барьера). Прежняя трактовка `sets` как плоского allow-list имён (только «какие поля разрешено писать», без источника) — заменена: без источника `incarnation.state` оставался пустым при успешном прогоне. Cross-host свёртка `sets` — last-wins по сортировке `SID`. **Контекст `sets` — `input.* / incarnation.* / soulprint.self.* / register.*`.** `register.*` (per-host probe-результат) пробрасывается через обратный канал `TaskEvent.register_data` → таблица `apply_task_register` → per-host register-карта после барьера (слайс 2 полной грамматики). `vars.* / essence.* / state.* / soulprint.hosts` и ключи `appends`/`modifies` (per-host коллекции) — future-расширение полной грамматики, недоступны (ошибка рендера → `error_locked`). **Грамматика расширена [ADR-057](0057-state-changes-crud-verbs.md) (см. amendment ниже):** `state_changes` — упорядоченный список CRUD-глаголов (`set`/`add`/`modify`/`remove`+`foreach`), map-форма (`sets`/`appends`/`modifies`) — deprecated. Грамматика — [`docs/scenario/orchestration.md §7.1`](../scenario/orchestration.md#71-грамматика-state_changes--список-crud-операций).
  - **Изоляция и `output:` destiny.** destiny публикует собственный результат через декларированный top-level `output:`; это симметрично `input:` и изоляцию destiny **не** нарушает (destiny не читает scenario-scope — `essence`/`register`/`vars`/`input` родителя/`incarnation.*`/топологию прогона). Спецификация — [`docs/destiny/output.md`](../destiny/output.md). **Уточнение (см. Amendment 2026-06-18):** стабильные self-факты целевого хоста (`soulprint.self.*`) — это **не** scenario-scope, а per-host свойство самого хоста, поэтому они доступны destiny-CEL; запрещён только cross-host (`soulprint.hosts`/`soulprint.where`).
- **Обоснование.** Один язык задач на оба слоя — нет двух диалектов, автор сценария не учит второй синтаксис. Probe (read-only `module:` + `register:`) становится естественным, чем и реализуется волатильная роль ([ADR-008](0008-coven-stable-tags.md#adr-008-coven--только-стабильные-логические-теги)). destiny не теряет свойств — она остаётся выбором по рекомендации, а не по принуждению.
- **Trade-off / риски.**
  - **Расширение scope [ADR-003](0003-destiny-format.md#adr-003-формат-destiny--yaml-с-типизированной-схемой-cuejson-schema).** Безопасный шаблонизатор отдельной фазой теперь обслуживает и scenario (`where:`-предикаты, `on:`-литералы, шаблоны). Этот вопрос (выбор шаблонизатора) закрыт [ADR-010](0010-templating.md#adr-010-шаблонизатор-cel-для-yaml-выражений-go-texttemplate-для-файлов). До ADR-010 он блокировал имплементацию scenario, не только destiny.
  - **Риск эрозии [ADR-007](0007-versioning-git-ref.md#adr-007-версионирование-артефактов--через-git-ref-а-не-через-поле-в-манифесте).** Инлайн-`module:`-мутации в scenario минуют независимое git-версионирование destiny (точечная правка `scenario/<name>/main.yml` не получает тег и ревью отдельного destiny-репо). **Смягчение:** граница-рекомендация выше + backlog lint-warn в [`docs/soul-lint.md`](../soul-lint.md) («инлайн-мутация в scenario без выноса в destiny»). Осознанно принятая плата за простоту типовых операций.
- **Связанные решения.** [ADR-008](0008-coven-stable-tags.md#adr-008-coven--только-стабильные-логические-теги) (probe вместо роли-как-Coven — реализуется именно снятием этого инварианта). Полная спецификация дельты — [`docs/scenario/`](../scenario/README.md).

**Amendment 2026-05-27 (invocation-time scope chunking — Tide).** [ADR-040](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override) вводит новую top-level сущность **Tide** — invocation-instance, разбивающую массовый прогон scenario на последовательные волны (**Surges**) с опциональным target/wave/concurrency-override. Принципиально:

- **Tide НЕ часть scenario-DSL** — она не записывается в YAML scenario, а параметризует **invocation** (HTTP/MCP request на `POST /v1/incarnations/{name}/scenarios/{scenario}`). Scenario-DSL ADR-009 остаётся неизменным; новые опт-поля (`target` / `wave` / `concurrency`) живут только в payload запроса, не в `main.yml`.
- **Декларация vs invocation — два уровня.** `serial:` / `run_once:` / `on:` / `where:` фиксируют разбиение и таргет **на уровне декларации scenario** (git-репо сервиса, авторское намерение). Tide добавляет **invocation-level** оркестрационный слой **над** scenario-runner-ом — оператор при запуске может разбить scope на waves, сузить таргет, переопределить concurrency, не меняя scenario. См. [`docs/scenario/orchestration.md §2.2.3`](../scenario/orchestration.md).
- **Совместимость с scenario-DSL:**
  - scenario без `run_once:` совместим с Tide безусловно.
  - scenario с `run_once: true` **несовместим** с invocation `wave` (`run_once` = ровно один хост таргета, разбивать на waves нечего) — ошибка валидации `tide_run_once_conflict` (422), [`docs/keeper/operator-api.md`](../keeper/operator-api.md).
  - scenario `concurrency:` (`serial:`) может быть **переопределён** invocation `concurrency:` (REPLACE-семантика — runtime-knob, не declared invariant).
- **Per-Surge barrier/state-commit — parity §7.** Каждый Surge — полноценный scenario-прогон со своим cross-host barrier и state-commit; barrier-инвариант §7 расширяется на каждый Surge независимо, а не на весь Tide целиком. Нормативно — механика per-Surge state-commit Tide (отдельный файл `docs/scenario/tide-state-commit.md` не создавался; Tide упразднён — см. [ADR-043](0043-voyage.md#adr-043-voyage--унифицированный-батчевый-прогон), per-Surge → per-incarnation state-commit).
- **AND-merge target-override (security-инвариант).** invocation `target.coven[]` / `target.where` только **сужают** scope (пересечение со scenario `on:` / `where:`), не расширяют. Оператор не может через invocation выйти за рамки декларированного scenario-таргета. Симметрично multi-label AND внутри `on:` — единый источник правды текста DSL [`docs/scenario/orchestration.md §3`](../scenario/orchestration.md).
- **`soulprint.hosts` / `incarnation.host_count` в Tide.** Аксессоры scenario-context резолвятся per-Surge от **target_resolved_souls** snapshot Tide (зафиксированный AND-merge на старте Tide), а не от исходного roster incarnation — каждый Surge видит только свою долю.

Цели и обоснование — [ADR-040](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override).

**Amendment 2026-05-29 (§7 — per-incarnation state-commit в Voyage = parity per-Surge).** [ADR-043](0043-voyage.md#adr-043-voyage--унифицированный-батчевый-прогон) (Voyage) поглощает Tide режимом `kind=scenario`. Подход **B1**: батч (**Leg**) = N **инкарнаций**, каждая инкарнация = полноценный scenario-run со своим cross-host barrier и **per-incarnation state-commit**. Это **прямая parity** per-Surge state-commit ([ADR-040](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override)) и сохраняет barrier-инвариант §7 без ослабления: `incarnation.state` коммитится только после успешного barrier-а **этой** инкарнации, частичный коммит (часть хостов инкарнации в одном Leg, остальные — в следующем) **запрещён**. Альтернатива B2 (Leg режет scope по хостам поперёк нескольких инкарнаций) **отвергнута именно потому, что ломает §7** — см. [ADR-043](0043-voyage.md#adr-043-voyage--унифицированный-батчевый-прогон) пункт 3. Внутри-инкарнационные волны по хостам остаются отдельной осью через scenario `serial:` (declaration-level), как и было.

**Amendment 2026-05-29 (Choir — additive `choirs[]` в scenario-аксессорах).** [ADR-044](0044-choir.md#adr-044-choir--именованная-топология-хостов-внутри-инкарнации) (Choir) добавляет к scenario-context-у **additive** поле `choirs[]` (список имён Choir-ов хоста в текущей инкарнации): `soulprint.self.choirs` (per-host, текущий хост) и `soulprint.hosts[].choirs` (cross-host аксессор хостов прогона, [`docs/scenario/orchestration.md §4.1`](../scenario/orchestration.md) — место будущей нормативной правки при S-T4). Это additive-расширение существующих стабильных полей `soulprint.hosts` (covens/sid/network/os) — не меняет грамматику `on:`/`where:`, доступно в `where:`-предикатах **без probe** (Choir — стабильный declared-факт, не волатильная роль). `on: [choir]` НЕ вводится в MVP (резолвер `on:` остаётся coven-only; сужение по Choir — через `where:`). Поглощённая declared-роль `soulprint.hosts[].role` ([ADR-044](0044-choir.md#adr-044-choir--именованная-топология-хостов-внутри-инкарнации) пункт 2) питается из Choir (через Voice), а не из `spec.hosts[].role`. Нормативная правка orchestration.md — слайсом S-T4, этот amendment её только декларирует.

**Amendment 2026-06-12 (опциональное поле задачи `id:` — стабильный адрес для per-task-changed-алертов).** Вводится новое **опциональное** поле DSL-ядра задачи `id:` — стабильный адрес задачи, по которому подписка из [ADR-052](0052-herald-notifications.md#adr-052-herald--tiding--уведомления-о-событиях-прогонов) (§h) адресует событие «эта таска изменила state» (per-task-changed-алерт). Решения:

- **Формат значения = register-формат** (тот же regex `^[a-z][a-z0-9_]*$`, что у `register:`): id и register живут в **одном адресном пространстве** подписки на per-task-changed-события, поэтому форма едина.
- **Опционально.** Отсутствие `id:` у задачи — норма; существующие scenario/destiny не ломаются.
- **`id:` ⊕ `register:` — взаимоисключимы.** Задача с `register:` уже адресуема по нему → `id:` избыточен и создаёт двусмысленность адреса. Оба ключа на одной задаче — ошибка валидации (`id_register_conflict`). `id:` предназначен для задач **без** `register:`.
- **Pilot-ограничение: `id:` только на module-задаче.** В первом заходе `id:` допустим лишь на module-задаче (у неё есть собственный changed-сигнал). На `block:`/`include:`-задаче — ошибка валидации (`id_unsupported_target`); снятие ограничения — отдельным заходом при запросе.
- **Уникальность `id:` по прогону** (после flatten `include`/`block`) — **в этом amendment не фиксируется**; это cross-ref-проверка линта (нужен flatten-контекст), отдельный слайс. Здесь — только per-task-формат + взаимоисключение с register + ограничение на module.
- **Резолв `id:` в RenderedTask, формирование событий и Tiding-привязка** — следующие слайсы, в эту правку не входят (только грамматика поля и валидация формы).

Per-task changed-адресация (адресное пространство `register ∪ id`) фиксируется со стороны уведомлений в [ADR-052](0052-herald-notifications.md#adr-052-herald--tiding--уведомления-о-событиях-прогонов) (§h). Документ-источник правды по полю задачи — [`docs/destiny/tasks.md §3`](../destiny/tasks.md#3-полный-список-блоков-задачи).

**Amendment 2026-06-18 (CEL-env прохода destiny дополнительно регистрирует read-only `soulprint.self.*`).** Совместно с [ADR-010](0010-templating.md#adr-010-шаблонизатор-cel-для-yaml-выражений-go-texttemplate-для-файлов) (amendment той же даты). Инвариант изоляции destiny **ослабляется**:

- **CEL-env прохода destiny дополнительно регистрирует read-only `soulprint.self.*`** — стабильный слой `SoulprintFacts` ([ADR-018](0018-soulprint-typed.md#adr-018-soulprint-typed-схема-mvp)): `os.*` (включая `arch`), `kernel`/`cpu`/`memory`/`network`, корневые `sid`/`hostname` и registry-проекции (`covens`/`choirs`/`role`). Источник — тот же per-host soulprint целевого хоста, что Keeper уже собирает для render_context `.tmpl`-фазы.
- **`soulprint.hosts` / `soulprint.where(...)` остаются scenario-only** (топология прогона — cross-host) — обращение к ним в destiny-проходе по-прежнему **ошибка изоляции**. Граница: self-факты целевого хоста — да; топология прогона — нет.
- **Симметрия с `.tmpl` render_context** ([ADR-012(d)](0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add)): `self` уже течёт в `core.file.rendered.render_context` per-host; теперь `soulprint.self.<path>` в CEL `.yml` ≡ `.self.<path>` в `.tmpl` — единая точка правды по self.
- **`apply: input:` остаётся валидным каналом** (не удаляется): передача топологии прогона (`soulprint.hosts.where(...)`) в destiny по-прежнему только через явный `apply: input:`-проброс. Идиома bootstrap-create (master discovery через apply:input) не ломается.
- **Обоснование.** `soulprint.self.*` — per-host свойство самого хоста, на котором destiny исполняется (а не scenario-scope: не `essence`/`register`/`vars`/`input` родителя). Destiny и так применяется к конкретному хосту; видеть его стабильные факты (arch для выбора пакета/архива, family для путей) — естественно и убирает лишний boilerplate `apply: input: { arch: "${ soulprint.self.os.arch }" }`. Безопасность не страдает: self read-only, без I/O, без cross-host.

Нормативные правки — [`docs/templating.md §3.2/§7`](../templating.md), [`docs/destiny/tasks.md §10`](../destiny/tasks.md#10-шаблонный-контекст) (уже отражал self в destiny-контексте). Реализация: `soulprint.self` в destiny-проходе доступен через render-pipeline (`keeper/internal/render`, `Vars.SoulprintSelf`), `soulprint.hosts`/`soulprint.where` отсекаются `AllowHosts=false` (`shared/cel`).

**Amendment ([ADR-056](0056-staged-render-passage.md), staged-render — Passage). Статус → amended.** Per-RUN min-width `serial:` (§2.2.1 orchestration.md) и open Q §8 «per-task гранулярность» формулировались на ограничении «один `ApplyRequest` на хост со всеми задачами скопом» — per-task dispatch не был реализован. [ADR-056](0056-staged-render-passage.md) вводит **Passage**: прогон = N упорядоченных этапов (render→dispatch→barrier→сбор register) по оси задач; per-task dispatch реализуется как N `ApplyRequest`/строк `apply_runs` на хост (по Passage). Главный драйвер — реальная работа probe→where (задача с `register.X` в `where:`/`apply: input:`/`params:`/`vars:` стратифицируется в Passage **после** probe, эмитящего `X`), которую этот ADR обещал каноном ([orchestration.md §4/§5](../scenario/orchestration.md#4-волатильный-предикат--where)), но keeper исполнял up-front до probe. **Barrier/state-commit-инвариант §7 НЕ ослаблен** — `incarnation.state` коммитится один раз после **последнего** Passage (Passage — ось задач, не ось коммита). Решения и слайс-карта — [ADR-056](0056-staged-render-passage.md).

**Amendment (S-2D1, 2D serial×passage). min-width `serial:` (§2.2.1 orchestration.md) — per-RUN → per-Passage.** С реализацией Passage `serial:` крутится совместно с осью задач: каждый Passage катит свои serial-волны по хостам ЭТОГО Passage, ширина выводится из задач **одного Passage**, а не всего прогона. Probe-Passage без `serial:` едет одной волной, даже когда другой Passage несёт `serial:N` (узкое окно одного Passage не просачивается в чужой — устранён destructive throttle probe-проходу). **В N=1 (один Passage) per-Passage и per-RUN тождественны бит-в-бит** — формулировка §2.2.1 для одно-проходного прогона не меняется. Рестрикт пилота `serial_staged_unsupported` снят. Детали — [ADR-056 §serial / §S4](0056-staged-render-passage.md).

**Amendment ([ADR-061](0061-onboarding-await-and-midrun-reresolve.md) §7, mid-run re-resolve roster). Статус → amended.** Прежний инвариант §7 — «roster прогона резолвится up-front перед первым Passage и mid-run НЕ растёт» (`soulprint.hosts` / `on: [incarnation.name]` фиксированы на весь прогон). **Amendment 2026-06-23 (новый task-level ключ `assert:` — keeper-side render-time precondition). Пилот.** Вводится новый task-level ключ DSL-ядра scenario `assert:` (форма Ansible-модуля `assert`):

```yaml
- name: топология cluster совпадает
  when: "input.redis_type == 'cluster'"
  assert:
    that:
      - "size(soulprint.hosts) == (has(input.shards) ? int(input.shards) : 0) * (1 + int(input.replicas_per_shard))"
    message: "topology mismatch: hosts != shards*(1+replicas_per_shard)"
```

Решения:

- **Keeper-side render-time precondition.** `assert:` вычисляется **на этапе модели** (render-фаза Keeper), как `where:` — в полном scenario-CEL-контексте, **включая `soulprint.hosts`** (`AllowHosts=true`: assert видит roster прогона, в отличие от Soul-side flow-control-песочницы, где топология недоступна). Все предикаты `that[]` должны быть `true`; **первый `false` обрывает render понятной ошибкой** (`message` + текст непрошедшего предиката), и **ни одной задачи на Soul не уходит** — прогон не стартует. Это «fail на этапе модели» (главная цель): неверная топология/инвариант отклоняется ДО dispatch, а не вылавливается shell-командой на хосте.
- **`assert:` НЕ wire-сущность.** Proto Keeper↔Soul и Soul-контракт **не меняются** (как `block:` — render-конструкция). assert **не emit RenderedTask** (это проверка, не задача): после успешного assert задача «исчезает» из плана, индексы последующих задач НЕ резервируют под неё позицию.
- **Дискриминатор.** `assert:` — пятый член дискриминатора задачи (наряду с `module:`/`apply:`/`include:`/`block:`): взаимоисключим с ними (assert — проверка, не исполняемая работа). `that` — непустой список CEL-bool-строк; `message` — опционален (дефолт-сообщение, если пуст).
- **Гейтится `when:`.** assert подчинён `when:`: статически-false `when:` (placeholder-skip ADR-012(d), напр. `when: input.redis_type == 'cluster'` на standalone-прогоне) → assert **не вычисляется**. На неактивном режиме cluster-assert молчит.
- **Run-level.** assert вычисляется **один раз на прогон** (использует roster `soulprint.hosts`), не per-host: это инвариант топологии прогона, а не per-host-предикат.
- **Мотив.** Убрать `core.cmd.shell`-guard-хак (произвольный shell на Soul ради control-flow: `test "${ <cel> }" = "true" || { echo ...; exit 1; }`) → декларативная keeper-side проверка. Снижает attack-surface (нет shell-исполнения ради проверки) и переносит отказ с хоста на модель.
- **Пилот — один use-site.** Заменён cluster size-guard (`examples/service/redis/scenario/create/cluster.yml`, задача size-guard). shards-guard (conditional-required) НЕ тронут — это отдельная фича `required_when` (следующий слайс). Тираж на sentinel/sentinel_only — после review пилота.

Нормативные правки — [`docs/templating.md §2.1`](../templating.md) (assert.that[] в таблицу expression-ключей, сторона Keeper), [`docs/scenario/orchestration.md §2.3`](../scenario/orchestration.md), [`docs/naming-rules.md`](../naming-rules.md). Реализация — `shared/config/scenario_task.go` (грамматика `AssertSpec`), `keeper/internal/render` (run-level eval, abort на false), `soul-lint` (статвалидация `that[]`), `keeper/internal/trial` (harness `expect_render_error` для fail-кейса, [ADR-023](0023-trial-test-runner.md) amendment).

Решения:

- **Контекст правила — INPUT-ONLY** (выбор пользователя). `that`-предикат видит **только `input.*`** — тот же узкий cel-go-sandbox, что `required_when` (объявлена единственная переменная `input`). Ссылка на `essence.*`/`soulprint.*`/`register.*`/`vault()`/`now()` → compile-ошибка undeclared reference: **структурный input-only-барьер необъявленностью**, не текстовый guard. `validate:` покрывает инварианты ВВОДА; топология/roster-проверки (`size(soulprint.hosts) == …`) остаются за `assert:` — у него полный scenario-CEL-контекст с `soulprint.hosts`.
- **ДОПОЛНЯЕТ, не заменяет.** `validate:` сосуществует с `assert:` и `required_when` (все три остаются). Граница «какой механизм когда»:

  | Механизм | Контекст CEL | Что проверяет | Когда срабатывает | Код ошибки |
  |---|---|---|---|---|
  | `required_when` (input-schema ключ) | input-only | **присутствие** поля при условии над другими input | input-стадия (request-путь) | 422 `validation-failed` (`input_invalid`) |
  | `validate:` (top-level секция) | input-only | **кросс-полевой инвариант** ВВОДА (соотношение значений) | pre-flight (request-путь), ДО коммита/applying | 422 `validation-failed` |
  | `assert:` (task-дискриминатор) | полный scenario-контекст (`soulprint.hosts`) | **инвариант ТОПОЛОГИИ** прогона (roster) | pre-flight create + render fail-safe | 422 `assert-failed` |

- **Когда — pre-flight на request-пути.** `validate:` вычисляется в `scenario.ValidateInput` (та же фаза, что `required_when`: ПОСЛЕ merge/required/type-валидации над **смерженным** input, ДО коммита incarnation и входа в `applying`). Первый `false` → **422 `validation-failed`** оператору с `message` правила; incarnation НЕ создаётся, `error_locked` НЕ ставится. Покрывает оба request-пути — `CreateTyped` (`POST /v1/incarnations`) и `RunTyped` (`POST .../scenarios/{scenario}`).
- **Render fail-safe НЕ нужен** (контраст с `assert:`, форма A). `assert:` держит двухточечную eval, потому что roster (`soulprint.hosts`) мог измениться между pre-flight и стартом goroutine (TOCTOU). `validate:` детерминировано от `input.*`, который между request-путём и стартом прогона **не меняется** — одной точки (pre-flight) достаточно.
- **Отдельный sentinel от input-схемы.** `validate:`-провал → `scenario.ErrValidateFailed` (отличён от `ErrInputInvalid`); оба маппятся в 422 `validation-failed`, но различимы для handler-а (разный detail-текст: «input не матчит схему» vs «input-инвариант сценария нарушен»). Порядок строгий: сначала schema/required/type, потом validate-инварианты (правило `that` пишут в расчёте на корректные типы).
- **Форма элемента = бит-в-бит `assert.that`+`message`** (переиспользование eval-машины). `that` — обязательная непустая CEL-bool-строка (вся строка = CEL без обёртки, как `where:`); `message` — обязательная непустая строка (в отличие от `assert.message`, опционального: у assert есть имя задачи как fallback, у validate-правила имени нет → анонимный 422). Без второго CEL-движка: `that` компилируется тем же `compileRequiredWhen` (input-env), что и `required_when`.
- **Additive / forward-compat.** Секция опциональная; существующие сценарии не ломаются. `soul-lint` валидирует `validate:` статически (структура + парсимость `that` input-only через `LoadScenarioManifest`). В L0-trial validate-провал обрывает кейс тем же каналом, что `required_when` (ловится `expect_render_error`, [ADR-023](0023-trial-test-runner.md)).
- **Мотив / пилот.** Use-case — сервис `redis`, режим `sentinel`: `sentinel_quorum <= 1 + replicas` (кворум не может превышать число sentinel-демонов, иначе failover недостижим). Это соотношение двух input-полей — не `min/max` (потолок = другое поле) и не `required_when` (не про присутствие); до `validate:` его пришлось бы городить keeper-side shell-стражем или ронять как непонятный runtime-фейл. Первый прод-потребитель за пилотом — слайс B («TLS или порт»).

Нормативные правки — [`docs/scenario/orchestration.md §2.5`](../scenario/orchestration.md#25-validate--декларативные-input-инварианты), [`docs/input.md`](../input.md). Реализация — `shared/config` (`ScenarioManifest.Validate`/`ValidateRule`, `validateValidateBlock`, `EvalValidateRules` поверх узкого input-CEL-эвалуатора `required_when`), `keeper/internal/scenario` (`ValidateInput` → `ErrValidateFailed`), `keeper/internal/api/handlers/incarnation_typed.go` (маппинг в 422 на Create/Run), `soul-lint` (статвалидация через `LoadScenarioManifest`), `keeper/internal/trial` (harness `EvalValidateRules` → `expect_render_error`).

Пилот.** На include-задаче (`- include: <file>`) теперь допустим **статический** `when:` (рядом с `name:`) — условный include. - **Скоуп предиката — только статический.** include раскрывается в плоский список **ДО** фазы стратификации ([ADR-056](0056-staged-render-passage.md)): register предыдущих задач ещё не собран, per-host `soulprint` неизвестен. Динамический include-when — **отложен** (open Q, см. ниже): требует решения «отложить раскрытие include до пост-probe Passage», что меняет фазовую модель.
- **Механизм carry-through + group-drop.** `config.ExpandIncludes` при раскрытии условного include протаскивает include-when и **id группы** в КАЖДУЮ вклеенную задачу через НЕ-YAML поля `Task.IncludeWhen`/`Task.IncludeGroupID` (проставляются ТОЛЬКО раскрытием, forward-compat как `RenderInput.destinyIsolated` — не парсятся из YAML, не попадают в `taskKnownKeys`). Каждый условный include получает свой монотонный group-id (`0` зарезервирован за «вне условного include»); вложенный условный include сохраняет СВОЙ (более конкретный) group-id — `stampIncludeGroup` не перезаписывает уже застампленную вложенную группу. Render в начале цикла, **ДО** `emitStaticWhenSkip`, для задачи с `IncludeGroupID != 0` вычисляет include-when ОДИН раз на группу (кеш по group-id) тем же `evalStaticWhen`; false → `continue` без эмита `RenderedTask` и без `idx++`.
- **Сосуществование с block-static-skip (дискриминатор `IncludeGroupID` vs `block`).** Это **отдельный** механизм от placeholder-skip block-а со static-false `when:` (amendment block-static-skip выше): block-static-skip **резервирует** индекс и эмитит per-потомок placeholder со своим `register:` (держит in-file register-видимость); group-drop **не резервирует** индекс и физически удаляет задачи. Развод строгий по `IncludeGroupID`: задача условного include идёт веткой group-drop, block-задача — прежней веткой placeholder-skip. Оба сосуществуют в одном плане.
- **Безопасность (нет `ErrOnChangesUnknownRegister`).** Дроп группы мог бы оставить «висячий» внешний `onchanges:`, ссылающийся на `register:` дропнутой задачи (`resolveOnChanges` → `ErrOnChangesUnknownRegister`). Этого не происходит: **cross-file register-ссылка уже lint-запрещена офлайн** — per-file `validateTaskRefs` валидирует register-scope каждого файла отдельно, поэтому `onchanges:` основного файла на register included-файла отвергается как `unknown_register_reference` при загрузке (а нераскрытый include до render → `ErrUnexpandedInclude`). Внешний потребитель register дропнутой группы в валидном scenario невозможен → `resolveOnChanges` после group-drop dangling-ссылки не встречает.
- **Оба пути include-expansion (scenario + destiny), кеш per-проход раздельный.** Группа-дроп работает на **обоих** render-путях, раскрывающих include в плоский план: scenario-проход `Pipeline.Render` (`keeper/internal/render/pipeline.go`) и изолированный destiny-проход `renderApplyDestiny` (`keeper/internal/render/destiny.go`). Механика идентична (carry-through `Task.IncludeWhen`/`IncludeGroupID` ставит один и тот же `config.ExpandIncludes`, вызываемый для destiny на `keeper/internal/artifact/destiny.go` и в Trial-fixture-резолвере; ветка group-drop стоит первой в цикле, ДО `emitStaticWhenSkip` и block-обработки), но с двумя инвариантами destiny-пути: (1) include-when destiny-группы вычисляется в **изолированном destiny-env** (`evalIncludeWhen(destinyIn, …)` — `destinyIn.Input` = резолвнутый `apply.input` + schema-defaults, БЕЗ scenario-scope: scenario-input/vars/register не видны), НЕ в `parentIn`; (2) кеш `includeGroupKeep` — **отдельная карта на проход** (scenario-цикл и каждый destiny-проход держат свою), id-группы destiny не пересекаются со scenario. register-изоляция идентична scenario (cross-file register дропнутой destiny-группы lint-запрещён per-file тем же `validateTaskRefs`, поэтому `resolveOnChanges` dangling-ссылки не встречает). Дискриминатор `IncludeGroupID` ортогонален block: при `keep=false` группа дропается вся (включая destiny-block-задачу и её потомков) ДО ветки `renderDestinyBlock`.

Нормативные правки — [`docs/destiny/tasks.md §4`](../destiny/tasks.md) (include: разрешённый статический `when:`, group-drop на обоих путях), [`docs/destiny/tasks.md §12`](../destiny/tasks.md) (open Q: динамический include-when отложен). Реализация — `shared/config/include_expand.go` (carry-through + `include_when_dynamic_unsupported`), `shared/config/task_refs.go` (`IsStaticIncludeWhen`), `shared/config/scenario_task.go` (`Task.IncludeWhen`/`IncludeGroupID`), `keeper/internal/render/pipeline.go` (scenario group-drop ветка + `evalIncludeWhen`), `keeper/internal/render/destiny.go` (destiny group-drop ветка над изолированным `destinyIn`, раздельный кеш), `soul-lint/internal/validate/stage.go` (офлайн-статвалидация). **Amendment 2026-06-29 (механизм нескольких именованных create-сценариев + удаление хардкод-дефолта `create` + bare-инкарнация как lifecycle-путь). Статус → amended.** Прежняя модель: ровно один зарезервированный bootstrap-сценарий с именем `create`, который `incarnation.Create` (REST + MCP) прогонял первым на новой инкарнации; имя `create` было привилегированным (резолвер `ResolveCreateScenarios` безусловно вкладывал литерал `CreateScenarioName` в набор — «магический union»). Это не выражало сервис с **несколькими** разными стартовыми операциями (например Redis: «поднять VM с нуля» vs «развернуть на готовый roster» vs «мигрировать с внешнего кластера») — каждая = свой состав задач и своя `input:`-схема, не вариация одного `create`. Решения:

- **Create-способность сценария — top-level `create: true` в `scenario/<name>/main.yml`.** Сервис объявляет произвольное число стартовых сценариев, помечая их флагом `create: true` (опциональный bool-ключ DSL-ядра scenario, рядом с `name:`). Набор стартовых = **РОВНО** `{sc | sc.create == true}`. Декларация — **per-scenario во флаге**, НЕ список в `service.yml` (сценарии и так auto-discover-ятся по каталогу, [`docs/service/manifest.md`](../service/manifest.md); централизованный список дублировал бы их). `destroy` этим флагом НЕ помечается (teardown — спец-флоу `DELETE /v1/incarnations/{name}`, не run).
- **Имя `create` БОЛЬШЕ НЕ привилегировано.** Магический union с литералом `CreateScenarioName` **удалён** из `ResolveCreateScenarios`: сценарий `scenario/create/main.yml` попадает в стартовый набор **только если сам несёт `create: true`**, как любой другой. Прежний back-compat-шорткат («пустой выбор → дефолтный `create`») снят.
- **Выбор стартового сценария оператором — поле `create_scenario` (`POST /v1/incarnations`).** Трёхветочный контракт (`ValidateCreateScenarioChoice`):
  - **выбор непуст + в наборе** → этот сценарий стартует; **не в наборе / невалидное имя** → `ErrCreateScenarioNotEligible` (422 `validation_failed`). Защита от bootstrap-а инкарнации произвольным (например day-2 `add_user`) сценарием.
  - **выбор пуст + набор НЕпуст** (сервис предлагает create-сценарии) → `ErrCreateScenarioRequired` (422 `validation_failed`, с перечислением годных имён). Выбор **обязателен**: `input:` валидируется против схемы КОНКРЕТНОГО сценария — без выбора нечего применять. Это намеренный сдвиг от прежнего авто-`create`.
  - **выбор пуст + набор ПУСТ** (нет ни одного `create: true`) → **bare-инкарнация** (см. ниже).
- **Bare-инкарнация — новый lifecycle-путь.** Сервис **без единого** `create: true`-сценария при создании даёт инкарнацию **`StatusReady` БЕЗ прогона** (apply_id не генерируется, `incarnation.created_scenario = NULL`). Инкарнация сразу готова к day-2-операциям (оператор гоняет нужные сценарии обычным `run`). Это обобщение существовавшего пути `lifecycle.auto_create: false` (инкарнация в `ready` без bootstrap-прогона) на случай «нет bootstrap-сценария вовсе». Bare — НЕ ошибка и НЕ ослабление: barrier/state-commit-инвариант §7 не задет (прогона нет → коммитить нечего; первая мутация `incarnation.state` будет от первого day-2-сценария).
- **`created_scenario` стал NULLABLE** (миграция 090, амендмент к 089). `NULL` = bare (создана без bootstrap-сценария), непустое = имя создавшего сценария. **`rerun-create` на bare → 409** (`ErrRerunScenarioNotCreate`): перезапускать нечего. Для не-bare `rerun-create` перезапускает **именно создавший** сценарий (`incarnation.created_scenario`, не хардкод `create`) — инкарнация, созданная `create_from_souls`, при rerun перезапускает `create_from_souls`. Existing-инкарнации (`created_scenario = 'create'` из DEFAULT 089) остаются корректны — у Redis-сервиса `scenario/create/main.yml` теперь несёт `create: true` (Фаза 1), поэтому `'create'` для них по-прежнему валидный bootstrap, backfill не нужен.
- **Контракт-сдвиг public-поверхности (semantic-breaking, не wire).** Семантика поля `create_scenario` (OpenAPI `docs/keeper/openapi.yaml`, MCP-tool `keeper.incarnation.create`) **изменилась**: пустое значение БОЛЬШЕ НЕ означает «default `create`». Внешний API/MCP-клиент, славший create без `create_scenario` в расчёте на авто-`create`, теперь получает либо 422 `create_scenario_required` (если сервис предлагает create-сценарии), либо bare-инкарнацию (если нет). Wire-схема additive (поле существовало), но **смысл** breaking — отметить в release-notes. MCP — паритет с REST (та же трёхветочная логика).
- **Redis-сервис (пилот).** Три `create: true`-сценария: `create` (provision новых VM через `core.cloud.created` + онбординг, [ADR-061](0061-onboarding-await-and-midrun-reresolve.md)), `create_from_souls` (deploy на уже-онбордившийся roster, без provision-секции), `migrate_cluster` (миграция с внешнего Redis через нативную репликацию = `create_from_redis`). Общие деплой-ветки (`cluster.yml`/`sentinel.yml`) и provision-тело (`redis-provision.yml`) — service-level include (фоллбэк имени файла, §6) — делятся без дубликата; `input:`/`compute:`/`state_changes:` остаются inline per-scenario (общего include для них нет — известная плата, cross-file drift не ловится soul-lint).
- **Additive / forward-compat.** Флаг `create:` опционален; сервис с единственным `create:true`-сценарием `create` ведёт себя как раньше (выбор `create` явно или единственным членом набора). soul-lint валидирует `create:` структурно (bool, `shared/config`).
