# ADR-009. Scenario — the full destiny task DSL; the boundary with destiny is a recommendation

- **Context.** Initially an isolation invariant was in force: a scenario could only `apply: { destiny: … }` (plus `target`/orchestration), but **not** `module:` directly — all the "work" had to live in an isolated, versionable, molecule-testable destiny. In practice this forced creating a separate destiny repo for the sake of a one-or-two-step inline operation (role probe, one-off command, diagnostics) that is never reused. The invariant hindered even read-only steps (probe), without which the volatile role from [ADR-008](0008-coven-stable-tags.md#adr-008-coven--stable-logical-tags-only) cannot be implemented.
- **Decision.**
  - **The "scenario without `module:`" invariant is fully lifted** (not only for read-only). Scenario receives **the entire destiny task DSL core** from [`docs/destiny/tasks.md`](../destiny/tasks.md): `module:` (including mutating modules), `templates`, task-level `vars:`, `register:`, `loop:`, `block:`, `parallel:`, `onchanges:`/`onfail:`/`require:`, `changed_when:`/`failed_when:`, `retry:`, `timeout:` — plus an orchestration layer on top (`on:`/`where:` targeting, `apply: { destiny: … }`, `state_changes`).
  - **The DSL core is not duplicated.** The source of truth for the core is `docs/destiny/tasks.md`; the scenario delta is a separate normative document [`docs/scenario/orchestration.md`](../scenario/orchestration.md).
  - **destiny remains** a standalone entity: a reusable, independently versionable (git ref, [ADR-007](0007-versioning-git-ref.md#adr-007-artifact-versioning--via-git-ref-not-a-manifest-field)), isolated, molecule-testable brick.
  - **The destiny / scenario boundary is a recommendation, not a wall.** The criterion for extraction into a destiny is three "yes"es: (1) reuse, (2) molecule idempotency is needed, (3) it is isolable. All three → extract into a destiny via `apply:`; otherwise an inline `module:` implementation in the scenario is permissible.
  - **Barrier / state-commit invariant (not an option).** The commit of `incarnation.state` (`state_changes`) is a **cross-host final-barrier AFTER an unconditional wait for all parallel tasks of all hosts of the run**. If even a single one is finally-failed → the state is **not** committed → `error_locked`. This is an extension of the final-barrier from [destiny/tasks.md §6](../destiny/tasks.md#6-parallelism-parallel-true) from a single host to the cross-host scenario level. More details — [`docs/scenario/orchestration.md`](../scenario/orchestration.md#7-barrier--state-commit-invariant).
  - **`state_changes` carries the source of the value, not only an allow-list.** `sets` is a map `<field> → <CEL expression>`: the declaration specifies **both** which `incarnation.state` field to update **and** where to take the value from (rendered Keeper-side in the scenario-CEL environment after the barrier). The former reading of `sets` as a flat allow-list of names (only "which fields are allowed to be written", without a source) is replaced: without a source, `incarnation.state` stayed empty on a successful run. The cross-host fold of `sets` is last-wins by `SID` sort order. **The context of `sets` is `input.* / incarnation.* / soulprint.self.* / register.*`.** `register.*` (the per-host probe result) is threaded through the back channel `TaskEvent.register_data` → table `apply_task_register` → a per-host register map after the barrier (slice 2 of the full grammar). `vars.* / essence.* / state.* / soulprint.hosts` and the keys `appends`/`modifies` (per-host collections) are a future extension of the full grammar, unavailable (render error → `error_locked`). **The grammar is extended by [ADR-057](0057-state-changes-crud-verbs.md) (see amendment below):** `state_changes` is an ordered list of CRUD verbs (`set`/`add`/`modify`/`remove`+`foreach`), the map form (`sets`/`appends`/`modifies`) is deprecated. Grammar — [`docs/scenario/orchestration.md §7.1`](../scenario/orchestration.md#71-grammar-state_changes---list-of-crud-operations).
  - **Isolation and destiny `output:`.** A destiny publishes its own result through a declared top-level `output:`; this is symmetric to `input:` and does **not** violate destiny isolation (a destiny does not read scenario scope — the parent's `essence`/`register`/`vars`/`input`, `incarnation.*`, the run topology). Specification — [`docs/destiny/output.md`](../destiny/output.md). **Clarification (see Amendment 2026-06-18):** the target host's stable self-facts (`soulprint.self.*`) are **not** scenario scope but a per-host property of the host itself, therefore they are available to destiny-CEL; only cross-host (`soulprint.hosts`/`soulprint.where`) is forbidden.
- **Rationale.** One task language across both layers — no two dialects, the scenario author does not learn a second syntax. Probe (read-only `module:` + `register:`) becomes natural, which is exactly how the volatile role is implemented ([ADR-008](0008-coven-stable-tags.md#adr-008-coven--stable-logical-tags-only)). destiny does not lose its properties — it remains a choice by recommendation, not by compulsion.
- **Trade-off / risks.**
  - **Scope expansion of [ADR-003](0003-destiny-format.md#adr-003-destiny-format--yaml-with-a-typed-schema-cuejson-schema).** The safe templater as a separate phase now serves scenario as well (`where:` predicates, `on:` literals, templates). This question (choice of templater) is closed by [ADR-010](0010-templating.md#adr-010-templating-engine-cel-for-yaml-expressions-go-texttemplate-for-files). Prior to ADR-010 it blocked the implementation of scenario, not only destiny.
  - **Erosion risk of [ADR-007](0007-versioning-git-ref.md#adr-007-artifact-versioning--via-git-ref-not-a-manifest-field).** Inline `module:` mutations in a scenario bypass the independent git versioning of a destiny (a pinpoint edit of `scenario/<name>/main.yml` does not receive a tag and the review of a separate destiny repo). **Mitigation:** the boundary-recommendation above + a backlog lint-warn in [`docs/soul-lint.md`](../soul-lint.md) ("inline mutation in a scenario without extraction into a destiny"). A consciously accepted price for the simplicity of typical operations.
- **Related decisions.** [ADR-008](0008-coven-stable-tags.md#adr-008-coven--stable-logical-tags-only) (probe instead of role-as-Coven — implemented precisely by lifting this invariant). The full specification of the delta — [`docs/scenario/`](../scenario/README.md).

**Amendment 2026-05-27 (invocation-time scope chunking — Tide).** [ADR-040](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override) introduces a new top-level entity **Tide** — an invocation-instance that splits a mass scenario run into sequential waves (**Surges**) with an optional target/wave/concurrency override. Fundamentally:

- **Tide is NOT part of the scenario DSL** — it is not written into the scenario YAML, but rather parameterizes the **invocation** (HTTP/MCP request to `POST /v1/incarnations/{name}/scenarios/{scenario}`). The scenario DSL of ADR-009 stays unchanged; the new opt fields (`target` / `wave` / `concurrency`) live only in the request payload, not in `main.yml`.
- **Declaration vs invocation — two levels.** `serial:` / `run_once:` / `on:` / `where:` fix the split and the target **at the scenario declaration level** (the service's git repo, the author's intent). Tide adds an **invocation-level** orchestration layer **on top of** the scenario runner — the operator at launch time can split the scope into waves, narrow the target, override concurrency, without changing the scenario. See [`docs/scenario/orchestration.md §2.2.3`](../scenario/orchestration.md).
- **Compatibility with the scenario DSL:**
  - a scenario without `run_once:` is compatible with Tide unconditionally.
  - a scenario with `run_once: true` is **incompatible** with an invocation `wave` (`run_once` = exactly one target host, there is nothing to split into waves) — validation error `tide_run_once_conflict` (422), [`docs/keeper/operator-api.md`](../keeper/operator-api.md).
  - a scenario's `concurrency:` (`serial:`) may be **overridden** by the invocation's `concurrency:` (REPLACE semantics — a runtime knob, not a declared invariant).
- **Per-Surge barrier/state-commit — parity with §7.** Each Surge is a full-fledged scenario run with its own cross-host barrier and state-commit; the barrier invariant of §7 extends to each Surge independently, not to the whole Tide as a whole. Normatively — the mechanics of per-Surge state-commit for Tide (the separate file `docs/scenario/tide-state-commit.md` was not created; Tide is abolished — see [ADR-043](0043-voyage.md#adr-043-voyage--unified-batch-run), per-Surge → per-incarnation state-commit).
- **AND-merge target-override (security invariant).** The invocation's `target.coven[]` / `target.where` only **narrow** the scope (intersection with the scenario's `on:` / `where:`), they do not widen it. The operator cannot, via the invocation, go beyond the declared scenario target. Symmetric to the multi-label AND inside `on:` — a single source of truth for the DSL text [`docs/scenario/orchestration.md §3`](../scenario/orchestration.md).
- **`soulprint.hosts` / `incarnation.host_count` in Tide.** The scenario-context accessors are resolved per-Surge from the **target_resolved_souls** snapshot of the Tide (the AND-merge fixed at Tide start), not from the incarnation's original roster — each Surge sees only its own share.

Goals and rationale — [ADR-040](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override).

**Amendment 2026-05-29 (§7 — per-incarnation state-commit in Voyage = parity with per-Surge).** [ADR-043](0043-voyage.md#adr-043-voyage--unified-batch-run) (Voyage) absorbs Tide via the `kind=scenario` mode. Approach **B1**: a batch (**Leg**) = N **incarnations**, each incarnation = a full-fledged scenario run with its own cross-host barrier and **per-incarnation state-commit**. This is a **direct parity** with per-Surge state-commit ([ADR-040](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override)) and preserves the barrier invariant of §7 without weakening: `incarnation.state` is committed only after a successful barrier of **this** incarnation, a partial commit (part of the incarnation's hosts in one Leg, the rest in the next) is **forbidden**. Alternative B2 (a Leg slices the scope by hosts across several incarnations) is **rejected precisely because it breaks §7** — see [ADR-043](0043-voyage.md#adr-043-voyage--unified-batch-run) point 3. Intra-incarnation waves by hosts remain a separate axis via the scenario's `serial:` (declaration-level), as before.

**Amendment 2026-05-29 (Choir — additive `choirs[]` in scenario accessors).** [ADR-044](0044-choir.md#adr-044-choir--named-host-topology-within-an-incarnation) (Choir) adds an **additive** field `choirs[]` to the scenario context (the list of the host's Choir names in the current incarnation): `soulprint.self.choirs` (per-host, the current host) and `soulprint.hosts[].choirs` (cross-host accessor of the run's hosts, [`docs/scenario/orchestration.md §4.1`](../scenario/orchestration.md) — the site of a future normative edit at S-T4). This is an additive extension of the existing stable fields of `soulprint.hosts` (covens/sid/network/os) — it does not change the `on:`/`where:` grammar, and it is available in `where:` predicates **without a probe** (Choir is a stable declared fact, not a volatile role). `on: [choir]` is NOT introduced in the MVP (the `on:` resolver stays coven-only; narrowing by Choir is via `where:`). The absorbed declared role `soulprint.hosts[].role` ([ADR-044](0044-choir.md#adr-044-choir--named-host-topology-within-an-incarnation) point 2) is fed from Choir (via Voice), not from `spec.hosts[].role`. The normative edit to orchestration.md happens in slice S-T4; this amendment only declares it.

**Amendment 2026-06-12 (optional task field `id:` — a stable address for per-task-changed alerts).** A new **optional** task DSL-core field `id:` is introduced — a stable address of a task by which a subscription from [ADR-052](0052-herald-notifications.md#adr-052-herald--tiding--notifications-about-run-events) (§h) addresses the "this task changed state" event (per-task-changed alert). Decisions:

- **Value format = register format** (the same regex `^[a-z][a-z0-9_]*$` as `register:`): id and register live in **one address space** of the subscription to per-task-changed events, therefore the form is unified.
- **Optional.** The absence of `id:` on a task is normal; existing scenarios/destinies are not broken.
- **`id:` ⊕ `register:` are mutually exclusive.** A task with `register:` is already addressable by it → `id:` is redundant and creates address ambiguity. Both keys on one task is a validation error (`id_register_conflict`). `id:` is intended for tasks **without** `register:`.
- **Pilot restriction: `id:` only on a module task.** In the first pass `id:` is permitted only on a module task (it has its own changed signal). On a `block:`/`include:` task — a validation error (`id_unsupported_target`); lifting the restriction is a separate pass upon request.
- **Uniqueness of `id:` across the run** (after flattening `include`/`block`) is **not fixed in this amendment**; it is a lint cross-ref check (a flatten context is needed), a separate slice. Here — only the per-task format + mutual exclusion with register + the restriction to module.
- **Resolution of `id:` into RenderedTask, the formation of events, and the Tiding binding** are the next slices, not part of this edit (only the grammar of the field and form validation).

Per-task changed addressing (the address space `register ∪ id`) is fixed on the notifications side in [ADR-052](0052-herald-notifications.md#adr-052-herald--tiding--notifications-about-run-events) (§h). The source-of-truth document for the task field — [`docs/destiny/tasks.md §3`](../destiny/tasks.md#3-complete-list-of-task-blocks).

**Amendment 2026-06-18 (the destiny pass's CEL-env additionally registers read-only `soulprint.self.*`).** Jointly with [ADR-010](0010-templating.md#adr-010-templating-engine-cel-for-yaml-expressions-go-texttemplate-for-files) (an amendment of the same date). The destiny isolation invariant is **weakened**:

- **The destiny pass's CEL-env additionally registers read-only `soulprint.self.*`** — the stable layer `SoulprintFacts` ([ADR-018](0018-soulprint-typed.md#adr-018-soulprint-typed-schema-mvp)): `os.*` (including `arch`), `kernel`/`cpu`/`memory`/`network`, the root `sid`/`hostname` and the registry projections (`covens`/`choirs`/`role`). The source is the same per-host soulprint of the target host that Keeper already collects for the render_context of the `.tmpl` phase.
- **`soulprint.hosts` / `soulprint.where(...)` remain scenario-only** (run topology is cross-host) — a reference to them in the destiny pass is still an **isolation error**. The boundary: the target host's self-facts — yes; the run topology — no.
- **Symmetry with the `.tmpl` render_context** ([ADR-012(d)](0012-keeper-soul-grpc.md#adr-012-keepersoul-grpc-contract-one-eventstream-with-oneof-keeper-side-render-forward-compat-only-add)): `self` already flows into `core.file.rendered.render_context` per-host; now `soulprint.self.<path>` in CEL `.yml` ≡ `.self.<path>` in `.tmpl` — a single point of truth for self.
- **`apply: input:` remains a valid channel** (it is not removed): passing the run topology (`soulprint.hosts.where(...)`) into a destiny is still done only via an explicit `apply: input:` forwarding. The bootstrap-create idiom (master discovery via apply:input) is not broken.
- **Rationale.** `soulprint.self.*` is a per-host property of the very host on which the destiny is executed (and not scenario scope: not the parent's `essence`/`register`/`vars`/`input`). A destiny is applied to a concrete host anyway; seeing its stable facts (arch for choosing a package/archive, family for paths) is natural and removes the extra boilerplate `apply: input: { arch: "${ soulprint.self.os.arch }" }`. Security is not harmed: self is read-only, without I/O, without cross-host.

Normative edits — [`docs/templating.md §3.2/§7`](../templating.md), [`docs/destiny/tasks.md §10`](../destiny/tasks.md#10-template-context) (already reflected self in the destiny context). Implementation: `soulprint.self` in the destiny pass is available through the render pipeline (`keeper/internal/render`, `Vars.SoulprintSelf`), `soulprint.hosts`/`soulprint.where` are cut off by `AllowHosts=false` (`shared/cel`).

**Amendment ([ADR-056](0056-staged-render-passage.md), staged-render — Passage). Status → amended.** The per-RUN min-width `serial:` (§2.2.1 orchestration.md) and open Q §8 "per-task granularity" were formulated on the restriction "one `ApplyRequest` per host with all tasks in bulk" — per-task dispatch was not implemented. [ADR-056](0056-staged-render-passage.md) introduces **Passage**: a run = N ordered stages (render→dispatch→barrier→collection of register) along the task axis; per-task dispatch is implemented as N `ApplyRequest`s / `apply_runs` rows per host (per Passage). The main driver is the actual probe→where work (a task with `register.X` in `where:`/`apply: input:`/`params:`/`vars:` is stratified into a Passage **after** the probe that emits `X`), which this ADR promised as canon ([orchestration.md §4/§5](../scenario/orchestration.md#4-volatile-predicate---where)) but keeper executed up-front before the probe. **The barrier/state-commit invariant of §7 is NOT weakened** — `incarnation.state` is committed once after the **last** Passage (a Passage is the task axis, not the commit axis). Decisions and slice map — [ADR-056](0056-staged-render-passage.md).

**Amendment (S-2D1, 2D serial×passage). min-width `serial:` (§2.2.1 orchestration.md) — per-RUN → per-Passage.** With the implementation of Passage, `serial:` turns jointly with the task axis: each Passage rolls its own serial waves over the hosts of THAT Passage, the width is derived from the tasks of a **single Passage**, not the whole run. A probe Passage without `serial:` runs as a single wave, even when another Passage carries `serial:N` (the narrow window of one Passage does not leak into another — a destructive throttle of the probe pass is eliminated). **At N=1 (a single Passage) per-Passage and per-RUN are identical bit-for-bit** — the wording of §2.2.1 for a single-pass run does not change. The pilot restriction `serial_staged_unsupported` is lifted. Details — [ADR-056 §serial / §S4](0056-staged-render-passage.md).

**Amendment ([ADR-061](0061-onboarding-await-and-midrun-reresolve.md) §7, mid-run re-resolve of the roster). Status → amended.** The former §7 invariant — "the run's roster is resolved up-front before the first Passage and does NOT grow mid-run" (`soulprint.hosts` / `on: [incarnation.name]` fixed for the whole run). [ADR-061](0061-onboarding-await-and-midrun-reresolve.md) §S3 **weakens** it to **"the roster is stable WITHIN a Passage; at a refresh boundary it is re-resolved"**. A refresh boundary is a successful `core.soul.registered` step with `refresh_soulprint: true` (a new class of passage-defining edge, "roster-refresh", [ADR-056](0056-staged-render-passage.md)): before the next Passage the scenario-runner re-resolves the roster with a **live snapshot** of the incarnation's current online set (`topology.LoadIncarnationHosts`). This is NOT a monotonic growth `roster ∪ newly_online`, but a fresh snapshot: onboarded VMs become visible, while a host that went offline by the boundary is excluded (targeting the actually-online set). The target case is a single create run provision→onboarding→role. **The barrier/state-commit invariant of §7 is NOT weakened** — `incarnation.state` is committed once after the last Passage (re-resolve is the roster axis, not the commit axis). Details — [ADR-061 §S2/§S3](0061-onboarding-await-and-midrun-reresolve.md).

**Amendment ([ADR-057](0057-state-changes-crud-verbs.md), `state_changes` → an ordered list of CRUD verbs). Status → amended.** The `state_changes` grammar (§7.1) is extended: instead of an unordered map with the keys `sets`/`appends`/`modifies` — an **ordered list of operations**, each element = one CRUD verb `set`/`add`/`modify`/`remove` + a structural `foreach` (reuse of the migration DSL [ADR-019](0019-state-migration-dsl.md#adr-019-state_schema-migration-dsl)). Multiplicity of `modify`/`remove` is via a `match` predicate (all-by-default), not flags; the bindings `elem`/`key`/`value`; an optional `expect` (one/at_most_one/any) — a runtime assertion of cardinality; `add` with `on_conflict` (skip/replace/error, default skip = idempotent). It closes a **latent bug**: `appends`/`modifies` were no-op placeholders without a value source — `incarnation.state` did not grow (`add_replica`/`add_user`/`update_acl`). The barrier/state-commit invariant of §7 is **NOT weakened** (operations are applied after the cross-host barrier, one PG transaction, one `state_history` snapshot, a failure → `error_locked`). Transit: the map form is parsed for one release as DEPRECATED (dual-parse + soul-lint warn), then removed. Normative spec — [`docs/scenario/orchestration.md §7.1`](../scenario/orchestration.md#71-grammar-state_changes---list-of-crud-operations), fixed by — [ADR-057](0057-state-changes-crud-verbs.md).

**Amendment 2026-06-20 (read-only `incarnation.state.<path>` in the scenario-render CEL — Variant A).** Jointly with [ADR-010](0010-templating.md#adr-010-templating-engine-cel-for-yaml-expressions-go-texttemplate-for-files) (an amendment of the same date). It closes a gap: a scenario that reads the current `incarnation.state` for a diff (e.g. `update_acl`: `current: ${ incarnation.state.redis_users }`) had no way to reference the state — `state.*` in the scenario-CEL is **not declared** (only the migration-CEL declares it), and `incarnation`, although present in the context, did not materialize `.state`. Decisions:

- **The canonical form is `incarnation.state.<path>`** (not the bare `state.<path>`): the root `incarnation` is already in the scenario context, `state` becomes its subkey. Bare `state.*` in scenario/destiny → a `soul-lint` validation error (`state_naked_reference`) with a hint "use `incarnation.state.*`" — symmetric to how bare `soulprint.<path>` without `.self` is cut. The former wording (§5/§7: `state.*` is a future extension, unavailable) is **replaced**: access is opened under the `incarnation.` prefix.
- **Read-only snapshot.** `incarnation.state.<path>` is a snapshot of `incarnation.state` at the moment the run acquires the row-lock (`stateBefore` under `FOR UPDATE`). CEL **only reads** it, there is no path to mutate state via CEL (mutation is exclusively via `state_changes` after the barrier, §7). Destiny isolation is preserved: a destiny does NOT see `incarnation.state` (it receives values only via `apply: input:`, like the parent's `essence`/`register`/`vars`/`input`).
- **Invariant across all Passages (staged-render).** The snapshot is fixed **once** (`stateBefore`) and is identical at P0 and P1+ — it does **NOT** accumulate between Passages, unlike `register` (which is collected by the barriers of previous Passages). This is a deliberate distinction: `incarnation.state` is the pre-run state from the DB (last known-good), not an intermediate result of the current run.
- **Access.** `incarnation.state.<path>` is visible in `params:`/`where:`/`apply: input:`/`vars:` AND in the `state_changes` context (one projection point — `incarnationVars`). It is available in an ordinary scenario run, a failover-claim (Acolyte), and check-drift; in push mode and L0-trial without state, `incarnation.state.x` yields a regular no-such-key (backward-compat, not a compile-error).

**ADR-019 (migration DSL) is NOT affected**: migration isolation is preserved — there `state.*` stays **bare and mutable** (only `state` is declared, `incarnation`/`input`/… undeclared), Variant A did not leak into the migration-env. Implementation — `keeper/internal/render` (`RenderInput.State`, `incarnationVars`), `keeper/internal/scenario/run.go` (`State: stateBefore`), `shared/config` (`state_naked_reference` in `soul-lint`).

**Amendment 2026-06-23 (new task-level key `assert:` — a keeper-side render-time precondition). Pilot.** A new task-level key of the scenario DSL core `assert:` is introduced (the form of the Ansible module `assert`):

```yaml
- name: cluster topology matches
  when: "input.redis_type == 'cluster'"
  assert:
    that:
      - "size(soulprint.hosts) == (has(input.shards) ? int(input.shards) : 0) * (1 + int(input.replicas_per_shard))"
    message: "topology mismatch: hosts != shards*(1+replicas_per_shard)"
```

Decisions:

- **Keeper-side render-time precondition.** `assert:` is evaluated **at the model stage** (Keeper's render phase), like `where:` — in the full scenario-CEL context, **including `soulprint.hosts`** (`AllowHosts=true`: assert sees the run's roster, unlike the Soul-side flow-control sandbox where topology is unavailable). All predicates `that[]` must be `true`; **the first `false` aborts the render with a clear error** (`message` + the text of the predicate that failed), and **not a single task is sent to a Soul** — the run does not start. This is "fail at the model stage" (the main goal): an incorrect topology/invariant is rejected BEFORE dispatch, rather than being caught by a shell command on the host.
- **`assert:` is NOT a wire entity.** The Keeper↔Soul proto and the Soul contract **do not change** (like `block:` — it is a render construct). assert does **not** emit a RenderedTask (it is a check, not a task): after a successful assert the task "disappears" from the plan, and the indices of subsequent tasks do NOT reserve a position for it.
- **Discriminator.** `assert:` is the fifth member of the task discriminator (alongside `module:`/`apply:`/`include:`/`block:`): mutually exclusive with them (assert is a check, not executable work). `that` is a non-empty list of CEL-bool strings; `message` is optional (a default message if empty).
- **Gated by `when:`.** assert is subordinate to `when:`: a statically-false `when:` (placeholder-skip ADR-012(d), e.g. `when: input.redis_type == 'cluster'` on a standalone run) → assert is **not** evaluated. On an inactive mode the cluster-assert stays silent.
- **Run-level.** assert is evaluated **once per run** (it uses the roster `soulprint.hosts`), not per-host: it is a run-topology invariant, not a per-host predicate.
- **Motivation.** To remove the `core.cmd.shell`-guard hack (arbitrary shell on a Soul for the sake of control-flow: `test "${ <cel> }" = "true" || { echo ...; exit 1; }`) → a declarative keeper-side check. It reduces the attack surface (no shell execution for the sake of a check) and moves the rejection from the host to the model.
- **Pilot — one use-site.** The cluster size-guard is replaced (`examples/service/redis/scenario/create/cluster.yml`, the size-guard task). The shards-guard (conditional-required) is NOT touched — it is a separate feature `required_when` (the next slice). The rollout to sentinel/sentinel_only is after the pilot review.

**Amendment 2026-06-23 (assert is evaluated at TWO points — pre-flight + render fail-safe; form A). Pilot.** The original edition of `assert:` evaluated the predicate ONLY at the async render phase of the run: the failure occurred already AFTER the incarnation was moved to `applying`, which dropped the run into **`error_locked`** — that is, "rejection at the model stage" turned into a fail status and a post-factum analysis (unlock), even though the topology invariant can be checked BEFORE any mutation. The amendment introduces a **two-point evaluation** (the user's choice — form A):

- **(1) Pre-flight at run creation (the main UX).** The assert predicates of the `create` scenario are evaluated SYNCHRONOUSLY in the **request path** of the create handler (`POST /v1/incarnations`), **AFTER** synchronous input validation and **BEFORE** the commit of the incarnation / entry into `applying`. Any `false` → **422 `assert-failed`** to the operator with `message` + the text of the predicate that failed; **the incarnation is NOT created, no fail status (`error_locked`) is set, the run does not start**. This moves the rejection from the async phase (post-factum `error_locked`) to the synchronous request path — symmetric to how [`ValidateInput`](../keeper/operator-api.md) already moved input validation out of the run goroutine into the request path (closing the "created → error_locked" hole).
- **(2) Render fail-safe (a TOCTOU safeguard).** The render branch of `assert:` is NOT removed. Between pre-flight (the request path) and the actual start of the scenario goroutine the roster could have changed (connected souls dropped off/were added — TOCTOU). The render-assert at run start catches this rare case: a failure at render = `error_locked` via the same async-abort as before. Pre-flight is the main UX (422, without a record); the render-assert is a fail-safe (a rare `error_locked`).
- **Single source of eval (no dialect).** Both points evaluate the predicate with ONE function (`render.evalAssertTask`): pre-flight calls the public `render.Pipeline.EvalAsserts` (iterates only the assert tasks, without emitting a RenderedTask), the render branch calls the same `evalAssertTask` inside `Render`. The reuse invariant is covered by a test (one input through both points → a matching verdict). Duplicating the eval is forbidden — otherwise a render↔pre-flight dialect.
- **Roster-at-create.** Pre-flight resolves the roster of connected souls by the incarnation's root Coven label (= `incarnation.name`, ADR-008) — `topology.LoadIncarnationHosts` does NOT require an existing incarnation record (a filter by `souls.coven[]`, the declared roles are empty for a not-yet-created one — not an error). This is the roster of the imminent run (create Start-s immediately). 0 connected souls → a topology-assert (`size==N`) honestly does not hold → 422 (correct: you cannot create+start an N-shard cluster without N hosts).
- **Pilot scope — only the create path** (`POST /v1/incarnations`). The paths `incarnation.run` on an existing one / `destroy` / Voyage are follow-up (the render-assert fail-safe is their only gate for now). Use case — the redis cluster size-guard.
- **Error code — `assert-failed` (422), not `precondition-failed`.** 422 Unprocessable Entity (parity with `validation-failed`/input_invalid: the same class "the input semantics do not add up"), NOT 412 Precondition Failed (reserved for the conditional headers `If-Match`/`If-None-Match`). A separate URN from `validation-failed` — the UI/SDK distinguishes "the topology does not add up" from "an input field does not match the schema".

Normative edits — [`docs/keeper/operator-api.md`](../keeper/operator-api.md) (error catalog: `assert-failed` 422), [`docs/naming-rules.md`](../naming-rules.md) (HTTP error type + the two-point eval in the description of `assert:`), [ADR-027](0027-apply-work-queue.md#adr-027-apply-execution-model--work-queue--claim-acolyte-pool-ward-claim) amendment (the pre-flight gate in the run lifecycle BEFORE `applying`). **Amendment 2026-06-23 (new input-schema key `required_when` — conditional requiredness of a field).** An **optional** input-DSL key `required_when: "<CEL predicate over input.*>"` is introduced (the standard [`docs/input.md`](../input.md), common to destiny / scenario / the module manifest). The parameter becomes required **when** the predicate is true; when the predicate is false the absence of the parameter is permissible. This closes the gap between the unconditional `required: true` (always required) and "required only in one mode". Decisions:

- **Validated at Keeper's input stage** (`config.requireInputValues`, AFTER merging defaults, BEFORE the render phase and dispatch) — the same place as the unconditional `required`. Absence when the predicate is true = an input error (422 `input_invalid` / sync-`ValidateInput`); in L0-trial — an abort at input resolution (caught by `expect_render_error`, ADR-023). The error message carries the same recognizable form as `required` ("required … not provided"), plus the text of the condition.

A direct import `config → cel` is rejected (it would bloat the base config with the heavy engine and its surface). Instead — a **narrow standalone evaluator inside `shared/config`** on top of `cel-go` (the library is already in the `shared` module), an env with the single variable `input`. There is no cycle (`cel` also does not import `config`); the full Engine is not duplicated (the narrow evaluator carries less surface — exactly the input-only predicate).
- **Additive / forward-compat.** The field is optional; existing schemas are not broken. `soul-lint` validates `required_when` statically (CEL parsability).
- **Motivation.** It removes shell guards of conditional requiredness (`core.cmd.shell` with `test "${ has(input.X) }" = true || exit 1`) — arbitrary shell for control-flow on a Soul is replaced by declarative keeper-side input validation: the operator gets a rejection before the run starts, the attack surface does not grow. Use case — the `redis` service: `shards` when `redis_type == 'cluster'`, `master_ip` when `redis_type == 'sentinel_only'`.

**Amendment 2026-06-23 (new top-level scenario key `compute:` — computed run vars).** An **optional** top-level scenario block `compute:` is introduced (next to `input:`/`state_changes:`/`tasks:`) — a map `<name>: <CEL expression>` which Keeper resolves **ONCE per run** and makes available as `compute.<name>` in `apply: input:` **and** in `state_changes`. **Amendment 2026-06-23 (new top-level scenario section `validate:` — declarative input invariants).** An **optional** top-level scenario section `validate:` is introduced (next to `input:`) — a list of rules `[{that: <CEL-bool>, message: <str>}]`. Each rule declares an **input invariant** that must be true; the first `false` rejects the run to the operator **before the incarnation commit and before applying**. It closes the "must be X, not Y" gap for **cross-field preconditions** not expressible by a single schema key (`enum`/`min`/`max` — about one field) and not covered by `required_when` (about the **presence** of a field, not about the **relation** of values). Decisions:

- **The rule context is INPUT-ONLY** (the user's choice). The `that` predicate sees **only `input.*`** — the same narrow cel-go sandbox as `required_when` (the single variable `input` is declared). A reference to `essence.*`/`soulprint.*`/`register.*`/`vault()`/`now()` → a compile error undeclared reference: a **structural input-only barrier by undeclaredness**, not a textual guard. `validate:` covers invariants of the INPUT; topology/roster checks (`size(soulprint.hosts) == …`) remain the domain of `assert:` — it has the full scenario-CEL context with `soulprint.hosts`.
- **It COMPLEMENTS, does not replace.** `validate:` coexists with `assert:` and `required_when` (all three remain). The boundary "which mechanism when":

  | Mechanism | CEL context | What it checks | When it fires | Error code |
  |---|---|---|---|---|
  | `required_when` (input-schema key) | input-only | the **presence** of a field under a condition over other input | input stage (request path) | 422 `validation-failed` (`input_invalid`) |
  | `validate:` (top-level section) | input-only | a **cross-field invariant** of the INPUT (relation of values) | pre-flight (request path), BEFORE commit/applying | 422 `validation-failed` |
  | `assert:` (task discriminator) | full scenario context (`soulprint.hosts`) | an invariant of the run's TOPOLOGY (roster) | pre-flight create + render fail-safe | 422 `assert-failed` |

- **When — pre-flight on the request path.** `validate:` is evaluated in `scenario.ValidateInput` (the same phase as `required_when`: AFTER merge/required/type validation over the **merged** input, BEFORE the incarnation commit and entry into `applying`). The first `false` → **422 `validation-failed`** to the operator with the rule's `message`; the incarnation is NOT created, `error_locked` is NOT set. It covers both request paths — `CreateTyped` (`POST /v1/incarnations`) and `RunTyped` (`POST .../scenarios/{scenario}`).
- **A render fail-safe is NOT needed** (in contrast with `assert:`, form A). `assert:` holds a two-point eval because the roster (`soulprint.hosts`) could have changed between pre-flight and the goroutine start (TOCTOU). `validate:` is deterministic from `input.*`, which between the request path and the run start **does not change** — a single point (pre-flight) is sufficient.
- **A separate sentinel from the input schema.** A `validate:` failure → `scenario.ErrValidateFailed` (distinguished from `ErrInputInvalid`); both map to 422 `validation-failed`, but are distinguishable for the handler (different detail text: "input does not match the schema" vs "a scenario input invariant is violated"). The order is strict: first schema/required/type, then validate invariants (the `that` rule is written on the assumption of correct types).
- **The element form = bit-for-bit `assert.that`+`message`** (reuse of the eval machine). `that` is a mandatory non-empty CEL-bool string (the whole string = CEL without a wrapper, like `where:`); `message` is a mandatory non-empty string (unlike `assert.message`, which is optional: assert has the task name as a fallback, a validate rule has no name → an anonymous 422). Without a second CEL engine: `that` is compiled by the same `compileRequiredWhen` (input-env) as `required_when`.
- **Additive / forward-compat.** The section is optional; existing scenarios are not broken. `soul-lint` validates `validate:` statically (structure + parsability of `that` input-only via `LoadScenarioManifest`). In L0-trial a validate failure aborts the case via the same channel as `required_when` (caught by `expect_render_error`, [ADR-023](0023-trial-test-runner.md)).
- **Motivation / pilot.** Use case — the `redis` service, `sentinel` mode: `sentinel_quorum <= 1 + replicas` (the quorum cannot exceed the number of sentinel daemons, otherwise failover is unreachable). This is a relation of two input fields — not `min/max` (the ceiling = another field) and not `required_when` (not about presence); before `validate:` it would have had to be cobbled together with a keeper-side shell guard or dropped as an obscure runtime failure. The first production consumer after the pilot is slice B ("TLS or port").

Normative edits — [`docs/scenario/orchestration.md §2.5`](../scenario/orchestration.md#25-validate---declarative-input-invariants), [`docs/input.md`](../input.md). Implementation — `shared/config` (`ScenarioManifest.Validate`/`ValidateRule`, `validateValidateBlock`, `EvalValidateRules` on top of the narrow input-CEL evaluator of `required_when`), `keeper/internal/scenario` (`ValidateInput` → `ErrValidateFailed`), `keeper/internal/api/handlers/incarnation_typed.go` (mapping to 422 on Create/Run), `soul-lint` (static validation via `LoadScenarioManifest`), `keeper/internal/trial` (the harness `EvalValidateRules` → `expect_render_error`).

Therefore the include-when must be **static** — depending only on `input.*`/`essence.*`/`incarnation.*`/`vars.*` (the same `isStaticWhen` criterion as for the static-when placeholder-skip [ADR-012(d)](0012-keeper-soul-grpc.md)). A dynamic predicate (`register.*`/`soulprint.*`) → the error `include_when_dynamic_unsupported` (the expansion rejects it; `soul-lint` catches it offline by static validation of the include node without resolving the target). A dynamic include-when is **deferred** (an open Q, see below): it requires the decision "defer the include expansion until the post-probe Passage", which changes the phase model.
- **The carry-through + group-drop mechanism.** `config.ExpandIncludes`, when expanding a conditional include, threads the include-when and the **group id** into EVERY spliced task through the NON-YAML fields `Task.IncludeWhen`/`Task.IncludeGroupID` (set ONLY by the expansion, forward-compat like `RenderInput.destinyIsolated` — not parsed from YAML, not included in `taskKnownKeys`). Each conditional include gets its own monotonic group-id (`0` is reserved for "outside a conditional include"); a nested conditional include preserves ITS OWN (more specific) group-id — `stampIncludeGroup` does not overwrite an already-stamped nested group. Render, at the start of the loop, **BEFORE** `emitStaticWhenSkip`, for a task with `IncludeGroupID != 0` computes the include-when ONCE per group (a cache by group-id) with the same `evalStaticWhen`; false → `continue` without emitting a `RenderedTask` and without `idx++`.
- **Coexistence with block-static-skip (the discriminator `IncludeGroupID` vs `block`).** This is a **separate** mechanism from the placeholder-skip of a block with a static-false `when:` (the block-static-skip amendment above): block-static-skip **reserves** the index and emits a per-child placeholder with its own `register:` (holds the in-file register visibility); group-drop does **not** reserve the index and physically removes the tasks. The split is strict by `IncludeGroupID`: a conditional-include task goes through the group-drop branch, a block task through the former placeholder-skip branch. Both coexist in one plan.
- **Safety (no `ErrOnChangesUnknownRegister`).** Dropping a group could leave a "dangling" external `onchanges:` referencing the `register:` of a dropped task (`resolveOnChanges` → `ErrOnChangesUnknownRegister`). This does not happen: a **cross-file register reference is already lint-forbidden offline** — the per-file `validateTaskRefs` validates the register-scope of each file separately, therefore an `onchanges:` of the main file on a register of an included file is rejected as `unknown_register_reference` at load (and an unexpanded include before render → `ErrUnexpandedInclude`). An external consumer of a dropped group's register is impossible in a valid scenario → `resolveOnChanges` after a group-drop does not encounter a dangling reference.
- **Both include-expansion paths (scenario + destiny), the per-pass cache is separate.** The group-drop works on **both** render paths that expand include into a flat plan: the scenario pass `Pipeline.Render` (`keeper/internal/render/pipeline.go`) and the isolated destiny pass `renderApplyDestiny` (`keeper/internal/render/destiny.go`). The mechanics are identical (the carry-through `Task.IncludeWhen`/`IncludeGroupID` is set by the same `config.ExpandIncludes`, called for destiny at `keeper/internal/artifact/destiny.go` and in the Trial-fixture resolver; the group-drop branch stands first in the loop, BEFORE `emitStaticWhenSkip` and block handling), but with two invariants of the destiny path: (1) the include-when of a destiny group is computed in the **isolated destiny-env** (`evalIncludeWhen(destinyIn, …)` — `destinyIn.Input` = the resolved `apply.input` + schema-defaults, WITHOUT scenario-scope: the scenario-input/vars/register are not visible), NOT in `parentIn`; (2) the cache `includeGroupKeep` is a **separate map per pass** (the scenario loop and each destiny pass hold their own), the destiny group ids do not intersect with scenario's. The register-isolation is identical to scenario (a cross-file register of a dropped destiny group is lint-forbidden per-file by the same `validateTaskRefs`, therefore `resolveOnChanges` does not encounter a dangling reference). The discriminator `IncludeGroupID` is orthogonal to block: at `keep=false` the whole group is dropped (including the destiny-block task and its children) BEFORE the `renderDestinyBlock` branch.

Normative edits — [`docs/destiny/tasks.md §4`](../destiny/tasks.md) (include: a permitted static `when:`, group-drop on both paths), [`docs/destiny/tasks.md §12`](../destiny/tasks.md) (open Q: a dynamic include-when is deferred). **Amendment 2026-06-29 (a mechanism of several named create scenarios + removal of the hardcoded default `create` + a bare incarnation as a lifecycle path). Status → amended.** The former model: exactly one reserved bootstrap scenario named `create`, which `incarnation.Create` (REST + MCP) ran first on a new incarnation; the name `create` was privileged (the resolver `ResolveCreateScenarios` unconditionally put the literal `CreateScenarioName` into the set — a "magic union"). This did not express a service with **several** different starting operations (for example Redis: "bring up VMs from scratch" vs "deploy onto a ready roster" vs "migrate from an external cluster") — each = its own set of tasks and its own `input:` schema, not a variation of a single `create`. Decisions:

- **A scenario's create capability — the top-level `create: true` in `scenario/<name>/main.yml`.** A service declares an arbitrary number of starting scenarios by marking them with the flag `create: true` (an optional bool key of the scenario DSL core, next to `name:`). The set of starting scenarios = **EXACTLY** `{sc | sc.create == true}`. The declaration is **per-scenario in the flag**, NOT a list in `service.yml` (scenarios are auto-discovered by directory anyway, [`docs/service/manifest.md`](../service/manifest.md); a centralized list would duplicate them). `destroy` is NOT marked with this flag (teardown is a special flow `DELETE /v1/incarnations/{name}`, not a run).
- **The name `create` is NO LONGER privileged.** The magic union with the literal `CreateScenarioName` is **removed** from `ResolveCreateScenarios`: the scenario `scenario/create/main.yml` enters the starting set **only if it itself carries `create: true`**, like any other. The former back-compat shortcut ("an empty choice → the default `create`") is dropped.
- **The operator's choice of the starting scenario — the field `create_scenario` (`POST /v1/incarnations`).** A three-branch contract (`ValidateCreateScenarioChoice`):
  - **the choice is non-empty + in the set** → this scenario starts; **not in the set / an invalid name** → `ErrCreateScenarioNotEligible` (422 `validation_failed`). Protection against bootstrapping an incarnation with an arbitrary (e.g. a day-2 `add_user`) scenario.
  - **the choice is empty + the set is NON-empty** (the service offers create scenarios) → `ErrCreateScenarioRequired` (422 `validation_failed`, with an enumeration of the eligible names). The choice is **mandatory**: `input:` is validated against the schema of a SPECIFIC scenario — without a choice there is nothing to apply. This is a deliberate shift from the former auto-`create`.
  - **the choice is empty + the set is EMPTY** (there is not a single `create: true`) → a **bare incarnation** (see below).
- **A bare incarnation — a new lifecycle path.** A service **without a single** `create: true` scenario, on creation, yields an incarnation in **`StatusReady` WITHOUT a run** (no apply_id is generated, `incarnation.created_scenario = NULL`). The incarnation is immediately ready for day-2 operations (the operator runs the needed scenarios via the ordinary `run`). This is a generalization of the existing path `lifecycle.auto_create: false` (an incarnation in `ready` without a bootstrap run) to the case "there is no bootstrap scenario at all". Bare is NOT an error and NOT a weakening: the barrier/state-commit invariant of §7 is not affected (there is no run → nothing to commit; the first mutation of `incarnation.state` will come from the first day-2 scenario).
- **`created_scenario` became NULLABLE** (migration 090, an amendment to 089). `NULL` = bare (created without a bootstrap scenario), non-empty = the name of the creating scenario. **`rerun-create` on a bare one → 409** (`ErrRerunScenarioNotCreate`): there is nothing to rerun. For a non-bare one, `rerun-create` reruns **exactly the creating** scenario (`incarnation.created_scenario`, not the hardcoded `create`) — an incarnation created by `create_from_souls`, on rerun, reruns `create_from_souls`. Existing incarnations (`created_scenario = 'create'` from the 089 DEFAULT) remain correct — for the Redis service `scenario/create/main.yml` now carries `create: true` (Phase 1), therefore `'create'` for them is still a valid bootstrap, no backfill is needed.
- **A contract shift of the public surface (semantic-breaking, not wire).** The semantics of the field `create_scenario` (OpenAPI `docs/keeper/openapi.yaml`, the MCP tool `keeper.incarnation.create`) **changed**: an empty value NO LONGER means "the default `create`". An external API/MCP client that sent create without `create_scenario` counting on auto-`create` now receives either 422 `create_scenario_required` (if the service offers create scenarios) or a bare incarnation (if not). The wire schema is additive (the field existed), but the **meaning** is breaking — note it in the release notes. MCP has parity with REST (the same three-branch logic).
- **The Redis service (pilot).** Three `create: true` scenarios: `create` (provision new VMs via `core.cloud.created` + onboarding, [ADR-061](0061-onboarding-await-and-midrun-reresolve.md)), `create_from_souls` (deploy onto an already-onboarded roster, without a provision section), `migrate_cluster` (migration from an external Redis via native replication = `create_from_redis`). The common deploy branches (`cluster.yml`/`sentinel.yml`) and the provision body (`redis-provision.yml`) — a service-level include (the filename fallback, §6) — are shared without duplication; `input:`/`compute:`/`state_changes:` remain inline per-scenario (there is no common include for them — a known price, cross-file drift is not caught by soul-lint).
- **Additive / forward-compat.** The flag `create:` is optional; a service with a single `create:true` scenario `create` behaves as before (choosing `create` explicitly or as the sole member of the set). soul-lint validates `create:` structurally (bool, `shared/config`).

Normative edits — [`docs/service/manifest.md`](../service/manifest.md), [`docs/keeper/operator-api.md`](../keeper/operator-api.md), [`docs/keeper/openapi.yaml`](../keeper/openapi.yaml) (the description of `create_scenario`). Implementation — `keeper/internal/scenario/create_scenarios.go` (`ResolveCreateScenarios` without the literal, `ValidateCreateScenarioChoice` three-branch `(name, bare, err)`, `ErrCreateScenarioRequired`/`ErrCreateScenarioNotEligible`), `keeper/internal/api/handlers/incarnation_typed.go` (`CreateTyped` — the bare branch StatusReady without a run), `keeper/internal/mcp/incarnation_create.go` (parity), `keeper/internal/incarnation/crud.go` (`created_scenario` NULLABLE, rerun on NULL → `ErrRerunScenarioNotCreate`), migration `keeper/migrations/090_incarnation_created_scenario_nullable.{up,down}.sql`. Guard tests — `keeper/internal/scenario/create_scenarios_test.go`, `keeper/internal/mcp/incarnation_create_test.go`, L0 `examples/service/redis/scenario/create_from_souls/tests/*`.

**Amendment 2026-06-29 (a new top-level scenario entity `extends:` + a service-level fragment of a covenant file — inheritance of a common section contract).** An **optional** top-level scenario key `extends: <covenant-name>` is introduced: a scenario **inherits** a common service-level contract of the sections `input:` / `compute:` / `state_changes:` / `validate:` from a covenant fragment file in the **root of the service repo** (isomorphic to the already-accepted `types.yml` — a service-level shared catalog, [ADR-062](0062-input-types.md)). It closes the duplication between several create scenarios of one service (the amendment 2026-06-29 above): `create` (provision new VMs + onboarding) and `create_from_souls` (deploy onto a ready roster) share almost the entire input/state contract, but `input:`/`compute:`/`state_changes:`/`validate:` are written inline in each → the two copies diverge, soul-lint does not catch cross-file drift. `extends:` eliminates this drift **constructively** (a single source = the covenant file, not the author's discipline). Decisions:

- **`extends:` NAMES the covenant file (resolves `<name>.yml`).** The value of `extends:` is the **name of the covenant file without the extension**: `extends: <name>` resolves to the file **`<name>.yml`** in the root of the service repo (the mechanism supports an arbitrary name, symmetric to `apply: { destiny: <name> }`). This is **not** "a logical name with a fixed `covenant.yml`" — `extends:` carries the file name itself. **Convention:** the service's common section contract is called **`covenant`** → the file **`covenant.yml`**, the reference **`extends: covenant`** (the recommended layout of a typical service; the mechanism allows several covenant files, but one common `covenant.yml` is a convention, see the MVP restrictions below).
- **Add-only merge, fail-closed (deep merge is REJECTED).** A covenant is a **minimum** (a common base); a scenario **ADDS a delta** on top. The merge is **shallow by the top key** of a section (input field / compute name / an element of the `state_changes`/`validate` list): the base and the delta are joined by top keys, there is no recursive merging of values. A duplicate top key (one input field declared both in the covenant and in the scenario; one compute name; an intersection of `set` fields of `state_changes`) → the error `section_key_conflict` (**fail-closed**: NOT last-wins, NOT override). The intent "the base = the common, the scenario = only its own additional" is made explicit: a collision = an author's error, not a silent overwrite. Deep merging (merging nested map values) is rejected — it hides which part of the schema came from the covenant, and reproduces the "diverging override" footgun from [ADR-062](0062-input-types.md) (the same logic there: `$type` is either a reference or inline, not their merge).
  - **The delta of the create variants.** `create` adds the provision specifics (`+provision`: the provision section/fields), `create_from_souls` — a **zero delta** (fully covered by the covenant, it carries no sections of its own above the base). This is exactly the target gain: the common contract lives in one file.
- **`form:` is NOT merged (local-only).** A scenario's UI `form:` block ([ADR-045](0045-param-dsl.md)/[ADR-042](0042-backend-driven-ui.md)) is **not** inherited from the covenant — it remains a property of the specific scenario (the form layout for a specific operation). A covenant carries the data contract, not the presentation.
- **Append order — covenant FIRST.** When splicing ordered sections (`compute:` — an ordered list per [ADR-009 amendment 2026-06-23 §2.4](#); `state_changes:` — an ordered list of CRUD verbs §7.1; `validate:` — an ordered list of rules §2.5) the covenant's elements go **before** the local ones. The reason: a local `compute[i]` is entitled to reference a covenant `compute[j]` (`compute` resolves left to right, §2.4 "the declaration order matters"), a local `state_changes`/`validate` — to rely on what has already been applied from the covenant. The reverse order (locals first) would break the forward-reference covenant→local.
- **The resolution point — `LoadScenarioManifestResolved` (BEFORE input validation/render).** The merge is performed at a **single manifest resolution point** (the working name `LoadScenarioManifestResolved`) — isomorphic to how the `$type` resolution ([ADR-062](0062-input-types.md)) and the `include` expansion (`config.ExpandIncludes`, §6) are unfolded BEFORE the consumers: load the service's `covenant.yml` → merge the 4 sections into the scenario manifest (add-only, fail-closed on a conflict) → then the **ordinary** pipeline on the ALREADY-merged manifest (input-merge → required → render → dispatch). The downstream consumers (`ValidateInput`, `Pipeline.Render`, `EvalAsserts`, `EvalValidateRules`, soul-lint) see a whole manifest — there is **no fragment-aware code outside the resolution point** (just as `$type` does not reach the render phase, the covenant does not reach the section consumers as a separate entity).
- **MVP restrictions.**
  - **One `extends:` per scenario** (not a list of covenants): multiple inheritance would introduce an order-dependent merge and cross-covenant conflicts — disproportionate to the goal of wave 1.
  - **A covenant does NOT extend a covenant** (no recursion/chain): a covenant is a flat leaf, it does not inherit another covenant. This removes the need for cycle-detection at this layer (unlike `$type`, where type→type is legal and cycle-detection is mandatory).
  - **A covenant carries ONLY 4 sections** — `input:` / `compute:` / `state_changes:` / `validate:`. **NOT** `tasks:` / `name:` / `create:` / `form:` / `extends:` (any other top key in a covenant file → a validation error). A covenant is the contract of a run's **data**, not its steps/identity: tasks (`tasks:`) are shared via the service-level `include:` (§6, the filename fallback), not via a covenant.
  - **One `covenant.yml` per service is a convention (not a hard restriction).** The mechanism resolves `extends: <name>` → `<name>.yml`, therefore a service can technically have several covenant files; but it is **recommended** to keep one common section contract in `covenant.yml` (`extends: covenant`). Several covenant files are justified only for unrelated families of scenarios with different common contracts — for a typical service this is unnecessary fragmentation.
- **form validation under `extends:` — a POST-merge invariant (S2b).** The pilot uncovered a phase contradiction: the invariant "`form:` ⊆ input" (the form fields reference only existing input parameters) was checked at the **schema phase** (`LoadScenarioManifestFromBytes`, NO-I/O), but under `extends:` the **effective** input exists only **after** the covenant-merge (covenant ∪ local), and the merge requires reading the covenant file from FS (keeper-side). The decision: under `extends:` form validation is **moved** from the schema phase to the **Resolved stage** — after `MergeCovenant`, against the effective input. A scenario **without** `extends:` — form validation stays at the schema phase, **bit-for-bit as now**. This is the same phase logic as with `$type` ([ADR-062](0062-input-types.md)): the schema is not final until the Resolved resolution, invariants "over the final schema" are computed after it.
- **A single covenant resolver in `shared/config` (S2b, no dialect).** An FS wrapper over `MergeCovenant` + `LoadCovenantFragmentFromBytes` + the post-merge form validation — **one** resolution path called by **all three** consumers: the keeper request-path (via `artifact`), the trial-harness, and `soul-lint`. Duplicating the merge/validation per consumer is forbidden — otherwise a dialect "how the lint merges/validates a covenant" ≠ "how prod does" ≠ "how trial does", i.e. **the same drift class** that this feature eliminates (the single source of the contract would be devalued by three diverging resolvers). **The layer invariant:** the `*FromBytes` functions (`LoadScenarioManifestFromBytes`, `LoadCovenantFragmentFromBytes`) stay **NO-I/O** (a pure parse of bytes); reading a covenant file from FS lives **strictly** in the Resolved layer (the FS wrapper), it does not leak into the byte parsers — this conforms to the invariant "the resolution point is BEFORE the consumers, there is no fragment-aware code outside it" above.
- **Forward-compat.** The key `extends:` is **optional**; a scenario **without** `extends:` behaves **bit-for-bit as now** (the resolution point, in the absence of `extends:`, is a no-op pass-through, the manifest is not touched; form validation stays at the schema phase). Existing services without a `covenant.yml` are not broken.
- **Relation to `types.yml`/`$type` and `IncludeResolver`.** A covenant is the **third** service-level shared mechanism in the same row: `types.yml` shares **named input schemas** ([ADR-062](0062-input-types.md)), the service-level `include:` (the two-level `IncludeResolver`, §6) shares **sets of tasks**, `covenant.yml` shares the **common section contract** between scenarios. All three resolve BEFORE the consumers and remove copy-paste of a different nature (type / tasks / section-contract); none of them introduces a wire entity.

Normative edits — [`docs/scenario/orchestration.md`](../scenario/orchestration.md) (a § about `extends:`/covenant next to the task-include §6), [`docs/service/manifest.md`](../service/manifest.md) (`covenant.yml` in the root of the service repo). Implementation (plan) — `shared/config` (the `extends:` grammar + loading/validation of `covenant.yml` with a 4-section sub-schema), the resolution point `LoadScenarioManifestResolved` (add-only merge, `section_key_conflict`), the consumers (`ValidateInput`/`Pipeline.Render`/soul-lint) call the resolution point without knowledge of the fragment. The Redis service is the pilot (`create`/`create_from_souls` via `extends: <covenant>`, the create↔create_from_souls drift is eliminated).
