# ADR-009. Scenario — полная DSL задач destiny; граница с destiny — рекомендация

- **Контекст.** Изначально действовал инвариант изоляции: scenario мог только `apply: { destiny: … }` (плюс `target`/оркестрация), но **не** `module:` напрямую — вся «работа» обязана была жить в изолированной, версионируемой, molecule-тестируемой destiny. На практике это вынуждало заводить отдельный destiny-репо ради одно-двух-шаговой инлайн-операции (probe роли, разовая команда, диагностика), которая никогда не переиспользуется. Инвариант мешал даже read-only-шагам (probe), без которых не реализуется волатильная роль из [ADR-008](0008-coven-stable-tags.md#adr-008-coven--только-стабильные-логические-теги).
- **Решение.**
  - **Инвариант «scenario без `module:`» снят полностью** (не только для read-only). Scenario получает **всё DSL-ядро задач destiny** из [`docs/destiny/tasks.md`](../destiny/tasks.md): `module:` (включая изменяющие модули), `templates`, task-level `vars:`, `register:`, `loop:`, `block:`, `parallel:`, `onchanges:`/`onfail:`/`require:`, `changed_when:`/`failed_when:`, `retry:`, `timeout:` — плюс оркестрационный слой сверху (`on:`/`where:`-таргетинг, `apply: { destiny: … }`, `state_changes`).
  - **DSL-ядро не дублируется.** Источник правды по ядру — `docs/destiny/tasks.md`; scenario-дельта — отдельный нормативный документ [`docs/scenario/orchestration.md`](../scenario/orchestration.md).
  - **destiny остаётся** самостоятельной сущностью: переиспользуемый, независимо-версионируемый (git ref, [ADR-007](0007-versioning-git-ref.md#adr-007-версионирование-артефактов--через-git-ref-а-не-через-поле-в-манифесте)), изолированный, molecule-тестируемый кирпич.
  - **Граница destiny / scenario — рекомендация, не стена.** Критерий выноса в destiny — три «да»: (1) переиспользование, (2) нужна molecule-идемпотентность, (3) изолируемо. Все три → выноси в destiny через `apply:`; иначе допустима инлайн-`module:`-реализация в scenario.
  - **Инвариант barrier / state-commit (не опция).** Коммит `incarnation.state` (`state_changes`) — это **cross-host final-barrier ПОСЛЕ безусловного ожидания всех parallel-задач всех хостов прогона**. Если хоть одна finally-failed → state **не** коммитится → `error_locked`. Это расширение final-barrier из [destiny/tasks.md §6](../destiny/tasks.md#6-параллелизм-parallel-true) с одного хоста на cross-host scenario-уровень. Подробнее — [`docs/scenario/orchestration.md`](../scenario/orchestration.md#7-инвариант-barrier--state-commit).
  - **`state_changes` несёт источник значения, не только allow-list.** `sets` — это map `<поле> → <CEL-выражение>`: декларация задаёт **и** какое поле `incarnation.state` обновить, **и** откуда взять значение (рендерится Keeper-side в scenario-CEL-окружении после барьера). Прежняя трактовка `sets` как плоского allow-list имён (только «какие поля разрешено писать», без источника) — заменена: без источника `incarnation.state` оставался пустым при успешном прогоне. Cross-host свёртка `sets` — last-wins по сортировке `SID`. **Контекст `sets` — `input.* / incarnation.* / soulprint.self.* / register.*`.** `register.*` (per-host probe-результат) пробрасывается через обратный канал `TaskEvent.register_data` → таблица `apply_task_register` → per-host register-карта после барьера (слайс 2 полной грамматики). `vars.* / essence.* / state.* / soulprint.hosts` и ключи `appends`/`modifies` (per-host коллекции) — future-расширение полной грамматики, недоступны (ошибка рендера → `error_locked`). Грамматика — [`docs/scenario/orchestration.md §7.1`](../scenario/orchestration.md#71-грамматика-state_changes--поле--источник-значения).
  - **Изоляция и `output:` destiny.** destiny публикует собственный результат через декларированный top-level `output:`; это симметрично `input:` и изоляцию destiny **не** нарушает (destiny не читает внешний контекст). Спецификация — [`docs/destiny/output.md`](../destiny/output.md).
- **Обоснование.** Один язык задач на оба слоя — нет двух диалектов, автор сценария не учит второй синтаксис. Probe (read-only `module:` + `register:`) становится естественным, чем и реализуется волатильная роль ([ADR-008](0008-coven-stable-tags.md#adr-008-coven--только-стабильные-логические-теги)). destiny не теряет свойств — она остаётся выбором по рекомендации, а не по принуждению.
- **Trade-off / риски.**
  - **Расширение scope [ADR-003](0003-destiny-format.md#adr-003-формат-destiny--yaml-с-типизированной-схемой-cuejson-schema).** Безопасный шаблонизатор отдельной фазой теперь обслуживает и scenario (`where:`-предикаты, `on:`-литералы, шаблоны). Этот вопрос (выбор шаблонизатора) закрыт [ADR-010](0010-templating.md#adr-010-шаблонизатор-cel-для-yaml-выражений-go-texttemplate-для-файлов). До ADR-010 он блокировал имплементацию scenario, не только destiny.
  - **Риск эрозии [ADR-007](0007-versioning-git-ref.md#adr-007-версионирование-артефактов--через-git-ref-а-не-через-поле-в-манифесте).** Инлайн-`module:`-мутации в scenario минуют независимое git-версионирование destiny (точечная правка `scenario/<name>/main.yml` не получает тег и ревью отдельного destiny-репо). **Смягчение:** граница-рекомендация выше + backlog lint-warn в [`docs/soul-lint.md`](../soul-lint.md) («инлайн-мутация в scenario без выноса в destiny»). Осознанно принятая плата за простоту типовых операций.
- **Связанные решения.** [ADR-008](0008-coven-stable-tags.md#adr-008-coven--только-стабильные-логические-теги) (probe вместо роли-как-Coven — реализуется именно снятием этого инварианта). Полная спецификация дельты — [`docs/scenario/`](../scenario/README.md).

**Amendment 2026-05-27 (invocation-time scope chunking — Tide).** [ADR-040](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override) вводит новую top-level сущность **Tide** — invocation-instance, разбивающую массовый прогон scenario на последовательные волны (**Surges**) с опциональным target/wave/concurrency-override. Принципиально:

- **Tide НЕ часть scenario-DSL** — она не записывается в YAML scenario, а параметризует **invocation** (HTTP/MCP request на `POST /v1/incarnations/{name}/scenarios/{scenario}`). Scenario-DSL ADR-009 остаётся неизменным; новые опт-поля (`target` / `wave` / `concurrency`) живут только в payload запроса, не в `main.yml`.
- **Декларация vs invocation — два уровня.** `serial:` / `run_once:` / `on:` / `where:` фиксируют разбиение и таргет **на уровне декларации scenario** (git-репо сервиса, авторское намерение). Tide добавляет **invocation-level** оркестрационный слой **над** scenario-runner-ом — оператор при запуске может разбить scope на waves, сузить таргет, переопределить concurrency, не меняя scenario. См. [`docs/scenario/orchestration.md §2.2.3`](../scenario/orchestration.md).
- **Совместимость с scenario-DSL:**
  - scenario без `run_once:` совместим с Tide безусловно.
  - scenario с `run_once: true` **несовместим** с invocation `wave` (`run_once` = ровно один хост таргета, разбивать на waves нечего) — ошибка валидации `tide_run_once_conflict` (422), [`docs/keeper/operator-api.md`](../keeper/operator-api.md).
  - scenario `concurrency:` (`serial:`) может быть **переопределён** invocation `concurrency:` (REPLACE-семантика — runtime-knob, не declared invariant).
- **Per-Surge barrier/state-commit — parity §7.** Каждый Surge — полноценный scenario-прогон со своим cross-host barrier и state-commit; barrier-инвариант §7 расширяется на каждый Surge независимо, а не на весь Tide целиком. Нормативно — механика per-Surge state-commit Tide (отдельный файл `docs/scenario/tide-state-commit.md` не создавался; Tide упразднён — см. [ADR-043](0043-voyage.md#adr-043-voyage--унифицированный-батчевый-прогон), per-Surge → per-incarnation state-commit).
- **AND-merge target-override (security-инвариант).** invocation `target.coven[]` / `target.where` только **сужают** scope (пересечение со scenario `on:` / `where:`), не расширяют. Оператор не может через invocation выйти за рамки декларированного scenario-таргета. Симметрично multi-label AND внутри `on:` — единый источник правды текста DSL [`docs/scenario/orchestration.md §3`](../scenario/orchestration.md).
- **`soulprint.hosts` / `incarnation.host_count` в Tide.** Аксессоры scenario-context резолвятся per-Surge от **target_resolved_souls** snapshot Tide (зафиксированный AND-merge на старте Tide), а не от исходного roster incarnation — каждый Surge видит только свою долю.

Цели и обоснование — [ADR-040](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override).

**Amendment 2026-05-29 (§7 — per-incarnation state-commit в Voyage = parity per-Surge).** [ADR-043](0043-voyage.md#adr-043-voyage--унифицированный-батчевый-прогон) (Voyage) поглощает Tide режимом `kind=scenario`. Подход **B1**: батч (**Leg**) = N **инкарнаций**, каждая инкарнация = полноценный scenario-run со своим cross-host barrier и **per-incarnation state-commit**. Это **прямая parity** per-Surge state-commit ([ADR-040](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override)) и сохраняет barrier-инвариант §7 без ослабления: `incarnation.state` коммитится только после успешного barrier-а **этой** инкарнации, частичный коммит (часть хостов инкарнации в одном Leg, остальные — в следующем) **запрещён**. Альтернатива B2 (Leg режет scope по хостам поперёк нескольких инкарнаций) **отвергнута именно потому, что ломает §7** — см. [ADR-043](0043-voyage.md#adr-043-voyage--унифицированный-батчевый-прогон) пункт 3. Внутри-инкарнационные волны по хостам остаются отдельной осью через scenario `serial:` (declaration-level), как и было.

**Amendment 2026-05-29 (Choir — additive `choirs[]` в scenario-аксессорах).** [ADR-044](0044-choir.md#adr-044-choir--именованная-топология-хостов-внутри-инкарнации) (Choir) добавляет к scenario-context-у **additive** поле `choirs[]` (список имён Choir-ов хоста в текущей инкарнации): `soulprint.self.choirs` (per-host, текущий хост) и `soulprint.hosts[].choirs` (cross-host аксессор хостов прогона, [`docs/scenario/orchestration.md §4.1`](../scenario/orchestration.md) — место будущей нормативной правки при S-T4). Это additive-расширение существующих стабильных полей `soulprint.hosts` (covens/sid/network/os) — не меняет грамматику `on:`/`where:`, доступно в `where:`-предикатах **без probe** (Choir — стабильный declared-факт, не волатильная роль). `on: [choir]` НЕ вводится в MVP (резолвер `on:` остаётся coven-only; сужение по Choir — через `where:`). Поглощённая declared-роль `soulprint.hosts[].role` ([ADR-044](0044-choir.md#adr-044-choir--именованная-топология-хостов-внутри-инкарнации) пункт 2) питается из Choir (через Voice), а не из `spec.hosts[].role`. Нормативная правка orchestration.md — слайсом S-T4, этот amendment её только декларирует.

**Amendment 2026-06-12 (опциональное поле задачи `id:` — стабильный адрес для per-task-changed-алертов).** Вводится новое **опциональное** поле DSL-ядра задачи `id:` — стабильный адрес задачи, по которому подписка из [ADR-052](0052-herald-notifications.md#adr-052-herald--tiding--уведомления-о-событиях-прогонов) (§h) адресует событие «эта таска изменила state» (per-task-changed-алерт). Решения:

- **Формат значения = register-формат** (тот же regex `^[a-z][a-z0-9_]*$`, что у `register:`): id и register живут в **одном адресном пространстве** подписки на per-task-changed-события, поэтому форма едина.
- **Опционально.** Отсутствие `id:` у задачи — норма; существующие scenario/destiny не ломаются.
- **`id:` ⊕ `register:` — взаимоисключимы.** Задача с `register:` уже адресуема по нему → `id:` избыточен и создаёт двусмысленность адреса. Оба ключа на одной задаче — ошибка валидации (`id_register_conflict`). `id:` предназначен для задач **без** `register:`.
- **Pilot-ограничение: `id:` только на module-задаче.** В первом заходе `id:` допустим лишь на module-задаче (у неё есть собственный changed-сигнал). На `block:`/`include:`-задаче — ошибка валидации (`id_unsupported_target`); снятие ограничения — отдельным заходом при запросе.
- **Уникальность `id:` по прогону** (после flatten `include`/`block`) — **в этом amendment не фиксируется**; это cross-ref-проверка линта (нужен flatten-контекст), отдельный слайс. Здесь — только per-task-формат + взаимоисключение с register + ограничение на module.
- **Резолв `id:` в RenderedTask, формирование событий и Tiding-привязка** — следующие слайсы, в эту правку не входят (только грамматика поля и валидация формы).

Per-task changed-адресация (адресное пространство `register ∪ id`) фиксируется со стороны уведомлений в [ADR-052](0052-herald-notifications.md#adr-052-herald--tiding--уведомления-о-событиях-прогонов) (§h). Документ-источник правды по полю задачи — [`docs/destiny/tasks.md §3`](../destiny/tasks.md#3-полный-список-блоков-задачи).
