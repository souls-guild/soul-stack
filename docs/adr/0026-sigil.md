# ADR-026. Sigil — целостность плагинов (Keeper-signed digest-индекс)

> **Имплементация почти полная.** Keeper-side: **git-резолвер каталога плагинов** (A1-S1, `keeper/internal/plugingit` — go-git F-fetch, R-nested-кеш `<ns>-<name>/<commit_sha>/` + `current`, см. (g)), подпись + реестр (S3, `keeper/internal/sigil`, `shared/pluginhost.BuildSigilBlock`), operator-facing allow/revoke/list (S4a REST `POST/GET/DELETE /v1/plugins/sigils*` + S4b MCP `keeper.plugin.*`, permissions `plugin.allow|revoke|list`, audit `plugin.allowed|revoked`). **Host-side verify-against-Sigil — LIVE** (S6): TOFU-ветка first-load заменена на verify по Sigil + multi-anchor-набор (`shared/pluginhost`); TOFU-сверка перед каждым `exec` остаётся defense-in-depth. **Multi-anchor ротация ключей подписи — LIVE (R3, ADR-026(h))**: реестр `sigil_signing_keys` (миграция 037, `keeper/internal/sigil/keys.go`), multi-anchor Signer (`LoadSigner`), broadcast `SigilTrustAnchors` + Redis-канал `sigil:anchors-changed` (cluster reload), operator-facing rotation (R3-S7, REST `POST/GET /v1/sigil/keys`, `POST /v1/sigil/keys/{key_id}/primary`, `DELETE /v1/sigil/keys/{key_id}` + MCP `keeper.sigil.key.*`, permissions `sigil.key-introduce|retire|list|set-primary`, audit `sigil.key-introduced|retired|primary-set`). Bootstrap-reply отдаёт набор якорей **из живого источника** (R3-S7), а не снимка старта — новый Soul после ротации получает актуальный набор. Не сделано: колонка `commit_sha` в `plugin_sigils` (A1-S3 — audit-метка, ВНЕ подписи). Полная нормативная спека Integrity-model — [`docs/keeper/plugins.md → Integrity-model`](../keeper/plugins.md#integrity-model).

- **Контекст.** Целостность плагинов (`soul-mod-*` / `soul-cloud-*` / `soul-ssh-*`) сейчас держится на **TOFU** (trust-on-first-use, [`docs/keeper/plugins.md → Integrity-model`](../keeper/plugins.md#integrity-model)): при первой загрузке бинаря в кеш host-а host **сам** считает SHA-256 и доверяет «как есть», фиксируя его в sidecar `.sha256`; перед каждым последующим `exec` digest сверяется. Это закрывает подмену бинаря **после** деплоя в shared-кеш, но **не** закрывает first-load и first-fetch: если атакующий подменил бинарь в artifact-source (или в момент Keeper-checkout-а git-ref-а) **до** первого появления на host-е — malicious-плагин проходит integrity-gate и форкается с правами service-user-а (`keeper`-плагины имеют доступ к Vault / PG / PKI) → **RCE-вектор**. Этот gap явно зафиксирован как «Известный gap — TOFU first-load» в plugins.md и требует **keeper-signed индекса**. Параллельно: неподписанный `manifest.yaml` рядом с бинарём → задекларированные `side_effects` / `required_capabilities` ([ADR-020(f)/(g)](0020-plugin-infrastructure.md#adr-020-plugin-инфраструктура-формат-manifest-handshake-lifecycle)) **подделываемы** (атакующий правит manifest вместе с бинарём). Имя сущности «реестр + подпись допуска плагинов» в [naming-rules.md](../naming-rules.md) не было зафиксировано — нужен propose-and-wait.
- **Сущность — `Sigil`** (печать доверия Keeper-а к конкретному бинарю плагина: digest + подпись фрагмента allow-list-индекса). Внесена в [naming-rules.md](../naming-rules.md). Прошла propose-and-wait; trust-модель — **Вариант A (Keeper-signed digest-индекс)**.
- **Решение.** Вводится **Sigil** — Keeper-side allow-list плагинов с подписью допуска. Keeper — единственный authority над тем, какие бинари допущены к запуску.
  - **(a) Authority и реестр.** Keeper держит allow-list `(namespace, name, ref) → sha256` в Postgres (новая таблица `plugin_sigils`). Запись попадает в реестр, **когда Архонт ЯВНО допускает плагин** через OpenAPI/MCP — человек подтверждает «этот бинарь с этим хешем допущен». TOFU-семантика «host сам решает доверять» **заменяется** на «authoritative-список ведёт Keeper». Источник истины по хешам — Postgres, authority — Keeper.
  - **(b) Sigil = digest + подпись.** При раздаче плагина на host Keeper отдаёт бинарь **плюс** подписанный своим ключом фрагмент индекса: `Sigil = sign_keeper(manifest_digest_block)`, где подписываемый блок несёт `(namespace, name, ref, binary_sha256, manifest)`. Soul (или Keeper-host для своих плагинов) перед seal / `exec` проверяет: (1) SHA-256 фактического бинаря == `binary_sha256` в Sigil; (2) подпись Sigil валидна публичным ключом Keeper-а. Любое расхождение → отказ, бинарь не получает управления (как сейчас `ErrPluginDigestMismatch`).
  - **(c) Покрытие манифеста (идея C).** Подпись Sigil покрывает `manifest` с **пришитым** полем `binary_sha256`. Следствие: задекларированные `side_effects` / `required_capabilities` / `protocol_version` перестают быть подделываемыми — нельзя подменить manifest, не сломав подпись. Сейчас host доверяет неподписанному `manifest.yaml`; с Sigil манифест становится частью подписанного блока.
  - **(d) Корень доверия.** Подписывающий ключ Keeper-а живёт в **Vault KV** — по аналогии с `secret/keeper/jwt-signing-key` ([ADR-014](0014-operator-identity.md#adr-014-identity-модель-оператора-archon)). **Публичная** часть приезжает Soul-у в **bootstrap** вместе с CA-chain (тем же каналом `BootstrapReply`, что и mTLS CA, [ADR-012(f)](0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add)) — Soul получает trust-anchor подписи Sigil ровно тогда же, когда trust-anchor mTLS.
  - **(e) Транспорт — only-add.** Раздача Sigil Soul-у — only-add proto-message-ы в `oneof payload` существующего `EventStream` ([ADR-012(c)](0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add) forward-compat only-add; [ADR-002](0002-transport-grpc-ha.md#adr-002-транспорт-keeper--souls--grpc-bidirectional-stream-поверх-mtls-ha-кластер-keeper) «один стрим» соблюдён), нового RPC нет. Поле `binary_sha256` в `proto/plugin` manifest — новое **опциональное** поле (only-add, не breaking). Имена закреплены при имплементации S2b: `PluginSigil` (broadcast одной печати), `SigilSnapshot` (полный active-набор, ReplaceAll) и `SigilTrustAnchors` (полный набор якорей, ReplaceAll) в `FromKeeper.oneof`; нормативная семантика их совмещения — в (h).
  - **(f) Режимы push / pull.** Sigil работает в обоих. **Push** (Keeper передаёт плагины ОТ Keeper-а Soul-у по mTLS — Keeper уже trust-anchor): Sigil едет вместе с бинарём. **Pull** (Soul-демон): Soul верифицирует Sigil против публичного ключа Keeper-а из bootstrap-а. В обоих случаях верификация — до seal / exec.
  - **(g) Семантика `ref` — git-verified (Вариант A, F-fetch).** `ref` каталога `keeper.yml::plugins.{cloud_drivers,ssh_providers}` теперь **git-verified**: Keeper сам резолвит `source`+`ref` в локальный слот кеша через **go-git** (pure-Go, без форка системного `git`). Цепочка резолва — shallow `clone`→`fetch`→`ResolveRevision(<ref>^{commit})` (резолв ref-а в 40-hex `commit_sha`)→detached-HEAD `checkout`; из checkout-а извлекается **УЖЕ собранный** бинарь `dist/<binary-name>` + `manifest.yaml` (**F-fetch — компиляции на Keeper НЕТ**, бинарь приходит готовым из репозитория плагина). Граница «verified» = «Keeper зачекаутил именно этот `ref` и зафиксировал результат (`commit_sha` + `binary_sha256`)», **НЕ** bit-reproducibility сборки.
    - **Раскладка кеша — R-nested.** `<cacheRoot>/<ns>-<name>/<commit_sha>/` (иммутабельный слот per-commit: `manifest.yaml` + бинарь) + symlink `<cacheRoot>/<ns>-<name>/current → <commit_sha>` (атомарно переставляемый указатель на активный слот). **Single-active-на-пару** на Keeper-стороне (как и Soul-сторона — single-active): `current` указывает ровно на один `commit_sha`, но **несколько `commit_sha`-слотов под одной `(namespace, name)` сосуществуют** (прежнее утверждение «single-slot, сосуществование двух `ref` невозможно» снято). `plugin.allow` читает бинарь+manifest АКТИВНОГО слота через `current` (`pluginhost.ReadSlot`), `ref` в lookup слота по-прежнему не участвует.
    - **`commit_sha` — audit-метка происхождения, ВНЕ подписи.** DST остаётся `soul-stack/sigil/v1`, подпись Sigil — над блоком `(namespace, name, ref, binary_sha256, manifest_sha256)` (см. формат блока в [plugins.md](../keeper/plugins.md#формат-подписываемого-блока-нормативный-s3)); `commit_sha` в подписываемый блок **не входит** — он добавится колонкой в `plugin_sigils` как audit-метка («какой коммит дал этот бинарь») при S3 (помечено «вводится при impl»). **Authority целостности — `sha256` + подпись Keeper-а** (инвариант (b) не ослаблен); `commit_sha`/`ref` несут происхождение и читаемость, не доверие.
    - **Git-стек — go-git, hardening.** hooks **не исполняются** by-design (go-git их не запускает ни при clone, ни при checkout); submodules **не рекурсятся** (`RecurseSubmodules` не передаётся); транспорта `ext::` (исполнение внешней команды как «транспорта») в go-git **нет в принципе**; `file://` **заперт scheme-allowlist-ом** (`validateGitScheme`: prod-allowlist `https://`/`ssh://`/scp-форма; `file://` — только под env-флагом `SOUL_STACK_ALLOW_FILE_REPOS=1` для dev/test). **Нет runtime-зависимости от бинаря `git`** на keeper-host. Hardening: shallow (`Depth=1`) + context-timeout (`plugins.fetch_timeout`, дефолт 120s); рабочий клон (`plugins.work_root`) **СТРОГО вне `cacheRoot`** — `.git` и checkout не должны попадать в кеш-каталог, который читают Discover/ReadSlot; **size-limit по объёму** — `plugins.max_clone_size_mb` (дефолт 1024 MiB, рабочее дерево клона: du-walk checkout + `.git`, проверка после checkout до извлечения артефакта) + `plugins.max_artifact_size_mb` (дефолт 256 MiB, бинарь `dist/<binary-name>`: `os.Stat` перед copy + `io.LimitReader` во время copy).
    - **git-egress — HIGH security-риск.** Внешний git-вызов с config-управляемым `source` — поверхность атаки. `source` operator-asserted, но репозиторий недоверенный, а timeout ограничивает egress лишь по времени; **size-limit — РЕАЛИЗОВАН** (см. выше: clone-tree + artifact cap, fail-closed → слот не создаётся, плагину нечего допускать; sentinel-ы `ErrCloneTooLarge`/`ErrArtifactTooLarge`). Остаток security-pass-а перед прод (`noexec` на слот, sandbox git-операций) — обязательное отложенное (см. Consequences).
    - Per-entry резолв **fail-closed**: сломанная запись каталога (нерезолвящийся ref / нет manifest / нет `dist/<binary-name>` / недоступный remote) превращается в per-entry warning, Keeper не падает; sentinel-ы — `ErrRefNotResolved` / `ErrManifestNotFound` / `ErrArtifactNotFound` / `ErrSourceUnavailable`.
  - **(h) Совмещение single-vs-snapshot и multi-anchor — replace-семантика (Вариант A).** Транспорт раздаёт два рода сообщений; нормативная семантика их совмещения такова:
    - **`SigilSnapshot{repeated PluginSigil}` — ЕДИНСТВЕННЫЙ источник истины** active-набора допусков на Soul-стороне. Применяется как **ReplaceAll**: Soul **заменяет весь** свой локальный набор Sigil-ов этим списком, **не** делает upsert. Допуск, отсутствующий в snapshot, Soul **забывает** — так срабатывает revoke/retire: после revoke Архонтом Keeper шлёт новый snapshot без отозванного допуска, и доступ закрывается **near-instant**, без перезапуска Soul-а. Пустой `sigils[]` = ни один плагин не допущен.
    - **Одиночный `FromKeeper.plugin_sigil` (field 5) — роль broadcast-уведомления** о новом допуске, **НЕ мутация набора**. Soul по нему **НЕ делает upsert** в авторитетный набор; авторитет — **только snapshot**. Одиночное сообщение служит сигналом «появился новый Sigil» (например, чтобы Soul инициировал применение свежего snapshot-а), но само по себе active-набор не меняет.
    - **`SigilTrustAnchors{repeated pubkey_pem}` — ReplaceAll набора якорей** подписи. Runtime-доставка набора **полностью замещает** bootstrap-anchors (это **replace, не мёрж**): после получения `SigilTrustAnchors` Soul верифицирует подписи **именно этим** набором; anchor, отсутствующий в наборе, Soul забывает (retire старого ключа). Multi-anchor позволяет безразрывную ротацию ключа подписи (любой anchor из набора верифицирует подпись).
    - **`BootstrapReply`: приоритет set > single, ЖИВОЙ источник (R3-S7).** При непустом `sigil_pubkey_pem_set` (field 6) одиночное `sigil_pubkey_pem` (field 5) **игнорируется** (set — авторитет). **Оба пустых = Sigil выключен** на этом хосте (verify плагинов не выполняется). Bootstrap-handler читает набор **при каждом reply из живого источника** (`TrustAnchorSource`, тот же atomic-holder, что обновляет watcher `sigil:anchors-changed`), а **не из снимка старта** — иначе новый Soul, онбордящийся между ротацией и своим первым connect-ом, получил бы устаревший набор (опасно при Retire: доверял бы выведенному ключу либо отвергал свежий primary).
    - **Retire-инвариант (нормативный, R3-S7).** Вывод ключа из набора (`Retire`) безопасен **только** когда выполнены два условия: (1) новый набор **разошёлся по кластеру** — гарантирует `PublishAnchorsChanged` (Redis-канал `sigil:anchors-changed`) → `reloadAnchors` на каждой ноде (re-build Signer + re-broadcast `SigilTrustAnchors` + обновление holder-а), а **при потере pub/sub-сигнала** — TTL-fallback-тик (`sigil_anchors_reload_interval`, см. (h)) самоисцеляет отставшую ноду за интервал; распространение видно оператору по `keeper_sigil_anchors_last_delivered` **до** Retire; (2) bootstrap-источник **живой** — bootstrap-reply отдаёт актуальный набор (предыдущий пункт). Оба условия теперь держатся имплементацией. Дополнительно `Retire` запрещён для **primary** напрямую (сперва `SetPrimary` другому active-ключу) и для **последнего active** (набор не должен опустеть — симметрия self-lockout RBAC); инварианты — транзакционно под `FOR UPDATE` (`keeper/internal/sigil/keys.go`).
    - **Почему так.** Authority над целостностью — у Keeper; Soul не должен накапливать локальную историю допусков, расходящуюся с keeper-state. ReplaceAll делает revoke и ротацию атомарными и наблюдаемыми (один snapshot = полная картина), исключает рассинхрон «у Soul-а остался отозванный Sigil» без явной команды забвения. Одиночный broadcast как «уведомление, не мутация» снимает риск, что потеря/дубль одиночного сообщения исказит набор: набор всегда восстанавливается из следующего snapshot-а.
- **Нормативный инвариант безопасности.** Плагин **не получает управления**, пока не пройдена двойная проверка (digest бинаря == Sigil + валидная подпись Keeper-а). First-load перестаёт быть «доверяй как есть» — он становится «доверяй, только если Keeper явно допустил этот хеш и подписал его». `manifest` (а значит `side_effects` / `capabilities`) — под подписью, не подделываем.
- **Реконсиляция.** [ADR-020(f)/(g)](0020-plugin-infrastructure.md#adr-020-plugin-инфраструктура-формат-manifest-handshake-lifecycle) (`required_capabilities` / `side_effects` как strict-контракт) — **усиливается**: декларации становятся доверенными благодаря (c). [ADR-012(f)](0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add) (bootstrap несёт CA-chain) — расширяется публичным ключом Sigil тем же каналом. [ADR-014](0014-operator-identity.md#adr-014-identity-модель-оператора-archon) (Vault KV `secret/keeper/jwt-signing-key`) — задаёт паттерн для Sigil-ключа в Vault. [ADR-016](0016-parity-license.md#adr-016-стратегия-parity-с-saltstackansible-и-лицензия-soul-stack) (community-плагины) — на авторов плагинов **не ложится никакой нагрузки** (их ключи / подписи не нужны); допуск делает оператор через Keeper.
- **Consequences.**
  - **Proto-контракт раздачи (введён S2b, only-add в `EventStream` oneof):** `PluginSigil` (одна печать-broadcast, `FromKeeper` field 5), `SigilSnapshot{repeated PluginSigil}` (полный active-набор, ReplaceAll, field 7), `SigilTrustAnchors{repeated pubkey_pem}` (полный набор якорей, ReplaceAll, field 8) + **новое опциональное поле `binary_sha256`** в `proto/plugin` manifest (only-add). `BootstrapReply` несёт одиночный `sigil_pubkey_pem` (field 5) и multi-anchor `sigil_pubkey_pem_set` (field 6, приоритет set > single; см. (h)).
  - **Новая PG-таблица `plugin_sigils`** (`(namespace, name, ref)` → `sha256` + audit-поля допуска) — реестр allow-list-а. Точная схема — задача имплементации. **Колонка `commit_sha`** (git-verified происхождение бинаря, (g)) — **вводится при impl S3**, ВНЕ подписываемого блока (audit-метка, не trust).
  - **Git-резолвер каталога плагинов — `keeper/internal/plugingit`** (A1-S1, реализован). Keeper при старте резолвит `keeper.yml::plugins.{cloud_drivers,ssh_providers}` через go-git в R-nested-кеш `<cacheRoot>/<ns>-<name>/<commit_sha>/` + symlink `current` ((g)). Конвенция артефакта — `dist/<binary-name>` (F-fetch). Sentinel-ы резолва: `ErrRefNotResolved` / `ErrManifestNotFound` / `ErrArtifactNotFound` / `ErrSourceUnavailable` / `ErrCloneTooLarge` / `ErrArtifactTooLarge` (per-entry warnings, fail-closed). Новые config-поля — `plugins.work_root` (корень рабочих клонов, СТРОГО вне `cache_root`), `plugins.fetch_timeout` (дефолт 120s), `plugins.max_artifact_size_mb` (дефолт 256 MiB) и `plugins.max_clone_size_mb` (дефолт 1024 MiB); scheme-allowlist на `source` (`validateGitScheme`).
  - **Security-pass git-egress** (git-egress HIGH-риск ((g))): **size-limit — реализован** (clone-tree + artifact byte-cap, fail-closed, MEDIUM-1 security-аудита закрыт); остаток (`noexec` на слот / sandbox git-операций) — обязателен перед прод, отложен.
  - **GC старых `commit_sha`-слотов** (несколько слотов на пару после ротаций `ref`/коммитов) — отложен; кандидат на Reaper-правило, имя — отдельный propose-and-wait при impl.
  - **MCP / OpenAPI endpoint допуска** — Архонт допускает `(namespace, name, ref, sha256)` в реестр; новые RBAC-permissions (`plugin.allow` / `plugin.revoke` / `plugin.list`) и audit-events (`plugin.allowed` / `plugin.revoked` / `plugin.verify_failed`). CRUD-поверхность стартует как stub-каталог (записи в operator-api.md / mcp-tools.md / rbac.md по факту имплементации).
  - **Замена TOFU-ветки first-load** в `shared/pluginhost` на verify-against-Sigil. TOFU-сверка перед каждым `exec` остаётся как defense-in-depth для shared-кеша; меняется только first-load (вместо «доверяй как есть» — «верифицируй Sigil»). Раздача правки `shared/pluginhost` — отдельный slice после ревью ADR.
  - **Подписывающий ключ Keeper-а в Vault KV** (новый путь, по паттерну `secret/keeper/jwt-signing-key`); **публичный ключ в bootstrap** (`BootstrapReply`, рядом с CA-chain) — одиночный `sigil_pubkey_pem` плюс multi-anchor `sigil_pubkey_pem_set` для безразрывной ротации (R3, replace-семантика; runtime-доставка `SigilTrustAnchors` полностью замещает bootstrap-anchors, см. (h)).
  - **Multi-anchor реестр ключей подписи (R3, реализован).** Новая PG-таблица **`sigil_signing_keys`** (миграция 037): `key_id` (SHA-256 SPKI, hex) → `pubkey_pem` (SPKI) + `vault_ref` (приватник в Vault) + `is_primary` / `status` (active|retired) + audit-FK на `operators`. **Приватник НИКОГДА не в Postgres** — только публичная часть + ссылка на Vault. Инварианты: ровно один primary среди active (partial unique index `sigil_signing_keys_one_primary`), ≥1 active при Retire. Multi-anchor Signer (`keeper/internal/sigil`): primary подписывает новые Sigil-ы, набор anchors = публичные ключи всех active. Пустой реестр → fallback на одиночный `sigil.signing_key_ref` из cfg (старый путь не ломается).
  - **Cluster-wide ротация — Redis-канал `sigil:anchors-changed` (R3-S6) + TTL-fallback.** После мутации реестра ключей (introduce / set-primary / retire) мутирующая нода `PublishAnchorsChanged` в `sigil:anchors-changed`; **каждая** нода (включая мутирующую) по подписке `reloadAnchors`: re-build Signer (новый primary + якоря) → `SetSigner` в sigil-service → обновление verify-набора keeper-host-а → обновление holder-а connect-time broadcast → re-broadcast `SigilTrustAnchors` подключённым Soul-ам. Отдельный канал от `sigil:invalidate` (тот про допуски `plugin_sigils`, этот — про ключи их подписи). Pub/sub best-effort, но **поверх него — TTL-fallback-тик** (`sigil_anchors_reload_interval`, default 30s): тот же `reloadAnchors` периодически по тикеру, поэтому пропущенный сигнал самоисцеляется за интервал (образец TTL-poll RBAC [ADR-028(d)](0028-rbac-storage.md#adr-028-rbac-storage--postgres) / `Summons` poll-fallback [ADR-027](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim)). Тик поднимается **независимо от Redis** (при single-instance/dev без Redis — единственный путь доезда runtime-ротации без рестарта). `reloadAnchors` idempotent/fail-safe (на ошибке re-build состояние не трогается), поэтому двойной reload (тик + pub/sub) безопасен.
  - **Operator-facing rotation API (R3-S7).** REST `POST /v1/sigil/keys` (introduce: Keeper генерирует ed25519-пару, пишет приватник в Vault KV `secret/keeper/sigil-keys/<key_id>`, вставляет публичную часть в реестр; ответ несёт `key_id` + `pubkey_pem`, **приватник НИКОГДА не возвращается**), `GET /v1/sigil/keys` (list active), `POST /v1/sigil/keys/{key_id}/primary` (set-primary), `DELETE /v1/sigil/keys/{key_id}` (retire). Паритет MCP: `keeper.sigil.key.introduce|list|set-primary|retire`. Permissions `sigil.key-introduce|retire|list|set-primary` (resource `sigil`, hyphenated action — грамматика `<resource>.<action>`, 3-сегментный `sigil.key.*` = MCP-tool, не permission). Audit `sigil.key-introduced|retired|primary-set` (payload `key_id` + caller AID, **без приватника**). Метрики `keeper_sigil_signing_keys_active` (gauge числа active-ключей), `keeper_sigil_anchors_rebroadcast_total` (counter проходов re-broadcast набора якорей) и `keeper_sigil_anchors_last_delivered` (gauge числа Soul-ов, которым последний набор реально доехал — оператор видит распространение **ПЕРЕД** Retire). Гонка установки primary (one_primary index, 23505) → sentinel `ErrConcurrentPrimary` → 409 (retry).
  - **TTL-fallback для `sigil:anchors-changed` — ЗАКРЫТ (был KNOWN GAP).** Распространение нового набора якорей по кластеру было best-effort Redis pub/sub (`PublishAnchorsChanged` → `reloadAnchors`) **без периодического poll-tick-а**: потеря сигнала → отставшая нода держала старый набор якорей до рестарта (load-at-start), что опасно при **Retire** (нода локально *fail-open* доверяла бы выведенному ключу для Soul-а, онбордящегося через неё). **Закрыто** TTL-fallback-тиком в `reloadAnchors` (`runAnchorsReloadTicker`, интервал `sigil_anchors_reload_interval` default 30s, по образцу `rbacHolder.Run` TTL-poll / `Summons` poll-fallback): пропущенный сигнал самоисцеляется за интервал, тик независим от Redis. Наблюдаемость доставки re-broadcast-а — метрики `keeper_sigil_anchors_rebroadcast_total` (проходы) + `keeper_sigil_anchors_last_delivered` (сколько Soul-ам набор реально доехал) + лог числа active-якорей и delivered на каждый reload — оператор видит распространение набора **ПЕРЕД** Retire старого ключа.
  - **KNOWN GAP «orphan Vault-секрет при PG-fail Introduce».** `POST /v1/sigil/keys` (introduce) пишет приватник в Vault KV `secret/keeper/sigil-keys/<key_id>` **до** вставки публичной части в `sigil_signing_keys`. Если PG-вставка падает после успешной записи в Vault — приватник остаётся **сиротой** в Vault (нет соответствующей строки реестра). Reverse-cleanup (удаление из Vault на PG-fail) **сознательно не делается** — orphan безвреден (ключ нигде не объявлен primary/active, подписи им не производятся), а компенсирующая транзакция через два хранилища (Vault + PG) хрупка сама по себе. GC сиротских секретов — **кандидат на Reaper-правило**, отложен вместе с GC `commit_sha`-слотов ((g)). Имя правила — отдельный propose-and-wait при impl.
  - **Известный gap «TOFU first-load» — ЗАКРЫТ** (S6 LIVE): раздел Integrity-model в [plugins.md](../keeper/plugins.md) переписан под Sigil + multi-anchor; host-side first-load теперь verify-against-Sigil, не TOFU.
  - **Имена в [naming-rules.md](../naming-rules.md)** — `Sigil` (+ определение), proto-message `PluginSigil` / `SigilSnapshot` / `SigilTrustAnchors`, поле `sigil_pubkey_pem_set` (приоритет set > single), PG-таблицы `plugin_sigils` и `sigil_signing_keys`, RBAC-perms (`plugin.*` + `sigil.key-*`), audit-events (`plugin.*` + `sigil.key-*`), Redis-канал `sigil:anchors-changed`.
- **Отложенное (post-MVP, без breaking changes).** **Вариант B (author-signed коллекции / издатели)** — граница доверия на уровне коллекции по `(namespace, ref)` (один publisher → один ключ) — откладывается до module-collections ([module-collections.md, open Q №2](../module-collections.md)). Sigil (Вариант A) — допуск конкретных хешей — текущая модель MVP; коллекции/издатели расширяют её аддитивно, не ломая `(namespace, name, ref) → sha256`-реестр. **Security-pass git-egress** — size-limit клона/артефакта **реализован** ((g), MEDIUM-1 закрыт); остаток (`noexec` на слот / sandbox git-операций) обязателен перед прод, git-egress HIGH-риск. **GC старых `commit_sha`-слотов** — кандидат на Reaper-правило, имя — отдельный propose-and-wait. **Колонка `commit_sha` в `plugin_sigils`** — вводится при S3 (audit-метка происхождения, ВНЕ подписи).
- **Trade-off.** Допуск концентрируется на Keeper (single authority) — но Keeper **уже** корень mTLS / рендера Destiny / Vault-доступа, новой trust-поверхности почти не добавляется. **Нулевая нагрузка на авторов плагинов** (не нужны их ключи / подписи) — допуск целиком на операторе через Keeper; цена — оператор обязан явно допустить каждый `(namespace, name, ref, sha256)` (человеко-шаг на каждый новый хеш), но это и есть осознанный контроль supply-chain. Альтернатива — author-signed (Вариант B) — снимает человеко-шаг, но требует key-management у авторов community-плагинов; отложена.
