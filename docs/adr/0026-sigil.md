# ADR-026. Sigil — plugin integrity (Keeper-signed digest index)

> **Implementation almost complete.** Keeper-side: **git resolver for the plugin catalog** (A1-S1, `keeper/internal/plugingit` — go-git F-fetch, R-nested cache `<ns>-<name>/<commit_sha>/` + `current`, see (g)), signing + registry (S3, `keeper/internal/sigil`, `shared/pluginhost.BuildSigilBlock`), operator-facing allow/revoke/list (S4a REST `POST/GET/DELETE /v1/plugins/sigils*` + S4b MCP `keeper.plugin.*`, permissions `plugin.allow|revoke|list`, audit `plugin.allowed|revoked`). **Host-side verify-against-Sigil — LIVE** (S6): the TOFU first-load branch is replaced with verify by Sigil + a multi-anchor set (`shared/pluginhost`); the TOFU check before every `exec` remains as defense-in-depth. **Multi-anchor rotation of signing keys — LIVE (R3, ADR-026(h))**: registry `sigil_signing_keys` (migration 037, `keeper/internal/sigil/keys.go`), multi-anchor Signer (`LoadSigner`), broadcast `SigilTrustAnchors` + Redis channel `sigil:anchors-changed` (cluster reload), operator-facing rotation (R3-S7, REST `POST/GET /v1/sigil/keys`, `POST /v1/sigil/keys/{key_id}/primary`, `DELETE /v1/sigil/keys/{key_id}` + MCP `keeper.sigil.key.*`, permissions `sigil.key-introduce|retire|list|set-primary`, audit `sigil.key-introduced|retired|primary-set`). The bootstrap reply serves the anchor set **from a live source** (R3-S7), not a start-time snapshot — a new Soul after rotation receives the current set. Not done: the `commit_sha` column in `plugin_sigils` (A1-S3 — an audit marker, OUTSIDE the signature). Full normative Integrity-model spec — [`docs/keeper/plugins.md → Integrity-model`](../keeper/plugins.md#integrity-model).

- **Context.** Plugin integrity (`soul-mod-*` / `soul-cloud-*` / `soul-ssh-*`) currently rests on **TOFU** (trust-on-first-use, [`docs/keeper/plugins.md → Integrity-model`](../keeper/plugins.md#integrity-model)): on the first load of a binary into the host cache, the host **itself** computes the SHA-256 and trusts it "as is", recording it in a sidecar `.sha256`; before every subsequent `exec` the digest is checked. This closes binary substitution **after** deployment into the shared cache, but **does not** close first-load and first-fetch: if an attacker substituted the binary in the artifact source (or at the moment the Keeper checks out the git ref) **before** its first appearance on the host — a malicious plugin passes the integrity gate and is forked with the service-user's privileges (`keeper` plugins have access to Vault / PG / PKI) → **an RCE vector**. This gap is explicitly recorded as "Known gap — TOFU first-load" in plugins.md and requires a **keeper-signed index**. In parallel: an unsigned `manifest.yaml` next to the binary → the declared `side_effects` / `required_capabilities` ([ADR-020(f)/(g)](0020-plugin-infrastructure.md#adr-020-plugin-infrastructure-manifest-format-handshake-lifecycle)) are **forgeable** (the attacker edits the manifest together with the binary). The name of the entity "plugin allow-list registry + allow signature" was not fixed in [naming-rules.md](../naming-rules.md) — a propose-and-wait is needed.
- **Entity — `Sigil`** (the Keeper's seal of trust for a specific plugin binary: digest + signature of a fragment of the allow-list index). Added to [naming-rules.md](../naming-rules.md). Passed propose-and-wait; trust model — **Variant A (Keeper-signed digest index)**.
- **Decision.** **Sigil** is introduced — a Keeper-side plugin allow-list with an allow signature. The Keeper is the sole authority over which binaries are allowed to run.
  - **(a) Authority and registry.** The Keeper holds an allow-list `(namespace, name, ref) → sha256` in Postgres (new table `plugin_sigils`). A record enters the registry **when the Archon EXPLICITLY allows a plugin** via OpenAPI/MCP — a human confirms "this binary with this hash is allowed". The TOFU semantics "the host decides to trust on its own" is **replaced** with "the authoritative list is maintained by the Keeper". The source of truth for hashes is Postgres, the authority is the Keeper.
  - **(b) Sigil = digest + signature.** When distributing a plugin to a host, the Keeper serves the binary **plus** a fragment of the index signed with its key: `Sigil = sign_keeper(manifest_digest_block)`, where the signed block carries `(namespace, name, ref, binary_sha256, manifest)`. Before seal / `exec`, the Soul (or the Keeper host for its own plugins) checks: (1) the SHA-256 of the actual binary == `binary_sha256` in the Sigil; (2) the Sigil signature is valid under the Keeper's public key. Any mismatch → rejection, the binary does not gain control (as `ErrPluginDigestMismatch` today).
  - **(c) Manifest coverage (idea C).** The Sigil signature covers the `manifest` with the `binary_sha256` field **attached**. Consequence: the declared `side_effects` / `required_capabilities` / `protocol_version` stop being forgeable — the manifest cannot be substituted without breaking the signature. Currently the host trusts an unsigned `manifest.yaml`; with Sigil the manifest becomes part of the signed block.
  - **(d) Root of trust.** The Keeper's signing key lives in **Vault KV** — by analogy with `secret/keeper/jwt-signing-key` ([ADR-014](0014-operator-identity.md#adr-014-operator-identity-model-archon)). The **public** part arrives at the Soul in **bootstrap** together with the CA-chain (through the same `BootstrapReply` channel as the mTLS CA, [ADR-012(f)](0012-keeper-soul-grpc.md#adr-012-keepersoul-grpc-contract-one-eventstream-with-oneof-keeper-side-render-forward-compat-only-add)) — the Soul receives the Sigil signature trust-anchor at exactly the same time as the mTLS trust-anchor.
  - **(e) Transport — only-add.** Distributing a Sigil to the Soul — only-add proto messages in the `oneof payload` of the existing `EventStream` ([ADR-012(c)](0012-keeper-soul-grpc.md#adr-012-keepersoul-grpc-contract-one-eventstream-with-oneof-keeper-side-render-forward-compat-only-add) forward-compat only-add; [ADR-002](0002-transport-grpc-ha.md#adr-002-transport-keeper--souls--grpc-bidirectional-stream-over-mtls-ha-keeper-cluster) "one stream" preserved), no new RPC. The `binary_sha256` field in the `proto/plugin` manifest is a new **optional** field (only-add, not breaking). Names fixed at the S2b implementation: `PluginSigil` (broadcast of one seal), `SigilSnapshot` (full active set, ReplaceAll) and `SigilTrustAnchors` (full anchor set, ReplaceAll) in `FromKeeper.oneof`; the normative semantics of combining them — in (h).
  - **(f) Push / pull modes.** Sigil works in both. **Push** (the Keeper transfers plugins FROM the Keeper to the Soul over mTLS — the Keeper is already a trust-anchor): the Sigil rides along with the binary. **Pull** (Soul daemon): the Soul verifies the Sigil against the Keeper's public key from bootstrap. In both cases verification is before seal / exec.
  - **(g) `ref` semantics — git-verified (Variant A, F-fetch).** The `ref` of the catalog `keeper.yml::plugins.{cloud_drivers,ssh_providers}` is now **git-verified**: the Keeper itself resolves `source`+`ref` into a local cache slot via **go-git** (pure-Go, no fork of the system `git`). The resolution chain is shallow `clone`→`fetch`→`ResolveRevision(<ref>^{commit})` (resolving the ref into a 40-hex `commit_sha`)→detached-HEAD `checkout`; from the checkout the **ALREADY built** binary `dist/<binary-name>` + `manifest.yaml` is extracted (**F-fetch — NO compilation on the Keeper**, the binary comes ready from the plugin repository). The "verified" boundary = "the Keeper checked out exactly this `ref` and fixed the result (`commit_sha` + `binary_sha256`)", **NOT** bit-reproducibility of the build.
    - **Cache layout — R-nested.** `<cacheRoot>/<ns>-<name>/<commit_sha>/` (an immutable per-commit slot: `manifest.yaml` + binary) + symlink `<cacheRoot>/<ns>-<name>/current → <commit_sha>` (an atomically repointable pointer to the active slot). **Single-active-per-pair** on the Keeper side (as on the Soul side — single-active): `current` points to exactly one `commit_sha`, but **several `commit_sha` slots under one `(namespace, name)` coexist** (the earlier claim "single-slot, coexistence of two `ref`s is impossible" is withdrawn). `plugin.allow` reads the binary+manifest of the ACTIVE slot via `current` (`pluginhost.ReadSlot`), `ref` still does not participate in the slot lookup.
    - **`commit_sha` — a provenance audit marker, OUTSIDE the signature.** The DST remains `soul-stack/sigil/v1`, the Sigil signature is over the block `(namespace, name, ref, binary_sha256, manifest_sha256)` (see the block format in [plugins.md](../keeper/plugins.md#signed-block-format-normative-s3)); `commit_sha` is **not included** in the signed block — it will be added as a column in `plugin_sigils` as an audit marker ("which commit produced this binary") at S3 (marked "introduced at impl"). **The integrity authority is `sha256` + the Keeper's signature** (invariant (b) is not weakened); `commit_sha`/`ref` carry provenance and readability, not trust.
    - **Git stack — go-git, hardening.** hooks are **not executed** by-design (go-git runs them neither on clone nor on checkout); submodules are **not recursed** (`RecurseSubmodules` is not passed); the `ext::` transport (executing an external command as a "transport") **does not exist at all** in go-git; `file://` is **locked by a scheme allow-list** (`validateGitScheme`: prod allow-list `https://`/`ssh://`/scp-form; `file://` — only under the env flag `SOUL_STACK_ALLOW_FILE_REPOS=1` for dev/test). **No runtime dependency on the `git` binary** on the keeper host. Hardening: shallow (`Depth=1`) + context-timeout (`plugins.fetch_timeout`, default 120s); the working clone (`plugins.work_root`) is **STRICTLY outside `cacheRoot`** — `.git` and the checkout must not land in the cache directory read by Discover/ReadSlot; **size limit by volume** — `plugins.max_clone_size_mb` (default 1024 MiB, the clone working tree: a du-walk of checkout + `.git`, checked after checkout before extracting the artifact) + `plugins.max_artifact_size_mb` (default 256 MiB, the binary `dist/<binary-name>`: `os.Stat` before copy + `io.LimitReader` during copy).
    - **git-egress — HIGH security risk.** An external git call with a config-controlled `source` is an attack surface. `source` is operator-asserted, but the repository is untrusted, and the timeout limits egress only by time; **size limit — IMPLEMENTED** (see above: clone-tree + artifact cap, fail-closed → the slot is not created, there is nothing to allow for the plugin; sentinels `ErrCloneTooLarge`/`ErrArtifactTooLarge`). The remainder of the security pass before prod (`noexec` on the slot, sandboxing git operations) — mandatory, deferred (see Consequences).
    - Per-entry resolution is **fail-closed**: a broken catalog entry (an unresolvable ref / no manifest / no `dist/<binary-name>` / an unreachable remote) turns into a per-entry warning, the Keeper does not crash; sentinels — `ErrRefNotResolved` / `ErrManifestNotFound` / `ErrArtifactNotFound` / `ErrSourceUnavailable`.
  - **(h) Combining single-vs-snapshot and multi-anchor — replace semantics (Variant A).** The transport distributes two kinds of messages; the normative semantics of combining them is as follows:
    - **`SigilSnapshot{repeated PluginSigil}` — the SOLE source of truth** of the active allow set on the Soul side. Applied as **ReplaceAll**: the Soul **replaces its entire** local set of Sigils with this list, does **not** do an upsert. An allow entry absent from the snapshot the Soul **forgets** — this is how revoke/retire works: after an Archon revoke the Keeper sends a new snapshot without the revoked entry, and access closes **near-instant**, without restarting the Soul. An empty `sigils[]` = no plugin is allowed.
    - **A single `FromKeeper.plugin_sigil` (field 5) — the role of a broadcast notification** about a new allow, **NOT a set mutation**. On it the Soul does **NOT upsert** into the authoritative set; the authority is **only the snapshot**. The single message serves as a signal "a new Sigil has appeared" (for example, so the Soul initiates applying a fresh snapshot), but by itself does not change the active set.
    - **`SigilTrustAnchors{repeated pubkey_pem}` — ReplaceAll of the anchor set** of the signature. Runtime delivery of the set **fully replaces** the bootstrap anchors (this is a **replace, not a merge**): after receiving `SigilTrustAnchors` the Soul verifies signatures with **exactly this** set; an anchor absent from the set the Soul forgets (retire of the old key). Multi-anchor allows seamless rotation of the signing key (any anchor from the set verifies the signature).
    - **`BootstrapReply`: priority set > single, LIVE source (R3-S7).** When `sigil_pubkey_pem_set` (field 6) is non-empty, the single `sigil_pubkey_pem` (field 5) is **ignored** (the set is the authority). **Both empty = Sigil disabled** on this host (plugin verify is not performed). The bootstrap handler reads the set **on every reply from a live source** (`TrustAnchorSource`, the same atomic holder that the `sigil:anchors-changed` watcher updates), and **not from a start-time snapshot** — otherwise a new Soul onboarding between a rotation and its first connect would receive a stale set (dangerous on Retire: it would trust the retired key or reject the fresh primary).
    - **Retire invariant (normative, R3-S7).** Removing a key from the set (`Retire`) is safe **only** when two conditions hold: (1) the new set has **propagated across the cluster** — guaranteed by `PublishAnchorsChanged` (Redis channel `sigil:anchors-changed`) → `reloadAnchors` on each node (re-build Signer + re-broadcast `SigilTrustAnchors` + holder update), and **on loss of the pub/sub signal** — a TTL fallback tick (`sigil_anchors_reload_interval`, see (h)) self-heals the lagging node within the interval; propagation is visible to the operator via `keeper_sigil_anchors_last_delivered` **before** Retire; (2) the bootstrap source is **live** — the bootstrap reply serves the current set (the previous item). Both conditions now hold in the implementation. Additionally `Retire` is forbidden for the **primary** directly (first `SetPrimary` to another active key) and for the **last active** (the set must not empty out — symmetric with RBAC self-lockout); the invariants are transactional under `FOR UPDATE` (`keeper/internal/sigil/keys.go`).
    - **Why so.** The authority over integrity is the Keeper's; the Soul must not accumulate a local allow history diverging from the keeper state. ReplaceAll makes revoke and rotation atomic and observable (one snapshot = the full picture), eliminates the desync "the Soul kept a revoked Sigil" without an explicit forget command. The single broadcast as "notification, not mutation" removes the risk that loss/duplication of a single message distorts the set: the set is always restored from the next snapshot.
- **Normative security invariant.** A plugin **does not gain control** until a double check has passed (binary digest == Sigil + a valid Keeper signature). First-load stops being "trust as is" — it becomes "trust only if the Keeper explicitly allowed this hash and signed it". The `manifest` (and thus `side_effects` / `capabilities`) is under the signature, not forgeable.
- **Reconciliation.** [ADR-020(f)/(g)](0020-plugin-infrastructure.md#adr-020-plugin-infrastructure-manifest-format-handshake-lifecycle) (`required_capabilities` / `side_effects` as a strict contract) — **is strengthened**: the declarations become trusted thanks to (c). [ADR-012(f)](0012-keeper-soul-grpc.md#adr-012-keepersoul-grpc-contract-one-eventstream-with-oneof-keeper-side-render-forward-compat-only-add) (bootstrap carries the CA-chain) — is extended with the Sigil public key through the same channel. [ADR-014](0014-operator-identity.md#adr-014-operator-identity-model-archon) (Vault KV `secret/keeper/jwt-signing-key`) — sets the pattern for the Sigil key in Vault. [ADR-016](0016-parity-license.md) (community plugins) — places **no burden on plugin authors** (their keys / signatures are not needed); the allow is done by the operator through the Keeper.
- **Consequences.**
  - **Distribution proto contract (introduced at S2b, only-add in the `EventStream` oneof):** `PluginSigil` (one seal-broadcast, `FromKeeper` field 5), `SigilSnapshot{repeated PluginSigil}` (full active set, ReplaceAll, field 7), `SigilTrustAnchors{repeated pubkey_pem}` (full anchor set, ReplaceAll, field 8) + a **new optional field `binary_sha256`** in the `proto/plugin` manifest (only-add). `BootstrapReply` carries the single `sigil_pubkey_pem` (field 5) and the multi-anchor `sigil_pubkey_pem_set` (field 6, priority set > single; see (h)).
  - **New PG table `plugin_sigils`** (`(namespace, name, ref)` → `sha256` + allow audit fields) — the allow-list registry. The exact schema is an implementation task. **The `commit_sha` column** (git-verified binary provenance, (g)) — **introduced at impl S3**, OUTSIDE the signed block (an audit marker, not trust).
  - **Git resolver for the plugin catalog — `keeper/internal/plugingit`** (A1-S1, implemented). At startup the Keeper resolves `keeper.yml::plugins.{cloud_drivers,ssh_providers}` via go-git into the R-nested cache `<cacheRoot>/<ns>-<name>/<commit_sha>/` + symlink `current` ((g)). The artifact convention is `dist/<binary-name>` (F-fetch). Resolution sentinels: `ErrRefNotResolved` / `ErrManifestNotFound` / `ErrArtifactNotFound` / `ErrSourceUnavailable` / `ErrCloneTooLarge` / `ErrArtifactTooLarge` (per-entry warnings, fail-closed). New config fields — `plugins.work_root` (the root of working clones, STRICTLY outside `cache_root`), `plugins.fetch_timeout` (default 120s), `plugins.max_artifact_size_mb` (default 256 MiB) and `plugins.max_clone_size_mb` (default 1024 MiB); a scheme allow-list on `source` (`validateGitScheme`).
  - **git-egress security pass** (git-egress is a HIGH risk ((g))): **size limit — implemented** (clone-tree + artifact byte-cap, fail-closed, security-audit MEDIUM-1 closed); the remainder (`noexec` on the slot / sandboxing git operations) — mandatory before prod, deferred.
  - **GC of old `commit_sha` slots** (several slots per pair after `ref`/commit rotations) — deferred; a candidate for a Reaper rule, the name — a separate propose-and-wait at impl.
  - **MCP / OpenAPI allow endpoint** — the Archon allows `(namespace, name, ref, sha256)` into the registry; new RBAC permissions (`plugin.allow` / `plugin.revoke` / `plugin.list`) and audit events (`plugin.allowed` / `plugin.revoked` / `plugin.verify_failed`). The CRUD surface starts as a stub catalog (entries in operator-api.md / mcp-tools.md / rbac.md as actually implemented).
  - **Replacement of the TOFU first-load branch** in `shared/pluginhost` with verify-against-Sigil. The TOFU check before every `exec` remains as defense-in-depth for the shared cache; only first-load changes (instead of "trust as is" — "verify Sigil"). Distributing the `shared/pluginhost` change — a separate slice after the ADR review.
  - **The Keeper's signing key in Vault KV** (a new path, following the `secret/keeper/jwt-signing-key` pattern); **the public key in bootstrap** (`BootstrapReply`, next to the CA-chain) — the single `sigil_pubkey_pem` plus the multi-anchor `sigil_pubkey_pem_set` for seamless rotation (R3, replace semantics; runtime delivery of `SigilTrustAnchors` fully replaces the bootstrap anchors, see (h)).
  - **Multi-anchor registry of signing keys (R3, implemented).** New PG table **`sigil_signing_keys`** (migration 037): `key_id` (SHA-256 SPKI, hex) → `pubkey_pem` (SPKI) + `vault_ref` (the private key in Vault) + `is_primary` / `status` (active|retired) + an audit FK to `operators`. **The private key is NEVER in Postgres** — only the public part + a reference to Vault. Invariants: exactly one primary among active (partial unique index `sigil_signing_keys_one_primary`), ≥1 active on Retire. Multi-anchor Signer (`keeper/internal/sigil`): the primary signs new Sigils, the anchor set = the public keys of all active. An empty registry → fallback to the single `sigil.signing_key_ref` from cfg (the old path does not break).
  - **Cluster-wide rotation — Redis channel `sigil:anchors-changed` (R3-S6) + TTL fallback.** After a key-registry mutation (introduce / set-primary / retire) the mutating node does `PublishAnchorsChanged` to `sigil:anchors-changed`; **every** node (including the mutating one) on subscription does `reloadAnchors`: re-build Signer (new primary + anchors) → `SetSigner` into the sigil-service → update of the keeper host's verify set → update of the connect-time broadcast holder → re-broadcast `SigilTrustAnchors` to connected Souls. A separate channel from `sigil:invalidate` (that one is about the `plugin_sigils` allows, this one — about the keys signing them). Pub/sub is best-effort, but **on top of it — a TTL fallback tick** (`sigil_anchors_reload_interval`, default 30s): the same `reloadAnchors` periodically by a ticker, so a missed signal self-heals within the interval (the RBAC TTL-poll pattern [ADR-028(d)](0028-rbac-storage.md#adr-028-rbac-storage--postgres) / `Summons` poll-fallback [ADR-027](0027-apply-work-queue.md#adr-027-apply-execution-model--work-queue--claim-acolyte-pool-ward-claim)). The tick is raised **independently of Redis** (in single-instance/dev without Redis — the only path for runtime rotation to arrive without a restart). `reloadAnchors` is idempotent/fail-safe (on error the re-build leaves state untouched), so a double reload (tick + pub/sub) is safe.
  - **Operator-facing rotation API (R3-S7).** REST `POST /v1/sigil/keys` (introduce: the Keeper generates an ed25519 pair, writes the private key to Vault KV `secret/keeper/sigil-keys/<key_id>`, inserts the public part into the registry; the response carries `key_id` + `pubkey_pem`, **the private key is NEVER returned**), `GET /v1/sigil/keys` (list active), `POST /v1/sigil/keys/{key_id}/primary` (set-primary), `DELETE /v1/sigil/keys/{key_id}` (retire). MCP parity: `keeper.sigil.key.introduce|list|set-primary|retire`. Permissions `sigil.key-introduce|retire|list|set-primary` (resource `sigil`, hyphenated action — the grammar `<resource>.<action>`, the 3-segment `sigil.key.*` = an MCP tool, not a permission). Audit `sigil.key-introduced|retired|primary-set` (payload `key_id` + caller AID, **without the private key**). Metrics `keeper_sigil_signing_keys_active` (gauge of the number of active keys), `keeper_sigil_anchors_rebroadcast_total` (counter of anchor-set re-broadcast passes) and `keeper_sigil_anchors_last_delivered` (gauge of the number of Souls the last set actually reached — the operator sees propagation **BEFORE** Retire). A race on setting primary (the one_primary index, 23505) → sentinel `ErrConcurrentPrimary` → 409 (retry).
  - **TTL fallback for `sigil:anchors-changed` — CLOSED (was a KNOWN GAP).** Propagation of the new anchor set across the cluster was best-effort Redis pub/sub (`PublishAnchorsChanged` → `reloadAnchors`) **without a periodic poll tick**: signal loss → the lagging node held the old anchor set until restart (load-at-start), which is dangerous on **Retire** (the node would locally *fail-open* trust the retired key for a Soul onboarding through it). **Closed** by a TTL fallback tick in `reloadAnchors` (`runAnchorsReloadTicker`, interval `sigil_anchors_reload_interval` default 30s, following the `rbacHolder.Run` TTL-poll / `Summons` poll-fallback pattern): a missed signal self-heals within the interval, the tick is independent of Redis. Observability of re-broadcast delivery — metrics `keeper_sigil_anchors_rebroadcast_total` (passes) + `keeper_sigil_anchors_last_delivered` (how many Souls the set actually reached) + a log of the number of active anchors and delivered on each reload — the operator sees propagation of the set **BEFORE** Retire of the old key.
  - **KNOWN GAP "orphan Vault secret on a PG-fail Introduce".** `POST /v1/sigil/keys` (introduce) writes the private key to Vault KV `secret/keeper/sigil-keys/<key_id>` **before** inserting the public part into `sigil_signing_keys`. If the PG insert fails after a successful Vault write — the private key remains an **orphan** in Vault (no corresponding registry row). Reverse cleanup (deleting from Vault on a PG-fail) is **deliberately not done** — the orphan is harmless (the key is nowhere declared primary/active, no signatures are produced with it), and a compensating transaction across two stores (Vault + PG) is fragile by itself. GC of orphan secrets — a **candidate for a Reaper rule**, deferred together with GC of `commit_sha` slots ((g)). The rule name — a separate propose-and-wait at impl.
  - **Known gap "TOFU first-load" — CLOSED** (S6 LIVE): the Integrity-model section in [plugins.md](../keeper/plugins.md) is rewritten for Sigil + multi-anchor; host-side first-load is now verify-against-Sigil, not TOFU.
  - **Names in [naming-rules.md](../naming-rules.md)** — `Sigil` (+ definition), proto messages `PluginSigil` / `SigilSnapshot` / `SigilTrustAnchors`, the `sigil_pubkey_pem_set` field (priority set > single), PG tables `plugin_sigils` and `sigil_signing_keys`, RBAC perms (`plugin.*` + `sigil.key-*`), audit events (`plugin.*` + `sigil.key-*`), Redis channel `sigil:anchors-changed`.
- **Deferred (post-MVP, no breaking changes).** **Variant B (author-signed collections / publishers)** — a trust boundary at the collection level by `(namespace, ref)` (one publisher → one key) — is deferred until module-collections ([module-collections.md, open Q No. 2](../module-collections.md)). Sigil (Variant A) — allowing specific hashes — is the current MVP model; collections/publishers extend it additively, without breaking the `(namespace, name, ref) → sha256` registry. **git-egress security pass** — the clone/artifact size limit is **implemented** ((g), MEDIUM-1 closed); the remainder (`noexec` on the slot / sandboxing git operations) is mandatory before prod, git-egress is a HIGH risk. **GC of old `commit_sha` slots** — a candidate for a Reaper rule, the name — a separate propose-and-wait. **The `commit_sha` column in `plugin_sigils`** — introduced at S3 (a provenance audit marker, OUTSIDE the signature).
- **Trade-off.** The allow is concentrated on the Keeper (single authority) — but the Keeper is **already** the root of mTLS / Destiny render / Vault access, almost no new trust surface is added. **Zero burden on plugin authors** (their keys / signatures are not needed) — the allow is entirely on the operator through the Keeper; the price — the operator must explicitly allow each `(namespace, name, ref, sha256)` (a human step per new hash), but this is precisely deliberate supply-chain control. The alternative — author-signed (Variant B) — removes the human step, but requires key management by community-plugin authors; deferred.
