# ADR-066. Onboarding via Teleport on platforms without cloud-init userdata — environment constraints (WB) and working profiles

> **Status: active.** User's decisions 2026-07-01/02; recorded retrospectively 2026-07-03 (paying off the debt "an architectural decision goes into the record in the same move"). **The prod-profile is live-proven by run-23 E2E create** (2026-07-02, apply_id `01KWJKKQBT`: VM from scratch → external IP → Teleport enroll → full-install + onboarding automatically → `core.module.installed` → redis deploy 77 tasks SUCCESS → `ready` + soul connected). The demo-profile is live-proven 2026-07-01 (soul CONNECTED through a bastion tunnel to a local keeper).
>
> This is a **deployment-profile decision on top of existing machinery**: it builds on [ADR-063](0063-bootstrap-token-delivery.md) (Teleport by-name transport + full-install + init phase), [ADR-061](0061-onboarding-await-and-midrun-reresolve.md) (await barrier) and [ADR-017](0017-keeper-side-core.md) (cloud-init/self-onboard) — **it does not change their contracts** (not Amends). It fixes: which constraints the external environment imposes, which of the already existing onboarding paths is the standard one for it, and which environment requirements must be met.

**Context.** A create-run's provision→onboarding→role ([ADR-061](0061-onboarding-await-and-midrun-reresolve.md)) has two standard paths for delivering the bootstrap-setup to a freshly created VM: **cloud-init userdata** (B-flat / self-onboard "Variant T", [ADR-017(h)](0017-keeper-side-core.md)) and **push delivery** `core.bootstrap.delivered` ([ADR-063](0063-bootstrap-token-delivery.md), transports `direct`/`teleport`). The WB-cloud prod environment imposes hard constraints — each verified live 2026-07-01/02:

1. **`ci_user_data` is disabled on the WB side at the namespace level** — a feature flag, cut off BEFORE the key's permissions are checked. The matrix of direct gRPC checks (verify harness, 2026-07-01): all authorizing combinations — updated dev key (uuid) × dev/stage endpoint, prod key (len8) × prod endpoint, prod service account × prod endpoint — reached `CreateVmV3` and got `ci_user_data is not allowed for this user`. This is a property of the namespace/account, NOT of the key/endpoint/account type. → **Both userdata onboarding paths (B-flat and self-onboard T) are unavailable on this platform**; enabling it is a request on the WB side, outside Soul Stack's control.
2. **Direct SSH keeper→VM is closed** (kex breaks on connection) → `transport: direct` ([ADR-063](0063-bootstrap-token-delivery.md)) is unavailable.
3. **Access to the VM is only via Teleport**, moreover the Teleport Proxy stands behind a public L7-TLS balancer (which terminates TLS, the cert `*.tp.rwb.ru` is signed by WB's internal root, WB Root CA, and the SAN `teleport.cluster.local` is absent), while the VMs live behind NAT.

## Decision — prod-profile: full-install via `core.bootstrap.delivered` `transport: teleport`

The standard onboarding path on a "no userdata, VM behind Teleport" platform is the **full-install mode** of `core.bootstrap.delivered` ([ADR-063](0063-bootstrap-token-delivery.md) amendments "Teleport by-name transport" / "full-install" / "init phase"): the module installs the whole setup (`soulinstall.Blueprint`) over Teleport SSH, delivers the token, runs `soul init` and activates the unit. The machinery is in ADR-063; here the **mandatory environment requirements** revealed live are fixed:

**(a) Keeper's Teleport identity: a bot-identity without `pin_source_ip` — a prod requirement.** An identity issued by an interactive `tsh login --out` is unsuitable for background keeper push for three reasons: (1) it carries **PinnedIP** — the OID `1.3.9999.1.9` in the TLS cert = the source IP of the issuing machine (proven by decoding the identity 2026-07-01): a keeper dialing from a different src-IP gets access-denied, even though roles/expiry are identical to the interactive session; (2) the short TTL of an interactive session — manual reissue; (3) MFA binding (hardware key) — a background process cannot do the touch. Prod = **Teleport Machine ID / bot** (`tctl bots add keeper-push --roles=…`, an identity without `pin_source_ip`, rotation via `tbot`); alternatives — an identity issued with a pin on keeper's src-IP, or removing `pin_source_ip` from the role. All of it is on the Teleport-admin side, it does not touch Soul Stack code.

The history of the 403 diagnosis (so we don't get confused a third time): the 403 on `DialHost` was first attributed to PinnedIP; later the live finale revealed the actual wall was **our own code** — the order of ALPN `NextProtos`: the vendor `creds.TLSConfig()` set `["h2"]`, and appending the vendor proxy-client gave `["h2","teleport-proxy-ssh-grpc"]`, and Teleport's TLS-routing router picks the handler by the first ALPN → h2-first routed into the web stack → the gRPC ProxySSH received a plain-HTTP `403 text/plain`. The fix — `NextProtos = nil` after `creds.TLSConfig()` ([dial_teleport.go](../../keeper/internal/push/dial_teleport.go)); run-23 passed with a pinned identity from the issuing machine (the src-IP matched the pin). **PinnedIP is nonetheless an objective property of the identity** (the OID in the cert) and remains a prod constraint for a keeper whose src-IP ≠ pin, so the bot-identity requirement stands.

**(b) Dial config behind an L7-TLS balancer.** `push.teleport.alpn_upgrade: true` is mandatory ([ADR-063](0063-bootstrap-token-delivery.md) amendment 2026-06-30): the LB does not proxy a raw gRPC/SSH stream — an ALPN conn-upgrade tunnels it over WebSocket. In this case the connection = two TLS layers: the **outer** TLS to the LB is verified by the keeper process's system trust (the LB's private root — WB Root CA — must be added to the process's system trust; on the stand `SSL_CERT_FILE=<trust-bundle>` in the keeper process environment), the **inner** gRPC-mTLS to the proxy — by the combined trust of identity-CA ∪ system. `use_system_trust` is ignored in alpn mode (it is relevant only for direct access to the proxy with a public cert without an ALPN tunnel).

**(c) Enroll requirement: a working external IP on the VM.** A VM behind NAT dials the Teleport agent outward itself (`tp.rwb.ru`) — for enroll it needs an active external IP: (1) the VM profile must carry `set_external_ip: true` (WB's default internal-external without egress is unsuitable for enroll); (2) the VM readiness driver-probe (`finalizeCreate` of the `soul-cloud-wb` driver) counts the VM as provisioned only after the ordered external IP is **activated** — otherwise the keeper starts dialing before enroll and `join_wait_timeout` is spent in vain. The enroll itself is not a scale limit (WB has ~50k nodes), the time is variable (WB reconciler scan); the wait is covered by `join_wait_timeout` ([ADR-063](0063-bootstrap-token-delivery.md)).

## Demo/dev profile — bastion scheme (without a prod identity)

A working workaround for demo/dev when a bot-identity has not yet been issued (proven 2026-07-01, soul → CONNECTED):

- **keeper locally** on the operator's machine (dev stand PG/Redis/Vault);
- **an enrolled VM as a bastion**: a reverse tunnel `tsh ssh -R 0.0.0.0:<bootstrap>/<event_stream>` from the operator's machine (the Teleport agent binds `0.0.0.0` itself — sshd `GatewayPorts` is not needed); souls dial the bastion's IP → tunnel → local keeper;
- **soul onboarding — via an external Teleport shell-exec** from the operator's machine (`tsh scp` of the soul binary/CA/soul.yml + `tsh ssh 'SOUL_BOOTSTRAP_TOKEN=… soul init …; soul run'`): src-IP = the identity's issuing machine → PinnedIP does not interfere by construction.

Mandatory details: the keeper-cert must carry the **bastion's IP in the SAN** (`ip_sans`), otherwise soul TLS-verify fails; `keeper.endpoints` in soul.yml is the **bastion's IP, not an FQDN** (the VM does not resolve internal names). **Applicability boundaries: a demo/dev stand and manual onboarding debugging.** Not for prod: the tunnel lives in the operator's interactive session (SPOF, MFA re-login), throughput and resilience are not standardized. The recipe scripts are a local archive of the PM stand (`.pm/scripts`, outside the git canon; deliberately not moved into the canon — this is stand plumbing, not a product surface).

## Rejected alternatives

- **`core.teleport.shell` (plan B — explicit tsh-exec of bootstrap commands as a keeper-side module).** Rejected (architect verdict 2026-07-02, accepted by the user: keep only as a fallback): executing an external unsigned `tsh` binary in the keeper flow goes around the Sigil integrity model ([ADR-026](0026-sigil.md)); it pulls in an epic (~15 files + edits to three ADRs); it **does not solve PinnedIP** (same identity, same src-IP); the need to "explicitly specify teleport" is already covered by `transport: teleport`. Run-23 confirmed: the fallback was not needed.
- **Waiting for WB to enable `ci_user_data` as the only path.** Rejected as a blocker: enabling it is an external request outside the project's control. The profiles coexist: once userdata is enabled, self-onboard "Variant T" ([ADR-017(h)](0017-keeper-side-core.md)) will work on the same platform with no code changes — the teleport profile stays for platforms where userdata does not exist at all.
- **`transport: direct` (generic SSH + Vault host-CA, C1).** In this environment it is not a choice but an impossibility (SSH is closed by the platform); for environments with direct SSH the direct path remains standard ([ADR-063](0063-bootstrap-token-delivery.md)).

## Consequences

- No new names/config fields — the whole surface is already fixed by [ADR-063](0063-bootstrap-token-delivery.md) (`push.transport`, `push.teleport.{proxy_addr,identity_file,cluster,use_system_trust,alpn_upgrade}`, `join_wait_timeout`) and [ADR-017](0017-keeper-side-core.md) (profile/`set_external_ip` — Profile-registry parameters).
- The prod-hardening tail (Teleport-admin/WB side, noted in the roadmap): issuing a bot-identity without `pin_source_ip` + `tbot` rotation; optionally — a request to enable `ci_user_data` for the namespace.
- [ADR-020(m)](0020-plugin-infrastructure.md) (the dispatcher ignores `proxy_jump` on `soul-ssh-teleport`) — **does not block this profile**: bootstrap delivery goes through the keeper-side Teleport-Dialer by identity-file, and the `soul-ssh-teleport` plugin does not participate in the flow (see the amendment pointer in ADR-020).
