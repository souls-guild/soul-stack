# ADR-066. Онбординг через Teleport на платформах без cloud-init userdata — ограничения среды (WB) и рабочие профили

> **Статус: active.** Решения пользователя 2026-07-01/02; фиксация ретроспективно 2026-07-03 (погашение долга «архитектурное решение — в том же ходе»). **Прод-профиль live-доказан прогоном-23 E2E create** (2026-07-02, apply_id `01KWJKKQBT`: VM из нуля → external IP → Teleport enroll → full-install+онбординг автоматом → `core.module.installed` → деплой redis 77 задач SUCCESS → `ready` + soul connected). Demo-профиль live-доказан 2026-07-01 (soul CONNECTED через bastion-туннель в локальный keeper).
>
> Это **deployment-profile-решение поверх существующих механик**: строится на [ADR-063](0063-bootstrap-token-delivery.md) (Teleport by-name transport + full-install + init-фаза), [ADR-061](0061-onboarding-await-and-midrun-reresolve.md) (await-барьер) и [ADR-017](0017-keeper-side-core.md) (cloud-init/self-onboard) — **их контракты не меняет** (не Amends). Фиксирует: какие ограничения накладывает внешняя среда, какой из уже существующих путей онбординга является штатным для неё и какие требования к среде обязаны быть выполнены.

**Контекст.** У create-прогона provision→онбординг→роль ([ADR-061](0061-onboarding-await-and-midrun-reresolve.md)) два штатных пути доставки bootstrap-setup на свежесозданную VM: **cloud-init userdata** (B-flat / self-onboard «Вариант T», [ADR-017(h)](0017-keeper-side-core.md)) и **push-доставка** `core.bootstrap.delivered` ([ADR-063](0063-bootstrap-token-delivery.md), транспорты `direct`/`teleport`). Прод-среда WB-cloud накладывает жёсткие ограничения — каждое проверено live 2026-07-01/02:

1. **`ci_user_data` выключен на стороне WB на уровне namespace** — feature-flag, режется ДО проверки прав ключа. Матрица прямых gRPC-проверок (verify-харнесс, 2026-07-01): все авторизующиеся комбинации — обновлённый dev-ключ (uuid) × dev/stage-endpoint, prod-ключ (len8) × prod-endpoint, prod service-account × prod-endpoint — дошли до `CreateVmV3` и получили `ci_user_data is not allowed for this user`. Это свойство namespace/аккаунта, НЕ ключа/endpoint-а/типа учётки. → **Оба userdata-пути онбординга (B-flat и self-onboard T) на этой платформе недоступны**; включение — заявка на стороне WB, вне контроля Soul Stack.
2. **Прямой SSH keeper→VM закрыт** (обрыв kex на подключении) → `transport: direct` ([ADR-063](0063-bootstrap-token-delivery.md)) недоступен.
3. **Доступ к VM — только через Teleport**, причём Teleport Proxy стоит за публичным L7-TLS-балансировщиком (терминирует TLS, cert `*.tp.rwb.ru` подписан внутренним корнем WB Root CA, SAN `teleport.cluster.local` отсутствует), а VM живут за NAT.

## Решение — прод-профиль: full-install через `core.bootstrap.delivered` `transport: teleport`

Штатный путь онбординга на платформе «без userdata, VM за Teleport» — **full-install-режим** `core.bootstrap.delivered` ([ADR-063](0063-bootstrap-token-delivery.md) amendments «Teleport by-name transport» / «full-install» / «init-фаза»): модуль по Teleport SSH ставит весь setup (`soulinstall.Blueprint`), доставляет токен, делает `soul init` и активирует unit. Механика — в ADR-063; здесь фиксируются **обязательные требования к среде**, вскрытые live:

**(a) Teleport identity keeper-а: bot-identity без `pin_source_ip` — прод-требование.** Identity, выписанная интерактивным `tsh login --out`, для фонового keeper-push непригодна по трём причинам: (1) несёт **PinnedIP** — OID `1.3.9999.1.9` в TLS-cert = source-IP машины выписки (доказано декодом identity 2026-07-01): keeper, дилящий с другого src-IP, получает access-denied, хотя роли/expiry идентичны интерактивной сессии; (2) короткий TTL интерактивной сессии — ручной перевыпуск; (3) MFA-привязка (hardware key) — фоновый процесс touch пройти не может. Прод = **Teleport Machine ID / bot** (`tctl bots add keeper-push --roles=…`, identity без `pin_source_ip`, ротация через `tbot`); альтернативы — identity, выписанная с pin на src-IP keeper-а, либо снятие `pin_source_ip` из роли. Всё — сторона Teleport-admin, кода Soul Stack не касается.

История диагноза 403 (чтобы не путаться в третий раз): 403 на `DialHost` сперва атрибуцирован PinnedIP; позже live-финал вскрыл фактическую стену **нашего кода** — порядок ALPN `NextProtos`: vendor `creds.TLSConfig()` ставил `["h2"]`, append vendor-proxy-client давал `["h2","teleport-proxy-ssh-grpc"]`, TLS-routing-роутер Teleport выбирает обработчик по первому ALPN → h2-first уводил в web-стек → gRPC ProxySSH получал plain-HTTP `403 text/plain`. Фикс — `NextProtos = nil` после `creds.TLSConfig()` ([dial_teleport.go](../../keeper/internal/push/dial_teleport.go)); прогон-23 прошёл с pinned-identity с машины выписки (src-IP совпал с pin). **PinnedIP при этом — объективное свойство identity** (OID в серте) и остаётся прод-ограничением для keeper-а, чей src-IP ≠ pin, поэтому требование bot-identity в силе.

**(b) Конфиг dial за L7-TLS-балансировщиком.** `push.teleport.alpn_upgrade: true` обязателен ([ADR-063](0063-bootstrap-token-delivery.md) amendment 2026-06-30): LB не проксирует raw gRPC/SSH-stream — ALPN-conn-upgrade туннелирует его WebSocket-ом. При этом соединение = два TLS-слоя: **внешний** TLS к LB верифицируется системным trust-ом процесса keeper (приватный корень LB — WB Root CA — обязан быть добавлен в системный trust; на стенде `SSL_CERT_FILE=<trust-bundle>` окружением процесса keeper), **внутренний** gRPC-mTLS к proxy — объединённым trust-ом identity-CA ∪ системный. `use_system_trust` в alpn-режиме игнорируется (актуален только для прямого доступа к proxy с публичным cert без ALPN-туннеля).

**(c) Enroll-требование: рабочий external IP на VM.** VM за NAT сама дилит Teleport-агентом наружу (`tp.rwb.ru`) — для enroll ей нужен активный external IP: (1) VM-profile обязан нести `set_external_ip: true` (WB-дефолтный internal-external без egress для enroll непригоден); (2) driver-probe готовности VM (`finalizeCreate` драйвера `soul-cloud-wb`) считает VM provisioned только после **активации** заказанного external IP — иначе keeper начинает dial до enroll и `join_wait_timeout` тратится впустую. Сам enroll — не лимит по масштабу (у WB узлы ~50k), время переменное (reconciler-скан WB); ожидание покрывает `join_wait_timeout` ([ADR-063](0063-bootstrap-token-delivery.md)).

## Demo/дев-профиль — bastion-схема (без прод-identity)

Рабочий обход для демо/дев, когда bot-identity ещё не выдана (доказан 2026-07-01, soul → CONNECTED):

- **keeper локально** на машине оператора (dev-стенд PG/Redis/Vault);
- **enrolled-VM как bastion**: реверс-туннель `tsh ssh -R 0.0.0.0:<bootstrap>/<event_stream>` с машины оператора (Teleport-агент биндит `0.0.0.0` сам — `GatewayPorts` sshd не нужен); souls дилят IP bastion-а → туннель → локальный keeper;
- **онбординг soul — внешним Teleport shell-exec** с машины оператора (`tsh scp` soul-бинарь/CA/soul.yml + `tsh ssh 'SOUL_BOOTSTRAP_TOKEN=… soul init …; soul run'`): src-IP = машина выписки identity → PinnedIP не мешает by construction.

Обязательные детали: keeper-cert обязан нести **IP bastion-а в SAN** (`ip_sans`), иначе soul-TLS-verify падает; `keeper.endpoints` в soul.yml — **IP bastion-а, не FQDN** (VM внутренние имена не резолвит). **Границы применимости: демо/дев-стенд и ручная отладка онбординга.** Не для прода: туннель живёт в интерактивной сессии оператора (SPOF, MFA-релогин), пропускная способность и устойчивость не нормированы. Рецепты-скрипты — локальный архив стенда PM (`.pm/scripts`, вне git-канона; в канон не переносятся сознательно — это стендовая обвязка, не продуктовая поверхность).

## Отвергнутые альтернативы

- **`core.teleport.shell` (план Б — явный tsh-exec bootstrap-команд как keeper-side модуль).** Отклонён (вердикт architect 2026-07-02, пользователь принял: держать только как fallback): исполнение внешнего неподписанного `tsh`-бинаря в keeper-флоу идёт мимо Sigil-модели целостности ([ADR-026](0026-sigil.md)); тянет эпик (~15 файлов + правки трёх ADR); **не решает PinnedIP** (та же identity, тот же src-IP); потребность «явно указать teleport» уже закрыта `transport: teleport`. Прогон-23 подтвердил: fallback не понадобился.
- **Ждать включения `ci_user_data` у WB как единственный путь.** Отвергнуто как блокер: включение — внешняя заявка вне контроля проекта. Профили сосуществуют: при включении userdata self-onboard «Вариант T» ([ADR-017(h)](0017-keeper-side-core.md)) заработает на той же платформе без изменений кода — teleport-профиль останется для платформ, где userdata нет в принципе.
- **`transport: direct` (generic SSH + Vault host-CA, C1).** На этой среде не выбор, а невозможность (SSH закрыт платформой); для сред с прямым SSH direct-путь остаётся штатным ([ADR-063](0063-bootstrap-token-delivery.md)).

## Consequences

- Новых имён/конфиг-полей нет — вся поверхность уже зафиксирована [ADR-063](0063-bootstrap-token-delivery.md) (`push.transport`, `push.teleport.{proxy_addr,identity_file,cluster,use_system_trust,alpn_upgrade}`, `join_wait_timeout`) и [ADR-017](0017-keeper-side-core.md) (профиль/`set_external_ip` — параметры Profile-реестра).
- Прод-hardening хвост (сторона Teleport-admin/WB, отмечено в roadmap): выдача bot-identity без `pin_source_ip` + `tbot`-ротация; опционально — заявка на включение `ci_user_data` для namespace.
- [ADR-020(m)](0020-plugin-infrastructure.md) (dispatcher игнорирует `proxy_jump` у `soul-ssh-teleport`) — **не блокирует этот профиль**: bootstrap-доставка идёт keeper-side Teleport-Dialer-ом по identity-file, плагин `soul-ssh-teleport` в флоу не участвует (см. amendment-указатель в ADR-020).
