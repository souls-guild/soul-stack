# ADR-056. Staged-render — scenario run as N ordered Passages (probe->where actually works)

> **Status: active.** User decision (Variant A-full, N-stage; name of the new run phase — **Passage**) + architect design. The canon is fixed docs-first BEFORE code; implementation — separate slices S1-S5 (see §Slice map). This ADR **amends [ADR-009](0009-scenario-dsl.md), [ADR-012](0012-keeper-soul-grpc.md), [ADR-027](0027-apply-work-queue.md)** (see §Relation to ADR) and closes [open Q #24](../architecture.md#open-questions) (per-task granularity of `serial:` / per-task dispatch).

Amendments to S1 (register correlation, R1 onchanges/onfail remap) implemented; **R3 (cross-passage onchanges/onfail keeper-side gating) implemented — R2-reject removed** (see the "Amendment to S1 (onchanges/onfail-gating)" block below). The **global-vs-local-task_idx class closed 6/6**. Open limit — **onfail cross-passage FIRES unreachable under the current fail-stop barrier** (mechanism implemented+unit-proven, activates under continue-on-partial-failure semantics; future follow-up).

> **Amendment 2026-06-29 (keeper-target per-Passage dispatch)** — see the block of the same name below: the implementation assumption of the S1 pilot "keeper-side tasks execute in a single pass BEFORE host fan-out, one `apply_runs` row sid=keeper, passage 0" is **removed**; keeper tasks are stratified on a par with host tasks and execute per-Passage. Implements keeper->keeper register-chaining through Passage (`core.bootstrap.delivered` after `core.cloud.created`), which [ADR-061](0061-onboarding-await-and-midrun-reresolve.md) / [ADR-063](0063-bootstrap-token-delivery.md) already require.

**Context.** The canon [`docs/scenario/orchestration.md §4`/`§5`](../scenario/orchestration.md#4-volatile-predicate---where) presents probe->where as a **working** mechanism: a probe step (`module: core.exec.run` + `register: X` + `changed_when: false`) collects a per-host fact, and the **subsequent** task targets on it via `where: register.X.stdout == 'slave'` or carries it into `apply: input:` / `params:`. This mechanism is the basis of the volatile role from [ADR-008](0008-coven-stable-tags.md) (role is NOT Coven; actual-role is taken only from a live probe).

In practice **it is not implemented**: the Keeper renders the entire scenario in **one up-front pass** BEFORE dispatch. At render phase `in.Register` (the probe results map) is **empty** — the probe has not yet executed; the `Where` field in `RenderedTask` (`proto/keeper/v1`) **does not exist at all** — `where:` is resolved keeper-side at render time, i.e. **before** the probe returns `register`. Consequence:

- `where:` on `register.*` of a preceding probe always sees an empty register -> selects 0 hosts (or fails on a missing key);
- `apply: input:` / `params:` / `vars:` of the next task, reading `register.X`, also have no value at render phase;
- the canon promises a working idiom, the engine silently does not execute it — **doc-drift between orchestration.md and Keeper**.

> **What already works (boundary of the drift).** Flow-control CEL `when:`/`changed_when:`/`failed_when:` on `register.*` executes **Soul-side** (sandboxed, [ADR-012(d)](0012-keeper-soul-grpc.md)) — the Soul builds `register` itself from the accumulated task results within **one** `ApplyRequest`. This covers register dependencies **within a single-host slice**. The drift is specifically in **targeting** (`where:` — "on which hosts", resolved by the Keeper BEFORE probe) and in **cross-task render** (`apply: input:`/`params:` of the next task from the register of the previous), which require per-host register **on the Keeper after the probe barrier**.

This ADR introduces **staged-render** to implement what the canon already promises.

**Decision.** The scenario run executes as **N ordered Passages**.

- **(a) Passage — a run phase.** One Passage is an ordered cycle **render -> dispatch -> barrier -> register collection**. A scenario of K tasks executes as N >= 1 Passages; each Passage carries a subset of the run's tasks. Tasks without register dependencies fit into a single Passage (N=1 — current behavior, fast-path).

- **(b) Stratification by register dependency (topological N-stage).** Run tasks are **stratified** by the register dependency graph: a task reading `register.X` (in `where:` / `apply: input:` / `params:` / `vars:`) falls into a Passage **strictly AFTER** the probe step that emits `register: X`. A probe and its consumer cannot end up in the same Passage. This is a topological sort of tasks over the edges "probe X -> reader of register.X"; the number of Passages equals the number of levels (length of the dependency critical path + 1). Register dependency sources are found via a static parse of `where:` / interpolations `${ ... }` in `apply: input:`/`params:`/`vars:` for the presence of `register.<name>.*`.

- **(c) Cycle of one Passage.**
  1. **render** — the Keeper renders this Passage's tasks into `RenderedTask`, substituting the `register` of **all preceding** Passages (a per-host map accumulated by barriers). On the first Passage register is empty (as today); on the second and later — filled with probe facts from previous Passages.
  2. **dispatch** — `ApplyRequest` to the hosts that passed resolution of `on:`+`where:` **for this** Passage (now `where:` sees the real register).
  3. **barrier** — unconditional wait for the terminal of all per-host jobs of the Passage (cross-host barrier, parity with [ADR-009 §7](0009-scenario-dsl.md), but **within** a Passage).
  4. **register collection** — the per-host `register` map of the Passage is accumulated via the already existing channel `TaskEvent.register_data` -> `apply_task_register` (see §Reuse) and becomes available to the render of the next Passage.

- **(d) state-commit — happens ONCE after the last Passage (the §7 invariant is not split).** `incarnation.state` (`state_changes`) is committed **exactly once** after the successful barrier of the **last** Passage of the run — NOT per-Passage. This is a direct preservation of the barrier/state-commit invariant [ADR-009 §7](0009-scenario-dsl.md) / orchestration.md §7: a partial commit after an intermediate Passage is **forbidden** (otherwise `incarnation.state` would diverge from fact if the next Passage fails). Failure of any Passage (finally-failed after `retry:`) -> the run stops, subsequent Passages do not start, state is NOT committed -> `error_locked`. Passage is an axis of **tasks** (when to render/send what), not a commit axis.

**Reuse (half the mechanism already exists).** Staged-render reuses the existing post-barrier register channel — it does not introduce a new one:

- **`loadRegisterByHost` / `apply_task_register` / `SelectTaskRegistersByApplyID`** (post-barrier collection of `register` under `state_changes.sets`, [ADR-009 §7.1 "register.* as a source of sets"](0009-scenario-dsl.md)) — reused as a **Passage-loop**: the same per-host register collection after the barrier, but now it is read by the render of the next Passage, not only by the final `sets` render.
- **`evalWhere` / `renderParams`** already accept `in.Register` as a parameter — what is missing is not their signature, but the fact that `in.Register` is **empty** at the time they are called (up-front render). Staged-render fills `in.Register` between Passages.
- **What's missing (zone S1-S3):** the Passage cycle in run.go (run-goroutine / Acolyte) + a repeated `Render` with a filled `Register` on the second+ Passage + a `passage` field in proto/PG to correlate per-Passage jobs and terminals.

**Contract.**

- **proto — only-add ([ADR-012(c)](0012-keeper-soul-grpc.md) forward-compat, never-reuse field numbers):** `int32 passage` in `ApplyRequest` / `TaskEvent` / `RunResult` — the Passage index (0-based) to which the job/event/report belongs. This is an **only-add** to the existing messages `proto/keeper/v1/apply.proto`; field numbers are not reused, old fields are not removed. Exact field numbers are fixed in S1.
- **PG schema:** now **N** jobs land per run host (one per Passage), not one. The current composite PK `apply_runs (apply_id, sid)` ([ADR-027](0027-apply-work-queue.md), orchestration.md §2.2.1) is no longer unique per-host. Two options — **the choice is fixed in the S1 design**:
  - **(I)** extend the PK to `apply_runs (apply_id, sid, passage)` — N rows per host in the same table;
  - **(II)** a new table `apply_run_passages` (per-Passage jobs), `apply_runs` remains the per-host aggregate.
  The choice is made in S1 by criteria of compatibility with Ward-claim / reclaim / barrier read (`SelectStatusesByApplyID`) / Voyage back-link (`voyage_targets.apply_id`) / recovery — ~10 packages are affected, the migration is additive.

  **S1 — Variant I implemented** (migration 078: PK `apply_runs (apply_id, sid, passage)`, FK `apply_task_register -> apply_runs (apply_id, sid, passage)`).

  **Amendment to S1 (register correlation, migration 079).** The record in migration 078 that "`task_idx` is passage-wide" is **actually wrong**: `TaskEvent.task_idx` is the **LOCAL** position of the task in `ApplyRequest.tasks[]` of its own Passage. An `ApplyRequest` of a single Passage carries only a subset of that Passage's tasks, filtered further **per-host** by `where:` — hence `task_idx` is NOT unique either across Passages (probe Passage0/local 0 and action Passage1/local 0 share `task_idx=0`) or across hosts of the same Passage (different `where:` -> different local index for the same task). The former PK `apply_task_register (apply_id, sid, task_idx)` overwrote the probe register via `ON CONFLICT`, and `buildRegisterByHost` mapped `nameByIdx[Index]` (global) against `rows.task_idx` (local) — a name mismatch at `passage>0` (a latent S1 bug).

  **Register-correlation mechanism — Variant B (global `plan_index`)**, architect design. Proto only-add `RenderedTask.plan_index` (field 16) + `TaskEvent.plan_index` (field 8) — a **global, plan-wide task index** (across all Passages), = `RenderedTask.Index` on the Keeper. The Soul echoes back the received `plan_index` (alongside the local `task_idx`, whose semantics are NOT changed — it is used for Soul-side onchanges/onfail gating). `apply_task_register` is keyed by `plan_index` (migration 079: PK `(apply_id, sid, plan_index)`); `buildRegisterByHost` resolves `nameByIdx[Index]` against the global `plan_index` — both defects disappear (the global index is unique both across all Passages and all hosts). Variant A (`passage` in the register PK) is rejected: the local `task_idx` is ambiguous even WITHIN a single Passage across hosts. Backward-compat: N=1 -> `plan_index == task_idx` (BIT-FOR-BIT behavior); an old Soul (without the field) sends 0, but staged (N>1) is gated on the `passage` capability (S5). Guard tests: `TestBuildRegisterByHost_MultiTaskPassage0_NoCollision` / `_PerHostDifferentWhere_NoMismatch` (unit), `TestIntegration_MultiTaskPassage0_RegisterNotClobbered` / `_PerHostDifferentWhere_RegisterResolves` (integration).

  **Amendment to S1 (onchanges/onfail-gating, local-remap + cross-passage reject — amend R1+R2).** `RenderedTask.onchanges_idx`/`onfail_idx` reached the Soul in **global** `RenderedTask.Index` terms (resolution of register names of requisites in `resolveOnChanges`/`resolveOnFail`), while the Soul keys `registerByIdx` by the **LOCAL** position of the task in `ApplyRequest.tasks[]` (`applyrunner.go::skipOnChanges`/`skipOnFail` — the only consumers of the local index; `no_log`/register correlation are already converted to the global `plan_index` above). When the `ApplyRequest.tasks[]` slice != the full plan (per-host `where:` OR a staged passage subset), the global Index of the source does NOT match its local position -> the `registerByIdx[onchanges_idx]` lookup misses -> the onchanges task is silently SKIPPED, the onfail-rescue silently does NOT run. **★ Already reachable at N=1 with per-host `where:`** (a source filtered out on the host shifts local positions) — a latent bug OUTSIDE staged.

  - **R1 — same-passage remap (Keeper-side, inside the single converter `ToProtoTasks`).** The converter builds a map `RenderedTask.Index -> local position` over the **input** slice (`perHost[sid]`) and remaps `onchanges_idx`/`onfail_idx` of each task **global->local** before writing to the wire. Fixes ALL call sites (dispatch / l2_run / pushorch / Acolyte-claim) at once without changing the signature. A **source absent from the slice** (cross-passage OR filtered out by `where:` on this host) is encoded as an **out-of-range sentinel (`-1`)**, and NOT dropped (dropping would shift the remaining indices and break the AND semantics of several sources — "at least one changed -> run"). The Soul already treats `registerByIdx[neg]`=nil -> `changed/failed=false` (the invariant "an absent source neither saves from skip nor activates onfail", `applyrunner.go::skipOnChanges`/`skipOnFail`) — the correct behavior comes for free, **the Soul is left untouched**. N=1 without `where:` (Index==localPos) -> remap=identity, BIT-FOR-BIT behavior. Guard tests: `TestToProtoTasks_RemapOnChangesGlobalToLocal` / `_RemapOnFailGlobalToLocal` / `_RemapMissingSourceSentinel` / `_RemapIdentityBackwardCompat` (unit), `TestIntegration_RemapOnChanges_N1Where_NoMisfire` (★ N=1+where reverse guard, integration).

  - **R2 — cross-passage requisite explicit-reject (safety net, REMOVED in R3).** R1 remap fixes ONLY same-passage requisites; it encodes cross-passage ones with the sentinel meaning "the source does not save", which for a genuinely changed/failed source in another Passage is a silent misfire (Soul gating of one Passage does not see the register of another `ApplyRequest`). Before R3, a cross-passage requisite was detected after `Stratify` in `run.go` (`config.CrossPassageRequisite` on top of the `Stratify` result — requisites are NOT passage-defining, their source may land in any Passage) -> **fail-closed abort `cross_passage_requisite_unsupported`** (symmetric to `serial_staged_unsupported`), so cross-passage gating did NOT misfire silently. **R3 replaced the reject with full keeper-side gating** — the explicit reject is removed, only the detection utility (`config.CrossPassageRequisite`) remains as a foundation for the keeper resolve. The former `TestIntegration_CrossPassageRequisite_Rejected` -> R3 FIRES/SKIPS guard tests.

  - **R3 — cross-passage onchanges/onfail gating (Keeper-side, IMPLEMENTED; R2-reject REMOVED).** The Keeper resolves cross-passage requisites **per-host** against CHANGED/FAILED facts accumulated over previous Passages (`auditpg.SelectChangedTaskKeys` / `SelectFailedTaskKeys`, keyed by the global `plan_index`), OR across several sources of the same kind (at least one changed/failed -> the condition is satisfied), AND between kinds (onchanges AND onfail). Same-passage requisites remain on the Soul (R1 global->local remap). The R2-reject (`cross_passage_requisite_unsupported`) is **removed** — cross-passage is now supported; fail-closed remains ONLY when the source of audit facts is missing. When a cross-passage requisite is resolved by the Keeper, its `onchanges_idx`/`onfail_idx` are stripped from the wire (the Soul executes/skips by the Keeper's decision, not by local gating). Guard tests: `TestIntegration_CrossPassageOnChanges_Fires` / `_Skips` / `_PerHostDivergent` / `_WhereAndChanged` (integration, formerly `_Rejected` -> FIRES/SKIPS).

    **★ The global-vs-local-task_idx class is CLOSED 6/6.** All six instances of the mismatch "global plan-index vs local position in `ApplyRequest.tasks[]`" are closed, all converted to the global `plan_index`/`Index`:
    1. **register correlation** — PK `apply_task_register (apply_id, sid, plan_index)`, migration 079 (Variant B above);
    2. **checkdrift register resolve** — `buildRegisterByHost` maps `nameByIdx[Index]` against the global `plan_index`;
    3. **failure channel** — `failed_plan_index` (global), migration 081;
    4. **no_log barrier** — masking of secret fields by the global index (`dispatch.go`);
    5. **audit changed_tasks** — T3, audit of changed tasks by the global `plan_index`;
    6. **onchanges/onfail gating** — R1 global->local remap (same-passage) + R3 keeper-side gating by the global `plan_index` (cross-passage).

    **★ LIMIT (honestly): onfail cross-passage FIRES is architecturally unreachable under the current fail-stop barrier.** A FAILED source in an earlier Passage aborts the whole run (`classify` -> abort) BEFORE the consumer Passage starts — so cross-passage onfail-rescue **cannot fire live**. The mechanism is implemented and unit-proven (`SelectFailedTaskKeys` + the onfail branch of the gate), it activates WHEN the barrier gets continue-on-partial-failure / `failed_when` tolerance semantics (future follow-up, a separate fork). onchanges cross-passage FIRES **works** (changed does not abort the barrier — the consumer survives to its own Passage).
- **Change in terminal semantics:** now there are **N `RunResult`** per `(apply_id, sid, passage)` (one per Passage), not one per `(apply_id, sid)`. Each Passage's barrier waits for the terminals of its own slice; the final run outcome comes after the terminal of the last Passage. `RunResult` -> row correlation is by `(apply_id, sid, passage)` + attempt-fencing ([ADR-027](0027-apply-work-queue.md), unchanged within a Passage).
- **Forward-compat (old Soul under a staged scenario):** a Soul without the `passage` field (an old binary) under a scenario that stratified into N>1 Passages — gets an **explicit refusal**: render/dispatch fails with `render_failed` (reason: "the staged scenario requires a Passage-aware Soul; this SID does not support the `passage` field"), **NOT a hang** and **not a silent single-pass execution** (which would bring back the original drift — an empty register in `where:`). Detecting "the Soul understands passage" is by the Soul's version/capability (the exact indicator — S5). N=1 scenarios (without register dependencies) remain compatible with an old Soul without a refusal (passage=0, behavior identical to today).

**Canonical registry of Task register sources.** Stratification and the cross-ref validator must build the SAME register-dependency graph. The exhaustive list of **passage-defining** register-reading Task fields (fixed as the S2 canon):

| Field | Reference form | Why passage-defining |
|---|---|---|
| `where:` | bare CEL `register.<name>.*` | Passage host targeting — resolved by the Keeper BEFORE dispatch, must see the register of the preceding Passage |
| `vars:` | `${ register.<name>.* }` in values | render inputs of a Passage task |
| `params:` | `${ register.<name>.* }` in values (module task) | render inputs of the module |
| `apply.input:` | `${ register.<name>.* }` in values (applier task) | render inputs of the delegated destiny |
| `output:` | `${ register.<name>.* }` in values | the task's declared output, also rendered by the Keeper from register |
| `loop.items:` | `${ register.<name>.* }` in a scalar/list/map | render-time fan-out: the list of iterations is built from register |
| `block:` | recursion into nested tasks | block is an atomic Passage unit (orchestration.md §2.2.1); register of its descendants is addressed flat |

**Flow-control CEL** (`when:` / `changed_when:` / `failed_when:` / `retry.until:`) — **NOT passage-defining**: executes Soul-side (sandboxed, [ADR-012(d)](0012-keeper-soul-grpc.md)) from the register accumulated within a single `ApplyRequest`. **Not included in the stratification graph** (`collectTaskReads`) — see the amend (2026-06-21) below. `loop.when:` — an exception: remains passage-defining (the loop fan-out is built Keeper-side BEFORE dispatch, like `loop.items:`).

**The "roster-refresh" class — a SEPARATE axis of passage-defining edge (amend [ADR-061](0061-onboarding-await-and-midrun-reresolve.md) §S2).** Alongside the register sources of the table above, the stratifier accounts for a second, independent class of passage-defining edge — **roster-refresh**:

| Edge class | Emitter | Consumer | Semantics |
|---|---|---|---|
| roster-refresh | `core.soul.registered` with `refresh_soulprint: true` | roster reads: `on: [incarnation.name]` / an omitted `on:` / `soulprint.hosts` / `soulprint.where(...)` / `soulprint.self.*` | the consumer goes into a Passage strictly AFTER the refresh emitter (program-order), otherwise render would use the OLD roster (silent-wrong-target) |

This is **NOT a register graph**: the refresh boundary does not introduce register references, so the reads⊆refs invariant (below) does NOT apply to it — the roster axis is orthogonal to the register axis. At the refresh boundary the scenario-runner re-resolves the roster from a **live snapshot** of the current online set ([ADR-061 §S3](0061-onboarding-await-and-midrun-reresolve.md)). The logic — `shared/config/passage_refresh.go` (the edge is baked into `Stratify`); guard tests — `shared/config/passage_refresh_test.go`.

**Amend (2026-06-21): flow-control gating — stratifier narrow-fix + cross-passage UNSUPPORTED (FC-5).** The previous canon included flow-control (`when`/`changed_when`/`failed_when`/`retry.until`) in the stratifier's graph as a "conservative over-approximation". This was a **BUG**: a register-dependent `when:` **split** a probe and its same-passage consumer across different Passages (probe->Passage 0, `when: register.X` consumer->Passage 1), where the Soul cannot see the cross-passage register -> `no such key`, the task FAILED. `when` (Soul-side gating, [ADR-012(d)]) does NOT support cross-passage; `where` (Keeper-side targeting) does (the Keeper re-renders `where` with the register accumulated in the preceding Passage). The asymmetry is legitimate.

- **Narrow-fix:** `when`/`changed_when`/`failed_when`/`retry.until` are **removed from `collectTaskReads`** (passage-defining reads). A register-dependent `when` NO LONGER splits a Passage -> a probe and its same-passage when-consumer land in one Passage -> the Soul sees the register -> `when` works same-passage (as intended, [ADR-012(d)]).
- **Deliberate asymmetry `refs ⊋ passage-reads`:** `when`/`changed_when`/`failed_when` **REMAIN** in the cross-ref validator (`config.collectRefs`) — these are register-READING fields (checked for "does the register exist", `unknown_register_reference`), but NOT passage-DEFINING. The guard invariant below fixes: for flow-control `reads ⊊ refs` (`TestStratify_FlowControlInRefsNotPassageReads`), and a register-dependent `when` does NOT split a Passage (`TestStratify_RegisterDependentWhenDoesNotSplitPassage`).
- **Genuinely cross-passage flow-control gating — UNSUPPORTED:** after the narrow-fix flow-control itself does not split a Passage, BUT a probe can still land in an early Passage for a DIFFERENT reason (another task with `where: register.X` — passage-defining — pushed the X-emitter into Passage 0, while the flow-control consumer moved to Passage 1 by its own register dependency). Then `when: register.X` — is genuinely cross-passage -> the Soul of the consumer's Passage does not see register X -> `no such key` silently. The detector **`cross_passage_when_unsupported`** rejects this **fail-closed** (offline `soul-lint` + a runtime keeper-side safety net `config.CrossPassageWhenGating`), symmetric to `within_block_register_dependency`. `register.self` (same-task) is NOT counted (`ExtractRegisterRefs` strips it). Fix: use `where:` for cross-task register targeting OR `register.self` for same-task gating.

**Requisites** (`onchanges:` / `onfail:` / `require:`) — **NOT passage-defining**: these are addressed references to a task by register name for Soul-side gating by result indices, not an interpolation of a register value into render. They are NOT included in the stratification graph.

**Amend (2026-06-20): `block` and within-block register dependency.** `block:` is an **atomic Passage unit**: all of its descendants land in ONE Passage and are NOT split probe->where within the block (a block is not broken up by the barrier). Consequence: a probe and its register consumer **must be DIFFERENT top-level tasks** (different Passages). **Within-block** probe->where (a descendant reading `register.*` of a peer descendant of the SAME block) is physically impossible at render time: the peer register is available only Soul-side AFTER the probe, while the descendant's `where:` is resolved by the Keeper BEFORE the block is dispatched — register is empty -> silent-wrong-target. The detector **`within_block_register_dependency`** rejects such a dependency **fail-closed** (offline `soul-lint` + a runtime keeper-side safety net), symmetric to the silent-wrong-target risk above. **Valid**: an external top-level probe + a block-descendant consumer — a normal staged-render (probe in an early Passage, block in the next; the `restart`/`main.yml` idiom).

**Amend (2026-06-20): the probe-completeness idiom superseded by the fail-stop barrier.** The previous canon in orchestration.md §5 recommended an extra diagnostic predicate on the probe task, `failed_when: size(register.<own>) < incarnation.host_count` ("probe response completeness"). This idiom is **physically unexecutable** and is **removed from the canon/examples**:

- **`failed_when:` is Soul-side per-host** (flow-control CEL, [ADR-012(d)](0012-keeper-soul-grpc.md)), executed on the slice of a SINGLE host from the register accumulated by its own `ApplyRequest`. A cross-host aggregate ("how many hosts responded") cannot be expressed in this slice: `register.<probe_name>` — a sid->payload map — has not yet been assembled by the barrier at the moment of `failed_when:` on the probe task itself (this is its own register, which the Keeper aggregates AFTER the barrier of the Passage), and a CEL reference to the probe's own name -> `no such key`. That is, the predicate gives neither a true cross-host count nor even a valid expression.
- **Probe completeness is already guaranteed by the fail-stop barrier** (§(c) barrier + §(d) state-commit): a probe is a separate Passage; a probe that fails on any host (finally-failed after `retry:`) -> that Passage's barrier does not converge -> the run stops, **the where-Passage consumer does not start**, `state_changes` is not committed -> `error_locked`. The barrier gives the same safety (no destructive action on an incomplete probe) that the idiom only duplicated with diagnostics.
- **Register incompleteness is caught at the where-consumer:** if the probe did not produce a value for a host, `where: register.<name>.stdout == ...` in the next Passage -> an eval error (`no such key`) at resolve time — an observable error, not a silent-wrong-target.

The idiom has been removed from the redis service examples (`examples/service/redis/scenario/**`). Executable `failed_when:` on `register.self.*` (same-task result — for example a health gate for a replica in `restart`, `failed_when: "!contains(register.self.stdout, 'master_link_status:up')"`) — are **kept** (this is not a completeness idiom). **`failed_when:` / `when:` / `changed_when:` on the register of ANOTHER task in an EARLIER Passage do NOT work** (flow-control = Soul-side gating of one Passage, see the amend (2026-06-21) about the narrow-fix above): cross-passage flow-control gating is **UNSUPPORTED**, caught by `cross_passage_when_unsupported`. `remove_node` does NOT use cross-passage flow-control — its topology protection (current master with slots) is built through `soulprint.hosts` and the logic of the `community.redis` plugin.

**Maintenance invariant.** A new Keeper-renderable **passage-defining** register-reading Task field must **synchronously** update three places: `config.collectTaskReads` (the stratifier), `config.collectRefs` (the cross-ref validator), and this registry; plus the guard test **reads⊆refs consistency** (`shared/config/passage_test.go`, `TestStratify_ReadsEqRefsConsistency`), which proves that the set of passage-defining source fields of the stratifier matches what the validator covers in this class (a fixture with a ghost-register in each source field is caught by BOTH: `StratifyUnknownRegister` AND `unknown_register_reference`). Divergence between these graphs means either silent-wrong-target (the stratifier does not see the field -> a silently incomplete graph) OR a linter hole (an unknown-register survives to runtime instead of being caught offline). **Flow-control (`when`/`changed_when`/`failed_when`) is a deliberate asymmetry `refs ⊋ passage-reads`** (amend 2026-06-21): in `collectRefs` (register-reading, checked for existence), but not in `collectTaskReads` (NOT passage-defining). This asymmetry is fixed by `TestStratify_FlowControlInRefsNotPassageReads` and `TestStratify_RegisterDependentWhenDoesNotSplitPassage` — a regression "put flow-control back into `collectTaskReads`" turns them red (`Stratify` would again start splitting a probe<->when-consumer = FC-5).

**serial / run_once.** `serial:` is the axis of **target hosts** (waves of <=N hosts, orchestration.md §2.2.1); Passage is the axis of **tasks** (stratification by register). The axes are **orthogonal**, and now rotate jointly (**2D serial×passage implemented**, S-2D1): within EACH Passage its own serial barrier applies to the hosts of THAT Passage. The pilot restriction `serial_staged_unsupported` is **removed** (see §S4 below). `run_once:` + staged are **compatible** with no restrictions: `run_once` narrows the target of a Passage task down to the first by SID from the ALREADY resolved (with this Passage's fresh register) `on:`+`where:` target — resolution happens per-Passage in the repeated Render, so `run_once` sees the real register, not an empty one.

**Min-width — per-Passage, NOT per-RUN (S-2D1 amend).** Wave width is derived from the tasks of **one Passage** (`effectiveSerialWidth` on the `tasksForPassage`-filtered slice of `plans`), not the whole run. A probe-Passage without `serial:` runs in **one wave**, even when a subsequent Passage carries `serial:1` — the narrow window of one Passage does NOT leak into another (eliminates silent-wrong-width: a per-RUN min-width would give the probe pass a destructive throttle). The min-width invariant (the minimum positive width among the tasks) is preserved, but the aggregation scope is narrowed to a Passage. **At N=1 (a single Passage) per-Passage and per-RUN coincide bit-for-bit.** Reverse guard: `TestUnit_EffectiveSerialWidth_PerPassageSlice` + integration `TestIntegration_SerialStaged_ProbeNotThrottled`.

**Amendment 2026-06-29: keeper-target — per-Passage dispatch (keeper->keeper register-chaining).** The S1 pilot implemented keeper-side tasks (`on: keeper`, [ADR-017](0017-keeper-side-core.md)) as a **single pass BEFORE host fan-out**: all `on: keeper` tasks of a run were rendered and executed together in one row `apply_runs(apply_id, 'keeper', 0)` (passage 0), BEFORE the stratified host tasks. This ADR **did NOT introduce the invariant "keeper only Passage 0"** — it lived only in the keeper dispatch code (`keeper_dispatch.go` / `dispatch.go`) as an implementation assumption of the pilot. The assumption is **removed**: a keeper task reading `register.X` (in `params:` / `where:` / other passage-defining fields of the registry above) is **required** to go into a Passage strictly AFTER the emitter of that register — exactly like a host task. The canonical example — `core.bootstrap.delivered` (`hosts: ${ register.<provision>.hosts }`) goes into a Passage AFTER `core.cloud.created` (both `on: keeper`); a single pass would have merged them into one Passage 0, where `register.<provision>` is still empty -> token delivery to an empty host set (a silent-wrong-target on the keeper side).

- **Stratification.** Keeper-target is NOT a separate axis: keeper tasks go through the same `Stratify` as host tasks (the register graph §(b) + the roster-refresh axis [ADR-061](0061-onboarding-await-and-midrun-reresolve.md)). The side of the task (`on: keeper` vs host) determines **where** to dispatch within a Passage, **not** which Passage it belongs to. A Passage may carry only keeper tasks, only host tasks, or both.
- **Stage-loop.** Keeper tasks of the active Passage execute **on every iteration** of the stage-loop (a per-Passage render with the register of previous Passages accumulated), NOT in a single pass before the loop. The render of a keeper task of Passage P>0 sees the register of all Passages < P (the same `in.Register` as for host tasks) — keeper->keeper register-chaining works (`core.cloud.created` Passage 0 -> `core.bootstrap.delivered` Passage 1).
- **PG.** Keeper-target is now **N rows** `apply_runs(apply_id, 'keeper', passage)` — one per Passage that carries keeper tasks (rather than the single `(apply_id, 'keeper', 0)`). The Variant I PK (`apply_runs (apply_id, sid, passage)`, migration 078) **already allows** this — `sid='keeper'` is no different from a host SID in the composite. The barrier classifier accounts for the keeper-target of its **own** Passage (the terminal of that Passage's keeper job alongside the host terminals).
- **The state-commit invariant §(d) is UNAFFECTED.** `incarnation.state` is still committed **once** after the barrier of the last Passage — splitting keeper tasks across Passages only affects the task axis (when to render/send what on the keeper side), not the commit axis. Failure of a keeper task in any Passage (after `retry:`) -> fail-stop, subsequent Passages do not start, state is not committed -> `error_locked` — same as for host tasks.
- **Backward-compat.** A scenario whose keeper tasks have no register dependencies on each other stratifies into a single level (same as the previous single pass) — behavior for non-chaining keeper compositions is identical to the pilot. Implements the keeper->keeper Passage chain that [ADR-061](0061-onboarding-await-and-midrun-reresolve.md) (`core.cloud.created -> core.soul.registered`) and [ADR-063](0063-bootstrap-token-delivery.md) (`core.cloud.created -> core.bootstrap.delivered`) already describe as required — closes the doc-drift "the ADR promises a chain, the S1 engine executes it in a single pass" (symmetric to the original drift for which this ADR was created).

**Amendment 2026-07-02: keeper-register in the `state_changes` render scope (run-level underlay, host-wins).** The register of keeper-side tasks (`on: keeper`) is accumulated under the synthetic SID `keeper` (`accumulateKeeperRegister` -> bucket `RegisterByHost["keeper"]`), but the state_changes render (`stateChangesVars`) took register **only** from the per-host bucket of the host — `${ register.<keeper-task>.* }` in `sets` failed with "no such key" -> `error_locked` AFTER a fully successful deploy (a live bug in `provisioned_vm_ids` on a create run with provision). Fix:

- **keeper-register becomes available in the state_changes render scope as a run-level layer**: each host's register context = the keeper bucket (underlay) + the host's per-host bucket on top; **the host's per-host register takes priority (host-wins)** on a name collision (collisions are impossible due to the duplicate-register validator — this is a formal safety net). A single vars builder (`stateChangesVars`) covers all folding paths: `set`/`add`, `modify`/`remove` (`stateContextSnapshot`), legacy `sets`, `foreach`.
- **Additive scope extension**: the isolation of host tasks from keeper-register in `params:`/`when:` is **preserved** (the `hostRegister` fallback and the isolated channel `KeeperRegister`->`keeperVars` are unchanged) — only state_changes sees the underlay, and it is rendered keeper-side after the last Passage's barrier anyway (§(d)).
- The L0-harness (`trial`) masked this class (broadcasting register to all hosts) — the parity guard goes through the live path: `TestIntegration_KeeperRegisterInSets_CommitsToState` (integration), `TestStateChangesVars_KeeperRegisterVisible` / `_HostWinsOnCollision` / `_NoKeeperBucket_BitForBit` (unit).

**§S4 — pilot boundaries (implemented restrictions).**

- **`serial:` + staged (N>1) -> IMPLEMENTED (restriction removed, S-2D1).** The 2D matrix `serial`x`passage` is built: the Passage cycle in run.go executes each Passage in order, and `dispatchPassage` internally splits hosts into serial waves from the tasks of THIS Passage only (`effectiveSerialWidth` on the `tasksForPassage` slice — per-Passage width). The former fail-closed gate `serial_staged_unsupported` (run.go + soul-lint) is **REMOVED**; the broken fixture `scenario-serial-staged` is re-profiled into a valid golden (`passage_plan` HINT). Per-Passage barrier and fail-stop are inherited from the serial mechanics without changes: a host failing in a wave of Passage P -> the next wave and Passage P+1 do not start, state is not committed (last known-good). **The crash-recovery limit of staged-inline persists** (serial+staged goes through the same inline path as serial without staged -> Acolyte-reclaim does not cover it, see the next bullet) — this is **not a new regression**, but the inheritance of a general property of the inline path. Guard tests: `TestIntegration_SerialStaged_RollingPerPassage` (rolling by waves per-Passage), `_ProbeNotThrottled` (★ reverse per-passage width), `_FailStopInWave` (fail-stop + state not committed), `_RegisterAcrossWaves` (the register of all waves of Passage P collected before Passage P+1).

- **staged runs go INLINE; crash-recovery — the Voyage path is CLOSED via orphan-lock-release, standalone (direct run) is NOT yet covered.** A staged run (N>1) at any value of `AcolyteEnabled` executes via the OLD inline path (run.go gate `!staged`): Acolyte renders per-host once at claim time, not per-Passage, so Acolyte-claim is disabled for staged. Consequence for recovery: the Reaper rule `reclaim_apply_runs` (ADR-027 amend) is narrowed to `status='claimed'` with an expired lease — it returns to the queue only jobs that died BEFORE being handed off to the Soul. The inline path does NOT write `claimed` (it does `Insert(running)` -> `SendApply` directly, attempt=0), so `reclaim_apply_runs` does not pick it up. **The Voyage path (a staged incarnation inside a reclaimed Voyage) is CLOSED by the orphan-lock-release fix** ([ADR-027 amend (l)](0027-apply-work-queue.md), 2026-06-21): a reclaimed VoyageWorker detects an orphaned `applying` via the `voyage_targets.apply_id` back-link of the previous attempt and releases the lock (`applying->ready`, fenced) BEFORE re-run — a reclaimed Voyage no longer hangs on a staged incarnation, it restarts the run from a clean start (not a per-Passage resume — the whole scenario again; module idempotency + gate-1 fencing keep correctness). **STANDALONE staged (a direct run without Voyage — `POST /v1/incarnations/{name}/run`) remains OPEN**: it has no Voyage wrapper with a back-link, so there is no detect-anchor for the orphan seam — **a keeper-instance crash mid-run of such a run is not picked up**: the rows remain `running`, the incarnation — `applying`, the stage-loop does not resume. This is **NOT a silent loss/corruption**: the state is observable (the incarnation hangs in `applying`, apply_runs in `running`), it requires operator triage/unlock — not a silent-wrong-target and not a mismatch of `incarnation.state` (state is committed strictly ONCE after the last Passage, ADR-056 §d). Boundary note: this is a **general property of the inline path** (the serial guard and non-Acolyte instances also do not recover it if the run-goroutine crashes, BAG-1 `ensureTerminalApplyRun` only fires on an abort IN THE SAME goroutine, not on a process crash), staged merely inherits it. **Backstop follow-up for standalone:** an incarnation-lock owner column (epoch/kid on the `incarnation` row, like Ward) OR per-passage Acolyte-claim + per-passage reclaim/fencing (an ADR-027 amend at the `(apply_id, sid, passage)` level) — **deferred**. Guard test: `TestIntegration_StagedInlineNotAcolyteClaim` (staged + `AcolyteEnabled=true` goes inline — not a single `planned`/`claimed` row, the Acolyte path `dispatchPlanned` is not invoked).

- **3-stage restart re-probe — proven.** Topological stratification + the program-order edge of the emitter give `[0,1,1,1,2]` (Count=3) on the restart idiom (probe -> action -> re-probe -> action): the re-probe (`redis_role_after`) falls into a Passage AFTER the failover action (program-order: a register emitter does not precede preceding tasks), and the re-probe's consumer -> into Passage 2. Live proof of retargeting by the FRESH re-probe (not the first probe): `TestIntegration_Staged3Passage_ReprobeRetargets` (after a failover the master changes host-a->host-b; the Passage-2 action targets ONLY the new master via the Passage 1 re-probe).

**Rejected alternatives.**

- **Variant B (doc-only known-limitation).** Record in orchestration.md that probe->where does not work (a known limitation), and do not implement it. **Rejected by the user** in favor of implementation: probe->where is the basis of the volatile role ([ADR-008](0008-coven-stable-tags.md)), the canon already promises it, leaving the engine silently not delivering on that promise is unacceptable drift.
- **Fixed 2-stage** (exactly two passes: a probe pass, then an action pass). Rejected: **does not cover restart re-probe** (probe -> action -> re-probe -> action — three+ dependency levels, e.g. checking role before and after a failover). N-stage (topological) covers an arbitrary depth of register dependencies.

**Risks.**

- **silent-wrong-target on a stratification error.** If the register-dependency graph is parsed incorrectly (a consumer ends up in the same/an earlier Passage than its probe), `where:` again selects against an empty/stale register — a destructive operation silently lands on the wrong hosts. **Guard tests are mandatory (TDD, S2)**: stratification is verified by tests "a probe and its consumer are not in the same Passage" before the stage-loop is implemented. Symmetric to the silent-destructive-on-partial footgun (orchestration.md §5).
- **Forward-compat.** An old Soul under a staged scenario must receive an **explicit-reject** (`render_failed`), not silently run a single pass (which would bring back the original drift) and not hang. Tested in S5.
- **Regression of barrier / Voyage / recovery.** Changing a per-host job into a per-Passage job (PG-PK, ~10 packages: barrier reads, Ward-claim, reclaim, the `voyage_targets.apply_id` back-link, the history timeline, attempt-fencing) — a broad touch zone. Mitigation — the pilot order S1->S5 (contract+PG before stratification, stratification before stage-loop), an additive migration, guard tests at every slice.
