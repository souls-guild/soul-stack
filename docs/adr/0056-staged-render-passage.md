# ADR-056. Staged-render — прогон сценария как N упорядоченных Passage (probe→where реально работает)

> **Статус: active.** Решение пользователя (Variant A-полный, N-stage; имя новой фазы прогона — **Passage**) + дизайн architect-а. Канон фиксируется docs-first ДО кода; имплементация — отдельные слайсы S1–S5 (см. §Слайс-карта). Этот ADR **amends [ADR-009](0009-scenario-dsl.md), [ADR-012](0012-keeper-soul-grpc.md), [ADR-027](0027-apply-work-queue.md)** (см. §Связь с ADR) и закрывает [open Q №24](../architecture.md#открытые-вопросы) (per-task гранулярность `serial:` / per-task dispatch).
>
> **Прогресс имплементации.** S1–S5 реализованы (proto `passage`/`plan_index`, миграции 078/079/081, стратификация, stage-loop, Soul forward-compat, soul-lint). Поправки к S1 (register-корреляция, R1 onchanges/onfail remap) реализованы; **R3 (cross-passage onchanges/onfail keeper-side gating) реализован — R2-reject снят** (см. блок «Поправка к S1 (onchanges/onfail-gating)»). Класс **global-vs-local-task_idx закрыт 6/6**. Открытый лимит — **onfail cross-passage FIRES недостижим под текущим fail-stop барьером** (механизм реализован+unit-доказан, активируется при continue-on-partial-failure семантике; future follow-up).

**Контекст.** Канон [`docs/scenario/orchestration.md §4`/`§5`](../scenario/orchestration.md#4-волатильный-предикат--where) подаёт probe→where как **рабочий** механизм: probe-шаг (`module: core.exec.run` + `register: X` + `changed_when: false`) собирает per-host факт, а **последующая** задача таргетит по нему через `where: register.X.stdout == 'slave'` либо протаскивает его в `apply: input:` / `params:`. Этот механизм — основа волатильной роли из [ADR-008](0008-coven-stable-tags.md) (роль НЕ Coven; actual-роль берётся только живым probe).

На практике **он не реализован**: Keeper рендерит весь сценарий **одним up-front проходом** ДО dispatch-а. На render-фазе `in.Register` (карта результатов probe) **пуст** — probe ещё не исполнялся; поля `Where` в `RenderedTask` (`proto/keeper/v1`) **нет вовсе** — `where:` резолвится keeper-side в момент рендера, то есть **до** того, как probe вернёт `register`. Следствие:

- `where:` по `register.*` предыдущего probe всегда видит пустой register → отбирает 0 хостов (или падает на отсутствующем ключе);
- `apply: input:` / `params:` / `vars:` следующей задачи, читающие `register.X`, на render-фазе тоже не имеют значения;
- канон обещает работающую идиому, движок её молча не исполняет — **doc-drift между orchestration.md и keeper**.

> **Что уже работает (граница drift-а).** Flow-control CEL `when:`/`changed_when:`/`failed_when:` по `register.*` исполняется **Soul-side** (sandboxed, [ADR-012(d)](0012-keeper-soul-grpc.md)) — Soul строит `register` сам из накопленных результатов задач в рамках **одного** `ApplyRequest`. Это покрывает register-зависимости **внутри среза одного хоста**. Drift — именно в **таргетинге** (`where:` — «на каких хостах», резолвится Keeper-ом ДО probe) и в **cross-task render** (`apply: input:`/`params:` следующей задачи от register предыдущей), которые требуют per-host register **на Keeper-е после барьера probe**.

Этот ADR вводит **staged-render**, чтобы реализовать обещанное каноном.

**Решение.** Прогон сценария исполняется как **N упорядоченных Passage**.

- **(а) Passage — фаза прогона.** Один Passage — упорядоченный цикл **render → dispatch → barrier → сбор register**. Сценарий из K задач исполняется как N ≥ 1 Passage; каждый Passage несёт подмножество задач прогона. Задачи без register-зависимостей укладываются в один Passage (N=1 — текущее поведение, fast-path).

- **(б) Стратификация по register-зависимости (топологический N-stage).** Задачи прогона **стратифицируются** по графу register-зависимости: задача, читающая `register.X` (в `where:` / `apply: input:` / `params:` / `vars:`), попадает в Passage **строго ПОСЛЕ** probe-шага, эмитящего `register: X`. Probe и его потребитель не могут оказаться в одном Passage. Это топологическая сортировка задач по рёбрам «probe X → читатель register.X»; число Passage = число уровней (длина критического пути зависимостей + 1). Источник зависимостей register — статический разбор `where:` / интерполяций `${ … }` в `apply: input:`/`params:`/`vars:` на наличие `register.<name>.*`.

- **(в) Цикл одного Passage.**
  1. **render** — Keeper рендерит задачи **этого** Passage в `RenderedTask`, подставляя `register` **всех предыдущих** Passage (per-host карта, накопленная барьерами). На первом Passage register пуст (как сейчас); на втором и далее — заполнен фактами probe из предыдущих Passage.
  2. **dispatch** — `ApplyRequest` на хосты, прошедшие резолв `on:`+`where:` **этого** Passage (теперь `where:` видит реальный register).
  3. **barrier** — безусловное ожидание терминала всех per-host заданий Passage (cross-host barrier, parity [ADR-009 §7](0009-scenario-dsl.md), но **внутри** Passage).
  4. **сбор register** — per-host `register`-карта Passage накапливается через уже существующий канал `TaskEvent.register_data` → `apply_task_register` (см. §Reuse) и доступна render-у следующего Passage.

- **(г) state-commit — ОДИН раз после последнего Passage (инвариант §7 не дробится).** `incarnation.state` (`state_changes`) коммитится **единожды** после успешного барьера **последнего** Passage прогона — НЕ по-Passage. Это прямое сохранение barrier/state-commit-инварианта [ADR-009 §7](0009-scenario-dsl.md) / orchestration.md §7: частичный коммит после промежуточного Passage **запрещён** (иначе `incarnation.state` разойдётся с фактом при падении следующего Passage). Падение любого Passage (finally-failed после `retry:`) → прогон останавливается, последующие Passage не стартуют, state НЕ коммитится → `error_locked`. Passage — ось **задач** (когда что рендерить/слать), а не ось коммита.

**Reuse (половина механизма уже есть).** Staged-render переиспользует существующий post-barrier register-канал — не вводит новый:

- **`loadRegisterByHost` / `apply_task_register` / `SelectTaskRegistersByApplyID`** (post-barrier сбор `register` под `state_changes.sets`, [ADR-009 §7.1 «register.* как источник sets»](0009-scenario-dsl.md)) — переиспользуется как **Passage-loop**: тот же сбор per-host register после барьера, но теперь его читает render следующего Passage, а не только финальный `sets`-рендер.
- **`evalWhere` / `renderParams`** уже принимают `in.Register` параметром — недостающее не в их сигнатуре, а в том, что `in.Register` на момент их вызова **пуст** (up-front render). Staged-render заполняет `in.Register` между Passage.
- **Недостающее (зона S1–S3):** Passage-цикл в run.go (run-goroutine / Acolyte) + повторный `Render` с заполненным `Register` на втором+ Passage + поле `passage` в proto/PG для корреляции per-Passage заданий и terminal-ов.

**Контракт.**

- **proto — only-add ([ADR-012(c)](0012-keeper-soul-grpc.md) forward-compat, never-reuse field-номеров):** `int32 passage` в `ApplyRequest` / `TaskEvent` / `RunResult` — индекс Passage (0-based), к которому относится задание/событие/отчёт. Это **only-add** в существующие сообщения `proto/keeper/v1/apply.proto`; номера полей не реюзятся, старые поля не удаляются. Точные field-номера фиксируются в S1.
- **PG-схема:** на один хост прогона теперь приходится **N** заданий (по одному на Passage), а не одно. Текущий composite PK `apply_runs (apply_id, sid)` ([ADR-027](0027-apply-work-queue.md), orchestration.md §2.2.1) больше не уникален per-host. Два варианта — **выбор фиксируется в S1-дизайне**:
  - **(I)** расширить PK до `apply_runs (apply_id, sid, passage)` — N строк на хост в той же таблице;
  - **(II)** новая таблица `apply_run_passages` (per-Passage задания), `apply_runs` остаётся per-host агрегатом.
  Выбор делается в S1 по критериям совместимости с Ward-claim / reclaim / barrier-чтением (`SelectStatusesByApplyID`) / Voyage back-link (`voyage_targets.apply_id`) / recovery — затрагивается ~10 пакетов, миграция аддитивная.

  **S1 — реализован Variant I** (миграция 078: PK `apply_runs (apply_id, sid, passage)`, FK `apply_task_register → apply_runs (apply_id, sid, passage)`).

  **Поправка к S1 (register-корреляция, миграция 079).** Запись миграции 078 «`task_idx` сквозной по всем Passage» **фактически неверна**: `TaskEvent.task_idx` — это **ЛОКАЛЬНАЯ** позиция задачи в `ApplyRequest.tasks[]` своего Passage. `ApplyRequest` одного Passage несёт лишь подмножество задач этого Passage, отфильтрованное ещё и **per-host** через `where:` — поэтому `task_idx` НЕ уникален ни между Passage (probe Passage0/локальная 0 и действие Passage1/локальная 0 делят `task_idx=0`), ни между хостами одного Passage (разный `where:` → разный локальный индекс одной задачи). Прежний PK `apply_task_register (apply_id, sid, task_idx)` затирал probe-register по `ON CONFLICT`, а `buildRegisterByHost` мапил `nameByIdx[Index]` (глобальный) против `rows.task_idx` (локальный) — рассинхрон имён на `passage>0` (латентный баг S1).

  **Механизм register-корреляции — Вариант B (глобальный `plan_index`)**, дизайн architect-а. Proto only-add `RenderedTask.plan_index` (field 16) + `TaskEvent.plan_index` (field 8) — **глобальный сквозной индекс задачи по ВСЕМУ плану** (по всем Passage), = `RenderedTask.Index` на Keeper-е. Soul эхает полученный `plan_index` (рядом с локальным `task_idx`, чья семантика НЕ меняется — на нём Soul-side onchanges/onfail-gating). `apply_task_register` ключуется по `plan_index` (миграция 079: PK `(apply_id, sid, plan_index)`); `buildRegisterByHost` резолвит `nameByIdx[Index]` против глобального `plan_index` — оба дефекта уходят (глобальный индекс уникален и по всем Passage, и по всем хостам). Variant A (`passage` в PK register) отвергнут: локальный `task_idx` неоднозначен и ВНУТРИ одного Passage между хостами. Backward-compat: N=1 → `plan_index == task_idx` (поведение БИТ-В-БИТ); старый Soul (без поля) шлёт 0, но staged (N>1) gated на capability `passage` (S5). Guard-тесты: `TestBuildRegisterByHost_MultiTaskPassage0_NoCollision` / `_PerHostDifferentWhere_NoMismatch` (unit), `TestIntegration_MultiTaskPassage0_RegisterNotClobbered` / `_PerHostDifferentWhere_RegisterResolves` (integration).

  **Поправка к S1 (onchanges/onfail-gating, local-remap + cross-passage reject — amend R1+R2).** `RenderedTask.onchanges_idx`/`onfail_idx` доезжали до Soul-а в **глобальном** `RenderedTask.Index` (резолв register-имён requisite-ов в `resolveOnChanges`/`resolveOnFail`), а Soul ключует `registerByIdx` **ЛОКАЛЬНОЙ** позицией задачи в `ApplyRequest.tasks[]` (`applyrunner.go::skipOnChanges`/`skipOnFail` — единственные потребители локального индекса; `no_log`/`register`-корреляция уже переведены на глобальный `plan_index` выше). Когда срез `ApplyRequest.tasks[]` ≠ полный план (per-host `where:` ИЛИ staged passage-подмножество), глобальный Index источника НЕ совпадает с его локальной позицией → лукап `registerByIdx[onchanges_idx]` промахивается → onchanges-задача молча SKIPPED, onfail-rescue молча НЕ запускается. **★ Достижимо УЖЕ при N=1 с per-host `where:`** (отфильтрованный на хосте источник сдвигает локальные позиции) — латентный баг ВНЕ staged.

  - **R1 — same-passage remap (Keeper-side, внутри единого конвертера `ToProtoTasks`).** Конвертер строит карту `RenderedTask.Index → локальная позиция` по **входному** срезу (`perHost[sid]`) и ремапит `onchanges_idx`/`onfail_idx` каждой задачи **global→local** перед записью в wire. Чинит ВСЕ call-site (dispatch / l2_run / pushorch / Acolyte-claim) разом без смены сигнатуры. **Отсутствующий-в-срезе источник** (cross-passage ИЛИ отфильтрован `where:` на этом хосте) кодируется **out-of-range sentinel (`-1`)**, а НЕ выкидывается (выкидывание сместило бы остальные индексы и сломало бы AND-семантику нескольких источников «хотя бы один changed → выполнить»). Soul уже трактует `registerByIdx[neg]`=nil → `changed/failed=false` (инвариант «отсутствующий источник = не спасает от skip / не активирует onfail», `applyrunner.go::skipOnChanges`/`skipOnFail`) — поведение получается бесплатно, **Soul не трогаем**. N=1 без `where:` (Index==localPos) → remap=identity, поведение БИТ-В-БИТ. Guard-тесты: `TestToProtoTasks_RemapOnChangesGlobalToLocal` / `_RemapOnFailGlobalToLocal` / `_RemapMissingSourceSentinel` / `_RemapIdentityBackwardCompat` (unit), `TestIntegration_RemapOnChanges_N1Where_NoMisfire` (★ N=1+where реверс-guard, integration).

  - **R2 — cross-passage requisite explicit-reject (страховка, СНЯТА в R3).** R1-remap чинит ТОЛЬКО same-passage requisites; cross-passage он кодирует sentinel-ом = «источник не спасает», что для реально-changed/failed источника в другом Passage = молчаливый мисфайр (Soul gating одного Passage не видит register источника другого ApplyRequest). До R3 cross-passage requisite детектился после `Stratify` в `run.go` (`config.CrossPassageRequisite` поверх `Stratify`-результата — requisites НЕ passage-определяющие, их источник может оказаться в любом Passage) → **fail-closed abort `cross_passage_requisite_unsupported`** (симметрия `serial_staged_unsupported`), чтобы cross-passage gating НЕ мисфайрил молча. **R3 заменил reject на полноценный keeper-side gating** — explicit-reject снят, осталась только утилита детекта (`config.CrossPassageRequisite`) как опора для keeper-резолва. Бывший `TestIntegration_CrossPassageRequisite_Rejected` → FIRES/SKIPS guard-тесты R3.

  - **R3 — cross-passage onchanges/onfail gating (Keeper-side, РЕАЛИЗОВАН; R2-reject СНЯТ).** Keeper резолвит cross-passage requisite **per-host** по накопленным за предыдущие Passage CHANGED/FAILED-фактам (`auditpg.SelectChangedTaskKeys` / `SelectFailedTaskKeys`, ключ — глобальный `plan_index`), OR между несколькими источниками одного вида (хотя бы один changed/failed → условие выполнено), AND между видами (onchanges И onfail). Same-passage requisites остаются на Soul-е (R1 global→local remap). R2-reject (`cross_passage_requisite_unsupported`) **снят** — cross-passage поддержан; fail-closed остаётся ТОЛЬКО при отсутствии источника аудит-фактов. Когда cross-passage requisite разрешён keeper-ом, его `onchanges_idx`/`onfail_idx` убираются с wire (Soul выполняет/пропускает по keeper-решению, не по локальному gating). Guard-тесты: `TestIntegration_CrossPassageOnChanges_Fires` / `_Skips` / `_PerHostDivergent` / `_WhereAndChanged` (integration, бывший `_Rejected` → FIRES/SKIPS).

    **★ Класс global-vs-local-task_idx ЗАКРЫТ 6/6.** Все шесть инстанций рассинхрона «глобальный план-индекс vs локальная позиция в `ApplyRequest.tasks[]`» закрыты, все переведены на глобальный `plan_index`/`Index`:
    1. **register-correlation** — PK `apply_task_register (apply_id, sid, plan_index)`, миграция 079 (Вариант B выше);
    2. **checkdrift register-резолв** — `buildRegisterByHost` мапит `nameByIdx[Index]` против глобального `plan_index`;
    3. **failure-канал** — `failed_plan_index` (глобальный), миграция 081;
    4. **no_log-barrier** — маскирование секрет-полей по глобальному индексу (`dispatch.go`);
    5. **audit changed_tasks** — T3, аудит changed-задач по глобальному `plan_index`;
    6. **onchanges/onfail-gating** — R1 global→local remap (same-passage) + R3 keeper-side gating по глобальному `plan_index` (cross-passage).

    **★ ЛИМИТ (честно): onfail cross-passage FIRES архитектурно недостижим под текущим fail-stop барьером.** FAILED-источник в раннем Passage валит весь прогон (`classify` → abort) ДО того, как стартует Passage-потребитель — поэтому cross-passage onfail-rescue **не может сработать live**. Механизм реализован и unit-доказан (`SelectFailedTaskKeys` + onfail-ветка gate), он активируется, КОГДА барьер получит continue-on-partial-failure / `failed_when`-tolerance семантику (future follow-up, отдельная развилка). onchanges cross-passage FIRES **работает** (changed не валит барьер — потребитель доживает до своего Passage).
- **Смена семантики terminal:** теперь **N `RunResult` на `(apply_id, sid, passage)`** (по одному на Passage), а не один на `(apply_id, sid)`. Barrier каждого Passage ждёт терминалы своего среза; финальный итог прогона — после терминала последнего Passage. Корреляция `RunResult`→строка — по `(apply_id, sid, passage)` + attempt-fencing ([ADR-027](0027-apply-work-queue.md), не меняется внутри Passage).
- **Forward-compat (старый Soul под staged-сценарием):** Soul без поля `passage` (старый бинарь) под сценарием, который стратифицировался в N>1 Passage, — **явный отказ**: render/dispatch падает с `render_failed` (причина: «staged-сценарий требует Passage-aware Soul; этот SID не поддерживает поле `passage`»), **НЕ зависание** и **не молчаливое одно-проходное исполнение** (которое вернуло бы исходный drift — пустой register в `where:`). Детект «Soul понимает passage» — по версии/capability Soul-а (точный признак — S5). N=1-сценарии (без register-зависимостей) совместимы со старым Soul без отказа (passage=0, поведение идентично текущему).

**Канонический реестр register-источников Task.** Стратификация и cross-ref-валидатор обязаны строить ОДИН и тот же граф register-зависимостей. Исчерпывающий список **passage-определяющих** register-читающих полей Task (фиксируется как канон S2):

| Поле | Форма ссылки | Почему passage-определяющее |
|---|---|---|
| `where:` | голый CEL `register.<name>.*` | таргетинг хостов Passage — резолвится Keeper-ом ДО dispatch, обязан видеть register предыдущего Passage |
| `vars:` | `${ register.<name>.* }` в значениях | render-входы задачи Passage |
| `params:` | `${ register.<name>.* }` в значениях (module-задача) | render-входы модуля |
| `apply.input:` | `${ register.<name>.* }` в значениях (applier-задача) | render-входы делегируемого destiny |
| `output:` | `${ register.<name>.* }` в значениях | декларированный output задачи, тоже рендерится Keeper-ом из register |
| `loop.items:` | `${ register.<name>.* }` в скаляре/списке/map | render-time fan-out: список итераций строится из register |
| `block:` | рекурсия по вложенным задачам | block — атомарная единица Passage (orchestration.md §2.2.1); register его потомков адресуется плоско |

**Flow-control CEL** (`when:` / `changed_when:` / `failed_when:` / `retry.until:`) — **НЕ passage-определяющий**: исполняется Soul-side (sandboxed, [ADR-012(d)](0012-keeper-soul-grpc.md)) из register, накопленного в рамках одного `ApplyRequest`. **Не включается в граф стратификации** (`collectTaskReads`) — см. amend (2026-06-21) ниже. `loop.when:` — исключение: остаётся passage-определяющим (fan-out цикла строится Keeper-side ДО dispatch, как `loop.items:`).

**Amend (2026-06-21): flow-control gating — narrow-fix стратификатора + cross-passage UNSUPPORTED (FC-5).** Прежний канон включал flow-control (`when`/`changed_when`/`failed_when`/`retry.until`) в граф стратификатора как «conservative over-approximation». Это был **БАГ**: register-зависимый `when:` **расщеплял** probe и его same-passage-потребителя по разным Passage (probe→Passage 0, `when: register.X`-потребитель→Passage 1), где Soul cross-passage register **не видит** → `no such key`, задача FAILED. `when` (Soul-side gating, [ADR-012(d)]) cross-passage **не умеет**; `where` (Keeper-side targeting) — умеет (Keeper пере-рендерит `where` с накопленным register предыдущего Passage). Асимметрия легитимна.

- **Narrow-fix:** `when`/`changed_when`/`failed_when`/`retry.until` **убраны из `collectTaskReads`** (passage-определяющих reads). Register-зависимый `when` БОЛЬШЕ НЕ расщепляет Passage → probe и same-passage when-потребитель в одном Passage → Soul видит register → `when` работает same-passage (как задумано [ADR-012(d)]).
- **Намеренная асимметрия `refs ⊋ passage-reads`:** `when`/`changed_when`/`failed_when` **ОСТАЮТСЯ** в cross-ref-валидаторе (`config.collectRefs`) — это register-ЧИТАЮЩИЕ поля (проверка «register существует», `unknown_register_reference`), но НЕ passage-ОПРЕДЕЛЯЮЩИЕ. Guard-инвариант ниже фиксирует: для flow-control `reads ⊊ refs` (`TestStratify_FlowControlInRefsNotPassageReads`), и register-зависимый `when` НЕ расщепляет Passage (`TestStratify_RegisterDependentWhenDoesNotSplitPassage`).
- **Genuinely cross-passage flow-control gating — UNSUPPORTED:** после narrow-fix flow-control сам Passage не расщепляет, НО probe может оказаться в раннем Passage по ДРУГОЙ причине (иная задача с `where: register.X` — passage-определяющий — загнала эмиттер X в Passage 0, а flow-control-потребитель уехал в Passage 1 своей register-зависимостью). Тогда `when: register.X` — genuinely cross-passage → Soul Passage потребителя не видит register X → `no such key` молча. Детектор **`cross_passage_when_unsupported`** отвергает это **fail-closed** (офлайн `soul-lint` + runtime keeper-страховка `config.CrossPassageWhenGating`), симметрично `within_block_register_dependency`. `register.self` (same-task) НЕ считается (его режет `ExtractRegisterRefs`). Лечится: `where:` для cross-task register-таргетинга ИЛИ `register.self` для same-task gating.

**Requisites** (`onchanges:` / `onfail:` / `require:`) — **НЕ passage-определяющие**: это адресные ссылки на task по register-имени для Soul-side gating по индексам результатов, не интерполяция значения register в render. В граф стратификации они НЕ входят.

**Amend (2026-06-20): `block` и within-block register-зависимость.** `block:` — **атомарная единица Passage**: все его потомки попадают в ОДИН Passage и НЕ расщепляются probe→where внутри блока (block не дробится барьером). Следствие: probe и его register-потребитель **обязаны быть РАЗНЫМИ top-level задачами** (разными Passage). **Within-block** probe→where (потомок читает `register.*` peer-потомка ТОГО ЖЕ блока) физически невозможен на render: peer-register доступен только Soul-side ПОСЛЕ probe, а `where:` потомка резолвится Keeper-ом ДО dispatch блока — register пуст → silent-wrong-target. Детектор **`within_block_register_dependency`** отвергает такую зависимость **fail-closed** (офлайн `soul-lint` + runtime keeper-страховка), симметрично риску silent-wrong-target выше. **Валидно**: внешний top-level probe + block-потомок-потребитель — нормальный staged-render (probe в раннем Passage, блок в следующем; идиома `restart`/`main.yml`).

**Amend (2026-06-20): probe-completeness идиома вытеснена fail-stop барьером.** Прежний канон orchestration.md §5 рекомендовал на probe-задаче дополнительный диагностический предикат `failed_when: size(register.<own>) < incarnation.host_count` («полнота ответа probe»). Эта идиома **физически неисполнима** и **убрана из канона/примеров**:

- **`failed_when:` — Soul-side per-host** (flow-control CEL, [ADR-012(d)](0012-keeper-soul-grpc.md)), исполняется в срезе ОДНОГО хоста из register, накопленного его собственным `ApplyRequest`. Cross-host агрегат («сколько хостов ответило») в этом срезе **не выражается**: `register.<имя_probe>` — карта sid→payload — на момент `failed_when:` probe-задачи ещё не собрана барьером (это её собственный register, который keeper агрегирует ПОСЛЕ barrier Passage), а CEL ссылка на собственное имя probe → `no such key`. То есть предикат не даёт ни истинного cross-host счёта, ни даже исполнимого выражения.
- **Полнота probe уже гарантирована fail-stop барьером** (§(в) barrier + §(г) state-commit): probe — отдельный Passage; упавший probe на любом хосте (finally-failed после `retry:`) → барьер этого Passage не сходится → прогон останавливается, **where-Passage-потребитель не стартует**, `state_changes` не коммитится → `error_locked`. Барьер даёт ту же safety (нет destructive-действия на неполном probe), что идиома лишь дублировала диагностикой.
- **Неполнота register ловится на where-потребителе:** если probe не дал значения для хоста, `where: register.<name>.stdout == …` следующего Passage → eval-error (`no such key`) на резолве — наблюдаемая ошибка, не silent-wrong-target.

Идиома удалена из примеров redis-сервиса (`examples/service/redis/scenario/**`). Исполнимые `failed_when:` по `register.self.*` (same-task результат — например health-gate реплики в `restart`, `failed_when: "!contains(register.self.stdout, 'master_link_status:up')"`) — **сохранены** (это не completeness-идиома). **`failed_when:` / `when:` / `changed_when:` по register ДРУГОЙ задачи в БОЛЕЕ РАННЕМ Passage — НЕ работают** (flow-control = Soul-side gating одного Passage, см. amend (2026-06-21) о narrow-fix выше): cross-passage flow-control gating **UNSUPPORTED**, ловится `cross_passage_when_unsupported`. `remove_node` cross-passage flow-control **не использует** — его защита топологии (current master со слотами) строится через `soulprint.hosts` и логику плагина `community.redis`.

**Инвариант сопровождения.** Новое keeper-рендеримое **passage-определяющее** register-читающее поле Task → **синхронно** обновить три места: `config.collectTaskReads` (стратификатор), `config.collectRefs` (cross-ref-валидатор) и этот реестр; плюс guard-тест **reads⊆refs consistency** (`shared/config/passage_test.go`, `TestStratify_ReadsEqRefsConsistency`), доказывающий, что множество passage-определяющих source-полей стратификатора совпадает с покрытым валидатором в этом классе (фикстура с ghost-register в каждом source-поле ловится ОБОИМИ: `StratifyUnknownRegister` И `unknown_register_reference`). Расхождение этих графов = silent-wrong-target (стратификатор не видит поле → молча неполный граф) ИЛИ дыра линтера (unknown-register доживает до рантайма вместо офлайн-отлова). **Flow-control (`when`/`changed_when`/`failed_when`) — намеренная асимметрия `refs ⊋ passage-reads`** (amend 2026-06-21): ∈ `collectRefs` (register-читающий, проверка существования), но ∉ `collectTaskReads` (НЕ passage-определяющий). Эту асимметрию фиксируют `TestStratify_FlowControlInRefsNotPassageReads` и `TestStratify_RegisterDependentWhenDoesNotSplitPassage` — регресс «вернуть flow-control в `collectTaskReads`» краснит их (Stratify снова начнёт расщеплять probe↔when-потребителя = FC-5).

**serial / run_once.** `serial:` — ось **хостов таргета** (волны по ≤N хостов, orchestration.md §2.2.1); Passage — ось **задач** (стратификация по register). Оси **ортогональны**, и теперь крутятся совместно (**2D serial×passage реализован**, S-2D1): внутри КАЖДОГО Passage применяется свой serial-барьер по хостам ЭТОГО Passage. Pilot-рестрикт `serial_staged_unsupported` **снят** (см. §S4 ниже). `run_once:` + staged **совместимы** без ограничений: run_once режет таргет Passage-задачи до первого по SID из УЖЕ резолвнутого (свежим register этого Passage) `on:`+`where:`-таргета — резолв происходит per-Passage в повторном Render, поэтому run_once видит реальный register, не пустой.

**Min-width — per-Passage, НЕ per-RUN (S-2D1 amend).** Ширина волны выводится из задач **одного Passage** (`effectiveSerialWidth` на `tasksForPassage`-фильтрованном срезе `plans`), а не по всему прогону. Probe-Passage без `serial:` едет **одной волной**, даже когда последующий Passage несёт `serial:1` — узкое окно одного Passage **НЕ просачивается** в чужой (устранён silent-wrong-width: per-RUN min-width дал бы destructive throttle probe-проходу). Min-width-инвариант (минимальная положительная ширина среди задач) сохранён, но область агрегации сужена до Passage. **В N=1 (один Passage) per-Passage и per-RUN совпадают бит-в-бит.** Реверс-guard: `TestUnit_EffectiveSerialWidth_PerPassageSlice` + integration `TestIntegration_SerialStaged_ProbeNotThrottled`.

**§S4 — границы пилота (реализованные ограничения).**

- **`serial:` + staged (N>1) → РЕАЛИЗОВАНО (рестрикт снят, S-2D1).** 2D-матрица `serial`×`passage` построена: Passage-цикл run.go исполняет каждый Passage по порядку, а `dispatchPassage` внутри бьёт хосты на serial-волны из задач ИМЕННО ЭТОГО Passage (`effectiveSerialWidth` на `tasksForPassage`-срезе — per-Passage width). Бывший fail-closed-гейт `serial_staged_unsupported` (run.go + soul-lint) **УДАЛЁН**; broken-фикстура `scenario-serial-staged` перепрофилирована в valid-golden (`passage_plan` HINT). Per-Passage barrier и fail-stop наследуются от serial-механики без изменений: падение хоста в волне Passage P → следующая волна и Passage P+1 не стартуют, state не коммитится (last known-good). **Crash-recovery-лимит staged-inline сохраняется** (serial+staged идёт тем же inline-путём, что serial без staged → Acolyte-reclaim его не покрывает, см. следующий буллет) — это **не новая регрессия**, а наследование общего свойства inline-пути. Guard-тесты: `TestIntegration_SerialStaged_RollingPerPassage` (rolling по волнам per-Passage), `_ProbeNotThrottled` (★ реверс per-passage width), `_FailStopInWave` (fail-stop + state не коммитнут), `_RegisterAcrossWaves` (register всех волн Passage P собран до Passage P+1).

- **staged-прогоны идут INLINE; crash-recovery — Voyage-путь ЗАКРЫТ orphan-lock-release, standalone (прямой run) пока НЕ покрыт.** Staged-прогон (N>1) при любом значении `AcolyteEnabled` исполняется СТАРЫМ inline-путём (run.go-гейт `!staged`): Acolyte рендерит per-host при claim ОДИН раз, а не per-Passage, поэтому Acolyte-claim для staged отключён. Следствие по recovery: правило Reaper-а `reclaim_apply_runs` (ADR-027 amend) сужено до `status='claimed'` с истёкшим lease — оно возвращает в очередь задания, умершие ДО отдачи Soul-у. Inline-путь `claimed` НЕ пишет (он делает `Insert(running)` → `SendApply` напрямую, attempt=0), поэтому `reclaim_apply_runs` его не подхватывает. **Voyage-путь (staged-инкарнация внутри реклеймнутого Voyage) ЗАКРЫТ фиксом orphan-lock-release** ([ADR-027 amend (l)](0027-apply-work-queue.md), 2026-06-21): реклеймнутый VoyageWorker детектит осиротевший `applying` по back-link `voyage_targets.apply_id` прошлого attempt и снимает lock (`applying→ready`, fenced) ПЕРЕД re-run — реклеймнутый Voyage больше не виснет на staged-инкарнации, перезапускает прогон с чистого старта (не per-Passage resume — весь scenario заново; идемпотентность модулей + gate-1 fencing держат корректность). **STANDALONE staged (прямой run без Voyage — `POST /v1/incarnations/{name}/run`) остаётся ОТКРЫТ**: у него нет Voyage-обёртки с back-link, поэтому detect-якоря orphan-шва нет — **падение keeper-инстанса посреди такого прогона не подхватывается**: строки остаются `running`, incarnation — `applying`, stage-loop не возобновляется. Это **НЕ молчаливая потеря/порча**: состояние наблюдаемо (incarnation висит в `applying`, apply_runs в `running`), требует операторского триажа/анлока — не silent-wrong-target и не рассинхрон `incarnation.state` (state коммитится строго ОДИН раз после последнего Passage, ADR-056 §г). Замечание о границе: это **общее свойство inline-пути** (serial-guard и не-Acolyte-инстансы тоже его не recover-ят при краше run-goroutine, BAG-1 `ensureTerminalApplyRun` срабатывает только при abort В ТОЙ ЖЕ goroutine, не при краше процесса), staged лишь наследует его. **Backstop-follow-up для standalone:** колонка-владелец incarnation-lock (epoch/kid на строке `incarnation`, как у Ward) ЛИБО per-passage Acolyte-claim + per-passage reclaim/fencing (ADR-027 amend на уровне `(apply_id, sid, passage)`) — **отложены**. Guard-тест: `TestIntegration_StagedInlineNotAcolyteClaim` (staged + `AcolyteEnabled=true` идёт inline — ни одной `planned`/`claimed` строки, Acolyte-путь `dispatchPlanned` не вызывается).

- **3-stage restart re-probe — доказан.** Топологическая стратификация + program-order ребро эмиттера дают `[0,1,1,1,2]` (Count=3) на restart-идиоме (probe → действие → re-probe → действие): re-probe (`redis_role_after`) попадает в Passage ПОСЛЕ failover-действия (program-order: эмиттер register не опережает предшествующие задачи), а потребитель re-probe — в Passage 2. Live-доказательство retargeting-а по СВЕЖЕМУ re-probe (а не первому probe): `TestIntegration_Staged3Passage_ReprobeRetargets` (после failover master меняется host-a→host-b; Passage-2 действие таргетится ТОЛЬКО на нового master по re-probe Passage 1).

**Отвергнутые альтернативы.**

- **Variant B (doc-only known-limitation).** Зафиксировать в orchestration.md, что probe→where не работает (known-limitation), и не реализовывать. **Отвергнут пользователем** в пользу реализации: probe→where — основа волатильной роли ([ADR-008](0008-coven-stable-tags.md)), канон уже её обещает, оставлять движок молча не исполняющим обещанное — недопустимый drift.
- **Фиксированный 2-stage** (ровно два прохода: probe-проход, затем действие-проход). Отвергнут: **не покрывает restart re-probe** (probe → действие → re-probe → действие — три+ уровня зависимости, например проверка роли до и после failover). N-stage (топологический) покрывает произвольную глубину register-зависимостей.

**Риски.**

- **silent-wrong-target при ошибке стратификации.** Если граф register-зависимостей разобран неверно (потребитель попал в тот же/более ранний Passage, чем его probe), `where:` снова отберёт по пустому/устаревшему register — разрушительная операция уйдёт не на те хосты молча. **Guard-тесты обязательны (TDD, S2)**: стратификация проверяется тестами «probe и потребитель не в одном Passage» до реализации stage-loop. Симметрия footgun-у silent-destructive-on-partial (orchestration.md §5).
- **Forward-compat.** Старый Soul под staged-сценарием обязан получить **explicit-reject** (`render_failed`), а не молча отработать одним проходом (вернуло бы исходный drift) и не зависнуть. Тестируется в S5.
- **Регресс barrier / Voyage / recovery.** Смена per-host задания на per-Passage задание (PG-PK, ~10 пакетов: barrier-чтение, Ward-claim, reclaim, `voyage_targets.apply_id` back-link, history-timeline, attempt-fencing) — широкая зона касания. Митигация — пилот-порядок S1→S5 (контракт+PG до стратификации, стратификация до stage-loop), аддитивная миграция, guard-тесты на каждом слайсе.
