# ADR-051. Operator API codegen: OpenAPI → Go types (oapi-codegen), types-only → strict

> **Status: SUPERSEDED [ADR-054](0054-openapi-code-first.md) (2026-06-12), implementation REMOVED (2026-06-13, HEAD `fde65bf`).** The premise was inverted: it was **OpenAPI → Go types** (spec-first, oapi-codegen), it became **Go types → OpenAPI** (code-first, huma v2). User decision. **Rollout complete, the oapi-codegen scaffold of this ADR is FULLY removed from the code:** the handwritten `openapi.yaml` as source, the package `keeper/internal/api/oapi/` (committed `types.gen.go`/`server.gen.go`), the `oapi_strict.go`/`strictServer` bridge, the oapi-codegen toolchain and the Makefile targets `gen-api`/`check-gen-api`/`sync-openapi` — all removed. The source of shape is now Go huma types (see [ADR-054](0054-openapi-code-first.md)). The text below describes the already-retired spec-first approach and remains only as historical context of the transition.

> **Historical note (before supersede): Phase 1 (`types-only`) COMPLETE (2026-06-09/10), in baseline.** User decisions + architect design locked and implemented. All ~21 wire handlers of the Operator API migrated to the generated types `keeper/internal/api/oapi` (oapi-codegen **v2.7.1**, pinned): gen-target, `types.gen.go`, rollout of alias/converters (categories A–D), removal of `proto/operator/v1`, downgrade of the spec to `3.0.3` — all done. Strict-goal invariants (1) single source of shape and (3) codegen idempotence (d) are closed. **Phase 2 (`types-only → strict`, invariant (2) — shape validation by codegen, slice S6) — is the next step** by explicit decision.

**Context.** [ADR-011](0011-go-layout.md#adr-011-go-code-layout-gowork-with-per-side-modules) made the package `proto/operator/v1/` (6 `.proto`, messages for 16 HTTP endpoints) the source of the Operator API **shape**. But the Operator API transport is **HTTP/JSON** ([ADR-004](0004-binaries.md#adr-004-binary-layout--keeper-soul-soul-lint-push-mode-as-a-module-inside-keeper)), not gRPC; there is no gRPC service over `operator/v1`. As a result `proto/operator/v1` turned out to be a **parasitic second model of the shape** alongside [`docs/keeper/openapi.yaml`](../keeper/openapi.yaml): the handwritten handler DTO types silently drift away from the spec (request-drift), while the spec is the only thing that actually goes on the wire and is read by the web side. What is needed is a **single source of shape** = the OpenAPI spec, from which the Go types are generated, not written by hand.

**Decision.**

**(a) Tool — oapi-codegen v2 (pinned), MVP mode `types-only`.** The generator is `oapi-codegen` v2 with a pinned version. The codegen config in the MVP is **models only** (`generate: { models: true }`): the spec produces Go request/reply types, without a server scaffold. The target package is **`keeper/internal/api/oapi`**, the file **`types.gen.go`** (committed, like proto-gen per [ADR-011](0011-go-layout.md#adr-011-go-code-layout-gowork-with-per-side-modules)). The Makefile gets a target **`gen-api`** (generation) + **`check-gen-api`** (idempotence check — parity with the existing `check-gen` for proto). Auto-`HandlerFromMux` in the MVP is **not used** (see (d)).

**(b) Source of truth of the shape — `docs/keeper/openapi.yaml`.** The shape (request/reply schemas, required, enum, format) lives in the OpenAPI spec; the Go types are its derivative. `proto/operator/v1` is **abolished** (amends [ADR-011](0011-go-layout.md#adr-011-go-code-layout-gowork-with-per-side-modules), see below). Semantics/mapping (endpoint ↔ MCP-tool ↔ permission, transport rules, error-format) are still held by [`docs/keeper/operator-api.md`](../keeper/operator-api.md) — this is markdown normalization, not a second model of the shape.

**(c) Integration pattern — type-alias to the gen-type + a domain→wire converter at the boundary.** The generated type is not edited by hand; handler code is bound to it via a **type-alias**, and the domain is translated into the wire form by a **converter at the boundary** (in the same place as serialization). Rollout categories by this distinction:
- **A — request types** (incoming body): a pure **alias** to the gen-type. The handler decodes directly into the gen struct.
- **B — reply with scalar fields** (flat strings/numbers/bool): a pure **alias**.
- **C — reply with typed-enum / pointer-optional, read by tests**: **a converter at the boundary, NOT an alias**. The domain struct stays separate (it is asserted by tests), and on output it is mapped into the gen-wire-type. An alias is forbidden here — it would pull the gen shape (pointers/enum wrappers) into the domain and into the tests.
- **D — reply field byte-passthrough JSONB** (domain field = `json.RawMessage`, raw bytes from PG returned as-is): in the spec the field is marked `x-go-type: json.RawMessage` → the field's gen-type becomes `json.RawMessage` → the domain is **aliased cleanly** (the field's gen shape matches the domain). `x-go-type-import` is **NOT added**: oapi-codegen always imports `encoding/json` itself (marshal code of the gen file), and an explicit import directive would produce a duplicate import `json "encoding/json"` next to the implicit one → `json redeclared` (the gen file does not compile). The `map`↔`raw` converter is FORBIDDEN: PG-JSONB canonicalization ≠ Go `map`-marshal (lexicographic key order) → unmarshal→map→marshal **reorders the keys INSIDE the passthrough value** = a wire-change of the field value itself (not the harmless order of members of the enclosing object — see the load-bearing invariant (f)), which violates "zero wire-change": for byte-passthrough JSONB the raw bytes of the value are the contract, and reshuffling them changes the value. Distinguish from an ordinary `type: object` map domain (pushprovider/profile — domain `map[string]any`, gen-`map` already matches, x-go-type is **NOT** applied there). The marker of D: the field domain is `json.RawMessage` from untouched PG-JSONB. For every D-field a **byte-exact guard test** is mandatory — a non-empty multi-key payload with a NON-lexicographic key order, asserting the raw bytes of the passthrough value (not the unmarshal-equivalent); the test catches a regression of reintroducing the `map`-converter. (Optionally a symmetric request input with a RawMessage domain is also x-go-typed — this removes the converter on input, additive, wire-neutral.)

**Pattern rules for the domain rollout (the developer applies on every domain):**
- **The previous serialization of `time.Time` is checked BEFORE migration** — a bare `time.Time.MarshalJSON` yields nanoseconds, such a domain is **NOT truncated** (see (f)); a previous `.Format(time.RFC3339)` yields seconds, truncate.
- **Codegen initialism set** — `AID` / `SID` / `JWT` / `SHA256` / `PEM` (`additional-initialisms` in `oapi-codegen.yaml`). The normalizer touches only the Go field name, not the json tag — therefore extending the set **does not change the wire**, it merely collapses domains whose only C-reason was a normalizer-mismatch (`Sha256`/`PubkeyPem`/`Jwt`) into a pure alias.
- **Justify category C ONLY by a real reason** — a typed-enum or pointer-optional, read by tests. A normalizer-mismatch is **not** a reason for C (it is removed by the initialism). If, after regen, the domain matches the gen-type field-for-field — **collapse it into an alias**.
- **The `omitempty` tag — by the spec.** The domain must match the gen-tag. A "domain ↔ gen" divergence is either a **domain bug** (we fix the domain to match the spec) or a **spec bug** (propose-and-wait to the architect). An intentional wire-change (e.g. sigil `revoked_at` null→omit to converge with the spec) is fixed by a guard test.

**(d) The `types-only → strict` path.** The target strict state is defined by **three invariants**: (1) the spec is the single source of shape (no handwritten DTO duplicates); (2) `required`/`enum`/`format` are enforced by codegen (strict-server middleware), not by manual checks; (3) codegen is idempotent. **The MVP stops at `types-only`** — this closes invariants 1 and 3; **invariant 2 (shape validation by codegen) — after an explicit decision** to move to strict, as a separate slice **S6** (not in this epic). Additionally the design fixes **three strict invariants** of the transition:
- **Ban on auto-`HandlerFromMux`.** Route mounting is **manual**: an auto-generated router would hang the handlers bypassing the per-route RBAC/Audit middleware (a security regression — loss of enforcement on the endpoint). When moving to a strict-server, routes are mounted manually on top of the strict wrappers.
- **Own error handlers.** The codegen's `RequestErrorHandlerFunc` / `ResponseErrorHandlerFunc` are redirected into **`problem.Write`** (a single RFC 7807 format, [operator-api.md → Error format](../keeper/operator-api.md#error-format-rfc-7807)) — otherwise the strict layer returns its own non-problem+json error format, diverging from the cluster's error contract.
- **Spec completeness for query.** For query parameters (keyset pagination, filters) the spec must explicitly set `style`/`explode` — otherwise the strict query binding diverges from the actual parsing.

**(e) OpenAPI version / nullable — downgrade of the spec to `3.0.3`** (user choice). The reason for the downgrade is that **oapi-codegen v2 does not support OpenAPI 3.1**: on the `3.1.0` header it prints a warning on every run. Declaring `3.1.0` when the codegen can only do 3.0 is a latent risk: any future 3.1-only construct in the spec would break silently. The spec content is **100% compatible with 3.0** (there are no 3.1-only features — no `$defs`, `type: [T, null]`, `webhooks`, `const`, `prefixItems`, `unevaluatedProperties`), so the downgrade is **safe and lossless**. (Empirically verified by an S1b marshal comparison: oapi-codegen v2.7.1 reads `nullable` **identically on 3.1.0 and 3.0.3** — `types.gen.go` is byte-for-byte identical before and after the downgrade; the version affects only the codegen warning, not the emit. The prior premise "3.1.0 broke oapi-codegen: it ignored `nullable` → Go omit" is **refuted**.)

**The nullable Go↔TS contract — structurally consistent via two classes** (requires neither the downgrade nor special handling in the rollout):
- **required + nullable** → Go emits `*T` **without** `omitempty` (the value `null` is present in the JSON) = TS `T | null`. Symmetry `null ↔ null`.
- **optional + nullable** → Go emits `*T` **with** `omitempty` (the field may be absent) = TS `T | null` with `?` (absent is allowed). Go-omit ↔ TS-optional.

There is no divergence for critical fields. **Invariant (already satisfied, proven empirically S1b):** the Go emit (oapi-codegen) and the TS emit (openapi-typescript on [soul-stack-web](#)) eat **one** spec and yield consistent null semantics. **The rollout does NOT need special modeling of nullable** — the existing shape is consistent by construction. The downgrade of the spec header itself is a parallel slice **S1b** (an edit to `openapi.yaml`), not this document.

**The load-bearing invariant of the migration — "zero wire-change" = SEMANTIC wire equivalence.** The type-only translation of the handlers must preserve the wire form, but "zero wire-change" is normalized **semantically**, not byte-wise: the same set of keys, the same values, the same `null`-vs-omit semantics of the field. **The order of members of a JSON object is NOT normalized** — the field-order may change during codegen (alias/converter-at-the-boundary maps into a gen struct with a different field order), and this is **acceptable**: the order of members of a JSON object is not semantic (RFC 8259), the web side and any correct parser read the object by keys. Where "byte-for-byte" / "byte-exact" appears below (date-time precision, category-D guard tests) — this is about the values and null-vs-omit of specific fields, not about the positional order of object members. The decision was fixed by the user (2026-06-09/10).

**(f) The date-time format on the wire — RFC3339 UTC, precision is PRESERVED PER DOMAIN.** The date-time precision on the wire **does not change during the type migration** — a load-bearing invariant: the type-only translation = zero changes on the wire (in the sense of semantic equivalence, see above — value and null-vs-omit, not the order of object members). The domain→wire converter (c) is chosen by the previous serialization of the domain:
- A domain with a previous **SECOND-level** wire (the format was `.Format(time.RFC3339)` or equivalent, the fractional part was already dropped) → `.UTC().Truncate(time.Second)` (operator / service / audit / sigil).
- A domain with a previous **NANOSECOND-level** wire (a bare `time.Time.MarshalJSON` → RFC3339Nano, e.g. choir `created_at`/`added_at`) → `.UTC()` **WITHOUT** `Truncate` (truncation would break byte-for-byte).

The global `Truncate(time.Second)` of the first revision of (f) is **WITHDRAWN** — it contradicted the "zero wire-change" invariant for nanosecond domains. Uniform second-level precision across the whole API is **a separate optional contract normalization** (an intentional wire-change), outside the rollout, by an explicit decision.

**(g) Required nested objects on `types-only`.** A nested object-field that is semantically required is **NOT marked** `required:` in the schema in `types-only` mode: oapi-codegen emits a required object as a value type (not a pointer), which breaks nil-presence detection, and `types-only` gives no auto-validation. The presence of such an object is enforced **manually** — a 422 `application/problem+json` in the handler. When moving to strict (d, invariant 2) `required` is enforced by codegen, and this manual pattern is removed.

**(h) Web synchronization — an operational invariant.** [soul-stack-web](#) vendors **the same** spec (`vendor/openapi/keeper.yaml`) and runs it through `openapi-typescript` → `types.gen.ts`. The core→web sync is **cross-repo** and is **not caught** by core `make check`: when editing `openapi.yaml` a manual vendor-update in web + `npm run gen:api` is needed. This is fixed as an operational invariant (a candidate for a CI-guard, **backlog**) — otherwise a third copy of the spec (web-vendor) silently falls behind.

**Rationale.**
- **A single source of shape.** The request/reply shape stops living in two places (proto + spec); the handwritten DTO duplicates are removed, drift between the handler and the spec becomes impossible at the type level.
- **Go↔TS consistency.** Both codegens (Go and TS) are fed one spec; the null semantics are symmetric by construction (e): required+nullable → `null ↔ null`, optional+nullable → Go-omit ↔ TS-optional. The wire contract converges without special handling — the downgrade to 3.0.3 (e) does not affect this (its reason is the absence of 3.1 support in oapi-codegen, not nullable).
- **Security does not suffer.** The manual mount + own error handlers (d) preserve per-route RBAC/Audit and a single problem+json — the codegen does not undermine the existing security invariants of the Operator API.

**Consequences.**
- **Removal of `proto/operator/v1`** (6 `.proto`) + its gen + editing the Makefile gen-target (it loses the operator-proto section) — as a separate slice (amends [ADR-011](0011-go-layout.md#adr-011-go-code-layout-gowork-with-per-side-modules)).
- **A new package** `keeper/internal/api/oapi/` with a committed `types.gen.go`; the Makefile targets `gen-api` / `check-gen-api`.
- **`types-only` gives no auto-validation** of the shape (required/enum/format) until the move to strict (d, S6) — in the MVP presence/validation are in places manual (g).
- **A third copy of the spec** (web-vendor) lives outside `make check` (h) — an operational risk of desync, a candidate for a CI-guard.
- **OpenAPI / MCP / operator-api.md** — the semantics and mapping stay in [operator-api.md](../keeper/operator-api.md); the source spec is [openapi.yaml](../keeper/openapi.yaml) (docs-writer during implementation).

**Rejected alternatives.**
- **(a) The spec header `3.1.0`.** oapi-codegen v2 does not support 3.1 — a warning on every run + a latent risk of a silent break on a future 3.1-only feature. Rejected — a downgrade to `3.0.3` (e). (The nullable contract did **not** depend on the header version — empirically S1b; this was not the reason for rejection.)
- **(b) Auto-`HandlerFromMux`.** Would hang the handlers bypassing per-route RBAC/Audit (a security regression). Rejected — a manual mount (d).
- **(c) Keeping `proto/operator/v1` as the shape.** A second model of the shape alongside the spec = request-drift; there is no gRPC service over it. Rejected (b).
- **(d) An alias on category-C reply types.** Would pull the gen shape (pointer/enum wrappers) into the domain and the tests. Rejected — a converter at the boundary (c).
- **(e) `map`-re-marshal of category-D passthrough fields.** Unpacking byte-passthrough JSONB (`json.RawMessage`) into `map[string]any` and marshaling back at the boundary reorders the keys (Go-`map`-marshal is lexicographic, PG-JSONB canonicalization is not) = a wire-change of key order, violates "zero wire-change" (f). Rejected — `x-go-type: json.RawMessage` + a pure alias, the bytes are returned as-is (category D in (c)).

**Slice plan S.** S1a — canon (this ADR + the ADR-011 amendment). S1b — downgrade `openapi.yaml` to `3.0.3` (in parallel). S2+ — gen-target `gen-api`/`check-gen-api` + the package `keeper/internal/api/oapi/` + `types.gen.go`. S3+ — rollout of alias (categories A/B) + converters-at-the-boundary (category C). S-removal — removal of `proto/operator/v1` + its gen + editing the Makefile. S6 (after an explicit decision) — the `types-only → strict` transition (validation by codegen, invariant d.2).

**Amendment (2026-06-10, S6: the strict-rollout pattern of Phase 2 — "bridge", full-strict only greenfield).** oapi-codegen v2.7.1 in strict mode decodes the body leniently (`json.NewDecoder` without `DisallowUnknownFields`) inside the generated `strictHandler`, with no hook on the decode method; `StrictHTTPServerOptions` controls only the error handlers. Therefore the requirement "unknown field → 400" under the "zero wire-change" invariant is incompatible with the typed `request.Body` of the strict layer. **The canonical rollout pattern for existing domains:** `bridgeMiddleware` buffers the body and puts `(w,r,body)` into the `ctx`; the aggregate's strict method `strictServer` does **NOT** use the typed `request.Body`/typed-response, but retrieves `(w,r)`, rewinds the Body and delegates to the existing domain `func(w,r)`-handler (it carries the `DisallowUnknownFields`-decode, presence validation, `problem.Write` and the exact wire form); the method returns a no-op `handledResponse`. **Compiler guarantees of the bridge:** completeness of the `StrictServerInterface` operations (you cannot forget an operation), path/query binding by the wrapper, unified problem-error-handlers; a panic-stub on unmounted methods = an explicit map of what is not migrated. **We deliberately do NOT get:** compiler enforcement of the typed input/output and the status code (this is held by the tests and by categories A–D of Phase 1). **Greenfield endpoints** (without a legacy handler and without the zero-wire invariant, e.g. `POST /v1/incarnations/{name}/rerun-create`) are written full-strict right away (typed `request.Body` + typed-response). The strictness "unknown field → 400" on greenfield is added by a manual pre-check of the buffered body (`bridgeMiddleware`-buffer + `DisallowUnknownFields`, helper `strictDecodeProbe`) BEFORE the domain orchestration — the typed oapi decode is lenient, but the "unknown→400" decision is preserved for greenfield endpoints too. Typed and bridge methods are **compatible in one aggregate per-method** — a gradual bridge→typed conversion by separate slices is allowed. A global full-strict conversion of legacy = a repeat of Phase 1 (re-wire + re-proof of date-time/omitempty per-domain) — **rejected by the user** as disproportionate; backlog on a concrete need. **Normative mount invariants of Phase 2** (were in a code comment, promoted to canon): (1) the ban on auto-`HandlerFromMux` — mount only manual per-route with preservation of the RBAC/Audit/Toll/Tempo hangings; (2) own `Request`/`ResponseErrorHandlerFunc` → `problem.Write`; (3) wire-semantic equivalence of the migration.

**Amendment (2026-06-10, S6: normalization of query bind errors to 400).** The "zero-wire-change" invariant is refined: a bad-value of a typed-query-parameter (`Format: date-time`, `Type: integer/boolean`) is normalized to **400 `TypeMalformedRequest`** across the whole Operator API (bind `ServerInterfaceWrapper` as the single source). The previous 422 `TypeValidationFailed` on a bad-value of such query parameters (`operators.revoked`, `audit.started_after`/`started_before`, `errand.started_after`) change to 400 — a deliberate wire-change of the error class of the bind phase (user decision). A bad-value of the body and the enum semantics of string parameters stay 422. The domain tests are updated.
