# ADR-043. Voyage — унифицированный батчевый прогон.

> **Статус: реализован, поглощение Tide/ErrandRun завершено (Wave 5, 2026-05-30).** Слайсы S1–S5 раскатаны; стадия S6-deprecate пропущена по решению Wave 5 (прямой drop без обратной совместимости); S7 выполнен — Tide ([ADR-040](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override)) и ErrandRun ([ADR-041](0041-errandrun.md#adr-041-errandrun--multi-target-обвязка-над-errand)) физически удалены (migration `061`/`062`, соответствующие пакеты / эндпоинты / proto-сообщения / audit-семейства). Scenario-батчи теперь идут через `kind=scenario`, multi-target command — через `kind=command`. Single Errand ([ADR-033](0033-errand.md#adr-033-errand--pull-ad-hoc-exec-вне-scenario), `POST /v1/souls/{sid}/exec`) сохранён как sugar.

**Контекст.** К 2026-05-29 в системе сосуществуют три семантически близких механизма «запустить работу на множестве хостов»:
- **scenario-run** (classic single-run, [ADR-009](0009-scenario-dsl.md#adr-009-scenario--полная-dsl-задач-destiny-граница-с-destiny--рекомендация)) — один прогон scenario на incarnation с внутренними волнами по хостам через `serial:`;
- **Tide** ([ADR-040](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override)) — invocation-time chunking того же scenario-run-а на последовательные Surge-волны + target/concurrency-override;
- **ErrandRun** ([ADR-041](0041-errandrun.md#adr-041-errandrun--multi-target-обвязка-над-errand)) — multi-target обвязка над N ad-hoc Errand.

Три сущности, три PG-таблицы (`tides` / `errand_runs` / `apply_runs`), три worker-pool-а (TideWorker / ErrandRunWorker / Acolyte), три набора audit-семейств (`tide.*` / `errand_run.*` / `apply.*`), три ветки в UI и CLI. По сути это **одна операция «батчевый прогон»** в двух режимах: «применить scenario к набору инкарнаций» и «выполнить команду на наборе хостов». Дублирование плодит диалекты, удорожает поддержку, ломает единый «All-runs» вид для оператора.

**Решение (зафиксировано пользователем 2026-05-29).** Ввести единую top-level сущность **Voyage** — унифицированный батчевый прогон, поглощающий Tide + ErrandRun + scenario-run. Единица батча — **Leg** (один «отрезок пути»). Метафора — продолжение морской линии Soul Stack (Tide / Surge): Voyage — «плавание», Leg — «переход» внутри него.

1. **Дискриминатор `kind`.** Voyage имеет два режима, заданных полем `kind`:
   - **`scenario`** — применить named scenario к набору **инкарнаций** (поглощает Tide + classic scenario-run);
   - **`command`** — выполнить whitelisted-модуль на наборе **хостов** (поглощает ErrandRun + single-SID Errand).

2. **Единица батча — Leg.** Прогон бьётся на последовательные Leg-и. Параметры прогона (invocation-level, не declaration-level):
   - `batch_size` — размер одного Leg (число единиц в батче);
   - `concurrency` — степень параллелизма внутри Leg;
   - `dry_run` — bool (для `command` — только модули с `PlanReadSafe`, parity [ADR-031](0031-scry-drift.md#adr-031-scry--drift-detection-declarative-dry-run-reconcile)/[ADR-033](0033-errand.md#adr-033-errand--pull-ad-hoc-exec-вне-scenario));
   - `schedule_at` — отложенный старт (nullable; при наличии прогон входит в новое состояние `scheduled`);
   - `inter_batch_interval` — пауза между Leg-ами (контролируемая выкатка);
   - `on_failure` — `abort` | `continue` (политика перехода к следующему Leg при провале).

3. **Подход B1 (зафиксирован) — что есть единица батча.**
   - **`kind=scenario`: батч = N ИНКАРНАЦИЙ.** Каждая инкарнация в Leg — **полноценный scenario-run** со своим **per-incarnation state-commit** и cross-host barrier ([ADR-009 §7](0009-scenario-dsl.md#adr-009-scenario--полная-dsl-задач-destiny-граница-с-destiny--рекомендация)). Внутри-инкарнационные волны по хостам остаются **отдельной осью** — через scenario `serial:` (declaration-level), не через Leg.
   - **B2 (двухуровневые волны по хостам внутри Leg) ОТВЕРГНУТ.** Если бы Leg резал scope по хостам поперёк нескольких инкарнаций, это ломало бы barrier/state-commit-инвариант [ADR-009 §7](0009-scenario-dsl.md#adr-009-scenario--полная-dsl-задач-destiny-граница-с-destiny--рекомендация): частичный коммит `incarnation.state` (часть хостов одной инкарнации в одном Leg, остальные — в следующем) запрещён. Любая попытка B2 либо нарушает этот инвариант, либо схлопывается обратно в B1 (Leg = целая инкарнация). Поэтому host-level chunking живёт только внутри одного scenario-run-а через `serial:`, не на Voyage-слое.
   - **`kind=command`: батч = N ХОСТОВ.** Команда выполняется на наборе хостов; `incarnation.state` **не трогается** (наследует state-инвариант [ADR-033](0033-errand.md#adr-033-errand--pull-ad-hoc-exec-вне-scenario)). Здесь Leg = подмножество хостов — host-level chunking легален, потому что state-barrier-а нет.

4. **Модель данных.** Имена таблиц эскиза (`runs`/`run_targets`) уточнены в S1 до **`voyages`/`voyage_targets`** (чтобы имя несло сущность Voyage/Leg, а не было общим `runs`; см. комментарий migration `059`). Таблицы заведены рядом со старыми (пилот не падает):
   - **`voyages`** — top-level прогон: `voyage_id` (TEXT PK, ULID), `kind` (`scenario`|`command`), `scenario_name` (для `scenario`) | `module` (для `command`), `input`, `target_resolved` (snapshot набора единиц от старта прогона — parity Tide `target_resolved_souls`), `batch_size`, `concurrency`, `dry_run`, `schedule_at`, `inter_batch_interval`, `on_failure`, `total_batches`, `current_batch_index`, `status` (+ новое значение **`scheduled`** для отложенного старта), claim+lease-поля для failover-parity ([ADR-027](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim): `claimed_by_kid` / `claim_expires_at` / `last_renewed_at` / `attempt`), `started_by_aid` (FK `operators(aid)`), тайминги, `summary` (jsonb). Batch-режимы и -стратегии (migration `064`/`065`): `batch_mode` (`barrier`|`window`), `batch_percent`, `fail_threshold`, `inter_unit_interval`, `require_alive`.
   - **`voyage_targets`** — единицы прогона: `voyage_id` (FK), `target_kind` (`incarnation`|`sid`), `target_id`, `batch_index`, `status`, `apply_id` (для `scenario` — back-link на `apply_runs`) | `errand_id` (для `command` — back-link на `errands`). Даёт единый **All-runs** вид с two-level drill: для `scenario` — инкарнации → их хосты (через `apply_runs`); для `command` — сразу хосты.

5. **Advanced target-override УДАЛЁН.** Tide-конструкция invocation-time AND-merge `target.coven[]` / `target.where` ([ADR-040](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override) пункт 1, инвариант AND-merge) в Voyage **не переносится**. Фильтрация целей делается на **шаге ВЫБОРА цели** (оператор выбирает набор из уже-разрешённого RBAC-скоупа). Security-смысл «клиент не может расширить scope» **сохраняется** — но не как invocation-параметр, а как «выбор из уже-разрешённого набора»: оператор не может выбрать то, что вне его permission-скоупа. Это устраняет invocation-time AND-merge resolver и связанные error-codes (`tide_target_override_invalid`).

6. **RBAC разделён по `kind` (security-критичный guard).** Единый эндпоинт `/v1/voyages` выбирает permission по `kind` из тела запроса:
   - `kind=scenario` → переиспользует **`incarnation.run`** (тот же permission, что classic scenario-run / Tide — это та же операция);
   - `kind=command` → переиспользует **`errand.run`** (наследует whitelist [ADR-033](0033-errand.md#adr-033-errand--pull-ad-hoc-exec-вне-scenario)).
   - Guard выбора permission по `kind` — **fail-closed** и security-критичен: ошибка маппинга = privilege escalation (scenario-операция под command-permission или наоборот). Selector-семантика (`coven=` / `service=` / `host=`) наследуется из соответствующего permission-пространства без изменений.

7. **Audit — НОВЫЕ семейства** `scenario_run.*` (для `kind=scenario`) и `command_run.*` (для `kind=command`). Они **семантически заменяют** `tide.*` (→ `scenario_run.*`) и `errand_run.*` (→ `command_run.*`). Downstream audit-консьюмеры (UI audit-парсер в [`soul-stack-web`](0035-distribution-split.md#adr-035-distribution-split--core-apicli-vs-web-ui), [ADR-042](0042-backend-driven-ui.md#adr-042-backend-driven-dynamic-data-в-ui--ui-не-хардкодит-динамические-каталоги)) правятся **синхронно** при реализации соответствующего слайса (S2/S3). Имена зарегистрированы в [naming-rules.md → Audit-events](../naming-rules.md#audit-events).

8. **Инварианты, унаследованные от поглощённых сущностей** (переезжают в Voyage без ослабления):
   - **Per-incarnation state-commit (`kind=scenario`)** = parity per-Surge state-commit [ADR-040](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override) = parity barrier [ADR-009 §7](0009-scenario-dsl.md#adr-009-scenario--полная-dsl-задач-destiny-граница-с-destiny--рекомендация). Коммит `incarnation.state` — только после успешного cross-host barrier этой инкарнации; частичный коммит запрещён (B1).
   - **State-инвариант `command`-kind** — не мутирует `incarnation.state` ([ADR-033](0033-errand.md#adr-033-errand--pull-ad-hoc-exec-вне-scenario)).
   - **Failover claim+lease** — parity Acolyte-style PG-lease ([ADR-027(d)](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim) / [ADR-040 amendment](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override)): любой Keeper-инстанс подбирает осиротевший Voyage через PG-claim + TTL-lease, fencing-epoch через `attempt++`.

   **Default-политика recovery-правила `reclaim_voyages` — ВКЛЮЧЕНО через path-defaulting.** `reclaim_voyages` (механизм failover выше) исполняется по умолчанию, без обязательной записи в конфиге. Механизм — **path-defaulting в `reaper.dispatch`**: отсутствие ключа `reclaim_voyages` в `reaper.rules` ЛИБО явный `enabled: true` → правило РАБОТАЕТ (ON); выключить можно только явным `enabled: false`. Это **обратный дефолт** относительно остальных reaper-правил, которые map-driven (отсутствие ключа в `reaper.rules` = OFF, требуют явного включения); `reclaim_voyages` — особое recovery-правило, для которого «по умолчанию» значит «работает».
   - **(a) Почему default-ON.** Целевой масштаб — до 100k Souls; Keeper — горизонтально масштабируемый stateless-кластер ([ADR-002](0002-transport-grpc-ha.md#adr-002-транспорт-keeper--souls--grpc-bidirectional-stream-поверх-mtls-ha-кластер-keeper)), где рестарт/замена инстанса (деплой, OOM, авто-скейл) — штатное, регулярное событие, а не авария. Если правило выключено, осиротевший Voyage (владелец умер до финализации) висит в `running` без живого владельца до ручного вмешательства — неприемлемо для штатной эксплуатации. Поэтому recovery не должен зависеть от того, вписал ли оператор правило в `reaper.rules`: path-defaulting в `reaper.dispatch` поднимает `reclaim_voyages` при отсутствии ключа, оператор отключает его только явным `reaper.rules.reclaim_voyages.enabled: false`. (`reclaim_apply_runs` остаётся default-OFF, map-driven — у него иной риск: stale RunResult после отдачи прогона Soul-у, поэтому только явный opt-in.)
   - **(b) Безопасность default-ON — exactly-once на top-level Voyage.** Финализация опирается на CAS-ownership-guard: `voyage.Finalize` (`keeper/internal/voyage/crud.go`) пишет терминальный статус под условием `WHERE voyage_id = $1 AND claimed_by_kid = $2`. Если «мёртвый» воркер ожил уже после reclaim (claim перехвачен другим Keeper-ом, `attempt++`), его commit не совпадает по владельцу — UPDATE задевает 0 строк, caller ловит `ErrLeaseLost` и НЕ применяет результат. Дубль-commit одного Voyage двумя инстансами невозможен даже при ложном reclaim живого владельца — exactly-once на уровне top-level Voyage сохраняется.
   - **(c) Требование к настройке lease.** Интервал renew (`last_renewed_at`) ОБЯЗАН быть заметно короче lease-TTL (`claim_expires_at`): иначе живой, но временно подтормозивший воркер теряет lease и провоцирует ложный reclaim (повторный спавн Leg-ов). Короткий renew относительно TTL сужает окно двойного спавна до минимума; exactly-once на самом top-level Voyage при этом гарантирован CAS-guard-ом (b) независимо от настройки. Конкретные значения — в docs/keeper/reaper.md.
   - **Snapshot-scope** — набор единиц прогона фиксируется на старте (`voyages.target_resolved`), не «дрожит» от добавления/удаления Soul-ов между Leg-ами (parity Tide `target_resolved_souls`).
   - **Cross-incarnation легален на Voyage-слое** (`kind=scenario` оркестрирует N инкарнаций) — снятие запрета [ADR-008](0008-coven-stable-tags.md#adr-008-coven--только-стабильные-логические-теги) на верхнем слое; внутри одного scenario-run-а / scenario-resolver-а запрет «одна инкарнация» **сохраняется** (см. amendment ADR-008 ниже).

   - **(d) Идемпотентность `command`-Voyage и at-least-once.** (1) `command`-Voyage (`kind=command`) при failover имеет **at-least-once**-семантику исполнения per-target — как и весь Voyage-слой (см. failover-модель выше); exactly-once гарантируется CAS-guard-ом (b) только на финализации **top-level** Voyage, не на каждом спавне дочернего Errand. (2) Спавн дочернего Errand **фенсится через ctx-cancel**: `renewLoop` воркера (`keeper/internal/voyageorch/worker.go`) при потере lease (`ErrLeaseLost` от `RenewVoyageLease` — CAS `WHERE claim_by_kid=$2 AND attempt=$3` в `keeper/internal/voyage/claim.go`) вызывает `cancelLease`, и воркер, потерявший lease, **прекращает спавн** новых Errand-ов. (3) Поэтому остаточное окно дубля — **патологическое**: воркер должен зависнуть дольше lease-TTL, потом ожить и успеть заспавнить Errand **до** того, как поймает отмену контекста. Штатный failover (мёртвый воркер не оживает) дубля Errand **не даёт**. (4) **Рекомендация оператору:** command-модули whitelist-а (`core.cmd.shell` / `core.exec.run`, пункт 9) для multi-target `command`-Voyage желательно делать **идемпотентными** — оператор учитывает at-least-once-семантику при выборе модуля и формулировке команды. (5) `kind=scenario`-Voyage этим **не затронут**: его per-incarnation state-commit идёт через single-winner barrier ([ADR-009 §7](0009-scenario-dsl.md#adr-009-scenario--полная-dsl-задач-destiny-граница-с-destiny--рекомендация)), который отсекает дубль на уровне коммита состояния.

9. **Whitelist `command`-kind** — тот же, что [ADR-033](0033-errand.md#adr-033-errand--pull-ad-hoc-exec-вне-scenario): жёсткий список (`core.cmd.shell` / `core.exec.run`) + marker `ErrandReadSafe`. Single-SID `/v1/souls/{sid}/exec` остаётся sugar (см. amendment ADR-033 ниже).

**Отвергнутые альтернативы.**
- **(а) Оставить три сущности.** Дублирование диалектов, тройной UI/CLI/audit, нет единого All-runs. Решение пользователя — объединить.
- **(б) B2 (двухуровневые волны по хостам внутри Leg для `scenario`).** Ломает barrier/state-commit [ADR-009 §7](0009-scenario-dsl.md#adr-009-scenario--полная-dsl-задач-destiny-граница-с-destiny--рекомендация) (частичный коммит state). См. пункт 3.
- **(в) Сохранить invocation-time target-override.** Заменён выбором-из-разрешённого-набора (пункт 5) — тот же security-инвариант без resolver-сложности.
- **(г) Единый permission на оба `kind`.** Стёр бы границу scenario↔command в RBAC (privilege escalation). Permission выбирается по `kind` (пункт 6).

**Что отложено (post-MVP, как и в поглощённых ADR).**
- Cancel mid-flight (running-cancel через `DELETE /v1/voyages/{id}`) + cancel-policy — наследует deferred Tide/ErrandRun (pending/scheduled-cancel реализован, S5).
- SSE-progress единого Voyage-вида — после стабилизации core-orchestrator-а.
- Per-SID state-commit внутри упавшего Leg (`kind=scenario`, `on_failure: continue`) — наследует deferred [ADR-040](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override).
- Audit-log scaling массовых Voyage-прогонов — backlog [Открытые вопросы §26](../architecture.md#открытые-вопросы).
- per-coven concurrency sub-pools (`command`-kind) — наследует deferred [ADR-041](0041-errandrun.md#adr-041-errandrun--multi-target-обвязка-над-errand).

**Amendment (2026-05-29, ось Choir ортогональна Leg).** [ADR-044](0044-choir.md#adr-044-choir--именованная-топология-хостов-внутри-инкарнации) вводит **Choir** — именованную топологию хостов ВНУТРИ инкарнации. Это **внутри-инкарнационная** ось и она **не конфликтует с Leg**: для `kind=scenario` Leg = N **инкарнаций** (батч-ось верхнего слоя, B1), а Choir живёт внутри одной инкарнации (host-level позиция). Сужение по Choir — через scenario `where:` ([ADR-044](0044-choir.md#adr-044-choir--именованная-топология-хостов-внутри-инкарнации) пункт 5), а не через Voyage-параметры; Leg-разбиение и Choir-таргетинг — независимые оси.

**Amendment (2026-06-01, два batch-mode + батч-стратегии Salt-уровня — решение пользователя).** Voyage получает второй режим батчинга и набор additive-ключей parity Salt/Ansible. Всё — **forward-compat additive** (новые опт-поля payload, новый config-ключ; существующие прогоны без них работают как раньше — [ADR-012 forward-compat only-add](0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add)). Контракт `kind`/Leg/B1/§7 **не ослаблен**.

1. **Дискриминатор `batch_mode`: `barrier` | `window`.** Новое опт-поле invocation-payload (default `barrier` — текущее реализованное поведение).
   - **`barrier`** (реализован, пункт 2 выше): прогон бьётся на последовательные Leg-и (пачка `batch_size` единиц), между Leg-ами — барьер (дождаться терминала всех единиц Leg-а, опционально `inter_batch_interval`). `concurrency` = параллелизм **внутри** одного Leg, `batch_size` = размер пачки.
   - **`window`** (новый): полное скользящее окно — пул воркеров тянет единицы из общей очереди прогона, вернулся один → запускается следующий, постоянно держится `concurrency` активных. **Барьеров между пачками нет.** В window-режиме `batch_size` **НЕ используется** (нет пачек); ширину окна задаёт реюз существующего `concurrency` («держать N активных»). `batch_size` в payload с `batch_mode: window` → `422 validation-failed`.
   - **Окно по хостам — только `kind=command`.** Для `kind=command` window-окно скользит **по хостам** (state не трогается — host-level chunking легален, пункт 3). Для `kind=scenario` window-окно скользит **по инкарнациям**: внутри каждой инкарнации host-level cross-host barrier и per-incarnation state-commit **сохраняются без изменений** (B1, [ADR-009 §7](0009-scenario-dsl.md#adr-009-scenario--полная-dsl-задач-destiny-граница-с-destiny--рекомендация) не нарушается — окно лишь убирает барьер **между инкарнациями**, не внутри). B2 (окно по хостам поперёк инкарнаций) остаётся ОТВЕРГНУТЫМ по той же причине, что в пункте 3.

2. **`batch_percent` (новый опт int, 1..100).** Размер пачки как % от резолвнутого scope (parity Salt `-b 25%`), **взаимоисключающий** с `batch_size`. Аддитивен: `batch_size` остаётся `*int` (НЕ union-тип) — резолвер при наличии `batch_percent` вычисляет эффективный размер пачки от `len(target_resolved)`. Инвариант: **ровно один** из `{batch_size, batch_percent}` (оба заданы → `422`); в `batch_mode: window` оба не используются. Применим только к `batch_mode: barrier`.

3. **`fail_threshold: N` (новый опт int, request-level).** Порог абсолютного числа провалов: накоплено `N` провалившихся единиц → прогон останавливается (новые Leg-и / новые единицы окна не стартуют). **Соотношение с `on_failure`:** текущий `on_failure: abort` ≡ `fail_threshold: 1` (один провал → стоп); `on_failure: continue` ≡ отсутствие порога (`fail_threshold` не задан → бежать до конца). `fail_threshold: N` (N>1) — промежуточная толерантность. Работает в обоих `batch_mode`.
   > **Изменено amendment-ом 2026-06-09:** прежняя формулировка «В MVP порог **абсолютный** (не процент)» **снята**. Порог теперь выражается строковым полем `max_failures` (`N` абсолют | `N%` процент); `fail_threshold: N` (int) остаётся принимаемым, но **deprecated**. Полная семантика порога (единицы прогона по `kind`, `cancelled`/`no_match` не в счёт, `%` → `ceil(scope·pct/100)`) — в amendment-е 2026-06-09 ниже.

4. **`inter_unit_interval` (новый опт duration).** Per-unit пауза в `batch_mode: window` (parity к `inter_batch_interval`, который пауза **между Leg-ами** в `barrier`): задержка перед стартом каждой следующей единицы окна — мягкий throttle при скользящем окне. Применим только к `batch_mode: window`; в `barrier` не используется (там throttle = `inter_batch_interval`).

5. **`require_alive` (новый опт bool, default `false`).** Presence-фильтр живых: при `true` scope-резолв дополнительно отсекает Soul-ы без живого presence-lease (`SoulLeaseChecker` подключается к резолву scope, presence-lease — [ADR-006 Redis](0006-cache-redis.md#adr-006-кэш-и-координация--redis)). Снимок scope после фильтра фиксируется в `voyage.target_resolved` как обычно (snapshot-инвариант пункта 8 не ослаблен — фильтр применяется **на резолве**, не «дрожит» в процессе).

6. **`max_batch_size` (новый config-ключ Keeper, рядом с `voyage.max_scope`).** Оператор-инвариант — верхний предел размера батча/окна (`batch_size` для `barrier`, `concurrency` для `window`). Fail-closed: request с `batch_size` или `concurrency` > `max_batch_size` отвергается `422` с error-code **`voyage_batch_size_too_large`** (parity к `voyage_scope_too_large` — DoS-guard, ADR-002 масштаб 100k). Симметричен `voyage.max_scope`: дефолт + `0` = «без предела», отрицательное → `value_out_of_range`.

7. **Семантика `voyage_targets.batch_index` для двух режимов.** Колонка `voyage_targets.batch_index` (= Leg, к которому принадлежит единица):
   - **`batch_mode: barrier`** — индекс Leg-а единицы (0-based), как сейчас (Leg = пачка `batch_size`). Two-level drill в UI группирует по `batch_index`.
   - **`batch_mode: window`** — **`batch_index = 0` для всех единиц** (в окне нет пачек, прогон плоский). UI drill в window-режиме показывает плоский список единиц, без группировки по Leg-ам.

8. **Таблица «какой ключ в каком режиме».**

| Ключ | `batch_mode: barrier` | `batch_mode: window` | Уровень |
|---|---|---|---|
| `batch_size` | размер пачки (Leg) | **не используется** (`422` если задан) | request |
| `batch_percent` | % от scope (XOR с `batch_size`) | **не используется** | request |
| `concurrency` | параллелизм **внутри** Leg | **ширина окна** (держать N активных) | request |
| `inter_batch_interval` | пауза между Leg-ами | не используется | request |
| `inter_unit_interval` | не используется | per-unit throttle | request |
| `fail_threshold` | порог провалов → стоп (`abort` ≡ `1`) | порог провалов → стоп | request |
| `on_failure` | `abort`(=`fail_threshold:1`) \| `continue` | `abort` \| `continue` | request |
| `require_alive` | presence-фильтр на резолве scope | presence-фильтр на резолве scope | request |
| `dry_run` / `schedule_at` | как пункт 2 | как пункт 2 | request |
| `voyage.max_scope` | потолок резолвнутого scope | потолок резолвнутого scope | config |
| `max_batch_size` | потолок `batch_size` | потолок `concurrency` (= ширины окна) | config |

**Инварианты, которые НЕ затронуты:** B1 / per-incarnation state-commit / [ADR-009 §7](0009-scenario-dsl.md#adr-009-scenario--полная-dsl-задач-destiny-граница-с-destiny--рекомендация) cross-host barrier (window для scenario убирает барьер только **между** инкарнациями, внутри инкарнации barrier+commit сохранены); B2 остаётся отвергнут; snapshot-scope (пункт 8 исходного ADR); failover claim+lease; permission-по-`kind`. Реализация — отдельным слайсом (этот amendment — только фиксация дизайна).

**Amendment (2026-06-09, строковые batch-поля `batch`/`max_failures` + preview-эндпоинт — решение пользователя).** Два прежних int-поля (`batch_size`/`batch_percent` — размер пачки; `fail_threshold` — порог провалов) объединяются в **строковые** поля, удобные для CLI и для процента. Всё — **forward-compat additive без v2 и без миграций БД**: новое строковое поле на резолве **распадается на существующие колонки** (`batch_size`/`batch_percent`/`fail_threshold` в таблице `voyages`, migration `064`/`065`), старые int-поля остаются принимаемыми, контракт `kind`/Leg/B1/§7 не ослаблен.

1. **`batch` (новое строковое поле, request-level).** Единое поле размера пачки: `N` = абсолютное число хостов, `N%` = процент (1–100) от резолвнутого scope. **Парсит Keeper** (request-валидация): после `trim` строка матчится грамматикой `^(\d+)%?$`; невалид (пустой `N`, `0`, `%` без числа, мусор, `>100` для процента) — **fail-closed** → `422 validation-failed`. `N%` → эффективный размер пачки = `ceil(scope·pct/100)` от `len(target_resolved)` (естественно ∈ [1, scope]); `N` → абсолют (далее ограничен `max_batch_size`). Мотив — единое поле вместо пары `batch_size`/`batch_percent` + удобный CLI (`--batch 20%`). Применим только к `batch_mode: barrier` (в `window` пачек нет — пункты 1–2 amendment-а 2026-06-01).

2. **`max_failures` (новое строковое поле, request-level).** Порог провалов строкой: `N` (абсолют) | `N%` (процент). **Считается по единицам прогона** — для `kind=scenario` единица = **инкарнация**, для `kind=command` = **хост** (parity Leg). В счёт идут только провалившиеся единицы; `cancelled` и `no_match` (единица отрезана `where:`/presence-фильтром — для неё нечего исполнять) **НЕ считаются провалами**. `N%` → `ceil(scope·pct/100)` (естественно ∈ [1, scope]). Замещает `fail_threshold` (тот остаётся принимаемым, deprecated). Соотношение с `on_failure` — без изменений (`abort` ≡ `max_failures: 1`, `continue` ≡ порог не задан). Работает в обоих `batch_mode`.

3. **A1-сосуществование (additive, без breaking).** Старые поля `batch_size` (`*int`), `batch_percent` (`*int`), `fail_threshold` (`*int`) **остаются принимаемыми** и помечаются **deprecated**. Смешение нового строкового и старого int-формата в одном request-е (`batch` вместе с `batch_size`/`batch_percent`, либо `max_failures` вместе с `fail_threshold`) → **`422`** с error-code **`voyage_batch_spec_conflict`**. Без union-типов, без миграций БД: на резолве `batch` → (`batch_size` либо `batch_percent`), `max_failures` → `fail_threshold` (с расширением на процент). Forward-compat **без `v2`** (правило [ADR-012 only-add](0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add)): старые клиенты/прогоны работают как раньше.

4. **`POST /v1/voyages/preview` (новый эндпоинт).** Dry-resolve scope **без создания Voyage**: принимает тот же таргет-фрагмент + batch-настройки, что `POST /v1/voyages`, и возвращает `{scope_size, total_batches, effective_batch_size}`. **Не раскрывает SID-список** (только числа — для предпоказа числа батчей). Реюзит резолверы scope в **read-only** (без INSERT в `voyages`/`voyage_targets`), потолок `voyage.max_scope` и **RBAC-by-`kind`** (`incarnation.run` для scenario / `errand.run` для command — тот же guard, что у create). **Tempo rate-limit — ОТДЕЛЬНЫЙ bucket `voyage_preview`** ([ADR-050 amendment 2026-06-17](0050-tempo.md#adr-050-tempo--per-aid-rate-limiting-write-api), дефолт `30/60`, мягче create `10/20`): preview read-like по эффекту, но resolver-heavy по стоимости → собственный, более широкий лимит, **не делит квоту с `voyage_create`**. ~~Реюзит тот же rate-limit, что create~~ (отменено amendment-ом 2026-06-17). Назначение — предпоказ для **late-binding** таргета (`coven`/`require_alive`, где число хостов резолвит Keeper); для snapshot-таргета (явный `incarnations[]`/`sids[]`) число батчей считает клиент сам, эндпоинт не обязателен.

**Инварианты, которые НЕ затронуты amendment-ом 2026-06-09:** все из amendment-а 2026-06-01 (B1 / per-incarnation state-commit / §7 cross-host barrier / B2-отказ / snapshot-scope / failover claim+lease / permission-по-`kind`); `batch_mode` `barrier`|`window`, `window`/`max_batch_size`/`inter_unit_interval`/`require_alive` — без изменений; колонки таблицы `voyages` — без изменений (новые поля распадаются на существующие). Реализация — отдельным слайсом S1 (этот amendment — фиксация дизайна).
