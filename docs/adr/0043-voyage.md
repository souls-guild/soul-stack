# ADR-043. Voyage — unified batch run.

> **Status: implemented, absorption of Tide/ErrandRun completed (Wave 5, 2026-05-30).** Slices S1-S5 rolled out; the S6-deprecate stage was skipped by Wave 5 decision (direct drop without backward compatibility); S7 completed — Tide ([ADR-040](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override)) and ErrandRun ([ADR-041](0041-errandrun.md#adr-041-errandrun--multi-target-wrapper-over-errand)) physically removed (migration `061`/`062`, corresponding packages / endpoints / proto messages / audit families). Scenario batches now go through `kind=scenario`, multi-target command — through `kind=command`. Single Errand ([ADR-033](0033-errand.md#adr-033-errand--pull-ad-hoc-exec-outside-a-scenario), `POST /v1/souls/{sid}/exec`) is kept as sugar.

**Context.** As of 2026-05-29 the system has three semantically close mechanisms for "run work on a set of hosts":
- **scenario-run** (classic single-run, [ADR-009](0009-scenario-dsl.md#adr-009-scenario--the-full-destiny-task-dsl-the-boundary-with-destiny-is-a-recommendation)) — one scenario run on an incarnation with internal per-host waves via `serial:`;
- **Tide** ([ADR-040](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override)) — invocation-time chunking of that same scenario-run into sequential Surge waves + target/concurrency override;
- **ErrandRun** ([ADR-041](0041-errandrun.md#adr-041-errandrun--multi-target-wrapper-over-errand)) — multi-target wrapper over N ad-hoc Errands.

Three entities, three PG tables (`tides` / `errand_runs` / `apply_runs`), three worker pools (TideWorker / ErrandRunWorker / Acolyte), three sets of audit families (`tide.*` / `errand_run.*` / `apply.*`), three branches in UI and CLI. In essence this is **one operation "batch run"** in two modes: "apply scenario to a set of incarnations" and "execute a command on a set of hosts". The duplication breeds dialects, raises maintenance cost, and breaks a unified "All-runs" view for the operator.

**Decision (fixed by the user 2026-05-29).** Introduce a single top-level entity **Voyage** — a unified batch run absorbing Tide + ErrandRun + scenario-run. The unit of batching is the **Leg** (one "leg of the journey"). The metaphor continues the Soul Stack maritime line (Tide / Surge): Voyage — "voyage", Leg — "leg" within it.

1. **Discriminator `kind`.** Voyage has two modes, set by the `kind` field:
   - **`scenario`** — apply a named scenario to a set of **incarnations** (absorbs Tide + classic scenario-run);
   - **`command`** — execute a whitelisted module on a set of **hosts** (absorbs ErrandRun + single-SID Errand).

2. **Unit of batching — Leg.** A run is split into sequential Legs. Run parameters (invocation-level, not declaration-level):
   - `batch_size` — size of one Leg (number of units in the batch);
   - `concurrency` — degree of parallelism within a Leg;
   - `dry_run` — bool (for `command` — only modules with `PlanReadSafe`, parity with [ADR-031](0031-scry-drift.md#adr-031-scry--drift-detection-declarative-dry-run-reconcile)/[ADR-033](0033-errand.md#adr-033-errand--pull-ad-hoc-exec-outside-a-scenario));
   - `schedule_at` — delayed start (nullable; when present the run enters the new `scheduled` state);
   - `inter_batch_interval` — pause between Legs (controlled rollout);
   - `on_failure` — `abort` | `continue` (policy for transitioning to the next Leg on failure).

3. **Approach B1 (fixed) — what the unit of batching is.**
   - **`kind=scenario`: batch = N INCARNATIONS.** Each incarnation within a Leg is a **full scenario-run** with its own **per-incarnation state-commit** and cross-host barrier ([ADR-009 §7](0009-scenario-dsl.md#adr-009-scenario--the-full-destiny-task-dsl-the-boundary-with-destiny-is-a-recommendation)). Intra-incarnation waves across hosts remain a **separate axis** — via scenario `serial:` (declaration-level), not via the Leg.
   - **B2 (two-level host waves within a Leg) REJECTED.** If a Leg were to cut scope by hosts across multiple incarnations, this would break the barrier/state-commit invariant [ADR-009 §7](0009-scenario-dsl.md#adr-009-scenario--the-full-destiny-task-dsl-the-boundary-with-destiny-is-a-recommendation): partial commit of `incarnation.state` (some hosts of one incarnation in one Leg, the rest in the next) is prohibited. Any attempt at B2 either violates this invariant or collapses back into B1 (Leg = whole incarnation). Therefore host-level chunking lives only within a single scenario-run via `serial:`, not at the Voyage layer.
   - **`kind=command`: batch = N HOSTS.** The command runs on a set of hosts; `incarnation.state` is **not touched** (inherits the state invariant from [ADR-033](0033-errand.md#adr-033-errand--pull-ad-hoc-exec-outside-a-scenario)). Here Leg = a subset of hosts — host-level chunking is legal because there is no state barrier.

4. **Data model.** The sketch table names (`runs`/`run_targets`) were refined during S1 to **`voyages`/`voyage_targets`** (so the name carries the Voyage/Leg entity, not a generic `runs`; see the migration `059` comment). Tables were created alongside the old ones (the pilot doesn't break anything):
   - **`voyages`** — top-level run: `voyage_id` (TEXT PK, ULID), `kind` (`scenario`|`command`), `scenario_name` (for `scenario`) | `module` (for `command`), `input`, `target_resolved` (snapshot of the unit set from run start — parity with Tide `target_resolved_souls`), `batch_size`, `concurrency`, `dry_run`, `schedule_at`, `inter_batch_interval`, `on_failure`, `total_batches`, `current_batch_index`, `status` (+ new value **`scheduled`** for delayed start), claim+lease fields for failover parity ([ADR-027](0027-apply-work-queue.md#adr-027-apply-execution-model--work-queue--claim-acolyte-pool-ward-claim): `claimed_by_kid` / `claim_expires_at` / `last_renewed_at` / `attempt`), `started_by_aid` (FK `operators(aid)`), timings, `summary` (jsonb). Batch modes and strategies (migration `064`/`065`): `batch_mode` (`barrier`|`window`), `batch_percent`, `fail_threshold`, `inter_unit_interval`, `require_alive`.
   - **`voyage_targets`** — run units: `voyage_id` (FK), `target_kind` (`incarnation`|`sid`), `target_id`, `batch_index`, `status`, `apply_id` (for `scenario` — back-link to `apply_runs`) | `errand_id` (for `command` — back-link to `errands`). Gives a unified **All-runs** view with two-level drill: for `scenario` — incarnations → their hosts (via `apply_runs`); for `command` — hosts directly.

5. **Advanced target-override REMOVED.** The Tide construct of invocation-time AND-merge `target.coven[]` / `target.where` ([ADR-040](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override) item 1, the AND-merge invariant) is **not carried over** into Voyage. Filtering of targets is done at the **target-SELECTION step** (the operator selects a set from an already-RBAC-resolved scope). The security meaning "the client cannot expand scope" is **preserved** — but not as an invocation parameter, rather as "selection from an already-permitted set": the operator cannot select something outside their permission scope. This eliminates the invocation-time AND-merge resolver and the related error codes (`tide_target_override_invalid`).

6. **RBAC split by `kind` (security-critical guard).** The single endpoint `/v1/voyages` selects the permission by `kind` from the request body:
   - `kind=scenario` → reuses **`incarnation.run`** (the same permission as classic scenario-run / Tide — it's the same operation);
   - `kind=command` → reuses **`errand.run`** (inherits the whitelist from [ADR-033](0033-errand.md#adr-033-errand--pull-ad-hoc-exec-outside-a-scenario)).
   - The guard selecting permission by `kind` is **fail-closed** and security-critical: a mapping error = privilege escalation (a scenario operation under command permission or vice versa). Selector semantics (`coven=` / `service=` / `host=`) is inherited from the corresponding permission namespace without changes.

7. **Audit — NEW families** `scenario_run.*` (for `kind=scenario`) and `command_run.*` (for `kind=command`). They **semantically replace** `tide.*` (→ `scenario_run.*`) and `errand_run.*` (→ `command_run.*`). Downstream audit consumers (the UI audit parser in [`soul-stack-web`](0035-distribution-split.md#adr-035-distribution-split--core-apicli-vs-web-ui), [ADR-042](0042-backend-driven-ui.md#adr-042-backend-driven-dynamic-data-in-the-ui--the-ui-does-not-hardcode-dynamic-catalogs)) are fixed **synchronously** during implementation of the corresponding slice (S2/S3). Names are registered in [naming-rules.md → Audit-events](../naming-rules.md#audit-events).

8. **Invariants inherited from the absorbed entities** (carried over into Voyage without weakening):
   - **Per-incarnation state-commit (`kind=scenario`)** = parity with per-Surge state-commit [ADR-040](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override) = parity with barrier [ADR-009 §7](0009-scenario-dsl.md#adr-009-scenario--the-full-destiny-task-dsl-the-boundary-with-destiny-is-a-recommendation). Commit of `incarnation.state` — only after a successful cross-host barrier of that incarnation; partial commit is prohibited (B1).
   - **State invariant of `command`-kind** — does not mutate `incarnation.state` ([ADR-033](0033-errand.md#adr-033-errand--pull-ad-hoc-exec-outside-a-scenario)).
   - **Failover claim+lease** — parity with Acolyte-style PG-lease ([ADR-027(d)](0027-apply-work-queue.md#adr-027-apply-execution-model--work-queue--claim-acolyte-pool-ward-claim) / [ADR-040 amendment](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override)): any Keeper instance picks up an orphaned Voyage via PG-claim + TTL-lease, fencing epoch via `attempt++`.

   **Default policy of the recovery rule `reclaim_voyages` — ENABLED via path-defaulting.** `reclaim_voyages` (the failover mechanism above) runs by default, without requiring a config entry. The mechanism is **path-defaulting in `reaper.dispatch`**: absence of the `reclaim_voyages` key in `reaper.rules` OR explicit `enabled: true` → the rule RUNS (ON); it can only be disabled with an explicit `enabled: false`. This is the **reverse default** relative to the other reaper rules, which are map-driven (absence of a key in `reaper.rules` = OFF, requiring explicit enablement); `reclaim_voyages` is a special recovery rule for which "by default" means "runs".
   - **(a) Why default-ON.** The target scale is up to 100k Souls; Keeper is a horizontally scalable stateless cluster ([ADR-002](0002-transport-grpc-ha.md#adr-002-transport-keeper--souls--grpc-bidirectional-stream-over-mtls-ha-keeper-cluster)), where restart/replacement of an instance (deploy, OOM, auto-scale) is a routine, regular event, not an incident. If the rule is disabled, an orphaned Voyage (owner died before finalization) hangs in `running` without a live owner until manual intervention — unacceptable for routine operation. Therefore recovery must not depend on whether the operator wrote the rule into `reaper.rules`: path-defaulting in `reaper.dispatch` raises `reclaim_voyages` when the key is absent, and the operator disables it only with an explicit `reaper.rules.reclaim_voyages.enabled: false`. (`reclaim_apply_runs` remains default-OFF, map-driven — it has a different risk: a stale RunResult after the run has been handed off to the Soul, so only explicit opt-in.)
   - **(b) Safety of default-ON — exactly-once on the top-level Voyage.** Finalization relies on a CAS ownership guard: `voyage.Finalize` (`keeper/internal/voyage/crud.go`) writes the terminal status under the condition `WHERE voyage_id = $1 AND claimed_by_kid = $2`. If a "dead" worker wakes up after a reclaim (the claim was taken over by another Keeper, `attempt++`), its commit doesn't match by owner — the UPDATE affects 0 rows, the caller catches `ErrLeaseLost` and does NOT apply the result. A double commit of one Voyage by two instances is impossible even with a false reclaim of a live owner — exactly-once at the top-level Voyage level is preserved.
   - **(c) Lease tuning requirement.** The renew interval (`last_renewed_at`) MUST be noticeably shorter than the lease TTL (`claim_expires_at`): otherwise a live but temporarily lagging worker loses its lease and triggers a false reclaim (repeated spawn of Legs). A short renew relative to TTL narrows the window of double-spawn to a minimum; exactly-once at the top-level Voyage itself is nevertheless guaranteed by the CAS guard (b) regardless of tuning. Specific values are in docs/keeper/reaper.md.
   - **Snapshot-scope** — the set of run units is fixed at start (`voyages.target_resolved`), it does not "drift" from Souls being added/removed between Legs (parity with Tide `target_resolved_souls`).
   - **Cross-incarnation is legal at the Voyage layer** (`kind=scenario` orchestrates N incarnations) — a lifting of the ban from [ADR-008](0008-coven-stable-tags.md#adr-008-coven--stable-logical-tags-only) at the top layer; within a single scenario-run / scenario-resolver the "one incarnation" ban **remains in force** (see the ADR-008 amendment below).

   - **(d) Idempotency of `command`-Voyage and at-least-once.** (1) A `command`-Voyage (`kind=command`) has **at-least-once** execution semantics per target on failover — like the entire Voyage layer (see the failover model above); exactly-once is guaranteed by the CAS guard (b) only on finalization of the **top-level** Voyage, not on each spawn of a child Errand. (2) Spawning a child Errand is **fenced via ctx-cancel**: the worker's `renewLoop` (`keeper/internal/voyageorch/worker.go`) on lease loss (`ErrLeaseLost` from `RenewVoyageLease` — CAS `WHERE claim_by_kid=$2 AND attempt=$3` in `keeper/internal/voyage/claim.go`) calls `cancelLease`, and the worker that lost the lease **stops spawning** new Errands. (3) Therefore the residual duplication window is **pathological**: the worker must hang longer than the lease TTL, then wake up and manage to spawn an Errand **before** it catches the context cancellation. Routine failover (a dead worker doesn't wake up) does **not** produce Errand duplication. (4) **Recommendation to the operator:** whitelist command modules (`core.cmd.shell` / `core.exec.run`, item 9) for a multi-target `command`-Voyage should preferably be made **idempotent** — the operator accounts for at-least-once semantics when choosing the module and phrasing the command. (5) `kind=scenario`-Voyage is **unaffected** by this: its per-incarnation state-commit goes through a single-winner barrier ([ADR-009 §7](0009-scenario-dsl.md#adr-009-scenario--the-full-destiny-task-dsl-the-boundary-with-destiny-is-a-recommendation)), which cuts off duplication at the state-commit level.

9. **Whitelist of `command`-kind** — the same as [ADR-033](0033-errand.md#adr-033-errand--pull-ad-hoc-exec-outside-a-scenario): a hard list (`core.cmd.shell` / `core.exec.run`) + the `ErrandReadSafe` marker. Single-SID `/v1/souls/{sid}/exec` remains sugar (see the ADR-033 amendment below).

**Rejected alternatives.**
- **(a) Keep three entities.** Duplication of dialects, triple UI/CLI/audit, no unified All-runs. The user's decision — unify.
- **(b) B2 (two-level host waves within a Leg for `scenario`).** Breaks the barrier/state-commit [ADR-009 §7](0009-scenario-dsl.md#adr-009-scenario--the-full-destiny-task-dsl-the-boundary-with-destiny-is-a-recommendation) (partial state commit). See item 3.
- **(c) Keep invocation-time target-override.** Replaced by selection-from-a-permitted-set (item 5) — the same security invariant without resolver complexity.
- **(d) A single permission for both `kind`.** Would erase the scenario↔command boundary in RBAC (privilege escalation). Permission is selected by `kind` (item 6).

**Amendment (2026-05-29, the Choir axis is orthogonal to Leg).** [ADR-044](0044-choir.md#adr-044-choir--named-host-topology-within-an-incarnation) introduces **Choir** — a named topology of hosts WITHIN an incarnation. This is an **intra-incarnation** axis and it **does not conflict with Leg**: for `kind=scenario`, Leg = N **incarnations** (the top-layer batch axis, B1), while Choir lives within a single incarnation (host-level position). Narrowing by Choir is done via scenario `where:` ([ADR-044](0044-choir.md#adr-044-choir--named-host-topology-within-an-incarnation) item 5), not via Voyage parameters; Leg partitioning and Choir targeting are independent axes.

**Amendment (2026-06-01, two batch modes + batch strategies — user decision).** Voyage gets a second batching mode and a set of additive keys. Everything is **forward-compat additive** (new optional payload fields, a new config key; existing runs without them work as before — [ADR-012 forward-compat only-add](0012-keeper-soul-grpc.md#adr-012-keepersoul-grpc-contract-one-eventstream-with-oneof-keeper-side-render-forward-compat-only-add)). The contract `kind`/Leg/B1/§7 is **not weakened**.

1. **Discriminator `batch_mode`: `barrier` | `window`.** New optional invocation-payload field (default `barrier` — the current implemented behavior).
   - **`barrier`** (implemented, item 2 above): the run is split into sequential Legs (a batch of `batch_size` units), with a barrier between Legs (wait for the terminal state of all units of a Leg, optionally `inter_batch_interval`). `concurrency` = parallelism **within** one Leg, `batch_size` = batch size.
   - **`window`** (new): a full sliding window — a worker pool pulls units from the run's common queue, one returns → the next one starts, `concurrency` active units are constantly held. **There are no barriers between batches.** In window mode `batch_size` is **NOT used** (there are no batches); the window width is set by reusing the existing `concurrency` ("hold N active"). A `batch_size` in a payload with `batch_mode: window` → `422 validation-failed`.
   - **Window over hosts — `kind=command` only.** For `kind=command` the window slides **over hosts** (state is not touched — host-level chunking is legal, item 3). For `kind=scenario` the window slides **over incarnations**: within each incarnation the cross-host barrier and per-incarnation state-commit **remain unchanged** (B1, [ADR-009 §7](0009-scenario-dsl.md#adr-009-scenario--the-full-destiny-task-dsl-the-boundary-with-destiny-is-a-recommendation) is not violated — the window only removes the barrier **between** incarnations, not within one). B2 (a window over hosts across incarnations) remains REJECTED for the same reason as in item 3.

2. **`batch_percent` (new optional int, 1..100).** Batch size as a % of the resolved scope, **mutually exclusive** with `batch_size`. Additive: `batch_size` stays `*int` (NOT a union type) — the resolver, when `batch_percent` is present, computes the effective batch size from `len(target_resolved)`. Invariant: **exactly one** of `{batch_size, batch_percent}` (both set → `422`); in `batch_mode: window` neither is used. Applicable only to `batch_mode: barrier`.

3. **`fail_threshold: N` (new optional int, request-level).** Absolute failure-count threshold: once `N` failed units have accumulated → the run stops (no new Legs / no new window units start). **Relationship with `on_failure`:** the current `on_failure: abort` ≡ `fail_threshold: 1` (one failure → stop); `on_failure: continue` ≡ no threshold (`fail_threshold` not set → run to completion). `fail_threshold: N` (N>1) — intermediate tolerance. Works in both `batch_mode`.
   > **Changed by the 2026-06-09 amendment:** the previous wording "In MVP the threshold is **absolute** (not a percent)" is **removed**. The threshold is now expressed as a string field `max_failures` (`N` absolute | `N%` percent); `fail_threshold: N` (int) remains accepted but **deprecated**. The full threshold semantics (run units by `kind`, `cancelled`/`no_match` not counted, `%` → `ceil(scope·pct/100)`) — in the 2026-06-09 amendment below.

4. **`inter_unit_interval` (new optional duration).** Per-unit pause in `batch_mode: window` (parity with `inter_batch_interval`, which is the pause **between Legs** in `barrier`): a delay before starting each next window unit — soft throttling in a sliding window. Applicable only to `batch_mode: window`; not used in `barrier` (there throttle = `inter_batch_interval`).

5. **`require_alive` (new optional bool, default `false`).** Liveness presence filter: when `true`, scope resolution additionally excludes Souls without a live presence lease (`SoulLeaseChecker` plugs into scope resolution, presence lease — [ADR-006 Redis](0006-cache-redis.md#adr-006-cache-and-coordination--redis)). The scope snapshot after the filter is recorded in `voyage.target_resolved` as usual (the snapshot invariant of item 8 is not weakened — the filter is applied **at resolution**, it does not "drift" during the run).

6. **`max_batch_size` (new Keeper config key, alongside `voyage.max_scope`).** An operator invariant — the upper bound on batch/window size (`batch_size` for `barrier`, `concurrency` for `window`). Fail-closed: a request with `batch_size` or `concurrency` > `max_batch_size` is rejected with `422` and error code **`voyage_batch_size_too_large`** (parity with `voyage_scope_too_large` — a DoS guard, ADR-002 scale of 100k). Symmetric to `voyage.max_scope`: default + `0` = "no limit", negative → `value_out_of_range`.

7. **Semantics of `voyage_targets.batch_index` for the two modes.** The `voyage_targets.batch_index` column (= the Leg a unit belongs to):
   - **`batch_mode: barrier`** — the Leg index of the unit (0-based), as now (Leg = a batch of `batch_size`). UI two-level drill groups by `batch_index`.
   - **`batch_mode: window`** — **`batch_index = 0` for all units** (there are no batches in a window, the run is flat). UI drill in window mode shows a flat list of units, without grouping by Legs.

8. **Table "which key applies to which mode".**

| Key | `batch_mode: barrier` | `batch_mode: window` | Level |
|---|---|---|---|
| `batch_size` | batch size (Leg) | **not used** (`422` if set) | request |
| `batch_percent` | % of scope (XOR with `batch_size`) | **not used** | request |
| `concurrency` | parallelism **within** a Leg | **window width** (hold N active) | request |
| `inter_batch_interval` | pause between Legs | not used | request |
| `inter_unit_interval` | not used | per-unit throttle | request |
| `fail_threshold` | failure threshold → stop (`abort` ≡ `1`) | failure threshold → stop | request |
| `on_failure` | `abort`(=`fail_threshold:1`) \| `continue` | `abort` \| `continue` | request |
| `require_alive` | presence filter at scope resolution | presence filter at scope resolution | request |
| `dry_run` / `schedule_at` | as in item 2 | as in item 2 | request |
| `voyage.max_scope` | ceiling on resolved scope | ceiling on resolved scope | config |
| `max_batch_size` | ceiling on `batch_size` | ceiling on `concurrency` (= window width) | config |

**Invariants that are NOT affected:** B1 / per-incarnation state-commit / [ADR-009 §7](0009-scenario-dsl.md#adr-009-scenario--the-full-destiny-task-dsl-the-boundary-with-destiny-is-a-recommendation) cross-host barrier (window for scenario removes the barrier only **between** incarnations, within an incarnation the barrier+commit are preserved); B2 remains rejected; snapshot-scope (item 8 of the original ADR); failover claim+lease; permission-by-`kind`. Implementation — a separate slice (this amendment is only a design fixation).

**Amendment (2026-06-09, string batch fields `batch`/`max_failures` + preview endpoint — user decision).** The two former int fields (`batch_size`/`batch_percent` — batch size; `fail_threshold` — failure threshold) are merged into **string** fields, convenient for CLI and for percentages. All of it is **forward-compat additive, without a v2 and without DB migrations**: the new string field, upon resolution, **decomposes into the existing columns** (`batch_size`/`batch_percent`/`fail_threshold` in the `voyages` table, migration `064`/`065`), the old int fields remain accepted, the contract `kind`/Leg/B1/§7 is not weakened.

1. **`batch` (new string field, request-level).** A single field for batch size: `N` = an absolute number of hosts, `N%` = a percent (1-100) of the resolved scope. **Parsed by Keeper** (request validation): after `trim`, the string is matched against the grammar `^(\d+)%?$`; invalid (empty `N`, `0`, `%` without a number, garbage, `>100` for a percent) — **fail-closed** → `422 validation-failed`. `N%` → effective batch size = `ceil(scope·pct/100)` of `len(target_resolved)` (naturally ∈ [1, scope]); `N` → absolute value (further bounded by `max_batch_size`). Motive — a single field instead of the pair `batch_size`/`batch_percent` + a convenient CLI (`--batch 20%`). Applicable only to `batch_mode: barrier` (in `window` there are no batches — items 1-2 of the 2026-06-01 amendment).

2. **`max_failures` (new string field, request-level).** Failure threshold as a string: `N` (absolute) | `N%` (percent). **Counted by run units** — for `kind=scenario` a unit = an **incarnation**, for `kind=command` = a **host** (parity with Leg). Only failed units are counted; `cancelled` and `no_match` (a unit excluded by `where:`/the presence filter — nothing to execute for it) are **NOT counted as failures**. `N%` → `ceil(scope·pct/100)` (naturally ∈ [1, scope]). Replaces `fail_threshold` (which remains accepted, deprecated). The relationship with `on_failure` is unchanged (`abort` ≡ `max_failures: 1`, `continue` ≡ threshold not set). Works in both `batch_mode`.

3. **A1 coexistence (additive, non-breaking).** The old fields `batch_size` (`*int`), `batch_percent` (`*int`), `fail_threshold` (`*int`) **remain accepted** and are marked **deprecated**. Mixing the new string format and the old int format in one request (`batch` together with `batch_size`/`batch_percent`, or `max_failures` together with `fail_threshold`) → **`422`** with error code **`voyage_batch_spec_conflict`**. No union types, no DB migrations: at resolution, `batch` → (`batch_size` or `batch_percent`), `max_failures` → `fail_threshold` (extended for percent). Forward-compat **without `v2`** (the [ADR-012 only-add](0012-keeper-soul-grpc.md#adr-012-keepersoul-grpc-contract-one-eventstream-with-oneof-keeper-side-render-forward-compat-only-add) rule): old clients/runs work as before.

4. **`POST /v1/voyages/preview` (new endpoint).** Dry-resolve scope **without creating a Voyage**: accepts the same target fragment + batch settings as `POST /v1/voyages`, and returns `{scope_size, total_batches, effective_batch_size}`. **Does not reveal the SID list** (only numbers — for a preview of the batch count). Reuses the scope resolvers in **read-only** mode (without INSERT into `voyages`/`voyage_targets`), respects the `voyage.max_scope` ceiling and **RBAC-by-`kind`** (`incarnation.run` for scenario / `errand.run` for command — the same guard as create). **Tempo rate-limit — a SEPARATE bucket `voyage_preview`** ([ADR-050 amendment 2026-06-17](0050-tempo.md#adr-050-tempo--per-aid-rate-limiting-write-api), default `30/60`, softer than the create bucket's `10/20`): preview is read-like in effect but resolver-heavy in cost → its own, wider limit, **does not share quota with `voyage_create`**. ~~Reuses the same rate limit as create~~ (revoked by the 2026-06-17 amendment). Purpose — a preview for a **late-binding** target (`coven`/`require_alive`, where Keeper resolves the host count); for a snapshot target (explicit `incarnations[]`/`sids[]`) the client can count the number of batches itself, the endpoint is not mandatory.

**Amendment (2026-06-21, Voyage executor self-orphan-release of the applying lock before re-run).** A reclaimed VoyageWorker, BEFORE a repeat per-incarnation scenario-run, releases ITS OWN orphaned `incarnation.status='applying'`, left over from a run by a dead Voyage owner; without this, `lockRun` rejects the re-run and the reclaimed Voyage hangs on the very first applying incarnation. `reclaim_apply_runs` does **NOT** cover this gap (the lock is an orphaned flag on the `incarnation` row, not a Ward row of `apply_runs`; the `apply_run` itself is ephemeral or retention-purged). Detection — the `voyage_targets.apply_id` back-link of the previous attempt; fencing — no-live-rival + reclaimed-attempt + VerifyOwnership + single-winner CAS `applying→ready`. Always enabled together with default-ON `reclaim_voyages` (there is no separate opt-in — without this patch Voyage recovery is functionally broken). The full fixation (a residual double-apply class on par with `reclaim_apply_runs`, always-ON, the coexistence window with the `UpdateStateFromRun` status CAS) — [ADR-027 amend (l)](0027-apply-work-queue.md#adr-027-apply-execution-model--work-queue--claim-acolyte-pool-ward-claim).

**Invariants NOT affected by the 2026-06-09 amendment:** everything from the 2026-06-01 amendment (B1 / per-incarnation state-commit / §7 cross-host barrier / B2 rejection / snapshot-scope / failover claim+lease / permission-by-`kind`); `batch_mode` `barrier`|`window`, `window`/`max_batch_size`/`inter_unit_interval`/`require_alive` — unchanged; columns of the `voyages` table — unchanged (new fields decompose into existing ones). Implementation — a separate slice S1 (this amendment is only a design fixation).
