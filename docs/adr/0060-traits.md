# ADR-060. Trait ‚Äî operator-set key-value labels on the incarnation (relocated from Soul)

> **Status: amended (Trait relocated per-soul ‚Üí per-incarnation, R1).** Trait was originally introduced per-soul (operator-set labels on each Soul). **2026-06-25 (R1, fixed by the user) Trait is RELOCATED per-soul ‚Üí per-incarnation:** Trait belongs to the INCARNATION (an organizational label of the owner/product/namespace of the whole instance, not a single host). The source of truth is **`incarnation.traits`** (a jsonb column, modeled on `incarnation.covens`, migration `088_add_incarnation_traits`), set by the operator in `incarnation.spec.traits` on create. It is projected in a **MATERIALIZED** way into member hosts' `souls.traits` via a **sync hook** on incarnation create AND on host bind (`core.soul.registered`). `souls.traits` (migration `087`) REMAINS ‚Äî it becomes the **projection target**, the read layer is reused unchanged (`soulprint.self.traits` / `where:traits` / soul-lint / `topology.HostFacts.Traits`). The text below (items 1‚Äì6) describes the read/target mechanics, which remain in force; only the source of Trait moved from the host to the incarnation.
>
> **Implemented (R1 foundation, slice 1):** migration `088_add_incarnation_traits` (`incarnation.traits jsonb NOT NULL DEFAULT '{}'` + a GIN index, reversible), the field `incarnation.Incarnation.Traits` + round-trip in `keeper/internal/incarnation` (Create/scan/SelectAll), the bridge `incarnation.spec.traits ‚Üí incarnation.traits` (`incarnation.TraitsFromSpec`), the **sync hook `incarnation.SyncTraitsToHosts`** (reuses `soul.BulkReplaceTraits` with `BulkSelector{Incarnation}` + `BulkScope{Unrestricted}`), wired into (a) the incarnation-create path (`handlers.CreateTyped`, after insert, gated on "traits non-empty") and (b) the bind path (`scenario.keeper_dispatch.syncTraitsOnRegistered`, after a successful `core.soul.registered`). The `souls.traits` read layer is UNTOUCHED (0 changes).
>
> **Transitional state (R1):** the per-soul write path `POST /v1/souls/traits` (bulk-assign) and the MCP `soul.traits-assign` still work directly on `souls.traits`, BUT they get **overwritten by the projection** on the next `incarnation.traits` sync (an incarnation create with traits / a host bind). This is expected during the transition period until **relocating the per-soul bulk API ‚Üí per-incarnation** (the next slice, see below).
>
> **Implemented (R1 slice a ‚Äî API relocation per-soul ‚Üí per-incarnation):** a `traits` field in `IncarnationCreateRequest` (`POST /v1/incarnations` ‚Üí `incarnation.spec.traits` ‚Üí `incarnation.traits` + projection on create); the day-2 `PUT /v1/incarnations/{name}/traits` (`incarnation.UpdateTraits` in one FOR UPDATE tx + the sync hook), permission `incarnation.traits-set`, the audit event `incarnation.traits_changed` (KEYS only), the MCP tool `keeper.incarnation.traits-set`. The per-soul write path (`POST /v1/souls/traits` / `soul.traits-assign` / `keeper.soul.traits-assign`) is **DEPRECATED, NOT removed** (forward-compat). Details ‚Äî the Amendment block below.
>
> **NOT in slice 1/a (next slices):** **RBAC scope by traits on the incarnation dimension** (R1 unblocked it ‚Äî Trait is now incarnation-level data, scope-pushdown by incarnation-traits is possible; item 7 below is no longer blocked; slice a does NOT implement it); an operator-facing UI for `incarnation.spec.traits`; physically removing the per-soul path.
>
> **Historical header (the per-soul pilot, before R1):** the read/target path was implemented per-soul: migration `087_add_souls_traits` (`souls.traits jsonb` + a GIN index), `soul.Soul.Traits` + round-trip, the registry projection `soulprint.self.traits` (`cel_render.go`), `topology.HostFacts.Traits`, the whitelist entry `traits` (`shared/config/soulprint_paths.go`). This whole layer is reused as-is as the projection target.

**Context.** As of 2026-06-25, Soul has a single operator-set grouping axis ‚Äî **Coven** ([ADR-008](0008-coven-stable-tags.md#adr-008-coven--stable-logical-tags-only)): a flat `souls.coven TEXT[]`, stable logical tags (cluster / project / environment / datacenter). Coven is good for membership and coarse targeting (`$1 = ANY(coven)`, `coven && ARRAY[...]`), and RBAC scope-pushdown ([ADR-047](0047-purview.md#adr-047-purview--scoped-rbac-visibility-of-nodes-role-default_scope--extended-selector)) and the CEL predicate `'x' in soulprint.self.covens` are built on it too.

What's missing: **operator-set key-value host attributes** ‚Äî `owner=alice`, `product=aboba`, `namespace=dba-ns`. These can't be expressed as a flat tag without "gluing" (`owner-alice`), which breaks targeting by key (`owner == 'alice'`), doesn't allow list values (`owners: [alice, bob]`), and clutters the coven axis that RBAC and membership hang off of.

**Decision (fixed by the user on 2026-06-25, propose-and-wait closed).** Introduce **Trait** ‚Äî operator-set key-value labels on Soul, a **separate entity alongside the flat Coven**.

1. **Trait ‚Äî key-value (scalar | list), operator-set, separate from Coven (Variant B, locked).**
   - A Trait's value is a scalar (`namespace: dba-ns`, `product: aboba`) OR a list (`owners: [alice, bob]`). Arbitrary operator string keys.
   - The source is the **operator** (stable registry data, like Coven), **NOT Soul-reported**. Trait is an organizational label ("whose host / which product"), not a fact about the system.
   - **Coven is untouched.** Trait is a parallel axis: `souls.coven TEXT[]` remains a flat set of membership/targeting/RBAC tags; `souls.traits` is a new jsonb column for key-value data.

2. **Storage ‚Äî `souls.traits jsonb` (locked).** `NOT NULL DEFAULT '{}'::jsonb` + a GIN index for targeting (`traits @> '{"namespace":"dba-ns"}'` / containment by key). The `souls.coven TEXT[]` column is unchanged. jsonb because the value is polymorphic (scalar | list), which `TEXT[]` cannot express.

3. **Projection into `soulprint.self.traits` (a registry projection, like `covens`/`choirs`, locked).** When resolving CEL, Keeper puts `souls.traits` into `soulprint.self.traits` ‚Äî a virtual projection of registry data into the Soulprint namespace, exactly as `soulprint.self.covens` projects `souls.coven[]` ([ADR-018](0018-soulprint-typed.md#adr-018-soulprint-typed-schema-mvp): "`covens` is not in `SoulprintFacts`"). The source of truth is the registry, so the projection **overrides** a same-named reported key, had Soul ever sent one. **`traits` is NOT part of the `SoulprintFacts` proto** ‚Äî the Soul‚ÜíKeeper registration contract is untouched (forward-compat: Trait is operator data, Soul doesn't report it).

4. **Targeting ‚Äî `where:` over `soulprint.self.traits.<key>` (locked).** The canonical CEL form (via `soulprint.self`, [ADR-018](0018-soulprint-typed.md#adr-018-soulprint-typed-schema-mvp)):
   - a scalar: `where: soulprint.self.traits.namespace == 'dba-ns'`;
   - a list: `where: 'alice' in soulprint.self.traits.owners`.

   `soulprint` is declared in CEL as `DynType` ([ADR-010](0010-templating.md#adr-010-templating-engine-cel-for-yaml-expressions-go-texttemplate-for-files)), so `traits.<any_key>` resolves dynamically **with no CEL-env edit and no AST rewrite** ‚Äî `traits` is just a regular map field of the targeting element. Soul-lint statically checks only the `traits` prefix (see item 6), the second segment (the operator's key) is dynamic ‚Äî like `covens`.

5. **Mutability ‚Äî like Coven (amended R1, R1 slice a).** Trait is changed by the operator (stable registry data). **R1: the source of truth is `incarnation.traits`** (operator-set in `incarnation.spec.traits` on create), not a per-soul write. `souls.traits` is only mutated by the **projection** (the sync hook `SyncTraitsToHosts`), not directly by the operator. **R1 slice a: the operator-facing per-incarnation write path is IMPLEMENTED** ‚Äî `incarnation.spec.traits` on create + the day-2 `PUT /v1/incarnations/{name}/traits` (permission `incarnation.traits-set`, audit `incarnation.traits_changed`, MCP `keeper.incarnation.traits-set`); the per-soul bulk write `POST /v1/souls/traits` / `soul.traits-assign` is **DEPRECATED, NOT removed** (forward-compat, overwritten by the projection). Details ‚Äî the Amendment block.

   **Sync hook (R1, a new reactive mechanism ‚Äî Coven has no reactive equivalent, built for the first time here).** `incarnation.SyncTraitsToHosts` projects `incarnation.traits` in a MATERIALIZED way into member hosts' `souls.traits` via `soul.BulkReplaceTraits` (a full replace ‚Äî `incarnation.traits` is the sole source of truth, the projection aligns hosts to it). A member of the incarnation = a host whose incarnation name ‚àà `souls.coven[]` (ADR-008, `BulkSelector{Incarnation}`). scope = `Unrestricted` (a keeper-internal projection, not an operator bulk action ‚Äî it does not obey the initiator's coven scope). Two wiring points: **(a) incarnation create** (after insert, gated on "traits non-empty"), **(b) host bind** via `core.soul.registered` (after a successful registration ‚Äî a newly bound host picks up its incarnation's traits). Both are best-effort (a projection failure does not fail the create/run ‚Äî Trait is an organizational label, it will catch up on the next bind).

6. **Soul-lint ‚Äî `traits` in the `soulprint.self.*` whitelist as a map/dyn (locked).** Added to `soulprintSelfTopLevel` (`shared/config/soulprint_paths.go`) **without** statically checking the second segment ‚Äî Trait keys are dynamic, like `covens`/`choirs`. `soulprint.self.traits.<any_key>` passes lint (the third segment isn't captured by the regex, like the `os.family` tail after an array/scalar).

7. **RBAC scope by traits ‚Äî UNBLOCKED by R1, implementation is the next slice.** Before R1, scope-by-traits was deferred because of the per-soul shape (a Trait dimension on the host doesn't fit well into Purview's incarnation-scope). **R1: Trait is relocated onto the incarnation ‚Äî scope-pushdown by `incarnation.traits` becomes natural** (a Purview incarnation dimension, analogous to coven‚à™{name} from [ADR-047](0047-purview.md#adr-047-purview--scoped-rbac-visibility-of-nodes-role-default_scope--extended-selector)). The foundation (slice 1) does NOT implement it ‚Äî only unblocks it; the specific scope selector by traits is the next slice.

**Rejected alternatives.**

- **Variant A ‚Äî extend `souls.coven` to key-value.** Rejected: breaks the [ADR-008](0008-coven-stable-tags.md#adr-008-coven--stable-logical-tags-only) flat-tag semantics (Coven is a stable logical tag, not a key-value pair), as well as RBAC scope-pushdown `$1 = ANY(coven)` and the predicates `'x' in soulprint.self.covens` ‚Äî they rely on the `TEXT[]` shape. Converting coven to jsonb would drag along a migration of every scope/target path. Trait as a separate axis leaves Coven untouched.
- **Soul-reported traits.** Rejected: contradicts [ADR-008](0008-coven-stable-tags.md#adr-008-coven--stable-logical-tags-only)/[ADR-018](0018-soulprint-typed.md#adr-018-soulprint-typed-schema-mvp) ‚Äî Trait is an organizational label of ownership, not a fact about the host; Soul (potentially compromised) should not get to declare its own product/owner affiliation. The source is the operator, as with Coven.

**Naming.** **Trait** ‚Äî the operator's key-value labels on Soul. Difference from Coven: Coven = flat groups (cluster / environment / project, an RBAC/membership axis); Trait = attributes (owner / product / namespace). The name is outside the SaltStack dictionary (there's no direct analog there; grains are Soul-reported facts, which Trait semantically is NOT). Coven is taken for flat groups ‚Äî it can't be reused.

**Amends.** Supplements [ADR-008](0008-coven-stable-tags.md#adr-008-coven--stable-logical-tags-only) (a parallel operator-set axis appears alongside Coven) and [ADR-018](0018-soulprint-typed.md#adr-018-soulprint-typed-schema-mvp) (a new registry projection `soulprint.self.traits`, symmetric to `covens`/`choirs`; the `SoulprintFacts` proto is untouched).

**Amendment (2026-07-17, NIM-124 ‚Äî membership decoupled from coven==name).** The projection's member criterion ‚Äî "a member of the incarnation = a host whose incarnation name ‚àà `souls.coven[]`" (`soul.BulkSelector{Incarnation}`) ‚Äî is superseded: membership is now a first-class relation `incarnation_membership` (see [ADR-008 amendment 2026-07-17](0008-coven-stable-tags.md#amendment-2026-07-17-nim-124-incarnationname-is-not-a-coven--membership-is-a-first-class-relation)). `BulkSelector{Incarnation}` resolves members via that relation, not via `= ANY(coven)`. The `souls.traits` read/target layer (items 1‚Äì6) is otherwise untouched.

**Amendment (2026-06-25, R1 ‚Äî Trait relocated per-soul ‚Üí per-incarnation).** Fixed by the user. The source of truth for Trait moved from the host (`souls.traits`, operator-set-per-soul) to the incarnation (`incarnation.traits`, operator-set in `incarnation.spec.traits`). `souls.traits` is kept as the **projection target**; the read layer (items 1‚Äì6) is reused unchanged. A **sync hook** `SyncTraitsToHosts` is introduced (a MATERIALIZED projection of incarnation.traits ‚Üí member hosts' souls.traits; wired into create + bind). Migration `088_add_incarnation_traits` (mirroring `046_add_incarnation_covens`). Transitional state: the per-soul bulk write still works, but is overwritten by the projection (relocating the per-soul API + deprecating it is the next slice). RBAC scope by traits on the incarnation dimension is UNBLOCKED (implementation is the next slice).

**Amendment (2026-06-25, R1 slice a ‚Äî API relocation per-soul ‚Üí per-incarnation IMPLEMENTED).** Closes "relocate the bulk write per-soul ‚Üí per-incarnation," flagged as the next slice in the header. The operator-facing Trait write path is moved onto the incarnation:

- **`incarnation.spec.traits` on create** ‚Äî a `traits` field (a map of key ‚Üí scalar | list of scalars) in `IncarnationCreateRequest` (`POST /v1/incarnations`). It is put into `incarnation.traits` (the source of truth) and projected in a materialized way into member hosts' `souls.traits` (`TraitsFromSpec` + `SyncTraitsToHosts` on the create path). The domain validates the format/value (a nested object/array ‚Üí 422).
- **`PUT /v1/incarnations/{name}/traits`** (the day-2 replacement) ‚Äî wholesale REPLACES `incarnation.traits` (an empty/omitted `traits` = clear the labels) in one `SELECT ‚Ä¶ FOR UPDATE` tx (`incarnation.UpdateTraits`), then the sync hook projects it into member hosts' `souls.traits`. Permission **`incarnation.traits-set`** (scope incarnation/coven/service by the path `name`, the same selector as `incarnation.update-hosts`). Audit event **`incarnation.traits_changed`** (`source: api`, payload `{name, old_keys, new_keys}` ‚Äî KEYS only, trait VALUES are NOT put in the payload, secret hygiene symmetric to `soul.traits-changed`). Mirrored 1:1 by the MCP tool **`keeper.incarnation.traits-set`** (the same domain, body-scoped RBAC OR-Check, `source: mcp`), REST-only semantics aside.
- **The per-soul write path is DEPRECATED (forward-compat, NOT removed).** `POST /v1/souls/traits` (bulk-assign), permission `soul.traits-assign`, MCP tool `keeper.soul.traits-assign` are kept for role/client config compatibility, marked deprecated (the MCP description and the call-time log signal "use `keeper.incarnation.traits-set`"). The per-soul write is still overwritten by the `incarnation.traits` projection on the next sync (create / bind) ‚Äî the behavior from the R1 header remains in force.

**NOT in slice a (remains the next slice):** an RBAC scope selector by traits on the incarnation dimension (R1 unblocked it, slice a doesn't implement it ‚Äî only moves the write path); a UI for `incarnation.spec.traits` / `PUT .../traits`; physically removing the per-soul path.
