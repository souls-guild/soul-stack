# ADR-008. Coven — только стабильные логические теги

- **Контекст.** Изначально Coven использовался и для стабильной группировки (кластер / проект / окружение / ЦОД), и для роли хоста: `incarnation.name` становился корневой меткой, а под-роли — по convention `{incarnation.name}-{role}` (`…-master`, `…-replica`). Роль (кто сейчас master в Redis HA) **волатильна**: меняется при failover в любой момент, без участия Keeper-а. Привязка волатильной роли к Coven-метке означала бы, что метка либо устаревает (таргет по `…-master` бьёт по бывшему мастеру), либо требует механизма свежести / коллектора роли / волатильных Soulprint-фактов — лишний движущийся механизм в критическом пути разрушительных операций.
- **Решение.**
  - **Coven — только стабильные логические теги:** кластер, проект, окружение, ЦОД, тип железа. То, что не меняется само по себе в ходе работы сервиса.
  - `incarnation.name` **остаётся** корневой Coven-меткой всех хостов incarnation, назначается keeper-ом автоматически.
  - **Convention `{incarnation.name}-{role}` удаляется.** Под-ковенов по роли больше нет.
  - **Роль (master / replica) — не Coven.** Существуют две концептуально разные роли: **declared** — в `incarnation.spec.hosts[].role`, для bootstrap (`create`, где probe-ить ещё нечего), топологии и аудита; **actual** — волатильная, получается **живым probe-шагом** в сценарии (`module: core.exec.run` + `register:`), таргет следующего шага идёт по `where:` от `register:`. После failover — просто новый probe.
  - **Волатильное не живёт в Soulprint.** Soulprint хранит только стабильные / медленно меняющиеся факты. Нет волатильных Soulprint-фактов, нет коллектора роли, нет механизма свежести.
  - **Essence role-agnostic.** Из pipeline-а сборки essence удаляется ступень `role/<Y>.yaml`; порядок сборки — `default → os → coven → incarnation.spec`. Роль-зависимые параметры переезжают в destiny и передаются через `input:` по probe-роли.
- **Обоснование.** Стабильная метка не устаревает — таргетинг по Coven всегда корректен по Postgres. Волатильное (роль) запрашивается ровно в момент использования и не кэшируется — нет окна рассинхрона перед разрушительной операцией. Убираем целый класс механизмов (коллектор роли, свежесть, волатильные факты), который существовал только чтобы поддерживать роль-как-метку. P2P Soul↔Soul для обмена ролью отвергнут отдельно — ломает ADR-002/004 и модель безопасности.
- **Trade-off.** Сценарий, которому нужна actual-роль, обязан содержать явный probe-шаг перед таргетингом (нельзя просто написать `coven: …-master`). Это больше строк в сценарии, но цена — корректность разрушительных операций при failover. Принимаем.
- **Отменяет.** Раздел «Targeting и связь хостов» в части под-ковенов `{incarnation.name}-{role}`; essence-слой `role/*.yaml` и строку `role/{{ host.role }}.yaml` в `essence/_stack.yaml`. Полное решение пользователя зафиксировано; детали таргетинга — раздел [«Targeting и связь хостов»](../architecture.md#targeting-и-связь-хостов), детали scenario — [`docs/scenario/`](../scenario/README.md).
- **Amendment (2026-05-25, окружения = Coven, first-class `Environment` отвергнут).** На вопрос «поддержка окружений с фиксированным набором и разделением прав» зафиксировано: **окружение остаётся частным случаем Coven-метки** (как и было заявлено в «Решении»), отдельная ось/сущность `Environment` **не вводится** (отклонён ради единой оси таргетинга — иначе завтра попросят то же для ЦОД/проекта и расплодим оси). Из этого следуют две **planned**-задачи (не реализованы на 2026-05-25):
  - **(a) Per-Coven RBAC-scope для incarnation-операций — РЕАЛИЗОВАНО (C+D, 2026-05-25).** Известный gap (extractor incarnation-эндпоинтов клал в RBAC-context только `{incarnation: name}` без `coven`/`service`, поэтому роль вида `incarnation.* on coven=prod` не матчила — docs↔code дрейф) **закрыт**:
    - **(C) Declared environment-теги — колонка `incarnation.covens`** (миграция **046**): хранит declared-набор стабильных coven-меток incarnation (в т.ч. окружение). Снимает неопределённость «что значит coven incarnation-а» из исходной planned-формулировки — это не размазанные по хостам метки, а явно задекларированный набор уровня incarnation.
    - **(D) RBAC-context incarnation-роутов обогащён** до `service=` + **multi-value** `coven=` (= `incarnation.covens` ∪ `{incarnation.name}` — `incarnation.name` остаётся корневой Coven-меткой, см. «Решение»). Матчинг — **OR-Check** (`RequirePermissionMulti`): доступ есть, если permission покрывает **хотя бы одну** из меток набора. `enforcer.Matches` НЕ тронут (изменился только сбор context-а, не движок).
    - **`create` — body-scoped:** проверка идёт по `covens` из тела запроса, поэтому оператор со scope `coven=dev` не создаст incarnation с `covens=[prod]` (нельзя «прокрасться» в чужое окружение на создании).
    - **REST↔MCP паритет:** все 7 incarnation MCP-tools (`run` / `upgrade` / `destroy` / `get` / `history` / `unlock` / `create`) делают тот же scoped OR-Check, **fail-closed** (нет матчащего permission → отказ).
    - **Ограничение (backlog).** Селекторы — **single-key** (`coven=` ИЛИ `service=`); комбинированный multi-key **AND** (`coven=X,service=Y` — «только prod-окружение этого сервиса») грамматикой permission-селектора **не выражается**. Структура `Permission.Selector` AND-семантику уже умеет — не хватает только грамматики разбора. Расширение требует parser-extension + отдельного ADR (см. [Открытые вопросы](../architecture.md#открытые-вопросы)).
  - **(b) «Заранее заданный фиксированный набор» меток — валидируемый справочник coven-меток.** Сейчас `souls.coven` — свободные строки без справочника (`migrations/007`), опечатка молча плодит фантом. Planned: реестр допустимых coven-меток (опц. с типом `environment`/`datacenter`/…), валидация против него. Точка расширения — `CovenLabelValidator`-хук в Service-слое (no-op в MVP), чтобы не ломать API при появлении справочника.
- **Amendment (2026-05-29, cross-incarnation на Voyage-слое — сознательное решение пользователя).** Инвариант «кросс-incarnation таргетинг запрещён грамматикой» (см. «Решение» выше и [«Targeting и связь хостов»](../architecture.md#targeting-и-связь-хостов)) **снимается на верхнем Voyage-слое** ([ADR-043](0043-voyage.md#adr-043-voyage--унифицированный-батчевый-прогон)): `kind=scenario` Voyage **легально оркестрирует несколько инкарнаций** в одном прогоне (батч = N инкарнаций, B1). На уровне **scenario-resolver-а и одного scenario-run-а запрет СОХРАНЯЕТСЯ** без изменений: один scenario-run работает строго с одной инкарнацией (один cross-host barrier, один per-incarnation state-commit, [ADR-009 §7](0009-scenario-dsl.md#adr-009-scenario--полная-dsl-задач-destiny-граница-с-destiny--рекомендация)). Voyage не размывает границу инкарнации — он оркестрирует **N независимых scenario-run-ов**, каждый со своей инкарнацией и своим state-commit-ом. Cross-incarnation запрет внутри `on:`/`where:`-грамматики scenario не ослаблен.
- **Amendment (2026-05-29, Choir ≠ coven — явная граница).** [ADR-044](0044-choir.md#adr-044-choir--именованная-топология-хостов-внутри-инкарнации) вводит сущность **Choir** — именованную позицию хоста ВНУТРИ инкарнации (топологическая «партия»). **Choir НЕ является Coven** и не реанимирует удалённый под-coven `{incarnation.name}-{role}`: coven — глобальная стабильная ось таргетинга/RBAC (кластер / проект / окружение / ЦОД), Choir — внутри-инкарнационная топологическая ось без RBAC-семантики coven-а. При этом **Choir — стабильный факт** (как coven, не как волатильная роль): он декларируется оператором и не меняется сам по себе при работе сервиса, поэтому доступен в `where:`-предикатах **без probe-шага** (через `soulprint.self.choirs` / `soulprint.hosts[].choirs`, [ADR-044](0044-choir.md#adr-044-choir--именованная-топология-хостов-внутри-инкарнации) пункт 5). Волатильная **actual**-роль (кто сейчас master после failover) по-прежнему получается только probe + register + `where:` — Choir её НЕ заменяет (Choir = declared-топология, не actual-state).
