# ADR-048. Conductor — leader-elected исполнитель Cadence-расписаний

> **Статус: канон ДО кода (C0, 2026-06-02).** propose-and-wait по имени **Conductor** пройден (выбрано пользователем). Дизайн модели — architect. Слайсы C1–C5 — реализация. Кода по этому ADR пока нет: триггер Cadence сейчас живёт Reaper-правилом `spawn_due_cadence` ([ADR-046 §4](0046-cadence.md#adr-046-cadence--регулярные-запуски-scheduledrecurring-voyage)), миграция в Conductor — предмет этого ADR.

**Контекст.** [ADR-046](0046-cadence.md#adr-046-cadence--регулярные-запуски-scheduledrecurring-voyage) (Cadence) повесил спавн due-расписаний на **Reaper-лидера** новым правилом `spawn_due_cadence` — это была натяжка двух разных доменов на один исполнитель:

- **Reaper = cleanup-домен** (TTL-purge, recovery-reclaim, retention) — правила, которым редкий тик нормален: дефолт `reaper.interval` = **1h** ([reaper.md → Конфиг](../keeper/reaper.md#конфиг)).
- **Cadence = scheduling-домен** — расписанию нужен **частый** тик: оператор, заказавший «каждые 5 минут» или cron `*/2 * * * *`, на reaper-тике 1h получит **спавн раз в час вместо раза в 5 минут** — расписание молча не выполняется с заявленной частотой. Понизить `reaper.interval` до 15–30s ради Cadence нельзя: это прогонит **все** cleanup-правила (purge/reclaim/scry) сотни раз в час без нужды, нагрузив PG и Redis-lease впустую.

Домены конфликтуют по своему естественному ритму. Совмещать их на одном тике — либо тормозить scheduling, либо молотить cleanup. Нужен **отдельный исполнитель расписаний** со своим интервалом, не связанным с reaper.interval.

**Решение (зафиксировано пользователем 2026-06-02).** Вынести спавн Cadence из Reaper в отдельную leader-elected подсистему **Conductor** — дирижёр расписаний (метафора продолжает «музыкально-душевную» линию [Choir](0044-choir.md#adr-044-choir--именованная-топология-хостов-внутри-инкарнации)/Voice/[Cadence](0046-cadence.md#adr-046-cadence--регулярные-запуски-scheduledrecurring-voyage): дирижёр задаёт ритм исполнения). Conductor — это **только смена исполнителя триггера**; вся spawn-семантика Cadence ([ADR-046 §4/§5](0046-cadence.md#adr-046-cadence--регулярные-запуски-scheduledrecurring-voyage)) переносится **без изменений**.

> **Индустриальная коллизия имени принята осознанно (пользователь, 2026-06-02).** «Conductor» носят Netflix Conductor (workflow-orchestrator) и Temporal/Cadence (Uber) — там «Cadence» вообще имя workflow-движка. В Soul Stack Cadence = расписание, Conductor = его исполнитель; пересечение терминов с этими продуктами лексическое, не семантическое. Решение зафиксировано пользователем как приемлемое.

1. **Conductor — отдельная leader-elected подсистема внутри `keeper`** (не отдельный бинарь, как Reaper). Сидит на **generic leaderloop** (`keeper/internal/leaderloop`, вынесен слайсом C1 — на тот же общий примитив переводится и Reaper). Свой Redis-lease ([ADR-006](0006-cache-redis.md#adr-006-кэш-и-координация--redis)) с ключом **`conductor:leader`** (отдельный от `reaper:leader`) → ровно-один-исполнитель спавна в кластере, как у Reaper, но **независимое** лидерство (лидер Conductor и лидер Reaper — это могут быть разные инстансы).

2. **Свой tick-interval, независимый от `reaper.interval`.** Новый блок конфига **`cadence_scheduler`** в `keeper.yml` с собственным `interval` (рекомендация ~15–30s — достаточно частый для interval/cron-гранулярности Cadence, не настолько, чтобы нагружать PG как cleanup-loop). Conductor на каждом тике исполняет ровно ту due-выборку, что описана в [ADR-046 §4](0046-cadence.md#adr-046-cadence--регулярные-запуски-scheduledrecurring-voyage): `SELECT … FROM cadences WHERE enabled AND next_run_at <= NOW() FOR UPDATE SKIP LOCKED` → per due-строка в одной PG-tx: применить `overlap_policy` + Insert `voyages`/`voyage_targets` с `cadence_id` + пересчёт `next_run_at` + `last_run_at`.

3. **Переносится из Reaper — логика spawn целиком, не переписывается.** `CadenceSpawner` (логика из `keeper/internal/reaper/cadence_spawn.go`, [ADR-046 §4](0046-cadence.md#adr-046-cadence--регулярные-запуски-scheduledrecurring-voyage)) переезжает в Conductor **как есть**: due-выборка, три `overlap_policy` (`skip`/`queue`/`parallel`, [ADR-046 §5](0046-cadence.md#adr-046-cadence--регулярные-запуски-scheduledrecurring-voyage)), пересчёт `next_run_at` (cron-парсер / interval-anchored), spawn-tx. **Reaper теряет правило `spawn_due_cadence`** и связанный с ним `action: spawn` (closed-enum Reaper-action возвращается к `expire`/`delete`/`set_status`/`report`/`soft_delete`). `spawn` был единственным **активным** (insert-порождающим) action Reaper-а — после выноса Reaper снова чисто cleanup-домен.

4. **Audit-source остаётся `background`.** Spawn/skip-события Cadence (`cadence.spawned` / `cadence.skipped_overlap`, [ADR-046 §8](0046-cadence.md#adr-046-cadence--регулярные-запуски-scheduledrecurring-voyage)) после переезда пишутся с тем же **`source: background`** и `archon_aid: NULL` (автономная фоновая инициатива keeper-а; авторство рецепта по-прежнему живёт в `voyages.started_by_aid` дочернего Voyage). **Новый audit-source `scheduler` НЕ вводится** — enum audit-source ([ADR-022(b)](0022-audit-pipeline.md#adr-022-audit-pipeline-storage-schema-retention)) не расширяется; `background` уже покрывает «фоновое периодическое keeper-правило без оператора-инициатора», семантика спавна не изменилась от смены исполнителя.

5. **Default-ON при наличии Redis.** Conductor включён по умолчанию, **если настроен Redis** (lease-лидерство возможно). Это footgun-guard: Cadence без работающего планировщика молча не спавнит Voyage — оператор создал расписание, а оно «мёртвое» без видимой ошибки. Поэтому планировщик активен из коробки на любой Redis-инсталляции; выключение Cadence — per-Cadence через `enabled: false` ([ADR-046 §3](0046-cadence.md#adr-046-cadence--регулярные-запуски-scheduledrecurring-voyage)), не глобальным гашением Conductor. Без Redis (single-instance dev без Redis) leader-election невозможна — Conductor деградирует так же, как Reaper-лидер на одиночном инстансе.

6. **`scry_background` — аномалия Reaper, оставлена (backlog).** `scry_background` ([ADR-031](0031-scry-drift.md#adr-031-scry--drift-detection-declarative-dry-run-reconcile) Slice C, [reaper.md → Правила](../keeper/reaper.md#правила)) — тоже **активное** (dispatch-порождающее dry_run-прогоны) правило, сидящее в cleanup-домене Reaper по той же натяжке, что была у `spawn_due_cadence`. По-хорошему фоновый drift-скан — это тоже scheduling, не cleanup. Но в этом ADR его **не трогаем** (backlog): у scry своя частота (`scry_background.interval`, рекомендация 6h — близка к reaper-домену, не требует частого тика как Cadence), и его вынос — отдельное решение при появлении запроса. Зафиксировано как известная аномалия, не как долг этого слайса.

**Forward-compat.** Conductor — новая keeper-side подсистема, ничего не ломает. Таблица `cadences` и колонка `voyages.cadence_id` ([ADR-046](0046-cadence.md#adr-046-cadence--регулярные-запуски-scheduledrecurring-voyage)) не меняются. Proto Keeper↔Soul **не затрагивается** (Conductor спавнит Voyage существующим путём). Reaper-`action` enum **сужается** удалением `spawn` (C4) — это не breaking для конфигов: `spawn` использовался единственным правилом `spawn_due_cadence`, которое удаляется тем же слайсом; иных правил с `action: spawn` в `keeper.yml` нет и быть не могло. До C4 закрытие двойного спавна гарантируется атомарным переключением исполнителя (правило в reaper остаётся неактивным/удаляется одним шагом с включением Conductor).

**Связь с другими ADR.**

Меняется **только исполнитель триггера**: Reaper-лидер + `spawn_due_cadence` → Conductor-лидер. ADR-046 §4/§8 амендятся (триггер; audit-source background сохраняется).
- **[ADR-006](0006-cache-redis.md#adr-006-кэш-и-координация--redis)** — свой Redis-lease `conductor:leader` (отдельный от `reaper:leader`), single-executor спавна, default-ON при наличии Redis.
- **[ADR-027](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim) / Reaper** — Reaper после выноса возвращается к чистому cleanup-домену; перечень Reaper-правил теряет `spawn_due_cadence`, closed-enum action — `spawn`.
- **[ADR-022](0022-audit-pipeline.md#adr-022-audit-pipeline-storage-schema-retention)** — audit-source enum **не расширяется**: `cadence.spawned`/`cadence.skipped_overlap` остаются `source: background`.
- **leaderloop (C1)** — Conductor и Reaper делят generic lease-loop-примитив `keeper/internal/leaderloop`; два независимых lease-ключа, две независимых leader-election.

**Отвергнутые альтернативы.**
- **(а) Оставить спавн в Reaper, понизить `reaper.interval` до 15–30s** — прогонит все cleanup-правила сотни раз в час без нужды (PG/Redis нагрузка, бессмысленные purge-сканы). Домены имеют разный естественный ритм — нельзя один интервал на оба.
- **(б) Отдельный бинарь `conductor`** — Reaper и Conductor оба keeper-side single-executor-подсистемы; выделять бинарь под одну фоновую goroutine противоречит модели ([ADR-004](0004-binaries.md#adr-004-раскладка-бинарей--keeper-soul-soul-lint-push-режим--модуль-внутри-keeper): Reaper тоже не отдельный бинарь). Conductor живёт внутри `keeper`.
- **(в) Новый audit-source `scheduler`** — `background` уже семантически точен (фоновое периодическое keeper-правило без оператора-инициатора); смена исполнителя не меняет природу события. Расширять enum ради косметики — лишний breaking-риск.
- **(г) Заодно вынести `scry_background` в Conductor** — у scry другой ритм (6h, reaper-близкий) и своя модель (dry_run work-queue, не Voyage-spawn); вынос — отдельное решение при запросе, не часть этого слайса (см. §6).

**Что отложено (post-MVP / backlog).**
- Вынос `scry_background` из Reaper-домена (§6) — при появлении запроса/проблемы с частотой drift-скана.
- Возможное обобщение Conductor до универсального scheduler-домена (несколько типов расписаний на одном leader-loop) — пока единственный потребитель Cadence.

**Adaptive interval (амендмент 2026-06-07).** Шаг опроса Conductor больше **не фиксированный** `cadence_scheduler.interval`, а **адаптивный** — выводится из реального enabled-реестра Cadence на каждом тике:

- **Формула шага:** `clamp(derivedMinPeriod, poll_floor, poll_ceiling)`; **пустой** enabled-реестр (нет ни одного enabled-правила) → `poll_idle`.
- **`derivedMinPeriod`** = `min( MIN(interval_seconds) по enabled interval-правилам, 60s если есть хоть одно enabled cron-правило )`. cron-правила несут `interval_seconds = NULL` (в `MIN` не попадают), их вклад — фиксированные **60s** (cron-гранулярность — минута: чаще ближайшего cron-слота опрашивать незачем). Если interval-правил нет, а cron есть — `derivedMinPeriod = 60s`.
- **Профиль «Спокойный» (дефолт):** `poll_floor = 30s`, `poll_ceiling = 60s`, `poll_idle = 120s`. Частые расписания опрашиваются у floor, редкие (`interval = 1h`) кэпятся ceiling-ом, пустой реестр — idle. Все три — `duration`, hot-reload (читаются из свежего config-снимка в `IntervalFn`, как `interval`/`lock_ttl`). Валидация config: `poll_floor ≥ 30s` (абсолютный минимум), `poll_floor ≤ poll_ceiling`, `poll_idle ≥ poll_ceiling`.
- **`interval` — backcompat-alias.** Старый `cadence_scheduler.interval` сохранён: если задан, а `poll_ceiling` не задан → `poll_ceiling = max(interval, poll_floor)` — **clamp вверх до floor**. Если `interval < poll_floor` (dev-конфиги ставили суб-30s, напр. 5s), ceiling **поднимается до floor** и эмитится **warning-диагностика** (`value_clamped`, НЕ error): «cadence_scheduler.interval Xs ниже poll_floor 30s — поднято до 30s; для суб-30s реакции используйте Beacons (ADR-030)». Старый малый `interval` **не роняет конфиг** через инвариант `poll_floor ≤ poll_ceiling`. Если `interval ≥ poll_floor` — alias работает как прежде (`poll_ceiling = interval`), опрос адаптивен в коридоре `[poll_floor, interval]`. Error `poll_floor > poll_ceiling` остаётся только для случая ЯВНО заданных обоих полей с floor > ceiling (реальная конфиг-ошибка оператора).
- **Idle-вариант (a) — без нового Redis-канала.** Сигнал «реестр пуст» несёт **тот же** агрегатный MIN-запрос (`SELECT MIN(interval_seconds), bool_or(schedule_kind='cron') FROM cadences WHERE enabled` → обе NULL = пусто), отдельного pub/sub-канала «реестр изменился» **не вводится**. Запрос летит **только с Conductor-лидера** (в `IntervalFn` tick-loop-а); не-лидеры `IntervalFn` не зовут — лишней PG-нагрузки нет. Ошибка запроса → fallback на `poll_ceiling` + warn (лидер продолжает тикать).
- **Failover-safe by construction.** `IntervalFn` **stateless**: новый Conductor-лидер после failover пересчитывает шаг из PG (тот же реестр → тот же шаг), in-memory состояния опроса не существует.
- **Инвариант «poll ≤ period ⇒ ни одно правило не недополучает тики».** Адаптив гарантирует, что шаг опроса не превышает минимального периода активных правил, поэтому **рутинных пропусков** тиков (когда `interval` грубее самого частого расписания) больше нет. `NextRunAnchored` missed-slot policy ([ADR-046 §4](0046-cadence.md#adr-046-cadence--регулярные-запуски-scheduledrecurring-voyage)) остаётся **страховкой только на downtime лидера** (catch-up на первый будущий слот сетки после простоя) — два механизма не конфликтуют: адаптив убирает штатные пропуски, anchored grid-skip ловит лишь пропуски из-за реального простоя планировщика.
- **Floor мин-периода (`interval_seconds ≥ 30`)** — отдельный амендмент **ADR-046** (Pass B): write-path floor-reject + DB-CHECK + OpenAPI `minimum: 30`. Здесь, в Conductor, floor работает только как **clamp-нижняя-граница опроса** (`interval_seconds = 10` опрашивается раз в 30s; само создание такой Cadence отвергается на write-path (422) и DB-CHECK — ADR-046 floor, Pass B). См. Pass B / ADR-046.
