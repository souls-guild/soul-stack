# ADR-048. Conductor — leader-elected executor of Cadence schedules

> **Status: canon BEFORE code (C0, 2026-06-02).** Propose-and-wait on the name **Conductor** has passed (chosen by the user). Model design — architect. Slices C1–C5 — implementation. There is no code for this ADR yet: the Cadence trigger currently lives in the Reaper rule `spawn_due_cadence` ([ADR-046 §4](0046-cadence.md#adr-046-cadence--regular-runs-scheduledrecurring-voyage)), migration to Conductor is the subject of this ADR.

**Context.** [ADR-046](0046-cadence.md#adr-046-cadence--regular-runs-scheduledrecurring-voyage) (Cadence) hung the spawn of due schedules on the **Reaper leader** as a new rule `spawn_due_cadence` — this was stretching two different domains onto one executor:

- **Reaper = the cleanup domain** (TTL-purge, recovery-reclaim, retention) — rules for which an infrequent tick is normal: the default `reaper.interval` = **1h** ([reaper.md → Configuration](../keeper/reaper.md#configuration)).
- **Cadence = the scheduling domain** — a schedule needs a **frequent** tick: an operator who ordered "every 5 minutes" or cron `*/2 * * * *`, with a 1h reaper tick, would get a **spawn once an hour instead of once every 5 minutes** — the schedule silently doesn't run at the declared frequency. Lowering `reaper.interval` to 15–30s for Cadence's sake is not an option: it would run **all** cleanup rules (purge/reclaim/scry) hundreds of times an hour needlessly, loading PG and the Redis lease for nothing.

The domains conflict in their natural rhythm. Combining them on one tick means either throttling scheduling or hammering cleanup. A **separate schedule executor** is needed, with its own interval, decoupled from `reaper.interval`.

**Decision (fixed by the user on 2026-06-02).** Move the Cadence spawn out of Reaper into a separate leader-elected subsystem, **Conductor** — the conductor of schedules (the metaphor continues the "musical-soul" line of [Choir](0044-choir.md#adr-044-choir--named-host-topology-inside-an-incarnation)/Voice/[Cadence](0046-cadence.md#adr-046-cadence--regular-runs-scheduledrecurring-voyage): a conductor sets the tempo of performance). Conductor is **only a change of the trigger's executor**; all of Cadence's spawn semantics ([ADR-046 §4/§5](0046-cadence.md#adr-046-cadence--regular-runs-scheduledrecurring-voyage)) carry over **unchanged**.

> **The industry name collision is accepted deliberately (user, 2026-06-02).** "Conductor" is used by Netflix Conductor (a workflow orchestrator) and Temporal/Cadence (Uber) — there "Cadence" is itself the name of a workflow engine. In Soul Stack, Cadence = a schedule, Conductor = its executor; the term overlap with these products is lexical, not semantic. Recorded as acceptable by the user.

1. **Conductor — a separate leader-elected subsystem inside `keeper`** (not a separate binary, like Reaper). It sits on the **generic leaderloop** (`keeper/internal/leaderloop`, extracted in slice C1 — Reaper is moved onto the same shared primitive too). Its own Redis lease ([ADR-006](0006-cache-redis.md#adr-006-cache-and-coordination--redis)) with the key **`conductor:leader`** (separate from `reaper:leader`) → exactly-one spawn executor in the cluster, like Reaper, but with **independent** leadership (the Conductor leader and the Reaper leader can be different instances).

2. **Its own tick interval, independent of `reaper.interval`.** A new config block **`cadence_scheduler`** in `keeper.yml` with its own `interval` (recommendation ~15–30s — frequent enough for Cadence's interval/cron granularity, not so frequent as to load PG like a cleanup loop). On every tick, Conductor executes exactly the due-selection described in [ADR-046 §4](0046-cadence.md#adr-046-cadence--regular-runs-scheduledrecurring-voyage): `SELECT … FROM cadences WHERE enabled AND next_run_at <= NOW() FOR UPDATE SKIP LOCKED` → per due row, in one PG tx: apply `overlap_policy` + insert `voyages`/`voyage_targets` with `cadence_id` + recompute `next_run_at` + `last_run_at`.

3. **Moved wholesale from Reaper — the spawn logic, not rewritten.** `CadenceSpawner` (the logic from `keeper/internal/reaper/cadence_spawn.go`, [ADR-046 §4](0046-cadence.md#adr-046-cadence--regular-runs-scheduledrecurring-voyage)) moves into Conductor **as-is**: the due-selection, the three `overlap_policy` values (`skip`/`queue`/`parallel`, [ADR-046 §5](0046-cadence.md#adr-046-cadence--regular-runs-scheduledrecurring-voyage)), the `next_run_at` recompute (cron parser / interval-anchored), the spawn tx. **Reaper loses the rule `spawn_due_cadence`** and its associated `action: spawn` (the Reaper-action closed enum reverts to `expire`/`delete`/`set_status`/`report`/`soft_delete`). `spawn` was Reaper's only **active** (insert-generating) action — after the move, Reaper is a purely cleanup domain again.

4. **Audit source stays `background`.** Cadence's spawn/skip events (`cadence.spawned` / `cadence.skipped_overlap`, [ADR-046 §8](0046-cadence.md#adr-046-cadence--regular-runs-scheduledrecurring-voyage)) after the move are still written with the same **`source: background`** and `archon_aid: NULL` (an autonomous background initiative of keeper's; recipe authorship still lives in `voyages.started_by_aid` of the child Voyage). **A new audit-source `scheduler` is NOT introduced** — the audit-source enum ([ADR-022(b)](0022-audit-pipeline.md#adr-022-audit-pipeline-storage-schema-retention)) is not extended; `background` already covers "a periodic background keeper rule with no initiating operator," and the spawn semantics haven't changed just because the executor changed.

5. **Default-ON when Redis is available.** Conductor is enabled by default **if Redis is configured** (lease-leadership is possible). This is a footgun guard: without a working scheduler, Cadence would silently not spawn Voyages — the operator created a schedule, and it's "dead" with no visible error. So the scheduler is active out of the box on any Redis installation; disabling Cadence is done per-Cadence via `enabled: false` ([ADR-046 §3](0046-cadence.md#adr-046-cadence--regular-runs-scheduledrecurring-voyage)), not by globally killing Conductor. Without Redis (a single-instance dev setup without Redis), leader election is impossible — Conductor degrades the same way the Reaper leader does on a single instance.

6. **`scry_background` — a Reaper anomaly, left as-is (backlog).** `scry_background` ([ADR-031](0031-scry-drift.md#adr-031-scry--drift-detection-declarative-dry-run-reconcile) Slice C, [reaper.md → Rules](../keeper/reaper.md#rules)) is also an **active** rule (dispatch-generating dry_run runs) sitting in Reaper's cleanup domain under the same stretch that `spawn_due_cadence` was. Properly speaking, a background drift scan is also scheduling, not cleanup. But this ADR does **not** touch it (backlog): Scry has its own cadence (`scry_background.interval`, recommendation 6h — close to the reaper domain, doesn't need a frequent tick like Cadence), and moving it out is a separate decision if a request arises. Recorded as a known anomaly, not as debt of this slice.

**Forward-compat.** Conductor is a new keeper-side subsystem, it breaks nothing. The `cadences` table and the `voyages.cadence_id` column ([ADR-046](0046-cadence.md#adr-046-cadence--regular-runs-scheduledrecurring-voyage)) are unchanged. The Keeper↔Soul proto is **not affected** (Conductor spawns Voyage through the existing path). The Reaper `action` enum **narrows** by removing `spawn` (C4) — this is not breaking for configs: `spawn` was used only by the `spawn_due_cadence` rule, which is removed in the same slice; there are no other rules with `action: spawn` in `keeper.yml`, nor could there be. Before C4, closing off double-spawn is guaranteed by an atomic switch of the executor (the rule in reaper stays inactive/is removed in the same step that enables Conductor).

**Relation to other ADRs.**
- **[ADR-046](0046-cadence.md#adr-046-cadence--regular-runs-scheduledrecurring-voyage) (Cadence)** — Conductor **executes** Cadence schedules; the spawn semantics (model b, overlap policies, next_run_at recompute, single-executor) are preserved verbatim. **Only the trigger's executor** changes: the Reaper leader + `spawn_due_cadence` → the Conductor leader. ADR-046 §4/§8 are amended (the trigger; the `background` audit-source is preserved).
- **[ADR-006](0006-cache-redis.md#adr-006-cache-and-coordination--redis)** — its own Redis lease `conductor:leader` (separate from `reaper:leader`), single-executor spawn, default-ON when Redis is available.
- **[ADR-027](0027-apply-work-queue.md#adr-027-apply-execution-model--work-queue--claim-acolyte-pool-ward-claim) / Reaper** — after the move, Reaper returns to a pure cleanup domain; the Reaper rule list loses `spawn_due_cadence`, the closed-enum action loses `spawn`.
- **[ADR-022](0022-audit-pipeline.md#adr-022-audit-pipeline-storage-schema-retention)** — the audit-source enum is **not extended**: `cadence.spawned`/`cadence.skipped_overlap` remain `source: background`.
- **leaderloop (C1)** — Conductor and Reaper share the generic lease-loop primitive `keeper/internal/leaderloop`; two independent lease keys, two independent leader elections.

**Rejected alternatives.**
- **(a) Keep the spawn in Reaper, lower `reaper.interval` to 15–30s** — would run all cleanup rules hundreds of times an hour needlessly (PG/Redis load, pointless purge scans). The domains have a different natural rhythm — one interval can't serve both.
- **(b) A separate `conductor` binary** — Reaper and Conductor are both keeper-side single-executor subsystems; carving out a binary for one background goroutine contradicts the model ([ADR-004](0004-binaries.md#adr-004-binary-layout--keeper-soul-soul-lint-push-mode-as-a-module-inside-keeper): Reaper isn't a separate binary either). Conductor lives inside `keeper`.
- **(c) A new audit-source `scheduler`** — `background` is already semantically accurate (a periodic background keeper rule with no initiating operator); changing the executor doesn't change the nature of the event. Extending the enum for cosmetics alone is an unnecessary breaking risk.
- **(d) Also move `scry_background` into Conductor while at it** — Scry has a different rhythm (6h, close to reaper) and its own model (a dry_run work queue, not a Voyage spawn); moving it out is a separate decision on request, not part of this slice (see §6).

**What is deferred (post-MVP / backlog).**
- Moving `scry_background` out of the Reaper domain (§6) — if a request/problem with the drift-scan frequency arises.
- A possible generalization of Conductor into a universal scheduler domain (several schedule types on one leader loop) — Cadence is currently its only consumer.

**Adaptive interval (amendment 2026-06-07).** Conductor's poll step is **no longer fixed** at `cadence_scheduler.interval`, but **adaptive** — derived from the real enabled-registry of Cadence on every tick:

- **Step formula:** `clamp(derivedMinPeriod, poll_floor, poll_ceiling)`; an **empty** enabled registry (not a single enabled rule) → `poll_idle`.
- **`derivedMinPeriod`** = `min( MIN(interval_seconds) over enabled interval rules, 60s if there is at least one enabled cron rule )`. cron rules carry `interval_seconds = NULL` (they don't enter the `MIN`), their contribution is a fixed **60s** (cron granularity is a minute — no point polling more often than the nearest cron slot). If there are no interval rules but there are cron ones — `derivedMinPeriod = 60s`.
- **The "Calm" profile (default):** `poll_floor = 30s`, `poll_ceiling = 60s`, `poll_idle = 120s`. Frequent schedules are polled at the floor, rare ones (`interval = 1h`) are capped at the ceiling, an empty registry is idle. All three are `duration`, hot-reload (read from a fresh config snapshot in `IntervalFn`, like `interval`/`lock_ttl`). Config validation: `poll_floor ≥ 30s` (absolute minimum), `poll_floor ≤ poll_ceiling`, `poll_idle ≥ poll_ceiling`.
- **`interval` — a backcompat alias.** The old `cadence_scheduler.interval` is preserved: if set and `poll_ceiling` is not set → `poll_ceiling = max(interval, poll_floor)` — **clamped up to the floor**. If `interval < poll_floor` (dev configs set sub-30s, e.g. 5s), the ceiling is **raised to the floor** and a **warning diagnostic** is emitted (`value_clamped`, NOT an error): "cadence_scheduler.interval Xs is below poll_floor 30s — raised to 30s; for sub-30s reactions use Beacons (ADR-030)." An old small `interval` **does not break the config** through the `poll_floor ≤ poll_ceiling` invariant. If `interval ≥ poll_floor` — the alias works as before (`poll_ceiling = interval`), polling is adaptive within `[poll_floor, interval]`. The error `poll_floor > poll_ceiling` remains only for the case where BOTH fields are EXPLICITLY set with floor > ceiling (a genuine operator config error).
- **The idle variant (a) — no new Redis channel.** The "registry is empty" signal is carried by the **same** aggregate MIN query (`SELECT MIN(interval_seconds), bool_or(schedule_kind='cron') FROM cadences WHERE enabled` → both NULL = empty), no separate "registry changed" pub/sub channel is introduced. The query is issued **only from the Conductor leader** (in the tick loop's `IntervalFn`); non-leaders don't call `IntervalFn` — no extra PG load. A query error → fall back to `poll_ceiling` + warn (the leader keeps ticking).
- **Failover-safe by construction.** `IntervalFn` is **stateless**: the new Conductor leader after a failover recomputes the step from PG (same registry → same step), no in-memory polling state exists.
- **The invariant "poll ≤ period ⇒ no rule is ever under-served."** The adaptive step guarantees that the poll interval never exceeds the minimum period of the active rules, so there are no more **routine** missed ticks (when `interval` is coarser than the most frequent schedule). `NextRunAnchored`'s missed-slot policy ([ADR-046 §4](0046-cadence.md#adr-046-cadence--regular-runs-scheduledrecurring-voyage)) remains a **safety net only for leader downtime** (catch-up to the first future grid slot after an outage) — the two mechanisms don't conflict: the adaptive step removes routine misses, the anchored grid-skip only catches misses from actual scheduler downtime.
- **The minimum-period floor (`interval_seconds ≥ 30`)** — a separate amendment to **ADR-046** (Pass B): a write-path floor-reject + DB CHECK + OpenAPI `minimum: 30`. Here, in Conductor, the floor only works as a **clamp lower bound on polling** (`interval_seconds = 10` is polled once every 30s; creating such a Cadence in the first place is rejected on the write path (422) and by the DB CHECK — ADR-046 floor, Pass B). See Pass B / ADR-046.
