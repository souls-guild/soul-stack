# ADR-006. Cache and Coordination — Redis

- **Context.** An HA Keeper cluster needs: a hot cache for frequently updated fields (Soul heartbeats), a lease/lock mechanism for coordination (which Keeper "holds" a Soul, which Keeper is currently the Reaper), and pub/sub for coordination between instances. Doing all of this on PG leads to an UPDATE storm and heavy lock tables.
- **Decision.** Redis as the horizontal cache and coordination bus of the Keeper cluster. Roles:
  - **(a) Heartbeat cache.** Souls' `last_seen_at` and `last_seen_by_kid` live in Redis; a periodic batched flush to Postgres. The "list of active Souls" query is read from a Redis structure (Set/HSET), not from PG.
  - **(b) SID lease.** Which Keeper holds the active stream to a given Soul (`SET sid:lock kid NX EX <ttl>` with renewal). On disconnect the TTL expires and the next Keeper freely takes over. Force-release on a detected-crash reconnect (presence-gated CAS re-acquisition from a provably-dead holder, shrinking the invisibility window from the SID-lease TTL down to the Conclave-presence TTL) — see [ADR-027(n)](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim).
  - **(c) Pub/Sub.** Signals between Keeper instances: "a new Soul joined", "Destiny updated", "SoulSeed revoked"; moves away from polling PG.
  - **(d) Leader for background jobs.** The Reaper and similar singleton jobs (see the "Reaper" section) take a leader lease in Redis.
- **Rationale.** The standard "PG as source of truth + Redis as a hot layer" pattern — it removes the UPDATE storm and gives cheap TTLs/locks without reinventing the wheel.
- **Trade-off.** Yet another mandatory external dependency in production. We accept this for the same reasons as ADR-005: without Redis, a decently performing HA cluster cannot be built.
- **Cache of rendered Destiny / Soulprint (role d-prime)** — deferred until real load materializes; not part of the MVP.
- **Amendment (2026-05-25, Redis presence role + lease-aware `mark_disconnected`).** **Soul presence is derived from the SID lease (role b)**, not from PG `souls.status`: the target resolver derives the online set from live leases (batch-EXISTS), and PG presence is not written on the hot path (see the [ADR-002](0002-transport-grpc-ha.md#adr-002-transport-keeper--souls--grpc-bidirectional-stream-over-mtls-ha-keeper-cluster) amendment, invariant "hot → Redis"). The `connected`/`disconnected` field in PG `souls.status` is only a **lazy snapshot** for the Operator API; it is reconciled by the `mark_disconnected` reaper rule, which became lease-aware and a **bidirectional** reconcile: (1) `connected`→`disconnected` for candidates with a stale `last_seen` and a dead SID lease; (2) `disconnected`→`connected` for candidates with a live SID lease (the snapshot does not latch after a reconnect — without this direction an online Soul would remain `disconnected` in the snapshot forever). Both phases: PG candidates → Redis lease filter; fail-safe: on a Redis error, neither direction is marked; falls back to pure SQL (one-directional) without a Redis client. The "last seen at T" snapshot for the Operator API is provided by the existing throttled heartbeat flush (role a). This closes the deferred slice "explicit reconciliation with the Redis heartbeat cache" from role (a).
- **Amendment (2026-05-25, Keeper-instance presence — Conclave, new Redis role e).** In addition to Souls presence (role b), Redis now also holds **presence of the Keeper instances themselves** — the **Conclave** registry (`keeper/internal/redis/conclave.go`): every live instance writes the key `keeper:instance:<kid>` with TTL `DefaultConclaveTTL=30s` and renews it via a renewal goroutine every `DefaultConclaveRenewInterval=10s` (the same lease pattern as the SID lease and the reaper leader), removing the key on graceful shutdown. `LiveKIDs()`/`CountLive()` give the current set of live instances; `RegisterInstance` rejects an already-taken KID (`ErrConclaveKIDTaken`) — protection against a duplicate KID. Application: the **refuse-startup-guard** ([ADR-027](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim) (k)/(h)) — an instance with no local Acolyte workers (`acolytes:0`) refuses to start when `Conclave.CountLive()>1` (a dangerous "single-path multi-keeper" setup where the dispatcher cannot execute a claim); opt-out via `allow_unsafe_single_path_multi_keeper` / `KEEPER_ALLOW_UNSAFE_MULTI_KEEPER`. Conclave is also the foundation for soul-shedding coordination (see the [ADR-002](0002-transport-grpc-ha.md#adr-002-transport-keeper--souls--grpc-bidirectional-stream-over-mtls-ha-keeper-cluster) amendment). This closes the open slice "registry of live Keeper instances" from [ADR-027](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim) (h).
- **Amendment (2026-05-25, load snapshot in Conclave for Shepherd — PLANNED/backlog).** For load balancing (**Shepherd**, [ADR-002](0002-transport-grpc-ha.md#adr-002-transport-keeper--souls--grpc-bidirectional-stream-over-mtls-ha-keeper-cluster) amendment), the Conclave record `keeper:instance:<kid>` is extended with an **instance load snapshot** (at minimum, the number of active EventStream streams; optionally, Acolyte queue depth), updated on the same renew tick (10s). Then any instance can derive the fair share `sum of streams / CountLive` and see its own skew. This gives a "basic understanding of the system" (in the user's own words) for both auto-drain and manual/semi-auto, without a separate channel. **Implementation is deferred** along with Shepherd; Keeper-instance presence (above) does not depend on this.
- **Amendment (role (c), cluster-mode SSE/apply-events routing — applybus over pub/sub).** The in-memory `applybus.EventBus` (M0.7.c) lives inside a single Keeper instance. So that an SSE subscriber `GET /mcp/events?apply_id=X` on Keeper-A receives events for a run whose `TaskEvent`/`RunResult`/`ErrandResult` is being streamed by a Soul connected to Keeper-B, the publisher, after local delivery, additionally PUBLISHes the event to Redis pub/sub (role c), and every instance holds a subscription bridge and forwards cross-Keeper events to its local subscribers. A self-filter on `origin_kid` (the KID of the publishing instance) drops the echo of its own publications (otherwise local Publish + Redis echo would cause double delivery). Wire format — a JSON envelope with fields `origin_kid`/`kind`/`apply_id`/`at`/`payload` (payload is `json.RawMessage`, marshaled once on the publisher side). Fire-and-forget semantics: pub/sub without persistence, the subscriber must subscribe before the run starts (the SSE contract "subscribe → tools/call → wait for events").
  - **(c.1) Channel sharding — `events:shard:<n>`, K=256 (S2 applybus bottleneck).** The previous per-applyID channel `apply:<apply_id>` produced exactly as many Redis subscriptions per instance as there were concurrent runs observed via SSE/Errand, and at large scale it ran into `maxclients`. The channel is collapsed into a **fixed set of K=256 shards** `events:shard:<n>`; applyID is deterministically mapped to a shard by the hash **fnv32a(applyID) mod K**. The Redis bridge is now raised **per-shard, not per-applyID**: the first Subscribe for any applyID of a given shard raises one subscription to the shard channel, subsequent applyIDs of the same shard reuse it (refs are counted per shard; the last unsubscribe of the last held applyID of a shard closes the subscription). The forward loop filters incoming events by `envelope.apply_id` and delivers them **only** to the local subscribers of the matching applyID — a collision of two runs in one shard (frequency ≈ 1/K) does not mix their payloads. The prefix `events` (instead of `apply`) was chosen as not tied to a single family of opaque IDs: the channel carries both `apply.*` and `errand.*` events. K=256 is a compromise between "events of different runs almost never share a forward loop" and "a bounded number of Redis subscriptions per instance regardless of the number of Souls". Implementation — `keeper/internal/redis/applybus.go` (`ApplyBusShardCount`/`ApplyBusShardIndex`/`ApplyBusChannel`) + `keeper/internal/applybus/bus.go` (per-shard `bridges`, filter in `deliverFromCluster`).
  - **(c.2) holder==self → skip per-applyID bridge (S1).** A caller that already knows the event will only come from a local publisher of the same instance (the lease holder of the target SID == self-KID) subscribes via `SubscribeWithBridge(applyID, wantBridge=false)` — this drops the Redis Subscribe for that run (local delivery works through the same instance). If the holder is unknown or may change, the caller must request `wantBridge=true` (conservatively), otherwise a cross-Keeper event will not reach the subscription (on a holder flip after a skip decision, the dispatcher terminates via the `timed_out` wait-timer instead of hanging). The `heldBridge` invariant: shard refs are incremented/decremented only by held subscribers (`wantBridge=true`); local-only subscribers do not touch refs — otherwise it would drop the bridge of a neighbor in the same shard.
  - **(c.3) Forward-compat protocol for wire changes (a NORM for the future, NOT applied in the first release).** Any future incompatible change to the pub/sub channel's wire format (prefix rename, K change, envelope format) is done via a **two-release protocol**: release N — instances publish to both the old AND the new channel, and read both (dual-write/dual-read); release N+1 — the old channel is removed. This prevents event loss during a rolling restart of the cluster, where instances of two versions are alive at the same time. In the **first** release the protocol is NOT applied (greenfield — a single wire format is rolled out immediately, there is no "before" version in production); it is fixed as a mandatory norm for all subsequent wire changes to the applybus channel.
- **Amendment (2026-06-22, Redis topology — standalone / sentinel / cluster + vault-resolve for the password).** The keeper-side Redis client (`keeper/internal/redis`) supports three topologies, selected by the explicit field `redis.mode` in `keeper.yml` (empty/omitted = `standalone` — forward-compat for existing configs):
  - **standalone** (`redis.NewClient`) — a single node, address in `redis.addr` (as before).
  - **sentinel** (`redis.NewFailoverClient`) — Redis Sentinel HA: the client finds the master through sentinel nodes. Requires `redis.master_name` + `redis.sentinels` (sentinel node addresses host:port); `redis.addr` is not used in this mode. The password of the sentinel nodes themselves is `redis.sentinel_password_ref` (optional).
  - **cluster** (`redis.NewClusterClient`) — Redis Cluster (slot-based sharding). Requires `redis.nodes` (node addresses host:port for bootstrap discovery); `redis.addr`/sentinel fields are not used.

    All three implement `redis.UniversalClient`, so the neighboring subsystems of the package (lease/soullease/conclave/herald/applybus/ratelimit) work uniformly through the shared wrapper `*keeperredis.Client`; external signatures were not changed.

    **Cluster blockers (mandatory, otherwise cluster breaks silently/loudly):**
    - **Herald reliable queue (`heralddelivery.go`).** `Claim` performs an atomic `BRPOPLPUSH pending→processing` — both KEYS must live in the same slot, otherwise Redis Cluster rejects the multi-key command with `CROSSSLOT`. All delivery-queue keys carry a common hash tag `{q}` (`herald:delivery:{q}:pending` / `…:{q}:processing` / `…:{q}:lease:<id>`), so the entire delivery keyspace lands in one slot. In standalone/sentinel the hash tag is just part of the key name.
    - **Conclave presence registry (`conclave.go`).** Presence keys `keeper:instance:<kid>` of different KIDs deliberately do NOT share a hash tag (otherwise the entire presence keyspace would land on one node) → they are sharded across slots/nodes. A plain `SCAN` on `*redis.ClusterClient` only walks ONE node and would undercount presence, silently breaking the "I'm not alone" refuse-guard (ADR-027) and soul-shedding. So `LiveKIDs`/`CountLive` in cluster mode do a per-master SCAN via `ClusterClient.ForEachMaster` and merge the result (type-switch on the actual `underlying()` type); standalone/sentinel keep the previous single SCAN.
    - **pub/sub — classic broadcast.** The event channel (role c, shards `events:shard:<n>`) and other PUBLISH/SUBSCRIBE remain ordinary (broadcast is correct in cluster). Sharded `SPUBLISH`/`SSUBSCRIBE` is a separate GA slice, NOT introduced in this change.
    - Single-key Lua scripts (`lease.go`/`soullease.go`/`ratelimit.go`) are cluster-safe by construction (one KEYS → one slot), untouched.

    **Vault-resolve for the password.** `redis.password_ref` / `redis.sentinel_password_ref` in the form `vault:<mount>/<path>[#field]` are resolved at `NewClient` via the keeper vault client (brought up before Redis in `setupVault`), symmetric with `pg.ResolveDSN`: default field `password`, override via `#field`; a `vault:` ref without a vault client → error `ErrVaultClientRequired`; a non-`vault:` string is plaintext as-is (tests). The former stub `ErrPasswordResolveNotImplemented` was removed.
