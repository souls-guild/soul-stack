# ADR-006. Кэш и координация — Redis

- **Контекст.** При HA-кластере Keeper-а нужны: горячий кэш для часто обновляемых полей (heartbeat Souls), механизм lease/lock для координации (кто из Keeper-ов «держит» Soul, кто из Keeper-ов сейчас Жнец), и pub/sub для согласования между инстансами. Делать всё это на PG ведёт к UPDATE-шторму и тяжёлым lock-таблицам.
- **Решение.** Redis как horizontal-кэш и координационная шина Keeper-кластера. Роли:
  - **(a) Heartbeat-кэш.** `last_seen_at` и `last_seen_by_kid` Souls живут в Redis; периодический flush в Postgres батчем. Запрос «список активных Souls» читается из Redis-структуры (Set/HSET), не из PG.
  - **(b) Lease на SID.** Какой Keeper держит активный стрим к данному Soul (`SET sid:lock kid NX EX <ttl>` с продлением). При разрыве — TTL истекает, следующий Keeper свободно занимает. Force-release при detected-crash reconnect (presence-gated CAS-перезахват у доказанно-мёртвого holder-а, сокращает окно невидимости с TTL SID-lease до TTL Conclave-presence) — см. [ADR-027(n)](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim).
  - **(c) Pub/Sub.** Сигналы между Keeper-инстансами: «новый Soul зашёл», «Destiny обновлён», «отзыв SoulSeed»; уход от поллинга PG.
  - **(d) Лидер для фоновых задач.** Reaper и подобные одиночные задачи (см. раздел «Reaper / Жнец») берут лидерский lease в Redis.
- **Обоснование.** Стандартный паттерн «PG как source of truth + Redis как горячий слой» — снимает UPDATE-шторм и даёт дешёвые TTL/lock без нашего велосипеда.
- **Trade-off.** Ещё одна обязательная внешняя зависимость в продакшене. Принимаем по тем же причинам, что в ADR-005: без Redis HA-кластер с приличной производительностью не делается.
- **Кэш rendered Destiny / Soulprint (роль d-prime)** — отложен до появления реальной нагрузки; не часть MVP.
- **Amendment (2026-05-25, presence-роль Redis + `mark_disconnected` lease-aware).** **Presence Soul-а — производная от SID-lease (роль b)**, а не от PG-`souls.status`: таргет-резолвер деривирует online-набор из живых lease (batch-EXISTS), PG-presence на горячем пути не пишется (см. [ADR-002](0002-transport-grpc-ha.md#adr-002-транспорт-keeper--souls--grpc-bidirectional-stream-поверх-mtls-ha-кластер-keeper) amendment, инвариант «горячее → Redis»). Часть `connected`/`disconnected` в PG-`souls.status` — лишь **ленивый snapshot** для Operator API; его согласует reaper-правило `mark_disconnected`, ставшее lease-aware и **двунаправленным** reconcile: (1) `connected`→`disconnected` для кандидатов со stale `last_seen` и мёртвым SID-lease; (2) `disconnected`→`connected` для кандидатов с живым SID-lease (снимок не латчится после реконнекта — без этого направления online-Soul навсегда оставался бы `disconnected` в снимке). Обе фазы: PG-кандидаты → Redis lease-фильтр; fail-safe: при ошибке Redis не метит ни в одну сторону; fallback на чисто-SQL (одностороннее) без Redis-клиента. Snapshot «last seen at T» для Operator API даёт существующий throttled heartbeat-flush (роль a). Закрывает отложенный slice «явная сверка с Redis heartbeat-кэшем» из роли (a).
- **Amendment (2026-05-25, presence Keeper-инстансов — Conclave, новая Redis-роль e).** Помимо presence Souls (роль b), Redis теперь держит **presence самих Keeper-инстансов** — реестр **Conclave** (`keeper/internal/redis/conclave.go`): каждый живой инстанс пишет ключ `keeper:instance:<kid>` с TTL `DefaultConclaveTTL=30s` и продлевает renewal-goroutine каждые `DefaultConclaveRenewInterval=10s` (тот же lease-паттерн, что SID-lease и reaper-лидер), снимает ключ на graceful-shutdown. `LiveKIDs()`/`CountLive()` дают актуальный набор живых инстансов; `RegisterInstance` отвергает занятый KID (`ErrConclaveKIDTaken`) — защита от дубль-KID. Применение: **refuse-startup-guard** ([ADR-027](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim) (k)/(h)) — инстанс без локальных Acolyte-воркеров (`acolytes:0`) при `Conclave.CountLive()>1` отказывается стартовать (опасный «single-path multi-keeper» сетап, где диспетчер не может исполнить claim); opt-out через `allow_unsafe_single_path_multi_keeper` / `KEEPER_ALLOW_UNSAFE_MULTI_KEEPER`. Conclave — также основа для soul-shedding-координации (см. [ADR-002](0002-transport-grpc-ha.md#adr-002-транспорт-keeper--souls--grpc-bidirectional-stream-поверх-mtls-ha-кластер-keeper) amendment). Закрывает open-slice «реестр живых Keeper-инстансов» из [ADR-027](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim) (h).
- **Amendment (2026-05-25, снимок нагрузки в Conclave для Shepherd — PLANNED/backlog).** Под балансировку нагрузки (**Shepherd**, [ADR-002](0002-transport-grpc-ha.md#adr-002-транспорт-keeper--souls--grpc-bidirectional-stream-поверх-mtls-ha-кластер-keeper) amendment) Conclave-запись `keeper:instance:<kid>` расширяется **снимком нагрузки инстанса** (минимум — число активных EventStream-стримов; опц. — глубина Acolyte-очереди), обновляемым тем же renew-тиком (10s). Тогда любой инстанс деривирует справедливую долю `сумма стримов / CountLive` и видит свой перекос. Это даёт «базовое понимание системы» (по формулировке пользователя) и для авто-drain, и для ручного/полу-авто, без отдельного канала. **Реализация отложена** вместе с Shepherd; presence keeper-инстанса (выше) от этого не зависит.
- **Amendment (роль (c), cluster-mode SSE/apply-events routing — applybus поверх pub/sub).** In-memory `applybus.EventBus` (M0.7.c) живёт внутри одного Keeper-инстанса. Чтобы SSE-подписчик `GET /mcp/events?apply_id=X` на Keeper-A получал события прогона, чьи `TaskEvent`/`RunResult`/`ErrandResult` льёт Soul, подключённый к Keeper-B, publisher после local-доставки дополнительно PUBLISH-ит событие в Redis pub/sub (роль c), а каждый инстанс держит подписку-bridge и форвардит cross-Keeper-события в свои local-subscriber-ы. Self-filter по `origin_kid` (KID публикующего инстанса) отсекает эхо собственных публикаций (иначе local-Publish + Redis-echo дали бы двойную доставку). Wire-формат — JSON-envelope с полями `origin_kid`/`kind`/`apply_id`/`at`/`payload` (payload — `json.RawMessage`, маршалится один раз на стороне publisher-а). Семантика fire-and-forget: pub/sub без persistence, подписчик обязан подписаться до старта прогона (контракт SSE «subscribe → tools/call → ждать events»).
  - **(c.1) Шардирование канала — `events:shard:<n>`, K=256 (S2 applybus-bottleneck).** Прежний per-applyID канал `apply:<apply_id>` давал ровно столько Redis-подписок на инстанс, сколько одновременных прогонов наблюдается через SSE/Errand, и на больших флотах упирался в `maxclients`. Канал сведён к **фиксированному множеству из K=256 шардов** `events:shard:<n>`; applyID детерминированно отображается в шард хешем **fnv32a(applyID) mod K**. Redis-bridge теперь поднимается **per-shard, а не per-applyID**: первый Subscribe любого applyID данного shard-а поднимает одну подписку на shard-канал, последующие applyID того же shard-а её переиспользуют (refs считаются по shard-у; last-unsubscribe последнего held-applyID шарда закрывает подписку). Forward-loop фильтрует входящие события по `envelope.apply_id` и раздаёт **только** в local-subscriber-ов соответствующего applyID — коллизия двух прогонов в один shard (частота ≈ 1/K) не смешивает их payload-ы. Префикс `events` (вместо `apply`) выбран как не привязанный к одному семейству opaque-id: канал несёт и `apply.*`, и `errand.*`-события. K=256 — компромисс «события разных прогонов почти не делят forward-loop» против «ограниченное число Redis-подписок на инстанс независимо от размера флота». Реализация — `keeper/internal/redis/applybus.go` (`ApplyBusShardCount`/`ApplyBusShardIndex`/`ApplyBusChannel`) + `keeper/internal/applybus/bus.go` (per-shard `bridges`, фильтр в `deliverFromCluster`).
  - **(c.2) holder==self → skip per-applyID bridge (S1).** Caller, заранее знающий, что событие придёт только от local-publisher-а того же инстанса (lease-holder целевого SID == self-KID), подписывается через `SubscribeWithBridge(applyID, wantBridge=false)` — это снимает Redis-Subscribe для такого прогона (local-доставка работает через тот же инстанс). Если holder неизвестен или может смениться — caller обязан запросить `wantBridge=true` (консервативно), иначе cross-Keeper-событие до подписки не дойдёт (при holder-flip после skip-решения dispatcher завершается по wait-timer-у `timed_out`, не зависает). `heldBridge`-инвариант: refs шарда инкрементит/декрементит только held-subscriber (`wantBridge=true`); local-only subscriber refs не трогает — иначе уронил бы bridge соседа того же shard-а.
  - **(c.3) Forward-compat протокол wire-изменений (НОРМА на будущее, в первом релизе НЕ применяется).** Любое будущее несовместимое изменение wire-формата pub/sub-канала (переименование префикса, смена K, формат envelope) делается **двухрелизным протоколом**: релиз N — инстансы публикуют в старый И новый канал, читают оба (dual-write/dual-read); релиз N+1 — старый канал удаляется. Это исключает потерю событий при rolling-restart кластера, где одновременно живут инстансы двух версий. В **первом** релизе протокол НЕ применяется (greenfield — единый wire-формат разворачивается сразу, версий «до» в проде нет); зафиксирован как обязательная норма для всех последующих wire-изменений applybus-канала.
