# ADR-025. Augur — Keeper-side брокер внешнего доступа Soul

> **Статус: MVP-1 реализован (2026-05), в baseline; MVP-2 (делегация) отложен.** Полная нормативная спека — [`docs/keeper/augur.md`](../keeper/augur.md).
>
> **Amendment 2026-06-16 (статус ↔ код).** Исходная шапка «это дизайн, имплементации нет» устарела — MVP-1 (брокер, `delegate=false`) реализован и в baseline. **Закрыто (MVP-1):**
> - **Сущности и реестры** — таблицы `omens` (миграция [`032_create_omens`](../../keeper/migrations/032_create_omens.up.sql)) и `rites` (миграция [`033_create_rites`](../../keeper/migrations/033_create_rites.up.sql)), CRUD + валидация формы (`source_type` closed-enum `vault`/`prometheus`/`elk`, allow-shape по типу, token-поля только vault-delegate) — [`keeper/internal/augur/`](../../keeper/internal/augur/) (`augur.go`/`crud.go`).
> - **Брокер (`delegate=false`)** — keeper-side резолв (Omen → SID→covens → Rite → allow-list) и per-source fetch Vault / Prometheus / ELK через SSRF-guarded egress, данные текут inline через Keeper — [`resolve.go`](../../keeper/internal/augur/resolve.go), `broker_*.go`, `egress.go`, `service.go`, `metrics.go`.
> - **Транспорт** — `proto/keeper/v1/augur.proto` (`AugurRequest`/`AugurReply`, only-add в `EventStream`-oneof); `ScopedVaultToken`/`ScopedStaticCred` присутствуют **как контракт** (резолв их не выдаёт — см. ниже).
> - **Soul-side `core.augur.fetch`** — read-probe (`changed=false` конструктивно), зарегистрирован в Soul-side registry — [`soul/internal/coremod/augur/augur.go`](../../soul/internal/coremod/augur/augur.go), README — [`docs/module/core/augur/README.md`](../module/core/augur/README.md).
> - **CRUD / RBAC / audit-поверхность** — OpenAPI-роуты `/v1/augur/omens` + `/v1/augur/rites` (huma full-typed) и MCP, permission-семьи `omen.*` / `rite.*`.
>
> **Отложено (MVP-2, делегация `delegate=true`):** минтинг scoped Vault-токена (`auth/token/create` под master-AppRole) и scoped static read-cred для prom/elk; зависит от Vault-AppRole ([ADR-014](0014-operator-identity.md#adr-014-identity-модель-оператора-archon)). На уровне кода `delegate=true` Rite **резолвится в `denied`** (см. [`resolve.go`](../../keeper/internal/augur/resolve.go)), proto-сообщения `ScopedVaultToken`/`ScopedStaticCred` заведены, но не заполняются. Порядок реализации MVP-2 относительно прочего backlog-а — за PM/пользователем.

- **Контекст.** Действующая модель доступа Soul-а к внешним системам — **pre-resolved**: всё (CEL-фаза, `vault(...)`) резолвится Keeper-side **до** отправки `ApplyRequest` на Soul ([ADR-012(d)](0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add)); на Soul нет Vault-токенов / `cel-go` / Vault-клиента. Эта модель **не покрывает** случаи, когда внешнее значение нужно **live** — во время рендера / apply, а не заранее: короткоживущий dynamic-secret Vault (протухнет при resolve на Keeper-е заранее), live-query в Prometheus как условие шага, чтение из ELK в ходе прогона. Нужен узкий авторизованный канал «Soul просит внешнее значение прямо сейчас», не ломающий security-инвариант «master-credential не на Soul».
- **Решение.** Вводится **Augur** — Keeper-side подсистема-брокер внешнего доступа. Реализуется в **две фазы** (один ADR):
  - **(a) Сущности.** **Augur** (брокер), **Omen** (внешняя система — vault/prometheus/elk; реестр Postgres `omens`, аналог Provider), **Rite** (grant/policy-mapping: субъект `coven` XOR `sid` × Omen → allow-list + `delegate` + ttl/uses; реестр Postgres `rites`). Внесены в [naming-rules.md](../naming-rules.md). Managed через OpenAPI / MCP, как Provider / Profile.
  - **(b) MVP-1 — брокер (`delegate=false`).** Soul просит через Keeper, Keeper сам читает (Vault — существующий `ReadKV`) и возвращает значение inline через себя. Данные текут через Keeper; токен/credential на Soul **не попадает**; AppRole не нужен. **Границу [ADR-012(d)](0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add) НЕ трогает** — меняется только момент доступа (live), не сторона.
  - **(c) MVP-2 — делегация (`delegate=true`).** Vault: Keeper **минтит** scoped / short-TTL / limited-use Vault-токен (`auth/token/create` под master-AppRole) → Soul читает Vault напрямую (salt-vault-модель); требует AppRole + token-create в `keeper/internal/vault`. Prometheus / ELK: Vault-style минтинга нет — делегация = scoped **статический** pre-scoped read-key (отдельный read-key в `auth_ref`, **не** master-cred).
  - **(d) Транспорт — only-add, без нового RPC.** Два сообщения `AugurRequest` / `AugurReply` добавляются в `oneof payload` существующего `EventStream` ([ADR-012(c)](0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add) forward-compat only-add; [ADR-002](0002-transport-grpc-ha.md#adr-002-транспорт-keeper--souls--grpc-bidirectional-stream-поверх-mtls-ha-кластер-keeper) «один стрим» соблюдён), новый файл `proto/keeper/v1/augur.proto`. `AugurRequest{request_id, apply_id, omen_name, query}` (SID — из mTLS peer cert, не в payload, [ADR-012(i)](0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add)); `AugurReply{request_id, status: ok/denied/error, oneof result: inline_data | scoped_vault_token | scoped_static_cred, error}`.
  - **(e) Авторизация (Keeper).** Omen существует → Soul (по mTLS→SID→covens) в Rite → query в allow-list → ветвление по `delegate` / `source_type`. Иначе — `denied`.
- **Нормативный инвариант безопасности.** **Soul НИКОГДА не получает master-credential** — только эфемерный scoped-токен (vault) / scoped read-only static cred (prom/elk), либо вообще не получает credential (`delegate=false` — inline через Keeper). `delegate=true` (MVP-2) — **узкое осознанное исключение из [ADR-012(d)](0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add)**: на Soul появляется минимальный fetch-клиент, но master-cred / `cel-go` / sprig-render-контекст по-прежнему не на Soul. MVP-1 границу ADR-012(d) не трогает.
- **Реконсиляция ADR.** [ADR-017](0017-keeper-side-core.md#adr-017-keeper-side-core-модули-расширены-corecloudprovisioned-corevaultkv-read) (`core.vault.kv-read`) — **обобщается Augur-ом**: kv-read остаётся для render-фазы (pre-resolve), Augur — live-доступ при apply (два разных момента). [ADR-014](0014-operator-identity.md#adr-014-identity-модель-оператора-archon) (Vault AppRole — post-MVP) → AppRole становится **зависимостью MVP-2**. [ADR-012(d)](0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add) получает пункт-исключение для delegate-режима. [ADR-002](0002-transport-grpc-ha.md#adr-002-транспорт-keeper--souls--grpc-bidirectional-stream-поверх-mtls-ha-кластер-keeper) соблюдён (only-add).
- **Consequences.**
  - **Новый файл [`docs/keeper/augur.md`](../keeper/augur.md)** — полная нормативная спека (2 фазы, модель данных `omens` / `rites`, транспорт, авторизация, audit, дизайн-форки).
  - **Имена в [naming-rules.md](../naming-rules.md)** — Augur / Omen / Rite (+ определения), `source_type` enum (`vault` / `prometheus` / `elk`), proto-имена (`AugurRequest` / `AugurReply` / `ScopedVaultToken` / `ScopedStaticCred`), RBAC-perms (`omen.create/list/delete`, `rite.create/list/delete`), audit-events (`augur.fetch_brokered` / `augur.token_minted` / `augur.cred_issued` / `augur.access_denied` + `omen.created/revoked` + `rite.created/revoked`), PG-таблицы (`omens` / `rites`).
  - **CRUD / RBAC / audit-поверхность** — OpenAPI + MCP-tools для Omen / Rite стартуют как **stub-каталог** (записи в operator-api.md / mcp-tools.md / rbac.md по факту имплементации). Live-fetch от Soul пишется в audit-категорию `soul_grpc` ([ADR-022(b)](0022-audit-pipeline.md#adr-022-audit-pipeline-storage-schema-retention)), `archon_aid: NULL`, `correlation_id = apply_id`; **нового значения в `source` enum не вводит**.
  - **Дизайн-форки (приняты).** Rite → Omen `ON DELETE CASCADE`; минтуемый Vault-токен — orphan (`no_parent`, переживает Keeper-restart); prom / elk delegate — через отдельный pre-scoped read-key в `auth_ref` (не master); response-wrapping — post-MVP hardening.
- **Trade-offs.**
  - **Дубль каналов доступа к Vault** (pre-resolve `core.vault.kv-read` + live Augur) — два пути к Vault. Принимаем: разные моменты (рендер vs apply), каждый под свой use case; Augur не заменяет pre-resolved-дефолт.
  - **MVP-2 ослабляет [ADR-012(d)](0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add)** на узкий fetch-клиент на Soul. Принимаем осознанно: master-доступ остаётся за Keeper-ом, Soul получает только эфемерный scoped credential; компенсируется коротким TTL / num_uses, дефолтом `delegate=false` и явным opt-in-ом оператора на Rite.
  - **Orphan-токен не отзывается каскадом** при revoke родителя — цена за переживание Keeper-restart. Компенсируется коротким TTL / num_uses из Rite.
