# ADR-025. Augur — Keeper-side broker for Soul external access

> **Status: MVP-1 implemented (2026-05), in baseline; MVP-2 (delegation) deferred.** Full normative spec — [`docs/keeper/augur.md`](../keeper/augur.md).
>
> **Amendment 2026-06-16 (status ↔ code).** The original header "this is design, no implementation" is outdated — MVP-1 (broker, `delegate=false`) is implemented and in baseline. **Closed (MVP-1):**
> - **Entities and registries** — tables `omens` (migration [`032_create_omens`](../../keeper/migrations/032_create_omens.up.sql)) and `rites` (migration [`033_create_rites`](../../keeper/migrations/033_create_rites.up.sql)), CRUD + form validation (`source_type` closed-enum `vault`/`prometheus`/`elk`, allow-shape by type, token fields only for vault-delegate) — [`keeper/internal/augur/`](../../keeper/internal/augur/) (`augur.go`/`crud.go`).
> - **Broker (`delegate=false`)** — keeper-side resolve (Omen → SID→covens → Rite → allow-list) and per-source fetch of Vault / Prometheus / ELK via SSRF-guarded egress, data flows inline through the Keeper — [`resolve.go`](../../keeper/internal/augur/resolve.go), `broker_*.go`, `egress.go`, `service.go`, `metrics.go`.
> - **Transport** — `proto/keeper/v1/augur.proto` (`AugurRequest`/`AugurReply`, only-add in the `EventStream` oneof); `ScopedVaultToken`/`ScopedStaticCred` are present **as a contract** (resolve does not issue them — see below).
> - **Soul-side `core.augur.fetch`** — read-probe (`changed=false` by construction), registered in the Soul-side registry — [`soul/internal/coremod/augur/augur.go`](../../soul/internal/coremod/augur/augur.go), README — [`docs/module/core/augur/README.md`](../module/core/augur/README.md).
> - **CRUD / RBAC / audit surface** — OpenAPI routes `/v1/augur/omens` + `/v1/augur/rites` (huma full-typed) and MCP, permission families `omen.*` / `rite.*`.

> **Deferred (MVP-2, delegation `delegate=true`):** minting of a scoped Vault token (`auth/token/create` under the master AppRole) and a scoped static read-cred for prom/elk; depends on the Vault AppRole ([ADR-014](0014-operator-identity.md#adr-014-operator-identity-model-archon)). At the code level a `delegate=true` Rite **resolves to `denied`** (see [`resolve.go`](../../keeper/internal/augur/resolve.go)), the proto messages `ScopedVaultToken`/`ScopedStaticCred` are defined but not populated. The implementation order of MVP-2 relative to the rest of the backlog is up to PM/the user.

- **Context.** The current model of Soul access to external systems is **pre-resolved**: everything (the CEL phase, `vault(...)`) is resolved Keeper-side **before** the `ApplyRequest` is sent to the Soul ([ADR-012(d)](0012-keeper-soul-grpc.md#adr-012-keepersoul-grpc-contract-one-eventstream-with-oneof-keeper-side-render-forward-compat-only-add)); the Soul has no Vault tokens / `cel-go` / Vault client. This model **does not cover** cases where an external value is needed **live** — during render / apply, not in advance: a short-lived Vault dynamic-secret (would expire if resolved on the Keeper ahead of time), a live query to Prometheus as a step condition, a read from ELK during a run. A narrow authorized channel is needed for "the Soul requests an external value right now" that does not break the security invariant "the master credential is not on the Soul".
- **Decision.** **Augur** is introduced — a Keeper-side external-access broker subsystem. Implemented in **two phases** (one ADR):
  - **(a) Entities.** **Augur** (broker), **Omen** (external system — vault/prometheus/elk; Postgres registry `omens`, analog of Provider), **Rite** (grant/policy-mapping: subject `coven` XOR `sid` × Omen → allow-list + `delegate` + ttl/uses; Postgres registry `rites`). Added to [naming-rules.md](../naming-rules.md). Managed via OpenAPI / MCP, like Provider / Profile.
  - **(b) MVP-1 — broker (`delegate=false`).** The Soul requests through the Keeper, the Keeper reads it itself (Vault — the existing `ReadKV`) and returns the value inline through itself. Data flows through the Keeper; the token/credential **does not reach** the Soul; no AppRole is needed. **It does NOT touch the [ADR-012(d)](0012-keeper-soul-grpc.md#adr-012-keepersoul-grpc-contract-one-eventstream-with-oneof-keeper-side-render-forward-compat-only-add) boundary** — only the moment of access changes (live), not the side.
  - **(c) MVP-2 — delegation (`delegate=true`).** Vault: the Keeper **mints** a scoped / short-TTL / limited-use Vault token (`auth/token/create` under the master AppRole) → the Soul reads Vault directly (salt-vault model); requires an AppRole + token-create in `keeper/internal/vault`. Prometheus / ELK: no Vault-style minting — delegation = a scoped **static** pre-scoped read-key (a separate read-key in `auth_ref`, **not** the master cred).
  - **(d) Transport — only-add, no new RPC.** Two messages `AugurRequest` / `AugurReply` are added to the `oneof payload` of the existing `EventStream` ([ADR-012(c)](0012-keeper-soul-grpc.md#adr-012-keepersoul-grpc-contract-one-eventstream-with-oneof-keeper-side-render-forward-compat-only-add) forward-compat only-add; [ADR-002](0002-transport-grpc-ha.md#adr-002-transport-keeper--souls--grpc-bidirectional-stream-over-mtls-ha-keeper-cluster) "one stream" preserved), new file `proto/keeper/v1/augur.proto`. `AugurRequest{request_id, apply_id, omen_name, query}` (SID — from the mTLS peer cert, not in the payload, [ADR-012(i)](0012-keeper-soul-grpc.md#adr-012-keepersoul-grpc-contract-one-eventstream-with-oneof-keeper-side-render-forward-compat-only-add)); `AugurReply{request_id, status: ok/denied/error, oneof result: inline_data | scoped_vault_token | scoped_static_cred, error}`.
  - **(e) Authorization (Keeper).** Omen exists → the Soul (by mTLS→SID→covens) is in a Rite → the query is in the allow-list → branch by `delegate` / `source_type`. Otherwise — `denied`.
- **Normative security invariant.** **The Soul NEVER receives a master credential** — only an ephemeral scoped token (vault) / a scoped read-only static cred (prom/elk), or no credential at all (`delegate=false` — inline through the Keeper). `delegate=true` (MVP-2) is a **narrow deliberate exception to [ADR-012(d)](0012-keeper-soul-grpc.md#adr-012-keepersoul-grpc-contract-one-eventstream-with-oneof-keeper-side-render-forward-compat-only-add)**: a minimal fetch client appears on the Soul, but the master cred / `cel-go` / sprig render context are still not on the Soul. MVP-1 does not touch the ADR-012(d) boundary.
- **ADR reconciliation.** [ADR-017](0017-keeper-side-core.md#adr-017-keeper-side-core-modules-extended-corecloudprovisioned-corevaultkv-read) (`core.vault.kv-read`) — **is generalized by Augur**: kv-read remains for the render phase (pre-resolve), Augur — live access at apply (two different moments). [ADR-014](0014-operator-identity.md#adr-014-operator-identity-model-archon) (Vault AppRole — post-MVP) → the AppRole becomes a **dependency of MVP-2**. [ADR-012(d)](0012-keeper-soul-grpc.md#adr-012-keepersoul-grpc-contract-one-eventstream-with-oneof-keeper-side-render-forward-compat-only-add) gets an exception clause for the delegate mode. [ADR-002](0002-transport-grpc-ha.md#adr-002-transport-keeper--souls--grpc-bidirectional-stream-over-mtls-ha-keeper-cluster) is preserved (only-add).
- **Consequences.**
  - **New file [`docs/keeper/augur.md`](../keeper/augur.md)** — full normative spec (2 phases, the `omens` / `rites` data model, transport, authorization, audit, design forks).
  - **Names in [naming-rules.md](../naming-rules.md)** — Augur / Omen / Rite (+ definitions), the `source_type` enum (`vault` / `prometheus` / `elk`), proto names (`AugurRequest` / `AugurReply` / `ScopedVaultToken` / `ScopedStaticCred`), RBAC perms (`omen.create/list/delete`, `rite.create/list/delete`), audit events (`augur.fetch_brokered` / `augur.token_minted` / `augur.cred_issued` / `augur.access_denied` + `omen.created/revoked` + `rite.created/revoked`), PG tables (`omens` / `rites`).
  - **CRUD / RBAC / audit surface** — OpenAPI + MCP tools for Omen / Rite start as a **stub catalog** (entries in operator-api.md / mcp-tools.md / rbac.md as actually implemented). A live fetch from a Soul is written to the audit category `soul_grpc` ([ADR-022(b)](0022-audit-pipeline.md#adr-022-audit-pipeline-storage-schema-retention)), `archon_aid: NULL`, `correlation_id = apply_id`; it **introduces no new value in the `source` enum**.
  - **Design forks (accepted).** Rite → Omen `ON DELETE CASCADE`; the minted Vault token is an orphan (`no_parent`, survives a Keeper restart); prom / elk delegate — via a separate pre-scoped read-key in `auth_ref` (not the master); response-wrapping — post-MVP hardening.
- **Trade-offs.**
  - **Duplicate Vault access channels** (pre-resolve `core.vault.kv-read` + live Augur) — two paths to Vault. Accepted: different moments (render vs apply), each for its own use case; Augur does not replace the pre-resolved default.
  - **MVP-2 weakens [ADR-012(d)](0012-keeper-soul-grpc.md#adr-012-keepersoul-grpc-contract-one-eventstream-with-oneof-keeper-side-render-forward-compat-only-add)** down to a narrow fetch client on the Soul. Accepted deliberately: master access stays with the Keeper, the Soul receives only an ephemeral scoped credential; compensated by a short TTL / num_uses, the `delegate=false` default, and an explicit operator opt-in per Rite.
  - **The orphan token is not cascade-revoked** when its parent is revoked — the price of surviving a Keeper restart. Compensated by a short TTL / num_uses from the Rite.
