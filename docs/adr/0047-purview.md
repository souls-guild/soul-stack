# ADR-047. Purview — scoped RBAC visibility of nodes (role default_scope + extended selector)

**Context.** Before this ADR, RBAC scope existed in exactly one dimension and one place: `Enforcer.CovenScope` ([`keeper/internal/rbac/enforcer.go`](../../keeper/internal/rbac/enforcer.go)) returned `(covens []string, unrestricted bool)` — a union of `coven=` values across matching permissions plus a bool "unrestricted". That's enough for bulk-coven-mutation, but doesn't cover a real need: a `db-operator` operator still sees **all** Souls on `GET /v1/souls`, not just their own scope; `incarnation.list`, Cadence/Vigil/Voyage targeting also ignore scope. `.*` target meant "all Souls," not "everything available to me." Scope needs to be extended not only by coven, but also by regex (SID), a soulprint predicate (CEL `soulprint.self.*`), and a state predicate (CEL over `incarnation.state`) — and `CovenScope` with its `(covens, bool)` signature can't hold that.

**Decision.** Introduce a **Purview** entity — the operator's allowed-scope resolver. Contract:

```
ResolveScope(aid, resource, action string) → Purview
```

`Purview` is a typed result with dimensions: `covens` (exact labels), `regexes` (over SID, RE2), `soulprint` (CEL predicates `soulprint.self.*`), `state` (CEL predicates over `incarnation.state`), plus terminal flags `unrestricted` (no restrictions at all) and `deny` (access denied). Purview **generalizes** `Enforcer.CovenScope`: covens+unrestricted remain a special case (the `coven` dimension + a flag). Multiple values within a dimension (multi-coven, multi-regex) are a union (OR within the dimension).

**(a) default_scope at the role level.** A new field `rbac_roles.default_scope` — a scope selector inherited by **all** permissions of the role. A per-permission selector (`on coven=X` in the permission string) **overrides** default_scope for that specific permission. In other words: the role sets the base scope once, individual permissions can narrow/change it pointwise.

**(b) Default-deny per dimension with three exceptions.** If a role **explicitly introduces** a scope dimension (via `default_scope` or a per-perm selector) but the specific dimension in it is empty → that dimension gets **deny**. BUT:
- **`*`-permission (cluster-admin)** = explicit allow-all, does NOT obey default-deny (`*` literally means "everything," otherwise the bootstrap cluster-admin would lock itself out — [ADR-013](0013-bootstrap-archon.md#adr-013-bootstrap-the-first-archon)/[ADR-014](0014-operator-identity.md#adr-014-operator-identity-model-archon)).
- **bare permission** (no role `default_scope` AND no per-perm selector) = `unrestricted` (backward compatibility: existing roles don't break — this is the same semantics as `Permission.Matches` with `Selector==nil` → `true`, and as `CovenScope` for a bare permission → `unrestricted`).
- An empty dimension gives deny only when it is **explicitly introduced** — the absence of a dimension ≠ an empty dimension.

This removes the blocker "don't lock out the bootstrap cluster-admin when default-deny is enabled."

**(c) Selector extension (future slices).** The exact keys `coven`/`host`/`incarnation`/`service` ([ADR-008](0008-coven-stable-tags.md#adr-008-coven--stable-logical-tags-only)) are extended with: `regex` (over SID, RE2 without backtracking), `soulprint` (a CEL predicate `soulprint.self.*`, [ADR-010](0010-templating.md#adr-010-templating-engine-cel-for-yaml-expressions-go-texttemplate-for-files)/[ADR-018](0018-soulprint-typed.md#adr-018-soulprint-typed-schema-mvp)), `state` (a CEL predicate over `incarnation.state`). The specific key grammar in the permission string is **propose-and-wait when the corresponding slice is implemented** (S2).

**(d) Scoped visibility everywhere.** The souls-list, incarnations-list, Cadence/Vigil/Voyage targeting give/target **only the operator's Purview**. `.*` = "everything available TO ME," not all Souls. Relation to invocation-time scope: regex+where in the target ([ADR-040](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override)/[ADR-043](0043-voyage.md#adr-043-voyage--unified-batch-run)) intersects with Purview — Purview is the **upper bound**, and invocation only narrows it. The security invariant "invocation narrows" falls out naturally.

**Two-layer authorization of read endpoints (ADR-047 §d amendment 2026-06-04).** Read visibility of nodes is authorized in TWO layers, served by different enforcer methods:
1. **Gate (existence)** — `HoldsAction(aid, resource, action)`: does the operator hold the action in ANY scope at all (`ResolvePurview` is non-empty: `Unrestricted` OR any dimension coven/regex/soulprint/state is populated, AND `!Deny`). Doesn't hold → 403 before the handler. The gate does NOT operate on scope-context — the read endpoint doesn't carry host/coven/state in the request at the middleware stage (scope is resolved from DB rows that don't exist yet at the gate stage).
2. **Scope narrowing (handler)** — `ResolvePurview` + per-resource resolvers (`soulpurview`/`statepredicate`): after fetching rows, the handler keeps only nodes within the operator's scope boundary (coven-pushdown / regex-keyset / state-CEL). A node outside scope → excluded from the list / 404 on a single-read (no existence leak).

**Invariant:** the scope-aware `Check(...,ctx)` for read visibility is NOT applied at the gate layer — it answers "does the permission apply in this scope context," while the read endpoint's gate asks "does the operator hold the action at all, in principle." `Check(...,nil)` for a scoped permission gives a false deny (the selector key is absent in a nil context), which broke a scoped operator's access to their own list (closed by slices G0–G2). Mutating endpoints with a scope context known from the body/path (`incarnation.run on coven=…`, `soul.coven-assign`) continue to use the scope-aware `Check`/`RequirePermissionMulti` — for them the gate AND the scope coincide in one question. Per-host history endpoints (`GET /v1/souls/{sid}/history`) go through the same handler-InScope gate as get/soulprint.

**Slice map.**
- **S0** (this ADR + Purview refactor): introduce `ResolveScope`/`Purview` as a generalization of `CovenScope`, **WITHOUT changing `Permission.Matches` semantics** — a pure refactor, observable behavior does not change.
- **S1**: role `default_scope` + inheritance/override + default-deny with the `*` exception (amends the "empty selector = allow" semantics from [ADR-028](0028-rbac-storage.md#adr-028-rbac-storage--postgres)).
- **S2**: selector extension one dimension at a time: `regex` → `soulprint` → `state`.
- **S3**: scoped visibility list→target.
- **S4**: merge with regex+where late-eval target ([ADR-040](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override)/[ADR-043](0043-voyage.md#adr-043-voyage--unified-batch-run)).

**Gotchas (resolve BEFORE the corresponding slice).**
- **Perf of the CEL scope filter on souls-list** — up to 100k evals per request. Needs a perf strategy (SQL-pushdown by `coven`/`regex` vs page-limited CEL over `soulprint`/`state`) — **decide in an ADR/design BEFORE S2b/S3**.
- **`subset.go` least-privilege** (granting rights ⊆ one's own, [`keeper/internal/rbac/subset.go`](../../keeper/internal/rbac/subset.go)) must be extended for **every** new scope dimension, otherwise an operator could grant a right wider than their own → escalation.
- **Regex ReDoS** — RE2 without backtracking is inherently safe, but limit the pattern length.
- **Default-deny migration of cluster-admin** — the built-in `cluster-admin` (`*`) must not lock itself out when default-deny is enabled (exception (b)).

**Relation to other ADRs.** [ADR-028](0028-rbac-storage.md#adr-028-rbac-storage--postgres) (RBAC storage; S1 changes the `Matches` "empty selector = allow" semantics), [ADR-013](0013-bootstrap-archon.md#adr-013-bootstrap-the-first-archon)/[ADR-014](0014-operator-identity.md#adr-014-operator-identity-model-archon) (identity/bootstrap — don't lock out cluster-admin), [ADR-040](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override)/[ADR-043](0043-voyage.md#adr-043-voyage--unified-batch-run) (target-scope), [ADR-010](0010-templating.md#adr-010-templating-engine-cel-for-yaml-expressions-go-texttemplate-for-files)/[ADR-018](0018-soulprint-typed.md#adr-018-soulprint-typed-schema-mvp) (CEL/soulprint), [ADR-008](0008-coven-stable-tags.md#adr-008-coven--stable-logical-tags-only) (coven).

**Amendment (2026-06-09, Synod — a group of Archons [ADR-049](0049-synod.md#adr-049-synod--a-group-of-archons)).** With the introduction of the intermediate level **[Synod](0049-synod.md#adr-049-synod--a-group-of-archons)** (a group of Archons bundling roles), an operator's role set = direct ∪ via Synod. For Purview this is **transparent**: the union is assembled in the enforcer's snapshot build ([ADR-028(d)](0028-rbac-storage.md#adr-028-rbac-storage--postgres)) **before** resolution — `ResolveScope`/`ResolvePurview`/`HoldsAction` receive an already-merged role set, **the Purview matching layer does not change** (Synod is invisible below the snapshot). **Group-scope is NOT introduced:** Synod has no `default_scope` of its own — **scope lives on roles** ([ADR-047(a)](#adr-047-purview--scoped-rbac-visibility-of-nodes-role-default_scope--extended-selector) `rbac_roles.default_scope` + per-perm selector), Synod merely groups already-scoped roles and adds nothing to scope resolution. A second scope layer on the group (intersection group-scope ∩ role-scope) — rejected in the MVP as an additive extension with no confirmed need ([ADR-049](0049-synod.md#adr-049-synod--a-group-of-archons) §Rejected (a)). **Least-privilege subset** ([§Gotchas](#adr-047-purview--scoped-rbac-visibility-of-nodes-role-default_scope--extended-selector), [`subset.go`](../../keeper/internal/rbac/subset.go)) must compute the initiator's effective rights as direct ∪ via Synod — otherwise an operator could grant a right wider/narrower than their own ([ADR-049 §f](0049-synod.md#adr-049-synod--a-group-of-archons)).

#### S3b — scoped visibility of list endpoints (souls + incarnations)

A concretization of slice S3 for **read visibility** (separate from the S4 target). S3 is split into S3a (target, merged with late-eval) and **S3b (list/get visibility)**, because these two paths have a different security invariant and a different perf strategy.

**Entities.**
- **`keeper/internal/soulpurview`** (new, ADR-047 S3b) — a souls analog of `keeper/internal/statepredicate`. Resolver: accepts an already-resolved `rbac.Purview` **as a parameter** (does NOT go to the enforcer — `ResolvePurview` calls the handler; the S4 target filter reuses the same translation on top of its own Purview intersection) → translates the scope boundary into souls-query parameters. A one-directional dependency `soulpurview→rbac`, with no import cycle.
- **incarnations-list/get** scoped visibility **reuses** the existing `statepredicate.ResolveIncarnations` (it already knows how to do SQL-pushdown by `service`/`coven` + page-by-page CEL over state) — no new incarnations package is introduced.

**Endpoints in S3b scope.** `GET /v1/souls` (list) + `GET /v1/souls/{sid}` (get) + `GET /v1/incarnations` (list) + `GET /v1/incarnations/{name}` (get). Each gives the **intersection** of the requested set with the operator's Purview for `<resource>.list` (souls-get/soulprint/history go under the same `soul.list` permission — the service/omen/vigil pattern).

**Resolver coverage (implementation sequence).**
1. **S3b-0 (pilot, this slice) — souls-list, coven dimension only.** SQL-pushdown `souls.coven && ARRAY[purview.Covens]` (reuses `appendScopeClause`, shared with bulk coven-assign — a single fail-closed semantics for the souls layer). offset/total are correct without drift (coven-pushdown is complete), no keyset needed. Wiring: `SoulHandler.List` resolves `ResolvePurview(aid, "soul", "list")` → `soulpurview.Resolve` → `soul.ListScope` → `soul.SelectAll`. The previous `CovenScoper` (S0 `(covens, unrestricted)`) is **generalized** into `PurviewResolver` (`ResolvePurview`) — one resolver for list visibility AND bulk coven-assign (the latter unpacks the coven dimension of Purview into `BulkScope`).
2. **S3b-1** — souls single-get (`GET /v1/souls/{sid}`): the same Purview, a single-host membership check.
3. **S3b-2** — regex/soulprint dimensions for souls: a **page-by-page CEL post-filter** on top of the coven-narrowed set + a **keyset cursor** (offset pagination over a Go post-filter drifts — a keyset is needed). Closes the "perf of the CEL scope filter on souls-list" gotcha (ADR-047 §Gotchas): coven → SQL-pushdown now, soulprint/regex → page-CEL+keyset.
4. **S3b-3** — incarnations list+get: reuse `statepredicate.ResolveIncarnations`; incarnation-scope = `coven ∪ {incarnation.name}` (the incarnation's name is the root Coven label, [ADR-008](0008-coven-stable-tags.md#adr-008-coven--stable-logical-tags-only)).

**Fail-closed (security invariant, OPPOSITE of presence fail-safe).** On scope ambiguity, the result is an EMPTY list, NOT all Souls:
- an empty Purview (`Purview{}`: no dimension at all, not `Unrestricted`) → empty;
- no claims / scoper not configured → empty;
- **scope-eval-error fail-CLOSED** — an error computing scope (a broken CEL at runtime, a resolver failure) hides, does NOT show.

This is **explicitly the opposite** of the presence-overlay on `GET /v1/souls` (`overlayPresence`): on a Redis failure, presence is fail-SAFE (returns the PG snapshot). Scope, when in doubt, **hides**; presence, when in doubt, **shows** — the two layers do NOT borrow each other's strategy. Order: scope narrows the set BEFORE the presence overlay (the scope filter is in SQL, presence is layered on top of the already-scoped set).

`Unrestricted`/`*`-permission → the full available list, no scope filter (bare-with-no-default → `Unrestricted`, backcompat).

**S3b ↔ S4 boundary.** S3b is **visibility** (what the operator SEES in list/get). S4 is **target** (what the operator can TOUCH in a run): invocation-time scope (regex+where, [ADR-040](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override)/[ADR-043](0043-voyage.md#adr-043-voyage--unified-batch-run)) intersects with Purview as the upper bound. Both read the same `ResolvePurview`; the difference is in application (read-filter vs target ∩ Purview).

**Perf.** coven — SQL-pushdown (now, S3b-0). regex/soulprint — page-CEL post-filter + keyset (S3b-2). incarnations — `statepredicate.ResolveIncarnations` page-by-page (S3b-3). At 100k Souls: coven-narrowing in SQL cuts off the bulk before CEL-eval.

#### S4 — target ∩ Purview for Voyage's command path (security fix)

A concretization of slice S4 ([§Slice map](#adr-047-purview--scoped-rbac-visibility-of-nodes-role-default_scope--extended-selector)) for Voyage's **command path** (`kind=command`, permission `errand.run`, [ADR-043 §6](0043-voyage.md#adr-043-voyage--unified-batch-run)).

**Problem (security).** The command path resolves the target as a **bare NoSelector** and expands it **cluster-wide** — without intersecting with Purview. A scoped Archon with the right `errand.run on coven=A` could launch a command-Voyage on `coven=B` (their target wasn't narrowed to their scope). This is an **asymmetry with the scenario path** (`kind=scenario`, `incarnation.run`), where a per-incarnation scope check already exists and the target doesn't cross the operator's boundary. The command path is an escalation hole.

**Decision.** Command target resolution **intersects with the operator's Purview** by **reusing `soulpurview`** ([§S3b](#adr-047-purview--scoped-rbac-visibility-of-nodes-role-default_scope--extended-selector)) — the same resolver that filters `GET /v1/souls`. Command coverage = the target allowed by the request **∩** `ResolvePurview(aid, "soul", …)` (the upper bound, as in [§d](#adr-047-purview--scoped-rbac-visibility-of-nodes-role-default_scope--extended-selector): "invocation narrows"). The intersection happens **in the resolver**, not in the handler: `POST /v1/voyages/preview` ([ADR-043 amendment](0043-voyage.md#adr-043-voyage--unified-batch-run)) **inherits** the narrowed scope automatically (shared resolver).

**No new RBAC selector keys are introduced.** The intersection reuses the existing Purview dimensions (coven / host / regex / soulprint / state); RBAC selector keys remain limited to {`service`, `coven`, `incarnation`, `host`} ([ADR-008](0008-coven-stable-tags.md#adr-008-coven--stable-logical-tags-only)/[§c](#adr-047-purview--scoped-rbac-visibility-of-nodes-role-default_scope--extended-selector)) — the command path adds none of its own.

**Semantics — a hybrid (user's choice, 2026-06-09).** Three branches by target shape:
1. **An explicit foreign host in `sids[]`** (the operator listed specific SIDs, some outside their Purview) → **403** (anti-escalation, parity with the scenario path, where an explicit excursion beyond scope = a denial, not a silent truncation). Explicitly naming a foreign host is an escalation attempt, not a broad filter.
2. **A wide target** (`coven=…` / a `where:` predicate, late-binding) → **truncate** to `target ∩ Purview` (like list visibility: the operator gets what's inside their boundary, no denial).
3. **Empty intersection** (nothing left after truncation) → **422 `voyage_empty_target`** (a valid request, but nothing to execute — distinguished from a 403 escalation).

**Existence-gate — unified, via `ResolvePurview("errand","run")` (the [§d G1](#adr-047-purview--scoped-rbac-visibility-of-nodes-role-default_scope--extended-selector) pattern).** "Does the operator hold the right `errand.run` in any scope at all" is checked by the **same resolve** that gives the scope boundary for intersecting with the target — with no separate preliminary bare-check: `Scope.Empty` (an empty Purview: no dimension and not `Unrestricted`) = no right in any context → denied BEFORE resolving the target. This removes the nil-context bare-check gotcha for a scoped role (like souls G1: a lone `Check(nil)` falsely denies a role with a non-empty scope). **The denial reason is classified** (`Scope.Empty` itself doesn't carry it — revocation and no-perm are merged in the resolver): via the enforcer inside the `Empty` branch — **revoked → `TypeOperatorRevokedToken` (401)**, **no-perm → 403** (`operator lacks required permission errand.run`). The classifier is safe (nil-context): scope has already been proven empty in any context, a scoped role cannot be falsely denied — the enforcer here only distinguishes the reason, not a second gate. Parity of error semantics with the scenario path (`incarnation.run`) and the cluster-wide branch (scoper not configured).

**soulprint/state — `Partial` until S3b-2b.** In command-∩-Purview, the `soulprint`/`state` dimensions remain **under-shown** (`Partial` — never over-show: with incomplete dimension support, the resolver would rather cut too much than show someone else's) until S3b-2b is closed; coven/regex/host work in full. This is the safe direction of error (fail-closed, [§S3b fail-closed](#adr-047-purview--scoped-rbac-visibility-of-nodes-role-default_scope--extended-selector)).

**Compatibility.** For **scoped roles** — this is a **behavior change** (previously cluster-wide command, now ∩ Purview): recorded as a **security fix in release notes**. For **Unrestricted / cluster-admin** (`*`-permission) — **unchanged** (`Unrestricted` → all Souls, as before — [§b](#adr-047-purview--scoped-rbac-visibility-of-nodes-role-default_scope--extended-selector)).

**S4 ↔ S3b boundary.** S3b is **visibility** (what the operator SEES in list/get). S4 is **target** (what the operator can TOUCH): both the scenario and command paths of Voyage intersect the invocation-target with Purview as the upper bound. Both read the same `ResolvePurview`; S4-command closes the asymmetry, bringing the command path up to the same invariant as scenario.

**Slice plan P.** S-P0 — canon (this amendment). S-P1 — scoper-wire + `ResolveSIDsInScope` (resolve ∩ Purview for the command target). S-P2 — `createCommand` (the three hybrid-semantics branches) + guard tests (403-explicit-foreign / truncate-wide / 422-empty / Unrestricted-unchanged). S-P3 — e2e (a scoped Archon does not exceed its coven, parity with scenario).

## Amendment (2026-07-18, NIM-128 — boolean scope grammar)

Scope stops being a single `key=value` selector and becomes a **boolean expression**. Fixed by the user; realises the extended-selector intent of §c with a different shape and a **reduced** selector enum. Full spec — [rbac.md → Selector grammar](../keeper/rbac.md); grammar EBNF — [rbac.md → Format permissions](../keeper/rbac.md).

**Selector enum (§c revised).** The closed set of selector types is **`{coven, service, incarnation, host, trait}`**. The pattern-matching types `regex`, `soulprint`, `state` are **removed** — RBAC visibility by SID-regex, host soulprint facts, and incarnation state is dropped as a product decision. Pattern matching on host identity moves to a `host matches <glob>` condition (glob-only, see below). `trait` moves to the dot-notation form `trait.<key>=v` (from the former `trait=key:value`, [ADR-060](0060-traits.md#adr-060-trait--operator-set-key-value-labels-on-the-incarnation-relocated-from-soul) p. 7).

**Boolean scope.** Conditions combine with `AND` / `OR` and parenthesised groups. Precedence in the parser: **`AND` binds tighter than `OR`**, both left-associative, parentheses override. The UI condition-builder emits **homogeneous groups** (each group is all-`AND` or all-`OR`, mixed logic via nesting) with explicit parentheses, so a UI-authored scope round-trips 1:1; hand-written strings may rely on precedence. `OR` within one dimension is `in (a, b)`. A size cap (≤ 32 atoms, ≤ 4 nesting levels) is enforced on load. `host matches <glob>` is **glob-only** (`*`, `?`, full anchored), compiled internally to an anchored RE2 (ReDoS-safe, length cap 256) and pushed down to SQL `LIKE` where possible; RE2 syntax is not exposed.

**Wire format = string, no schema migration.** The scope is stored RAW in the existing TEXT columns (`rbac_role_permissions.permission`, `rbac_roles.default_scope`) — no column/jsonb migration. An AST-JSON wire format was rejected (string is human-readable, diff-friendly, back-compatible with existing `coven=a,b` rows, single Go-parser source of truth). The parser normalises the string into a `ScopeExpr` AST. `MyPermissionScope` in openapi is reworked: the `regex`/`soulprint`/`state` fields are dropped, a canonical `expr: string` (the rendered scope tree) is added, `unrestricted` stays.

**Least-privilege subset containment (`subset.go`) — fail-closed.** Containment over a boolean scope = **DNF single-conjunct subsumption** (every disjunct of the issued scope must imply one allowing conjunct of the caller — a dimension the caller constrains but the grant leaves free is NOT covered, the key anti-escalation invariant) **+ an exact fast-path** (a disjunct of only exact/`in`-list atoms is covered the old way — decomposed to `(key,value)` points against the union of the caller's permissions — preserving today's behaviour 1:1). Glob containment is **conservative**: `host = x ⊆ host matches g` (exact under glob) and equal globs only; general glob-subsumption and full boolean SAT are deferred. On any undecidable case → DENY.

**Removed type in an old permission = fail-closed.** After removal, a stored permission still carrying `regex`/`soulprint`/`state` fails to parse → the enforcer snapshot build fails → the previous snapshot is held with a warn (never a silent unrestricted, which would be an escalation). Therefore the **data migration must run before the new binary**: [migration `100_rbac_drop_pattern_selectors`](../../keeper/migrations/100_rbac_drop_pattern_selectors.up.sql) is a fail-closed guard — it aborts with a report if any role permission / `default_scope` still uses a removed type (or the old `trait=key:value` form), requiring manual remediation (removing a selector would widen the permission = escalation, so no auto-scrub). The loss of `regex`/`soulprint`/`state` visibility is a **security/feature note for release notes**.

**Single scope resolver + full AST→SQL pushdown.** Scope evaluation is unified into **one resolver with the full node context** (coven + host + trait + incarnation + service), replacing the former split "Check-context vs per-resource resolver" (a boolean mixing dimensions under one `OR`, e.g. `coven=a OR trait.owner=x`, cannot be evaluated in either place alone). Souls/incarnations visibility (S3b) and target ∩ Purview (S4) translate the scope AST **fully into a SQL `WHERE`** (coven array-overlap + host glob→`LIKE` + trait jsonb + incarnation/service), rather than a gate + conservative post-filter. Fail-closed remains the invariant (on scope ambiguity → hide, [§S3b fail-closed](#s3b--scoped-visibility-of-list-endpoints-souls--incarnations)).

## Amendment (2026-07-19, NIM-128 — scoped wildcard `* on <scope>` + incarnation glob)

Two follow-ups to the boolean-scope amendment above (backend implemented, guard-tests green). Grammar EBNF / condition catalog — [rbac.md → Format permissions](../keeper/rbac.md) / [→ Selector grammar](../keeper/rbac.md).

**(A) Scoped wildcard `* on <scope>` — a "scoped super-admin".** The full-wildcard `*` gains an optional boolean scope: `permission := "*" | "*" on <scope> | <resource>.<action> (on <scope>)?`. **Bare `*` stays a cluster-admin** (all actions on every resource, no boundary, [§b](#adr-047-purview--scoped-rbac-visibility-of-nodes-role-default_scope--extended-selector)); **scoped `* on X`** is a super-admin **confined to the predicate X** — every action on every resource, but only where the boolean scope `X` holds.

- **Enforced on the write path.** `Permission.Matches` for a wildcard now branches on the scope: `Scope == nil → true` (bare `*`, unchanged); otherwise `evalScope(Scope, ctx)` — the scoped `*` matches **only** a context in which the predicate is true. A cluster-level action carrying **no** scope context (e.g. `operator.create` without a coven) is therefore **denied** under `* on coven=X` — the wildcard no longer blanket-allows once it is scoped. This closes the gap where a scoped `*` would have behaved like a bare cluster-admin on context-less operations.
- **`ResolvePurview`.** Bare `*` → `Unrestricted` (as before). Scoped `* on X` does **not** lift to `Unrestricted`: it contributes `X` into the `Exprs` of **every** `(resource, action)` the caller resolves — a real scope boundary carried through the normal pushdown, not an allow-all. Role `default_scope` continues **not** to touch `*` ([§b](#adr-047-purview--scoped-rbac-visibility-of-nodes-role-default_scope--extended-selector) exception #1 is preserved for **bare** `*`; a scoped `*` already carries its own predicate, so default_scope is moot).
- **Least-privilege subset (`subset.go`).** Granting `* on X` is allowed **only** if the caller holds bare `*` **or** `* on X'` with `X ⊆ X'` (DNF single-conjunct subsumption of the wildcard scope, § boolean subset). Granting **bare** `*` still requires the caller to hold bare `*`. A caller holding `* on X` also **covers** the issue of a narrower `resource.action on Y` whenever `Y ⊆ X` — but is **not** treated as `Unrestricted` (fail-closed containment, never an escalation).
- **Self-lockout is bare-only.** `HasWildcard` / `ClusterAdmins` count an operator as a cluster-admin **only via a bare `*`** — a scoped `*` does **not** count. The bootstrap "≥1 active bare-`*` admin must survive" invariant ([ADR-013](0013-bootstrap-archon.md#adr-013-bootstrap-the-first-archon) / [§ self-lockout](../keeper/rbac.md)) is therefore intact: a scoped super-admin can never be mistaken for the last cluster-admin and cannot lock the cluster.
- Guard tests: bare-`*`→scoped-`*` grant ok; scoped-`*`→bare-`*` grant deny; `* on X` issuing `* on X'`/`resource.action on Y` only under `⊆`; out-of-scope context deny; context-less cluster-op deny under scoped `*`.

**(B) `incarnation matches <glob>`.** The glob condition, until now `host`-only, is extended to `incarnation`: `incarnation matches "prod-*"` is a valid scope atom. Evaluation, `PurviewSQL` (a `LIKE` pushdown over `incarnation.name`, same anchored-glob→`LIKE` translation as `host`) and least-privilege subset (the same per-atom glob containment: `incarnation = x ⊆ incarnation matches g` iff `g` matches `x`, equal globs only) all reuse the `host`-glob machinery. `coven` / `service` stay **exact / `in`-list only** (no glob) — matching by name-pattern is meaningful only for the two identity dimensions (`host` SID, `incarnation` name).
