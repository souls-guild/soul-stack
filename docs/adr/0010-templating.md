# ADR-010. Шаблонизатор: CEL для YAML-выражений, Go text/template для файлов

- **Контекст.** Soul Stack использует выражения в двух принципиально разных контекстах: предикаты и интерполяция в YAML (scenario, destiny, keeper.yml, миграции, essence pipeline) и рендер файлов конфигов на хосте. [ADR-003](0003-destiny-format.md#adr-003-формат-destiny--yaml-с-типизированной-схемой-cuejson-schema) зарезервировал «безопасный шаблонизатор отдельной фазой», но конкретный движок не выбирал. Open Q «Шаблонизатор Destiny и сценариев» блокировал имплементацию scenario и destiny одновременно (фиксировано в [scenario/orchestration.md §8](../scenario/orchestration.md#8-открытые-вопросы-расширения-не-закрывать-молча) — поднимало scope с одного слоя на оба).
- **Решение.** Два движка с строгой границей по файлу:
  - **CEL** (google/cel-go) — все YAML-выражения. Top-level expression-ключи (`where:`, `when:`, `changed_when:`, `failed_when:`, `until:`) — вся строка = CEL без обёртки. Интерполяция в строковых контекстах — маркер `${ … }`.
  - **Go text/template** + sprig (allowlist) — рендер файлов в `templates/<path>.tmpl`, выполняется новым core-модулем `core.file.rendered`.

  Граница строгая по файлам: `.yml` — CEL, `.tmpl` — text/template. В одном файле — один движок. Это не пересечение, а последовательная передача данных через `vars` шага `core.file.rendered`.

  Полная нормативная спецификация (CEL-функции, sprig allowlist, security model, фазы обработки, escaping, диагностика) — [docs/templating.md](../templating.md).
- **Обоснование.** CEL разработан Google как **sandbox by design**: нет syscall, нет файлов, нет сети, нет произвольного исполнения. Это закрывает класс рисков, которыми болеют Jinja2/Starlark в роли DSL для конфигурации. Type-checking CEL даёт основу для статической проверки в `soul-lint` поверх схем из `input:`/manifest/output-схем модулей. Go text/template — родной для Go, маленькая поверхность, обкатан в k8s/helm. sprig через **whitelist** (а не denylist) удерживает поверхность атаки фиксированной даже при upgrade sprig. Разделение «CEL для логики, text/template для файлов» исключает SSTI через данные тремя одновременными барьерами: strict-mode, allowlist без `exec`, изолированный контекст рендера.
- **Consequences.**
  - `soul-lint` получает статическую проверку выражений по схемам (CEL-AST + типы из `input:`/output-схем модулей/essence-схем). Незакрытые типы (`soulprint.*` до open Q №6) — `dyn`, узел теряет static-check, но не падает.
  - Расширение шаблонных файлов меняется с `.j2` на `.tmpl` (исторические примеры переписываются отдельным pilot+batch).
  - `render(<path>)`-функция (jinja-style) заменяется явным шагом `core.file.rendered` — новый core-модуль (см. [naming-rules.md](../naming-rules.md)), параллель `core.file.present` / `core.file.absent` по той же state-форма-семантике.
  - `soulprint.hosts.where(<predicate>)` / `soulprint.where(<predicate>)` — **compile-time rewrite статического литерал-предиката в native CEL filter-comprehension**, не runtime-исполнение строки и не CEL-макрос пользователя. Всё выражение парсится в AST, вызовы `.where("<pred>")` на `soulprint.hosts`/`soulprint.where(...)` переписываются в `soulprint.hosts.filter(<iter>, <pred>)` (поля предиката квалифицируются полем элемента, внешний контекст — как есть), дерево компилируется один раз. Следствия безопасности/контракта: предикат **обязан** быть строковым литералом (динамическая склейка отвергается на валидации), `.where` разрешён только на `soulprint.hosts`/`soulprint.where(...)` (generic `.where` запрещён), первый элемент — `[0]` (`.first` не вводится). `soulprint.hosts` — **scenario-only** аксессор: в destiny-проходе обращение к нему — ошибка изоляции (топология приходит в destiny только через `apply: input:`). Нормативно — [templating.md §2.3/§7.1](../templating.md#23-зарегистрированные-cel-функции-стартовый-минимум), [scenario/orchestration.md §4.1](../scenario/orchestration.md#41-soulprinthosts--список-хостов-прогона-scenario-only-аксессор).
  - Jinja-фильтры (`| int`, `| length`) переписываются в CEL-функции (`int(…)`, `size(…)`); Jinja-блоки (`{% for %}` / `{% if %}`) — только в `.tmpl` через text/template.
  - Маркер интерполяции в YAML — `${ … }` (не `{{ … }}`), чтобы визуально отделить CEL-в-YAML от text/template-в-файлах и исключить копипаст-ошибки между движками.
  - Open Q «Шаблонизатор Destiny и сценариев» закрыт; пункт 1 «Шаблонизатор теперь блокирует и scenario» в [scenario/orchestration.md §8](../scenario/orchestration.md#8-открытые-вопросы-расширения-не-закрывать-молча) **удалён** (вопрос закрыт настоящим ADR).
- **Trade-offs.**
  - Два движка вместо одного — два словаря у автора сценария. Закрывается чёткой границей «выражение vs render файла» и единым правилом «расширение файла → движок».
  - CEL менее выразителен, чем Jinja2: нет циклов, нет присваиваний, нет произвольных функций. Это **сознательное ограничение** — безопасность важнее. Циклы по списку хостов прогона реализуются через `loop:`-ключ задачи + CEL-выражения над `soulprint.hosts`, не через шаблонный `for`.
  - Маркер `${ … }` вместо привычного `{{ … }}` — необходимая переписка существующих примеров. Компенсация — визуальное разделение движков на уровне синтаксиса.
