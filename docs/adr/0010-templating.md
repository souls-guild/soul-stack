# ADR-010. Шаблонизатор: CEL для YAML-выражений, Go text/template для файлов

- **Контекст.** Soul Stack использует выражения в двух принципиально разных контекстах: предикаты и интерполяция в YAML (scenario, destiny, keeper.yml, миграции, essence pipeline) и рендер файлов конфигов на хосте. [ADR-003](0003-destiny-format.md#adr-003-формат-destiny--yaml-с-типизированной-схемой-cuejson-schema) зарезервировал «безопасный шаблонизатор отдельной фазой», но конкретный движок не выбирал. Open Q «Шаблонизатор Destiny и сценариев» блокировал имплементацию scenario и destiny одновременно (фиксировано в [scenario/orchestration.md §8](../scenario/orchestration.md#8-открытые-вопросы-расширения-не-закрывать-молча) — поднимало scope с одного слоя на оба).
- **Решение.** Два движка с строгой границей по файлу:
  - **CEL** (google/cel-go) — все YAML-выражения. Top-level expression-ключи (`where:`, `when:`, `changed_when:`, `failed_when:`, `until:`) — вся строка = CEL без обёртки. Интерполяция в строковых контекстах — маркер `${ … }`.
  - **Go text/template** + sprig (allowlist) — рендер файлов в `templates/<path>.tmpl`, выполняется новым core-модулем `core.file.rendered`.

  Граница строгая по файлам: `.yml` — CEL, `.tmpl` — text/template. В одном файле — один движок. Это не пересечение, а последовательная передача данных через `vars` шага `core.file.rendered`.

  Полная нормативная спецификация (CEL-функции, sprig allowlist, security model, фазы обработки, escaping, диагностика) — [docs/templating.md](../templating.md).
- **Обоснование.** CEL разработан Google как **sandbox by design**: нет syscall, нет файлов, нет сети, нет произвольного исполнения. Это закрывает класс рисков, которыми болеют Jinja2/Starlark в роли DSL для конфигурации. Type-checking CEL даёт основу для статической проверки в `soul-lint` поверх схем из `input:`/manifest/output-схем модулей. Go text/template — родной для Go, маленькая поверхность, обкатан в k8s/helm. sprig через **whitelist** (а не denylist) удерживает поверхность атаки фиксированной даже при upgrade sprig. Разделение «CEL для логики, text/template для файлов» исключает SSTI через данные тремя одновременными барьерами: strict-mode, allowlist без `exec`, изолированный контекст рендера.
- **Consequences.**
  - `soul-lint` получает статическую проверку выражений по схемам (CEL-AST + типы из `input:`/output-схем модулей/essence-схем). Незакрытые типы (`soulprint.*` до open Q №6) — `dyn`, узел теряет static-check, но не падает.
  - Расширение шаблонных файлов меняется с `.j2` на `.tmpl` (исторические примеры переписываются отдельным pilot+batch).
  - `render(<path>)`-функция (jinja-style) заменяется явным шагом `core.file.rendered` — новый core-модуль (см. [naming-rules.md](../naming-rules.md)), параллель `core.file.present` / `core.file.absent` по той же state-форма-семантике.
  - `soulprint.hosts.where(<predicate>)` / `soulprint.where(<predicate>)` — **compile-time rewrite статического литерал-предиката в native CEL filter-comprehension**, не runtime-исполнение строки и не CEL-макрос пользователя. Всё выражение парсится в AST, вызовы `.where("<pred>")` на `soulprint.hosts`/`soulprint.where(...)` переписываются в `soulprint.hosts.filter(<iter>, <pred>)` (поля предиката квалифицируются полем элемента, внешний контекст — как есть), дерево компилируется один раз. Следствия безопасности/контракта: предикат **обязан** быть строковым литералом (динамическая склейка отвергается на валидации), `.where` разрешён только на `soulprint.hosts`/`soulprint.where(...)` (generic `.where` запрещён), первый элемент — `[0]` (`.first` не вводится). `soulprint.hosts` / `soulprint.where(...)` — **scenario-only** аксессоры (cross-host топология прогона): в destiny-проходе обращение к ним — ошибка изоляции (топология приходит в destiny только через `apply: input:`). **`soulprint.self.*`** (стабильный self-факт целевого хоста) — **доступен и в destiny-проходе** (см. Amendment 2026-06-18): граница изоляции проходит по self vs топология, не по «весь soulprint». Нормативно — [templating.md §2.3/§7.1](../templating.md#23-зарегистрированные-cel-функции-стартовый-минимум), [scenario/orchestration.md §4.1](../scenario/orchestration.md#41-soulprinthosts--список-хостов-прогона-scenario-only-аксессор).
  - Jinja-фильтры (`| int`, `| length`) переписываются в CEL-функции (`int(…)`, `size(…)`); Jinja-блоки (`{% for %}` / `{% if %}`) — только в `.tmpl` через text/template.
  - Маркер интерполяции в YAML — `${ … }` (не `{{ … }}`), чтобы визуально отделить CEL-в-YAML от text/template-в-файлах и исключить копипаст-ошибки между движками.
  - Open Q «Шаблонизатор Destiny и сценариев» закрыт; пункт 1 «Шаблонизатор теперь блокирует и scenario» в [scenario/orchestration.md §8](../scenario/orchestration.md#8-открытые-вопросы-расширения-не-закрывать-молча) **удалён** (вопрос закрыт настоящим ADR).
- **Trade-offs.**
  - Два движка вместо одного — два словаря у автора сценария. Закрывается чёткой границей «выражение vs render файла» и единым правилом «расширение файла → движок».
  - CEL менее выразителен, чем Jinja2: нет циклов, нет присваиваний, нет произвольных функций. Это **сознательное ограничение** — безопасность важнее. Циклы по списку хостов прогона реализуются через `loop:`-ключ задачи + CEL-выражения над `soulprint.hosts`, не через шаблонный `for`.
  - Маркер `${ … }` вместо привычного `{{ … }}` — необходимая переписка существующих примеров. Компенсация — визуальное разделение движков на уровне синтаксиса.

**Amendment 2026-06-22 (CEL-примитив `merge(m, m...) -> map`, SHALLOW last-wins).** Вводит pure CEL-функцию **`merge(map, map, ...) -> map`**: возвращает новый map, в который последовательно слиты аргументы слева направо по ключу **верхнего уровня** — **SHALLOW** (вложенные map НЕ сливаются глубоко: правый аргумент **целиком замещает** значение совпавшего верхнего ключа), **last-wins** (правый перекрывает левый). Pure (без I/O/секретов/крипты/eval-time состояния — симметрично `glob()`/stdlib). Не-map аргумент → ошибка. Регистрируется в **Keeper-env (full, `shared/cel.New`)** и **Soul flow-control-env (`shared/cel.NewFlowControl`)**; в **migration-CEL (`shared/cel.NewMigration`, [ADR-019](0019-state-migration-dsl.md#adr-019-state_schema-migration-dsl)) НЕ регистрируется** (hermetic-песочница с минимумом surface area — расширение требует отдельного ADR, как `glob()`). **Мотивация:** трансляция «простой типизированный ввод оператора → детальная конфигурация» (общий примитив, не redis-специфичный): база CEL не имеет оператора `map + map`, а граница движков не позволяет sprig-merge в `.yml`. Slot для слияния авторского пресет-конфига с passthrough-`input`-map при render-е. **Тип результата** — `map(dyn, dyn)` (как остальные узлы до закрытия type-model [§2.4](../templating.md#24-type-model)). **Безопасность (BLOCKER, security):** `merge` сохраняет ключи верхнего уровня без переименования, поэтому merged-map с vault-значением маскируется выходным слоем (`shared/audit.MaskSecrets`: по sensitive-имени ключа назначения и по vault-ref-маркеру) **идентично** прямому `${ vault(...) }` — секрет не протекает в логи/OTel/RunResult.

Дополнение той же даты (additive, без breaking change): `merge()` **дополнительно принимает один аргумент `list(map)`** — overload **`merge(list(map)) -> map`** (отдельная сигнатура по типу аргумента, не конфликтует с varargs `merge(m, m...)`: cel-go выбирает overload по типу). Семантика — тот же SHALLOW last-wins, но flatten списка map-ов **слева направо** (правый элемент списка бьёт левый). Пустой список → пустой map; элемент-не-map → внятная ошибка. Закрывает случай, когда коллекция приходит из CEL-comprehension `.map(...)` (даёт **список** map-ов), а в шаблон её нужно передать **map-ом** «имя→объект» ради **детерминизма порядка строк** (Go text/template range по map сортирует ключи, по list — сохраняет, возможно недетерминированный, порядок итерации Go-map; см. нормативное правило в [`docs/templating.md §6`](../templating.md#6-передача-данных-между-движками-конвейер)). Нормативные правки — [`docs/templating.md §2.3` (таблица CEL-функций), `§5`, `§6`](../templating.md).

**Amendment 2026-06-23 (CEL-примитив `default(x, y)` — значение-или-дефолт, parity Ansible `| default()`).** Вводит **`default(x, y)`**: если `x` присутствует/доступно — вернуть `x`, иначе `y`. Сокращает канонический has()-guard ([`docs/input.md`](../input.md#чтение-optional-без-default-input-канонический-has-guard)): `has(essence.tls_enable) ? essence.tls_enable : false` → `default(essence.tls_enable, false)`.

- **Механизм — custom macro (compile-time AST-rewrite), как `vault()`** (`parser.NewGlobalMacro`, `shared/cel/default.go` ≡ `vault.go`). CEL вычисляет аргументы **жадно**, поэтому `default(essence.tls_enable, false)`, будь это обычная функция, упал бы «no such key» на отсутствующем ключе **до** вызова. Макрос видит AST первого аргумента **до** eval и переписывает его в проверенную форму **`has(x) ? x : y`** (для select-chain) — та же техника обхода жадности, что у `vault()`. Это compile-time rewrite, а не runtime-исполнение строки.
- **Ограничение (семантика Ansible-default): `x` обязан быть select-chain или идентификатором** (`essence.tls_enable`, `input.shards`, `a.b.c`, либо корневой `input`/`essence`/…). `has()` в CEL применим **только** к доступу к полю (select), поэтому: **Select** `x` разворачивается в `has(x) ? x : y`; **голый идентификатор-корень** (`input`/`essence`/…) всегда присутствует в активации (`shared/cel.Vars.activation` биндит корни как пустой map, не «отсутствуют») → разворачивается в сам `x` (fallback недостижим — это корректная вырожденная семантика). Аргумент-**выражение** (`default(size(x), 0)`, `default(a+b, 0)`, индекс-доступ `default(a['k'], 0)`) макрос **отвергает внятной compile-ошибкой** (`*common.Error`): это и есть корректная семантика — default над переменной/полем, не над вычислением (для последнего у автора есть тернар).
- **Pure** (без I/O/секретов/крипты/eval-time состояния — симметрично `merge()`/`glob()`/stdlib). Env: регистрируется в **Keeper-env (full, `shared/cel.New`)** и **Soul flow-control-env (`shared/cel.NewFlowControl`)**; в **migration-CEL (`shared/cel.NewMigration`, [ADR-019](0019-state-migration-dsl.md#adr-019-state_schema-migration-dsl)) НЕ регистрируется** (hermetic-песочница с минимумом surface area, симметрия с `merge()`/`glob()`).
- **Безопасность (masking-инвариант):** `default(x, y)` — синтаксический сахар над `has(x) ? x : y`, он **не переименовывает ключ назначения**. Поэтому секрет, подставленный через `default(essence.x, vault('…#password'))` или результат `default(...)`, присвоенный ключу с sensitive-именем (`password`/`secret`/`token`/`tls_key`/…), маскируется выходным слоем (`shared/audit.MaskSecrets`) **идентично** прямой подстановке; несекретные значения проходят без over-masking. `default` границу маскинга не расширяет и не сужает (симметрия с `merge()`).

Нормативные правки — [`docs/templating.md §2.3` (таблица CEL-функций), `§7.1`](../templating.md), [`docs/input.md`](../input.md#чтение-optional-без-default-input-канонический-has-guard).

Когда секрет поймал **только** regex (schema/vault/seal по этому пути молчали) — инкрементится метрика **`keeper_mask_regex_fallback_total`** + warn-лог: сигнал пробела декларатива (полное удаление regex — отдельный слайс после structured-замены этого класса). Метафора — «запечатанные пути». Нормативно — [`docs/templating.md §7.4`](../templating.md#74-secret-маскинг).

**Amendment 2026-06-24 (var → var внутри слоя `vars.*` — eager-topological CEL-AST resolve).** Совместно с [ADR-009](0009-scenario-dsl.md#adr-009-scenario--полная-dsl-задач-destiny-граница-с-destiny--рекомендация) (amendment той же даты, полный текст и обоснование — там). Интерполируемое значение `vars.*` (file-level `vars.yml` либо task-level `vars:`) **может** ссылаться на другую переменную **того же слоя** через `${ vars.<other> }`. Извлечение ссылок — новый `shared/cel.Engine.VarRefs(raw) ([]string, error)`: зеркало `DetectSealed` (`scanInterpolation` → per-block `parseNoMacro` → `PostOrderVisit` → `selectBaseField` где base=="vars"), **AST-обход, не regex** — `vars.x` в литеральном тексте вне `${ … }` или в строковом CEL-литерале (`"vars.x"`) ссылкой не считается. Слой резолвится eager-topological (`keeper/internal/render.resolveVarLayer`, зеркало `resolveCompute`): граф зависимостей → топосорт (Kahn) → резолв в порядке с накоплением в `base.Vars`; **порядок объявления безразличен**. Цикл → `ErrVarCycle` с трассой, несуществующая ссылка → `ErrVarUnknownRef` (eager — даже у неиспользуемого var), index-форма `vars['k']` → `ErrVarIndexForm` (детерминированно; имя ключа обязано быть статически известно из AST, симметрично «строковый литерал предиката `.where`» выше). **Изоляция НЕ ослаблена:** var→var строго внутри слоя (file↔task запрещены — межслойная граница), restricted-env не расширяется (в активацию добавляется ТОЛЬКО ключ `vars`, `register.*`/`soulprint.hosts` в значении var по-прежнему ошибка). **migration-CEL** ([ADR-019](0019-state-migration-dsl.md#adr-019-state_schema-migration-dsl)) **не задет** — `vars` остаётся undeclared в migration-env. Нормативно — [`docs/destiny/vars.md`](../destiny/vars.md).

**Amendment 2026-06-26 (Вариант B: `render_context` корень += УСЛОВНЫЙ `input`; seal S-1).** Корень text/template-контекста `core.file.rendered` ([§3.2 templating.md](../templating.md#32-контекст-рендера)) расширяется с `{ vars, self, role, essence }` до **`{ vars, self, role, essence }` + УСЛОВНО `input`**: ключ `input` = резолвнутый operator-input прохода (host-инвариантный) добавляется **только когда шаблон реально читает `.input.*`**. Шаблон читает `.input.<name>` **напрямую** — убирается redundant passthrough `params.vars` для каждого input-поля (`.vars.*` остаётся каналом ПРОИЗВОДНЫХ, вычисленных CEL-ом значений). **★Условная инъекция (детект по AST).** Keeper до per-host цикла читает `.tmpl` один раз и определяет обращение к корневому `.input` через **парс text/template + обход parse-AST** (`tmpl.Engine.UsesRootField` — не string-search: упоминание `.input` в литеральном тексте/комментарии тела шаблона **не** считается обращением; обращение — только `FieldNode` с первым идентификатором `input` внутри action/range/if/with/pipeline-аргумента/вложенного `define`). Есть обращение → ключ `input` кладётся в `render_context` (`buildRenderContext`, источник — `cel.Vars.Input`); иначе **не кладётся**, `render_context` остаётся `{ vars, self, role, essence }` как до Варианта B (шаблоны на одних `.vars` — например, redis — не получают раздутый контекст, их deep-equal-фикстуры стабильны, секреты в `render_context.input` не попадают). Soul передаёт корень движку без изменений (proto не меняется — едет внутри `RenderedTask.params.render_context`, A1). **★Security — seal S-1.** Убрав passthrough, теряем seal-провенанс секретов из сырых params (раньше `${ input.secret }` физически в `params.vars` → `collectSealed`/`DetectSealed` ловил по AST). Провенанс восстанавливается **декларативно**: для шага `core.file.rendered`, **когда `input` реально инъектится** (тот же AST-гейт), Keeper помечает sealed путь `render_context.input.<name>` для каждого input-поля, объявленного `secret: true` (или с `vault_scope`) в активной схеме прохода (`render.sealRenderContextInput`, источник — `secretInputNames`). Гейт держит seal-набор синхронным реальному составу `render_context` — vars-only шаблон не плодит мёртвые sealed-пути на несуществующую ячейку. Маскинг (`audit.MaskSecretsSealed`) прячет `render_context.input.<secret>` в наблюдаемых каналах (status_details/error/логи); wire-значение реальное. **Ограничение (открыто, security):** в **destiny-проходе** (`apply: destiny`) destiny-input-схема в пилоте **не пробрасывается** в seal-источник, поэтому secret-поля destiny-input, доехавшие в `render_context.input`, S-1 **не запечатывает** — для них остаётся vault-происхождение (если значение `vault:`-ref) + regex-last-resort.
