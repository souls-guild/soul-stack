# ADR-003. Destiny Format — YAML with a Typed Schema (CUE/JSON Schema)

- **Context.** Destiny is authored by admins and operations teams; the format must be familiar, diffable, version-controlled in git, and validated **before** it ships to a host.
- **Decision.** The source of truth is YAML. On top of it — a typed schema (JSON Schema to start, with an eye toward CUE for richer constraints). Templating is a separate safe engine (no Jinja-style arbitrary execution), split out into an explicit rendering phase before validation. The specific engines and their security model — [ADR-010](0010-templating.md#adr-010-templating-engine-cel-for-yaml-expressions-go-texttemplate-for-files). The format of the destiny input contract (the `input:` block) and the surface validation DSL are described separately in the ["Destiny: entry contract and validation"](../architecture.md#destiny-entry-contract-and-validation) section.
- **Rationale.** YAML is a familiar, widely-used configuration format. The schema catches typos and wrong types locally in `soul-lint` and in CI before Destiny reaches Soul. The strict "render → validate → apply" separation rules out classes of errors and attacks typical of templating magic.
- **Trade-off.** Less expressive than Jsonnet/Starlark; complex branching is less elegant to express. We accept this — better weaker but safer and more predictable.
