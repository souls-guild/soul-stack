# ADR-003. Формат Destiny — YAML с типизированной схемой (CUE/JSON Schema)

- **Контекст.** Destiny описывают админы и команды эксплуатации; формат должен быть знакомым, диффабельным, версионируемым в git и проверяемым **до** выезда на хост.
- **Решение.** Источник истины — YAML. Поверх — типизированная схема (на старте JSON Schema, с прицелом на CUE для более богатых ограничений). Шаблонизация — отдельным безопасным движком (без Jinja-style произвольного исполнения), вынесена в явную фазу рендеринга перед валидацией. Конкретные движки и их security model — [ADR-010](0010-templating.md#adr-010-шаблонизатор-cel-для-yaml-выражений-go-texttemplate-для-файлов). Формат входного контракта destiny (блок `input:`) и поверхностный DSL валидации описаны отдельно в разделе [«Destiny: входной контракт и валидация»](../architecture.md#destiny-входной-контракт-и-валидация).
- **Обоснование.** YAML привычен для Salt/Ansible-аудитории. Схема ловит опечатки и неверные типы локально на `soul-lint` и в CI до того, как Destiny дойдёт до Soul. Жёсткое разделение «render → validate → apply» исключает классы ошибок и атак, характерных для шаблонной магии.
- **Trade-off.** Меньше выразительности, чем у Jsonnet/Starlark; сложные ветвления делаются менее элегантно. Принимаем — лучше слабее, но безопаснее и предсказуемее.
