# ADR-017. Keeper-side core-модули расширены: `core.cloud.provisioned`, `core.vault.kv-read`

- **Контекст.** Прецедент keeper-side core (`core.soul.registered`, [`docs/keeper/modules.md`](../keeper/modules.md)) показал шаблон: модуль исполняется на keeper-инстансе (диспетчер `on: keeper`), оперирует реестрами Postgres/Redis, имеет тот же `<namespace>.<module>.<state>`-контракт, что и Soul-side core. В существующих примерах (`service.yml`, [Service — структура и manifest](../architecture.md#service--структура-и-manifest)) фигурирует destiny `cloud-provision` с `on: keeper`. Это семантический drift: destiny — пакет задач для Soul, не keeper-side операция. Аналогично, «vault-resolve фаза» в [ADR-010](0010-templating.md#adr-010-шаблонизатор-cel-для-yaml-выражений-go-texttemplate-для-файлов) сейчас неявная — стоит формализовать как явный модульный шаг, чтобы аудит/RBAC/OTel работали единообразно.
- **Решение.** Вводятся два новых keeper-side core-модуля:

  **(a) `core.cloud.provisioned`.**
  - State-формы: `created` / `destroyed`.
  - Назначение: создание / удаление VM или другой cloud-сущности через `CloudDriver`-плагин (`soul-cloud-*`, [ADR-011](0011-go-layout.md#adr-011-раскладка-go-кода-gowork-с-модулями-по-сторонам), [Plugin-инфраструктура](../architecture.md#plugin-инфраструктура)).
  - Адресация шага в scenario: `module: core.cloud.created` (или `core.cloud.destroyed`), `on: keeper`, `params:` содержит `provider` / `profile` / `count` / прочие cloud-параметры. (Историческая форма `module: core.cloud.provisioned` в раннем тексте этого ADR — НЕ существующий адрес; см. amendment (u)/(v) 2026-06-26.)
  - Заменяет старый паттерн «destiny `cloud-provision`» — соответствующие примеры в `service.yml` обновляются (отдельная задача).
  - Output (register-payload): список созданных хостов с их `sid` / `coven[]` / cloud-метаданными. Через `register:` и [`soulprint.hosts`](../scenario/orchestration.md) доступен последующим шагам scenario.

  **(b) `core.vault.kv-read`.**
  - State-форма: `read` (verb).
  - Назначение: чтение Vault KV-секрета на keeper-стороне в момент рендера scenario. Заменяет неявную «vault-resolve фазу» ([ADR-010](0010-templating.md#adr-010-шаблонизатор-cel-для-yaml-выражений-go-texttemplate-для-файлов)) как явный модульный шаг для случаев, когда оператор хочет явный аудит чтения.
  - Адресация: `module: core.vault.kv-read`, `on: keeper`, `params: { path: secret/redis/admin-password, fields: [password] }`.
  - Output: запрошенные поля как register-payload (secret-masking — на выходе через стандартный механизм, см. [ADR-010](0010-templating.md#adr-010-шаблонизатор-cel-для-yaml-выражений-go-texttemplate-для-файлов)).
  - Implicit vault-resolve в CEL-выражениях (`${ vault('secret/.../password') }`) — **остаётся** для удобства; explicit-модуль для случаев, когда нужна явная задача-шаг.

  **(c) Адресация и контракт идентичны Soul-side core** — намеренно. Authors keeper-side custom-модулей (в perspective) реализуют тот же SoulModule-интерфейс из `sdk/module/`.

- **Consequences.**
  - В [`docs/keeper/modules.md`](../keeper/modules.md) дополняются спецификации двух новых модулей.
  - В примерах `service.yml` и destiny-каталоге паттерн «destiny `cloud-provision`» заменяется на «scenario-task `module: core.cloud.created` с `on: keeper`» — это работа на отдельной задаче миграции примеров.
  - В [naming-rules.md → раздел «Конкретные core-модули»](../naming-rules.md) дополняются имена.
  - **Cascade `destroyed`-state** (миграции 016+017): после успешного `PluginHost.Destroy` модуль `core.cloud.provisioned` в одной PG-транзакции переводит `souls.status → 'destroyed'` (terminal-state, расширение enum), активные `soul_seeds.status → 'orphaned'` (новый статус; `revoked` НЕ перетирается — precedence revoked > orphaned), ещё-не-использованные `bootstrap_tokens` → `used_at = NOW(), used_by_kid = 'system-cloud-destroy'` (anti-replay-инвариант, специальный маркер вне формата реального KID). `destroyed` намеренно НЕ входит в default-set `purge_souls.statuses` (forensic > GC). Полная семантика — в [`docs/soul/identity.md`](../soul/identity.md).
  - Open Q №5 — частично закрыт ([ADR-015](0015-core-modules-mvp.md#adr-015-core-модули-mvp-точный-список) + этот).
- **Trade-offs.**
  - Дублирование (implicit vault-resolve + explicit `core.vault.kv-read`) — два пути для одной задачи. Принимаем: implicit дёшев для рендера, explicit нужен для аудита (destination-шаг в scenario). Каждый под свой use case.
  - Существующие примеры с destiny `cloud-provision` остаются временно невалидными, пока миграция не пройдена — допустимое промежуточное состояние, помечается в [open вопросах].

Отложено pre-тираж: `Status` / `List` тоже должны принимать credential (only-add `StatusRequest.credentials` / `ListRequest.credentials`) — вводится перед реализацией соответствующих методов.
  - **(f) Shared SDK-каркас (`sdk/clouddriver`).** Общая (не provider-specific) часть драйвера выносится в SDK: **error-таксономия** (классификация cloud-ошибок на retryable / terminal / auth), **Retry** (backoff) и **WaitUntilReady** (anti-orphan — драйвер дожидается, пока VM перейдёт в running/ready, чтобы не вернуть half-created ресурс без отслеживания). Provider-specific часть каждого драйвера минимальна — это **тираж-гейт** (pilot задаёт pattern, остальные тиражируют по нему, см. [CLAUDE.md → Массовые операции](../../CLAUDE.md)).
  - **(g) Шесть официальных провайдеров (закрывает [open Q №13](../architecture.md#открытые-вопросы)).** В первый набор входят: **AWS / GCP / Azure / Yandex Cloud / Proxmox / OpenStack**; бинари — `soul-cloud-aws` / `soul-cloud-gcp` / `soul-cloud-azure` / `soul-cloud-yc` / `soul-cloud-proxmox` / `soul-cloud-openstack` (имя Yandex Cloud сокращено до `yc`). **vSphere** — community / отложен (не в первом наборе). **AWS — пилот** (reference-реализация, на которой зафиксирован pattern перед тиражом остальных пяти). Стоп-правило тиража — форма provider-specific credentials-struct и WaitUntilReady на on-prem (Proxmox без региона, OpenStack Keystone-auth).
  - **(h) Bootstrap soul-агента на VM — рекомендованный путь cloud-init userdata (по [open Q №14](../architecture.md#открытые-вопросы)).** Основной путь — **cloud-init userdata**: `CreateRequest.userdata` несёт blob, разворачивающий soul-агент на новой VM. **Per-VM-токен-в-userdata отложен** (chicken-egg: SID присваивается **после** create — на момент формирования userdata индивидуального токена под конкретный SID ещё нет). В MVP — **общий batch-blob userdata** + онбординг через CSR ([ADR-012](0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add) `Bootstrap`, приватник не покидает хост). Альтернативные пути (`keeper.push` после create / gold image с soul в AMI) остаются возможны, но cloud-init — рекомендованный default для cloud-create.
  - **(i) Отложено pre-тираж (edge-hardening).** Помимо `Status` / `List` credentials (см. (e)): driver-side **wait-deadline** (жёсткий таймаут ожидания ready), **terminal-state probe** (распознавание провалившегося create вместо бесконечного ожидания), **partial-batch** (корректная обработка `count>1`, когда часть VM создалась, часть нет). Вводятся перед батч-тиражом провайдеров.

- **Amendment 2026-05-27 (cloud-init bootstrap MVP, B-flat закреплён).** Уточняет (h) под фактический код. Прежний текст «per-VM-токен отложен (chicken-egg)» **устарел**: реализованный код `keeper/internal/coremod/cloud/provisioned.go::applyCreated` уже выписывает per-VM bootstrap-токены ПОСЛЕ Create-фазы (когда SID = FQDN VM известен из `VmInfo.fqdn`), складывая их в `register.<step>.hosts[i].bootstrap_token` (см. inline-комментарий «security, H1»). Это **B-flat** вариант: per-VM-токен есть, но доставляется на VM **не через cloud-init userdata**, а **отдельным шагом сценария** (типично `keeper.push` через SSH-провайдер). Cloud-init userdata НЕСЁТ ТОЛЬКО: установка `soul`-бинаря (curl с pinned-CA), embedded PEM CA Keeper-а (`/etc/soul/tls/keeper-ca.pem`), минимальный `soul.yml` с `keeper.endpoints` (LB host:port), systemd-unit `soul.service`. **БЕЗ токенов** — cloud-provider API хранит userdata в plaintext metadata, доступной процессам VM (security floor). Late-binding callback `/v1/bootstrap-tokens/claim` для self-bootstrap в одном цикле cloud-init — отложен пост-MVP (требует нового token-kind, endpoint-а, отдельного TOFU-mitigation).
  - **(m) Реализация.** Пакет [`keeper/internal/cloudinit`](../../keeper/internal/cloudinit/): `Config` + `GenerateUserdata(cfg)` рендерит cloud-config YAML через `go:embed` template `templates/cloud-init.tmpl`; `Resolver.Resolve(ctx, cfg)` тянет CA PEM из Vault по `tls_ca_ref`. Опциональный блок `keeper.yml::cloud_init` (новое поле `KeeperConfig.CloudInit` в `shared/config`): `bootstrap_endpoint` (LB host:port), `tls_ca_ref` (`vault:<mount>/<path>`, поле `ca`), `soul_binary_url` (HTTPS, проверяется), `soul_version` (опц. метка). Hot-reload работает через `config.Store.Get()` — следующий cloud-create-шаг подхватит обновлённый snapshot без рестарта.
  - **(n) Контракт scenario.** Параметр `core.cloud.provisioned` `state: created`: `generate_userdata: true` (bool, optional, default false) — рендерит userdata из keeper.yml::cloud_init. Mutually exclusive с явным `userdata: "<blob>"` (одновременное присутствие → fail с понятной ошибкой). Без `generate_userdata` и без `userdata` — провайдер получает пустой userdata (legacy-поведение, удобно для gold-image-флоу).
  - **(o) Per-provider templates и Ignition — отложены.** MVP: один общий cloud-config YAML на все 6 провайдеров. CoreOS/Flatcar Ignition (`#!/ignition` JSON-формат) — отдельный slice по запросу. soul-binary signature verification (sigstore / detached signature) — отдельный slice.
  - **(p) Закрывает рекомендацию по [open Q №14](../architecture.md#открытые-вопросы)** для типового cloud-create-флоу. Альтернативные пути (`keeper.push` после create без cloud-init / gold image с soul в AMI) остаются возможны.

- **Amendment (2026-06-22, прозрачная поддержка KV v1/v2 — версия резолвится probe-ом, а не эвристикой).** Уточняет (b) `core.vault.kv-read` и весь keeper-side Vault KV-доступ под фактический код `keeper/internal/vault/client.go`. Касается всех ~15 callers `Vault.ReadKV`/`WriteKV` (bootstrap / sigil / push / cloudinit / cloud-credentials / `core.vault.kv-read` / CEL `${ vault(...) }` / augur / herald / toll) — поверхность прозрачна, публичные сигнатуры методов не менялись.
  - **(q) Версия KV mount-а резолвится в `vault.Client`.** `kv_mount` теперь поддерживает и KV v1, и KV v2; версия определяется ленивым thread-safe probe-ом `sys/internal/ui/mounts/<mount>` (читается `data.type == "kv"` + `data.options.version` ∈ {"1","2"}), результат кешируется per-mount. `ReadKV`/`WriteKV` роутятся на `KVv1`/`KVv2` соответственно; контракт возврата `ReadKV` — единый плоский payload (поля секрета) для обеих версий. Probe **строго ленивый** (первый реальный `ReadKV`/`WriteKV`), НЕ в конструкторе — иначе сломался бы bootstrap-путь (`keeper init` поднимает Client до выдачи KV-доступа, конструктор делает только `Ping` на `sys/health`).
  - **(r) Explicit override `vault.kv_version`.** Новое опциональное поле `keeper.yml::vault.kv_version` (`"1"`/`"2"`; пусто → auto). Override побеждает probe **без round-trip-а** — нужно, когда ACL закрывает probe-endpoint. Additive (старые конфиги работают). `ListKV`/`ReadKVMetadata` — v2-only (у KV v1 нет `<mount>/metadata/`-пути); на v1-mount-е отдают понятную ошибку «требует KV v2», а не молча сломанный путь.
  - **(s) Fail-closed на неопределимой версии.** Если probe не дал однозначной версии (нет `options.version` / неожиданное значение / `type != "kv"` / permission-denied без override) — **явная ошибка** с подсказкой про `vault.kv_version`, а НЕ молчаливый дефолт v2.
  - **(t) ОТВЕРГНУТ прежний механизм «discriminator по классу ошибки `KVv2.Get`».** Раннее предполагалось читать KV прозрачно через «попробовать v2, при `ErrSecretNotFound` упасть на v1». Эмпирически провалился: **обычный v1-секрет неотличим от v2-missing** — `KVv2.Get` на v1-mount возвращает тот же `ErrSecretNotFound`, что и реально отсутствующий v2-секрет, поэтому валидные v1-секреты НИКОГДА не читались, а v2-missing рисковал замаскироваться под v1-чтение. Probe-механизм держит этот инвариант **конструктивно** (версия известна до round-trip-а), а не эвристикой. Латентный спутник: `core.vault.kv-read` ранее прогонял payload через `extractKVData`, ждавший несуществующую обёртку `{data,metadata}` (`ReadKV` отдаёт плоский payload и так) — удалён.

- **Amendment (2026-06-26, выравнивание author-facing module-адреса cloud — `core.cloud.created` / `core.cloud.destroyed`, не `core.cloud.provisioned`).** Уточняет написание адреса задачи в `module:` под фактический реестр (`keeper/internal/coremod/registry.go`) + код (`keeper/internal/coremod/cloud/provisioned.go`) + integration-тест + канон [naming-rules.md → `core.cloud`](../naming-rules.md). **Меняется ТОЛЬКО написание module-адреса** — state-формы (`created`/`destroyed`), контракт `CloudDriver`, credentials-flow (Вариант A), cascade `destroyed`-state и cloud-init bootstrap из амендментов (a)–(t) выше **не затрагиваются**.
