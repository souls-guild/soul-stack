# ADR-049. Synod — a group of Archons

> **Status: implemented (2026-06-09/10), in baseline.** The name **Synod** was chosen by the user (propose-and-wait passed); model design — architect. The implementation is **fully closed** — the backend (migrations `synods` / `synod_operators` / `synod_roles`, an addition to the enforcer's snapshot build, security invariants (f), the API `/v1/synods*` with the entire `synod.*` permission family, including `PATCH /v1/synods/{name}` for editing `description` — amend below) and the UI. Operator-to-role membership is now three-tier: direct rows `rbac_role_operators` ([ADR-028](0028-rbac-storage.md#adr-028-rbac-storage--postgres)) ∪ roles via Synod.

**Context.** The RBAC model is currently two-tier: **Archon → Roles** (membership rows `rbac_role_operators`, [ADR-028](0028-rbac-storage.md#adr-028-rbac-storage--postgres)). To give N Archons the same set of roles, the operator has to grant each of them every role by hand — `rbac_role_operators` grows as `archons × roles`, and changing a team's "standard permission set" means walking through all its members. There is no named unit "a role set for a group of operators" that can be set up once and have Archons added/removed from it. This is a typical RBAC gap: a **group of operators** is missing between "operator" and "role."

**Decision.** Introduce the **Synod** entity (Greek "assembly") — a **group of Archons** that bundles a set of roles. The model becomes three-tier: **Archon → Synod → Roles**.

> **Name chosen by the user (2026-06-09).** `Conclave` is a synonym for "assembly," but it is **taken** by the live-Keeper-instance registry ([ADR-006](0006-cache-redis.md#adr-006-кэш-и-координация--redis) amendment) — it can't be reused. `Synod` is free and keeps the "Greek-administrative" line alongside [Archon](0014-operator-identity.md#adr-014-identity-модель-оператора-archon).

**(a) Synod bundles roles, it has NO scope of its own.** Synod carries a **set of roles** — and only that. Synod has **no** `default_scope` of its own: scope lives on roles ([ADR-047(a)](0047-purview.md#adr-047-purview--scoped-rbac-visibility-of-nodes-role-default_scope--extended-selector) `rbac_roles.default_scope` + per-perm selector) — Synod adds nothing to scope resolution, it merely groups already-scoped roles. A Synod member gets the Synod's roles with exactly the scope set on the roles themselves. **Group-scope (a separate scope on Synod that narrows the group's roles) is NOT introduced** — an additive extension for the future (see "What is deferred"), not part of the MVP.

**(b) Flat groups.** Synod does **NOT** contain other Synods — there is no nesting (a flat model). A Synod member is only an Archon, not a Synod. There is no transitive expansion of groups within groups; effective roles are computed over a single grouping level. Nesting is an additive extension for the future (see "What is deferred").

**(c) An Archon can belong to several Synods.** The "Archon ↔ Synod" membership is many-to-many. An Archon's effective roles = **direct** (`rbac_role_operators`) **∪** roles via **all** of their Synods (a union over `synod_operators` ⋈ `synod_roles`). A duplicate role (via a direct grant AND via a Synod, or via two Synods) is idempotent — this is a set union, not a multiset.

**(d) Data model — three PG tables `synod*`** (the `rbac_*` pattern from [ADR-028](0028-rbac-storage.md#adr-028-rbac-storage--postgres)):

| Table | Role |
|---|---|
| **`synods`** | The group catalog. `name` PK (kebab-case, a CHECK on the format — like `rbac_roles.name`), `description`, `builtin` BOOL (protection from `synod.delete`, symmetric to `rbac_roles.builtin`), `created_at`, `created_by_aid` FK→`operators(aid)` nullable (NULL — seed groups with no initiating Archon). |
| **`synod_operators`** | The Synod ↔ Archon **membership**. `synod_name` FK→`synods(name)` `ON DELETE CASCADE`, `aid` FK→`operators(aid)`, `added_at`, `added_by_aid` FK→`operators(aid)` nullable, PK `(synod_name, aid)`. |
| **`synod_roles`** | The Synod ↔ role **bundle**. `synod_name` FK→`synods(name)` `ON DELETE CASCADE`, `role_name` FK→`rbac_roles(name)` `ON DELETE CASCADE`, `granted_at`, `granted_by_aid` FK→`operators(aid)` nullable, PK `(synod_name, role_name)`. CASCADE on both sides: deleting a Synod cleans up its own bundle rows, deleting a role ([ADR-028](0028-rbac-storage.md#adr-028-rbac-storage--postgres) `role.delete`) removes it from all Synods. |

Full symmetry with the RBAC tables: `synods` ↔ `rbac_roles` (catalog + `builtin`), `synod_operators` ↔ `rbac_role_operators` (membership), `synod_roles` — a new bundle tier.

**(e) Resolution — an addition to the snapshot build, matching is untouched.** An Archon's effective roles (direct ∪ via Synod) are **assembled at the enforcer's in-memory snapshot build stage** ([ADR-028(d)](0028-rbac-storage.md#adr-028-rbac-storage--postgres) `LoadSnapshot`): two more SELECTs (`synod_operators` ⋈ `synod_roles`) are added to the existing three (`rbac_roles` ⋈ `rbac_role_permissions` ⋈ `rbac_role_operators`), and the membership map `AID → []*Role` is **augmented** with roles via Synod before the snapshot is fixed. From there on, nothing changes: `Check` / `ResolvePurview` / `HoldsAction` ([ADR-047](0047-purview.md#adr-047-purview--scoped-rbac-visibility-of-nodes-role-default_scope--extended-selector)) see an already-merged role set and **the Purview matching layer is unchanged** — Synod is invisible below the snapshot build. Snapshot invalidation ([ADR-028(d)](0028-rbac-storage.md#adr-028-rbac-storage--postgres) topic `rbac:invalidate`) fires on a Synod/its membership/its bundle mutation the same way it does on a role mutation.

**(f) Security invariant — least-privilege and self-lockout MUST account for roles via Synod.** This is a critical point: both invariants compute "the operator's effective rights," and both break if they only count direct roles.

- **Least-privilege subset check** ([`subset.go`](../../keeper/internal/rbac/subset.go), [ADR-047 §Gotchas](0047-purview.md#adr-047-purview--scoped-rbac-visibility-of-nodes-role-default_scope--extended-selector)): "you can't grant wider than your own rights." If the subset check for `synod.grant-role` / `synod.add-operator` (and for regular `role.grant-operator`) compares the right being granted only against the initiator's **direct** roles, an operator whose rights partially come via a Synod will falsely get a denial **or** — worse — the "own rights" check will undercount them and let through a grant of a right the initiator doesn't hold directly. The initiator's effective rights in the subset check = direct ∪ via Synod.
- **The self-lockout invariant** ([ADR-013(c)](0013-bootstrap-archon.md#adr-013-bootstrap-первого-архонта)/[ADR-028(f)](0028-rbac-storage.md#adr-028-rbac-storage--postgres)): "the cluster must not be left without an active AID with effective `*`." `*` can now reach an Archon **via a Synod** (a Synod bundles a role with `*`). The check "at least 1 active AID with effective `*` remains" must expand Synod membership — otherwise `synod.remove-operator` / `synod.revoke-role` / `synod.delete`, which remove the last path to `*`, would lock the cluster unnoticed. So the Synod paths are added to ADR-028(f)'s three paths (`role.delete` / `role.update` / `role.revoke-operator`): `synod.delete`, `synod.revoke-role`, `synod.remove-operator` — each checks the invariant, accounting for the Synod expansion, before the mutation; otherwise `409 would-lock-out-cluster`.

**(g) API/MCP + the `synod.*` permission family.** Management via OpenAPI/MCP (the `role.*` pattern [ADR-028(e)](0028-rbac-storage.md#adr-028-rbac-storage--postgres), `operator.*` [ADR-014(d)](0014-operator-identity.md#adr-014-identity-модель-оператора-archon)):

- `synod.create` — create a group (+ description).
- `synod.delete` — delete a group (cascades membership + bundle; forbidden on `builtin=true`; the self-lockout guard (f)).
- `synod.list` — list groups with their members and roles.
- `synod.add-operator` — add an Archon to a group (`synod_operators`); under the least-privilege subset (f).
- `synod.remove-operator` — remove an Archon from a group (the self-lockout guard (f)).
- `synod.grant-role` — add a role to the group's bundle (`synod_roles`); under the least-privilege subset (f).
- `synod.revoke-role` — remove a role from the bundle (the self-lockout guard (f)).

Endpoints: `/v1/synods` (`synod.create` POST / `synod.list` GET / `synod.delete` DELETE `{name}`), `/v1/synods/{name}/operators` (`synod.add-operator` POST / `synod.remove-operator` DELETE `{aid}`), `/v1/synods/{name}/roles` (`synod.grant-role` POST / `synod.revoke-role` DELETE `{role_name}`). The `synod.*` permission family is **NoSelector** (group management is a cluster-level operation, not scoped by coven/host; like `role.*` and `operator.*`). The HTTP facade / MCP-tool mapping is normalized as a separate task during implementation (the ADR-028(e) pattern).

**Amendment (2026-06-09, editing a Synod).** Synod is **mutable on `description`**: a permission **`synod.update`** is added, plus the endpoint **`PATCH /v1/synods/{name}`** (MCP tool `keeper.synod.update`), changing **ONLY `description`**. The `synod.*` family is now 8 permissions. **`name` (PK) is IMMUTABLE — renaming is deliberately REJECTED** (symmetric to `rbac_roles.name` / `souls.sid` / `incarnation.name`: renaming a PK would cause audit drift — the old name in `audit_log` loses its link to the row; a desync window in the enforcer snapshot — `synod_operators`/`synod_roles` reference `synod_name`; an asymmetry with `role.update`, which also doesn't rename the role). `synod.update` is **NoSelector** (like the rest of the family); **editing a `builtin` one IS ALLOWED** (`description` is cosmetic for UI/audit, not behavior: it grants/removes no rights, so the `synod.delete` builtin boundary doesn't extend here); **no subset check and no self-lockout check** (description doesn't change rights — both invariants in (f) don't apply); **no snapshot invalidation** (the enforcer snapshot (e) carries only `name`/roles/membership — `description` doesn't enter matching, authorization doesn't change from editing it). Audit event **`synod.updated`** (`api`/`mcp`, payload `{name, description}`), alongside the HTTP/MCP write path (an RBAC-topology mutation is audited, [ADR-022](0022-audit-pipeline.md#adr-022-audit-pipeline-storage-schema-retention)).

**Rationale.**
- **Consistency with [ADR-014](0014-operator-identity.md#adr-014-identity-модель-оператора-archon)/[ADR-028](0028-rbac-storage.md#adr-028-rbac-storage--postgres).** Synod follows the same pattern as RBAC storage: PG tables with real FKs into `operators`/`rbac_roles`, hot-add via the API, audit via FKs + audit events, a snapshot in the enforcer + Redis invalidation. No new infrastructure — the snapshot machinery from [ADR-028(d)](0028-rbac-storage.md#adr-028-rbac-storage--postgres) is reused.
- **Security first.** A group is a manageable unit of revocation: `synod.remove-operator` removes the group's entire role set from an Archon in one mutation (+ Redis invalidation, the effect is instant across all nodes). But that same power requires that both security invariants in (f) expand Synod — this is an explicit ADR invariant, not an "implementation detail."

**Consequences.**
- **Three new PG tables** `synods` / `synod_operators` / `synod_roles` ([keeper/storage.md](../keeper/storage.md) is supplemented during implementation).
- **The `synod.*` permission family** (8 permissions, incl. `synod.update` — amend 2026-06-09) — [naming-rules.md](../naming-rules.md), catalog [rbac.md → Permissions directory](../keeper/rbac.md#permissions-directory) (docs-writer).
- **The enforcer's snapshot build** is supplemented with two SELECTs and a membership-map merge step; the `Check` / `ResolvePurview` / `HoldsAction` interfaces **do not change**.
- **The self-lockout invariant** is extended to `synod.delete` / `synod.revoke-role` / `synod.remove-operator` (f); **the least-privilege subset** — to `synod.grant-role` / `synod.add-operator` (f).
- **Audit events** `synod.created` / `synod.updated` (amend) / `synod.deleted` / `synod.operator-added` / `synod.operator-removed` / `synod.role-granted` / `synod.role-revoked` (category `api` / `mcp`, [ADR-022](0022-audit-pipeline.md#adr-022-audit-pipeline-storage-schema-retention)) — [naming-rules.md → Audit-events](../naming-rules.md#audit-events); normalized via a PR when the write path is implemented.

**Relation to other ADRs.**
- **[ADR-014](0014-operator-identity.md#adr-014-identity-модель-оператора-archon)** — an Archon's effective roles = direct ∪ via Synod (amendment).
- **[ADR-028](0028-rbac-storage.md#adr-028-rbac-storage--postgres)** — +3 Synod tables, the snapshot build is supplemented with group roles, `rbac:invalidate` invalidation covers Synod mutations (amendment).
- **[ADR-047](0047-purview.md#adr-047-purview--scoped-rbac-visibility-of-nodes-role-default_scope--extended-selector)** — effective roles are resolved accounting for Synod; group-scope is NOT introduced, scope lives on roles (amendment).
- **[ADR-013](0013-bootstrap-archon.md#adr-013-bootstrap-первого-архонта)** — the self-lockout invariant expands Synod membership (f).

**Rejected alternatives.**
- **(a) `default_scope` on Synod (group-scope).** Would have given "narrow the group's roles with a shared scope." Rejected for the MVP: scope already lives on roles ([ADR-047](0047-purview.md#adr-047-purview--scoped-rbac-visibility-of-nodes-role-default_scope--extended-selector)), a second scope layer on the group complicates resolution (intersecting group-scope ∩ role-scope) and Purview matching with no confirmed need. Additive — introduced later without a breaking change (a new nullable column `synods.default_scope`).
- **(b) Nested Synods (a group within a group).** Transitive expansion, a risk of cycles, a more expensive snapshot build. Rejected for the MVP — the flat model covers "a role set for a team." Additive on real demand.
- **(c) Store group membership in a JWT claim.** Would repeat BUG-1 ([ADR-028](0028-rbac-storage.md#adr-028-rbac-storage--postgres)): a claim is informational, the enforcer resolves from the DB snapshot. Synod membership is `synod_operators` rows, the authority is the DB, not a claim.
- **(d) Renaming a Synod (a mutable `name` PK).** A `synod.update` that changes `name` was rejected (amend 2026-06-09): renaming a PK causes audit drift (the old name in `audit_log` loses its link to the row), a desync window in the enforcer snapshot (`synod_operators`/`synod_roles` reference `synod_name`), and an asymmetry with `role.update` (a role isn't renamed either). `synod.update` changes **only `description`** — the name is immutable, like `rbac_roles.name` / `souls.sid` / `incarnation.name`. If renaming is ever needed — a separate transactional operation (rename + a cascading FK rewrite + an audit trail), not part of `synod.update`.

**What is deferred (post-MVP / backlog, additive).**
- **Group-scope** (`synods.default_scope`, narrowing the group's roles) — (a).
- **Nested Synods** (a group within a group) — (b).
- Both are extensible without a breaking change on top of the three MVP tables.
