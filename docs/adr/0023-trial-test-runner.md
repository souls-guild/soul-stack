# ADR-023. Тест-раннер Trial (`soul-trial`) и DSL-coverage

- **Контекст.** Нужно полноценное тестирование Destiny/Scenario с двумя ортогональными метриками покрытия: (1) **DSL-coverage** в духе Ansible/Molecule — какие задачи / CEL-ветки (`where:`/`when:`/…) / `enum:`-значения / `state_changes`-переходы реально прогнаны тестами; (2) **code coverage** обычного Go-кода ядра. Эталон формата теста уже есть — `migrations/<NNN>_to_<MMM>/tests/<case>.yml` (ADR-019, `state_before → state_after`). Полная спека и open Q — [`docs/destiny/testing.md`](../destiny/testing.md); этот ADR фиксирует только архитектурные решения, прошедшие propose-and-wait.
- **Сущность — `Trial`** (испытание Destiny/Scenario перед воплощением), бинарь `soul-trial`, метрика `trial coverage`. Внесено в [naming-rules.md](../naming-rules.md). Прошло propose-and-wait (развилки (а)–(д), решения пользователя — `.pm/tasks/2026-05-22-testing-framework/decisions.md`).
- **Решения.**
  - **Уровни тестов.** **L0 render-only (hermetic)** — vault-resolve→CEL→text/template дают ожидаемый `RenderedTask`/итоговый `incarnation.state` по fixtures, in-process, без хоста. **L1 migration** — state_schema-DSL (ADR-019) `state_before→transform→state_after`. **L2 single-host** (docker, реальные модули) — **реализован** (см. ниже). **L3 multi-host** (оркестрация) — **отложен** (см. ниже).
  - **L2 single-host — РЕАЛИЗОВАН (Вариант A, test-only под build-tag `integration`).** Прежняя формулировка «L2 post-MVP/открыт» — **снята** (был drift docs↔код). Фактическая реализация — **Вариант A**: oneshot `soul apply` в эфемерном docker-контейнере (testcontainers), план задач — **тот же in-process Keeper-side render** (`renderCase`, общий старт с L0), без поднятия Keeper-gRPC. Код — [`keeper/internal/trial/l2_run.go`](../../keeper/internal/trial/l2_run.go) / [`l2_case.go`](../../keeper/internal/trial/l2_case.go), пилот — node-exporter ([`l2_test.go`](../../keeper/internal/trial/l2_test.go)). **Test-only под build-tag `integration`**: L2 гоняется в Go-тестах keeper-модуля, в проде-`Run` ([`run.go`](../../keeper/internal/trial/run.go)) L2-кейс маркируется `Skipped`. Включение L2 в прод-CLI (`soul-trial run` через docker-SDK напрямую) — **отдельная отложенная задача**, не в этом ADR.
  - **L3 multi-host — отложен.** Оркестрация на топологии кластера (кто master/replica, ассерты на топологию и committed `incarnation.state` cross-host). Зависит от **Phase 3 ([ADR-027](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim)) распределённого serial** и от **multi-host `stand:`** — формат топологии стенда под propose-and-wait, не закрывается молча (расширение open Q №3 testing.md).
  - **`assert`-секции по уровням.** `assert.rendered_tasks` / `assert.state_changes` / `assert.state_after` — **L0** (герметичны, без хоста). `assert.state_after` — детерминированный итоговый `incarnation.state` (базовый `fixtures.state` + отрендеренные `state_changes.sets`, зеркало прод-коммита run.go `mergeStateChanges`, полная сверка как L1). `assert.dispatch` — **L3** (dispatch-план осмыслен только на multi-host; на single-host один синтетический хост). L0-загрузчик отвергает `assert.dispatch` strict-декодом как unknown-key — намеренно (приведение к нереализованной секции — явная ошибка, не silent-skip).
  - **Граница MVP — L0 + L1.** Уточнение по факту кода: **L0 — настоящий пилот фреймворка**, т.к. он ассертит уже написанный рендер-пайплайн (`keeper/internal/render`, `shared/cel`, `shared/tmpl`). **L1 опирается на движок ADR-019, который в коде пока существует только как пилот-merge** (`keeper/internal/scenario/state.go` — `rename`/`set`/`delete`/`move`), а НЕ как полный forward-only migration-движок с `foreach`+CEL-в-`set.value`+атомарной PG-tx. `keeper/internal/migrate` — это SQL-схема Keeper-а (golang-migrate), НЕ ADR-019. Следствие: L1 в MVP требует достройки движка ADR-019 первым; L0 готов к реализации сразу.
  - **Раскладка — вариант A: `keeper/cmd/soul-trial`** (второй бинарь-артефакт модуля `keeper`, параллель `cmd/keeper`). Импортит `keeper/internal/render`/`scenario`/migration-движок напрямую. Вариант B (вынос в `shared/`) отвергнут — нарушает инвариант ADR-011 «`shared/vault/` только клиентская часть» (render тянет server-side Vault/topology/incarnation). Поправка [ADR-004](0004-binaries.md#adr-004-раскладка-бинарей--keeper-soul-soul-lint-push-режим--модуль-внутри-keeper) (3→4 артефакта, граница `soul-lint`↔`soul-trial`) и [ADR-011](0011-go-layout.md#adr-011-раскладка-go-кода-gowork-с-модулями-по-сторонам) (`keeper/cmd/soul-trial`) применены.
  - **DSL-coverage hook — `CoverageSink` в `shared/cel`.** Точка — `Engine.EvalExpression` **после успешного `prg.Eval`** (НЕ `compile`: `Engine` кеширует `cel.Program`, хук на compile занизил бы метрику при повторных eval). `EvalInterpolation` сам вызывает `EvalExpression` → один хук ловит и top-level expression-ключи, и интерполяционные блоки (дублировать не нужно). Контракт: опциональное поле `Engine.sink CoverageSink` (nil по умолчанию → no-op в проде, ненулевой оверхед только при `soul-trial`). Sink получает `(expr string, out ref.Val)` — достаточно для учёта truthy/falsy ветки. Гранулярность MVP — «выражение truthy/falsy», под-ветки внутри одного CEL вне scope.
  - **code coverage (Go)** — стандартный `go test -coverprofile`, не изобретать; держится отдельно от DSL-coverage, **два разных числа, два имени** (`code coverage` / `trial coverage`).
  - **Формат case.yml** (детали — testing.md, НЕ здесь): расширение migration-эталона секциями `fixtures:` (input/essence/soulprint/vault) + `mocks.register` + опциональный `assert:` с тремя независимыми подсекциями `rendered_tasks` / `dispatch` / `state_after` (кейс выбирает нужное).
  - **L0 для standalone destiny — через scenario-обёртку `apply:destiny`.** L0 для переиспользуемой (standalone) destiny выполняется через минимальную scenario-обёртку с единственной задачей `apply:destiny` — destiny не рендерится отдельным путём; единственная точка инвокации destiny (scenario `apply:` / Keeper) сохраняется и в тестах. Каноническая раскладка обёртки — подкаталог самой destiny `destiny-<name>/_trial/scenario/apply/`. Это уточнение, не разворот: harness завязан на `scenario/<name>/main.yml`, и обёртка переиспользует тот же путь рендера, а не вводит второй. Рабочий пример — [`examples/destiny/destiny-node-exporter/_trial/`](../../examples/destiny/destiny-node-exporter/_trial/scenario/apply/main.yml).
  - **L0-резолв `apply:destiny` зеркалит прод-модель (slice A).** Резолвер L0 ([`keeper/internal/trial/destiny.go`](../../keeper/internal/trial/destiny.go)) повторяет двухступенчатый прод-резолв ([`keeper/internal/scenario/destiny.go`](../../keeper/internal/scenario/destiny.go)): (1) `apply:destiny`-имя → запись в `service.yml::destiny[]` → `{name, ref, git?}`, **необъявленная зависимость отвергается** (та же ошибка, что в проде, ADR-007); (2) URL по гибридному правилу — per-entry `git`-override побеждает, иначе имя подставляется в `default_destiny_source`. В L0 источник обязан быть **герметичным `file://`** (не-`file://`-схема → явная ошибка «L0 герметичен»); `default_destiny_source` для L0 живёт в `case.yml::fixtures` (то же имя ключа, что у keeper.yml), путь — относительно service-root кейса (для standalone-обёртки service-root = `_trial/`, поэтому обёртка несёт собственный `_trial/service.yml` с объявленной зависимостью). **Граница securejoin — destiny-root** (доверенная часть URL до сегмента с `{name}`), не service-root: `{name}` не может вырваться через `../` за объявленный каталог destiny. **Прежний эвристический перебор каталогов** (`[serviceRoot, parent, grandparent]` + конвенция `destiny-<name>/`) **отвергнут**: он не отражал прод и врал на cross-location раскладке (сервис и standalone destiny в разных поддеревьях `examples/`). Мини-пилот контракта L0-резолвера — service-redis (3 сценария) + три `_trial`-обёртки + service-pilot-destiny.
- **Что закрывает / оставляет открытым.** Закрывает open Q №1/6/7/8 в [testing.md](../destiny/testing.md) (форма инструмента, хранение coverage, CI-gate, состав метрик). С реализацией L2 (Вариант A) закрываются: **№4** (real vs mock) → real-only (mock-уровень = L0 render-only, отдельного mock-L2 нет); **№5** (идемпотентность) → обязательна с opt-out (`expect_idempotent` default true); **№3** (sandbox) → закрыт **частично** — single-host docker-стенд (`stand: {driver, image, mode: push}`, ephemeral container per-case), **multi-host `stand:` остаётся открытым** (расширение для L3, propose-and-wait по формату топологии). Нормировка решений — [testing.md](../destiny/testing.md). НЕ нормирует здесь: формат `case.yml`/`verify:`/`expect:`, раскладку `tests/`, multi-host `stand:` (живут в testing.md/orchestration.md).
- **Trade-off.** `soul-trial` в keeper-модуле — расхождение «где код» vs «роль инструмента», принято ради отказа от рефакторинга рабочего рендер-пайплайна. CoverageSink в `shared/cel` — минимальная аддитивная инъекция, не плодит второй парсер (уважает ADR-010 «один движок»).
