# ADR-023. Test Runner Trial (`soul-trial`) and DSL Coverage

- **Context.** We need full-fledged testing of Destiny/Scenario with two orthogonal coverage metrics: (1) **DSL coverage** in the spirit of Ansible/Molecule — which tasks / CEL branches (`where:`/`when:`/…) / `enum:` values / `state_changes` transitions are actually exercised by tests; (2) regular **code coverage** of the core's Go code. A reference test format already exists — `migrations/<NNN>_to_<MMM>/tests/<case>.yml` (ADR-019, `state_before → state_after`). The full spec and open Q's are in [`docs/destiny/testing.md`](../destiny/testing.md); this ADR fixes only the architectural decisions that went through propose-and-wait.
- **Entity — `Trial`** (a trial of Destiny/Scenario before it is enacted), binary `soul-trial`, metric `trial coverage`. Entered into [naming-rules.md](../naming-rules.md). Went through propose-and-wait (forks (a)–(d), the user's decisions — `.pm/tasks/2026-05-22-testing-framework/decisions.md`).
- **Decisions.**
  - **Test levels.** **L0 render-only (hermetic)** — vault-resolve→CEL→text/template produce the expected `RenderedTask`/final `incarnation.state` from fixtures, in-process, without a host. **L1 migration** — the state_schema DSL (ADR-019) `state_before→transform→state_after`. **L2 single-host** (docker, real modules) — **implemented** (see below). **L3 multi-host** (orchestration) — **deferred** (see below).
  - **L2 single-host — IMPLEMENTED (Variant A, test-only under build-tag `integration`).** The previous wording "L2 post-MVP/open" is **lifted** (there was a docs↔code drift). The actual implementation is **Variant A**: a oneshot `soul apply` in an ephemeral docker container (testcontainers), the task plan is **the same in-process Keeper-side render** (`renderCase`, shared start with L0), without bringing up Keeper-gRPC. Code — [`keeper/internal/trial/l2_run.go`](../../keeper/internal/trial/l2_run.go) / [`l2_case.go`](../../keeper/internal/trial/l2_case.go), pilot — node-exporter ([`l2_test.go`](../../keeper/internal/trial/l2_test.go)). **Test-only under build-tag `integration`**: L2 runs in the keeper module's Go tests, in production `Run` ([`run.go`](../../keeper/internal/trial/run.go)) the L2 case is marked `Skipped`. Enabling L2 in the production CLI (`soul-trial run` via the docker SDK directly) is a **separate deferred task**, not in this ADR.
  - **L3 multi-host — deferred.** Orchestration over cluster topology (who is master/replica, assertions on topology and the committed cross-host `incarnation.state`). Depends on **Phase 3 ([ADR-027](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim)) distributed serial** and on **multi-host `stand:`** — the stand-topology format is under propose-and-wait, not closed silently (an extension of open Q No. 3 in testing.md).
  - **Amendment 2026-06-22. L0 render-multi-host roster — CLOSED; L3 dispatch-multi-host remains deferred.** What used to be lumped together under "multi-host deferred" is now split by level:
    - **CLOSED (L0, render-only):** `fixtures.hosts: [...]` in `case.yml` — a roster of N hosts (a host record `sid`/`covens`/`role`/`soulprint`/`choirs`). The multi-host render engine already works (`keeper/internal/render/hosts_test.go` runs a 3-host roster: `soulprint.hosts` projection, `.where(...)`/`size()`, order determinism, `loop:` over `soulprint.hosts`, `state_changes`). The only ceiling was `keeper/internal/trial/harness.go::fixtureHosts`, which hard-coded returning a single host; the ceiling is lifted. This unblocks L0 cases with a `size(soulprint.hosts)` guard, node determinism, and cross-host CEL projection for cluster/sentinel/sentinel_only. `fixtures.soulprint` (a single map) remains **single-host sugar** (zero back-compat cost); `fixtures.soulprint` and `fixtures.hosts` in the same case are **mutually exclusive** (a strict decode error, in the spirit of the harness's strict decode).
    - **REMAINS DEFERRED (L3, dispatch):** `assert.dispatch` (who actually executes as master), the committed cross-host `incarnation.state` after the barrier, a docker `stand:` over topology. L0-render-multi-host does **not** give dispatch — it only checks the Keeper-side render plan/projections over the roster, without real per-host execution. L3 still depends on Phase 3 of ADR-027 and multi-host `stand:` (open Q No. 3 in testing.md, propose-and-wait on the topology format). Do not close L3 silently.
  - **`assert` sections by level.** `assert.rendered_tasks` / `assert.state_changes` / `assert.state_after` — **L0** (hermetic, no host). `assert.state_after` — the deterministic final `incarnation.state` (base `fixtures.state` + rendered `state_changes.sets`, mirroring the production commit's `run.go` `mergeStateChanges`, a full check like L1). `assert.dispatch` — **L3** (a dispatch plan only makes sense on multi-host; on single-host there is one synthetic host). The L0 loader rejects `assert.dispatch` via strict decode as an unknown key — deliberately (referencing an unimplemented section is an explicit error, not a silent skip).
  - **MVP boundary — L0 + L1.** A clarification based on the actual code: **L0 is the real pilot of the framework**, since it asserts an already-written render pipeline (`keeper/internal/render`, `shared/cel`, `shared/tmpl`). **L1 relies on the ADR-019 engine, which in the code currently exists only as a pilot merge** (`keeper/internal/scenario/state.go` — `rename`/`set`/`delete`/`move`), and NOT as the full forward-only migration engine with `foreach`+CEL-in-`set.value`+atomic PG tx. `keeper/internal/migrate` is Keeper's SQL schema (golang-migrate), NOT ADR-019. Consequence: L1 in the MVP requires finishing the ADR-019 engine first; L0 is ready for implementation right away.
  - **Layout — Variant A: `keeper/cmd/soul-trial`** (a second binary artifact of the `keeper` module, parallel to `cmd/keeper`). It imports `keeper/internal/render`/`scenario`/the migration engine directly. Variant B (moving it into `shared/`) is rejected — it breaks the ADR-011 invariant that "`shared/vault/` is client-side only" (render pulls in server-side Vault/topology/incarnation). The amendment to [ADR-004](0004-binaries.md#adr-004-binary-layout--keeper-soul-soul-lint-push-mode-as-a-module-inside-keeper) (3→4 artifacts, the `soul-lint`↔`soul-trial` boundary) and [ADR-011](0011-go-layout.md#adr-011-раскладка-go-кода-gowork-с-модулями-по-сторонам) (`keeper/cmd/soul-trial`) have been applied.
  - **DSL-coverage hook — `CoverageSink` in `shared/cel`.** The hook point is `Engine.EvalExpression` **after a successful `prg.Eval`** (NOT `compile`: `Engine` caches the `cel.Program`, a hook on compile would undercount the metric on repeated evals). `EvalInterpolation` itself calls `EvalExpression` → a single hook catches both top-level expression keys and interpolation blocks (no need to duplicate). Contract: an optional field `Engine.sink CoverageSink` (nil by default → a no-op in production, non-zero overhead only under `soul-trial`). The sink receives `(expr string, out ref.Val)` — enough to record the truthy/falsy branch. MVP granularity is "expression truthy/falsy"; sub-branches inside a single CEL expression are out of scope.
  - **Code coverage (Go)** — standard `go test -coverprofile`, no need to reinvent it; kept separate from DSL coverage, **two different numbers, two names** (`code coverage` / `trial coverage`).
  - **`case.yml` format** (details — testing.md, NOT here): an extension of the migration reference with `fixtures:` sections (input/essence/soulprint/vault) + `mocks.register` + an optional `assert:` with three independent subsections `rendered_tasks` / `dispatch` / `state_after` (the case picks whichever it needs).
  - **L0 for standalone destiny — via a scenario wrapper `apply:destiny`.** L0 for reusable (standalone) destiny is run via a minimal scenario wrapper with a single task `apply:destiny` — destiny is not rendered through a separate path; the single invocation point for destiny (scenario `apply:` / Keeper) is preserved in tests too. The canonical layout of the wrapper is the destiny's own subdirectory `<name>/_trial/scenario/apply/`. This is a clarification, not a reversal: the harness is tied to `scenario/<name>/main.yml`, and the wrapper reuses the same render path rather than introducing a second one. A working example — [`examples/destiny/node-exporter/_trial/`](../../examples/destiny/node-exporter/_trial/scenario/apply/main.yml).
  - **L0 resolution of `apply:destiny` mirrors the production model (slice A).** The L0 resolver ([`keeper/internal/trial/destiny.go`](../../keeper/internal/trial/destiny.go)) repeats the two-stage production resolution ([`keeper/internal/scenario/destiny.go`](../../keeper/internal/scenario/destiny.go)): (1) an `apply:destiny` name → a record in `service.yml::destiny[]` → `{name, ref, git?}`, **an undeclared dependency is rejected** (the same error as in production, ADR-007); (2) the URL follows a hybrid rule — a per-entry `git` override wins, otherwise the name is substituted into `default_destiny_source`. In L0 the source must be **hermetic `file://`** (a non-`file://` scheme → an explicit "L0 is hermetic" error); `default_destiny_source` for L0 lives in `case.yml::fixtures` (the same key name as in keeper.yml), the path is relative to the case's service-root (for a standalone wrapper the service-root = `_trial/`, so the wrapper carries its own `_trial/service.yml` with the declared dependency). **The securejoin boundary is the destiny-root** (the trusted part of the URL before the `{name}` segment), not the service-root: `{name}` cannot escape the declared destiny directory via `../`. **The previous heuristic directory search** (`[serviceRoot, parent, grandparent]` + the `destiny-<name>/` convention) **is rejected**: it did not reflect production and lied on cross-location layouts (a service and a standalone destiny in different `examples/` subtrees). The mini-pilot of the L0-resolver contract is redis (3 scenarios) + three `_trial` wrappers + a pilot destiny.
- **What it closes / leaves open.** Closes open Q No. 1/6/7/8 in [testing.md](../destiny/testing.md) (the tool's form, coverage storage, CI gate, metric composition). With L2 implemented (Variant A), the following are closed: **No. 4** (real vs mock) → real-only (the mock level = L0 render-only, there is no separate mock-L2); **No. 5** (idempotency) → mandatory with opt-out (`expect_idempotent` default true); **No. 3** (sandbox) → closed **partially** — a single-host docker stand (`stand: {driver, image, mode: push}`, an ephemeral container per case), **multi-host `stand:` remains open** (an extension for L3, propose-and-wait on the topology format). The normalization of decisions is in [testing.md](../destiny/testing.md). Not normalized here: the `case.yml`/`verify:`/`expect:` format, the `tests/` layout, multi-host `stand:` (they live in testing.md/orchestration.md).
- **Trade-off.** `soul-trial` living in the keeper module is a divergence between "where the code lives" and "the tool's role," accepted in order to avoid refactoring the working render pipeline. `CoverageSink` in `shared/cel` is a minimal additive injection, it does not spawn a second parser (it respects ADR-010's "one engine").

- **Amendment 2026-06-23 (`expect_render_error` — L0 can assert a render abort). Pilot.** The previous L0 harness only asserted a **successful** render (`rendered_tasks`/`state_changes`/`state_after`) — it could not assert a **render error** (a render abort). With the introduction of `assert:` (a keeper-side render-time precondition, [ADR-009](0009-scenario-dsl.md#adr-009-scenario--полная-dsl-задач-destiny-граница-с-destiny--рекомендация) amendment 2026-06-23), the first `false` predicate **aborts the render** — a failure case of this mechanism was inexpressible with the previous harness. The decision:
  - **A top-level field `expect_render_error: "<substring>"`** in `case.yml`: the case EXPECTS the render to fail with an error containing the substring. A render success when `expect_render_error` is set → the case FAILs; a render error that doesn't match the substring → FAILs; a render error that matches → PASSes.
  - **Mutually exclusive with `assert.rendered_tasks`.** `expect_render_error` (expecting an abort) and `assert.rendered_tasks` (expecting a plan) describe opposite outcomes → having both in the same case at once is a strict validation error. When `expect_render_error` is set, the `assert` section is empty (there is no plan).
  - **A minimal addition, not a harness rewrite.** One opt-in field + a branch in `RunCase`; it does not touch the coverage/state sections of the mechanism. Also an enabler for the future `required_when` (also a render abort).
  - Implementation — `keeper/internal/trial` (`Case.ExpectRenderError`, a branch in `RunCase`/`validate`).
