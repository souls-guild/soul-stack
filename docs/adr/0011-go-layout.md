# ADR-011. Раскладка Go-кода: go.work с модулями по сторонам

- **Контекст.** Soul Stack — три бинаря (`keeper`, `soul`, `soul-lint`, [ADR-004](0004-binaries.md#adr-004-раскладка-бинарей--keeper-soul-soul-lint-push-режим--модуль-внутри-keeper)) плюс плагины как отдельные репозитории (`soul-mod-*`, `soul-cloud-*`, `soul-ssh-*` — см. [Plugin-инфраструктура](../architecture.md#plugin-инфраструктура)). Open Q №9 «Раскладка Go-кода» оставался открытым: моно-модуль с тремя `cmd/` vs go.work с несколькими модулями. Решение должно адресовать (1) изоляцию `soul` от server-side кода Keeper (минимальная поверхность Soul, требование безопасности [ADR-002](0002-transport-grpc-ha.md#adr-002-транспорт-keeper--souls--grpc-bidirectional-stream-поверх-mtls-ha-кластер-keeper)/[ADR-004](0004-binaries.md#adr-004-раскладка-бинарей--keeper-soul-soul-lint-push-режим--модуль-внутри-keeper)), (2) shared-пакеты для сквозных требований (метрики, OTel, hot-reload, ротация логов, Vault-клиент, CEL/text/template — [ADR-010](0010-templating.md#adr-010-шаблонизатор-cel-для-yaml-выражений-go-texttemplate-для-файлов)), (3) публичный SDK для авторов плагинов без необходимости тянуть весь основной код.
- **Решение.** Принимается **вариант B — go.work с модулями по сторонам**. Структура верхнего уровня:

  ```
  soul-stack/                                 # repo root, НЕ модуль
  ├── go.work                                 # use ./keeper ./soul ./soul-lint ./shared ./sdk ./proto ./proto/plugin
  ├── proto/                  # go.mod | внутренние Keeper↔Soul (keeper/v1) + gen/go/  (Operator API — OpenAPI-спека + oapi-codegen, ADR-051; operator/v1 упразднён)
  ├── proto/plugin/           # go.mod | ВЛОЖЕННЫЙ модуль для plugin-контрактов (soulmodule/clouddriver/sshprovider) + gen/go/
  ├── shared/                 # go.mod | obs/ log/ config/ vault/ tlsx/ cel/ tmpl/  (vault/ — только клиентская часть)
  ├── sdk/                    # go.mod | публичный SDK: module/ clouddriver/ sshprovider/ handshake/
  ├── keeper/                 # go.mod | cmd/keeper + cmd/soul-trial + internal/{apiserver,mcpserver,rbac,store,reaper,push,pluginhost,cloud,vault,coremod,render,scenario}
  │                           #          (soul-trial — офлайн-раннер Trial, ADR-023; ассертит internal/render+scenario)
  ├── soul/                   # go.mod | cmd/soul + internal/{agent,connection,modules,coremod,cache}  — НЕ require keeper
  ├── soul-lint/              # go.mod | cmd/soul-lint + internal/  — НЕ require keeper/soul/sdk
  ├── examples/               # только non-Go артефакты (YAML и т.п.)
  ├── docs/
  └── tools/                  # вспомогательные tools (без собственной Go-логики)
  ```

  Ключевые свойства:
  - **Изоляция Soul гарантируется Go-компилятором.** `soul/go.mod` не содержит `require .../keeper`; попытка импорта серверного кода Keeper-а из Soul физически не скомпилируется. Это поднимает безопасностный инвариант [ADR-002](0002-transport-grpc-ha.md#adr-002-транспорт-keeper--souls--grpc-bidirectional-stream-поверх-mtls-ha-кластер-keeper)/[ADR-004](0004-binaries.md#adr-004-раскладка-бинарей--keeper-soul-soul-lint-push-режим--модуль-внутри-keeper) с уровня CI-линтера на уровень модульной границы.
  - **Общий контракт core/custom-модулей.** Core-модули на обеих сторонах (`keeper/internal/coremod/`, `soul/internal/coremod/`) реализуют тот же интерфейс из `sdk/module/`, что и custom-модули в отдельных репозиториях.
  - **Plugin-proto подмодуль.** `proto/plugin/` — отдельный вложенный go.mod; плагин-авторы тянут только три service-контракта + handshake, без Keeper↔Soul или Operator API. Минимальная зависимость.
  - **Vault-клиент изолирован.** `shared/vault/` — только клиентская часть (Soul-safe: token lookup, transit decrypt). Server-side Vault операции (CA для SoulSeed, чтение Essence, PKI engine) живут в `keeper/internal/vault/` и **не экспортируются** через `shared/`.
  - **Module path placeholder** = `github.com/soul-stack/soul-stack/<module>` на момент локальной разработки. При push на удалённый хостинг — sed-замена по всем `go.mod`.
  - **Версионная политика — совместные теги.** Один git-tag на корневой репо = одна логическая версия всех модулей (`keeper/`, `soul/`, `soul-lint/`, `shared/`, `sdk/`, `proto/`, `proto/plugin/` все получают `v0.1.0` при `git tag v0.1.0`). Плагин-авторам нужна одна цифра, не семь.
  - **Generated Go-код из proto — committed** в `proto/gen/go/` и `proto/plugin/gen/go/` (повторяемые билды, без protoc у разработчика). CI проверяет идемпотентность `make gen`.
  - **`cmd/` внутри модулей** оставлен (`keeper/cmd/keeper/main.go`) даже при одном бинаре на модуль — пригодится для будущих утилит (`keeper-migrate` и т.п.). Первый такой кейс — **`keeper/cmd/soul-trial`** (второй бинарь-артефакт модуля `keeper`): офлайн-раннер Trial ([ADR-004](0004-binaries.md#adr-004-раскладка-бинарей--keeper-soul-soul-lint-push-режим--модуль-внутри-keeper), [ADR-023](0023-trial-test-runner.md#adr-023-тест-раннер-trial-soul-trial-и-dsl-coverage)). Лежит в keeper-модуле, т.к. ассертит `keeper/internal/render`/`scenario`/migration-движок напрямую (вариант B — вынос в `shared/` — нарушил бы инвариант «`shared/vault/` только клиентская часть», т.к. render тянет server-side Vault/topology/incarnation). `go.work` править не нужно: модуль `./keeper` уже в use-листе.
  - **`shared/` — один модуль** на старте, с подпакетами; дробить при росте.
  - **`examples/` — только non-Go артефакты** (YAML, конфиги, шаблоны). Runnable Go-примеры — в `tools/` или отдельных репо. Это закрывает риск «случайно завести Go-модуль в examples».
- **Consequences.**
  - Первый коммит кода создаёт 7 `go.mod`-файлов + `go.work` + пустые `cmd/<binary>/main.go` для трёх бинарей + placeholder-`proto/keeper/v1/keeper.proto` с одним RPC `Ping` для верификации генерации.
  - Плагин-авторы (`soul-mod-*`, `soul-cloud-*`, `soul-ssh-*`) в своих репозиториях пишут `require github.com/soul-stack/soul-stack/proto/plugin v0.X.Y` и `require github.com/soul-stack/soul-stack/sdk v0.X.Y` — без зависимости от keeper/soul/shared.
  - Внешние интеграторы (для Operator API client-SDK) тянут `require github.com/soul-stack/soul-stack/proto v0.X.Y` либо будущий `sdk/api` модуль (отдельная задача после первого релиза Operator API).
  - Изменение proto-контракта Keeper↔Soul правится в `proto/keeper/v1/`, regenerate, правка хост-стороны — всё в одном PR в одном репо. **Форма Operator API** правится не в proto, а в **OpenAPI-спеке** ([`docs/keeper/openapi.yaml`](../keeper/openapi.yaml)) → `make gen-api` (oapi-codegen → `keeper/internal/api/oapi/`, [ADR-051](0051-operator-api-codegen.md#adr-051-operator-api-codegen-openapi--go-типы-oapi-codegen-types-only--strict)); markdown-нормирование транспорта и mapping endpoint ↔ MCP-tool ↔ permission — [`docs/keeper/operator-api.md`](../keeper/operator-api.md). `proto/operator/v1` упразднён (см. Amendment ниже).
  - Изменение plugin-контракта правится в `proto/plugin/v1/`, regenerate `proto/plugin/gen/go/`, поднимается версия `proto/plugin/` (через общий repo-tag); обновление зависимых плагин-репо — отдельная синхронизация.
  - Версия Go-модулей внутри основного репо — semver-тег корневого репо. Это **явно артикулированное исключение в [ADR-007](0007-versioning-git-ref.md#adr-007-версионирование-артефактов--через-git-ref-а-не-через-поле-в-манифесте)** для Go-модулей (`go.mod` `require` требует semver-тег).
  - **Server-side runtime-зависимости в `<binary>/internal/`, не в `shared/`.** Тяжёлые server-side драйверы (Postgres `pgx`, server-side Vault, server-side Redis-операции, server-side OTel-коллектор) живут в `<binary>/internal/<subsystem>/`, не в `shared/`. В `shared/` допустимы: interfaces, clients (Soul-safe), helpers без тяжёлого init-graph, типы данных и enums. Прецедент: `shared/vault/` (client-only) vs `keeper/internal/vault/` (server-side); по той же логике `shared/audit/` держит `Writer`-interface + типы + masking + ULID-helper, а pgx-реализация — в `keeper/internal/auditpg/`. Обобщённо: любой импорт server-side driver-а в `shared/` тащит package-level init (pgtype-регистрация, `sync.Pool`, и т.п.) в **каждый** бинарь, включая Soul, что нарушает компиляторную Soul-изоляцию выше.
  - Open Q №9 закрыт; open Q №5 и №2 остаются открытыми и могут уточняться без правки этого ADR.
- **Trade-offs.**
  - Семь `go.mod`-файлов на старте — больше boilerplate, чем у моно-module-варианта. Закрывается тем, что изоляция Soul гарантируется компилятором, а не CI-линтером (требование безопасности).
  - IDE и `go build ./...` обходят workspace — небольшая overhead в IDE-отзывчивости и build-времени. Не критично при семи модулях.
  - SDK как Go-library версионируется semver-тегом — явно артикулированное исключение в [ADR-007](0007-versioning-git-ref.md#adr-007-версионирование-артефактов--через-git-ref-а-не-через-поле-в-манифесте) для Go-модулей (отличается от «версия артефакта = git ref»).
  - Если экосистема плагинов вырастет до десятков внешних авторов и потребуется отдельный SDK-репо — миграция B→C (вынос `sdk/` + `proto/plugin/` в `github.com/soul-stack/soul-stack-sdk`) механическая, не блокирует решение сейчас.

- **Amendment (OpenAPI-эпик, 2026-06-09): упразднение `proto/operator/v1`.** Транспорт Operator API — **HTTP/JSON** ([ADR-004](0004-binaries.md#adr-004-раскладка-бинарей--keeper-soul-soul-lint-push-режим--модуль-внутри-keeper)), не gRPC; gRPC-сервиса над `operator/v1` нет. `proto/operator/v1` (6 `.proto`) был **паразитной второй моделью формы** рядом с OpenAPI-спекой — рукописные DTO молча расходились со спекой (request-drift). Источник правды **формы** Operator API → **OpenAPI-спека** ([`docs/keeper/openapi.yaml`](../keeper/openapi.yaml)), Go-типы из неё генерирует **oapi-codegen** в пакет **`keeper/internal/api/oapi/`** ([ADR-051](0051-operator-api-codegen.md#adr-051-operator-api-codegen-openapi--go-типы-oapi-codegen-types-only--strict)).
  - `proto/operator/v1` и его generated Go **удаляются** (отдельным слайсом). Makefile gen-target теряет operator-proto-секцию.
  - **`go.work` и структура модулей НЕ меняются:** модуль `proto/` остаётся — в нём живёт `proto/keeper/v1/` (контракт Keeper↔Soul). Инвариант изоляции `proto/plugin/` (плагин-авторы тянут минимальную зависимость) не затрагивается.
  - Все упоминания «Operator API (operator/v1)» / «messages для N HTTP-эндпоинтов» в тексте выше читаются как **«Operator API — OpenAPI-спека + oapi-codegen» ([ADR-051](0051-operator-api-codegen.md#adr-051-operator-api-codegen-openapi--go-типы-oapi-codegen-types-only--strict))**.

- **Amendment (разворот на code-first, 2026-06-12): huma-реестр операций vs пакет `oapi` — переходное сосуществование.** [ADR-054](0054-openapi-code-first.md) развернул источник формы Operator API: **Go-типы → OpenAPI** (huma v2 + humachi) вместо **OpenAPI → Go-типы** (oapi-codegen, ADR-051). По завершении тиража пакет **`keeper/internal/api/oapi/`** (committed `types.gen.go`/`server.gen.go`) и мост **`oapi_strict.go`** **умирают**, форму несут Go-struct-ы huma-операций (`keeper/internal/api/huma_*.go`), спека — производная (huma-dump). **Переходное состояние (pilot + тираж):** оба каркаса СОСУЩЕСТВУЮТ под одним `chi.Mux` — часть роутов на huma (pilot: `POST /v1/cadences`), остальное на oapi-strict. `go.work` и структура модулей НЕ меняются: huma-операции живут в `keeper/internal/api/` рядом с существующими handler-ами; новый прямой require `github.com/danielgtaylor/huma/v2` в `keeper/go.mod`. Снос `oapi`-пакета + Makefile-таргетов `gen-api`/`check-gen-api` — отдельным слайсом ПОСЛЕ завершения тиража.
  - **Финализация (2026-06-13, HEAD `fde65bf`): teardown ВЫПОЛНЕН.** Тираж завершён, пакет **`keeper/internal/api/oapi/`** (committed `types.gen.go`/`server.gen.go`), мост **`oapi_strict.go`**/`strictServer`, пакет `keeper/internal/api/meta/` (embed-источник спеки) и Makefile-таргеты `gen-api`/`check-gen-api`/`sync-openapi` — **удалены** (заменены `gen-openapi`/`check-openapi`). Форму Operator API целиком несут Go-struct-ы huma-операций (`keeper/internal/api/huma_*.go` + агрегатор `huma_full_spec.go`, native enum-каталог `huma_enums.go`); `docs/keeper/openapi.yaml` — производный снимок. Дерево комментария вверху файла (стаб-ссылка `Operator API — OpenAPI-спека + oapi-codegen, ADR-051; operator/v1 упразднён`) читается как **«Operator API — huma code-first, ADR-054; oapi-codegen и operator/v1 упразднены»**.
