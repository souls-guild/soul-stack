# ADR-007. Artifact Versioning — via git ref, not a manifest field

- **Context.** The source code of Soul Stack artifacts (**Service**, **Destiny**, **Module**, plugins — see the table in the ["Soul Stack artifacts: what's in git, what's in the database"](../architecture.md#soul-stack-artifacts-whats-in-git-whats-in-the-database) section) lives in git repositories. Originally each artifact declared its own version via a `version:` field in the manifest (`service.yml`, `destiny.yml`, `manifest.yaml`), and dependencies were resolved by semver range (`version: ">= 2.0"`). This created two related problems:
  1. **Double source of truth.** The `version:` field in the file is a second record of the version alongside the git tag. Authors forget to keep the two in sync when cutting a new tag, or set a value that does not match the tag. Any mechanism that checks this invariant (a CI guard, a hook) is one more moving part.
  2. **An unnecessary version resolver in Keeper.** A semver range on dependencies requires Keeper to read the list of tags from git, parse semver, and pick the best match. This is code on the critical path of artifact loading — and its own source of bugs around pre-releases, gate tags, and formats.
- **Decision.**
  - **An artifact's version = the git ref** under which it is committed. The `version:` field in `service.yml`, `destiny.yml`, and a module manifest **does not exist**.
  - **Dependencies are declared strictly via `ref:`** — either a git tag (`ref: v2.0.0`) or a branch (`ref: main`). No semver ranges, no ranges of any kind.
  - In Keeper's registry of services and plugins (`keeper.yml → services:`, `keeper.yml → plugins.*:`) — the same `ref:` field. Keeper does a plain git checkout at this ref, with no version-selection logic.
  - Exceptions (these are **not** "artifact versions"):
    - **`state_schema_version`** (an integer) in `service.yml` — this is the **version of the `incarnation.state` data structure** in the DB, needed for migrations (see the ["Versioning and state_schema migrations"](../architecture.md#versioning-and-state_schema-migrations) section). This is a different entity; confusing it with the "service version" is a mistake.
    - **`protocol_version`** in a plugin manifest — this is the **version of the plugin-API contract** (shared by `SoulModule`, `CloudDriver`, `SshProvider`; see [ADR-020](0020-plugin-infrastructure.md#adr-020-plugin-infrastructure-manifest-format-handshake-lifecycle)), needed by the `soul`/`keeper` host to check compatibility of the plugin protocol with the host itself. A runtime compatibility flag, not a module version.
    - **`service_version`** in the `incarnation` table — this is the pin of the git tag under which a given incarnation runs (see the incarnation fields table). Already ref-based; nothing changes here.
    - **Go modules inside the main repo** (`sdk/`, `shared/`, `proto/`, `proto/plugin/`, `keeper/`, `soul/`, `soul-lint/`) are versioned by a **shared semver tag of the main repo** (`v0.1.0`, `v0.2.0`, ...). This exception is dictated by the requirements of the Go toolchain (`require` in `go.mod` needs a semver tag) and applies only to **Go-library dependencies**, not to Service / Destiny / Module / Plugin as artifacts — for those the "version = git ref" rule stands unchanged. See [ADR-011](0011-go-layout.md#adr-011-go-code-layout-gowork-with-per-side-modules).
- **Rationale.** One source of truth for an artifact's version — the git tag. It is immutable, checked by git, visible via `git tag -l`, and shows up in `git log` and CI without extra glue. It is impossible to "forget to bump the version in the file" — because the file doesn't have one. The version resolver in Keeper reduces to `git checkout <ref>`. Branch-based dependencies (`ref: main`) work trivially for dev environments and feature branches; tag-based (`ref: v2.0.0`) for production. The same mechanism covers both scenarios.
- **Trade-off.** We lose the ability to write "dependency >= 2.0, < 3.0" and automatically pull in compatible patch versions. The operator must explicitly pin a tag in `ref:`. We accept this as a **deliberate trade for predictability**: bumping a version is an explicit action in `service.yml` / `keeper.yml`, visible in the git diff, going through review. If a range resolver is needed in the future (large installations with dozens of services and frequent updates), it can be added as a separate extension through the same CloudDriver-plugin mechanism, without touching the base model.
