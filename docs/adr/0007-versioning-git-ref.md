# ADR-007. Версионирование артефактов — через git ref, а не через поле в манифесте

- **Контекст.** Исходный код артефактов Soul Stack (**Service**, **Destiny**, **Module**, плагины — см. таблицу в разделе [«Артефакты Soul Stack: что в git, что в БД»](../architecture.md#артефакты-soul-stack-что-в-git-что-в-бд)) живёт в git-репозиториях. Изначально каждый артефакт декларировал собственную версию полем `version:` в манифесте (`service.yml`, `destiny.yml`, `manifest.yaml`), а зависимости разрешались по semver-range (`version: ">= 2.0"`). Это создавало две связанные проблемы:
  1. **Двойная истина.** Поле `version:` в файле — это вторая запись версии рядом с git-tag-ом. Авторы забывают синхронизировать одно с другим при выпуске нового тега, или ставят значение, не совпадающее с тегом. Любой механизм проверки этого инварианта (CI-гард, hook) — это ещё один движущийся элемент.
  2. **Лишний резолвер версий в Keeper-е.** Semver-range на зависимости требует, чтобы Keeper читал список тегов из git, парсил semver, выбирал лучшее совпадение. Это код в критическом пути загрузки артефактов — и собственный источник багов вокруг pre-release, gate-tag-ов и форматов.
- **Решение.**
  - **Версия артефакта = git ref**, под которым он закоммичен. Поле `version:` в `service.yml`, `destiny.yml`, manifest-е модуля **отсутствует**.
  - **Зависимости декларируются строго через `ref:`** — либо git-tag (`ref: v2.0.0`), либо ветка (`ref: main`). Никаких semver-range, никаких диапазонов.
  - В реестре сервисов и плагинов keeper-а (`keeper.yml → services:`, `keeper.yml → plugins.*:`) — то же поле `ref:`. Keeper делает простой git checkout по этому ref-у, никакой логики выбора версии.
  - Исключения (это **не** «версии артефакта»):
    - **`state_schema_version`** (целое) в `service.yml` — это **версия структуры данных** `incarnation.state` в БД, нужна для миграций (см. раздел [«Versioning и миграции state_schema»](../architecture.md#versioning-и-миграции-state_schema)). Это другая сущность; путать с «версией сервиса» — ошибка.
    - **`protocol_version`** в манифесте плагина — это **версия plugin-API-контракта** (общая для `SoulModule`, `CloudDriver`, `SshProvider`; см. [ADR-020](0020-plugin-infrastructure.md#adr-020-plugin-инфраструктура-формат-manifest-handshake-lifecycle)), нужна `soul`/`keeper`-хосту для проверки совместимости плагин-протокола с самим хостом. Runtime-флаг совместимости, не версия модуля.
    - **`service_version`** в `incarnation`-таблице — это пин git-tag-а, под которым работает конкретный incarnation (см. таблицу полей incarnation). Уже ref-based; ничего не меняется.
    - **Go-модули внутри основного репо** (`sdk/`, `shared/`, `proto/`, `proto/plugin/`, `keeper/`, `soul/`, `soul-lint/`) версионируются **совместным semver-тегом основного репо** (`v0.1.0`, `v0.2.0`, …). Это исключение обусловлено требованиями Go-toolchain (`require` в `go.mod` требует semver-тег) и распространяется только на **Go-library-зависимости**, не на Service / Destiny / Module / Plugin как артефакты — для них правило «версия = git ref» сохраняется без изменений. См. [ADR-011](0011-go-layout.md#adr-011-раскладка-go-кода-gowork-с-модулями-по-сторонам).
- **Обоснование.** Один source of truth для версии артефакта — git tag. Он immutable, проверяется git-ом, виден через `git tag -l`, попадает в `git log` и CI без отдельной обвязки. Невозможно «забыть обновить версию в файле» — потому что в файле её нет. Резолвер версий в Keeper-е сводится к `git checkout <ref>`. Branch-based зависимости (`ref: main`) тривиально работают для dev-окружений и feature-веток; tag-based (`ref: v2.0.0`) — для прод. Один и тот же механизм покрывает оба сценария.
- **Trade-off.** Теряется возможность писать «зависимость >= 2.0, < 3.0» и автоматически подтягивать совместимые патч-версии. Оператор должен явно зафиксировать tag в `ref:`. Принимаем как **сознательную плату за предсказуемость**: подъём версии — это явное действие в `service.yml` / `keeper.yml`, видимое в git diff, проходящее ревью. Если в будущем понадобится range-резолвер (большие инсталляции с десятками сервисов и частыми обновлениями), его можно добавить отдельным расширением через тот же механизм CloudDriver-плагинов, не трогая базовую модель.
