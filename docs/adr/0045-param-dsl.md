# ADR-045. Module Param-DSL — typed input fields for the Run Command UI form

> **Status: in implementation (slice map at the end of the ADR; the architect slices it in parallel).** The user's decisions were taken on 2026-05-31, fixed in the canon BEFORE the code (the rule "an architectural decision in the same move"). Absorbs the remainder of [ADR-044](0044-choir.md#adr-044-choir--named-host-topology-within-an-incarnation) S-T1 in the part of `format: sid` / `source:` for module input (the key names are already fixed by ADR-044 S-T1, see below).

**Context.** By 2026-05-31 two parallel input DSLs live in the system, and the Run Command UI cannot build a form for a module:
- **scenario/destiny `input:`** ([docs/input.md](../input.md), `config.InputSchema`) — rich: `type` / `required` / `secret` / `pattern` / `enum` / `format: sid` / `source: {incarnation_hosts, choir}` / `min`/`max` / `min_items`/`max_items` (the last three blocks — [ADR-044](0044-choir.md#adr-044-choir--named-host-topology-within-an-incarnation) S-T1);
- **module input schema** (`shared/plugin.InputParamDef`, the `states.<state>.input` section of a module's manifest per [ADR-020](0020-plugin-infrastructure.md#adr-020-plugin-инфраструктура-формат-manifest-handshake-lifecycle)) — **reduced**: `Type` / `Required` / `Secret` / `Pattern` / `Description` / `Default`, and nothing more.

Because of this the Run Command UI (`kind=command` Voyage, [ADR-043](0043-voyage.md#adr-043-voyage--унифицированный-батчевый-прогон); single-SID `/v1/souls/{sid}/exec`) **cannot render a form** for the selected module: there is no `enum` (a dropdown of allowed values), no `format: sid` + `source:` (a cluster-aware host-selection field with autocomplete), and although `pattern` exists in `InputParamDef`, it is **not passed through** to the DTO of the `/v1/modules` endpoint. The operator is forced to enter params blindly. This directly contradicts [ADR-042](0042-backend-driven-ui.md#adr-042-backend-driven-dynamic-data-в-ui--ui-не-хардкодит-динамические-каталоги) (the UI does not hardcode dynamics, it fetches the catalog from the backend).

**Decision (fixed by the user on 2026-05-31).** Bring `plugin.InputParamDef` closer to `config.InputSchema`, **reusing** the already-existing `input:` model — NOT introducing a third DSL and NOT copying its mechanisms. A unified field format for **core** (`shared/coremanifest`) AND **plugin** (`manifest.yaml`).

1. **Augment the module DSL with fields from `config.InputSchema` (additive, only-add).** To `plugin.InputParamDef` are added:
   - **`enum`** — the list of allowed values (the UI draws a dropdown; `soul-lint` verifies the literals; the backend validates membership). The semantics and constraints by `type` are the same as in [docs/input.md](../input.md) (enum is supported for scalar types; the error codes `input_enum_*` already exist, see [naming-rules.md](../naming-rules.md)).
   - **`format: sid`** + **`source: {incarnation_hosts, choir}`** — a cluster-aware SID field, **reusing the [ADR-044](0044-choir.md#adr-044-choir--named-host-topology-within-an-incarnation) S-T1 model** (the `format`/`source` keys are already fixed there for scenario/destiny `input:`). No new `type` is introduced — see rejected (a).
   - **`pattern`** — already exists in `InputParamDef`, but is now **passed through** to the DTO of the `/v1/modules` endpoint (previously lost). Purely a delivery fix, not a DSL extension.
   - **`items` under `type: map`** (amend S7) — **reuses** the already-introduced `items *InputParamDef` to declare the **value type** of a map: `map[string]<items>`. Flat string maps (env / headers / render-vars) are declared as `items: { type: string }` → the UI draws a KEY→VALUE editor; arbitrary maps (cloud `profile`) remain **without `items`** → the UI falls back to a JSON editor. No new key is introduced — this is the second use of `items` (value for a map, element for a list). `validateInputParam` accepts `items` under a map the same way as under a list (value type ∈ known-types); `enum` is not attached to a map.

2. **SID field — the exact form (parity with `config.InputSchema`).**
   - **Single SID:** `type: string` + `format: sid`.
   - **List of SIDs:** `type: array`, `items: { format: sid }`.
   - **Map with a scalar value:** `type: map`, `items: { type: <scalar> }` — describes the value type (`map[string]string`, etc.).
   - **Cluster-aware autocomplete** — the UI pulls live SID candidates from a **scoped form-prep endpoint** (live SIDs by the run's target, prefix-search), as already works for scenario input per [ADR-044](0044-choir.md#adr-044-choir--named-host-topology-within-an-incarnation) S-T1. `source:` indicates the catalog source (`incarnation_hosts` — the incarnation's hosts; `choir` — the voices of the selected Choir).
   - **Backend validation — FQDN structure, WITHOUT a hard-reject of membership (deliberately).** The backend checks that the value looks like a valid SID (FQDN structure, the same `sid_invalid_format` as everywhere), but does **NOT** reject a SID because the host is not yet in the registry/incarnation: the operator legally enters a host that is "about to connect". Hard membership would break this scenario. This is symmetric to the tolerant validation of scenario-input SID fields.

2-bis. **Map field — the UI widget selection rule (amend S7).** The decision is symmetric to a list field (a typed list vs a free list of strings):
   - **`type: map` + scalar `items.type`** (`string`/`int`/`bool`/...) → the UI draws a **KEY→VALUE editor**, the value widget is chosen by `items.type`. Flat string maps are declared this way: `core.cmd`/`core.exec` `env`, `core.http`/`core.url` `headers`, `core.file.rendered` `vars` — all `items: { type: string }`.
   - **`type: map` without `items`** (or `items.type ∈ {map, object}`, a nested collection) → the UI falls back to a **JSON editor** of arbitrary structure. This is how `core.cloud.provisioned` `profile` (a backend-specific object) is declared. The UI determines the widget **generically by the schema**, without hardcoding field names — this is the invariant of [ADR-042](0042-backend-driven-ui.md#adr-042-backend-driven-dynamic-data-в-ui--ui-не-хардкодит-динамические-каталоги).

3. **Delivery of core manifests to Keeper under the [ADR-011](0011-go-layout.md#adr-011-раскладка-go-кода-gowork-с-модулями-по-сторонам) isolation.** The Keeper binary does **NOT** import `soul/internal/*` (Soul isolation is guaranteed by the compiler). The source of truth for **core** modules' input schemas for the Keeper side (Validate + lint + form building) is the package **`shared/coremanifest`** (lives in `shared/`, available to both binaries, bypasses the isolation). The `/v1/modules` handler reads `coremanifest` for core params; plugin params are taken from the plugin's loaded `manifest.yaml` (as now). `coremanifest` is a single source for three consumers: runtime Validate, `soul-lint`, and the UI form.

4. **Coverage — core + plugin, prod variant.** Not only Soul-side core modules ([ADR-015](0015-core-modules-mvp.md#adr-015-core-модули-mvp-точный-список)): keeper-side core (`core.cloud.provisioned` / `core.vault.kv-read` / `core.choir` — [ADR-017](0017-keeper-side-core.md#adr-017-keeper-side-core-модули-расширены-corecloudprovisioned-corevaultkv-read), [ADR-044](0044-choir.md#adr-044-choir--named-host-topology-within-an-incarnation) S-T5) also get `coremanifest` records with typed input fields. Plugin modules — via the augmented `manifest.yaml`.

5. **Impact on the plugin SDK.** External module authors will be able to declare `enum` / `format: sid` / `source:` fields in `manifest.yaml`. **Forward-compat (only-add, [ADR-012](0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add) in spirit, [ADR-020(c)](0020-plugin-infrastructure.md#adr-020-plugin-инфраструктура-формат-manifest-handshake-lifecycle)):** via `protocol_version` (compat flag) + **lenient acceptance of new optional keys** — an old `soul-lint` / old host does NOT break upon encountering an unknown optional key of an input field (it ignores it, does not reject). This preserves compatibility with old plugins and old linters.

6. **Validation = an ADDITION, not a replacement of the imperative one.** The declarative schema covers the form / `soul-lint` / `required` / `enum` membership / `format` structure. **The imperative Apply validation of the module is preserved on top** — for cross-field invariants and type checks that the declarative schema cannot express. A consistency test ([ADR-044](0044-choir.md#adr-044-choir--named-host-topology-within-an-incarnation) S-T1 style) guards that the declared schema does not diverge from what Apply actually accepts. The declarative layer does not cancel the imperative one — it cuts off invalid input earlier (fail-fast at the form/lint).

**Rejected alternatives.**
- **(a) A new `type: sid`** (instead of `format: sid`). Rejected: it breeds a **second model** of the SID field, diverging from `config.InputSchema` (where SID is `type: string` + `format: sid`). The goal of the ADR is to BRING the two DSLs closer, not to separate them even more. `format` on top of the existing `type` is additive and compatible.
- **(b) A Go method `ParamsSchema()` on the module interface** (schema as code, not as a manifest). Rejected: `manifest.yaml` is already the **source of truth** of the input schema ([ADR-020(a)](0020-plugin-infrastructure.md#adr-020-plugin-инфраструктура-формат-manifest-handshake-lifecycle), `soul-lint` reads exactly the manifest without running the module). Duplicating the schema in a Go method = two sources, guaranteed drift; for core — `coremanifest` plays the same role as the manifest.

**Cross-ref.** [ADR-044](0044-choir.md#adr-044-choir--named-host-topology-within-an-incarnation) S-T1 (`format: sid` / `source:` are fixed there — this ADR extends their application from scenario/destiny input to module input); [ADR-015](0015-core-modules-mvp.md#adr-015-core-модули-mvp-точный-список) (core MVP — carriers of the new input fields); [ADR-011](0011-go-layout.md#adr-011-раскладка-go-кода-gowork-с-модулями-по-сторонам) (Soul isolation — `coremanifest` in `shared/` as a bridge); [ADR-042](0042-backend-driven-ui.md#adr-042-backend-driven-dynamic-data-в-ui--ui-не-хардкодит-динамические-каталоги) (backend-driven form — the UI fetches the schema, does not hardcode it).
