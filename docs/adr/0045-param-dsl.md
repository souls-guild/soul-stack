# ADR-045. Param-DSL модулей — типизированные input-поля для UI-формы Run Command

> **Статус: в реализации (slice-карта в конце ADR; architect нарежет слайсы параллельно).** Решения пользователя приняты 2026-05-31, фиксируются в каноне ДО кода (правило «архитектурное решение в том же ходе»). Поглощает остаток [ADR-044](0044-choir.md#adr-044-choir--именованная-топология-хостов-внутри-инкарнации) S-T1 в части `format: sid` / `source:` для модульного input (имена ключей уже закреплены ADR-044 S-T1, см. ниже).

**Контекст.** К 2026-05-31 в системе живут **две параллельные input-DSL**, и UI Run Command не может построить форму под модуль:
- **scenario/destiny `input:`** ([docs/input.md](../input.md), `config.InputSchema`) — богатая: `type` / `required` / `secret` / `pattern` / `enum` / `format: sid` / `source: {incarnation_hosts, choir}` / `min`/`max` / `min_items`/`max_items` (последние три блока — [ADR-044](0044-choir.md#adr-044-choir--именованная-топология-хостов-внутри-инкарнации) S-T1);
- **модульная input-схема** (`shared/plugin.InputParamDef`, секция `states.<state>.input` манифеста модуля по [ADR-020](0020-plugin-infrastructure.md#adr-020-plugin-инфраструктура-формат-manifest-handshake-lifecycle)) — **урезанная**: `Type` / `Required` / `Secret` / `Pattern` / `Description` / `Default`, и больше ничего.

Из-за этого UI Run Command (`kind=command` Voyage, [ADR-043](0043-voyage.md#adr-043-voyage--унифицированный-батчевый-прогон); single-SID `/v1/souls/{sid}/exec`) **не может отрисовать форму** под выбранный модуль: нет `enum` (выпадающий список допустимых значений), нет `format: sid` + `source:` (cluster-aware поле выбора хоста с автокомплитом), а `pattern` хоть и есть в `InputParamDef`, но **не прокидывается** в DTO эндпоинта `/v1/modules`. Оператор вынужден вбивать params вслепую. Это прямо противоречит [ADR-042](0042-backend-driven-ui.md#adr-042-backend-driven-dynamic-data-в-ui--ui-не-хардкодит-динамические-каталоги) (UI не хардкодит динамику, фетчит каталог с backend-а).

**Решение (зафиксировано пользователем 2026-05-31).** Сблизить `plugin.InputParamDef` с `config.InputSchema`, **переиспользуя** уже существующую модель `input:` — НЕ вводя третью DSL и НЕ копируя её механизмы. Единый формат поля для **core** (`shared/coremanifest`) И **plugin** (`manifest.yaml`).

1. **Дополнить модульную DSL полями из `config.InputSchema` (additive, only-add).** В `plugin.InputParamDef` добавляются:
   - **`enum`** — список допустимых значений (UI рисует выпадающий список; `soul-lint` сверяет литералы; backend валидирует членство). Семантика и ограничения по `type` — те же, что в [docs/input.md](../input.md) (enum поддержан для скалярных типов; код ошибок `input_enum_*` уже существует, см. [naming-rules.md](../naming-rules.md)).
   - **`format: sid`** + **`source: {incarnation_hosts, choir}`** — cluster-aware SID-поле, **переиспользуя модель [ADR-044](0044-choir.md#adr-044-choir--именованная-топология-хостов-внутри-инкарнации) S-T1** (`format`/`source`-ключи там уже закреплены для scenario/destiny `input:`). НЕ вводится новый `type` — см. отвергнутое (а).
   - **`pattern`** — уже есть в `InputParamDef`, но **прокидывается** в DTO эндпоинта `/v1/modules` (раньше терялся). Чисто delivery-фикс, не расширение DSL.
   - **`items` под `type: map`** (amend S7) — **переиспользует** уже введённый `items *InputParamDef` для декларации **типа значения** map: `map[string]<items>`. Плоские строковые map (env / headers / render-vars) объявляются `items: { type: string }` → UI рисует KEY→VALUE-редактор; произвольные map (cloud `profile`) остаются **без `items`** → UI откатывается на JSON-редактор. НЕ вводится новый ключ — это второе применение `items` (значение для map, элемент для list). `validateInputParam` принимает `items` под map так же, как под list (тип значения ∈ known-types); `enum` на map не вешается.

2. **SID-поле — точная форма (parity `config.InputSchema`).**
   - **Одиночный SID:** `type: string` + `format: sid`.
   - **Список SID:** `type: array`, `items: { format: sid }`.
   - **Map со скалярным значением:** `type: map`, `items: { type: <скаляр> }` — описывает тип значения (`map[string]string` и т.п.).
   - **Cluster-aware автокомплит** — UI тянет live-SID-кандидаты из **scoped form-prep эндпоинта** (live-SID по target прогона, prefix-search), как это уже работает для scenario-input по [ADR-044](0044-choir.md#adr-044-choir--именованная-топология-хостов-внутри-инкарнации) S-T1. `source:` указывает источник каталога (`incarnation_hosts` — хосты инкарнации; `choir` — голоса выбранного Choir).
   - **Backend-валидация — FQDN-структура, БЕЗ hard-reject членства (сознательно).** Backend проверяет, что значение похоже на валидный SID (FQDN-структура, тот же `sid_invalid_format`, что и везде), но **НЕ** отвергает SID за то, что хост ещё не в реестре/инкарнации: оператор легально вводит хост, который «вот-вот подключится». Жёсткое членство ломало бы этот сценарий. Это симметрично tolerant-валидации SID-полей scenario-input.

2-bis. **Map-поле — правило выбора виджета UI (amend S7).** Решение симметрично list-полю (типизированный список vs свободный список строк):
   - **`type: map` + `items.type` скалярный** (`string`/`int`/`bool`/...) → UI рисует **KEY→VALUE-редактор**, виджет значения подбирается по `items.type`. Так объявлены плоские строковые map: `core.cmd`/`core.exec` `env`, `core.http`/`core.url` `headers`, `core.file.rendered` `vars` — все `items: { type: string }`.
   - **`type: map` без `items`** (или `items.type ∈ {map, object}`, вложенная коллекция) → UI откатывается на **JSON-редактор** произвольной структуры. Так объявлен `core.cloud.provisioned` `profile` (backend-специфичный объект). UI определяет виджет **генерически по схеме**, без хардкода имён полей — это инвариант [ADR-042](0042-backend-driven-ui.md#adr-042-backend-driven-dynamic-data-в-ui--ui-не-хардкодит-динамические-каталоги).

3. **Доставка core-манифестов в Keeper при изоляции [ADR-011](0011-go-layout.md#adr-011-раскладка-go-кода-gowork-с-модулями-по-сторонам).** Keeper-бинарь **НЕ** импортирует `soul/internal/*` (изоляция Soul гарантируется компилятором). Источник истины по input-схемам **core**-модулей для Keeper-side (Validate + lint + построение формы) — пакет **`shared/coremanifest`** (живёт в `shared/`, доступен обоим бинарям, обходит изоляцию). Handler `/v1/modules` читает `coremanifest` для core-params; plugin-params берутся из загруженного `manifest.yaml` плагина (как сейчас). `coremanifest` — единый источник для трёх потребителей: рантайм-Validate, `soul-lint`, и форма UI.

4. **Охват — core + plugin, prod-вариант.** Не только Soul-side core-модули ([ADR-015](0015-core-modules-mvp.md#adr-015-core-модули-mvp-точный-список)): keeper-side core (`core.cloud.provisioned` / `core.vault.kv-read` / `core.choir` — [ADR-017](0017-keeper-side-core.md#adr-017-keeper-side-core-модули-расширены-corecloudprovisioned-corevaultkv-read), [ADR-044](0044-choir.md#adr-044-choir--именованная-топология-хостов-внутри-инкарнации) S-T5) тоже получают `coremanifest`-записи с типизированными input-полями. Plugin-модули — через дополненный `manifest.yaml`.

5. **Влияние на plugin-SDK.** Внешние авторы модулей смогут объявить `enum` / `format: sid` / `source:` поля в `manifest.yaml`. **Forward-compat (only-add, [ADR-012](0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add) по духу, [ADR-020(c)](0020-plugin-infrastructure.md#adr-020-plugin-инфраструктура-формат-manifest-handshake-lifecycle)):** через `protocol_version` (compat-флаг) + **нестрогий приём новых опциональных ключей** — старый `soul-lint` / старый host НЕ падает, встретив незнакомый опциональный ключ input-поля (игнорирует его, не reject). Это сохраняет совместимость со старыми плагинами и старыми линтерами.

6. **Валидация = ДОПОЛНЕНИЕ, не замена императивной.** Декларативная схема покрывает форму / `soul-lint` / `required` / `enum`-членство / `format`-структуру. **Императивная Apply-валидация модуля сохраняется поверх** — для cross-field-инвариантов и типовых проверок, которые декларативной схемой не выразить. Consistency-test ([ADR-044](0044-choir.md#adr-044-choir--именованная-топология-хостов-внутри-инкарнации) S-T1-стиль) стережёт, что declared-схема не расходится с тем, что Apply реально принимает. Декларативный слой не отменяет императивный — он отрезает невалидный вход раньше (fail-fast на форме/lint).

**Отвергнутые альтернативы.**
- **(а) Новый `type: sid`** (вместо `format: sid`). Отвергнуто: плодит **вторую модель** SID-поля, расходящуюся с `config.InputSchema` (там SID — это `type: string` + `format: sid`). Цель ADR — СБЛИЗИТЬ две DSL, а не развести их ещё сильнее. `format` поверх существующего `type` — additive и совместимо.
- **(б) Go-метод `ParamsSchema()` на интерфейсе модуля** (схема как код, не как манифест). Отвергнуто: `manifest.yaml` уже **источник истины** input-схемы ([ADR-020(a)](0020-plugin-infrastructure.md#adr-020-plugin-инфраструктура-формат-manifest-handshake-lifecycle), `soul-lint` читает именно манифест без запуска модуля). Дублировать схему в Go-методе = два источника, гарантированный drift; для core — `coremanifest` играет ту же роль манифеста.
