# ADR-012. Keeper↔Soul gRPC contract: one EventStream with oneof, Keeper-side render, forward-compat only-add

- **Context.** [ADR-002](0002-transport-grpc-ha.md#adr-002-transport-keeper--souls--grpc-bidirectional-stream-over-mtls-ha-keeper-cluster) fixed the transport (gRPC bidi over mTLS, initiated by Soul) but not the message contract. [Pilot scaffolding ADR-011](0011-go-layout.md#adr-011-go-code-layout-gowork-with-per-side-modules) placed a placeholder `proto/keeper/v1/keeper.proto` with the RPC `Ping`. A single consolidated decision is needed on the service structure, the layout of `.proto` files, the versioning policy, the location of Destiny rendering, and the relationship of `TaskEvent` to the `SoulModule`'s `ApplyEvent`. Some items overlap with open Q #6 (the Soulprint schema), #7 (protobuf compat) and #12 (Heartbeat).
- **Decision.**

  **(a) One `service Keeper` + one long-lived bidi stream with a `oneof` payload.** The service consists of two RPC methods:

  ```
  service Keeper {
    rpc Bootstrap(BootstrapRequest) returns (BootstrapReply);     // unary, server-only TLS
    rpc EventStream(stream FromSoul) returns (stream FromKeeper); // bidi, mTLS
  }
  ```

  `FromSoul` and `FromKeeper` are wrapper messages with a `oneof payload` enumerating all types per direction. This matches ADR-002's phrasing "one long-lived bidirectional gRPC stream" and allows adding new message types without a breaking change.

  **(b) Thematic layout of the `.proto` files inside `proto/keeper/v1/`:**

  ```
  proto/keeper/v1/
  ├── keeper.proto        # service Keeper; FromSoul / FromKeeper oneof
  ├── onboarding.proto    # BootstrapRequest / BootstrapReply
  ├── lifecycle.proto     # Hello / HelloReply, SeedRotationRequest / SeedRotationReply
  ├── apply.proto         # ApplyRequest, TaskEvent, RunResult, CancelApply
  ├── soulprint.proto     # SoulprintReport
  └── common.proto        # types reused across files (RenderedTask, TaskError, ...)
  ```

  One file = one semantic axis. All are imported by the top-level `keeper.proto`. Generated Go code is built by `make gen` into `proto/gen/go/keeper/v1/` (see [ADR-011](0011-go-layout.md#adr-011-go-code-layout-gowork-with-per-side-modules)).

  **(c) Versioning — forward-compat, only-add within `proto/keeper/v1/`. Closes open Q #7.**
  - Never delete fields and never reuse field numbers.
  - All new fields are added as optional (proto3 default behavior).
  - Breaking changes — only via a new package `proto/keeper/v2/`, not by editing `v1/`.
  - This allows upgrading Keeper separately from Souls: the typical operational scheme "Keeper first, then Souls in waves" becomes something the contract grammar supports, not a mere agreement.

  **(d) The render boundary — by external access, not by location.** The security invariant ADR-012(d) is **re-anchored** (reason: the prod blocker of the golden-path Real-Linux E2E `core.file.rendered: template_content missing` + a user+architect decision). The earlier phrasing was "all rendering Keeper-side"; now the boundary runs along **who reaches external systems**, and NOT along the physical location of rendering:

  - **The CEL phase (rendering params, `${ … }` interpolation, vault) — Keeper-side.** `vault(...)` / `soulprint.*` / `register.*` / `input.*` / `essence.*` in `params:`/`vars:`/`on:` are resolved on Keeper. Vault tokens do not reach the host — external access stays with Keeper.
  - **text/template COMPUTE — Soul-side.** The final pass over `templates/<path>.tmpl` (Go text/template + sprig-allowlist) is executed on Soul by the `core.file.rendered` module: this is **local compute without I/O** (no files/network/Vault/environment — the three sandbox barriers of ADR-010 are preserved on Soul, the sprig-allowlist excludes `env`/`exec`/`getHostByName` and everything that reads the FS/network). Soul pulls only `shared/tmpl` (text/template + sprig-allowlist).
  - **Flow-control CEL — Soul-side, sandboxed, without external access.** The predicates `when:`/`changed_when:`/`failed_when:` depend on `register.*` (the results of previous tasks), known only on Soul during the run, not on Keeper during rendering. Therefore they are evaluated Soul-side by a trimmed cel-go sandbox (`shared/cel.NewFlowControl`). This is consistent with the "by external access" boundary: flow-control CEL is local compute without I/O (like text/template COMPUTE). The sandbox registers only functions without external access (`size`/`contains`/`has`/comprehensions/conversions/operators/`duration`) and the variables `register.*` (Soul gathers them from task results by register name) + `flow_context` (delivered by Keeper). Forbidden by construction (the symbol is not registered → compile error): `vault(...)` and `now()` (external access/non-determinism — symmetric with the migration-CEL sandbox [ADR-019](0019-state-migration-dsl.md#adr-019-state_schema-migration-dsl)), `soulprint.hosts`/`soulprint.where` (cross-host, scenario-only), any `__` identifier. Soul pulls `cel-go` but NOT the `vault` client: there are still no Vault tokens on the host, external access is keeper-only. **Implemented Soul-side:** `when:` (gating BEFORE Apply) + `changed_when:`/`failed_when:` (overriding `changed`/`failed` AFTER Apply; the order is `changed_when` then `failed_when`, `failed` takes precedence). `failed_when: false` on a failed module = ignore_errors (the status is NOT FAILED, the run does not break, the original error is preserved in `register.<name>.ignored_error` + `TaskEvent.error`); it does NOT apply to `TIMED_OUT` (a timeout is an infrastructural fail-stop). The activation of `changed_when:`/`failed_when:` additionally includes `register.self.*` (the fresh result of the current task). `until:` (the retry loop) is deferred.
  - **`flow_context` — a literal per-host snapshot of the non-register part of the CEL context** `{ input, vars, essence, incarnation, self }` (the same as for rendering params, minus `soulprint.hosts` and loop), delivered in `RenderedTask.flow_context`. Soul reads it as data (binds `soulprint.self` ← `flow_context.self`, the rest top-level), and makes no external access. It is host-variant (`self` per-host), excluded from the per-host check of params host-invariance. `register.*` is NOT put into `flow_context` — Soul builds it itself from accumulated task results (by the `RenderedTask.register` name). New `RenderedTask` fields (only-add, [ADR-012(c)](#adr-012-keepersoul-grpc-contract-one-eventstream-with-oneof-keeper-side-render-forward-compat-only-add)): `when`/`changed_when`/`failed_when` (CEL strings), `flow_context`, `register` (the register name of the result). **Static-when placeholder-skip (Variant b).** If `when:` is static (register-/soulprint-independent) and evaluates Keeper-side to `false`, params are NOT rendered — the task becomes a skip placeholder (`flow_context` is assembled, `params` are empty; Soul will evaluate the same `when:false` → SKIPPED). This fixes multi-action destiny: tasks of the inactive branch read an optional input that is absent → without skip they would fail with `no-such-key`. **Static-when precedes loop fan-out:** for a loop task the static-when decision is made BEFORE `resolveLoopItems`; an unresolvable `items:` in a skipped task yields one placeholder, a resolvable one yields N (parity with the active branch). **Static-when precedes the DSL guard (`guardPilotDSL`/`guardDestinyTask`):** a statically-false gated-off task is skipped at the start of the task walk BEFORE any DSL validation, so unsupported DSL (`parallel:`/`block:`) in the INACTIVE branch does not block rendering of the active one — it is rejected by `ErrUnsupportedDSL` ONLY on activation of that branch (per-action validation; not masking — the task physically does not execute). This fixes multi-action destiny, where different `action`s carry a different set of supported/unsupported constructs (e.g. a `diagnose` branch with `parallel:` under `action: update_acls`). **Contract (intentional, not a bug):** a statically-false (`when:false`, register-/soulprint-independent) task is gated off **BEFORE** `guardPilotDSL`/`guardDestinyTask`; unsupported DSL in its **INACTIVE** branch is **NOT** rejected at parse time — per-action validation happens on activation (when `when:` becomes `true`). This is intentional: a gated-off task physically does not execute, so its DSL is never reached. The reverse guard `StaticWhenTrue_UnsupportedDSL_StillRejected` pins the opposite side of the invariant (statically-true unsupported DSL is still rejected). This is recorded so that the order "static-when BEFORE the DSL guard" is not later mistaken for a bug and "fixed" by eager validation of all branches — that would break multi-action destiny.
  - **Keeper delivers the literal template content + the assembled per-host context root.** The `ApplyRequest` for the `core.file.rendered` step carries the literal `template_content` (Keeper read the `.tmpl` from the service/destiny snapshot after the CEL phase, without executing text/template) + `render_context` — the **per-host** root of the text/template context assembled by Keeper per [templating.md §3.2](../templating.md): `{ vars, self, role, essence }` (CEL-rendered `params.vars` under the `vars` key; `self` — the same `soulprint.self` projection as in the CEL phase, the single source of truth of ADR-018; `role` — the host's declared role; `essence` — the effective layer). Soul passes `render_context` as the **root** into text/template, so the template sees `.vars.*`/`.self.*`/`.role`/`.essence.*`. **The A1 variant (MVP):** both `template_content` and `render_context` travel inside `RenderedTask.params` as ordinary keys — **without proto changes**; Keeper removes the flat `vars` key and the path key `template` from params (Soul reads the root only from `render_context`). **Host-invariance of the template context is NOT required:** `render_context` and `template_content` are excluded from the per-host check of params (`self`-dependent templates → each host has its OWN `render_context`). For the golden path (one host in the run) this is trivial; the full per-host dispatch of self-dependent templates (a separate `ApplyRequest` per host) is a deferred orchestrator layer. The root cause of the prod blocker BUG-A (Real-Linux E2E `map has no entry for key "self"`): Soul supplied a flat `vars` as the root, and the template `{{ .self.network.primary_ip }}` failed in strict-mode. The two-level resolution of `.tmpl` (scenario-local→service-level, [ADR-009](0009-scenario-dsl.md#adr-009-scenario--полная-dsl-задач-destiny-граница-с-destiny--рекомендация)) is done by Keeper on read.

  Pipeline phases: `vault-resolve(author refs in params) → input-resolve(merge → scoped operator-input-vault-resolve → value validation) → CEL-render → delivery of the literal .tmpl` (Keeper) → `text/template-render` (Soul, in `core.file.rendered`). The full phases — [templating.md §4](../templating.md).

  - **Scoped resolution of a `vault:`-ref in operator input — a separate restricted channel.** The value of a secret field with a declared `vault_scope` (the input contract, [docs/input.md → "vault_scope"](../input.md#vault_scope-scoped-резолв-vault-ref-в-operator-input)) may be a `vault:`-ref; Keeper resolves it keeper-side ONCE in the input phase with a check of the `vault_scope` prefix + a hard deny-list (`secret/keeper/*`/`secret/internal/*`, not disableable by config, extended by `keeper.yml → vault.input_deny_paths`). Default-deny: a `vault:` value in a field without `vault_scope` is an error. Each resolution (ok/denied) is audited (`input.vault_resolved`, the path is not a secret, the secret value is not logged); denied is a security signal. Value validation (`pattern`/`enum`) is on the ALREADY-resolved value. This does **not** touch the trusted author channel (`vault:`-ref in `params:` and `${ vault(...) }`) — it stays as-is.

  Advantages (preserved): no Vault tokens on Soul (security first); centralized audit of CEL and vault resolution on Keeper; no `cel-go`/`vault` client in `soul/go.mod`.

  Trade-off: probe steps in a scenario cause a Keeper↔Soul round-trip after each probe — Keeper holds the scenario state and sends the next step **after** receiving the per-host `register:` from Soul. This is done anyway: the scenario dispatcher lives on Keeper ([ADR-009](0009-scenario-dsl.md#adr-009-scenario--полная-dsl-задач-destiny-граница-с-destiny--рекомендация)).

  **(e) `TaskEvent` ↔ SoulModule `ApplyEvent` — aggregation on Soul (for the MVP).** Soul waits for the SoulModule sub-process to finish, forms the final `TaskEvent` (with `register.<name>.*` fields: `.changed`, `.failed`, `.timed_out`, plus the fields declared in the destiny `output:`) and sends it to Keeper. **The progress of long-running steps** does not go into the Keeper↔Soul stream in the MVP. Post-MVP — it will be added by duplicating the `ApplyEvent` structure in `apply.proto`, **without** a cross-import between `proto/keeper/v1/` and `proto/plugin/v1/` (forbidden by [ADR-011](0011-go-layout.md#adr-011-go-code-layout-gowork-with-per-side-modules)).

  **(f) The Bootstrap RPC — a separate listener, server-only TLS.** `rpc Bootstrap` listens on a separate port with **server TLS without a client cert** (Soul does not yet have a SoulSeed, the identity claim is provided by `bootstrap_token`). `rpc EventStream` — the main port with mTLS. The explicit split "no identity" / "has identity" simplifies RBAC, audit and network segmentation.

  **(g) The Soulprint payload — `google.protobuf.Struct` until open Q #6 is closed.** `SoulprintReport.facts` has the type `google.protobuf.Struct`; the exact typed schema will be added after #6 is settled as a new field without a breaking change (deprecating the old one, only-add).

  **(h) Heartbeat — not introduced.** `last_seen_at` is updated on any app message in the EventStream, plus gRPC keepalive on the transport. Open Q #12 remains open, but the current contract does not depend on it — a dedicated `Heartbeat` message will be added if operations show it is needed.

  **(i) SID in the payload — an echo, not an identity claim.** The authoritative source of the SID is the `Subject Alternative Name` of the mTLS session's peer cert. In the proto comments of the corresponding fields, explicitly: `// SID echoed for logging only; authoritative source is the mTLS peer certificate`.

  **(j) Message names** are fixed in [naming-rules.md → the section "Keeper↔Soul proto messages"](../naming-rules.md#сообщения-proto-keepersoul). `StateReport` from early discussions was rejected as a conflict with the reserved "State" (= `incarnation.state`); the final name of the final run report is **`RunResult`**.

  **(k) Amendment (2026-05-25, `WardRoster` — Soul reconcile).** A new `FromSoul` **only-add** message `WardRoster { repeated ActiveApply active }` (`ActiveApply { string apply_id; int32 attempt }`, field 3 reserved for `status`, never-reuse), `FromSoul.oneof` **field 8**. On (re)connect (right after Hello, ReplaceAll semantics) Soul declares the `apply_id`s it is actually driving — Keeper terminates orphaned `dispatched` ones ([ADR-027(g)](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim) IMPLEMENTED). Forward-compat: an old Soul does not send it → the Keeper sweep is a no-op (fail-safe). The name — in naming-rules.

- **Consequences.**
  - Six `.proto` files are created in `proto/keeper/v1/` (plus the existing `keeper.proto` is extended). `make gen` builds into `proto/gen/go/keeper/v1/`, the generated code is committed (ADR-011).
  - `cel-go` appears in `soul/go.mod` (sandboxed flow-control CEL — `when`/`changed_when`/`failed_when`, via `shared/cel.NewFlowControl`), but NOT the `vault` client (external access is keeper-only). This is a **deliberate re-anchoring of BUG-A** — symmetric with the one already done for text/template COMPUTE (the boundary is "by external access", not "where the compute is"), not a silent edit: flow-control predicates depend on `register.*`, which is not on Keeper during rendering. `sprig` (via `shared/tmpl`) is also **present** on Soul — the text/template COMPUTE of the `core.file.rendered` step. There is still no external access (Vault tokens / `vault` client / network to Vault) on Soul.
  - Keeper's configuration formalizes two sub-listeners under `listen.grpc.{bootstrap,event_stream}` ([config.md → listen](../keeper/config.md#listen)). The TLS parameters are independent (Bootstrap — server-only TLS without `ca`; EventStream — mTLS with `ca`); identical file paths for cert/key between the sub-listeners are allowed by the grammar. Schema validation adds the diag codes `grpc_bootstrap_listener_required` / `grpc_event_stream_listener_required` (both required by ADR-002) and `bootstrap_eventstream_port_conflict` (the addresses must differ).
  - `proto/keeper/v1/` does not import `proto/plugin/v1/`; event structures are duplicated between them — a deliberate price for the isolation of the plugin ecosystem (ADR-011).
  - `soul-lint` does not strictly validate `SoulprintReport.facts` until open Q #6 is closed.
  - **Open Q #7 is closed** (the forward-compat only-add rule). Open Q #6 and #12 remain open but do not block the v1 contract.
  - `service Keeper` lives in a single service block (see (a)); push mode ([keeper.push](../architecture.md#push-режим-keeperpush)) uses a **different** channel (SSH, not gRPC) and is not present in `service Keeper`.
- **Trade-offs.**
  - One EventStream is a simple model but requires careful ordering in both directions (especially on cancel/abort in parallel with a long apply). gRPC's FIFO guarantees within a single direction cover this.
  - Keeper-side CEL + external access => probe steps make a Keeper↔Soul round-trip. For a typical scenario of 5–10 steps this is tens of ms RTT per step — acceptable. The "by external access" boundary (ADR-012(d)): CEL with external access (vault/render params) — Keeper, local compute without I/O — Soul (text/template COMPUTE + flow-control CEL `when:`/…); only compute is left on Soul, with no loss in attack surface (the ADR-010 sandbox barriers + the sandbox-by-undeclaration flow-control env are preserved on Soul).
  - Forward-compat only-add clutters `v1/` with deprecated fields over time. Better than breaking the wire format on upgrades.
  - Bootstrap on a separate port = +1 listener in Keeper's config. A negligible operational overhead, a gain in security.
  - Duplication of the `ApplyEvent` structure between `proto/keeper/v1/apply.proto` and `proto/plugin/v1/soulmodule.proto` (when needed) — a drift risk. Accepted: ADR-011 explicitly forbids cross-import.

**Amendment ([ADR-056](0056-staged-render-passage.md), staged render — the dispatch model "N ApplyRequests per host by Passage"). Status → amended.** The phrasing **(d)/(e)** "one `ApplyRequest` per host with all its tasks at once, aggregation of one `TaskEvent`/`RunResult` per host" is **refined**: a scenario run is executed as **N ordered Passages** ([Passage](../naming-rules.md#сущности-предметной-области) — the stage render→dispatch→barrier→gathering register), and a single host now receives **N `ApplyRequest`s** (one per Passage), **N `TaskEvent`/`RunResult`s** respectively. This is the implementation of the **deferred post-MVP per-task dispatch** ([orchestration.md §2.2.1/§8](../scenario/orchestration.md), open Q #24/#25 architecture.md), needed so that probe→where actually works (the render of the next Passage substitutes the `register` of the previous ones). The contract is **only-add** ([ADR-012(c)](#adr-012-keepersoul-grpc-contract-one-eventstream-with-oneof-keeper-side-render-forward-compat-only-add) forward-compat, never-reuse): a new field `int32 passage` in `ApplyRequest`/`TaskEvent`/`RunResult`. The render boundary **(d)** (Keeper-side CEL with external access / Soul-side compute) does not change. Forward-compat: an old Soul without the `passage` field under a staged scenario (N>1) → **explicit-reject** at render (`render_failed`), not a hang and not a silent single-pass execution. Decisions and the slice map — [ADR-056](0056-staged-render-passage.md).

**Amendment ([ADR-065](0065-core-module-installed.md), `FetchModule` — a third RPC in `service Keeper`).** The phrasing **(a)** "the service consists of two RPC methods" is extended with a third — the server-streaming `rpc FetchModule(PluginFetchRequest) returns (stream PluginChunk)`: delivery of the bytes of a SoulModule plugin binary to Soul on a `core.module.installed` request. Served by the **same mTLS listener as EventStream** (Bootstrap stays on its own server-only TLS listener, (f)); there is no new port. The artifact bytes travel over a **separate HTTP/2 stream**, NOT through EventStream — megabytes of a binary do not choke the control plane. **Content-addressed:** Keeper serves only bytes whose `sha256` is in the active Sigil allowance `kind: soul_module` ([ADR-026](0026-sigil.md), reusing `plugin_sigils`); authorization — the mTLS peer cert (the SID from the SAN, (i)). The contract is **only-add** per (c): a new method + new messages, existing messages untouched; an old Soul does not call the method; a new Soul against an old Keeper gets `Unimplemented` → the install step fails with `module_fetch_failed` (explicit-reject, not a hang). Guard-rails: `plugins.max_artifact_size_mb` + a rate limit on parallel fetches per-SID. The exact set of fields and placement in the thematic layout (b) — slice S1. Decisions and the slice map — [ADR-065](0065-core-module-installed.md).
