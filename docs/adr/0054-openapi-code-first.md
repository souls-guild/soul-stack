# ADR-054. Operator API — разворот на code-first (Go-типы → OpenAPI) через huma v2

> **Статус: active.** Заменяет ([supersede](#связь-с-adr)) [ADR-051](0051-operator-api-codegen.md) — посылка инвертирована: было **OpenAPI → Go-типы** (spec-first, oapi-codegen), стало **Go-типы → OpenAPI** (code-first, huma). Решение пользователя (2026-06-12). Эпик начат **PILOT-SPIKE** по правилу массовых операций: канон (этот ADR + аменд ADR-011) + ОДИН роут (`POST /v1/cadences`) через huma — доказательство интеграции ДО тиража. Снос oapi-каркаса и тираж ~20 доменов — ПОСЛЕ доказанного pilot.

> **Amendment 2026-06-13 (ТИРАЖ ЗАВЕРШЁН + СНОС oapi-каркаса — финальное состояние, HEAD `fde65bf`).** Разворот реализован полностью. **Все ~19 доменов Operator API — handler-native huma full-typed** (`keeper/internal/api/huma_*.go`), `oapi.X`-типов в коде НЕ осталось. **served-spec = runtime huma-дамп** из кода: `GET /openapi.yaml` отдаёт `servedOpenAPIHandler` → `HumaFullSpecYAML` (агрегатор `buildFullOpenAPISpec` в `huma_full_spec.go`, единая 3.1-спека всех доменов), embed-копии больше нет. **committed [`docs/keeper/openapi.yaml`](../keeper/openapi.yaml) — ПРОИЗВОДНЫЙ huma-генерат** (для UI-vendor + git-ревью), пишется `make gen-openapi`, drift-guard `make check-openapi` (committed == huma-dump байт-в-байт). Удалены: пакет `keeper/internal/api/oapi/` (`types.gen.go`/`server.gen.go`), `keeper/internal/api/meta/` (embed-источник), рукописная `openapi.yaml` как ИСТОЧНИК, мост `oapi_strict.go`/`strictServer`, oapi-codegen-тулчейн, Makefile-таргеты `gen-api`/`check-gen-api`/`sync-openapi` (заменены `gen-openapi`/`check-openapi`). Формулировки «переходное сосуществование» / «часть роутов huma, часть рукопись» / «pilot — один роут» ниже по тексту — **исторические**, относятся к ходу эпика; финальное состояние — единственный источник формы = Go-типы huma, спека целиком производна. Спека-мерж двух источников (см. §Спека-мерж) **отпал** — рукописный хвост исчез вместе с oapi-каркасом, остался один huma-dump (3.1).

> **Amendment 2026-06-12 (FULL-TYPED разворот pilot-1).** Решение пользователя: huma-вход/выход для всех доменов — **FULL-TYPED** (typed `Body` + typed output), а НЕ делегация через `RawBody`-мост. Ранее отвергнутая альтернатива «(б) full-typed для существующих доменов» (см. ниже) — **ПРИНЯТА**; дороговизну ре-wire каждого домена пользователь принял осознанно (цена one-time, выигрыш — честная спека и ноль доменного `DisallowUnknownFields`). Соответственно `RawBody`-мост (бывший pilot-1 pattern, §Инвариант-2 «нюанс делегации») — **ОТВЕРГНУТ**, заменён FULL-TYPED read+write. Pilot-1 (`POST /v1/cadences`) переписан на эталонный FULL-TYPED pattern, который тиражируется на ~20 доменов. Мотив: один источник формы (Go-struct с huma-тегами) даёт и спеку, и валидацию, и output без параллельной доменной декод-логики; делегационный мост оставлял строгость за доменным кодом (две точки правды формы). Конкретика pattern — §Pattern и §Инвариант-2 ниже.

> **Amendment 2026-06-13 (третий tier Pattern — PATCH-presence через `Optional[T]`).** Решение пользователя: PATCH-роуты, где явный `null` обязан отличаться от «поле опущено» (presence-семантика), переводятся на generic `Optional[T]` (presence В ТИПЕ), а НЕ на `RawBody []byte` + ручной presence-парс. `RawBody []byte` для этого ОТВЕРГНУТ вторично (рецидив §-(д) RawBody-моста): huma на поле `RawBody []byte` безусловно ставит `application/octet-stream` в `requestBody`, что ломает web-codegen при мерже спеки. `Optional[T]` (`UnmarshalJSON` + `huma.SchemaProvider`) держит presence в типе и оставляет `requestBody` чистым `application/json` (nullable-схема вложенного `T`). Tier **РЕДКИЙ**: на 2026-06-13 единственный носитель — `PATCH /v1/roles/{name}/permissions` (`default_scope`); прочие PATCH (cadence/service/synod/incarnation-hosts) presence НЕ требуют — им хватает read-modify-write `*T omitempty` (nil = не трогать). Эталон + guard — §Pattern «третий tier» ниже.

> **Amendment 2026-06-13 (ЧЕТВЁРТЫЙ tier Pattern — read-with-typed-query, решение A).** Решение пользователя (вариант **A** — СОХРАНИТЬ контракт `400/422`, продолжение [ADR-051](0051-operator-api-codegen.md) Amendment, НЕ product-fork): READ-list-эндпоинты с query-фильтрами переводятся на huma full-typed с **типизированной query** (`query:"…"`-теги: `[]string`/`string`/`time.Time`/`int`). Эталон — `GET /v1/audit`; tier тиражируется на ~13-15 list-эндпоинтов (audit/operators-list/souls-list/incarnations-list/voyages-list/cadences-list/…). КОНТРАКТ `400/422` сохранён без изменения wire: bad parse-значение typed-query (date-time/int/bool/float) → **400 `TypeMalformedRequest`**, bad enum-значение → **422 `TypeValidationFailed`** — ровно как давала strict bind-фаза (ADR-051 Amendment 2026-06-10). Развилку детекта (parse-error vs enum-mismatch на одинаковом `query.`-Location) разрешает Message-литерал huma-`parseInto`, НЕ Location. Эталон + guard + coupling-комментарий — §Pattern «четвёртый tier» ниже.

> **Amendment 2026-06-15 (OpenAPI-вьювер `GET /docs` + served-spec за JWT — механизм A).** Над производной huma-спекой поднят **визуальный OpenAPI-вьювер** (рекорд реализованного, решения пользователя приняты; не новые решения). Два следствия: (1) новый `GET /docs` — публичный shell-каркас вьювера (RapiDoc web-component, go:embed-ассеты), сама API-поверхность приходит лишь после ввода JWT; (2) `GET /openapi.yaml` сменил security **публичный → за JWT** (усиление — нет анонимной разведки API-поверхности; UI-vendor/soulctl не затронуты, берут committed-файл). Вьювер = **RapiDoc** (выбран ради встроенного full-text-поиска по операциям — `allow-advanced-search`); для inline-рендера keeper отдаёт спеку И JSON-ом (`GET /openapi.json` за тем же JWT, тот же `buildFullOpenAPISpec`), т.к. `RapiDoc.loadSpec` принимает разобранный ОБЪЕКТ, а не URL/YAML-текст. Детали и обоснование механизма A — §OpenAPI-вьювер (`GET /docs`) ниже.

**Контекст.** [ADR-051](0051-operator-api-codegen.md) положил источником **формы** Operator API рукописную OpenAPI-спеку ([`docs/keeper/openapi.yaml`](../keeper/openapi.yaml)), из которой `oapi-codegen` генерирует Go-типы (`keeper/internal/api/oapi/`). Фаза 2 (strict-server, S6) для существующих доменов реализована «мостом» (`bridgeMiddleware` + `strictServer`-делегация в доменные `func(w,r)`-handler-ы), потому что typed-вход oapi-strict несовместим с инвариантом «ноль wire-change» при лояльном декоде. В итоге spec-first дал высокую церемонию: рукописная спека + committed `types.gen.go` + `server.gen.go` (≈900 КБ generated) + мост-обвязка ради сохранения доменных handler-ов, при этом строгость (`unknown→400`) и точная wire-форма всё равно держатся доменным кодом, а не сгенерённым strict-слоем. Источником **формы** де-факто остаётся Go-код handler-а, а спека — параллельная рукопись, которую нужно держать в синхроне (drift-test + `make sync-openapi`).

**Разворот.** Источник правды формы — **Go-типы**; OpenAPI-спека **производна** (генерируется из кода). Инструмент — **huma v2** (`github.com/danielgtaylor/huma/v2`, выбор пользователя; pilot на `v2.38.0`) с адаптером `humachi` (`adapters/humachi`) поверх существующего `chi.Mux`.

**Три strict-инварианта Фазы 2 переносятся в НОВОЙ (инвертированной) форме:**

1. **Источник = Go-типы, спека производная.** Форма request/reply живёт в Go-struct-ах с huma-тегами (`required`/`enum`/`doc`); OpenAPI генерируется из них (`huma.SchemaFromType` / `api.OpenAPI()`), а не пишется руками. Рукописная `openapi.yaml` упраздняется при завершении тиража (на pilot — сосуществует, см. ADR-011 amend).

2. **Валидация `required`/`enum`/`unknown→400` — из huma-тегов struct (НАТИВНО, FULL-TYPED).** huma валидирует typed `Body`-вход по схеме, сгенерённой из Go-тегов; `additionalProperties:false` — **честный** (НЕ снимается, как было в отвергнутом `RawBody`-мосте). Доменный `DisallowUnknownFields` БОЛЬШЕ НЕ участвует в валидации wire-границы — её целиком держит huma:
>    - **unknown-поле → 400.** huma добавляет в `res.Errors` запись `ErrorDetail{Message: "unexpected property"}` (`validation.MsgUnexpectedProperty`). Глобальный error-override (`humaProblemError`) детектит этот message и классифицирует ошибку как **400 `TypeMalformedRequest`** (единый problem+json-контракт кластера: unknown→400, исторически — доменный `DisallowUnknownFields`). Без детекта huma вернул бы дефолтный 422 — расхождение с контрактом.
>    - **missing-required / enum-mismatch / type-mismatch → 422.** Прочие validation-ошибки huma (нет `unexpected property`) → **422 `TypeValidationFailed`** (совпадает с доменной классификацией 422).
>    
>    Override смотрит на `errs ...error`, переданные `huma.NewError` из `WriteErr(api, ctx, errStatus, "validation failed", res.Errors...)`: есть `unexpected property` → 400, иначе → статус huma (422). Это и есть «один источник формы»: один Go-struct с тегами даёт схему OpenAPI, валидацию входа и классификацию ошибок — без второй точки правды (доменного декода). `RawBody` уходит из input-struct полностью.

3. **Committed-gen ИНВЕРТИРУЕТСЯ.** Раньше коммитили `types.gen.go`/`server.gen.go`, check-target (`check-gen-api`) проверял идемпотентность кодогена. Теперь коммитим **производный `openapi.yaml`** (dump из huma-кода), а check-target проверяет **`huma-dump == committed openapi.yaml`** (спека в репо не отстала от Go-кода). Генератора Go-типов больше нет — Go-типы и есть источник.

**Security (сохраняется без регресса).**
- **RBAC/Audit-mount.** huma-операция монтируется ВНУТРИ существующей chi-группы `r.With(RequireJWT, RequirePermission, …).Group(...)` — `humachi.New(router, cfg)` подключает операцию к ЭТОЙ группе обычным chi-роутом, вся навеска (JWT/RBAC/Audit/Toll/Tempo/maxBody/metrics) применяется как к любому chi-роуту. `chi.Walk` видит huma-роут → drift-test и per-route RBAC работают без изменений. Авто-роутер huma НЕ используется (как запрет `HandlerFromMux` в ADR-051) — mount ручной per-route.
- **Единый problem+json.** `huma.NewError` переопределяется глобально на наш RFC 7807 `problem.*` (`humaProblemError`: несёт `problem.Details`, реализует `huma.StatusError` + `ContentTypeFilter`→`application/problem+json`). huma-сгенерированные ошибки (415, body-limit, missing-required-422) выходят нашим problem-форматом, не huma-`ErrorModel`. Доменные ошибки и так идут через `problem.Write` внутри handler-а.
- **Audit-проводка — ЗОНА РИСКА (урок S6). Два класса доменов:**
>    - **Self-audit домены (`cadence`/`incarnation`/`errand` и т.п.).** Пишут audit САМ внутри доменной функции через `h.auditW` (не через Audit-middleware). При FULL-TYPED извлечении доменной функции (`CreateTyped`) audit-emit остаётся ВНУТРИ неё (вызывается на успешном write-пути, до возврата typed reply) — huma-обёртка его не задевает. **huma-audit-middleware НЕ нужен.** Pilot-1 cadence — этого класса (guard `TestHumaCadence_Create_AuditRecorded`: непустой payload на 201).
>    - **Middleware-audit домены (audit идёт через `apimiddleware.Audit`/StatusRecorder).** Audit-запись делает middleware, читая статус ответа. huma не отдаёт `http.ResponseWriter` доменной функции (typed output), поэтому middleware-audit для них реализуется **вариантом B — huma `UseMiddleware`** (PILOT-2 PROVEN, role.create). Эталон этого класса — **pilot-2** (`POST /v1/roles`, не входит в pilot-1).
>
>    **PILOT-2 PROVEN — железный pattern тиража middleware-audit-доменов (дословно):**
>
>    1. **Spike `hctx.Status()` — нативно, БЕЗ fallback-обёртки.** `api.UseMiddleware` регистрирует middleware `func(hctx huma.Context, next func(huma.Context))`; huma строит цепочку `api.Middlewares().Handler(endpoint)` (`chain.go`) — middleware ОБОРАЧИВАЕТ huma core-handler. Core-handler КАЖДЫЙ путь ответа завершает `ctx.SetStatus(status)` (success → `output.Status`; validation-fail → `WriteErr`→`SetStatus(errStatus)`; domain-`StatusError` → `transformAndWrite`→`SetStatus`). `humachi.chiContext.SetStatus(code)` пишет `code` в **поле** `c.status` И зовёт `w.WriteHeader`. `chiContext.Status()` возвращает `c.status` (ПОЛЕ структуры, НЕ чтение из `http.ResponseWriter`); тот же `*chiContext` тредится через `next` → ПОСЛЕ `next(hctx)` `hctx.Status()` == реальный финальный статус, даже что headers уже сflush-ились. Early-flush НЕ ломает чтение статуса → huma-аналог `StatusRecorder` НЕ требуется.
>    2. **Skip-условие — parity `apimiddleware.Audit`.** После `next`: `status>=300 || status==0` → skip (4xx/5xx — операция не выполнена; 0 — panic/ранний huma-отказ до `SetStatus`); 2xx → пишем.
>    3. **Claims-проброс.** `RequireJWT` (chi-middleware группы) кладёт claims в request-context ДО `humachi`; middleware достаёт их `apimiddleware.ClaimsFromContext(hctx.Context())`. Нет claims → warn + skip.
>    4. **Payload-проброс handler → middleware — мутируемый carrier.** huma-middleware исполняется СНАРУЖИ handler-closure (она возвращает `*output`, huma.Context после возврата недоступен), а huma.Context иммутабелен (`huma.WithValue` создаёт новый context) — классический `SetAuditPayload` (он `*r.WithContext`) к huma неприменим. Решение: middleware seed-ит пустой `*HumaAuditCarrier` в request-context через `huma.WithValue(hctx, HumaAuditCarrierKey{}, carrier)` ДО `next` и тредит обновлённый hctx; huma-handler-closure (`ctx` == этот context) кладёт payload через `middleware.SetHumaAuditPayload(ctx, …)`; ПОСЛЕ `next` middleware читает ТОТ ЖЕ указатель. Указатель общий → мутация видна (параллель `SetAuditPayload`, но carrier seed-ится заранее, без `*r.WithContext`).
>    5. **Write.** `writer.Write(context.Background(), &audit.Event{EventType:evt, Source:SourceAPI, ArchonAID:claims.Subject, Payload:carrier.Payload})` — Background-ctx (request-ctx может быть отменён сразу после ответа), ошибка best-effort (parity `Audit`). `writer==nil` (dev без audit) → middleware прозрачен.
>    6. **Mount.** Одна `huma.API` на chi-группу + один audit-event-тип: `newHumaAuditAPI(r, writer, evt, logger)` = `newHumaCadenceAPI(r)` + `api.UseMiddleware(humaAuditMiddleware(writer, evt, logger))`. Доменная функция кладёт payload в huma-конверте (`registerHumaRole`), а НЕ внутри `XTyped` (в отличие от self-audit-доменов).
>
>    - **S6-guard-инвариант для huma.** На каждый write-домен тиража — обязательный audit-guard-тест (E2E через router), ловящий регресс «huma проглотил/не довёл write-путь до audit». Повторение урока S6 в huma-форме. Pilot-2: `TestHumaAudit_RoleCreate_RecordsOnSuccess` (непустой payload на 201) + `TestHumaAudit_NoAudit_OnReject` (422 → 0 событий) + `TestHumaAudit_NoAudit_OnRBACDeny` (403 → 0 событий).

**OpenAPI-вьювер (`GET /docs`) + served-spec за JWT — механизм A (Amendment 2026-06-15).** Над производной huma-спекой ([§Спека-мерж](#спека-мерж-ключевой-архитектурный-вопрос-тиража): единый huma-dump 3.1) поднят визуальный вьювер для оператора; реализация в `keeper/internal/api/docs_viewer.go` + `keeper/internal/api/docsassets/`.

- **(а) `GET /docs` — визуальный вьювер ([RapiDoc](https://rapidocweb.com), go:embed).** Публичный HTML-shell вне `/v1` (без auth). Ассет web-компонента (`rapidoc-min.js`, один файл — RapiDoc держит стили в Shadow DOM, отдельного CSS нет) **вшит go:embed** (пакет `docsassets`) и раздаётся под `GET /docs/assets/*` — НЕ тянется с CDN (air-gapped/закрытый контур, «безопасность на первом месте», офлайн-инсталляции; один бинарь несёт всё). RapiDoc распространяется под MIT (совместимо с Apache-2.0 репо; атрибуция транзитивных зависимостей — `rapidoc-min.js.LICENSE.txt` рядом с бандлом на диске). Mount — `mountDocsViewer(r)` рядом с health/meta, ВНЕ `/v1` → без RBAC/audit/maxBody/metrics-навески (как `/healthz`/`/openapi.yaml`).

- **(б) `GET /openapi.yaml` сменил security публичный → за JWT; добавлен `GET /openapi.json` за тем же JWT.** Раньше served-spec был публичен (ADR-051(c)/T4c); теперь — required Bearer (тот же `RequireJWT`, что и `/v1`), но БЕЗ `/v1`-обвязки (maxBody/metrics/audit/RBAC): спека статична, mount остаётся ВНЕ `/v1` (URL не менялся). Усиление безопасности — **нет анонимной разведки полной API-поверхности**. Для inline-рендера вьювера keeper отдаёт ту же спеку дополнительно **JSON-ом** (`GET /openapi.json`, тот же `buildFullOpenAPISpec` — единый source-of-truth, только формат другой): `RapiDoc.loadSpec` принимает разобранный ОБЪЕКТ, поэтому страница фетчит JSON с Bearer и подаёт parsed-объект. UI-vendor (`openapi-typescript` на soul-stack-web) и `soulctl` не затронуты: они берут **committed** [`docs/keeper/openapi.yaml`](../keeper/openapi.yaml) (производный снимок), а не live-served-spec.

- **(в) `bearerAuth` securityScheme в агрегатор-спеке.** В huma-агрегатор (`HumaFullSpecYAML`) добавлен `components.securitySchemes.bearerAuth` (`type: http`, `scheme: bearer`, `bearerFormat: JWT`) + global `security: [{bearerAuth: []}]`. Дают вьюверу описание auth-механизма для «Try It» (тот же JWT) и явно фиксируют в спеке, что все `/v1`-операции требуют Bearer (симметрия [§Auth](../keeper/operator-api.md#auth)).

- **(г) Механизм A — обоснование.** Браузер при навигации (открытие `/docs`) НЕ может проставить `Authorization: Bearer …` заголовок, а спека теперь за JWT — поэтому **shell-страница публична, спека загружается отдельным `fetch` с Bearer**, который JS добавляет после ввода токена. Оператор вставляет Archon-JWT в поле → `fetch('/openapi.json', {Authorization: 'Bearer …'})` → `401` ⇒ сообщение «invalid/expired JWT», `2xx` ⇒ inline-render спеки. RapiDoc рендерит через метод **`loadSpec(объект)`** (разобранный JSON спеки), а НЕ передачей spec-URL: `loadSpec(строка)` RapiDoc трактует как spec-URL и фетчит её сам БЕЗ нашего Bearer → упёрся бы в `401`. Поэтому keeper и отдаёт спеку JSON-ом, а страница подаёт `resp.json()` в `loadSpec`. Тот же JWT прокидывается в `<rapi-doc>` для «Try It» через `setApiKey('bearerAuth', jwt)` — **ЧИСТЫЙ jwt без префикса**: RapiDoc сам добавляет `'Bearer '` для http/bearer-схемы `bearerAuth`, ручной префикс дал бы двойной Bearer. Full-text-поиск по операциям — встроенный (`allow-advanced-search`), это и есть причина выбора RapiDoc. Токен держится в **`sessionStorage`** (per-tab, не переживает закрытие вкладки), НЕ в `localStorage`; передаётся Bearer-заголовком, НЕ в URL (исключает утечку в access-логи/referer/историю). **Cookies/sessions НЕ вводятся** — консистентно [ADR-014](0014-operator-identity.md#adr-014-identity-модель-оператора-archon) (JWT-bearer — единственный auth-метод MVP). Неавторизованный посетитель `/docs` видит только поле ввода — API-поверхность не раскрыта.

- **Имя роута `/docs`.** Индустрия-стандартное имя вьювера, выбрано пользователем над словарными вариантами (`/scriptorium/codex`) ради узнаваемости оператором — осознанное исключение из «словарь Soul Stack» (зафиксировано в [naming-rules.md → Идентификаторы](../naming-rules.md)).

**Pattern (эталон тиража — вариант (б) тонкий-конверт).** Каждый write-домен переводится на huma по единой схеме (доказана pilot-1 cadence; дословная инструкция тиража):

1. **Typed input.** `xCreateInput struct { Body xCreateHumaBody }` — huma-теги (`required`/`enum`/`additionalProperties:false` честный, `doc`) на полях `xCreateHumaBody`. huma валидирует тело из тегов (НЕ `RawBody`). Path/query/header-параметры — отдельные поля input-struct с huma-тегами.
2. **Извлечённая доменная функция.** Из `Handler.X(w,r)` извлекается чистая `XTyped(ctx, claims, req xCreateRequest) (xCreateReply, error)`: вся бизнес-логика (RBAC-guards, валидация, persist tx, audit-emit, invalidation) — без `http.ResponseWriter`/`*http.Request`. Все `problem.Write(w, …)` → `return zeroReply, &problemError{<problem.Details>}` (typed-обёртка над `problem.Details`, тот же тип семейства, что `humaProblemError`). RBAC-guards возвращают `error` вместо `(denied bool + Write)`. Старый `X(w,r)` ОСТАЁТСЯ тонкой оболочкой над `XTyped` (декод + claims + `problem.Write` при error) — для strict-моста и прочих вызовов, пока oapi жив.
3. **Конверт.** huma-handler: typed `input.Body` → доменная-модель `xCreateRequest` (поле-в-поле) → `XTyped(ctx, claims, req)` → доменная `xCreateReply` → конверт в typed `xCreateOutput` (`Status`/`header:"Location"`/`Body`). Конверт — единственный «клей»; оба конца типизированы.
4. **Typed output.** `xCreateOutput struct { Status int; Location string \`header:"Location"\`; Body xCreateReplyHumaBody }`. Заменяет oapi-reply-тип. omitempty/nullable полей фиксируется golden-JSON snapshot (wire-регресс-guard тиража).
5. **huma.Register.** full-typed `func(ctx, *xCreateInput) (*xCreateOutput, error)`. unknown→400 / required→422 — нативно huma (§Инвариант-2). Self-audit домены — audit внутри `XTyped`; middleware-audit домены — вариант B (§Audit).

**Pattern «третий tier» — PATCH-presence через `Optional[T]` (Amendment 2026-06-13, РЕДКИЙ).** Применять ТОЛЬКО к PATCH-полю, где явный `null` (сброс) обязан отличаться от «поле опущено» (не трогать) — `*T omitempty` схлопывает оба случая в `nil`. Generic `Optional[T] struct { Set bool; Null bool; Value T }` (`keeper/internal/api/huma_optional.go`) несёт presence В ТИПЕ:
- **`UnmarshalJSON`** — стандартный json-декодер зовёт метод ТОЛЬКО когда ключ физически в теле → `Set=true`; `null` → `Null=true` (zero `Value`); иначе `json.Unmarshal` в `Value`. Поле опущено → метод не вызван → `Set=false` (zero).
- **`huma.SchemaProvider` (`Schema(r huma.Registry) *huma.Schema`)** — отдаёт схему вложенного `T` (`r.Schema(reflect.TypeFor[T](), …)`) с `Nullable:true`. Для `Optional[string]` schema-фрагмент = `{type: [string, null]}` в чистом `application/json` — БЕЗ `application/octet-stream`. Поле НЕ в `required` (управляется тегом `required:"false"`).
- **Конверт.** huma-обёртка маппит `Optional[T]` в доменный presence-контракт: presence-флаг (`SetX`) ← `Set`; значение (`*T`) ← `optionalToPtr` (`Set && !Null` → `&Value`, иначе `nil`). Доменная функция трактует `(SetX=false)` как «не трогать», `(SetX=true, ptr=nil)` как «сброс», `(SetX=true, ptr=&v)` как «установить».

**Эталон.** `PATCH /v1/roles/{name}/permissions` (`default_scope`, ADR-047 scope-narrowing): `roleUpdatePermissionsHumaBody.DefaultScope Optional[string]` → `handlers.UpdatePermissionsInput{SetDefaultScope, DefaultScope}` → `rbac.Service`. `RawBody []byte` + `jsonBodyHasKey` (бывший api-side presence-мост) **убраны полностью**.

**RawBody []byte для presence — ОТВЕРГНУТ вторично.** Рецидив §-(д) RawBody-моста: huma на поле `RawBody []byte` безусловно ставит `application/octet-stream` в `requestBody` (вместо `application/json`), что ломает web-codegen (`openapi-typescript`) при мерже спеки. `Optional[T]` устраняет артефакт по построению.

**Guard.** golden OpenAPI-фрагмента ассертит ОТСУТСТВИЕ `application/octet-stream` (`TestHumaRole_PatchPermissions_RequestBody_JSONOnly`, `TestHumaRole_OpenAPIFragment_3_1`); presence-конверт 3 ветки (omitted/null/value) проверены route-level через реальный PATCH-роут до доменного дна (`TestHumaRole_UpdatePermissions_ScopePresence`, SECURITY-relevant — RBAC scope-narrowing); `UnmarshalJSON`/`Schema`/round-trip — `huma_optional_test.go`.

**Tier РЕДКИЙ.** На 2026-06-13 единственный носитель — `default_scope` в PATCH-permissions. Прочие PATCH (cadence/service/synod/incarnation-hosts) — read-modify-write `*T omitempty` (nil = не трогать), presence НЕ нужна. Не плодить `Optional[T]` там, где явного сброса в семантике поля нет.

**Pattern «четвёртый tier» — read-with-typed-query (Amendment 2026-06-13, решение A).** READ-list-эндпоинт с query-фильтрами переводится на huma full-typed с типизированной query. Эталон — `GET /v1/audit` (`keeper/internal/api/huma_audit_endpoint{,_op}.go`); тираж — ~13-15 list-эндпоинтов:

1. **Typed query-input.** `xListInput struct { … }` — КАЖДОЕ поле фильтра несёт `query:"<name>"`-тег; huma биндит из `url.Values` и валидирует по схеме из тегов. RequestBody у GET НЕТ (huma не выводит body без `Body`-поля; guard ассертит отсутствие `requestBody`). Типы: `[]string` (multi-value `?k=a&k=b`, OR-семантика в доменной функции, ОБЯЗАТЕЛЬНО `,explode` — см. §1b), `string` (exact), `time.Time` (date-time-bind huma), `int32` (offset/limit — см. §1c).
   - **§1b. Multi-value `[]string`-query → `query:"<name>,explode"` (правило тиража).** huma-дефолт для query-array — `explode=false`: он читает comma-separated `?k=a,b` КАК ОДНО `"a,b"` (берёт лишь первое значение ключа + `strings.Split`, huma v2.38 `huma.go:1724-1729`) И эмитит `explode: false` в спеку. Это ломает OR-фильтр: повторяющийся ключ `?source=a&source=b` теряет всё, кроме `a`, а сгенерённый клиент кодирует список как составное значение. `,explode` в query-теге → `pfi.Explode=true` (`huma.go:157`): huma читает все повторяющиеся ключи `Query()[name]` И эмитит `explode: true` (`openapi.go` `Param.Explode`-указатель), совпадая с committed-спекой `style:form` + `explode:true`. Применять КО ВСЕМ multi-value `[]string`-полям list-фильтров (`type`/`source`/…) — без диалектов.
   - **§1c. Pagination — `int32`, НЕ Go-`int`.** huma на `reflect.Int` (64-бит платформа) эмитит `format:int64` (`schema.go:804-810`), а committed-спека (`OffsetQuery`/`LimitQuery`) и ADR-текст несут `int32`; пагинация (offset, limit≤1000) влезает в int32. Поля input — `int32` → huma эмитит `format:int32` (match committed, минимальный drift). Конверт в доменный filter (`int`) — расширяющий `int(in.Offset)` без потери. Применять ко ВСЕМ pagination-полям тиража.
2. **enum — `enum:"v1,v2,…"`-тег на `[]string`-поле = ПОЛНЫЙ доменный valid-set (правило тиража, исправлено Amendment 2026-06-13).** huma отбивает значение вне набора на **422** (schema-validate). Набор enum = доменный valid-set (для audit-`source` — `audit.Source.Valid()`: `signal,api,mcp,keeper_internal,soul_grpc,background,config_bootstrap` — ВСЕ семь). **Code-first: Go-домен — источник правды, committed-спека синхронизируется С ДОМЕНОМ, а не домен под спеку.** `config_bootstrap` реально эмитится (`push/auto_import.go`) и принимается доменом; его пропуск в enum-теге отбивал бы рабочий фильтр `?source=config_bootstrap` на **422** (wire-regression 200→422 vs легаси strict, который enum на bind НЕ валидировал). Committed `openapi.yaml`/`meta` source-enum были stale (drift от домена) — выровнены под домен. Guard `TestHumaAudit_ConfigBootstrapSource_200` фиксирует 200 (НЕ 422).
3. **Извлечённая доменная `XListTyped(ctx, filter) (reply, error)`.** Чистая read-логика без `http.ResponseWriter`/`*http.Request`: доменная валидация (enum defense-in-depth → 422), фильтр, чтение страницы, проекция в typed-envelope. Старый `X(w,r)` остаётся тонкой оболочкой (парсит query → тот же доменный filter → `XListTyped`).
4. **Typed envelope-output.** `Body` = тот же list-envelope `{items, offset, limit, total}`, что отдавала легаси `writeJSON` — golden-JSON byte-exact == legacy (главный guard tier-а; `items` non-nil `[]`).
5. **КОНТРАКТ 400/422 — инвариант tier-а (решение A).** Override (`newHumaProblemError`) расширен ADDITIVE-веткой `hasQueryParseError`: `*huma.ErrorDetail` с `strings.HasPrefix(Location, "query.")` И Message из parse-набора huma-`parseInto` (`invalid date/time` ПРЕФИКС / `invalid integer` / `invalid boolean` / `invalid float`, huma v2.38 `huma.go:1685-1769`) → **400**. Enum-mismatch (Schema-validate, Message `expected value to be one of …`, тот же `query.`-Location) в набор НЕ попадает → дефолтная **422**-ветка. **Дискриминатор parse-vs-enum — именно Message-литерал, не Location** (оба `query.`).
6. **Границы пагинации — БЕЗ huma `minimum`/`maximum`.** `offset`/`limit` (тип `int32`, см. §1c) несут только `default` (0/50, совпадает с `api.ParsePage`); диапазон (offset≥0, limit∈[1,1000]) НЕ выражен schema-min/max СОЗНАТЕЛЬНО — иначе huma вернул бы out-of-range на **422**, а легаси/strict-контракт даёт **400** (`ParsePage`). Диапазон enforce-ит доменная `XListTyped` через единый `api.CheckPageBounds` (тот же источник границ и сообщений, что `ParsePage`) → 400. Иначе wire-change на `limit=0/1001/offset<0`. (committed `OffsetQuery`/`LimitQuery` несут `minimum`/`maximum` лишь как документацию схемы — huma-вход их НЕ повторяет, enforce доменный.)
7. **zero-time → no-filter.** Опущенный date-time-параметр huma оставляет zero-value `time.Time` (parseInto не вызван) → доменная функция трактует `IsZero` как «без временной границы» (parity легаси `if param != ""`).
8. **READ — без audit-middleware** (чтение само audit-event не порождает; для audit_log ещё и рекурсия).

**Coupling на huma-Message-литералы.** `queryParseMessages` — копия точных строк huma-`parseInto`; при bump huma свериться с ним. Guard-набор (особенно КЛЮЧЕВОЙ `BadSource_422`) ловит регресс детекта при рассинхроне.

**Guard-набор (TDD).** `huma_audit_endpoint_test.go`: `BadStartedAfter_400` (bad date-time→400), `BadOffset_400`/`BadLimit_400` (bad int→400), `OutOfRangePagination_400` (limit=0/1001/offset<0→400, parity ParsePage), **`BadSource_422`** (bad enum→422 — КЛЮЧЕВОЙ, доказывает что parse-детект НЕ ловит enum), **`ConfigBootstrapSource_200`** (валидный config_bootstrap-source → 200, НЕ 422 — enum-тег = домен-valid-set, MAJOR-регрессия-guard), `ValidFilters_200`/`DefaultPagination_200` (200 + golden envelope byte-exact, границы совпадают с ParsePage), `OpenAPIFragment_3_1` (typed query-params date-time/int32/enum-вкл-config_bootstrap; `explode: true` на multi-value; нет `int64`; нет requestBody у GET).

**Обоснование.**
- **Один источник формы — Go-типы.** Устраняет рукопись спеки + committed gen + drift-test между ними: спека производна, рассинхрон невозможен по построению.
- **Меньше церемонии.** Уходит мост-обвязка oapi-strict (`bridgeMiddleware`/`strictServer`/15-арный конструктор) — huma валидирует typed `Body` из тегов и зовёт извлечённую доменную функцию (`XTyped`) через тонкий конверт, без `humachi.Unwrap`/`RawBody`-моста и без 900 КБ generated.
- **Безопасность не страдает.** Ручной mount в chi-группу с навеской + глобальный problem-override + per-домен audit-guard сохраняют RBAC/Audit/problem+json. Доказано pilot.

**Отвергнутые альтернативы.**
- **(а) Остаться на spec-first (ADR-051).** Источником формы де-факто стал Go-код handler-а (мост держит wire-форму и строгость), спека — параллельная рукопись с drift-риском + 900 КБ generated. Отвергнут пользователем — разворот на code-first.
- **(б) Full-typed huma для существующих доменов (typed-вход/выход + извлечённая доменная функция).** Изначально отвергался как несоразмерный (повторная Фаза 1: re-wire + ре-доказательство date-time/omitempty per-домен). **ПРИНЯТ Amendment 2026-06-12** (решение пользователя): дороговизна one-time оправдана — единый источник формы (Go-struct), честная спека, ноль доменного `DisallowUnknownFields`/мост-обвязки, golden-JSON митигирует wire-регресс. Стал эталоном тиража (§Pattern).
- **(д) Делегация через `RawBody`-мост (бывший pilot-1 pattern).** huma-операция несёт `RawBody []byte`, снимает `additionalProperties:false`, делегирует в доменный `func(w,r)` (строгость за доменным `DisallowUnknownFields`). **ОТВЕРГНУТ Amendment 2026-06-12** в пользу (б): мост оставлял две точки правды формы (huma-схема + доменный декод), требовал `humachi.Unwrap`/context-bridge-middleware, не давал typed output. FULL-TYPED проще по построению и устраняет рассинхрон схема↔декод. **Вторично ОТВЕРГНУТ Amendment 2026-06-13** как механизм PATCH-presence: huma на поле `RawBody []byte` безусловно ставит `application/octet-stream` в `requestBody` → ломает web-codegen при мерже; presence несёт `Optional[T]` в типе (§Pattern «третий tier»).
- **(в) Авто-роутер huma (`humachi.New` + auto-mount без chi-группы).** Навесил бы операции мимо per-route RBAC/Audit (security-регресс, как `HandlerFromMux` в ADR-051). Отвергнут — ручной mount в chi-группу с навеской.
- **(г) huma-`ErrorModel` как формат ошибок.** Разошёлся бы с problem+json-контрактом кластера. Отвергнут — глобальный `huma.NewError`-override на `problem.*`.
