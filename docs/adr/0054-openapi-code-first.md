# ADR-054. Operator API - reversal to code-first (Go types -> OpenAPI) via huma v2

> **Status: active.** Supersedes ([supersede](#connection-to-adr)) [ADR-051](0051-operator-api-codegen.md) - the premise is inverted: it was **OpenAPI -> Go types** (spec-first, oapi-codegen), it became **Go types -> OpenAPI** (code-first, huma). User decision (2026-06-12). Epic started as **PILOT-SPIKE** per the bulk-operations rule: canon (this ADR + amendment to ADR-011) + ONE route (`POST /v1/cadences`) via huma - proof of integration BEFORE rollout. Demolition of the oapi scaffold and rollout across ~20 domains - AFTER the pilot is proven.

> **Amendment 2026-06-13 (ROLLOUT COMPLETE + oapi-scaffold DEMOLISHED - final state, HEAD `fde65bf`).** The reversal is fully implemented. **All ~19 Operator API domains are handler-native huma full-typed** (`keeper/internal/api/huma_*.go`), no `oapi.X` types remain in the code. **served-spec = runtime huma-dump** from code: `GET /openapi.yaml` returns `servedOpenAPIHandler` -> `HumaFullSpecYAML` (aggregator `buildFullOpenAPISpec` in `huma_full_spec.go`, a single 3.1 spec across all domains), there is no longer an embed copy. **committed [`docs/keeper/openapi.yaml`](../keeper/openapi.yaml) - a DERIVED huma-generated artifact** (for UI-vendor + git review), written by `make gen-openapi`, drift-guard `make check-openapi` (committed == huma-dump byte-for-byte). Removed: package `keeper/internal/api/oapi/` (`types.gen.go`/`server.gen.go`), `keeper/internal/api/meta/` (embed source), the handwritten `openapi.yaml` as SOURCE, the bridge `oapi_strict.go`/`strictServer`, the oapi-codegen toolchain, Makefile targets `gen-api`/`check-gen-api`/`sync-openapi` (replaced by `gen-openapi`/`check-openapi`). Native enum catalog - `huma_enums.go`. The wording "transitional coexistence" / "part of the routes on huma, part handwritten" / "pilot - one route" further down in the text - is **historical**, refers to the course of the epic; the final state is a single source of form = Go types via huma, the spec is entirely derived. The spec-merge of two sources (see §Spec-merge) **falls away** - the handwritten tail disappeared along with the oapi scaffold, leaving a single huma-dump (3.1).

> **Amendment 2026-06-12 (FULL-TYPED reversal pilot-1).** User decision: huma input/output for all domains - **FULL-TYPED** (typed `Body` + typed output), and NOT delegation via a `RawBody` bridge. The previously rejected alternative "(b) full-typed for existing domains" (see below) - **ACCEPTED**; the user consciously accepted the extra cost of re-wiring every domain (a one-time price, the payoff - an honest spec and zero domain-level `DisallowUnknownFields`). Accordingly the `RawBody` bridge (formerly the pilot-1 pattern, §Invariant-2 "delegation nuance") - **REJECTED**, replaced by FULL-TYPED read+write. Pilot-1 (`POST /v1/cadences`) is rewritten to the reference FULL-TYPED pattern, which is rolled out to ~20 domains. Motive: a single source of form (a Go struct with huma tags) yields the spec, validation, and output alike without parallel domain decode logic; the delegation bridge left strictness to domain code (two points of truth for the form). Specifics of the pattern - §Pattern and §Invariant-2 below.

> **Amendment 2026-06-12 (PILOT-2 - huma-audit-middleware variant B PROVEN).** `POST /v1/roles` is switched to FULL-TYPED huma following the pilot-1 reference + a **huma-native audit-middleware (variant B)** is added - the reference for rolling out ~30 middleware-audit domains (role/operator/souls/synod/service/sigil/augur/oracle/...). The `huma.Context.Status()` spike (huma v2.38.0 + humachi) **is allowed natively**: a fallback wrapper is NOT needed (details - §Audit below). S6-guard `TestHumaAudit_RoleCreate_RecordsOnSuccess` is green (audit with a non-empty payload on 201), negative guards (422/403 -> audit is not written) are green. The pattern is solid (§Audit "PILOT-2 PROVEN"). Pilot-2 is NOT a rollout - only `role.create`; the remaining role routes stay on oapi-strict.

> **Amendment 2026-06-13 (third tier Pattern - PATCH presence via `Optional[T]`).** User decision: PATCH routes where an explicit `null` must be distinguishable from "field omitted" (presence semantics) switch to a generic `Optional[T]` (presence IN THE TYPE), and NOT to `RawBody []byte` + manual presence parsing. `RawBody []byte` is REJECTED a second time for this purpose (a recurrence of §-(d) the RawBody bridge): huma unconditionally sets `application/octet-stream` on a `RawBody []byte` field in `requestBody`, which breaks web-codegen when merging the spec. `Optional[T]` (`UnmarshalJSON` + `huma.SchemaProvider`) keeps presence in the type and leaves `requestBody` a clean `application/json` (a nullable schema of the wrapped `T`). Tier **RARE**: as of 2026-06-13 the sole bearer is `PATCH /v1/roles/{name}/permissions` (`default_scope`); other PATCH routes (cadence/service/synod/incarnation-hosts) do not need presence - read-modify-write `*T omitempty` (nil = do not touch) suffices for them. Reference + guard - §Pattern "third tier" below.

> **Amendment 2026-06-13 (FOURTH tier Pattern - read-with-typed-query, decision A).** User decision (variant **A** - PRESERVE the `400/422` contract, continuation of the [ADR-051](0051-operator-api-codegen.md) Amendment, NOT a product fork): READ-list endpoints with query filters are switched to huma full-typed with a **typed query** (`query:"..."` tags: `[]string`/`string`/`time.Time`/`int`). Reference - `GET /v1/audit`; the tier is rolled out to ~13-15 list endpoints (audit/operators-list/souls-list/incarnations-list/voyages-list/cadences-list/...). The `400/422` CONTRACT is preserved without a wire change: a bad parse value for a typed query (date-time/int/bool/float) -> **400 `TypeMalformedRequest`**, a bad enum value -> **422 `TypeValidationFailed`** - exactly as the strict bind phase gave before (ADR-051 Amendment 2026-06-10). The fork in detection (parse-error vs enum-mismatch on the same `query.` Location) is resolved by the Message literal from huma's `parseInto`, NOT by Location. Reference + guard + coupling comment - §Pattern "fourth tier" below.

> **Amendment 2026-06-15 (OpenAPI viewer `GET /docs` + served-spec behind JWT - mechanism A).** On top of the derived huma spec a **visual OpenAPI viewer** has been raised (a record of what was implemented, user decisions already taken; not new decisions). Two consequences: (1) a new `GET /docs` - a public shell scaffold for the viewer (RapiDoc web component, go:embed assets), the actual API surface only arrives after a JWT is entered; (2) `GET /openapi.yaml` changed security from **public -> behind JWT** (a hardening - no anonymous reconnaissance of the API surface; UI-vendor/soulctl are unaffected, they take the committed file). The viewer = **RapiDoc** (chosen for its built-in full-text search over operations - `allow-advanced-search`); for the inline render keeper also serves the spec as **JSON** (`GET /openapi.json` behind the same JWT, the same `buildFullOpenAPISpec`), since `RapiDoc.loadSpec` accepts a parsed OBJECT, not a URL/YAML text. Details and rationale for mechanism A - §OpenAPI viewer (`GET /docs`) below.

**Context.** [ADR-051](0051-operator-api-codegen.md) made the handwritten OpenAPI spec ([`docs/keeper/openapi.yaml`](../keeper/openapi.yaml)) the source of **form** for the Operator API, from which `oapi-codegen` generates Go types (`keeper/internal/api/oapi/`). Phase 2 (strict-server, S6) for existing domains was implemented as a "bridge" (`bridgeMiddleware` + `strictServer` delegation to domain `func(w,r)` handlers), because oapi-strict's typed input was incompatible with the "zero wire-change" invariant under a lenient decode. As a result spec-first gave high ceremony: a handwritten spec + committed `types.gen.go` + `server.gen.go` (~900 KB generated) + bridge glue for the sake of keeping domain handlers, while strictness (`unknown->400`) and the exact wire form were still held by domain code, not by the generated strict layer. The de facto source of **form** remained the Go handler code, and the spec was a parallel manuscript that had to be kept in sync (drift-test + `make sync-openapi`).

**Reversal.** The source of truth for form is now **Go types**; the OpenAPI spec is **derived** (generated from code). The tool is **huma v2** (`github.com/danielgtaylor/huma/v2`, the user's choice; pilot on `v2.38.0`) with the `humachi` adapter (`adapters/humachi`) on top of the existing `chi.Mux`.

**The three strict invariants of Phase 2 carry over in the NEW (inverted) form:**

1. **Source = Go types, the spec is derived.** The request/reply form lives in Go structs with huma tags (`required`/`enum`/`doc`); OpenAPI is generated from them (`huma.SchemaFromType` / `api.OpenAPI()`), not written by hand. The handwritten `openapi.yaml` is retired once the rollout completes (during the pilot it coexists, see the ADR-011 amendment).

2. **Validation `required`/`enum`/`unknown->400` - from struct huma tags (NATIVE, FULL-TYPED).** huma validates the typed `Body` input against the schema generated from Go tags; `additionalProperties:false` is **honest** (NOT lifted, as it was in the rejected `RawBody` bridge). Domain `DisallowUnknownFields` no longer participates in wire-boundary validation - huma owns it entirely:
>    - **unknown field -> 400.** huma adds a `ErrorDetail{Message: "unexpected property"}` entry to `res.Errors` (`validation.MsgUnexpectedProperty`). The global error override (`humaProblemError`) detects this message and classifies the error as **400 `TypeMalformedRequest`** (the cluster's uniform problem+json contract: unknown->400, historically the domain `DisallowUnknownFields`). Without the detection huma would return the default 422 - a divergence from the contract.
>    - **missing-required / enum-mismatch / type-mismatch -> 422.** Other huma validation errors (no `unexpected property`) -> **422 `TypeValidationFailed`** (matches the domain classification of 422).
>
>    The override looks at the `errs ...error` passed to `huma.NewError` from `WriteErr(api, ctx, errStatus, "validation failed", res.Errors...)`: if there is `unexpected property` -> 400, otherwise -> huma's status (422). This is exactly "one source of form": a single Go struct with tags gives OpenAPI schema, input validation, and error classification - without a second point of truth (domain decode). `RawBody` is entirely removed from the input struct.

3. **Committed-gen is INVERTED.** Previously we committed `types.gen.go`/`server.gen.go`, and a check target (`check-gen-api`) verified codegen idempotency. Now we commit the **derived `openapi.yaml`** (a dump from huma code), and the check target verifies **`huma-dump == committed openapi.yaml`** (the spec in the repo has not fallen behind the Go code). There is no longer a Go-type generator - the Go types are the source themselves.

**Security (preserved without regression).**
- **RBAC/Audit mount.** The huma operation is mounted INSIDE the existing chi group `r.With(RequireJWT, RequirePermission, ...).Group(...)` - `humachi.New(router, cfg)` attaches the operation to THIS group as a normal chi route, and all the tack-on (JWT/RBAC/Audit/Toll/Tempo/maxBody/metrics) applies as to any chi route. `chi.Walk` sees the huma route -> the drift-test and per-route RBAC work unchanged. huma's auto-router is NOT used (the same prohibition as `HandlerFromMux` in ADR-051) - the mount is manual, per route.
- **Uniform problem+json.** `huma.NewError` is overridden globally with our RFC 7807 `problem.*` (`humaProblemError`: carries `problem.Details`, implements `huma.StatusError` + `ContentTypeFilter` -> `application/problem+json`). huma-generated errors (415, body-limit, missing-required-422) come out in our problem format, not huma's `ErrorModel`. Domain errors already go through `problem.Write` inside the handler.
- **Audit wiring - RISK ZONE (lesson from S6). Two classes of domains:**
>    - **Self-audit domains (`cadence`/`incarnation`/`errand`, etc.).** They write audit THEMSELVES inside the domain function via `h.auditW` (not through the Audit middleware). With a FULL-TYPED extraction of the domain function (`CreateTyped`) the audit-emit stays INSIDE it (called on the successful write path, before returning the typed reply) - the huma wrapper does not touch it. **The huma-audit-middleware is NOT needed.** Pilot-1 cadence - belongs to this class (guard `TestHumaCadence_Create_AuditRecorded`: a non-empty payload on 201).
>    - **Middleware-audit domains (audit goes through `apimiddleware.Audit`/StatusRecorder).** The audit entry is written by middleware, reading the status of the response. huma does not give the domain function the `http.ResponseWriter` (typed output), so middleware-audit for these is implemented via **variant B - huma `UseMiddleware`** (PILOT-2 PROVEN, role.create). The reference for this class is **pilot-2** (`POST /v1/roles`, not part of pilot-1).
>
>    **PILOT-2 PROVEN - a solid rollout pattern for middleware-audit domains (verbatim):**
>
>    1. **Spike `hctx.Status()` - native, WITHOUT a fallback wrapper.** `api.UseMiddleware` registers middleware `func(hctx huma.Context, next func(huma.Context))`; huma builds the chain `api.Middlewares().Handler(endpoint)` (`chain.go`) - middleware WRAPS the huma core handler. The core handler ends EVERY response path with `ctx.SetStatus(status)` (success -> `output.Status`; validation failure -> `WriteErr` -> `SetStatus(errStatus)`; domain `StatusError` -> `transformAndWrite` -> `SetStatus`). `humachi.chiContext.SetStatus(code)` writes `code` into the **field** `c.status` AND calls `w.WriteHeader`. `chiContext.Status()` returns `c.status` (a struct FIELD, NOT a read from `http.ResponseWriter`); the same `*chiContext` is threaded through `next` -> AFTER `next(hctx)` `hctx.Status()` == the real final status, even though headers have already been flushed. Early flush does NOT break status reading -> a huma analogue of `StatusRecorder` is NOT required.
>    2. **Skip condition - parity with `apimiddleware.Audit`.** After `next`: `status>=300 || status==0` -> skip (4xx/5xx - the operation did not go through; 0 - panic/an early huma rejection before `SetStatus`); 2xx -> write.
>    3. **Claims propagation.** `RequireJWT` (a chi middleware of the group) puts claims in the request context BEFORE `humachi`; the middleware fetches them via `apimiddleware.ClaimsFromContext(hctx.Context())`. No claims -> warn + skip.
>    4. **Payload propagation handler -> middleware - a mutable carrier.** huma middleware executes OUTSIDE the handler closure (it returns `*output`, and huma.Context becomes unavailable after return), while huma.Context is immutable (`huma.WithValue` creates a new context) - so the classic `SetAuditPayload` (a `*r.WithContext`) is not applicable to huma. Solution: middleware seeds an empty `*HumaAuditCarrier` into the request context via `huma.WithValue(hctx, HumaAuditCarrierKey{}, carrier)` BEFORE `next` and threads the updated hctx; the huma handler closure (`ctx` == this context) puts the payload via `middleware.SetHumaAuditPayload(ctx, ...)`; AFTER `next` the middleware reads the SAME pointer. A shared pointer -> the mutation is visible (parallel to `SetAuditPayload`, but the carrier is seeded upfront, without `*r.WithContext`).
>    5. **Write.** `writer.Write(context.Background(), &audit.Event{EventType:evt, Source:SourceAPI, ArchonAID:claims.Subject, Payload:carrier.Payload})` - a Background context (the request context may be cancelled right after the response), the error is best-effort (parity with `Audit`). `writer==nil` (dev without audit) -> the middleware is transparent.
>    6. **Mount.** One `huma.API` per chi group + one audit event type: `newHumaAuditAPI(r, writer, evt, logger)` = `newHumaCadenceAPI(r)` + `api.UseMiddleware(humaAuditMiddleware(writer, evt, logger))`. The domain function puts the payload in the huma envelope (`registerHumaRole`), not inside `XTyped` (unlike self-audit domains).
>
>    - **The S6-guard invariant for huma.** For every write domain of the rollout - a mandatory audit guard test (E2E through the router), catching the regression "huma swallowed / did not carry the write path through to audit". A repeat of the S6 lesson in huma form. Pilot-2: `TestHumaAudit_RoleCreate_RecordsOnSuccess` (non-empty payload on 201) + `TestHumaAudit_NoAudit_OnReject` (422 -> 0 events) + `TestHumaAudit_NoAudit_OnRBACDeny` (403 -> 0 events).

**OpenAPI viewer (`GET /docs`) + served-spec behind JWT - mechanism A (Amendment 2026-06-15).** On top of the derived huma spec ([§Spec-merge](#spec-merge-key-architectural-question-of-the-rollout): a single huma-dump 3.1) a visual viewer has been raised for the operator; the implementation is in `keeper/internal/api/docs_viewer.go` + `keeper/internal/api/docsassets/`.

- **(a) `GET /docs` - the visual viewer ([RapiDoc](https://rapidocweb.com), go:embed).** A public HTML shell outside `/v1` (no auth). The web-component asset (`rapidoc-min.js`, a single file - RapiDoc keeps its styles in the Shadow DOM, there is no separate CSS) is **embedded via go:embed** (package `docsassets`) and served under `GET /docs/assets/*` - it is NOT pulled from a CDN (air-gapped/closed environments, "security first", offline installs; a single binary carries everything). RapiDoc is distributed under MIT (permissive - includable in the core distribution under BSL 1.1; attribution of transitive dependencies - `rapidoc-min.js.LICENSE.txt` next to the bundle on disk). Mount - `mountDocsViewer(r)` alongside health/meta, OUTSIDE `/v1` -> without the RBAC/audit/maxBody/metrics tack-on (like `/healthz`/`/openapi.yaml`).

- **(b) `GET /openapi.yaml` changed security from public -> behind JWT; `GET /openapi.json` behind the same JWT was added.** Previously the served spec was public (ADR-051(c)/T4c); now it requires a Bearer (the same `RequireJWT` as `/v1`), but WITHOUT the `/v1` tack-on (maxBody/metrics/audit/RBAC): the spec is static, the mount stays OUTSIDE `/v1` (the URL did not change). A security hardening - **no anonymous reconnaissance of the full API surface**. For the inline render of the viewer keeper also serves the same spec as **JSON** (`GET /openapi.json`, the same `buildFullOpenAPISpec` - a single source of truth, just a different format): `RapiDoc.loadSpec` accepts a parsed OBJECT, so the page fetches JSON with a Bearer and feeds the parsed object in. UI-vendor (`openapi-typescript` on soul-stack-web) and `soulctl` are unaffected: they take the **committed** [`docs/keeper/openapi.yaml`](../keeper/openapi.yaml) (a derived snapshot), not the live-served spec.

- **(c) `bearerAuth` securityScheme in the aggregator spec.** The huma aggregator (`HumaFullSpecYAML`) gained `components.securitySchemes.bearerAuth` (`type: http`, `scheme: bearer`, `bearerFormat: JWT`) + global `security: [{bearerAuth: []}]`. These give the viewer a description of the auth mechanism for "Try It" (the same JWT) and explicitly fix in the spec that all `/v1` operations require a Bearer (symmetric with [§Auth](../keeper/operator-api.md#auth)).

- **(d) Mechanism A - rationale.** A browser navigating to a page (opening `/docs`) CANNOT set an `Authorization: Bearer ...` header, and the spec is now behind JWT - so the **shell page is public, the spec is loaded by a separate `fetch` with a Bearer**, which JS adds after the token is entered. The operator pastes the Archon JWT into a field -> `fetch('/openapi.json', {Authorization: 'Bearer ...'})` -> `401` => a message "invalid/expired JWT", `2xx` => inline-render the spec. RapiDoc renders via the **`loadSpec(object)`** method (a parsed JSON of the spec), NOT by passing a spec URL: `loadSpec(string)` is treated by RapiDoc as a spec URL and it fetches it itself WITHOUT our Bearer -> it would hit a `401`. That is why keeper also serves the spec as JSON, and the page feeds `resp.json()` into `loadSpec`. The same JWT is forwarded into `<rapi-doc>` for "Try It" via `setApiKey('bearerAuth', jwt)` - a **BARE jwt with no prefix**: RapiDoc itself adds `'Bearer '` for the http/bearer scheme `bearerAuth`, a manual prefix would produce a double Bearer. Full-text search over operations - built in (`allow-advanced-search`), which is exactly why RapiDoc was chosen. The token is kept in **`sessionStorage`** (per-tab, does not survive closing the tab), NOT in `localStorage`; it is sent as a Bearer header, NOT in the URL (excludes leaking into access logs/referer/history). **Cookies/sessions are NOT introduced** - consistent with [ADR-014](0014-operator-identity.md#adr-014-operator-identity-model-archon) (JWT-bearer is the sole MVP auth method). An unauthenticated visitor to `/docs` sees only an input field - the API surface is not exposed.

- **Route name `/docs`.** An industry-standard viewer name, chosen by the user over dictionary variants (`/scriptorium/codex`) for operator recognizability - a deliberate exception from the "Soul Stack dictionary" (recorded in [naming-rules.md -> Identifiers](../naming-rules.md)).

Amendment to [ADR-011](0011-go-layout.md).

**Spec-merge (key architectural question of the rollout).** huma emits **OpenAPI 3.1.0**; the handwritten `meta/openapi.yaml` is **3.0.3** (a downgrade per ADR-051(e) for the sake of oapi-codegen, which does not support 3.1). During the **transition period** (pilot + start of the rollout) two spec sources coexist: huma fragments of migrated routes (3.1) + the handwritten tail of what is not yet migrated (3.0.3). The pilot does NOT merge the huma fragment into the served spec: `GET /openapi.yaml` still returns the handwritten 3.0.3 (it already describes `POST /v1/cadences`, the form matches - the drift-test is green). **The rollout strategy** (to be fixed as it is proven, in the backlog pending a choice): once the rollout is complete the sole source becomes the huma-dump (3.1) - the handwritten tail disappears along with the oapi scaffold, and merging two sources falls away. During the transition (part of the routes on huma, part handwritten) - option (a) is a make target that merges huma `paths`/`components` into the handwritten spec before embedding (`meta/openapi.yaml`), bumping the header to 3.1; the check target catches a discrepancy between the huma-dump and the committed file. **Header -> 3.1 - a one-time change, on the FIRST merge batch** (when the huma fragment is first merged into the served spec): `openapi: 3.0.3` -> `3.1.0` is edited once, then it stays 3.1 through the end of the rollout. Pilot-1 does NOT merge the huma fragment - `GET /openapi.yaml` still returns the handwritten 3.0.3 (the cadence path is already described there, the form matches, the drift-test is green); the pilot-1 huma-dump lives only in the guard test. **3.1 consequence for web:** `openapi-typescript` on soul-stack-web must digest 3.1 (the spec is currently 3.0.3) - **FLAGGED** for verification at the first merge of a huma fragment into the served spec (not in the pilot slice).

**Pattern (rollout reference - option (b), the thin envelope).** Every write domain is converted to huma following a single scheme (proven by pilot-1 cadence; a verbatim instruction for the rollout):

1. **Typed input.** `xCreateInput struct { Body xCreateHumaBody }` - huma tags (`required`/`enum`/`additionalProperties:false` honest, `doc`) on the fields of `xCreateHumaBody`. huma validates the body from the tags (NOT `RawBody`). Path/query/header parameters - separate fields of the input struct with huma tags.
2. **Extracted domain function.** From `Handler.X(w,r)` a pure `XTyped(ctx, claims, req xCreateRequest) (xCreateReply, error)` is extracted: all business logic (RBAC guards, validation, persist tx, audit-emit, invalidation) - without `http.ResponseWriter`/`*http.Request`. All `problem.Write(w, ...)` become `return zeroReply, &problemError{<problem.Details>}` (a typed wrapper over `problem.Details`, the same type family as `humaProblemError`). RBAC guards return an `error` instead of `(denied bool + Write)`. The old `X(w,r)` REMAINS a thin shell over `XTyped` (decode + claims + `problem.Write` on error) - for the strict bridge and other callers, while oapi is still alive.
3. **Envelope.** The huma handler: typed `input.Body` -> domain model `xCreateRequest` (field-for-field) -> `XTyped(ctx, claims, req)` -> domain `xCreateReply` -> envelope into typed `xCreateOutput` (`Status`/`header:"Location"`/`Body`). The envelope is the only "glue" - both ends are typed.
4. **Typed output.** `xCreateOutput struct { Status int; Location string \`header:"Location"\`; Body xCreateReplyHumaBody }`. Replaces the oapi reply type. omitempty/nullable fields are fixed by a golden-JSON snapshot (a wire-regression guard for the rollout).
5. **huma.Register.** full-typed `func(ctx, *xCreateInput) (*xCreateOutput, error)`. unknown->400 / required->422 - natively huma (§Invariant-2). Self-audit domains - audit inside `XTyped`; middleware-audit domains - variant B (§Audit).

**Pattern "third tier" - PATCH presence via `Optional[T]` (Amendment 2026-06-13, RARE).** Apply ONLY to a PATCH field where an explicit `null` (reset) must be distinguishable from "field omitted" (do not touch) - `*T omitempty` collapses both cases into `nil`. The generic `Optional[T] struct { Set bool; Null bool; Value T }` (`keeper/internal/api/huma_optional.go`) carries presence IN THE TYPE:
- **`UnmarshalJSON`** - the standard json decoder calls the method ONLY when the key is physically present in the body -> `Set=true`; `null` -> `Null=true` (zero `Value`); otherwise `json.Unmarshal` into `Value`. Field omitted -> the method is not called -> `Set=false` (zero).
- **`huma.SchemaProvider` (`Schema(r huma.Registry) *huma.Schema`)** - returns the schema of the wrapped `T` (`r.Schema(reflect.TypeFor[T](), ...)`) with `Nullable:true`. For `Optional[string]` the schema fragment is `{type: [string, null]}` in a plain `application/json` - WITHOUT `application/octet-stream`. The field is NOT in `required` (controlled by the `required:"false"` tag).
- **Envelope.** The huma wrapper maps `Optional[T]` into the domain presence contract: the presence flag (`SetX`) <- `Set`; the value (`*T`) <- `optionalToPtr` (`Set && !Null` -> `&Value`, otherwise `nil`). The domain function treats `(SetX=false)` as "do not touch", `(SetX=true, ptr=nil)` as "reset", `(SetX=true, ptr=&v)` as "set".

**Reference.** `PATCH /v1/roles/{name}/permissions` (`default_scope`, ADR-047 scope-narrowing): `roleUpdatePermissionsHumaBody.DefaultScope Optional[string]` -> `handlers.UpdatePermissionsInput{SetDefaultScope, DefaultScope}` -> `rbac.Service`. `RawBody []byte` + `jsonBodyHasKey` (the former API-side presence bridge) **are removed entirely**.

**`RawBody []byte` for presence - REJECTED a second time.** A recurrence of §-(d) the RawBody bridge: huma unconditionally sets `application/octet-stream` on a `RawBody []byte` field in `requestBody` (instead of `application/json`), which breaks web-codegen (`openapi-typescript`) when merging the spec. `Optional[T]` eliminates the artifact by construction.

**Guard.** A golden OpenAPI fragment asserts the ABSENCE of `application/octet-stream` (`TestHumaRole_PatchPermissions_RequestBody_JSONOnly`, `TestHumaRole_OpenAPIFragment_3_1`); the presence envelope's 3 branches (omitted/null/value) are checked at route level through a real PATCH route down to the domain floor (`TestHumaRole_UpdatePermissions_ScopePresence`, SECURITY-relevant - RBAC scope-narrowing); `UnmarshalJSON`/`Schema`/round-trip - `huma_optional_test.go`.

**Tier RARE.** As of 2026-06-13 the sole bearer is `default_scope` in PATCH-permissions. Other PATCH routes (cadence/service/synod/incarnation-hosts) - read-modify-write `*T omitempty` (nil = do not touch), presence is NOT needed. Do not proliferate `Optional[T]` where the field's semantics have no explicit reset.

**Pattern "fourth tier" - read-with-typed-query (Amendment 2026-06-13, decision A).** A READ-list endpoint with query filters is converted to huma full-typed with a typed query. Reference - `GET /v1/audit` (`keeper/internal/api/huma_audit_endpoint{,_op}.go`); rollout - ~13-15 list endpoints:

1. **Typed query-input.** `xListInput struct { ... }` - EVERY filter field carries a `query:"<name>"` tag; huma binds from `url.Values` and validates against the schema built from the tags. The GET has NO RequestBody (huma does not emit a body without a `Body` field; a guard asserts the absence of `requestBody`). Types: `[]string` (multi-value `?k=a&k=b`, OR semantics in the domain function, MANDATORY `,explode` - see §1b), `string` (exact), `time.Time` (huma's date-time bind), `int32` (offset/limit - see §1c).
   - **§1b. Multi-value `[]string` query -> `query:"<name>,explode"` (rollout rule).** huma's default for a query array is `explode=false`: it reads a comma-separated `?k=a,b` AS ONE `"a,b"` (takes only the first value of the key + `strings.Split`, huma v2.38 `huma.go:1724-1729`) AND emits `explode: false` into the spec. This breaks the OR filter: a repeated key `?source=a&source=b` loses everything except `a`, and a generated client encodes the list as a compound value. `,explode` in the query tag -> `pfi.Explode=true` (`huma.go:157`): huma reads all repeated keys `Query()[name]` AND emits `explode: true` (`openapi.go` the `Param.Explode` pointer), matching the committed spec's `style:form` + `explode:true`. Apply to ALL multi-value `[]string` fields of list filters (`type`/`source`/...) - no dialects.
   - **§1c. Pagination - `int32`, NOT Go `int`.** huma on `reflect.Int` (a 64-bit platform) emits `format:int64` (`schema.go:804-810`), while the committed spec (`OffsetQuery`/`LimitQuery`) and the ADR text carry `int32`; pagination (offset, limit<=1000) fits into an int32. Input fields - `int32` -> huma emits `format:int32` (matching the committed spec, minimal drift). Conversion into the domain filter (`int`) - a widening `int(in.Offset)` without loss. Apply to ALL pagination fields of the rollout.
2. **enum - `enum:"v1,v2,..."` tag on a `[]string` field = the FULL domain valid-set (a rollout rule, corrected by the Amendment 2026-06-13).** huma rejects a value outside the set with **422** (schema validation). The enum set = the domain valid-set (for audit's `source` - `audit.Source.Valid()`: `signal,api,mcp,keeper_internal,soul_grpc,background,config_bootstrap` - ALL seven). **Code-first: the Go domain is the source of truth, the committed spec is synced WITH THE DOMAIN, not the domain fitted to the spec.** `config_bootstrap` is really emitted (`push/auto_import.go`) and accepted by the domain; omitting it from the enum tag would reject a working filter `?source=config_bootstrap` with **422** (a wire regression 200->422 versus the legacy strict server, which did NOT validate enums on bind). The committed `openapi.yaml`/`meta` source-enum were stale (drift from the domain) - aligned to the domain. The guard `TestHumaAudit_ConfigBootstrapSource_200` pins 200 (NOT 422).
3. **Extracted domain `XListTyped(ctx, filter) (reply, error)`.** Pure read logic without `http.ResponseWriter`/`*http.Request`: domain validation (enum defense-in-depth -> 422), filtering, page reads, projection into the typed envelope. The old `X(w,r)` remains a thin shell (parses the query -> the same domain filter -> `XListTyped`).
4. **Typed envelope-output.** `Body` = the same list envelope `{items, offset, limit, total}` that legacy `writeJSON` returned - a golden-JSON byte-exact match to legacy (the tier's main guard; `items` non-nil `[]`).
5. **CONTRACT 400/422 - the tier invariant (decision A).** The override (`newHumaProblemError`) is extended with an ADDITIVE branch `hasQueryParseError`: a `*huma.ErrorDetail` whose `strings.HasPrefix(Location, "query.")` IS true AND whose Message is from the parse set of huma's `parseInto` (`invalid date/time` PREFIX / `invalid integer` / `invalid boolean` / `invalid float`, huma v2.38 `huma.go:1685-1769`) -> **400**. An enum mismatch (schema-validate, Message `expected value to be one of ...`, the same `query.` Location) does NOT fall into the set -> the default **422** branch. **The discriminator parse-vs-enum is precisely the Message literal, not Location** (both are `query.`).
6. **Pagination boundaries - WITHOUT huma `minimum`/`maximum`.** `offset`/`limit` (type `int32`, see §1c) carry only `default` (0/50, matching `api.ParsePage`); the range (offset>=0, limit in [1,1000]) is deliberately NOT expressed as a schema min/max - otherwise huma would return an out-of-range value as **422**, whereas the legacy/strict contract gives **400** (`ParsePage`). The range is enforced by the domain `XListTyped` via the shared `api.CheckPageBounds` (the same source of bounds and messages as `ParsePage`) -> 400. Otherwise there would be a wire change on `limit=0/1001/offset<0`. (The committed `OffsetQuery`/`LimitQuery` carry `minimum`/`maximum` only as schema documentation - the huma input does not repeat them, enforcement is domain-side.)
7. **zero-time -> no-filter.** An omitted date-time parameter is left by huma as a zero-value `time.Time` (parseInto is not called) -> the domain function treats `IsZero` as "no time bound" (parity with legacy `if param != ""`).
8. **READ - without audit-middleware** (reading itself does not produce an audit event; for audit_log there would also be recursion).

**Coupling on huma Message literals.** `queryParseMessages` - a copy of the exact strings from huma's `parseInto`; check against it on a huma bump. The guard set (especially the KEY `BadSource_422`) catches a regression in the detection on a desync.

**Guard set (TDD).** `huma_audit_endpoint_test.go`: `BadStartedAfter_400` (bad date-time -> 400), `BadOffset_400`/`BadLimit_400` (bad int -> 400), `OutOfRangePagination_400` (limit=0/1001/offset<0 -> 400, parity with ParsePage), **`BadSource_422`** (bad enum -> 422 - KEY, proves the parse detector does NOT catch an enum), **`ConfigBootstrapSource_200`** (a valid config_bootstrap source -> 200, NOT 422 - the enum tag is the domain valid-set, a MAJOR-regression guard), `ValidFilters_200`/`DefaultPagination_200` (200 + a golden envelope byte-exact match, bounds matching ParsePage), `OpenAPIFragment_3_1` (typed query params date-time/int32/enum-including-config_bootstrap; `explode: true` on multi-value; no `int64`; no requestBody on the GET).

**Rationale.**
- **A single source of form - Go types.** Eliminates the handwritten spec + committed gen + a drift-test between them: the spec is derived, a desync is impossible by construction.
- **Less ceremony.** The oapi-strict bridge glue (`bridgeMiddleware`/`strictServer`/the 15-argument constructor) is gone - huma validates the typed `Body` from tags and calls the extracted domain function (`XTyped`) through a thin envelope, without `humachi.Unwrap`/the `RawBody` bridge and without 900 KB of generated code.
- **Security does not suffer.** Manual mounting into the chi group with the tack-on + the global problem override + per-domain audit guards preserve RBAC/Audit/problem+json. Proven by the pilot.

**Rejected alternatives.**
- **(a) Stay on spec-first (ADR-051).** The de facto source of form became the Go handler code (the bridge holds the wire form and strictness), the spec was a parallel manuscript with drift risk + 900 KB generated. Rejected by the user - the reversal to code-first.
- **(b) Full-typed huma for existing domains (typed input/output + an extracted domain function).** Initially rejected as disproportionate (a repeat of Phase 1: re-wiring + re-proving date-time/omitempty per domain). **ACCEPTED by Amendment 2026-06-12** (a user decision): the one-time cost is justified - a single source of form (a Go struct), an honest spec, zero domain `DisallowUnknownFields`/bridge glue, golden-JSON mitigates the wire regression. Became the rollout reference (§Pattern).
- **(d) Delegation via a `RawBody` bridge (the former pilot-1 pattern).** The huma operation carries `RawBody []byte`, lifts `additionalProperties:false`, delegates to the domain `func(w,r)` (strictness held by the domain's `DisallowUnknownFields`). **REJECTED by Amendment 2026-06-12** in favor of (b): the bridge left two points of truth for the form (the huma schema + domain decode), required `humachi.Unwrap`/a context-bridge middleware, and gave no typed output. FULL-TYPED is simpler by construction and eliminates the schema<->decode desync. **Rejected a second time by Amendment 2026-06-13** as a PATCH-presence mechanism: huma unconditionally sets `application/octet-stream` on a `RawBody []byte` field in `requestBody` -> breaks web-codegen when merging; presence is carried by `Optional[T]` in the type (§Pattern "third tier").
- **(c) huma's auto-router (`humachi.New` + auto-mount without a chi group).** Would have hung operations bypassing per-route RBAC/Audit (a security regression, like `HandlerFromMux` in ADR-051). Rejected - manual mounting into the chi group with the tack-on.
- **(g) huma's `ErrorModel` as the error format.** Would have diverged from the cluster's problem+json contract. Rejected - a global `huma.NewError` override to `problem.*`.
