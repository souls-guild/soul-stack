# ADR-004. Binary Layout ‚Äî `keeper`, `soul`, `soul-lint`; push mode as a module inside `keeper`

- **Context.** The "security first" principle demands a minimal attack surface on the agent. Souls run on hosts owned by others, and any Keeper code that ends up in their binary widens the attack surface. In parallel, we need a tool that validates Destiny and Essence correctness locally (without Keeper, in CI, on a developer machine). We also need the ability to manage hosts without an agent (push model).
- **Decision.** Four separate binaries (`soul-trial` was added by [ADR-023](0023-trial-test-runner.md#adr-023-test-runner-trial-soul-trial-and-dsl-coverage); the heading anchor was left unchanged for cross-link stability):
  - `keeper` ‚Äî the central server. The system's control node. Includes the `keeper.push` module for SSH delivery to hosts without a Soul agent (see the "Push mode" section).
  - `soul` ‚Äî the agent daemon on a managed host (runs as a service).
  - `soul-lint` ‚Äî the standard linter for Destiny and Essence: parsing, rendering, schema validation, static analysis. Works offline, without a connection to Keeper.
  - `soul-trial` ‚Äî the offline test runner (**Trial**, [ADR-023](0023-trial-test-runner.md#adr-023-test-runner-trial-soul-trial-and-dsl-coverage)) for Destiny/Scenario and state_schema migrations. Like `soul-lint`, it works offline without a connection to Keeper. **Boundary with `soul-lint`:** `soul-lint` is pure static analysis (does not execute); `soul-trial` **runs** (executes the render pipeline / migration DSL against fixtures and asserts the result; L0/L1 are hermetic in-process, L2/L3 are a dynamic run on ephemeral hosts via docker, post-MVP). The dividing line: "does not execute ‚Üí soul-lint, executes ‚Üí soul-trial". The `soul-` prefix means "offline authoring-cycle tool" (like `soul-lint`), not a Soul role/agent ‚Äî server-side code in it is acceptable (see ADR-011: the code lives in the `keeper` module).

  No **mixing of Keeper and Soul roles** in one binary: Keeper server-side code does not end up in `soul`, and vice versa. A subcommand structure **within a single role** is acceptable and used for lifecycle operations (`soul init`/`run`/`apply`/`help`, `keeper init`/`run`/`help`). The prohibition applies to roles, not subcommands: `soul apply` is the push-oneshot mode of the same Soul role (stdin = rendered `ApplyRequest` protojson; stdout = NDJSON `TaskEvent`+`RunResult`; exit 0 on `success`, 1 otherwise), a single proto contract shared with pull (ADR-012); the identity of a push host is the SSH target + a `souls.transport=ssh` record, without mTLS/SoulSeed.
- **Rationale.** A minimal `soul`: only what the agent needs, no Keeper server-side code, no MCP server, no OpenAPI facade, no Vault server-side integration. `soul-lint` is separated from Keeper so validation works in CI/IDE/locally without network access and without Keeper access rights. Push mode is a server-side function (executed on behalf of an operator, requires Keeper RBAC and audit), so it belongs in `keeper`, not in a separate client binary. The operator's primary interface to Keeper is **OpenAPI and MCP**; an administrative CLI is acceptable as a thin wrapper over the API, but not as the primary path.
- **Trade-off.** A bit more work in the build/release pipeline (four artifacts), and some code will need to be carefully factored out into shared packages. We accept this as the cost of security and developer convenience. `soul-trial` physically lives in the `keeper/` Go module (`keeper/cmd/soul-trial`, [ADR-011](0011-go-layout.md#adr-011-go-code-layout-gowork-with-modules-per-side)) ‚Äî it needs direct access to `keeper/internal/render`/`scenario`/the migration engine as its assertion target; this is a deliberate divergence between "code module (keeper)" and "tool role (offline, alongside soul-lint)". It does not break Soul isolation (`soul-trial` is not a Soul agent).
