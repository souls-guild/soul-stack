# ADR-004. Раскладка бинарей — `keeper`, `soul`, `soul-lint`; push-режим — модуль внутри `keeper`

- **Контекст.** Тезис «безопасность на первом месте» требует минимальной поверхности атаки на агенте. Souls стоят на чужих хостах, и любой код Keeper, попавший в их бинарь, расширяет атакующую поверхность. Параллельно нужен инструмент, который локально (без Keeper, в CI, на машине разработчика) проверяет корректность Destiny и Essence. Также нужна возможность управлять хостами без агента (push-модель).
- **Решение.** Четыре отдельных бинаря (`soul-trial` добавлен [ADR-023](0023-trial-test-runner.md#adr-023-тест-раннер-trial-soul-trial-и-dsl-coverage); заголовок-якорь оставлен прежним ради стабильности cross-link-ов):
  - `keeper` — центральный сервер. Управляющий узел системы. Включает в себя модуль `keeper.push` для SSH-доставки на хосты без Soul-агента (см. раздел «Push-режим»).
  - `soul` — демон-агент на управляемом хосте (запускается как служба).
  - `soul-lint` — стандартный линтер для Destiny и Essence: разбор, рендер, проверка по схеме, статический анализ. Работает офлайн, без подключения к Keeper.
  - `soul-trial` — офлайн-раннер испытаний (**Trial**, [ADR-023](0023-trial-test-runner.md#adr-023-тест-раннер-trial-soul-trial-и-dsl-coverage)) Destiny/Scenario и state_schema-миграций. Как и `soul-lint`, работает офлайн без подключения к Keeper. **Граница с `soul-lint`:** `soul-lint` — чистая статика (не исполняет); `soul-trial` — **прогон** (исполняет render-пайплайн / миграционный DSL на fixtures и ассертит результат; L0/L1 герметично in-process, L2/L3 — динамический прогон на эфемерных хостах через docker, post-MVP). Линия раздела: «не исполняет → soul-lint, исполняет → soul-trial». Префикс `soul-` = «офлайн-инструмент авторского цикла» (как `soul-lint`), а не Soul-роль/агент — серверный код в нём допустим (см. ADR-011: код живёт в модуле `keeper`).

  Никакого **смешения ролей Keeper и Soul** в одном бинаре: серверный код Keeper не попадает в `soul`, и наоборот. Subcommand-структура **внутри одной роли** допустима и используется для операций жизненного цикла (`soul init`/`run`/`apply`/`help`, `keeper init`/`run`/`help`). Запрет касается ролей, а не подкоманд: `soul apply` — push-oneshot режим той же Soul-роли (stdin = отрендеренный `ApplyRequest` protojson; stdout = NDJSON `TaskEvent`+`RunResult`; exit 0 при `success`, 1 иначе), единый proto-контракт с pull (ADR-012); идентичность push-хоста — SSH-target + запись `souls.transport=ssh`, без mTLS/SoulSeed.
- **Обоснование.** Минимальный `soul`: только то, что нужно агенту, никакого серверного кода Keeper, MCP-сервера, OpenAPI-фасада и Vault-серверной интеграции. `soul-lint` отделён от Keeper, чтобы валидация работала в CI/IDE/локально без сети и без прав доступа к Keeper. Push-режим — это серверная функция (выполняется от лица оператора, требует RBAC и аудита Keeper), его место — в `keeper`, а не в отдельном клиентском бинаре. Основной интерфейс оператора к Keeper — **OpenAPI и MCP**; CLI для администрирования допустим как тонкая обёртка над API, но не как первичный путь.
- **Trade-off.** Чуть больше работы в build/release-пайплайне (четыре артефакта), часть кода придётся аккуратно вынести в общие пакеты. Принимаем как стоимость безопасности и удобства разработчика. `soul-trial` физически живёт в Go-модуле `keeper/` (`keeper/cmd/soul-trial`, [ADR-011](0011-go-layout.md#adr-011-раскладка-go-кода-gowork-с-модулями-по-сторонам)) — ему нужен прямой доступ к `keeper/internal/render`/`scenario`/migration-движку как к цели ассерта; это сознательное расхождение «модуль кода (keeper) vs роль инструмента (офлайн, рядом с soul-lint)». Изоляцию Soul не нарушает (`soul-trial` — не Soul-агент).
- **Amendment 2026-05-26 (push Variant C):** Кроме исходного push-режима «Keeper доставляет soul-бинарь + module-кеш через SSH», добавлен ad-hoc multi-host orchestrator: `POST /v1/push/apply` принимает inventory_sids[]+destiny_ref+input и выкатывает destiny на несколько хостов параллельно через SSH (без incarnation/state/scenario). См. [ADR-032](0032-push-orchestrator.md#adr-032-push-orchestrator-variant-c--multi-host-destiny-push-без-incarnationscenario).
