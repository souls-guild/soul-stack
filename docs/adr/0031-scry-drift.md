# ADR-031. Scry — drift-detection (declarative dry-run reconcile)

> **Status: design accepted, implementation in progress (on-demand pilot).** This ADR fixes the normative design of the **Scry** subsystem, the machine-readable `Plan` contract, and only-add proto fields; the implementation lands in slices (on-demand pilot → background scan). Closes [open Q No. 15](../architecture.md#open-questions) (drift-detection). The name **Scry** was chosen by the user (propose-and-wait passed, entered into [naming-rules.md](../naming-rules.md)).

- **Context.** Soul Stack is batch-declarative, but has no circuit for "has the host's real state drifted from the declaration." Between runs, an operator makes manual changes / another system does / a service rewrites its own config on its own — and **nobody finds out** until the next apply (see [Atomicity and `error_locked`](../architecture.md#atomicity-and-error_locked): drift accumulates silently). [Open Q No. 15](../architecture.md#open-questions) deferred this circuit until a concrete request; the request has now appeared. A cheap way is needed to answer "would even one resource re-apply if reconcile ran right now."
- **What drift is (a normative definition).** **drift = a dry-run reconcile would show `changed=true` on at least one resource**, among **declared** resources. This is **declarative drift**, and **NOT** a full dump of host state — the system has no such full dump and isn't meant to. Scry catches a mismatch **only for what is declared** in the scenario / `incarnation.state`; unmanaged host resources outside the declaration are invisible to Scry (by design — there's no "how it should be" source for them).
- **Decision — the Scry subsystem.**
  - **(a) The entity.** **Scry** — a read-only drift-detection subsystem. **MVP — on-demand**: the operator triggers a check via `keeper.incarnation.check-drift` (REST/MCP), Keeper renders a plan as for a normal apply, but in **dry-run mode**, collects per-resource `changed`, and forms a report. **Then — a background scan** (a Reaper-like leader periodically checks incarnations). Staging: **on-demand pilot → background** (the background scan isn't turned on until the on-demand path is proven).
  - **(b) The `Plan` contract → pure-read.** The `Plan` RPC in `SoulModule` ([ADR-020](0020-plugin-infrastructure.md#adr-020-plugin-infrastructure-manifest-format-handshake-lifecycle), methods `Validate`/`Plan`/`Apply`) is currently **a no-op in every module**. This ADR makes `Plan` **pure-read**: the module reads the resource's real state, compares it to the desired one, and **returns a machine-readable `changed`** (with no host mutation). This amends `Plan`'s semantics for all 17 core modules ([ADR-015](0015-core-modules-mvp.md#adr-015-core-modules-mvp-exact-list)).
  - **(c) only-add proto.** In `proto/plugin/v1` (the plugin SDK) — **`PlanEvent.changed`** (a machine flag for a per-resource mismatch). In the keeper↔soul transport — **`ApplyRequest.dry_run`** ([ADR-012(c)](0012-keeper-soul-grpc.md#adr-012-keepersoul-grpc-contract-one-eventstream-with-oneof-keeper-side-render-forward-compat-only-add) forward-compat only-add): with `dry_run=true`, Soul calls `Plan` instead of `Apply` (the same task plan, a read-only pass). Both are only-add, never reusing field numbers.
  - **(d) The `drift` status — informational, NOT blocking.** The incarnation status **`drift`** is added to `ValidStatus` + a CHECK migration (removes the "post-MVP" mark from the [status table](../architecture.md#atomicity-and-error_locked); `provisioning` stays post-MVP). `drift` is **informational** — it **does NOT block remediation** (unlike `error_locked`/`destroy_failed`): remediating drift = a **normal apply** from the `drift` state (run the scenario → the host converges to the declaration → status `ready`). It signals "there's a mismatch," it doesn't lock the incarnation.
  - **(e) The mismatch report — `DriftReport`.** A name alongside `RunResult` / `SoulprintReport` ([naming-rules → Proto Keeper↔Soul Messages](../naming-rules.md#messages-proto-keepersoul)): an aggregated report of "which resources on which hosts would show `changed=true`."
  - **(f) Security — default-deny for community modules.** A community `soul_module` **without a read-safe capability** → **drift is not invoked** (`Plan` isn't run, default-deny). Reason: `Plan` is declared pure-read, but for a third-party plugin this can't be guaranteed without an explicit declaration — an undeclared plugin isn't poked in dry-run mode (fail-closed). Core modules are read-safe by construction.
- **Amends.**
  - [ADR-015](0015-core-modules-mvp.md#adr-015-core-modules-mvp-exact-list) — Plan semantics across all core modules: **no-op → pure-read** with a machine-readable `changed`.
  - [ADR-012](0012-keeper-soul-grpc.md#adr-012-keepersoul-grpc-contract-one-eventstream-with-oneof-keeper-side-render-forward-compat-only-add) / [ADR-020](0020-plugin-infrastructure.md#adr-020-plugin-infrastructure-manifest-format-handshake-lifecycle) — only-add `PlanEvent.changed` (plugin SDK) + `ApplyRequest.dry_run` (keeper↔soul).
  - [ADR-027](0027-apply-work-queue.md#adr-027-apply-execution-model--work-queue--claim-acolyte-pool-ward-claim) — a drift check goes through the same **work-queue in Plan mode** (Acolyte claims the task, renders from the recipe, runs `Plan` instead of `Apply`); no second execution circuit.
  - **The incarnation status machine** — `drift` is added to `ValidStatus` + a CHECK migration (see (d)); removes the post-MVP mark on `drift` from the [status table](../architecture.md#atomicity-and-error_locked).
- **Trade-offs.**
  - **Declarative drift, not a full dump** — Scry sees a mismatch only for declared resources; unmanaged host changes are invisible. Accepted: there is no full "how it should be" for unmanaged resources, trying to compare everything = noise + false positives.
  - **Plan pure-read depends on the module's honesty** — for core we guarantee this by code; for community — default-deny (see (f)). Cost — a community plugin without a read-safe declaration is silently skipped in the drift scan (doesn't break the scan, but isn't covered either). Accepted.
  - **`drift` is informational, not blocking** — we trade away "hard protection against drifting" so that the drift status doesn't lock the incarnation and remediation runs as a normal apply. If "forbid operations on a drift-incarnation" is needed later — a separate decision.
- **Implementation slices.** **On-demand pilot** (`keeper.incarnation.check-drift` + `Plan` pure-read in core + only-add proto + the `drift` status + `DriftReport`) → **background scan** (a Reaper-like leader).
- **Amendment (Slice B implemented).** The on-demand pilot is fully committed:
  - **scenario-kind `converge`** ([naming-rules → Scenario kinds](../naming-rules.md)). A service supporting drift-detection MUST place a `scenario/converge/main.yml` in the repo. Discovery is auto-discover by name — no new YAML field; the file's absence → check-drift returns 422 "drift check unavailable" (`ErrConvergeMissing`, informational), the incarnation is untouched.
  - **converge-input auto-resolve from `incarnation.state`.** The "by name" convention ([`scenario.resolveDriftInput`](../../keeper/internal/scenario/checkdrift.go)): for every converge-schema parameter, the value is taken from the override body → `incarnation.state.<name>` → the schema default; a required field with no source → 422 `ErrDriftInputMissing`. This gives "auto-drift from state" without the operator manually passing parameters.
  - **REST + MCP.** `POST /v1/incarnations/{name}/check-drift` ([operator-api.md](../keeper/operator-api.md)) + `keeper.incarnation.check-drift` ([mcp-tools.md](../keeper/mcp-tools.md)). **A sync operation**: the handler blocks until the `DriftReport` is assembled and returns it in a 200 response. **`DriftReport`** — a keeper-internal aggregated view + an API/MCP response (NOT a Keeper↔Soul proto message): `{checked_at, incarnation, scenario_ref, hosts:[{sid, status, tasks:[{idx, module, action, changed, message}]}], summary:{hosts_drifted, hosts_clean, hosts_unsupported, hosts_failed}}`. Per-host `status` ∈ `clean`/`drifted`/`unsupported`/`failed`.
  - **The Acolyte flow with DryRun threaded through.** check-drift goes through the same work-queue as a normal run ([ADR-027](0027-apply-work-queue.md#adr-027-apply-execution-model--work-queue--claim-acolyte-pool-ward-claim)): `Recipe.DryRun=true` (a forward-compat field in `apply_runs.recipe` jsonb) → Acolyte builds `ApplyRequest{dry_run:true}` ([claim.go](../../keeper/internal/scenario/claim.go)). The inline path (acolytes=0) does **not** support check-drift (fail-closed: dry_run isn't threaded through).
  - **A status-machine amendment.** Migration `047_incarnation_status_drift` added `drift` to the CHECK `incarnation_status_valid`. A normal apply (`scenario.lockRun`) and an upgrade (`incarnation.PrepareUpgrade`) accept `drift` as a starting status on par with `ready` — drift does NOT block. Destroy also allows `drift` (`canDestroyFrom`).
  - **RBAC.** A new permission `incarnation.check-drift` ([rbac.md](../keeper/rbac.md)) with the same per-coven/service scope as `incarnation.run` (`IncarnationCovenContexts`).
  - **Audit.** A new event `incarnation.drift_checked` (`source=api`/`mcp`, `correlation_id=apply_id`, payload `{name, scenario, apply_id, drift_summary}`) — written by the handler after the report is assembled.
  - **Default-deny security confirmed.** A `plan.unsupported` task (a community module without the `PlanReadSafe` capability) on a host → `DriftStatusUnsupported` (NOT `failed`): the report separates "not supported" from a real Plan error.
  - **Lock strategy (on-demand pilot, a deliberate choice).** CheckDrift does NOT take a `lockRun`-equivalent on the incarnation: the window between `InsertPlanned` and the terminal Acolyte-apply theoretically overlaps with a parallel `upgrade`/`destroy` on the same incarnation. **Accepted as safe-by-design for the on-demand pilot:** (a) check-drift is an operator-initiated read-only operation; operators don't run it in parallel with destructive commands on the same incarnation. (b) `MarkDriftStatus` uses `WHERE status IN ('ready','drift')` — a late write while `status=applying`/`destroying` is a no-op (consistency). (c) On the Soul side, `dry_run` ≠ `Apply` (a Slice A structural guarantee): a parallel dry_run-flight + a real-apply-flight on the same host don't semantically conflict (dry_run is pure-read). **A full lock** (serializing check-drift with upgrade/destroy via `lockRun` or a drift-aware Acolyte-claim gate + a persistent task_idx→module map) — a **Slice C** task (the background automatic scan by a Reaper-like leader, where concurrency with other operations becomes real).
  - **Not part of Slice B.** Rolling Plan pure-read out to the remaining 14 core modules (3 pilots ready: pkg/file/service from Slice A); the background periodic scan (Slice C — see the lock strategy above); a per-VM token in the bootstrap flow. These are future slices.
- **Amendment (2026-05-26, the Plan pure-read rollout from Slice A applied to the other 14 core modules, commit `c2d8181`).** The Slice A pattern (`PlanReadSafe` capability + a pure-read `Plan` implementation) is applied to the remaining 14 Soul-side core modules. Total: 12 stateful modules with drift-detection + 5 without coverage (2 deferred, 3 verbs not applicable).
  - **9 modules with `PlanReadSafe` (full drift-detection).** `core.group` / `core.user` / `core.cron` / `core.mount` / `core.sysctl` / `core.archive` / `core.line` / `core.repo` / `core.firewall`. All follow the Slice A pattern exactly: a read-only probe of the resource → comparison to the desired state → a machine-readable `PlanEvent.changed` with no host mutation.
  - **2 modules WITHOUT `PlanReadSafe` (deferred to Slice C).** **`core.git`** — the `pulled` state form requires a network `git fetch` to the remote to determine the mismatch between local HEAD and the remote tip; there's no pure-read in the Apply flow (`fetch` mutates local refs). **`core.url` without a checksum** — the mismatch can only be determined via a HEAD request (Content-Length / ETag), which turns Plan into a network call; without a declared `checksum:` (sha256/sha1) there's no safe read-only probe. In Slice C: add a read-only mode for `core.git` (`git ls-remote` with no mutation) and accept a checksum-only Plan for `core.url` (without a checksum — `plan.unsupported`, like a community plugin without read-safe).
  - **3 verb modules WITHOUT `PlanReadSafe` (correct default-deny).** `core.exec.run` / `core.cmd.shell` / `core.http.probe` — verb modules **have no desired state** (there's no notion of a "declared result" for exec/cmd/http-probe; they're imperative / a read-probe by construction). `plan.unsupported` for them is correct behavior, not a coverage gap. In `DriftReport`, the per-host `status` for these tasks is `unsupported`, separating "the module doesn't apply to drift" from a real Plan error.
  - **Total drift-detection coverage:** 12 stateful core modules (3 Slice A pilots: pkg/file/service + the current 9) with an honest `changed` signal; 2 deferred to Slice C (`git`/`url`); 3 verb modules correctly return `unsupported`. **17 − 5 = 12** stateful modules covered.
  - **What is NOT part of this rollout.** The Slice C layer (a background periodic scan by a Reaper-like leader + a full lock strategy with upgrade/destroy + drift for `git`/`url`) — a separate slice, not done by this amendment.
- **Slice C status (2026-05-26):** The background scan is implemented, default OFF (`reaper.scry_background.enabled=false`). The contract:
  - The `scry_background` Reaper rule (keeper/internal/reaper/scry.go) scans candidates by the predicate `last_drift_check_at IS NULL OR < NOW() - interval` with a keyset iterator.
  - Each candidate: a validate-tx (TX1 lock → InsertPlanned dry_run scenario → COMMIT) with throttling via `CountActiveDryRuns()` (max_concurrent=10 by default).
  - The columns `incarnation.last_drift_check_at`, `last_drift_summary` (jsonb) were added by migration 050.
  - Default OFF — a guard rail: enabling it in prod is an explicit decision (the check-drift load multiplies by the number of incarnations).
  - Audit-source — `SourceBackground` (shared/audit/source.go).
- **Amendment (2026-06-10, converge → an operational scenario-kind).** The `converge` scenario is moved out of the special `LifecycleScenarioNames` set (only `create`/`destroy` remain there). Consequences: (1) `converge` is marked `kind=operational` in the `GET /v1/services/{name}/scenarios` catalog and becomes runnable via a normal `run` (Apply mode: actually converges hosts to the declared state, a changed/no-changes report; also run by Cadence like any operational scenario). (2) The drift-check function does NOT change: `check-drift` loads `scenario/converge/main.yml` via the name constant `ConvergeScenarioName` (auto-discover), not via membership in the lifecycle set; `ErrConvergeMissing`, the background Scry scan, MCP `incarnation.check-drift`, and audit `incarnation.drift_checked` work unchanged. So `converge` carries a dual role: as an operational `run` — an Apply-reconcile; as a dry-run target for `check-drift` — declarative drift-detection. Only the treatment of `converge` as a lifecycle-kind leaves ADR-031, not the drift circuit itself.
- **Amendment (2026-06-27, upgrade → drift as a drift source).** `keeper.incarnation.upgrade` now, on a successful upgrade transaction, puts the incarnation into **`status = drift`** (rather than `ready`): the migration changed `incarnation.state`/versions in the DB, but the hosts stayed on the old rollout — a state↔fact mismatch that used to accumulate silently until the next apply. This is a **second source of the `drift` status** alongside the Scry scan and on-demand `check-drift`; the status semantics are exactly (d): drift is **informational, not blocking**, remediation = a **normal apply** (`drift → ready`). Upgrade-drift is distinguished from Scry-drift in triage by a separate zero-diff `state_history` record with `scenario: upgrade-pending-apply`. The drift circuit itself (Plan pure-read, `DriftReport`, the background Scry scan) is unchanged — only one more way to enter the `drift` status is added. The normative record of the upgrade transition — [ADR-019 Amendment](0019-state-migration-dsl.md#amendment-upgrade--drift-the-final-status-2026-06-27); implementation — [`keeper/internal/incarnation/crud.go`](../../keeper/internal/incarnation/crud.go).
