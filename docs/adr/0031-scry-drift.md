# ADR-031. Scry — drift-detection (declarative dry-run reconcile)

> **Статус: дизайн принят, реализация в работе (on-demand-пилот).** Этот ADR фиксирует нормативный дизайн подсистемы **Scry**, машинный контракт `Plan` и only-add proto-поля; имплементация идёт срезами (on-demand-пилот → фоновый скан). Закрывает [open Q №15](../architecture.md#открытые-вопросы) (drift-detection). Имя **Scry** выбрано пользователем (propose-and-wait пройден, внесено в [naming-rules.md](../naming-rules.md)).

- **Контекст.** Soul Stack — пакетный декларативный, но без контура «расходится ли реальное состояние хоста с декларацией». Между прогонами оператор внёс изменения руками / другой системой / сервис самовольно переписал конфиг — и об этом **никто не узнает** до следующего apply (см. [Атомарность и `error_locked`](../architecture.md#атомарность-и-error_locked): «в Ansible drift копится тихо»). [open Q №15](../architecture.md#открытые-вопросы) откладывал контур до конкретного запроса; запрос появился. Нужен дешёвый способ ответить «применился бы хоть один ресурс заново, если прогнать reconcile прямо сейчас».
- **Что такое drift (нормативное определение).** **drift = dry-run reconcile показал бы `changed=true` хотя бы на одном ресурсе** по **декларированным** ресурсам. Это **declarative drift** (как Salt `test=True` / Ansible `--check`), а **НЕ** полный дамп состояния хоста — полного дампа у системы нет и не предполагается. Scry ловит расхождение **только по тому, что объявлено** в scenario / `incarnation.state`; неуправляемые ресурсы хоста вне декларации Scry не видит (by design — нет источника «как должно быть» для них).
- **Решение — подсистема Scry.**
  - **(a) Сущность.** **Scry** — read-only подсистема drift-детекта. **MVP — on-demand**: оператор запускает проверку командой `keeper.incarnation.check-drift` (REST/MCP), Keeper рендерит план как для обычного apply, но в **dry-run-режиме**, собирает per-ресурсный `changed` и формирует отчёт. **Затем — фоновый скан** (Reaper-подобный лидер периодически проверяет incarnation-ы). Этапность: **on-demand-пилот → фон** (фон не запускается, пока on-demand-путь не отлажен).
  - **(b) Контракт `Plan` → pure-read.** RPC `Plan` в `SoulModule` ([ADR-020](0020-plugin-infrastructure.md#adr-020-plugin-инфраструктура-формат-manifest-handshake-lifecycle), методы `Validate`/`Plan`/`Apply`) сейчас **no-op во всех модулях**. Этим ADR `Plan` становится **pure-read**: модуль читает реальное состояние ресурса, сравнивает с желаемым и **возвращает машинный `changed`** (без мутаций хоста). Это амендит семантику `Plan` для всех 17 core-модулей ([ADR-015](0015-core-modules-mvp.md#adr-015-core-модули-mvp-точный-список)).
  - **(c) only-add proto.** В `proto/plugin/v1` (плагин-SDK) — **`PlanEvent.changed`** (машинный флаг расхождения по ресурсу). В транспорте keeper↔soul — **`ApplyRequest.dry_run`** ([ADR-012(c)](0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add) forward-compat only-add): при `dry_run=true` Soul вызывает `Plan` вместо `Apply` (тот же план задач, read-only-проход). Оба — only-add, never-reuse field-номеров.
  - **(d) Статус `drift` — информационный, НЕ блокирующий.** incarnation-статус **`drift`** добавляется в `ValidStatus` + CHECK-миграцию (снимает «пост-MVP»-пометку из [таблицы статусов](../architecture.md#атомарность-и-error_locked); `provisioning` остаётся пост-MVP). `drift` **информационный** — он **НЕ блокирует remediation** (в отличие от `error_locked`/`destroy_failed`): remediation drift-а = **обычный apply** из состояния `drift` (прогнать scenario → хост сходится к декларации → статус `ready`). Сигнализирует «есть расхождение», не запирает incarnation.
  - **(e) Отчёт о расхождении — `DriftReport`.** Имя в ряд с `RunResult` / `SoulprintReport` ([naming-rules → Сообщения proto Keeper↔Soul](../naming-rules.md#сообщения-proto-keepersoul)): агрегированный отчёт «какие ресурсы на каких хостах показали бы `changed=true`».
  - **(f) Безопасность — default-deny для community-модулей.** Community `soul_module` **без read-safe-capability** → **drift не вызывается** (`Plan` не запускается, default-deny). Причина: `Plan` объявлен pure-read, но для стороннего плагина это нельзя гарантировать без явной декларации — недекларированный плагин в dry-run-режиме не дёргается (fail-closed). Core-модули read-safe по конструкции.
- **Затрагивает (amends).**
  - [ADR-015](0015-core-modules-mvp.md#adr-015-core-модули-mvp-точный-список) — Plan-семантика во всех core-модулях: **no-op → pure-read** с машинным `changed`.
  - [ADR-012](0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add) / [ADR-020](0020-plugin-infrastructure.md#adr-020-plugin-инфраструктура-формат-manifest-handshake-lifecycle) — only-add `PlanEvent.changed` (plugin-SDK) + `ApplyRequest.dry_run` (keeper↔soul).
  - [ADR-027](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim) — drift-проверка идёт через тот же **work-queue в Plan-режиме** (Acolyte клеймит задание, рендерит из рецепта, гонит `Plan` вместо `Apply`); никакого второго исполнительного контура.
  - **Status-машина incarnation** — `drift` входит в `ValidStatus` + CHECK-миграция (см. (d)); снимает пост-MVP-пометку `drift` из [таблицы статусов](../architecture.md#атомарность-и-error_locked).
- **Trade-offs.**
  - **Declarative drift, не полный дамп** — Scry видит расхождение только по объявленным ресурсам; неуправляемые изменения хоста невидимы. Принято: полного «как должно быть» для неуправляемых ресурсов нет, попытка сравнивать всё = шум + ложные срабатывания.
  - **Plan pure-read зависит от честности модуля** — для core гарантируем кодом; для community — default-deny (см. (f)). Цена — community-плагин без read-safe-декларации в drift-скане молча пропускается (не ломает скан, но и не покрыт). Принято.
  - **`drift` информационный, не блокирующий** — жертвуем «жёсткой защитой от ухода в drift» ради того, чтобы drift-статус не запирал incarnation и remediation шёл обычным apply. Если позже понадобится «запретить операции на drift-incarnation» — отдельное решение.
- **Срезы реализации.** **on-demand-пилот** (`keeper.incarnation.check-drift` + `Plan` pure-read в core + only-add proto + статус `drift` + `DriftReport`) → **фоновый скан** (Reaper-подобный лидер).
- **Amendment (Slice B реализован).** On-demand-пилот закоммичен полностью:
  - **scenario-kind `converge`** ([naming-rules → Сценарий-kind-ы](../naming-rules.md)). Сервис, поддерживающий drift-детект, ОБЯЗАН положить `scenario/converge/main.yml` в репо. Обнаружение auto-discover по имени — никакого нового YAML-поля; отсутствие файла → check-drift возвращает 422 «drift-проверка недоступна» (`ErrConvergeMissing`, информационно), incarnation не тронут.
  - **converge-input авто-резолв из `incarnation.state`.** Конвенция «по имени» ([`scenario.resolveDriftInput`](../../keeper/internal/scenario/checkdrift.go)): для каждого параметра converge-схемы значение берётся из override-body → `incarnation.state.<имя>` → default схемы; required без источника → 422 `ErrDriftInputMissing`. Это даёт «auto-drift из state» без ручной передачи параметров оператором.
  - **REST + MCP.** `POST /v1/incarnations/{name}/check-drift` ([operator-api.md](../keeper/operator-api.md)) + `keeper.incarnation.check-drift` ([mcp-tools.md](../keeper/mcp-tools.md)). **Sync-операция**: handler блокируется до сборки `DriftReport` и возвращает его 200-ответом. **`DriftReport`** — keeper-internal aggregated view + API/MCP-response (НЕ proto Keeper↔Soul): `{checked_at, incarnation, scenario_ref, hosts:[{sid, status, tasks:[{idx, module, action, changed, message}]}], summary:{hosts_drifted, hosts_clean, hosts_unsupported, hosts_failed}}`. Per-host `status` ∈ `clean`/`drifted`/`unsupported`/`failed`.
  - **Acolyte-флоу с DryRun-пробросом.** check-drift идёт через тот же work-queue, что обычный run ([ADR-027](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim)): `Recipe.DryRun=true` (forward-compat поле в `apply_runs.recipe` jsonb) → Acolyte строит `ApplyRequest{dry_run:true}` ([claim.go](../../keeper/internal/scenario/claim.go)). Inline-путь (acolytes=0) check-drift **не поддержан** (fail-closed: dry_run не пробрасывается).
  - **Status-машина amendment.** Миграция `047_incarnation_status_drift` добавила `drift` в CHECK `incarnation_status_valid`. Обычный apply (`scenario.lockRun`) и upgrade (`incarnation.PrepareUpgrade`) принимают `drift` как стартовый статус наравне с `ready` — drift НЕ блокирует. Destroy тоже допускает `drift` (`canDestroyFrom`).
  - **RBAC.** Новая permission `incarnation.check-drift` ([rbac.md](../keeper/rbac.md)) с тем же per-coven/service-scope, что `incarnation.run` (`IncarnationCovenContexts`).
  - **Audit.** Новое событие `incarnation.drift_checked` (`source=api`/`mcp`, `correlation_id=apply_id`, payload `{name, scenario, apply_id, drift_summary}`) — пишется handler-ом после сборки отчёта.
  - **Безопасность default-deny подтверждена.** `plan.unsupported`-task (community-модуль без `PlanReadSafe`-capability) на хосте → `DriftStatusUnsupported` (НЕ `failed`): отчёт отделяет «не поддержано» от реальной ошибки Plan.
  - **Lock-strategy (on-demand-пилот, осознанно).** CheckDrift НЕ берёт `lockRun`-эквивалент на incarnation: окно между `InsertPlanned` и terminal Acolyte-апплая теоретически пересечётся с параллельным `upgrade`/`destroy` на ту же incarnation. **Принято safe-by-design для on-demand-пилота:** (a) check-drift — оператор-инициированная read-only-операция; операторы не запускают её параллельно с разрушительными командами на ту же incarnation. (b) `MarkDriftStatus` использует `WHERE status IN ('ready','drift')` — поздняя запись на `status=applying`/`destroying` no-op-ится (consistency). (c) На Soul-side `dry_run` ≠ `Apply` (Slice A структурная гарантия): параллельный dry_run-flight + real-apply-flight на тот же хост семантически не конфликтуют (dry_run pure-read). **Полноценный lock** (сериализация check-drift c upgrade/destroy через `lockRun` или drift-aware Acolyte-claim-гейт + persistent task_idx→module-map) — задача **Slice C** (фоновый автоматический скан Reaper-подобным лидером, где конкурентность с другими операциями становится реальной).
  - **Не входит в Slice B.** Тираж Plan pure-read на остальные 14 core-модулей (готовы 3 пилотных: pkg/file/service из Slice A); фоновый периодический скан (Slice C — см. lock-strategy выше); per-VM-токен в bootstrap-флоу. Это будущие срезы.
- **Amendment (2026-05-26, Plan-тираж pure-read из Slice A применён к 14 остальным core-модулям, commit `c2d8181`).** Pattern Slice A (`PlanReadSafe`-capability + pure-read-имплементация `Plan`) применён к оставшимся 14 Soul-side core-модулям. Итого 12 stateful-модулей с drift-детектом + 5 без покрытия (2 deferred, 3 verbs не применимы).
  - **9 модулей с `PlanReadSafe` (полный drift-детект).** `core.group` / `core.user` / `core.cron` / `core.mount` / `core.sysctl` / `core.archive` / `core.line` / `core.repo` / `core.firewall`. Все ушли строго по pattern-у Slice A: read-only-проба ресурса → сравнение с желаемым → машинный `PlanEvent.changed` без мутаций хоста.
  - **2 модуля БЕЗ `PlanReadSafe` (отложено в Slice C).** **`core.git`** — state-форма `pulled` требует сетевой операции `git fetch` к remote-у, чтобы определить расхождение local HEAD vs remote tip; pure-read в Apply-флоу нет (`fetch` мутирует local refs). **`core.url` без checksum** — определить расхождение можно только через HEAD-запрос (Content-Length / ETag), что превращает Plan в сетевой вызов; без декларированного `checksum:` (sha256/sha1) безопасной read-only-пробы нет. В Slice C: добавить read-only-режим для `core.git` (`git ls-remote` без mutation) и принять `core.url` checksum-only-Plan (без checksum — `plan.unsupported`, как community-плагин без read-safe).
  - **3 verb-модуля БЕЗ `PlanReadSafe` (correct default-deny).** `core.exec.run` / `core.cmd.shell` / `core.http.probe` — verb-модули **не имеют желаемого состояния** (нет понятия «декларированный результат exec/cmd/http-probe»; они императивны / read-probe по конструкции). `plan.unsupported` для них — корректное поведение, не пробел покрытия. В `DriftReport` per-host `status` для этих задач — `unsupported`, отделяя «модуль не применим к drift» от реальной ошибки Plan.
  - **Итого покрытия drift-детектом:** 12 stateful core-модулей (3 пилотных Slice A: pkg/file/service + 9 нынешних) с честным `changed`-сигналом; 2 отложены в Slice C (`git`/`url`); 3 verb-модуля корректно дают `unsupported`. **17 - 5 = 12** stateful-модулей покрыты.
  - **Что НЕ входит в этот тираж.** Слой Slice C (фоновый периодический скан Reaper-подобным лидером + полноценный lock-strategy с upgrade/destroy + drift для `git`/`url`) — отдельный срез, не делается этим амендментом.
- **Slice C status (2026-05-26):** Background-scan реализован, default OFF (`reaper.scry_background.enabled=false`). Контракт:
  - Reaper-rule `scry_background` (keeper/internal/reaper/scry.go) сканирует кандидатов по предикату `last_drift_check_at IS NULL OR < NOW() - interval` с keyset-iterator-ом.
  - Each-candidate: validate-tx (TX1 lock → InsertPlanned dry_run scenario → COMMIT) с throttle через `CountActiveDryRuns()` (max_concurrent=10 по умолчанию).
  - Колонки `incarnation.last_drift_check_at`, `last_drift_summary` (jsonb) добавлены миграцией 050.
  - Default OFF — guard-rail: на проде включается explicit decision (нагрузка на check-drift умножается на число incarnation-ов).
  - Audit-source — `SourceBackground` (shared/audit/source.go).
- **Amendment (2026-06-10, converge → operational scenario-kind).** Сценарий `converge` выводится из специального набора `LifecycleScenarioNames` (там остаются только `create`/`destroy`). Следствия: (1) `converge` размечается `kind=operational` в каталоге `GET /v1/services/{name}/scenarios` и становится запускаемым обычным `run`-ом (Apply-режим: реально сводит хосты к декларированному состоянию, отчёт changed/no-changes; гоняется в т.ч. Cadence-ом как любой operational-сценарий). (2) Drift-check-функция НЕ меняется: `check-drift` грузит `scenario/converge/main.yml` по константе имени `ConvergeScenarioName` (auto-discover), а не через членство в lifecycle-наборе; `ErrConvergeMissing`, фоновый Scry-скан, MCP `incarnation.check-drift` и audit `incarnation.drift_checked` работают без изменений. Таким образом `converge` несёт двойную роль: как operational-`run` — Apply-reconcile; как dry-run target `check-drift` — declarative drift-detect. Из ADR-031 уходит только трактовка `converge` как lifecycle-kind, не сам drift-контур.
