# ADR-032. Push-orchestrator (Variant C) — multi-host destiny push without incarnation/scenario.

**Context.** push mode ([ADR-004](0004-binaries.md#adr-004-binary-layout--keeper-soul-soul-lint-push-mode-as-a-module-inside-keeper)) originally described "deliver the soul binary + ad-hoc apply over SSH". MVP+ required a multi-host parallel rollout of a destiny to a list of SIDs without the scenario/incarnation/state scaffolding (ad-hoc operations, operator-driven, not part of the soul-lifecycle).

**Decision.**
1. **Endpoint `POST /v1/push/apply`** (see [docs/keeper/operator-api.md:806-837](../keeper/operator-api.md)):
   - Body: `{inventory_sids[], destiny_ref ("<name>@<ref>"), ssh_provider, input, cleanup_stale}`.
   - Response: `202 + {apply_id}` (ULID).
   - Async model: best-effort parallel rollout, without a cross-host barrier.
2. **Storage — a separate table `push_runs`** (migration 051, does not overlap with `apply_runs`):
   - Fields: `apply_id` PK, `inventory_sids` text[], `destiny_ref`, `ssh_provider`, `input` jsonb, `cleanup_stale` bool, `status`, `started_at`, `finished_at`, `started_by_aid` FK→operators, `started_by_kid`, `summary` jsonb.
   - Partial indexes WHERE status IN ('pending','running').
3. **Statuses push_runs.status:** `pending` | `running` | `succeeded` | `partial_failed` | `failed`.
   - `partial_failed` — some hosts passed, some failed; effectively a "success", the operator decides on a retry of the failed ones themselves.
4. **RBAC permissions (new):** `push.apply` (launch), `push.read` (reading `GET /v1/push/{apply_id}`).
5. **Render context for push:**
   - Available: `input.*`, `soulprint.self.*`, `vault(...)`.
   - Unavailable: `register.*`, `incarnation.state.*`, `essence.*`, `soulprint.hosts` — fail-render with an explicit error (push has no scenario/incarnation/cross-host runtime).
   - Mechanism: a synthetic one-task scenario + `destinyIsolated=true` in render.Pipeline (see the architect verdict push S4).
6. **Recovery — best-effort via status + Reaper.**
   - On a Keeper restart, pending/running push runs → orphans.
   - The Reaper rule `purge_orphan_push_runs` (default TTL 1h): orphan → status=`failed`, summary={orphan_purged: true}.
   - Without a complex cross-keeper lease model (push is oneshot, fail-fast is simpler).
7. **Audit-events:** `push.applied` (start), `push.completed`/`push.failed`/`push.partial_failed` (finish). See [naming-rules.md](../naming-rules.md#audit-events).
8. **Symmetry with pull.** The analog `POST /v1/souls/{sid}/exec` for a pull-ad-hoc (single core commands over the Soul stream) — a separate entity Errand, ADR-033 in progress.

**Invariants.**
- Push does NOT read/mutate `incarnation.state` (there is no incarnation in the loop).
- The render pipeline is reused via the synthetic scenario; `RenderDestinyStandalone` is **not** introduced (drift risk).
- The `DestinyResolver` for push is separate (`pushorch.pushDestinyResolver`), it does not overlap with the scenario-side `DestinySource` (which is tied to service.yml).
- `topology.LoadByInventory(sids)` — a new only-add point, without an incarnation phase and declared-role.

**Rejected alternatives.**
- (a) Reuse `apply_runs` for push — rejected, the application lifecycles of a scenario and push differ (no state_changes, no attempt-fencing, no barrier).
- (b) `RenderDestinyStandalone` as a separate public API in render/ — rejected (drift risk, two entry points into CEL+template).
- (c) Push via the Acolyte work-queue — deferred (out of MVP scope); push is oneshot, a work-queue is redundant.

**What is deferred.**
- Vault token-renewal for long push runs (>100 hosts): a separate slice.
- Caching the destiny-tree between runs: `DestinyLoader` already carries a snapshot cache; if it hits git-clone latency — a separate case.
- Cancelling a push (by `apply_id`): not in the MVP, a separate slice.

**Amendment (2026-05-26, S6 pilot wire-up SshDispatcher).** ADR-032 aims at the production form of the push-orchestrator with the PG table push_providers, souls.ssh_target jsonb and push.host_ca_refs[] (multi-CA, multi-provider routing) — that is S7, a separate slice. Until S7 closes, "production" is enabled via a **pilot form** through 3 inline fields in `keeper.yml::push:`:

- **`push.targets[]`** — per-SID SSH credentials (host=SID, ssh_port/ssh_user/soul_path with defaults 22/root/`/usr/local/bin/soul`). Without the migration `souls.ssh_target jsonb` — pilot. A SID with no record → `ErrTargetNotConfigured` on resolve in the SshDispatcher (fail-closed).
- **`push.providers[]`** — per-provider `params` (an opaque form), serialized to JSON and injected into the env `SOUL_SSH_<UPPER_SNAKE(name)>_PARAMS` of the plugin at spawn (ADR-020 amendment l). Without the PG table `push_providers` — pilot.
- **`push.host_ca_ref`** — a single Vault-ref to the PEM public host-CA (the field `public_key`). Single-CA (multi-CA `host_ca_refs[]` — S7). Plaintext-inline-PEM is rejected (`vault_ref_invalid` in the schema phase).

`setupPushDispatchers` (`keeper/cmd/keeper/daemon.go`) brings up a single-provider single-CA SshDispatcher on the first discovered SshProvider plugin; multi-provider routing by `push_runs.ssh_provider` is deferred to S7. The spawn is done once at startup, the plugin handle is closed in LIFO order BEFORE Redis/Pool. The normative grammar of the block — [docs/keeper/config.md → push](../keeper/config.md#push); the wire-up description — [docs/keeper/push.md → Wire-up of SshDispatcher in the daemon](../keeper/push.md#wire-up-of-sshdispatcher-in-the-daemon-s6-pilot-2026-05-26).

The pilot **deliberately** does not introduce new entities (a dual registry `push.ssh_providers[]`, `private_key_vault_path`, a multi-CA list, multi-provider routing) — this is fully covered by the S7 canon and is not simplified in the long run. Until S7 the pilot is the only canon; after S7 the inline forms `push.targets[]` / `push.providers[]` are rejected as `unknown_key` (the deprecation window is fixed in the S7 amendment).

**Amendment (2026-05-26, S7-1 wire-up `souls.ssh_target jsonb` as the PG canon).** The first slice of S7: per-host SSH credentials move from the inline `push.targets[]` into the `souls` registry. Changes:

- **Column `souls.ssh_target jsonb`** (migration 053) + CHECK `souls_ssh_target_shape` (typed shape: integer/text/text). A NULL semantics of the whole field means "target not configured".
- **Operator API:** `PUT /v1/souls/{sid}/ssh-target` with body `{ssh_port, ssh_user, soul_path}`. Permission `soul.ssh-target-update` (with the selector `host=<sid>`), audit `soul.ssh-target.updated`. MCP-tool: `keeper.soul.ssh-target.update` (the 3-segment pattern `keeper.sigil.key.<verb>` ↔ permission `sigil.key-<verb>`).
- **Push-flow resolver:** `PGFallbackTargetResolver` (PG-first; `souls.ssh_target` → defaults port=22/user=root/soul-path=/usr/local/bin/soul for omitted fields). PG-row.ssh_target IS NULL → a check of the flag `push.allow_legacy_push_targets` (default `false`): false → `ErrTargetNotConfigured`; true → a one-time WARN deprecation + fallback onto `ConfigTargetResolver` over `push.targets[]`.
- **PM-decisions S7-1:** (1) push-providers/conduit as an "SSH Provider" variant of Provider (S7-2, not this slice); (2) the deprecation policy of `keeper.yml::push.targets[]` — 1 release WARN → hard-cut; (3) the audit-event name `soul.ssh-target.updated`; (4) the permission `soul.ssh-target-update`; (5) a typed JSON-shape, validated by CHECK; (6) **no auto-import** of keeper.yml targets[] into PG in S7-1 (deferred to S7-4, requires explicit consent + idempotency); (7) priority PG > keeper.yml (DB is the source of truth).
- **What is NOT in S7-1:** the push_providers PG table (S7-2), multi-CA `push.host_ca_refs[]` (S7-3), auto-import of legacy targets (S7-4).

**Amendment (2026-05-26, S7-2 wire-up `push_providers` PG table as the PG canon).** The second slice of S7: the per-provider env-payload params of SSH plugins move from the inline `push.providers[]` (pilot S6 / S7-1) into a PG table. Changes:

- **Table `push_providers`** (migration 054): PK `name TEXT` (regex `^[a-z][a-z0-9-]{0,62}$` — env-var-name-safe for translation into `SOUL_SSH_<UPPER_SNAKE(name)>_PARAMS`), `params JSONB`, `created_by_aid` NOT NULL REFERENCES operators, `updated_at`/`updated_by_aid` for triage audit.
- **Operator API:** 5 CRUD endpoints `POST/GET/PUT/DELETE /v1/push-providers[/{name}]` + `GET /v1/push-providers` (list). 5 permissions `push-provider.{create,update,delete,list,read}`. Audit-events `push-provider.created` / `.updated` / `.deleted` (payload `{name, params_keys}` without values — records the mutation without disclosing secrets). MCP-tools `keeper.push-provider.{create,update,delete,list,read}` (5 new tools, total catalog 53→58).
- **Sensitive params (PM-decision S7-2 #5):** the keys `secret_id` / `token` / `password` / `private_key` MUST be vault-refs (`vault:<path>`) — validation on the service layer (`pushprovider.Service.validateSensitive`), 422 on plaintext.
- **Push-flow resolver:** `PGFallbackProviderResolver` (`keeper/internal/push/provider_pg.go`) at the start of `setupPushDispatchers` resolves the env-payload params for the discovered SSH plugin: PG-first → fallback `keeper.yml::push.providers[]` under the flag `push.allow_legacy_push_providers` (default false; a one-shot WARN deprecation). If there is no record in PG and legacy is off — the plugin starts without an env-payload (like pilot S6).
- **Hot-reload (PM-decision S7-2 #6):** spawn-on-change via the Redis pub/sub `push-providers:changed` (`keeper/internal/redis/pushproviderchanged.go`); a REST/MCP mutation publishes, each node is subscribed via `SubscribePushProvidersChanged`. The subscription infrastructure is brought up; the actual re-spawn of the SSH plugin with the new env-payload — a separate follow-up slice (the current S7-2 implementation only logs the incoming messages).
- **PM-decisions S7-2:** (1) Extend the Provider entity (Cloud Provider + SSH Provider — two domains of one entity; different tables and permission areas); (2) audit-event naming `push-provider.created/updated/deleted` (kebab single-section); (3) Redis topic `push-providers:changed`; (4) deprecation 1 release WARN → hard-cut; (5) sensitive params as vault-refs; (6) hot-reload = spawn-on-change.
- **What is NOT in S7-2:** auto-import of legacy `keeper.yml::push.providers[]` into PG (S7-4), per-host invalidation (the multi-provider routing slice), a plugin's `sensitive_keys[]` capabilities manifest (post-MVP).

**Amendment (2026-05-26, S7-3 multi-CA `push.host_ca_refs[]`).** The third slice of S7: the single Vault-ref `push.host_ca_ref` → an array of structs `push.host_ca_refs[{ref, name}]` for verifying host keys over SSH. Changes:

- **Grammar:** `push.host_ca_refs[]` (`KeeperPushCARef = {ref: vault-ref, name: kebab-case}`). Each `ref` — vault-ref-only (plaintext-PEM forbidden, symmetric to the singular); `name` — operator-defined, used as the label value in `keeper_push_host_ca_used_total{ca_name=...}` and in diag messages. The names in the set must be unique.
- **Backward-compat singular:** `push.host_ca_ref` remains under a 1-release WARN deprecation window (like S7-1/S7-2). With the singular filled and `host_ca_refs[]` empty the daemon auto-adapts the singular into the singleton `HostCARefs[0]` with the auto-name `default` and writes a one-time WARN. The simultaneous presence of singular + plural is rejected by the schema phase (`mutually_exclusive_keys`).
- **Verify logic:** `ssh.CertChecker.IsHostAuthority` does an OR-check over all loaded CAs (closed by `hostCertCallback(cas, onMatch)` in [`keeper/internal/push/session.go`](../../keeper/internal/push/session.go)). A host-cert signed by any CA from the set → trusted; otherwise reject (refusal of TOFU). On a match the callback `onMatch(caName)` increments `keeper_push_host_ca_used_total{ca_name=...}` + a debug log.
- **Per-provider CA-override — DEFERRED.** In the MVP all-providers-trust-all-CAs: one set `host_ca_refs[]` applies to all SSH handshakes (target + proxy_jump). A separate CA per provider/route — a post-MVP open Q after S7.
- **PM-decisions S7-3:** (1) format — an array of structs (not a flat array of strings) to extract `name` for the label value; (2) backward-compat singular auto-adapt into a singleton + WARN; (3) mutually exclusive singular+plural; (4) verify via `CertChecker.IsHostAuthority` OR-loop; (5) per-provider CA-override deferred (all-trust-all in the MVP); (6) deprecation policy — 1 release WARN (like S7-1/S7-2).
- **What is NOT in S7-3:** per-provider CA-override / per-route CA (deferred, open Q post-S7), auto-import of legacy targets (S7-4).
- **Teleport by-name transport (ADR-063 amendment) — NOT via `push.host_ca_refs[]`.** Bootstrap delivery of the token in the mode `push.transport: teleport` (keeper-side `core.bootstrap.delivered`) does the host-verify via the Teleport identity-file (the Teleport host-CA from the `known_hosts` identity), and NOT via the `push.host_ca_refs[]` Vault host-CA. `host_ca_refs[]` remains mandatory only for the generic modes (a push run of a Destiny + `transport: direct` bootstrap delivery). See [ADR-063 §Amendment](0063-bootstrap-token-delivery.md).

**Amendment (2026-05-26, S7-4 auto-import legacy on start + deprecation timeline).** The final slice of S7: closes the migration loop — the pilot and the canon coexist under the previous flags, while an explicit opt-in moves the inline blocks into the PG sources in one pass at startup. Changes:

- **Grammar:** two new fields in `keeper.yml::push:`:
  - `push.auto_import_legacy_targets: false` (default) — an opt-in one-shot migration `push.targets[]` → `souls.ssh_target` jsonb;
  - `push.auto_import_legacy_providers: false` (default) — likewise for `push.providers[]` → the PG table `push_providers`.
- **One-shot semantics.** [push.AutoImporter] (`keeper/internal/push/auto_import.go`) runs as the step `runLegacyAutoImport` in the pipeline after `setupPushProviderSvc` and BEFORE `setupAPIServer`. Idempotent: already-imported SIDs / provider names are skipped (the PG canonical source is not overwritten), a repeated start with no new records in `keeper.yml` — a no-op. A missing `souls` row for a config-target SID — WARN-skip (not fatal: the soul is not yet registered, importing the credential is pointless; the operator will PUT it later via `/v1/souls/{sid}/ssh-target`). A PG read/write fail — `errSetupFailed` (the operator turned the flag on, they must not silently get "half imported").
- **Audit-events** (a new `source` enum: `config_bootstrap`, a system action, `archon_aid: NULL`):
  - `soul.ssh-target.imported_from_config` — per-row, payload `{sid, ssh_port, ssh_user, soul_path}`;
  - `push-provider.imported_from_config` — per-row, payload `{name, params_keys}` (the values of sensitive keys are NOT written to the audit, symmetric to `push-provider.created`).
- **System-AID `archon-system`.** Imported `push_providers` rows carry `created_by_aid='archon-system'` (to separate them from Archon-creates). The FK on `operators(aid)` requires the row `archon-system` to exist before the first auto-import; a reserved-AID migration — a separate slice (the pilot assumes an operator adding it by hand, or the bootstrap semantics of the next slice).
- **Deprecation timeline (final):**
  - **S7 wave (the current commit):** pilot + canon coexist. PG-first resolvers (`PGFallbackTargetResolver` / `PGFallbackProviderResolver`); yml fallback under `push.allow_legacy_push_targets` / `push.allow_legacy_push_providers` (default false); auto-import under `push.auto_import_legacy_targets` / `push.auto_import_legacy_providers` (default false). A 1-release WARN on the use of any legacy fallback / on the auto-adapt of a singular host_ca_ref.
  - **S8 (after the next prod release):** hard-cut. `keeper.yml::push.targets[]` / `push.providers[]` / singular `host_ca_ref` → `unknown_key` in the schema phase. The `allow_legacy_push_*` / `auto_import_legacy_*` flags are removed. The PG resolvers are simplified to PG-only (without `Fallback`/`AllowLegacy` fields).
- **Operator-driven migration path (if auto-import does not fit):** `soulctl souls ssh-target set <sid> --port … --user … --soul-path …` (S7-1) and `soulctl push-providers create <name> --params=…` (S7-2). Both paths are idempotent and audited (`soul.ssh-target.updated` / `push-provider.created`).
- **PM-decisions S7-4:** (1) auto-import — opt-in (default false, explicit operator consent); (2) one-shot semantics + idempotent (ON CONFLICT / PG-row presence check); (3) audit-events `soul.ssh-target.imported_from_config` + `push-provider.imported_from_config`; (4) a new `source` enum `config_bootstrap` (to separate it from `keeper_internal` semantically); (5) deprecation 1-release WARN → S8 hard-cut; (6) system-AID `archon-system` for imported rows.
- **What is NOT in S7-4:** the S8 hard-cut (a separate slice after the prod release); the reserved-AID `archon-system` migration (a separate slice). The TODO `push-providers:changed` re-spawn of the SshDispatcher plugin-handle — closed by ADR-032 amendment 2026-05-27 below (S7-2 closure).

**Amendment (2026-05-27, S7-2 closure: runtime re-spawn of the SshDispatcher plugin-handle).** Closes the TODO from S7-2 "the actual re-spawn of the SSH plugin with the new env-payload". S7-2 brought up the subscription infrastructure + logged the incoming `push-providers:changed` messages; this amendment connects the actual re-spawn. Changes:

- **`SshDispatcher.RefreshProvider(ctx, providerName)`** ([`keeper/internal/push/dispatcher.go`](../../keeper/internal/push/dispatcher.go)) — a public method that, under `d.mu.Lock`, swaps the internal `Deps.Provider`/`Deps.ProviderCloser` for a fresh plugin handle. Name reconciliation: the single-provider pilot holds exactly one provider (`Deps.ProviderName`), messages about another plugin are ignored (a no-op without error); an empty name (mass invalidation) is treated as "ours". The sentinel `push.ErrRespawnNotSupported` — when `Deps.Respawner == nil` (unit tests / single-instance dev without CRUD invalidation).
- **`push.ProviderRespawner`** — a narrow interface `RespawnProvider(ctx, name, oldCloser) (SshProvider, io.Closer, error)`. The implementation `pushProviderRespawner` ([`keeper/cmd/keeper/push_respawn.go`](../../keeper/cmd/keeper/push_respawn.go)) holds `*pluginhost.Host` + `[]pluginhost.Discovered` + `*push.PGFallbackProviderResolver`: finds the discovered one by name, closes the old handle, resolves fresh params (PG-first → legacy-fallback), spawns a new one with `WithEnv(SOUL_SSH_<UPPER_SNAKE(name)>_PARAMS)`, wraps it in `pluginhost.SshProviderPlugin`. The push package is independent of pluginhost/PG — the respawner's surface is narrow.
- **Daemon-listener** (`runPushProviderInvalidationListener`, [`keeper/cmd/keeper/daemon.go`](../../keeper/cmd/keeper/daemon.go)) — the TODO logging implementation is replaced by an actual call to `pushSshDispatcher.RefreshProvider`. The error `ErrRespawnNotSupported` is logged as WARN (not our plugin / no respawner); an actual spawn-fail — ERROR + the dispatcher goes into a degraded state until the next mutation/restart. The listener does not crash on an error — it keeps reading the channel.
- **Concurrency invariant:** RefreshProvider holds the `sync.RWMutex` Lock over the whole re-spawn cycle; SendApply/Cleanup on the hot path take a snapshot of the reference via `provider()` under RLock and run to the end of the session without blocking. The old handle stays alive until the moment the respawner calls its Close (synchronously inside RefreshProvider); a running job holds the gRPC conn open itself, `BasePlugin.Close` is idempotent.
- **Tests:** unit `keeper/internal/push/refresh_test.go` (happy-path / wrong-name no-op / empty-name mass-invalidate / spawn-fail degraded state / mutex concurrency); integration `keeper/internal/push/refresh_integration_test.go` (under `-tags=integration`, a miniredis fixture): publish → subscription → listener → RefreshProvider → respawner round-trip.
- **PM-decisions amendment 2026-05-27:** (1) the pilot shape is kept — `SshDispatcher` holds exactly ONE provider (single-provider pilot), not a map[name]; multi-provider routing — a separate slice post-S7, an extension without a breaking change via `map[string]providerEntry`; (2) the Respawner — a separate interface in the push package, the concrete implementation in `keeper/cmd/keeper/` (where the pluginhost+resolver dependencies live); (3) the listener does not crash on errors — best-effort with a structured log; (4) RWMutex (Read for the hot path SendApply, Write only on re-spawn) — so that the re-spawn does not delay running jobs.
- **What is NOT in this amendment:** multi-provider routing (several SshDispatchers on one node, each under its own `pluginName`); per-host invalidation (the current channel is cluster-wide); a graceful drain of the old plugin-handle before Close (old in-flight RPCs run out via their own snapshot, there is no explicit await — the pluginhost-side conn is closed on Close, unfinished RPCs will get cancelled, but this is not our hot path — the SendApply/Cleanup RPCs to the plugin are short).

**Amendment (2026-05-27, P2 Multi-provider routing).** Closes "multi-provider routing" from the known gap of the S7-2 closure: the keeper holds SIMULTANEOUSLY several SshProvider plugins and per-SID routes the SIDs between them. Changes:

- **Dispatcher single → map.** `push.Deps.Provider/ProviderName/ProviderCloser/Respawner` (single-provider pilot) removed; instead of them `Deps.Providers map[string]ProviderEntry` (`ProviderEntry = {Provider, Closer}`). `SendApply`/`Cleanup` receive a `providerName` from the caller and do a lookup in the map under RLock; an unknown name → `push.ErrProviderUnknown`. `SshDispatcher.RefreshProvider(name)` swaps one entry under Lock without affecting the others; `name=""` (mass invalidate) → re-spawn of all. The old single-provider-shape tests are rewritten onto a map-based wire-up via the helper `testProviderName + map[string]ProviderEntry{...}`.
- **Selector R1 (3-tier resolve)** in [`keeper/internal/push/router.go::PGRouter`](../../keeper/internal/push/router.go) (`ProviderRouter` interface, `RouteFor(ctx, sid) (name, source, err)`). Levels:
  - **Level 1:** `souls.ssh_target.ssh_provider` — an optional field in the shape-CHECK `souls_ssh_target_shape` (migration 056 extends 053 with the regex `^[a-z][a-z0-9-]{0,62}$` for the optional `ssh_provider`). Source `soul`.
  - **Level 2:** `push.coven_default_providers: { <coven>: <provider_name> }` — a map in keeper.yml. The tiebreak on a multiple coven-match of a Soul — **alphabetical order of the covens** (determinism). Source `coven`.
  - **Level 3:** `push.cluster_default_provider` — a cluster fallback. Source `cluster`.
  - All three empty → `ErrProviderNotRouted` → fail per-host (status=`error`, error_code=`provider_not_routed`). **WITHOUT a provider-chain fallback** (a security invariant: the auth-perimeter of different providers is different, a silent fallback breaks trust).
- **α-compat** REST/MCP `POST /v1/push/apply::ssh_provider`: with a non-empty field — the per-job preset applies to ALL SIDs, the router is NOT called. The source is treated as `soul`. With an empty one — a router resolve per-SID.
- **Eager spawn.** `setupPushDispatchers` spawns ALL discovered SshProvider plugins at Keeper startup (UX predictability + the plugin-start cost is a one-time one). A spawn-fail of any plugin → `errSetupFailed` (the operator explicitly declared it in the catalog).
- **Audit/Metrics.** The per-SID routing decision is **NOT** written as a separate audit-event (redundant noise). The actual SshProvider is stored in `push_runs.summary.hosts[sid].ssh_provider`. The counter `keeper_push_provider_routed_total{provider, decision_source}` (low cardinality ~N\_providers × 3) — on each successful RouteFor + on the α-compat preset path.
- **Hot-reload** for Level 2/3: `RouterConfigSource.Snapshot()` pulls a fresh `*KeeperConfig` via `config.Store.Get()` on each RouteFor (an atomic read, no PGRouter rebuild).
- **Operator surface:**
  - Per-SID: an extended `PUT /v1/souls/{sid}/ssh-target` body + the `keeper.soul.ssh-target.update` MCP-tool + `soulctl souls ssh-target set --ssh-provider=<name>`.
  - Bulk per-coven: `soulctl souls ssh-target bulk-set --coven=<name> --ssh-provider=<name>` (a client-side fan-out over list+PUT; a server-side bulk-route is not introduced).
  - Cluster / per-coven default: editing `keeper.yml::push.{coven_default_providers,cluster_default_provider}` + SIGHUP / API-reload.
- **PM-decisions amendment 2026-05-27 (P2):** (1) selector R1 — 3-tier (per-SID → per-coven → cluster), without β/γ variants in the MVP; (2) fail-per-host WITHOUT a provider-chain (security-first); (3) audit-summary-only (without a separate per-routing event); (4) eager spawn (not lazy); (5) the α-compat preset overrides routing; (6) the bulk operator command — a client-side fan-out over list+PUT.
- **What is NOT in P2:** a soul-label-selector `souls.attributes` (a new entity; propose-and-wait, deferred); per-job inline `routing_rules` (the γ-variant per-run override map; post-MVP); provider-chain fallback; lazy spawn; a soul-level per-incarnation override.
