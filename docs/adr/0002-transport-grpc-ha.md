# ADR-002. Transport Keeper ↔ Souls — gRPC bidirectional stream over mTLS, HA Keeper cluster

- **Context.** Souls run on managed hosts where keeping open ports is undesirable. We need a reliable bidirectional channel with encryption, mutual authentication, streaming of commands and events, an externally exposed OpenAPI/MCP facade, and scalability to at least thousands of Souls per installation.
- **Decision.**
  - Soul initiates the connection to Keeper and holds a long-lived bidirectional gRPC stream. Keeper → Soul commands travel over this stream, and Soul → Keeper events and results travel over the same stream. The transport is protected by mTLS.
  - **Keeper scales horizontally** as a set of stateless instances on top of a shared Postgres ([ADR-005](0005-storage-postgres.md#adr-005-keeper-state-storage--postgres)) and Redis ([ADR-006](0006-cache-redis.md#adr-006-кэш-и-координация--redis)). Each Soul connection lands on one Keeper instance chosen by the LB or by priorities in the Soul config; the source of truth is the DB and Redis.
  - **Each Keeper instance has a KID** (Keeper ID) — a stable identifier for binding leases, audit, and metrics.
  - **Soul on the client side** holds a fallback list of endpoints with priorities; the connection and failback algorithm is described in the "Soul connection: priority and failback" section.
- **Rationale.** No open ports on Souls; mTLS covers mutual authentication without a separate layer; gRPC provides streaming, a protobuf-based contract, and a convenient gRPC-Gateway for OpenAPI on top of the same core. HA via a shared DB and Redis cache is a standard stateless pattern, with no custom consensus.
- **Trade-off.** This introduces mandatory external dependencies (Postgres, Redis), raising the operational bar. We accept this — without them a correct HA Keeper cluster cannot be built, and without HA the "security and availability out of the box" requirement is not met.
- **Amendment (2026-05-25, presence — Redis-derived).** Soul presence (online/offline) **is not materialized in PG** (invariant "hot → Redis, not PG", scale target 100k VMs): the authority for "Soul online" is a live **SID lease in Redis** ([ADR-006(b)](0006-cache-redis.md#adr-006-кэш-и-координация--redis)), acquired on EventStream session-open and held by a renewal goroutine, expiring by TTL on teardown/disconnect. The **target resolver** (`keeper/internal/topology/resolver.go`) derives the online set in two phases: SQL selects candidates by Coven membership + `status NOT IN (pending/revoked/expired/destroyed)`, then a batch EXISTS check against the SID lease filters out those actually alive — `rosterSQL` **no longer** filters on `status='connected'`. `souls.status` now carries only lifecycle (`pending`/`revoked`/`expired`/`destroyed`) plus a lazy `connected`/`disconnected` snapshot for the Operator API's "last known" view (reconciled by the reaper, not a presence source). This closes the bug "reconnect after a planned Keeper restart did not restore presence → the reconnected Souls were invisible to the scheduler (`no_hosts`)" (found by the live multi-Keeper test on 2026-05-25) — without synchronous PG presence writes on the hot connect/disconnect path (an idle Soul does not flap either: presence is held by the renewal-goroutine lease, not by app traffic).
- **Amendment (2026-05-25, soul-shedding in the failback model).** Failback from the "Soul connection" section was **Soul-initiated only** (Soul itself decides to return to a higher-priority endpoint). A symmetric **Keeper-initiated shedding** mechanism (soul-shedding) has been added: an isolated Keeper instance that has lost contact with shared state (PG **and/or** Redis) **actively closes its Soul streams**, so the Souls fail back to healthy instances instead of hanging on a "half-alive" Keeper. The mechanism is `Watchman` (`keeper/internal/watchman/`): a probe loop pings PG+Redis, a debounce state machine (a threshold of consecutive failures to enter, consecutive successes to exit) moves the instance to `isolated` and calls `StreamManager.CloseAll()`, which cancels the per-stream `ctx` of every active EventStream → Soul receives **EOF** (not a separate go-away message; the proto was not touched) → the normal failback list kicks in. The reverse transition `isolated`→`healthy` does not "call anything back" — Souls return on their own via priority/spray. It is triggered **only by real isolation** (not by a one-off timeout), so a healthy cluster does not flap. Related to the refuse-guard and the registry of live instances — see [ADR-006](0006-cache-redis.md#adr-006-кэш-и-координация--redis) amendment (Conclave) and [ADR-027](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim) (k).
- **Amendment (2026-05-25, Shepherd — load balancing on scale-out, PLANNED/backlog).** Watchman only sheds streams on **isolation** (binary, all-or-nothing). A separate problem is **scale-out**: when a new Keeper instance is added behind the LB, existing long-lived EventStream streams **stick** to the old instances (the LB only balances new connections; Soul does not reconnect on its own until the stream breaks; failback only kicks in when `priority>1`). The new instance sits idle until natural churn (hours to days). The solution is the **Shepherd** subsystem (name chosen by the user; propose-and-wait completed): an instance that sees, via [Conclave](0006-cache-redis.md#adr-006-кэш-и-координация--redis), that its load skews above a fair share sheds the **excess** of its streams (a partial `StreamManager.CloseAll`, not all of them — unlike Watchman) with jitter/cap → Souls reconnect and redistribute. **The balancing domain is the priority group / a single VIP** (different priorities form a failover hierarchy, not a balance; within a group EventStream does a shuffle — see [docs/soul/connection.md](../soul/connection.md); behind a single VIP the LB itself balances via least-conn). An alternative/complement is passive max-connection-age stream recycling (gRPC server `MaxConnectionAge` / soul `max_stream_age`+spray) — a "dumb" background process with a constant handshake cost, so Shepherd is preferred. **Drain mode** (auto-by-threshold / semi-auto / manual) is TBD. **Implementation is deferred** (backlog 2026-05-25); it requires amending ADR-006 (Conclave carries a load snapshot) + an architect design + checking interaction with in-flight apply (ADR-027 Ward/recovery). The name is in the dictionary — [naming-rules.md](../naming-rules.md).
