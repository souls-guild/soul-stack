# ADR-002. Транспорт Keeper ↔ Souls — gRPC bidirectional stream поверх mTLS, HA-кластер Keeper

- **Контекст.** Souls стоят на управляемых хостах, на которых нежелательно держать открытые порты. Нужен надёжный двусторонний канал с шифрованием, аутентификацией обеих сторон, потоковой передачей команд и событий, выставление наружу OpenAPI/MCP-фасада, и масштабируемость хотя бы до тысяч Souls на инсталляцию.
- **Решение.**
  - Soul инициирует подключение к Keeper и держит долгоживущий bidirectional gRPC-стрим. Команды Keeper → Soul идут по этому стриму, события и результаты Soul → Keeper — по нему же. Транспорт защищён mTLS.
  - **Keeper горизонтально масштабируется** как набор stateless-инстансов поверх общей Postgres ([ADR-005](0005-storage-postgres.md#adr-005-хранилище-состояния-keeper--postgres)) и Redis ([ADR-006](0006-cache-redis.md#adr-006-кэш-и-координация--redis)). Каждое подключение Soul приземляется на один Keeper-инстанс по выбору LB или приоритетов в Soul-конфиге; источник правды — БД и Redis.
  - **Каждый Keeper-инстанс имеет KID** (Keeper ID) — стабильный идентификатор для привязки lease, аудита и метрик.
  - **Soul на стороне клиента** держит fallback-list endpoints с приоритетами; алгоритм подключения и failback описан в разделе «Подключение Soul: priority и failback».
- **Обоснование.** Нет открытых портов на Souls; mTLS закрывает аутентификацию обеих сторон без отдельного слоя; gRPC даёт поток, контракт через protobuf и удобный gRPC-Gateway для OpenAPI поверх того же ядра. HA через общую БД и Redis-кэш — стандартный stateless-паттерн, без собственного консенсуса.
- **Trade-off.** Появляются обязательные внешние зависимости (Postgres, Redis), что повышает порог эксплуатации. Принимаем — без них корректный HA-кластер Keeper-а не сделать, а без HA не выполняется требование «безопасность и доступность из коробки».
- **Amendment (2026-05-25, presence — Redis-derived).** Presence Soul-а (online/offline) **не материализуется в PG** (инвариант «горячее → Redis, не PG», цель масштаба 100k VM): авторитет «Soul online» — живой **SID-lease в Redis** ([ADR-006(b)](0006-cache-redis.md#adr-006-кэш-и-координация--redis)), берётся на session-open EventStream и держится renewal-goroutine, гаснет по TTL на teardown/обрыве. **Таргет-резолвер** (`keeper/internal/topology/resolver.go`) деривирует online-набор двухфазно: SQL отбирает кандидатов по членству в Coven + `status NOT IN (pending/revoked/expired/destroyed)`, затем batch-EXISTS по SID-lease отфильтровывает реально живых — `rosterSQL` **больше не** фильтрует `status='connected'`. `souls.status` несёт только lifecycle (`pending`/`revoked`/`expired`/`destroyed`) + ленивый snapshot `connected`/`disconnected` для Operator API «последнее известное» (согласуется reaper-ом, не источник presence). Закрывает баг «реконнект после планового рестарта Keeper не возвращал presence → переподключённый флот невидим планировщику (`no_hosts`)» (выявлен live multi-keeper тестом 2026-05-25) — без синхронных PG-presence-записей на горячем connect/disconnect-пути (idle-Soul тоже не флапает: presence держит renewal-goroutine lease, не app-трафик).
- **Amendment (2026-05-25, soul-shedding в failback-модели).** Failback из раздела «Подключение Soul» был **только Soul-инициируемым** (Soul сам решает вернуться на более приоритетный endpoint). Добавлен симметричный **Keeper-инициируемый сброс** (soul-shedding): изолированный Keeper-инстанс, потерявший связь с общим состоянием (PG **и/или** Redis), **активно закрывает свои Soul-стримы**, чтобы флот ушёл failback-ом на здоровые инстансы, а не висел на «полу-живом» Keeper-е. Механизм — `Watchman` (`keeper/internal/watchman/`): probe-loop пингует PG+Redis, debounce-state-machine (порог подряд-фейлов на вход, подряд-успехов на выход) переводит инстанс в `isolated` и дёргает `StreamManager.CloseAll()`, который отменяет per-stream `ctx` каждого активного EventStream → Soul получает **EOF** (не отдельное go-away-сообщение, proto не трогали) → отрабатывает обычный failback-list. Обратный переход `isolated`→`healthy` ничего не «зовёт назад» — Souls возвращаются сами по priority/spray. Триггерится **только реальной изоляцией** (а не разовым таймаутом), поэтому здоровый кластер не флапает. Связано с refuse-guard и реестром живых инстансов — см. [ADR-006](0006-cache-redis.md#adr-006-кэш-и-координация--redis) amendment (Conclave) и [ADR-027](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim) (k).
- **Amendment (2026-05-25, Shepherd — балансировка нагрузки при scale-out, PLANNED/backlog).** Watchman сбрасывает стримы только при **изоляции** (бинарно, все). Отдельная проблема — **scale-out**: при добавлении нового Keeper-инстанса за LB существующие долгоживущие EventStream-стримы **залипают** на старых инстансах (LB балансирует только новые соединения; Soul сам не реконнектится, пока стрим не оборвётся; failback срабатывает лишь при `priority>1`). Новый инстанс простаивает до естественного churn (часы-дни). Решение — подсистема **Shepherd** (имя выбрано пользователем; propose-and-wait пройден): инстанс, видящий через [Conclave](0006-cache-redis.md#adr-006-кэш-и-координация--redis) перекос своей нагрузки выше справедливой доли, сбрасывает **излишек** своих стримов (частичный `StreamManager.CloseAll`, не все — в отличие от Watchman) с jitter/cap → Souls реконнектятся и раскидываются. **Домен балансировки = priority-группа / один VIP** (разные priority = failover-иерархия, не баланс; внутри группы EventStream делает shuffle — см. [docs/soul/connection.md](../soul/connection.md); за одним VIP балансирует сам LB least-conn). Альтернатива/дополнение — пассивный max-connection-age рециклинг стримов (gRPC server `MaxConnectionAge` / soul `max_stream_age`+spray) — «тупой» фон с постоянной handshake-ценой, поэтому Shepherd предпочтительнее. **drain-режим** (авто-по-порогу / полу-авто / ручной) — TBD. **Реализация отложена** (backlog 2026-05-25); требует amend ADR-006 (Conclave несёт снимок нагрузки) + architect-дизайн + проверку взаимодействия с in-flight apply (ADR-027 Ward/recovery). Имя в словаре — [naming-rules.md](../naming-rules.md).
