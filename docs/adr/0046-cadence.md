# ADR-046. Cadence ‚Äî recurring runs (scheduled/recurring Voyage)

> **Status: canon BEFORE code (S0, 2026-06-01).** propose-and-wait passed (name + design of model b + interval/cron + three overlap policies approved by the user). Slices S1‚ÄìS5 ‚Äî implementation. There is not yet any code/proto/openapi/migrations for this ADR.

**Context.** [ADR-043](0043-voyage.md#adr-043-voyage--unified-batch-run) (Voyage) gives a **one-off** deferred start ‚Äî the field `schedule_at` (nullable; when present, the run enters the `scheduled` state and starts once at the specified moment). There is no **recurring** launch: "run converge every 5 minutes", "catalog-refresh by cron `0 */6 * * *`", "nightly drift-check at 02:00" the operator currently cannot express ‚Äî each run must be initiated by hand or by an external scheduler (a cron on the operator's side hitting `POST /v1/voyages`). This offloads the recurrence logic outward, loses the history "this series of runs is one entity", does not survive a Keeper-cluster restart as a single owner of the schedule, and does not integrate with Soul Stack's RBAC/audit. A **first-class schedule entity** is needed that spawns ordinary Voyages by time on its own.

**Decision (fixed by the user on 2026-06-01).** Introduce the entity **Cadence** ‚Äî a schedule that by time **spawns** an ordinary [Voyage](0043-voyage.md#adr-043-voyage--unified-batch-run) run. Cadence stores the "recipe" of the run (what and where to launch) + the repetition rule (interval or cron) + the overlap policy. The names `Rite` ([ADR-025](0025-augur.md#adr-025-augur--keeper-side-broker-for-soul-external-access)) and `Vigil` ([ADR-030](0030-vigil-oracle.md#adr-030-vigil--oracle--event-driven-monitoring-beacons--reactor)) are taken ‚Äî `Cadence` ("rhythm, measured beat") is free and continues the "musical-soul" line next to Choir/Voice.

1. **Model (variant b) ‚Äî Cadence is a separate entity that outlives the runs.** Cadence is a row in the Postgres table **`cadences`**, living independently of the runs. When the time comes, Cadence **spawns a new Voyage** (Insert into `voyages` + `voyage_targets`) ‚Äî the entire claim / Leg / batch / failover / audit mechanism of Voyage ([ADR-043](0043-voyage.md#adr-043-voyage--unified-batch-run), [ADR-027](0027-apply-work-queue.md#adr-027-apply-execution-model--work-queue--claim-acolyte-pool-ward-claim)) works **without changes**. The relation "parent Cadence ‚Üí child Voyages" (one-to-many). The Voyage invariant **"one Voyage = one run" IS PRESERVED**: Cadence creates **new** Voyage rows, it does **not resurrect or reuse** old ones. Manual (non-Cadence) runs live as before.
   - **Rejected variant (a) "recurrence fields directly in Voyage".** It would make Voyage both a single run and a schedule at once ‚Äî Voyage would cease to be "one run", breaking the ADR-043 invariant and its lifecycle (`scheduled ‚Üí pending ‚Üí running ‚Üí terminal`). A schedule is a **recipe**, not a run; it needs its own entity.

2. **Back-link `voyages.cadence_id`.** Voyage gets a **nullable** column `cadence_id` (FK to `cadences(id)`, without `ON DELETE CASCADE` ‚Äî deleting a Cadence does not carry away the history of the runs it spawned; see p.9). A manual run ‚Üí `cadence_id = NULL`; a spawn from a Cadence ‚Üí populated. This is an additive only-add column ([ADR-012 forward-compat only-add](0012-keeper-soul-grpc.md#adr-012-keepersoul-grpc-contract-one-eventstream-with-oneof-keeper-side-render-forward-compat-only-add) ‚Äî the style of an inverted back-link, like `voyage_targets.apply_id` in [ADR-027 amendment](0027-apply-work-queue.md#adr-027-apply-execution-model--work-queue--claim-acolyte-pool-ward-claim)): the link is stored on the child row, `voyages` stays orchestrator-agnostic regarding Cadence.

   - **Amend (T4-0, 2026-06-12): `cadence_id` reaches the run's terminal event.** A schedule's child Voyage passes `cadence_id` through the spawn chain (`voyages.cadence_id` ‚Üí `VoyageWorker` ‚Üí `ScenarioSpawner.SpawnScenarioRun` ‚Üí `scenario.RunSpec.CadenceID`), and `scenario.Runner` puts it into the payload of the run's terminal event `incarnation.run_completed` ([ADR-052 amend ¬ßk](0052-herald-notifications.md#adr-052-herald--tiding--notifications-about-run-events)) ‚Äî **only** when the run was spawned by a schedule (a manual run carries no `cadence_id` key). This lets a persistent [Tiding](0052-herald-notifications.md#adr-052-herald--tiding--notifications-about-run-events) rule with a cadence selector catch the **RESULTS** of a schedule's runs (`incarnation.run_completed` with `status: success`/`failed`), and not only spawned/skipped events (`cadence.spawned` / `cadence.skipped_overlap`, p.8).

3. **Fields of `cadences`.**
   - `id` (TEXT PK, ULID), `name`, `enabled` (bool ‚Äî pause without deletion).
   - **Repetition rule:** `schedule_kind` (`interval` | `cron`); **`interval_seconds`** (int, for `interval`) **XOR** **`cron_expr`** (TEXT, standard 5-field cron, for `cron`). Exactly one of the two is set per `schedule_kind` (the other NULL); a violation ‚Üí `422 validation-failed`.
   - **`overlap_policy`** (`skip` | `queue` | `parallel`) ‚Äî the behavior when the previous spawned Voyage is not yet terminal and the time for the next one has come (see p.5).
   - **Run recipe** (the same set as `VoyageCreateRequest` [ADR-043](0043-voyage.md#adr-043-voyage--unified-batch-run)): `kind` (`scenario` | `command`), `scenario_name` (for scenario) | `module` (for command), `target` (a choice from the creator's RBAC scope), `input`, batch settings (`batch_size` / `concurrency` / `batch_mode` / `batch_percent` / `fail_threshold` / `inter_batch_interval` / `inter_unit_interval` / `on_failure` / `require_alive` / `dry_run`). Cadence stores the recipe **as is** (a vault-ref in `input` is not revealed ‚Äî resolved just-in-time at the claim of the spawned Voyage, invariant A [ADR-027(f)](0027-apply-work-queue.md#adr-027-apply-execution-model--work-queue--claim-acolyte-pool-ward-claim)). `schedule_at` in the recipe **has no meaning** (the start moment is dictated by Cadence) ‚Äî it is ignored/forbidden at validation.
   - **Computed timings:** `next_run_at` (timestamptz ‚Äî when to spawn the next Voyage), `last_run_at` (timestamptz nullable ‚Äî the moment of the last spawn).
   - `created_by_aid` (FK `operators(aid)`), `created_at` / `updated_at`.

4. **Trigger ‚Äî a leader-elected executor (Conductor).** The spawn is driven by a **single-executor** leader ([ADR-006](0006-cache-redis.md#adr-006-cache-and-coordination--redis) Redis lease), which gives exactly-one-spawn per tick without a race between Keeper instances. On each tick: `SELECT ‚Ä¶ FROM cadences WHERE enabled AND next_run_at <= NOW() FOR UPDATE SKIP LOCKED` ‚Üí for each due row in **one PG transaction**: (apply `overlap_policy`, p.5) + Insert `voyages`/`voyage_targets` with `cadence_id` (if the policy allows a spawn) + recompute `next_run_at` + `last_run_at = NOW()`. The recompute of `next_run_at` is **anchored to the planned slot** (`next_run_at` BEFORE the recompute), not to `last_run_at`/`NOW()`: for `interval` ‚Äî `planned_slot + interval_seconds`, for `cron` ‚Äî the next cron moment by the parser from the planned slot. The anchor is precisely the planned slot, because `last_run_at = NOW()` in the same tx, and `last_run_at + interval` would again give a now-anchor (the tick arrives at `NOW() ‚âà planned_slot + Œµ`, the drift Œµ would accumulate in `next_run_at` and the ticker would miss the grid). **Missed-slot policy** (after downtime, the planned slot is in the past): `next_run_at` is recomputed to the **first future grid slot strictly after `NOW()`** ‚Äî one catch-up spawn per current due, WITHOUT re-spawning every missed slot (anti-storm); symmetric for cron and interval. The spawned Voyage enters the ordinary Voyage lifecycle (`pending` ‚Üí claim by the VoyageWorker ‚Üí `running` ‚Üí terminal); Cadence does **not** drive it further (a one-shot spawn, like `current_apply_id` in the Tide-spawn path of ADR-027).
   - **Executor ‚Äî Conductor, a separate leader-elected subsystem ([ADR-048](0048-conductor.md#adr-048-conductor--leader-elected-executor-of-cadence-schedules)).** **Amendment 2026-06-02:** initially (S0) the spawn was hung on the **Reaper leader** via a new Reaper rule `spawn_due_cadence` (`action: spawn`). This has been moved into a separate subsystem **Conductor** with its own Redis lease `conductor:leader` and its own tick-interval (`cadence_scheduler.interval`, ~15‚Äì30s) ‚Äî independent of `reaper.interval` (1h): the cleanup domain of Reaper and the scheduling domain of Cadence have different natural rhythms, a common 1h tick does not let Cadence spawn frequently (see [ADR-048 "Context"](0048-conductor.md#adr-048-conductor--leader-elected-executor-of-cadence-schedules)). **The spawn semantics (due selection, three `overlap_policy`, recompute of `next_run_at`, single-executor) is carried over unchanged** ‚Äî only the executor changes. After the extraction (C4) Reaper **loses** the rule `spawn_due_cadence` and `action: spawn`; the closed-enum Reaper action returns to `expire`/`delete`/`set_status`/`report`/`soft_delete`. Conductor is **default-ON when Redis is present** (a schedule without a working scheduler silently does not spawn ‚Äî a footgun guard); it is turned off per-Cadence via `enabled: false`, not by a global kill.

5. **`overlap_policy` ‚Äî three policies (the operator chooses at creation).** When `next_run_at` has come, but the previous Voyage spawned by this Cadence is not yet terminal (`pending`/`running`):
   - **`skip`** ‚Äî the spawn is skipped; the audit event `cadence.skipped_overlap` is written; `next_run_at` is recomputed anyway (the series does not "stick"). Protection against accumulating runs when the period is shorter than the run duration.
   - **`queue`** ‚Äî the spawn is postponed: the next Voyage starts only after the previous one's terminal (Cadence does not spawn while there is a live child; on the next due tick, if the previous one is still running, it stays waiting). Guarantees strict sequentiality without losing launches.
   - **`parallel`** ‚Äî the spawn happens regardless of the state of the previous ones: the new Voyage starts overlapping (several children of one Cadence can be `running` simultaneously). Workers/concurrency limits are at the level of the Voyage/Acolyte itself.
   - **All three ‚Äî an MVP goal** (the user's decision; `queue` is not postponed). The check "is there a live previous child" ‚Äî `EXISTS(SELECT 1 FROM voyages WHERE cadence_id=$1 AND status NOT IN (<terminal>))` in the same spawn tx.

6. **API ‚Äî CRUD `/v1/cadences`** (sketch, normalized in S4/S5):
   - `POST /v1/cadences` ‚Äî create (body = recipe + `schedule_kind`/`interval_seconds`|`cron_expr` + `overlap_policy`).
   - `GET /v1/cadences` / `GET /v1/cadences/{id}` ‚Äî list / detail.
   - `PATCH /v1/cadences/{id}` ‚Äî edit (incl. the `enabled` toggle ‚Äî pause/resume).
   - `DELETE /v1/cadences/{id}` ‚Äî delete the schedule (the history of child Voyages is preserved, p.9).
   - `GET /v1/cadences/{id}/runs` ‚Äî child Voyages (`WHERE cadence_id=$1`), **reuse of the Voyage DTO** ([ADR-043](0043-voyage.md#adr-043-voyage--unified-batch-run) All-runs view) ‚Äî the drill "schedule ‚Üí its runs ‚Üí their units".

7. **RBAC ‚Äî new permissions** (catalog-driven, [ADR-042](0042-backend-driven-ui.md#adr-042-backend-driven-dynamic-data-in-the-ui--the-ui-does-not-hardcode-dynamic-catalogs)): `cadence.create` / `cadence.list` / `cadence.update` / `cadence.delete` in `catalog.go` ‚Äî the UI fetches the catalog, does not hardcode. **A two-level guard (security-critical):** the `cadence.*` right controls the schedule itself; but the recipe spawns a Voyage, so upon **creating** a Cadence the creator must also have the corresponding Voyage permission by the recipe's `kind` (`incarnation.run` for `kind=scenario` / `errand.run` for `kind=command`, [ADR-043 ¬ß6](0043-voyage.md#adr-043-voyage--unified-batch-run)) ‚Äî otherwise Cadence would become a privilege-escalation bypass of RBAC. The target in the recipe is a choice from the creator's RBAC scope at the moment of creation (parity with ADR-043 ¬ß5). The spawn on a tick is executed **on behalf of the `created_by_aid`** of the Cadence (the spawned Voyage inherits this Archon as the initiator).

8. **Audit ‚Äî new event types:** `cadence.created` / `cadence.updated` / `cadence.deleted` (CRUD, write-path API) + `cadence.spawned` (the executor spawned a Voyage ‚Äî `{cadence_id, voyage_id, scheduled_for}`) + `cadence.skipped_overlap` (the `skip` policy skipped a spawn ‚Äî `{cadence_id, scheduled_for, reason: overlap}`). Written into the common audit pipeline ([ADR-022](0022-audit-pipeline.md#adr-022-audit-pipeline-storage-schema-retention)); spawn/skip ‚Äî with **`source: background`** (parity with `scry_background` ‚Äî a background periodic keeper rule, [ADR-022(b)](0022-audit-pipeline.md#adr-022-audit-pipeline-storage-schema-retention) / [reaper.md](../keeper/reaper.md)), `archon_aid: NULL` (a background keeper rule has no identified operator-initiator). **The audit source is preserved after the move to Conductor ([ADR-048](0048-conductor.md#adr-048-conductor--leader-elected-executor-of-cadence-schedules)):** although the executor of the spawn changes from the Reaper leader to the Conductor leader, the nature of the event does not change ‚Äî it is the same autonomous background initiative of the keeper, so it remains `source: background`; **a new source `scheduler` is NOT introduced** (the audit-source enum [ADR-022(b)](0022-audit-pipeline.md#adr-022-audit-pipeline-storage-schema-retention) is not extended). **Separating two fields (do NOT confuse):** the initiator-authorship of the recipe lives in `voyages.started_by_aid` of the child Voyage, which inherits the `created_by_aid` of the Cadence (see ¬ß7, spawn on behalf of the creator) ‚Äî but in the audit event spawn/skip itself `archon_aid` = `NULL`, because the write-path of this event is the autonomous background initiative of the keeper, not an operator call. The names are entered in [naming-rules.md ‚Üí Audit-events](../naming-rules.md#audit-events).

9. **Deleting a Cadence ‚âÝ deleting the history.** `DELETE /v1/cadences/{id}` removes the schedule, but the spawned Voyages remain (`cadence_id` FK without `ON DELETE CASCADE` ‚Äî either `ON DELETE SET NULL` so that orphan children stay in the All-runs view, or forbidding deletion when there are live children; the specifics ‚Äî S5). The Reaper retention of `voyages` (`purge_voyages`, if introduced) cleans children by its own rules independently of the Cadence cycle. **Exception ‚Äî persistent notify rules of the schedule form** ([ADR-052 ¬ßm](0052-herald-notifications.md#m-a-permanent-tiding-from-the-notify-block-in-the-cadence-form)): Tidings created from the `notify` block of the Cadence form are linked via `tidings.created_from_cadence_id` FK **`ON DELETE CASCADE`** (migration 074) and are **removed** together with the Cadence ‚Äî an orphaned auto-rule without the schedule that spawned it is meaningless (unlike the run history, which is self-valuable). Rules with the same `cadence` selector but `created_from_cadence_id = NULL` (manually created by the operator) are **preserved** when the schedule is removed ‚Äî the cascade distinguishes origin, not selector.

**Floor of the minimum period (amendment 2026-06-07, Pass B).** The minimum period of an interval Cadence is **30s** (`interval_seconds ‚â• poll_floor`, default floor = 30s, [ADR-048 "Adaptive interval"](0048-conductor.md#adr-048-conductor--leader-elected-executor-of-cadence-schedules)). Creating/changing a Cadence with `interval < floor` ‚Üí **reject (422 validation-failed)** with a hint: for reactive launch faster than 30s ‚Äî **Beacons** (Vigil/Oracle, [ADR-030](0030-vigil-oracle.md#adr-030-vigil--oracle--event-driven-monitoring-beacons--reactor)), not Cadence (Cadence is a timer schedule, not a reactive event trigger). cron Cadences (minute granularity, ‚â• 60s) do not fall under the floor ‚Äî `interval_seconds = NULL` (CHECK `interval_seconds IS NULL OR interval_seconds ‚â• 30`). The floor is set by a **single source** together with the adaptive polling ‚Äî config `cadence_scheduler.poll_floor` (not a hardcoded 30 in two places): the write-path floor-reject reads the same `ResolvedPollFloor` as the Conductor's `IntervalFn`. Relation to [ADR-048](0048-conductor.md#adr-048-conductor--leader-elected-executor-of-cadence-schedules): the floor is exactly the lower bound at which the adaptive step's clamp-floor does not "under-receive" ticks (polling does not drop below the floor, so a sub-floor period would not get more frequent polling anyway); the anti-storm grid-skip ([ADR-046 ¬ß4](#adr-046-cadence--recurring-runs-scheduledrecurring-voyage), `NextRunAnchored` missed-slot) ‚Äî **unchanged**. Three levels of defense (defence-in-depth): **(a) write-path** floor-reject in `cadence.ValidateIntervalFloor` (422 on CREATE/PATCH); **(b) DB CHECK** `cadences_interval_seconds_floor` (`interval_seconds IS NULL OR interval_seconds ‚â• 30`, under a separate name from the sane-bound `_positive`); **(c) OpenAPI** `interval_seconds.minimum: 30` (client-side hint). The migration carries a **pre-flight data guard** (RAISE on existing sub-30 rows BEFORE `ADD CONSTRAINT`) ‚Äî fail-fast, NOT a silent `UPDATE` (silently changing the period of someone else's schedule is unacceptable; the operator himself raises it to 30s / switches to cron / deletes it).

**Rejected alternatives.**
- **(a) Recurrence fields in Voyage itself** (variant a) ‚Äî breaks "one Voyage = one run" (see p.1).
- **(b) Only `overlap_policy: skip` in the MVP** ‚Äî the user explicitly required all three (`skip`/`queue`/`parallel`), the operator chooses at creation.
- **(c) Only interval OR only cron in the MVP** ‚Äî both at once (`schedule_kind`), the user required it.
- **(d) An external scheduler (the operator's cron ‚Üí `POST /v1/voyages`)** ‚Äî loses history-as-entity, the RBAC/audit integration, the single owner of the schedule via the Reaper leader.

**What is deferred (post-MVP).**
- `down` semantics/catch-up of ones missed during a long Keeper-cluster downtime (MVP: missed ticks are not "re-spawned" retroactively ‚Äî `next_run_at` is recomputed to the first future grid slot strictly after `NOW()`, one catch-up per current due, see ¬ß4 missed-slot policy).
- A per-Cadence limit on concurrent children under `overlap_policy: parallel` (MVP: the Voyage/Acolyte layer limits it).
- Timezone-aware cron (MVP: cron in UTC; a tz field ‚Äî an additive extension on request).
- A retention rule `purge_cadences` for old schedules ‚Äî a separate slice when volume appears (a schedule is an active entity, not history; it does not need age hygiene, see ¬ß9).

**`purge_voyages` ‚Äî IMPLEMENTED (amendment 2026-06-12, path to beta).** Initially this rule was listed here as deferred ("if introduced"). On the beta Souls, `voyages` is the only run-history table without retention (every manual launch and every Cadence spawn = a new row, growth without a ceiling), so the rule was made mandatory. Implementation:
- **Migration 075** (`075_create_purge_voyages.up.sql`) ‚Äî the SQL function `purge_voyages(max_age interval, batch_limit integer DEFAULT 1000)`: batch-delete of finished Voyages (`succeeded`/`failed`/`partial_failed`/`cancelled` with `finished_at IS NOT NULL`) older than `max_age`; does not touch `scheduled`/`pending`/`running`. Age ‚Äî from `finished_at` (parity with `purge_apply_runs`, 021). `voyage_targets` are carried away by `ON DELETE CASCADE` (059). The LIMIT parameter is named `batch_limit` (not `batch_size`), because `voyages` carries a column of the same name ‚Äî otherwise `LIMIT batch_size` would give `column reference ambiguous` (42702).
- **Go wrapper** `reaper.Purger.PurgeVoyages` (`keeper/internal/reaper/purger.go`) ‚Äî one batch through the common `callIntervalBatch` path of Reaper rules.
- **Cascades and the correlation invariant:** the soft links `voyage_targets.apply_id`/`errand_id` (to `apply_runs`/`errands`) and `tidings.voyage_id` (ephemeral) ‚Äî are NOT FKs to `voyages`, purge does not delete them and leaves no broken references (ephemeral Tidings are removed earlier by the rule `purge_orphan_ephemeral_tidings`). `voyages.cadence_id` ‚Üí `cadences` `ON DELETE SET NULL` (066): purge of children does NOT touch the schedule. The default window is aligned with `purge_apply_runs` (30d), so that the drill "voyage ‚Üí apply_runs" does not lose one of the sides.
