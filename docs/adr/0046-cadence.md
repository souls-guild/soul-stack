# ADR-046. Cadence — регулярные запуски (scheduled/recurring Voyage)

> **Статус: канон ДО кода (S0, 2026-06-01).** propose-and-wait пройден (имя + дизайн модели b + interval/cron + три overlap-политики утверждены пользователем). Слайсы S1–S5 — реализация. Кода/proto/openapi/миграций по этому ADR пока нет.

**Контекст.** [ADR-043](0043-voyage.md#adr-043-voyage--унифицированный-батчевый-прогон) (Voyage) даёт **разовый** отложенный старт — поле `schedule_at` (nullable; при наличии прогон входит в состояние `scheduled`, стартует один раз в указанный момент). **Повторяющегося** запуска нет: «прогонять converge каждые 5 минут», «catalog-refresh по cron `0 */6 * * *`», «nightly drift-check в 02:00» сейчас оператор не может выразить — каждый прогон надо инициировать руками или внешним планировщиком (cron на стороне оператора, дёргающий `POST /v1/voyages`). Это перекладывает recurrence-логику наружу, теряет историю «эта серия прогонов — одна сущность», не переживает рестарт Keeper-кластера как единый владелец расписания и не интегрируется с RBAC/audit Soul Stack-а. Нужна **первоклассная сущность расписания**, которая по времени сама спавнит обычные Voyage.

**Решение (зафиксировано пользователем 2026-06-01).** Ввести сущность **Cadence** — расписание, которое по времени **спавнит** обычный [Voyage](0043-voyage.md#adr-043-voyage--унифицированный-батчевый-прогон)-прогон. Cadence хранит «рецепт» прогона (что и где запускать) + правило повторения (interval или cron) + политику наложения. Имена `Rite` ([ADR-025](0025-augur.md#adr-025-augur--keeper-side-брокер-внешнего-доступа-soul)) и `Vigil` ([ADR-030](0030-vigil-oracle.md#adr-030-vigil--oracle--event-driven-мониторинг-beacons--reactor)) заняты — `Cadence` («ритм, размеренный такт») свободен и продолжает «музыкально-душевную» линию рядом с Choir/Voice.

1. **Модель (вариант b) — Cadence отдельная сущность, переживающая прогоны.** Cadence — строка в Postgres-таблице **`cadences`**, живёт независимо от прогонов. По наступлении времени Cadence **спавнит новый Voyage** (Insert в `voyages` + `voyage_targets`) — весь claim / Leg / batch / failover / audit-механизм Voyage ([ADR-043](0043-voyage.md#adr-043-voyage--унифицированный-батчевый-прогон), [ADR-027](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim)) работает **без изменений**. Отношение «родитель Cadence → дети Voyage» (one-to-many). Voyage-инвариант **«один Voyage = один прогон» СОХРАНЯЕТСЯ**: Cadence создаёт **новые** Voyage-строки, **не воскрешает и не переиспользует** старые. Ручные (не-Cadence) прогоны живут как раньше.
   - **Отвергнут вариант (a) «recurrence-поля прямо в Voyage».** Сделал бы Voyage и единичным прогоном, и расписанием одновременно — Voyage перестал бы быть «один прогон», ломая инвариант ADR-043 и его lifecycle (`scheduled → pending → running → terminal`). Расписание — это **рецепт**, а не прогон; ему нужна своя сущность.

2. **Back-link `voyages.cadence_id`.** Voyage получает **nullable** колонку `cadence_id` (FK на `cadences(id)`, без `ON DELETE CASCADE` — удаление Cadence не уносит историю порождённых прогонов; см. п.9). Ручной прогон → `cadence_id = NULL`; спавн от Cadence → populated. Это additive only-add колонка ([ADR-012 forward-compat only-add](0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add) — стиль инвертированного back-link-а, как `voyage_targets.apply_id` в [ADR-027 amendment](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim)): связь хранится на дочерней строке, `voyages` остаётся orchestrator-agnostic относительно Cadence.

   - **Amend (T4-0, 2026-06-12): `cadence_id` доезжает до терминального события прогона.** Дочерний Voyage расписания прокидывает `cadence_id` через цепочку спавна (`voyages.cadence_id` → `VoyageWorker` → `ScenarioSpawner.SpawnScenarioRun` → `scenario.RunSpec.CadenceID`), и `scenario.Runner` кладёт его в payload терминального события прогона `incarnation.run_completed` ([ADR-052 amend §k](0052-herald-notifications.md#adr-052-herald--tiding--уведомления-о-событиях-прогонов)) — **только** когда прогон спавнен расписанием (ручной прогон ключ `cadence_id` не несёт). Это позволяет постоянному [Tiding](0052-herald-notifications.md#adr-052-herald--tiding--уведомления-о-событиях-прогонов)-правилу с cadence-селектором ловить **РЕЗУЛЬТАТЫ** прогонов расписания (`incarnation.run_completed` со `status: success`/`failed`), а не только spawned/skipped-события (`cadence.spawned` / `cadence.skipped_overlap`, п.8).

3. **Поля `cadences`.**
   - `id` (TEXT PK, ULID), `name`, `enabled` (bool — пауза без удаления).
   - **Правило повторения:** `schedule_kind` (`interval` | `cron`); **`interval_seconds`** (int, для `interval`) **XOR** **`cron_expr`** (TEXT, стандартный 5-полевой cron, для `cron`). Ровно один из двух — задан по `schedule_kind` (второй NULL); нарушение → `422 validation-failed`.
   - **`overlap_policy`** (`skip` | `queue` | `parallel`) — поведение, когда предыдущий порождённый Voyage ещё не терминальный, а время следующего наступило (см. п.5).
   - **Рецепт прогона** (то же множество, что `VoyageCreateRequest` [ADR-043](0043-voyage.md#adr-043-voyage--унифицированный-батчевый-прогон)): `kind` (`scenario` | `command`), `scenario_name` (для scenario) | `module` (для command), `target` (выбор из RBAC-скоупа создателя), `input`, batch-настройки (`batch_size` / `concurrency` / `batch_mode` / `batch_percent` / `fail_threshold` / `inter_batch_interval` / `inter_unit_interval` / `on_failure` / `require_alive` / `dry_run`). Cadence хранит рецепт **как есть** (vault-ref в `input` не раскрывается — резолв just-in-time при claim спавнутого Voyage, инвариант A [ADR-027(f)](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim)). `schedule_at` в рецепте **не имеет смысла** (момент старта диктует Cadence) — игнорируется/запрещается на валидации.
   - **Расчётные тайминги:** `next_run_at` (timestamptz — когда спавнить следующий Voyage), `last_run_at` (timestamptz nullable — момент последнего спавна).
   - `created_by_aid` (FK `operators(aid)`), `created_at` / `updated_at`.

4. **Триггер — leader-elected исполнитель (Conductor).** Спавн ведёт **single-executor** лидер ([ADR-006](0006-cache-redis.md#adr-006-кэш-и-координация--redis) Redis-lease), что даёт ровно-один-спавн на тик без гонки между Keeper-инстансами. На каждом тике: `SELECT … FROM cadences WHERE enabled AND next_run_at <= NOW() FOR UPDATE SKIP LOCKED` → для каждой due-строки в **одной PG-транзакции**: (применить `overlap_policy`, п.5) + Insert `voyages`/`voyage_targets` с `cadence_id` (если политика разрешает спавн) + пересчёт `next_run_at` + `last_run_at = NOW()`. Пересчёт `next_run_at` **anchored к плановому слоту** (`next_run_at` ДО пересчёта), а не к `last_run_at`/`NOW()`: для `interval` — `плановый_слот + interval_seconds`, для `cron` — следующий cron-момент по парсеру от планового слота. Anchor — именно плановый слот, потому что `last_run_at = NOW()` в той же tx, и `last_run_at + interval` снова дал бы now-anchor (тик приходит на `NOW() ≈ плановый_слот + ε`, дрейф ε накапливался бы в `next_run_at` и тикер промахивался бы мимо сетки). **Missed-slot policy** (после простоя плановый слот в прошлом): `next_run_at` пересчитывается на **первый будущий слот сетки строго после `NOW()`** — один catch-up-спавн за текущий due, БЕЗ доспавна каждого пропущенного слота (anti-storm); симметрично cron и interval. Спавнутый Voyage входит в обычный Voyage-lifecycle (`pending` → claim VoyageWorker-ом → `running` → terminal); Cadence его дальше **не ведёт** (one-shot спавн, как `current_apply_id` в Tide-spawn-пути ADR-027).
   - **Исполнитель — Conductor, отдельная leader-elected подсистема ([ADR-048](0048-conductor.md#adr-048-conductor--leader-elected-исполнитель-cadence-расписаний)).** **Амендмент 2026-06-02:** изначально (S0) спавн вешался на **Reaper-лидера** новым Reaper-правилом `spawn_due_cadence` (`action: spawn`). Это вынесено в отдельную подсистему **Conductor** со своим Redis-lease `conductor:leader` и своим tick-interval (`cadence_scheduler.interval`, ~15–30s) — независимым от `reaper.interval` (1h): cleanup-домен Reaper и scheduling-домен Cadence имеют разный естественный ритм, общий тик 1h не даёт Cadence спавнить часто (см. [ADR-048 «Контекст»](0048-conductor.md#adr-048-conductor--leader-elected-исполнитель-cadence-расписаний)). **Spawn-семантика (due-выборка, три `overlap_policy`, пересчёт `next_run_at`, single-executor) переносится без изменений** — меняется только исполнитель. После выноса (C4) Reaper **теряет** правило `spawn_due_cadence` и `action: spawn`; closed-enum Reaper-action возвращается к `expire`/`delete`/`set_status`/`report`/`soft_delete`. Conductor — **default-ON при наличии Redis** (расписание без работающего планировщика молча не спавнит — footgun-guard); выключается per-Cadence через `enabled: false`, не глобальным гашением.

5. **`overlap_policy` — три политики (оператор выбирает при создании).** Когда `next_run_at` наступил, а предыдущий порождённый этой Cadence Voyage ещё не терминален (`pending`/`running`):
   - **`skip`** — спавн пропускается; пишется audit-event `cadence.skipped_overlap`; `next_run_at` всё равно пересчитывается (серия не «залипает»). Защита от накопления прогонов, когда период короче длительности прогона.
   - **`queue`** — спавн откладывается: следующий Voyage стартует только после терминала предыдущего (Cadence не спавнит, пока есть живой ребёнок; на следующем due-тике, если предыдущий ещё бежит, остаётся в ожидании). Гарантирует строгую последовательность без потери запусков.
   - **`parallel`** — спавн происходит независимо от состояния предыдущих: новый Voyage запускается наложенно (несколько детей одной Cadence могут быть `running` одновременно). Воркеры/concurrency-лимиты — на уровне самого Voyage/Acolyte.
   - **Все три — MVP-цель** (решение пользователя; `queue` не откладывается). Проверка «есть ли живой предыдущий ребёнок» — `EXISTS(SELECT 1 FROM voyages WHERE cadence_id=$1 AND status NOT IN (<terminal>))` в той же spawn-tx.

6. **API — CRUD `/v1/cadences`** (эскиз, нормируется в S4/S5):
   - `POST /v1/cadences` — создать (тело = рецепт + `schedule_kind`/`interval_seconds`|`cron_expr` + `overlap_policy`).
   - `GET /v1/cadences` / `GET /v1/cadences/{id}` — список / деталь.
   - `PATCH /v1/cadences/{id}` — правка (вкл. `enabled`-toggle — пауза/возобновление).
   - `DELETE /v1/cadences/{id}` — удалить расписание (история детей-Voyage сохраняется, п.9).
   - `GET /v1/cadences/{id}/runs` — дочерние Voyage (`WHERE cadence_id=$1`), **reuse Voyage-DTO** ([ADR-043](0043-voyage.md#adr-043-voyage--унифицированный-батчевый-прогон) All-runs вид) — drill «расписание → его прогоны → их единицы».

7. **RBAC — новые permission** (catalog-driven, [ADR-042](0042-backend-driven-ui.md#adr-042-backend-driven-dynamic-data-в-ui--ui-не-хардкодит-динамические-каталоги)): `cadence.create` / `cadence.list` / `cadence.update` / `cadence.delete` в `catalog.go` — UI фетчит каталог, не хардкодит. **Двухуровневый guard (security-критичный):** право `cadence.*` управляет самим расписанием; но рецепт спавнит Voyage, поэтому при **создании** Cadence создатель обязан иметь и соответствующий Voyage-permission по `kind` рецепта (`incarnation.run` для `kind=scenario` / `errand.run` для `kind=command`, [ADR-043 §6](0043-voyage.md#adr-043-voyage--унифицированный-батчевый-прогон)) — иначе Cadence стала бы privilege-escalation-обходом RBAC. Target в рецепте — выбор из RBAC-скоупа создателя на момент создания (parity ADR-043 §5). Спавн на тике исполняется **от имени `created_by_aid`** Cadence (порождённый Voyage наследует этого Архонта как инициатора).

8. **Audit — новые event-types:** `cadence.created` / `cadence.updated` / `cadence.deleted` (CRUD, write-path API) + `cadence.spawned` (исполнитель заспавнил Voyage — `{cadence_id, voyage_id, scheduled_for}`) + `cadence.skipped_overlap` (политика `skip` пропустила спавн — `{cadence_id, scheduled_for, reason: overlap}`). Пишутся в общий audit-pipeline ([ADR-022](0022-audit-pipeline.md#adr-022-audit-pipeline-storage-schema-retention)); spawn/skip — с **`source: background`** (parity `scry_background` — фоновое периодическое keeper-правило, [ADR-022(b)](0022-audit-pipeline.md#adr-022-audit-pipeline-storage-schema-retention) / [reaper.md](../keeper/reaper.md)), `archon_aid: NULL` (у фонового keeper-правила нет идентифицированного оператора-инициатора). **Audit-source сохраняется после переезда в Conductor ([ADR-048](0048-conductor.md#adr-048-conductor--leader-elected-исполнитель-cadence-расписаний)):** хотя исполнитель спавна меняется с Reaper-лидера на Conductor-лидера, природа события не меняется — это та же автономная фоновая инициатива keeper-а, поэтому остаётся `source: background`; **новый source `scheduler` НЕ вводится** (enum audit-source [ADR-022(b)](0022-audit-pipeline.md#adr-022-audit-pipeline-storage-schema-retention) не расширяется). **Разведение двух полей (НЕ путать):** инициатор-авторство рецепта живёт в `voyages.started_by_aid` дочернего Voyage, который наследует `created_by_aid` Cadence (см. §7, спавн от имени создателя) — но в самом audit-event spawn/skip `archon_aid` = `NULL`, потому что write-path этого события — автономная фоновая инициатива keeper-а, а не операторский вызов. Имена внесены в [naming-rules.md → Audit-events](../naming-rules.md#audit-events).

9. **Удаление Cadence ≠ удаление истории.** `DELETE /v1/cadences/{id}` снимает расписание, но порождённые Voyage остаются (`cadence_id` FK без `ON DELETE CASCADE` — либо `ON DELETE SET NULL`, чтобы дети-сироты оставались в All-runs виде, либо запрет удаления при живых детях; конкретика — S5). Reaper-retention `voyages` (`purge_voyages`, если будет введён) чистит детей по своим правилам независимо от Cadence-цикла. **Исключение — постоянные notify-правила формы расписания** ([ADR-052 §m](0052-herald-notifications.md#m-постоянный-tiding-из-блока-notify-формы-cadence)): Tiding-и, созданные из блока `notify` формы Cadence, привязаны через `tidings.created_from_cadence_id` FK **`ON DELETE CASCADE`** (миграция 074) и **сносятся** вместе с Cadence — осиротевшее автоправило без породившего расписания бессмысленно (в отличие от истории прогонов, которая самоценна). Правила с тем же `cadence`-селектором, но `created_from_cadence_id = NULL` (вручную заведённые оператором), при сносе расписания **сохраняются** — каскад различает происхождение, а не селектор.

**Floor минимального периода (амендмент 2026-06-07, Pass B).** Минимальный период interval-Cadence — **30s** (`interval_seconds ≥ poll_floor`, дефолт floor = 30s, [ADR-048 «Adaptive interval»](0048-conductor.md#adr-048-conductor--leader-elected-исполнитель-cadence-расписаний)). Создание/изменение Cadence с `interval < floor` → **reject (422 validation-failed)** с подсказкой: для реактивного запуска быстрее 30s — **Beacons** (Vigil/Oracle, [ADR-030](0030-vigil-oracle.md#adr-030-vigil--oracle--event-driven-мониторинг-beacons--reactor)), не Cadence (Cadence — расписание по таймеру, не реактивный event-trigger). cron-Cadence (гранулярность минута, ≥ 60s) под floor не попадают — `interval_seconds = NULL` (CHECK `interval_seconds IS NULL OR interval_seconds ≥ 30`). Floor задаётся **единым источником** с адаптивным опросом — config `cadence_scheduler.poll_floor` (не хардкод 30 в двух местах): write-path floor-reject читает тот же `ResolvedPollFloor`, что `IntervalFn` Conductor-а. Связь с [ADR-048](0048-conductor.md#adr-048-conductor--leader-elected-исполнитель-cadence-расписаний): floor — это и есть нижняя граница, при которой clamp-floor адаптивного шага не «недополучает» тики (опрос не опускается ниже floor, поэтому суб-floor период всё равно не получил бы более частого опроса); anti-storm grid-skip ([ADR-046 §4](#adr-046-cadence--регулярные-запуски-scheduledrecurring-voyage), `NextRunAnchored` missed-slot) — **без изменений**. Три уровня защиты (defence-in-depth): **(а) write-path** floor-reject в `cadence.ValidateIntervalFloor` (422 на CREATE/PATCH); **(б) DB-CHECK** `cadences_interval_seconds_floor` (`interval_seconds IS NULL OR interval_seconds ≥ 30`, отдельным именем от sane-bound `_positive`); **(в) OpenAPI** `interval_seconds.minimum: 30` (client-side hint). Миграция несёт **pre-flight data-guard** (RAISE при существующих суб-30 строках ПЕРЕД `ADD CONSTRAINT`) — fail-fast, НЕ тихий `UPDATE` (молчаливое изменение периода чужого расписания недопустимо; оператор сам поднимает до 30s / переводит в cron / удаляет).

Триггер реализуется подсистемой Conductor ([ADR-048](0048-conductor.md#adr-048-conductor--leader-elected-исполнитель-cadence-расписаний)) — новая keeper-side подсистема, ничего не ломает (в S0-дизайне планировался Reaper-`action: spawn`, вынесен в Conductor). Таблица `cadences` — новая, ничего не ломает. Proto Keeper↔Soul **не затрагивается** (Cadence — чисто keeper-side управляющая сущность; спавн идёт через существующий Voyage-путь, который уже шлёт `ApplyRequest`).

**Связь с другими ADR.**
- **[ADR-043](0043-voyage.md#adr-043-voyage--унифицированный-батчевый-прогон) (Voyage)** — Cadence **спавнит** Voyage, не подменяет его. Voyage-инвариант «один Voyage = один прогон» сохранён (новые строки, не воскрешение). Разовый `schedule_at` ADR-043 остаётся для одноразового отложенного старта; Cadence — для повторяющегося.
- **[ADR-048](0048-conductor.md#adr-048-conductor--leader-elected-исполнитель-cadence-расписаний) (Conductor)** — исполнитель триггера Cadence. **Амендмент 2026-06-02:** спавн вынесен из Reaper-лидера (`spawn_due_cadence`) в отдельную leader-elected подсистему Conductor со своим lease `conductor:leader` и tick-interval, независимым от `reaper.interval`. Spawn-семантика §4/§5 сохранена дословно; меняется только исполнитель. §4 и §8 этого ADR амендированы.
- **[ADR-027](0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim) / Reaper** — изначально `spawn_due_cadence` расширял перечень Reaper-правил; после [ADR-048](0048-conductor.md#adr-048-conductor--leader-elected-исполнитель-cadence-расписаний) правило **переезжает в Conductor**, Reaper возвращается к чистому cleanup-домену. claim/Ward/fencing спавнутого Voyage наследуются без изменений.
- **[ADR-006](0006-cache-redis.md#adr-006-кэш-и-координация--redis)** — single-executor спавна через лидера (Redis-lease); после [ADR-048](0048-conductor.md#adr-048-conductor--leader-elected-исполнитель-cadence-расписаний) — отдельный lease-ключ `conductor:leader` (не `reaper:leader`).
- **[ADR-022](0022-audit-pipeline.md#adr-022-audit-pipeline-storage-schema-retention)** — новые audit-event-семейства в общий pipeline; `source: background` сохраняется после переезда в Conductor (новый source не вводится).
- **[ADR-014](0014-operator-identity.md#adr-014-identity-модель-оператора-archon)** — `cadences.created_by_aid` FK на `operators(aid)`; спавн от имени создателя.

**Отвергнутые альтернативы.**
- **(а) Recurrence-поля в самой Voyage** (вариант a) — ломает «один Voyage = один прогон» (см. п.1).
- **(б) Только `overlap_policy: skip` в MVP** — пользователь явно затребовал все три (`skip`/`queue`/`parallel`), оператор выбирает при создании.
- **(в) Только interval ИЛИ только cron в MVP** — оба сразу (`schedule_kind`), пользователь затребовал.
- **(г) Внешний планировщик (cron оператора → `POST /v1/voyages`)** — теряет историю-как-сущность, RBAC/audit-интеграцию, single-owner расписания через Reaper-лидера.

**Что отложено (post-MVP).**
- `down`-семантика/catch-up пропущенных при долгом downtime Keeper-кластера (MVP: пропущенные тики не «доспавниваются» задним числом — `next_run_at` пересчитывается на первый будущий слот сетки строго после `NOW()`, один catch-up за текущий due, см. §4 missed-slot policy).
- Per-Cadence лимит конкурентных детей при `overlap_policy: parallel` (MVP: лимитирует Voyage/Acolyte-слой).
- Timezone-aware cron (MVP: cron в UTC; tz-поле — additive-расширение при запросе).
- Retention-правило `purge_cadences` для старых расписаний — отдельным слайсом при появлении объёма (расписание = активная сущность, не история; возрастная гигиена ей не нужна, см. §9).

**`purge_voyages` — РЕАЛИЗОВАН (амендмент 2026-06-12, путь к бете).** Изначально это правило значилось здесь как отложенное («если будет введён»). На бета-флоте `voyages` — единственная run-history таблица без retention (каждый ручной запуск и каждый спавн Cadence = новая строка, рост без потолка), поэтому правило сделано обязательным. Реализация:
- **Миграция 075** (`075_create_purge_voyages.up.sql`) — SQL-функция `purge_voyages(max_age interval, batch_limit integer DEFAULT 1000)`: batch-delete finished-Voyage (`succeeded`/`failed`/`partial_failed`/`cancelled` с `finished_at IS NOT NULL`) старше `max_age`; `scheduled`/`pending`/`running` не трогает. Возраст — от `finished_at` (parity `purge_apply_runs`, 021). `voyage_targets` уносятся `ON DELETE CASCADE` (059). Параметр LIMIT назван `batch_limit` (а не `batch_size`), т.к. `voyages` несёт одноимённую колонку — иначе `LIMIT batch_size` дал бы `column reference ambiguous` (42702).
- **Go-обёртка** `reaper.Purger.PurgeVoyages` (`keeper/internal/reaper/purger.go`) — один batch через общий `callIntervalBatch`-путь Reaper-правил.
- **Каскады и инвариант корреляции:** soft-link-и `voyage_targets.apply_id`/`errand_id` (на `apply_runs`/`errands`) и `tidings.voyage_id` (ephemeral) — НЕ FK на `voyages`, purge их не удаляет и битых ссылок не оставляет (ephemeral-Tiding-и снимаются раньше правилом `purge_orphan_ephemeral_tidings`). `voyages.cadence_id` → `cadences` `ON DELETE SET NULL` (066): purge детей НЕ трогает расписание. Default-окно выровнено на `purge_apply_runs` (30d), чтобы drill «voyage → apply_runs» не терял одну из сторон.
