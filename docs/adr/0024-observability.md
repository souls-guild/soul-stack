# ADR-024. Observability: Prometheus-primary + OTel-bridge

- **Контекст.** [`docs/requirements.md`](../requirements.md) фиксирует два сквозных требования — «у всех публикация метрик» и «поддержка OpenTelemetry из коробки». В коде `shared/obs` уже есть Prometheus-registry с HTTP-метриками (`keeper_http_*`, snake_case, `_total`/`_seconds`, dedicated registry), но **не нормировано**: какой канал телеметрии первичен (Prometheus pull vs OTel push), как называть метрики на разных компонентах, и как доменная идентичность инстанса (KID / SID) попадает в трейсы. Без этого имплементация Soul-side метрик и OTel-bridge будет угадывать конвенции и может разойтись с уже написанным Keeper-кодом.
- **Решение.**
  - **(a) Prometheus-primary + OTel-bridge.** Метрики первично экспонируются Prometheus-эндпоинтом `/metrics` (pull) — de-facto-стандарт, и Keeper уже имеет Prometheus-registry в `shared/obs`. **OpenTelemetry — мост** для трейсов (сквозная propagation оператор → Keeper → Soul через gRPC-метаданные) и **опционального** push метрик в OTLP-коллектор (для инсталляций без Prometheus-scrape). Не «OTel-primary»: OTLP-метрики — дополнительный канал, не замена scrape.
  - **(b) Namespace метрик — раздельные префиксы по компоненту:** `keeper_*` (Keeper-side) и `soul_*` (Soul-side). Различение **префиксом, а не label** `component=`: короче, греппабельнее, совпадает со стандартом Prometheus (per-exporter namespace). Имена — snake_case, единица в суффиксе (`_seconds` / `_bytes`, не `_ms`), `_total` для counters. Конкретный каталог метрик — **не нормируется здесь**, наполняется по факту имплементации подсистем (как [каталог audit-events](../naming-rules.md#audit-events)).
  - **(c) OTel resource-attributes — `service.name` + кастомные `soulstack.kid` / `soulstack.sid`.** Стандартный `service.name` = `"keeper"` \| `"soul"` (имя бинаря по словарю) + кастомные `soulstack.kid` ([KID](../naming-rules.md#идентификаторы)) и `soulstack.sid` ([SID](../naming-rules.md#идентификаторы)) для доменной идентичности инстанса в трейсах. Generic `service.instance.id` не даёт фильтра «трейсы конкретного Soul по FQDN» в терминах словаря системы. Высокая кардинальность KID/SID — причина класть их в OTel resource-attrs, **не** в Prometheus metric-labels (cardinality-blow-up).
- **Что нормирует, что откладывает.** Нормирует конвенции и архитектуру; **не нормирует** конкретный список метрик (по факту имплементации), типизацию `otel:`-блока конфига (config.md каждого бинаря, отдельная задача), OpenMetrics-format и exemplars (post-MVP). Полная спека — [`docs/observability.md`](../observability.md).
- **Consequences.**
  - **Новый файл [`docs/observability.md`](../observability.md)** — cross-cutting спека (один файл на оба бинаря, симметрично `shared/obs` как одному Go-модулю; обоснование «не per-binary» — в самом файле).
  - **Два новых имени в [`docs/naming-rules.md`](../naming-rules.md)** — namespace-префиксы `soul_*` / `keeper_*` и кастомные OTel resource-attrs `soulstack.kid` / `soulstack.sid`.
  - **`shared/obs` OTel-bridge** (trace-provider, OTLP-exporter, resource-attrs) — точка расширения существующего модуля, отдельная задача имплементации. Существующий Prometheus-код (`keeper_http_*`) уже консистентен с ADR — переименований не требует.
  - Раздел [«Сквозные требования»](../architecture.md#сквозные-требования-и-где-они-приземляются) расшифровывает «Метрики» / «OpenTelemetry» через ссылку на ADR-024.
- **Trade-offs.**
  - **Prometheus-primary vs OTel-primary.** Prometheus pull — зрелее в configuration-management-нише, проще debug (`curl /metrics`), у Keeper уже написан. OTel-primary потребовал бы коллектор как обязательную зависимость даже для одной метрики. Цена Prometheus-primary — метрики и трейсы идут разными каналами (нет единого OTLP-pipeline); закрывается OTel-bridge для тех, кому нужен единый push.
  - **Префикс vs label для компонента.** Префикс (`keeper_`/`soul_`) проигрывает «общему имени с label `component=`» в унификации дашбордов, но выигрывает в греппабельности и совпадает с per-exporter-стандартом Prometheus. Один scrape-таргет = один компонент, общего имени и не нужно.
