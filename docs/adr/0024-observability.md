# ADR-024. Observability: Prometheus-primary + OTel-bridge

- **Context.** [`docs/requirements.md`](../requirements.md) fixes two cross-cutting requirements — "metrics publishing for everyone" and "OpenTelemetry support out of the box." In the code, `shared/obs` already has a Prometheus registry with HTTP metrics (`keeper_http_*`, snake_case, `_total`/`_seconds`, a dedicated registry), but it is **not standardized**: which telemetry channel is primary (Prometheus pull vs OTel push), how to name metrics across different components, and how an instance's domain identity (KID / SID) reaches traces. Without this, implementing Soul-side metrics and the OTel-bridge would be guessing at conventions and could drift from the Keeper code already written.
- **Decision.**
  - **(a) Prometheus-primary + OTel-bridge.** Metrics are primarily exposed via the Prometheus `/metrics` endpoint (pull) — the de-facto standard, and Keeper already has a Prometheus registry in `shared/obs`. **OpenTelemetry is a bridge** for traces (end-to-end propagation operator → Keeper → Soul via gRPC metadata) and for an **optional** metrics push to an OTLP collector (for installations without Prometheus scrape). Not "OTel-primary": OTLP metrics are an additional channel, not a replacement for scrape.
  - **(b) Metrics namespace — separate prefixes per component:** `keeper_*` (Keeper-side) and `soul_*` (Soul-side). Distinguished by **prefix, not a `component=` label**: shorter, more greppable, matches the Prometheus standard (per-exporter namespace). Names — snake_case, a unit in the suffix (`_seconds` / `_bytes`, not `_ms`), `_total` for counters. The concrete metrics catalog is **not standardized here** — it fills in as subsystems are implemented (like the [audit-events catalog](../naming-rules.md#audit-events)).
  - **(c) OTel resource-attributes — `service.name` + custom `soulstack.kid` / `soulstack.sid`.** The standard `service.name` = `"keeper"` \| `"soul"` (the binary name per the dictionary) + custom `soulstack.kid` ([KID](../naming-rules.md#identifiers)) and `soulstack.sid` ([SID](../naming-rules.md#identifiers)) for an instance's domain identity in traces. A generic `service.instance.id` doesn't give a "traces for a specific Soul by FQDN" filter in the system dictionary's terms. The high cardinality of KID/SID is the reason to put them in OTel resource-attrs, **not** in Prometheus metric labels (cardinality blow-up).
- **What it standardizes, what it defers.** Standardizes conventions and architecture; **does not standardize** the concrete metrics list (fills in as implemented), typing of the config's `otel:` block (each binary's config.md, a separate task), the OpenMetrics format, and exemplars (post-MVP). Full spec — [`docs/observability.md`](../observability.md).
- **Consequences.**
  - **A new file [`docs/observability.md`](../observability.md)** — a cross-cutting spec (one file for both binaries, symmetric to `shared/obs` as one Go module; the "not per-binary" rationale is in the file itself).
  - **Two new names in [`docs/naming-rules.md`](../naming-rules.md)** — the namespace prefixes `soul_*` / `keeper_*` and the custom OTel resource-attrs `soulstack.kid` / `soulstack.sid`.
  - **The `shared/obs` OTel-bridge** (trace-provider, OTLP-exporter, resource-attrs) — an extension point of the existing module, a separate implementation task. The existing Prometheus code (`keeper_http_*`) is already consistent with the ADR — no renames needed.
  - The [end-to-end requirements](../architecture.md#end-to-end-requirements-and-where-they-land) section spells out "Metrics" / "OpenTelemetry" via a reference to ADR-024.
- **Trade-offs.**
  - **Prometheus-primary vs OTel-primary.** Prometheus pull is more mature in the configuration-management niche, easier to debug (`curl /metrics`), and Keeper already has it written. OTel-primary would require a collector as a mandatory dependency even for a single metric. The cost of Prometheus-primary — metrics and traces travel over different channels (no unified OTLP pipeline); this is closed by the OTel-bridge for those who need a unified push.
  - **A prefix vs a label for the component.** A prefix (`keeper_`/`soul_`) loses to "a shared name with a `component=` label" on dashboard unification, but wins on greppability and matches Prometheus's per-exporter standard. One scrape target = one component, no shared name needed.
- **Amendment ([ADR-067](0067-vector-log-shipping.md), the mandatory Vector log plane).** The observability model is extended with a **third dimension — logs (push)**: the [Vector](0067-vector-log-shipping.md) agent ships data-service logs to a central collector alongside metrics (Prometheus pull, §a) and traces/opt. OTLP metrics (OTel-bridge). The three layers are independent and don't overlap (metrics answer "is the service alive/loaded," logs answer "what exactly happened"). Mandatory on every data service (the `vector` destiny, a clone of the node-exporter reference; wired in via an unconditional `apply: destiny` at the end of `create`). The namespace conventions (§b) don't extend to logs — Vector has its own `sources → transforms → sinks` model. Full record — [ADR-067](0067-vector-log-shipping.md).
