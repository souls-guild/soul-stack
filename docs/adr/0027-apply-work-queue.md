# ADR-027. Модель исполнения apply — work-queue + claim (Acolyte-пул, Ward-claim)

> **Phase 0 / 1 / 2 + GATE-1 (deliver-once recovery) реализованы; Phase 3 (распределённый serial) — отложен.** Acolyte-пул, Ward-claim (`FOR UPDATE SKIP LOCKED`), Summons (`apply:summons` + poll-fallback), перерендер при claim (`keeper/internal/scenario/claim.go`, `keeper/internal/applyrun`), recovery-скан Reaper (`reclaim_apply_runs`, `keeper/internal/reaper`), attempt-fencing двусторонний (proto `ApplyRequest.attempt` + `RunResult.attempt` field 4 + Soul-guard + epoch-check на приёме), single-winner state-commit — в коде. **Lifecycle после GATE-1: `planned → claimed → dispatched → terminal`** (фаза `dispatched` записывается перед SendApply; `running` — vestigial, только старый синхронный путь; см. amend «GATE-1: deliver-once recovery» в конце ADR). **Recovery-скан `reclaim_apply_runs` по дефолту ВЫКЛЮЧЕН** (`enabled:false`): включение — операционный шаг под гейтом (после GATE-1 — раскатка fencing-Soul + S2–S5 done; см. Consequences + amend (e)). Распределённый serial-барьер (Phase 3) кодом не покрыт — serial клеймится целиком одним Acolyte.

- **Контекст.** Сейчас прогон исполняется **run-goroutine в памяти одного Keeper-инстанса** — того, который принял dispatch. Состояние барьера / serial / cancel-поллинга живёт в этой goroutine; в Postgres лежит только корреляция (`apply_runs`, `apply_task_register`, [storage.md](../keeper/storage.md)). Это даёт **дыру «висячий applying»**: если инстанс-владелец падает в середине прогона, `incarnation.status` остаётся `applying` навсегда, строки `apply_runs` зависают в `running`, и **никто** не подхватывает прогон — другие инстансы про эту goroutine не знают. Владение исполнением не персистентно и не передаётся. Нужна модель, где владение каждым per-host заданием **материализовано в Postgres** и может быть перехвачено живым инстансом после смерти владельца.
- **Решение.** Исполнение apply переводится на **work-queue + claim**: задания (per-host строки `apply_runs`) лежат в Postgres, пул воркеров на каждом инстансе атомарно клеймит их.
  - **(a) Сущности.** **Acolyte** — пул воркеров исполнения на каждом Keeper-инстансе (`keeper.acolytes:N`), заменяет одиночную run-goroutine; Acolyte клеймит задание, резолвит и рендерит его just-in-time, гонит apply и держит барьер для своего среза. **Ward** — claim (под-опека) задания: набор колонок `claim_by_kid` / `claim_at` / `claim_expires_at` / `attempt` + статусы `planned` / `claimed` на `apply_runs`; «взять Ward» = захватить задание. **Summons** — pub/sub-сигнал «появились planned-задания» (Redis, best-effort) + periodic poll fallback (Acolyte всё равно периодически сканирует `planned`, так что потеря сигнала не теряет задание). Имена внесены в [naming-rules.md](../naming-rules.md), propose-and-wait пройден.
  - **(b) Запись прогона.** scenario-runner на dispatch-е пишет строки `apply_runs` в статус `planned` (вместо прямого `running`) и шлёт `Summons`. Никакого in-memory-владения на этом шаге нет — задание уже персистентно и доступно любому Acolyte любого инстанса.
  - **(c) Just-in-time render.** Acolyte рендерит задание **в момент claim**, не на dispatch-е. В Postgres хранится **рецепт** прогона (scenario-ref + `input` с vault-ref **как есть**), не раскрытые секреты — см. инвариант A.
  - **(d) Claim — атомарный захват.** Acolyte берёт Ward через `UPDATE apply_runs SET status='claimed', claim_by_kid=$me, claim_at=NOW(), claim_expires_at=NOW()+lease, attempt=attempt+1 WHERE status='planned' … ` с семантикой **`SELECT … FOR UPDATE SKIP LOCKED`** (захват без блокировки конкурентов; занятые строки пропускаются). `RETURNING` отдаёт захваченные задания этому Acolyte. Конкурирующие Acolyte-ы разных инстансов не дерутся за одну строку.
  - **(e) claimed → running.** Получив Ward, Acolyte резолвит секреты + рендерит (инвариант A), отправляет `ApplyRequest` Soul-у и переводит задание `claimed → running`. RunResult-handler, как и прежде, переводит `running` в терминал (`success` / `failed` / `cancelled`).
  - **(f) Инвариант A — секреты не в plaintext.** В Postgres лежит **рецепт** (scenario-ref + `input` с vault-ref как есть), **не** раскрытые значения. Резолв (`vault(...)`, CEL) и рендер делаются **just-in-time при claim**; раскрытые секреты живут только в RAM Acolyte-а и на проводе (mTLS) до Soul-а. При **перехвате Ward** (recovery) новый Acolyte **перерендеривает** задание заново из рецепта — старый раскрытый материал нигде не персистится и не переезжает. Это **уточняет [ADR-012(d)](0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add)**: рендер остаётся Keeper-side, но привязан к моменту claim, а не dispatch, и повторяется при recovery.
  - **(g) Инвариант B — нет двойного исполнения (fencing).** Перехват протухшего Ward не должен дать **два** живых apply на один хост. Защита двухслойная: (1) **SID-lease** — один Keeper-инстанс владеет стримом данного Soul-а ([ADR-006](0006-cache-redis.md#adr-006-кэш-и-координация--redis)), apply на хост идёт только через владельца стрима; (2) **Soul-side guard** — Soul отвергает дубль / устаревший `ApplyRequest` по `apply_id` + `attempt`-epoch. **Не полагаемся на идемпотентность модулей.** Soul-guard требует **only-add proto-поле `attempt`** в `ApplyRequest` ([ADR-012(c)](0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add) forward-compat only-add, Phase 2); без него fencing деградирует до защиты по `apply_id` (отсекает дубль того же attempt, но не отличает stale-attempt).
  - **(h) Инвариант C — serial.** Не-serial прогон → **per-host claim** (каждый хост — отдельный Ward, параллелит весь пул). serial-прогон → задание клеймится **целиком одним Acolyte** (барьер / serial-окно держит он один, как run-goroutine сейчас). **Распределённый serial** (несколько Acolyte-ов кооперативно держат один serial-барьер) — **отдельный ADR**, отложен (Phase 3). barrier / serial-семантика [ADR-009](0009-scenario-dsl.md#adr-009-scenario--полная-dsl-задач-destiny-граница-с-destiny--рекомендация) сохранена.
  - **(i) Recovery-скан Reaper-лидера.** ⚠️ *До-GATE-1 формулировка; предикат сужен — см. amend (c)/(b).* Reaper-лидер ([ADR-006](0006-cache-redis.md#adr-006-кэш-и-координация--redis), [reaper.md](../keeper/reaper.md)) периодически сканирует `apply_runs` на **протухшие Ward** (после GATE-1: `status='claimed'`, было `IN ('claimed','running')`) `AND claim_expires_at < NOW()` → возвращает в `planned` (сбрасывает `claim_by_kid` / `claim_at` / `claim_expires_at`). Это **закрывает дыру «висячий applying»** на фазе **до отдачи** (недоставленное задание): строка мёртвого владельца снова попадает в очередь и её подхватит живой Acolyte; `attempt++` на следующем claim отсекает stale-`ApplyRequest` на Soul-е (инвариант B). **`dispatched` (отданное) НЕ реклеймится** (amend (c)). Имя Reaper-правила — `reclaim_apply_runs` (propose-and-wait пройден).
  - **(j) Single-winner state-commit.** Перевод `incarnation` из `applying` в терминал по итогам прогона — атомарный single-winner: `UPDATE incarnation SET status=… WHERE status='applying' RETURNING …`. Только один обработчик «выигрывает» строку; при гонке recovery / RunResult двойного коммита state не происходит.
  - **(k) Case 2 — изолированный Keeper.** Инстанс, потерявший связь с PG / Redis, но ещё живой для части Soul-ов: его **health-probe становится unhealthy** → LB уводит трафик; **soul-shedding** — инстанс сбрасывает свои Soul-стримы, Soul-ы переподключаются к рабочему Keeper-у (алгоритм подключения [ADR-002](0002-transport-grpc-ha.md#adr-002-транспорт-keeper--souls--grpc-bidirectional-stream-поверх-mtls-ha-кластер-keeper)) + retry. **Промоут БД-реплики — ВНЕ Soul Stack** (Patroni / облачный failover, [ADR-005](0005-storage-postgres.md#adr-005-хранилище-состояния-keeper--postgres)). Вариант «просить реплики переключиться» (failover БД средствами Keeper) **отклонён**; принят soul-shedding (Keeper не управляет своей БД-топологией). — **РЕАЛИЗОВАНО 2026-05-25** (propose-and-wait имена закрыты пользователем): (1) **Conclave** — Redis-реестр живых Keeper-инстансов (`keeper:instance:<kid>`, TTL 30s + renew 10s, паттерн lease.go; инвариант presence→Redis, [ADR-006](0006-cache-redis.md#adr-006-кэш-и-координация--redis)); даёт `CountLive` для refuse-guard. (2) **Watchman** — изоляция-детект: периодический ping PG+Redis (`/readyz`-набор) + debounce (изоляция после N подряд фейлов, дефолт 3) → **hard-close** локальных EventStream-стримов (`StreamManager.CloseAll` отменяет per-stream ctx; receive-loop вынесен в горутину, чтобы `stream.Recv()` был отменяем) → Soul получает EOF → reconnect по failback-list ([ADR-002](0002-transport-grpc-ha.md#adr-002-транспорт-keeper--souls--grpc-bidirectional-stream-поверх-mtls-ha-кластер-keeper); Soul-failback уже реализован). (3) **refuse-guard** — при `acolytes:0` + `Conclave.CountLive>1` Keeper отказывается стартовать (явный opt-out `allow_unsafe_single_path_multi_keeper` / env), закрывает Finding-A (см. (h)). **go-away отвергнут** — чистого EOF достаточно, proto Keeper↔Soul не трогается. Soul multi-endpoint failback — dev-config (стенд мега-теста хардкодил один endpoint). Имена Sentinel (коллизия Redis Sentinel) и Vigil (занято) отвергнуты.
- **Реконсиляция ADR.** [ADR-002](0002-transport-grpc-ha.md#adr-002-транспорт-keeper--souls--grpc-bidirectional-stream-поверх-mtls-ha-кластер-keeper) **усиливается** — stateless-кластер теперь распределяет ещё и **исполнение** (не только приём стримов): любой инстанс через Acolyte подхватывает любое задание. [ADR-005](0005-storage-postgres.md#adr-005-хранилище-состояния-keeper--postgres) — владение исполнением материализуется в **PG-строке** (`apply_runs` Ward-колонки), не в памяти. [ADR-006](0006-cache-redis.md#adr-006-кэш-и-координация--redis) — Summons на pub/sub, recovery-скан на Reaper-лидере; **claim-lease живёт в Postgres**, не в Redis (lease задания — холодное персистентное владение, в отличие от SID-lease). [ADR-009](0009-scenario-dsl.md#adr-009-scenario--полная-dsl-задач-destiny-граница-с-destiny--рекомендация) — barrier / serial сохранены; serial клеймится целиком. [ADR-012(d)](0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add) — уточняется: рендер привязан к claim и перерендеривается при recovery.
- **Consequences.**
  - **Миграция `apply_runs`** — аддитивная: новые статусы `planned` / `claimed`, новые колонки `claim_by_kid` / `claim_at` / `claim_expires_at` / `attempt`, плюс колонка **`recipe jsonb`** (см. ниже). Старый путь (прямой `running`) продолжает работать до Phase 1 (Phase 0).
  - **Acolyte-пул** на каждом инстансе (`keeper.acolytes:N` в [config.md](../keeper/config.md)) — заменяет одиночную run-goroutine.
  - **Перерендер при claim / recovery** — рендер переезжает с dispatch-а на claim; recovery перерендеривает из рецепта. **Рецепт прогона** (scenario-ref + `input` с vault-ref **как есть**, без раскрытых секретов) материализуется в колонке **`apply_runs.recipe jsonb`** (G4, propose-and-wait пройден — выбрана колонка, а не отдельная таблица: рецепт 1:1 с заданием, claim уже тянет строку). Инвариант A: секреты резолвятся только в RAM Acolyte при claim, в `recipe` — vault-ref как есть.
  - **Only-add proto-поле `attempt`** в `ApplyRequest` (`proto/keeper/v1/apply.proto`, **field 4** — реализовано, Phase 2) — fencing-epoch для Soul-guard ([ADR-012(c)](0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add) forward-compat only-add). Echo из `apply_runs.attempt` на момент claim: `ClaimNext` инкрементит его при каждом захвате Ward (`scenario/claim.go` кладёт `run.Attempt` в `ApplyRequest`), поэтому пере-claim протухшего задания приезжает с **бóльшим** attempt, чем оригинал. Soul-guard (`runtime.ApplyRunner`, `lastSeenAttempt map[apply_id]int32`, per-process) отвергает `ApplyRequest` с `attempt < lastSeenAttempt` — отсекает stale-дубль мёртвого Ward, чей apply ещё в полёте; отвергнутый дубль молча дропается (barrier-B1), оригинальный (бóльший attempt) закрывается своим `RunResult`. **`attempt = 0`/пусто = старый Keeper без fencing** (forward-compat): guard такой запрос не отвергает, fencing деградирует до защиты по `apply_id` + SID-lease. never-reuse field-номера сохранён. Метрика `soul_apply_fenced_total`.
  - **Reaper-правило recovery-скана — `reclaim_apply_runs`** (имя закреплено, propose-and-wait пройден; реализовано в `keeper/internal/reaper`): `UPDATE apply_runs SET status='planned' … WHERE status='claimed' AND claim_expires_at < NOW()` (после GATE-1; **было** `status IN ('claimed','running')` — см. amend (c)) (сброс `claim_by_kid`/`claim_at`/`claim_expires_at`; **`attempt` НЕ сбрасывается** — следующий `ClaimNext` его инкрементит). **Миграции не требует** — колонки `claim_expires_at`/`attempt` и partial-индекс `apply_runs_claim_scan_idx` пришли с миграцией 025. **По дефолту `enabled:false`** (см. ниже). Метрика — через `ObserveRule` (`keeper/internal/reaper`).
  - **Single-winner state-commit — вариант W1 (реализовано).** Перевод `incarnation` `applying`→терминал остаётся в run-goroutine, но защищён SQL-guard-ом: `UPDATE incarnation SET status=… WHERE name=$1 AND status IN ('applying','destroying') RETURNING …` (0 строк = строка уже финализирована/удалена → no-op, как destroy-S-D3). Дополнительно терминал `apply_runs` — append-only (терминал не перезаписывается терминалом). При гонке recovery/RunResult двойного commit-а state не происходит. **Вариант W2** (commit переносится в Acolyte) **увязан с Phase 3** distributed barrier — не сейчас.
  - **Lease-инвариант (граница применимости recovery).** ⚠️ *До-GATE-1 формулировка — ОСЛАБЛЕНА, см. amend (e).* До GATE-1 корректный recovery требовал `acolyte_lease > max(время-apply-одного-хоста, runTimeout)`, иначе reclaim возвращал в `planned` **живой** Ward (`running`) → второй SendApply. **После GATE-1 это неактуально**: reclaim берёт только `claimed` (фаза рендера/claim, ДО отдачи), живой долгий apply сидит в `dispatched` (reclaim не трогает). Достаточно `acolyte_lease > max-РЕНДЕР` — текущий дефолт **30s ок**, пересмотр lease и Acolyte lease-renew **для recovery-границы не требуются**.
  - **Включение recovery — гейт (операционный шаг, не одиночный `enabled:true`).** ⚠️ *До-GATE-1: 3-условный гейт — СМЯГЧЁН, см. amend (e).* После GATE-1 гейт: (а) **fencing-Soul раскатан по флоту** (все Soul-агенты понимают `ApplyRequest.attempt` — guard на исполнении + эхо `RunResult.attempt` для epoch-check на приёме) + (б) **слайсы S2–S5 (`dispatched`-фаза + mark-перед-send + reclaim-only-claimed + epoch-на-приёме) done** (есть). Условие «`acolyte_lease > max-apply` ИЛИ Acolyte lease-renew» — **снято** (lease-инвариант ослаблен). Перед включением — отдельный architect re-review связки. До выполнения гейта одиночный `enabled:true` опасен (старый не-fenced путь, см. amend (f)).
  - **Три слоя защиты от двойного исполнения (нормативно).** Безопасный recovery опирается на три независимых слоя, каждый закрывает свою стадию: **fencing** (`attempt`-epoch) — защита **ИСПОЛНЕНИЯ** на Soul-е от *меньшего* (устаревшего) attempt; **SID-lease** ([ADR-006](0006-cache-redis.md#adr-006-кэш-и-координация--redis)) — дедуп **ДОСТАВКИ** одного и того же epoch (apply на хост идёт только через владельца стрима); **single-winner** — защита **COMMIT** state (один обработчик выигрывает строку `incarnation`). Все три нужны одновременно — ни один не покрывает стадию другого: fencing не дедуплицирует доставку, SID-lease не различает attempt, single-winner не мешает двойному apply на хост.
  - **Summons** — Redis pub/sub-topic для сигнала planned-заданий; topic — **`apply:summons`** (зафиксировано, propose-and-wait пройден).
  - **Имена в [naming-rules.md](../naming-rules.md)** — Acolyte / Ward / Summons (+ определения).
  - **Обновление [storage.md](../keeper/storage.md)** — описание `apply_runs` (статусы + Ward-колонки + жизненный цикл; после GATE-1 — `planned → claimed → dispatched → terminal`, `running` vestigial, см. amend (a)).
  - **Full-roster render при claim (Phase 1 cutover).** Acolyte при claim рендерит **полный roster прогона** (как run-goroutine: `LoadIncarnationHosts` → весь roster → `Render` с `Hosts`=весь roster), затем берёт срез своего хоста `groupByHost(tasks, plans)[sid]`. Это устраняет диалект per-host↔full-roster: single-host render (`Hosts`=[host]) молча расходился со старым путём — `run_once` не резался (`applyRunOnce` при `len(targeted)≤1` возвращает таргет как есть → задача на ВСЕ хосты вместо одного), `soulprint.hosts`/`.where` и `incarnation.host_count` схлопывались до roster-а из одного хоста, cross-host `state_changes.sets` теряли соседей. Список full-roster-зависимостей **открытый** — поэтому воспроизводится полный roster, а не точечные guard-ы. SID-валидация «хост ещё в roster-е» (disconnected/revoked между dispatch и claim → ошибка) сохранена ДО render-а. Инвариант A не меняется: input-секреты scenario-level (резолвятся раз, не per-host), full-roster НЕ увеличивает секретный материал в RAM.
  - **KNOWN GAP «stuck-claimed» (Phase 1).** SendApply прошёл, но `MarkClaimedRunning` (claimed → running) упал (PG-сбой) — задание залипает в `claimed` навсегда. Локально **не чинится**: без attempt-fencing (Phase 2) переотправка / сброс в `planned` рискует **двойным SendApply** (apply уже ушёл на Soul). **Закрывается Phase 2**: recovery-скан Reaper (протухший Ward `claimed` → `planned`, (i)) + attempt-epoch отсекает stale-`ApplyRequest` на Soul-е ((g)).
  - **KNOWN GAP «`lastSeenAttempt` unbounded».** Soul-guard держит `lastSeenAttempt map[apply_id]int32` per-process без эвикции — карта растёт на каждый новый `apply_id` за время жизни процесса Soul-а. Для MVP приемлемо (apply-ы редки относительно жизни процесса, рестарт обнуляет — а после рестарта дубля физически нет). Кандидат на закрытие — TTL/LRU-эвикция записей. Отложен.
  - **KNOWN GAP «Acolyte lease-renew отсутствует».** ⚠️ *Неактуален для recovery-границы после GATE-1 (amend (e)).* Acolyte не продлевает `claim_expires_at` во время долгого apply — Ward держит фиксированный lease с момента claim. До GATE-1 это был корень lease-инварианта; **после GATE-1** reclaim берёт только `claimed` (фаза рендера/claim, не весь apply), поэтому фиксированный lease на recovery-корректность **не влияет** (долгий apply сидит в `dispatched`, reclaim не трогает). Renew — больше не блокер recovery; остаётся опциональным улучшением для предсказуемости claim-таймингов.

- **Поэтапность.**
  - **Phase 0 — миграция `apply_runs`** (аддитивна: статусы + Ward-колонки; старый путь работает) — **реализовано**.
  - **Phase 1 — Acolyte-пул + claim (`FOR UPDATE SKIP LOCKED`) + Summons + перерендер при claim** — **реализовано**.
  - **Phase 2 — recovery-скан Reaper (`reclaim_apply_runs`, `enabled:false` под гейтом) + fencing (proto `attempt` field 4 + Soul-guard) + single-winner state-commit (W1)** — **реализовано**. Case 2 (soul-shedding, (k)) — **РЕАЛИЗОВАНО 2026-05-25** (Conclave-реестр + Watchman изоляция-детект + EOF-shedding + refuse-guard, см. (k)); graceful-drain пула Acolyte при shutdown — реализован отдельно (`acolyte_drain_grace`).
  - **GATE-1 (deliver-once recovery) — реализовано** (см. amend выше): фаза `dispatched` + миграция 040, claimed→dispatched **перед** SendApply, reclaim сужен до `claimed`, epoch-check на приёме `RunResult.attempt` (proto only-add field 4 + метрика `keeper_runresult_stale_total`). Закрывает окно двойного apply из Phase 2. Soul-also-dead — known-gap, post-MVP через Soul-reconcile (amend (g)).
  - **Phase 3 — serial** (распределённый serial-барьер) — **отложен**, отдельным ADR.
- **Отложенное (post-MVP).** **Распределённый serial** (несколько Acolyte-ов на один serial-барьер) — отдельный ADR. **Durable Summons / outbox** (гарантированная доставка сигнала вместо best-effort pub/sub + poll) — при необходимости; poll-fallback покрывает потерю сигнала в MVP.
- **Trade-offs.**
  - **Перерендер при claim / recovery** — цена за инвариант «секреты не в plaintext в PG»: рендер дороже (повторяется при перехвате), но раскрытый материал нигде не персистится. Принято.
  - **claim-lease в Postgres, не в Redis** — владение заданием холодное и должно переживать рестарт Redis; цена — claim-операции бьют в PG (но это `UPDATE … SKIP LOCKED`, дешёвый), а не в горячий Redis.
  - **serial = claim целиком одним Acolyte в MVP** — теряем параллелизм внутри serial-прогона; принято как осознанное упрощение, распределённый serial отложен в Phase 3.
  - **Soul-fencing зависит от proto-поля `attempt`** — до Phase 2 fencing деградирует до защиты по `apply_id`; принято (SID-lease уже отсекает большинство дублей, `attempt` закрывает stale-attempt).
  - **Full-roster render при claim — O(N²) per-host CEL + N×per-host vault на прогон.** Каждый из N claim-ов рендерит **полный** roster (N хостов), поэтому суммарная стоимость рендера прогона = O(N²) CEL-вычислений и N×per-host vault-резолвов vs O(N) старого пути (run-goroutine рендерит roster один раз). Цена за устранение диалекта per-host↔full-roster (см. Consequences «Full-roster render при claim»): список full-roster-зависимостей открытый, точечные guard-ы fail-open. Приемлемо в **Phase 1** (единицы-десятки хостов); **сотни хостов** → кандидат на оптимизацию (симметрично «распределённый serial — Phase 3»). Принято.

> **Amend (GATE-1: deliver-once recovery, 2026-05-25).** Передизайн recovery-фазы (реализован, прошёл architect + review). Закрывает окно двойного apply, обнаруженное в Phase 2: claimed→running отмечался **после** SendApply — crash между отправкой и отметкой оставлял строку `claimed`, которую recovery-скан пере-клеймил → второй apply на тот же хост. Ниже — итоговая модель; формулировки (i), (j), lease-инварианта и гейта recovery-enable выше **заменены** на нижеследующие (старый текст оставлен помеченным как до-GATE-1, во избежание потери истории; при расхождении источник правды — этот amend).
>
> - **(a) Lifecycle apply_runs обновлён.** Цепочка фаз: **`planned → claimed → dispatched → terminal`** (`success` / `failed` / `cancelled`). Новая фаза **`dispatched`** — intent-маркер «задание отдано Soul-у»; записывается **АТОМАРНО** (коммит в PG) **СТРОГО ПЕРЕД SendApply** (`MarkDispatched` до отправки, не после). Статус **`running` СОХРАНЁН** в CHECK (no-backcompat-чистки нет), но в Acolyte-флоу **больше НЕ используется** (vestigial): его пишет только старый синхронный путь (`acolytes:0`, `dispatchWave → Insert(running)`). Миграция **040** расширяет CHECK (`status IN (… , dispatched)`); `down-040` корректна. Имя `dispatched` зафиксировано (propose-and-wait закрыт пользователем 2026-05-25; мотив — совпадает с уже используемым в коде `dispatch`/`grpc.apply_dispatch`, нулевой когнитивный разрыв) — внесено в [naming-rules.md](../naming-rules.md).
> - **(b) Перенос отметки до отправки (корень фикса).** Раньше claimed→running отмечался **ПОСЛЕ** SendApply → гонка: crash в окне между send и отметкой оставлял строку `claimed`, recovery-скан её пере-клеймил → **ДВОЙНОЙ apply**. Теперь claimed→`dispatched` **ПЕРЕД** SendApply это устраняет. Окна смерти Acolyte-пути после фикса: **W-a** (`MarkDispatched` ок, crash **до** SendApply) → строка `dispatched`, задание не доставлено, reclaim его **не трогает** → **безопасный ВИСЯК** (НЕ двойной apply); закрытие — Soul-reconcile, post-MVP (см. (g)). **W-b** (старый разрыв «отметка после send») — **УСТРАНЁН**.
> - **(c) reclaim сужен до `claimed`.** `reclaim_apply_runs` теперь `WHERE status='claimed'` (было `IN ('claimed','running')`). **Природа правила сменилась**: «добить **НЕДОСТАВЛЕННОЕ** задание» (владелец умер **до отдачи**, на фазе рендера/claim), а НЕ «реанимировать зомби-`running`». Фаза **`dispatched` НЕ реклеймится** — после отдачи прогоном владеет Soul; реклейм отданного = двойной apply.
> - **(d) Epoch-check на ПРИЁМЕ результата (симметрия fencing).** attempt-fencing становится **двусторонним**: (1) защита **исполнения** в Soul (`AcceptAttempt`, было в (g)) + (2) защита **ПРИЁМА** результата в Keeper. `RunResult` несёт эхо `attempt` (proto **only-add field 4**, [ADR-012(c)](0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add) forward-compat). `correlateRunResult` сверяет с `apply_runs.attempt`: `recvAttempt < rowAttempt` (и `!= 0`) → результат **устаревшей** попытки → **stale-drop** (метрика `keeper_runresult_stale_total`), state **НЕ коммитится**; `==` / `0` (forward-compat: старый Soul без эха) / `>` (невозможный, fail-safe + defensive warn-лог) → commit. Результат принимается на **ЛЮБОМ** Keeper-инстансе (Soul failback, [ADR-002](0002-transport-grpc-ha.md#adr-002-транспорт-keeper--souls--grpc-bidirectional-stream-поверх-mtls-ha-кластер-keeper)) — авторитет actuality в **общей PG** (`apply_runs.attempt`), а не в памяти инстанса. **Три ортогональных слоя** защиты от двойного исполнения (заменяют формулировку «Три слоя защиты» выше, дополняя её приёмом): **fencing исполнения** (Soul отвергает *меньший* attempt) / **SID-lease дедуп доставки** ([ADR-006](0006-cache-redis.md#adr-006-кэш-и-координация--redis)) / **single-winner commit + epoch-drop приёма** (Keeper). epoch-check на приёме ортогонален single-winner: epoch отсекает *устаревший* attempt, single-winner — *повторный терминал того же* attempt.
> - **(e) Lease-инвариант ОСЛАБЛЕН.** recovery-enable **больше НЕ требует** `acolyte_lease > max-apply` и **не требует** Acolyte-lease-renew. Достаточно `acolyte_lease > max-**РЕНДЕР**` (текущий дефолт **30s** — ок), потому что reclaim берёт только `claimed` (фаза рендера/claim **ДО** отдачи), а не весь apply: живой долгий apply сидит в `dispatched`, который reclaim не трогает. Это **снимает** старую формулировку lease-инварианта и **смягчает** гейт recovery-enable. Гейт теперь: **раскатка fencing-Soul** (нужна и для приёма RunResult-эха) + слайсы S2–S5 done. (Условие «`acolyte_lease > max-apply` ИЛИ Acolyte lease-renew» — снято; KNOWN GAP «Acolyte lease-renew отсутствует» выше становится неактуальным для recovery-границы.)
> - **(f) ДВА KNOWN-GAP (зафиксированы явно, чтобы не «починили» обратно).**
>   - **Старый синхронный путь без Reaper-recovery.** При `acolytes:0` (прод-дефолт; Acolyte — opt-in) задания пишутся прямо в `running`, и сужение reclaim до `claimed` означает, что эти `running`-строки Reaper **НЕ восстанавливает**. Это **НЕ регресс**: recovery старого пути — это in-memory run-goroutine инстанса-владельца (+`runTimeout`), а не Reaper-reclaim; при смерти владельца его прогон и так не восстанавливался (свойство до-Acolyte модели). **`reclaim_apply_runs` и `acolytes > 0` — связанная пара: включать reclaim при `acolytes:0` ЗАПРЕЩЕНО** — на не-fenced (старом) пути reclaim сам по себе небезопасен (двойной apply без epoch-защиты).
>   - **Cancel-видимость `dispatched`.** Guard `RequestCancel` **НЕ включает** `dispatched` (by design в pilot): уже-отданным Soul-ам Keeper `CancelApply` **НЕ шлёт** (best-effort cancel доставленных прогонов — отложенный admin-API слой через `Outbound.SendCancel`). barrier-чтение (`SelectStatusesByApplyID` по `apply_id`) `dispatched` **видит** — терминала дожидается корректно. Полное закрытие = отдельный под-слайс (`guard += 'dispatched'` **ВМЕСТЕ** с emit `CancelApply` на Soul + приём `RunStatus.CANCELLED`), свести с Soul-reconcile (см. (g)).
> - **(g) Soul-also-dead (Keeper и Soul оба мертвы после отдачи).** Окно, которое сам deliver-once **не закрывает**: `dispatched` зависнет — `RunResult` не придёт, reclaim строку не трогает. **РЕШЕНИЕ:** закрытие в **post-MVP через Soul-reconcile** — Soul при reconnect сообщает реально ведомые `apply_id`, Keeper сверяет и терминалит осиротевшие `dispatched`. На **MVP** — документированный known-gap (оператор перезапускает прогон вручную). **Reaper dispatched-timeout НЕ делаем** (terminal-by-timeout без подтверждения от Soul небезопасен). — **РЕАЛИЗОВАНО 2026-05-25 (S6 Soul-reconcile), known-gap снят:** Soul на (re)connect шлёт `WardRoster` (ReplaceAll-снимок ведомых `apply_id`+attempt; новое `FromSoul` only-add сообщение, oneof field 8 — [ADR-012(k)](0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add)) сразу после Hello; Keeper в `OrphanDispatched` epoch-fenced single-winner-сверкой терминалит `dispatched`-строки этого SID, которых нет в наборе → новый терминал **`orphaned`** (миграция 044; barrier классифицирует как fail → incarnation `error_locked`). Набор Soul-а — in-memory (active ∪ recently-finished ring TTL 30s; рестарт Soul → пуст → его dispatched законно сиротятся). Старый Soul без `WardRoster` → sweep no-op (fail-safe; под тем же гейтом раскатки fencing-Soul, что recovery-enable (e)).
> - **(h) Amendment (2026-05-25, multi-keeper guard).** Инвариант сделан **явным**: run-goroutine-путь (`acolytes:0`) — **single-keeper-only**; **N>1 Keeper-инстансов требует `acolytes>0`**. Live multi-keeper тест подтвердил footgun: при `acolytes:0` apply, созданный на keeper-A с Soul-ом на стриме keeper-B, отрабатывает на хосте (`apply_runs=success`), но incarnation **навсегда виснет в `applying`** — in-memory владелец на keeper-A не наблюдает `RunResult`, ушедший на keeper-B. Guard — **warn, не refuse** (без новой сущности реестра инстансов): (1) startup-advisory при `acolytes:0`; (2) точечный `WARN` в run-goroutine dispatch-пути, когда SID-lease целевого Soul-а принадлежит **другому KID** (footgun стрельнёт ровно здесь). Refuse-startup-guard + **реестр живых Keeper-инстансов** — **РЕАЛИЗОВАНЫ 2026-05-25** (реестр = **Conclave**, см. (k)): при `acolytes:0` + `Conclave.CountLive>1` Keeper отказывается стартовать (явный opt-out `allow_unsafe_single_path_multi_keeper` / env `KEEPER_ALLOW_UNSAFE_MULTI_KEEPER`); dispatch-time WARN оставлен runtime safety-net поверх refuse. Инвариант зафиксирован в [docs/keeper/config.md](../keeper/config.md).
> - **(i) Amendment (2026-05-25, FINDING-01: apply_runs не over-reports нецелевые).** На Acolyte-пути (`acolytes>0`) `dispatchPlanned` пишет `apply_runs`-строку на ВЕСЬ roster incarnation ДО резолва `on:`/`where:` (резолв — per-host при claim); barrier держит `wantHosts = len(roster)`. Раньше хост, у которого `on:`/`where:` отфильтровал все задачи (0 tasks), закрывался как `success` (no-op) → `apply_runs` ложно показывал «успех на всех» (выявлено HA-приёмкой Фаза 2: incarnation 9 хостов, `on:[cluster-a]` → задачи реально 3, но 9× success). Фикс: нецелевой no-op-хост → новый терминал **`no_match`** (миграция 045; `classify` засчитывает `no_match` как benign-терминал на success-сторону — barrier досчитывает `wantHosts`, incarnation `ready`; `failed`/`cancelled`/`orphaned` рядом по-прежнему fail-closed). `apply_runs` наружу по API сейчас НЕ отдаётся (history = `state_history`), потому impact был LOW; `no_match` делает запись честной до появления `/runs`-эндпоинта. **Остаточный диалект** (НЕ фикшен этим слайсом): старый путь `acolytes:0` нецелевые строки не материализует вовсе (target-only), Acolyte-путь пишет весь roster (`success`+`no_match`); полная унификация числа строк между режимами — отдельное решение.

> - **Extension 2026-05-27 (Tide-spawn apply_runs — back-link на `tides`).** [ADR-040](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override) вводит **второй путь** материализации строк `apply_runs`, не нарушающий инварианты этого ADR.
>
>   **Два пути spawn-а apply_runs:**
>
>   - **Классический (single-run, `tide_id IS NULL`):** scenario-runner на dispatch `POST /v1/incarnations/.../scenarios/.../run` **без** `wave`-поля пишет один `apply_run` на хост в `planned` и шлёт `Summons` — Acolyte-флоу работает как описано (a)–(j) выше.
>   - **Tide-путь (`tide_id` + `surge_index` populated):** **TideWorker** (отдельный pool, см. ниже) per-Surge спавнит N apply_run-строк **в одной PG-транзакции** с UPDATE `tides.current_surge_index=N` + `current_apply_id=Y` — same-tx invariant [ADR-040 amendment 2026-05-27](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override). Дальше Acolyte-флоу неотличим от классического (Acolyte claim → render → dispatch → terminal): Tide-orchestrator знает про эти `apply_run`-строки **только** через `current_apply_id` поле, не управляет их жизненным циклом напрямую.
>
>   **`apply_runs.tide_id` / `surge_index` — nullable back-link:**
>
>   - `tide_id` — nullable FK на `tides(tide_id)`; `surge_index` — int (nullable, всегда вместе с `tide_id`).
>   - Classic single-run: оба `NULL`.
>   - Tide-spawn: оба populated; композиция `(tide_id, surge_index)` идентифицирует Surge внутри Tide.
>   - **Reaper apply_runs-cleanup (`purge_apply_runs` / `reclaim_apply_runs`) НЕ зависит от `tide_id`** — apply_run-цикл независим от Tide-цикла. `apply_run` Surge-а чистится по своим правилам (max_age от `finished_at`); сам Tide живёт отдельным циклом и чистится `reclaim_tides` (recovery протухших claim-ов). Нет каскадного `ON DELETE` — `apply_runs.tide_id` без `ON DELETE`-clause (default NO ACTION); удаление `tides`-строки post-MVP (retention `tides`) потребует отдельного решения по orphan-apply_runs.
>
>   **TideWorker pool — отдельный от Acolyte:**
>
>   - **Не общий пул** — deadlock-prevention: при `acolytes: 1` + 1 Tide занял Acolyte-слот, Surge-у внутри Tide некуда было бы попасть. TideWorker и Acolyte конкурируют только за PG-соединения, не за worker-слоты.
>   - Конфигурация — отдельный блок `keeper.yml::tide:` (рекомендация — `tide_workers ≤ acolytes-1` per-instance). См. [`docs/keeper/config.md`](../keeper/config.md).
>   - Цикл воркера простой: `claim → orchestrate → poll/sleep`. Tide-операции редкие крупные (один Tide = N Surges за минуты-часы), без сложной wake-логики Acolyte-пути.
>
>   **Инварианты ADR-027 не меняются:**
>
>   - Lifecycle apply_run: `planned → claimed → dispatched → terminal` — тот же для обоих путей (Tide-spawn попадает в `planned` с готовым `tide_id`/`surge_index`).
>   - Fencing-эпоха (`attempt`-fencing исполнения + epoch-check приёма) — работает идентично; Tide-orchestrator не вмешивается в attempt-увеличение Acolyte.
>   - Single-winner state-commit — `UPDATE incarnation … WHERE status='applying'` остаётся корнем защиты; per-Surge state-commit реализуется отдельным SQL-guard-ом на стороне Tide-orchestrator (commit-only при подтверждённом lease ownership через `FinalizeWithOwnership`, [ADR-040 amendment 2026-05-27](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override)).
>   - SID-lease + Soul-fencing для дедупа доставки — также неизменны.
