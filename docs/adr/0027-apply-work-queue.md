# ADR-027. Apply execution model ‚Äî work-queue + claim (Acolyte-pool, Ward-claim)

> **Phase 0 / 1 / 2 + GATE-1 (deliver-once recovery) implemented; Phase 3 (distributed serial) ‚Äî deferred.** Acolyte-pool, Ward-claim (`FOR UPDATE SKIP LOCKED`), Summons (`apply:summons` + poll-fallback), re-render on claim (`keeper/internal/scenario/claim.go`, `keeper/internal/applyrun`), Reaper recovery scan (`reclaim_apply_runs`, `keeper/internal/reaper`), two-sided attempt-fencing (proto `ApplyRequest.attempt` + `RunResult.attempt` field 4 + Soul-guard + epoch-check on receive), single-winner state-commit ‚Äî in the code. **Lifecycle after GATE-1: `planned ‚Üí claimed ‚Üí dispatched ‚Üí terminal`** (the `dispatched` phase is written before SendApply; `running` is vestigial, only the old synchronous path; see the "GATE-1: deliver-once recovery" amend at the end of the ADR). **Recovery scan `reclaim_apply_runs` is DISABLED by default** (`enabled:false`): enabling it is an operational step behind a gate (after GATE-1 ‚Äî fencing-Soul rollout + S2‚ÄìS5 done; see Consequences + amend (e)). The distributed serial barrier (Phase 3) is not covered by code ‚Äî a serial run is claimed whole by a single Acolyte.

- **Context.** Currently a run is executed by a **run-goroutine in the memory of a single Keeper instance** ‚Äî the one that accepted the dispatch. The barrier / serial / cancel-polling state lives in this goroutine; Postgres holds only the correlation (`apply_runs`, `apply_task_register`, [storage.md](../keeper/storage.md)). This creates the **"hanging applying" hole**: if the owner instance dies mid-run, `incarnation.status` stays `applying` forever, `apply_runs` rows hang in `running`, and **nobody** picks up the run ‚Äî other instances don't know about this goroutine. Execution ownership is not persistent and is not handed off. We need a model where ownership of each per-host task is **materialized in Postgres** and can be taken over by a live instance after the owner's death.
- **Decision.** Apply execution is moved to a **work-queue + claim**: tasks (per-host `apply_runs` rows) sit in Postgres, and a pool of workers on each instance atomically claims them.
  - **(a) Entities.** **Acolyte** ‚Äî the execution worker pool on each Keeper instance (`keeper.acolytes:N`), replacing the single run-goroutine; an Acolyte claims a task, resolves and renders it just-in-time, drives apply, and holds the barrier for its slice. **Ward** ‚Äî the claim (sub-guardianship) of a task: the set of columns `claim_by_kid` / `claim_at` / `claim_expires_at` / `attempt` + the statuses `planned` / `claimed` on `apply_runs`; "take a Ward" = capture the task. **Summons** ‚Äî a pub/sub signal "planned tasks have appeared" (Redis, best-effort) + a periodic poll fallback (the Acolyte still periodically scans `planned`, so losing the signal does not lose the task). The names are recorded in [naming-rules.md](../naming-rules.md), propose-and-wait passed.
  - **(b) Run record.** The scenario-runner, on dispatch, writes `apply_runs` rows in the `planned` status (instead of directly `running`) and sends a `Summons`. There is no in-memory ownership at this step ‚Äî the task is already persistent and available to any Acolyte on any instance.
  - **(c) Just-in-time render.** The Acolyte renders the task **at the moment of claim**, not at dispatch. Postgres stores the **recipe** of the run (scenario-ref + `input` with vault-refs **as-is**), not the unwrapped secrets ‚Äî see invariant A.
  - **(d) Claim ‚Äî atomic capture.** The Acolyte takes a Ward via `UPDATE apply_runs SET status='claimed', claim_by_kid=$me, claim_at=NOW(), claim_expires_at=NOW()+lease, attempt=attempt+1 WHERE status='planned' ‚Ä¶ ` with **`SELECT ‚Ä¶ FOR UPDATE SKIP LOCKED`** semantics (capture without blocking competitors; busy rows are skipped). `RETURNING` hands the captured tasks to this Acolyte. Competing Acolytes on different instances do not fight over the same row.
  - **(e) claimed ‚Üí running.** Having received a Ward, the Acolyte resolves secrets + renders (invariant A), sends an `ApplyRequest` to the Soul, and moves the task `claimed ‚Üí running`. The RunResult handler, as before, moves `running` to a terminal (`success` / `failed` / `cancelled`).
  - **(f) Invariant A ‚Äî no plaintext secrets.** Postgres holds the **recipe** (scenario-ref + `input` with vault-refs as-is), **not** the unwrapped values. Resolution (`vault(...)`, CEL) and render are done **just-in-time at claim**; unwrapped secrets live only in the Acolyte's RAM and on the wire (mTLS) to the Soul. On a **Ward takeover** (recovery) the new Acolyte **re-renders** the task anew from the recipe ‚Äî the old unwrapped material is nowhere persisted and does not migrate. This **refines [ADR-012(d)](0012-keeper-soul-grpc.md#adr-012-–∫–æ–Ω—Ç—Ä–∞–∫—Ç-keepersoul-grpc-–æ–¥–∏–Ω-eventstream-—Å-oneof-keeper-side-—Ä–µ–Ω–¥–µ—Ä-forward-compat-only-add)**: render stays Keeper-side but is bound to the moment of claim, not dispatch, and is repeated on recovery.
  - **(g) Invariant B ‚Äî no double execution (fencing).** Taking over a stale Ward must not produce **two** live applies on one host. Protection is two-layered: (1) **SID-lease** ‚Äî a single Keeper instance owns the stream of a given Soul ([ADR-006](0006-cache-redis.md#adr-006-–∫—ç—à-–∏-–∫–æ–æ—Ä–¥–∏–Ω–∞—Ü–∏—è--redis)), and apply to a host goes only through the stream owner; (2) **Soul-side guard** ‚Äî the Soul rejects a duplicate / stale `ApplyRequest` by `apply_id` + `attempt`-epoch. **We do not rely on module idempotency.** The Soul-guard requires an **only-add proto field `attempt`** in `ApplyRequest` ([ADR-012(c)](0012-keeper-soul-grpc.md#adr-012-–∫–æ–Ω—Ç—Ä–∞–∫—Ç-keepersoul-grpc-–æ–¥–∏–Ω-eventstream-—Å-oneof-keeper-side-—Ä–µ–Ω–¥–µ—Ä-forward-compat-only-add) forward-compat only-add, Phase 2); without it fencing degrades to protection by `apply_id` (which cuts off a duplicate of the same attempt but does not distinguish a stale-attempt).
  - **(h) Invariant C ‚Äî serial.** A non-serial run ‚Üí **per-host claim** (each host is a separate Ward, parallelized across the whole pool). A serial run ‚Üí the task is claimed **whole by a single Acolyte** (the barrier / serial window is held by that one, like the run-goroutine today). **Distributed serial** (several Acolytes cooperatively holding one serial barrier) ‚Äî a **separate ADR**, deferred (Phase 3). The barrier / serial semantics of [ADR-009](0009-scenario-dsl.md#adr-009-scenario--–ø–æ–ª–Ω–∞—è-dsl-–∑–∞–¥–∞—á-destiny-–≥—Ä–∞–Ω–∏—Ü–∞-—Å-destiny--—Ä–µ–∫–æ–º–µ–Ω–¥–∞—Ü–∏—è) are preserved.
  - **(i) Recovery scan by the Reaper leader.** ‚öÝÔ∏è *Pre-GATE-1 wording; the predicate is narrowed ‚Äî see amend (c)/(b).* The Reaper leader ([ADR-006](0006-cache-redis.md#adr-006-–∫—ç—à-–∏-–∫–æ–æ—Ä–¥–∏–Ω–∞—Ü–∏—è--redis), [reaper.md](../keeper/reaper.md)) periodically scans `apply_runs` for **stale Wards** (after GATE-1: `status='claimed'`, was `IN ('claimed','running')`) `AND claim_expires_at < NOW()` ‚Üí returns them to `planned` (resetting `claim_by_kid` / `claim_at` / `claim_expires_at`). This **closes the "hanging applying" hole** at the **pre-delivery** phase (an undelivered task): the dead owner's row goes back into the queue and is picked up by a live Acolyte; `attempt++` on the next claim cuts off the stale `ApplyRequest` on the Soul (invariant B). **`dispatched` (delivered) is NOT reclaimed** (amend (c)). The Reaper rule name is `reclaim_apply_runs` (propose-and-wait passed).
  - **(j) Single-winner state-commit.** Moving an `incarnation` from `applying` to a terminal at the end of a run is an atomic single-winner: `UPDATE incarnation SET status=‚Ä¶ WHERE status='applying' RETURNING ‚Ä¶`. Only one handler "wins" the row; on a recovery / RunResult race there is no double state commit.
  - **(k) Case 2 ‚Äî isolated Keeper.** An instance that has lost connectivity to PG / Redis but is still alive for some Souls: its **health-probe becomes unhealthy** ‚Üí the LB steers traffic away; **soul-shedding** ‚Äî the instance drops its Soul streams, the Souls reconnect to a working Keeper (connection algorithm [ADR-002](0002-transport-grpc-ha.md#adr-002-transport-keeper--souls--grpc-bidirectional-stream-over-mtls-ha-keeper-cluster)) + retry. **Promoting a DB replica is OUTSIDE Soul Stack** (Patroni / cloud failover, [ADR-005](0005-storage-postgres.md#adr-005-keeper-state-storage--postgres)). The option "ask replicas to switch over" (DB failover by Keeper itself) is **rejected**; soul-shedding is adopted (Keeper does not manage its own DB topology). ‚Äî **IMPLEMENTED 2026-05-25** (propose-and-wait names closed by the user): (1) **Conclave** ‚Äî a Redis registry of live Keeper instances (`keeper:instance:<kid>`, TTL 30s + renew 10s, lease.go pattern; the presence‚ÜíRedis invariant, [ADR-006](0006-cache-redis.md#adr-006-–∫—ç—à-–∏-–∫–æ–æ—Ä–¥–∏–Ω–∞—Ü–∏—è--redis)); provides `CountLive` for the refuse-guard. (2) **Watchman** ‚Äî isolation detection: a periodic ping of PG+Redis (the `/readyz` set) + debounce (isolation after N consecutive failures, default 3) ‚Üí **hard-close** of local EventStream streams (`StreamManager.CloseAll` cancels the per-stream ctx; the receive-loop is moved into a goroutine so that `stream.Recv()` is cancellable) ‚Üí the Soul gets an EOF ‚Üí reconnect via the failback-list ([ADR-002](0002-transport-grpc-ha.md#adr-002-transport-keeper--souls--grpc-bidirectional-stream-over-mtls-ha-keeper-cluster); Soul-failback is already implemented). (3) **refuse-guard** ‚Äî when `acolytes:0` + `Conclave.CountLive>1` Keeper refuses to start (explicit opt-out `allow_unsafe_single_path_multi_keeper` / env), closing Finding-A (see (h)). **go-away rejected** ‚Äî a clean EOF is sufficient, the Keeper‚ÜîSoul proto is not touched. Soul multi-endpoint failback ‚Äî dev-config (the mega-test stand hardcoded a single endpoint). The names Sentinel (collision with Redis Sentinel) and Vigil (taken) are rejected.
- **ADR reconciliation.** [ADR-002](0002-transport-grpc-ha.md#adr-002-transport-keeper--souls--grpc-bidirectional-stream-over-mtls-ha-keeper-cluster) is **strengthened** ‚Äî the stateless cluster now also distributes **execution** (not only stream reception): any instance, through an Acolyte, picks up any task. [ADR-005](0005-storage-postgres.md#adr-005-keeper-state-storage--postgres) ‚Äî execution ownership is materialized in a **PG row** (`apply_runs` Ward columns), not in memory. [ADR-006](0006-cache-redis.md#adr-006-–∫—ç—à-–∏-–∫–æ–æ—Ä–¥–∏–Ω–∞—Ü–∏—è--redis) ‚Äî Summons over pub/sub, recovery scan on the Reaper leader; **the claim-lease lives in Postgres**, not in Redis (a task lease is cold persistent ownership, unlike the SID-lease). [ADR-009](0009-scenario-dsl.md#adr-009-scenario--–ø–æ–ª–Ω–∞—è-dsl-–∑–∞–¥–∞—á-destiny-–≥—Ä–∞–Ω–∏—Ü–∞-—Å-destiny--—Ä–µ–∫–æ–º–µ–Ω–¥–∞—Ü–∏—è) ‚Äî barrier / serial preserved; serial is claimed whole. [ADR-012(d)](0012-keeper-soul-grpc.md#adr-012-–∫–æ–Ω—Ç—Ä–∞–∫—Ç-keepersoul-grpc-–æ–¥–∏–Ω-eventstream-—Å-oneof-keeper-side-—Ä–µ–Ω–¥–µ—Ä-forward-compat-only-add) ‚Äî refined: render is bound to claim and is re-rendered on recovery.
- **Consequences.**
  - **`apply_runs` migration** ‚Äî additive: new statuses `planned` / `claimed`, new columns `claim_by_kid` / `claim_at` / `claim_expires_at` / `attempt`, plus a **`recipe jsonb`** column (see below). The old path (direct `running`) keeps working until Phase 1 (Phase 0).
  - **Acolyte-pool** on each instance (`keeper.acolytes:N` in [config.md](../keeper/config.md)) ‚Äî replaces the single run-goroutine.
  - **Re-render on claim / recovery** ‚Äî render moves from dispatch to claim; recovery re-renders from the recipe. The **run recipe** (scenario-ref + `input` with vault-refs **as-is**, without unwrapped secrets) is materialized in the **`apply_runs.recipe jsonb`** column (G4, propose-and-wait passed ‚Äî a column was chosen rather than a separate table: the recipe is 1:1 with the task, and claim already pulls the row). Invariant A: secrets are resolved only in the Acolyte's RAM at claim; in `recipe` ‚Äî the vault-ref as-is.
  - **Only-add proto field `attempt`** in `ApplyRequest` (`proto/keeper/v1/apply.proto`, **field 4** ‚Äî implemented, Phase 2) ‚Äî the fencing-epoch for the Soul-guard ([ADR-012(c)](0012-keeper-soul-grpc.md#adr-012-–∫–æ–Ω—Ç—Ä–∞–∫—Ç-keepersoul-grpc-–æ–¥–∏–Ω-eventstream-—Å-oneof-keeper-side-—Ä–µ–Ω–¥–µ—Ä-forward-compat-only-add) forward-compat only-add). Echoed from `apply_runs.attempt` at the moment of claim: `ClaimNext` increments it on every Ward capture (`scenario/claim.go` puts `run.Attempt` into `ApplyRequest`), so a re-claim of a stale task arrives with a **larger** attempt than the original. The Soul-guard (`runtime.ApplyRunner`, `lastSeenAttempt map[apply_id]int32`, per-process) rejects an `ApplyRequest` with `attempt < lastSeenAttempt` ‚Äî cutting off the stale duplicate of a dead Ward whose apply is still in flight; the rejected duplicate is silently dropped (barrier-B1), and the original (the larger attempt) closes with its own `RunResult`. **`attempt = 0`/empty = an old Keeper without fencing** (forward-compat): the guard does not reject such a request, and fencing degrades to protection by `apply_id` + SID-lease. The never-reuse of field numbers is preserved. Metric `soul_apply_fenced_total`.
  - **The Reaper recovery-scan rule ‚Äî `reclaim_apply_runs`** (name fixed, propose-and-wait passed; implemented in `keeper/internal/reaper`): `UPDATE apply_runs SET status='planned' ‚Ä¶ WHERE status='claimed' AND claim_expires_at < NOW()` (after GATE-1; it **was** `status IN ('claimed','running')` ‚Äî see amend (c)) (resetting `claim_by_kid`/`claim_at`/`claim_expires_at`; **`attempt` is NOT reset** ‚Äî the next `ClaimNext` increments it). **Requires no migration** ‚Äî the `claim_expires_at`/`attempt` columns and the partial index `apply_runs_claim_scan_idx` came with migration 025. **`enabled:false` by default** (see below). Metric ‚Äî via `ObserveRule` (`keeper/internal/reaper`).
  - **Single-winner state-commit ‚Äî variant W1 (implemented).** Moving `incarnation` `applying`‚Üíterminal stays in the run-goroutine but is protected by an SQL guard: `UPDATE incarnation SET status=‚Ä¶ WHERE name=$1 AND status IN ('applying','destroying') RETURNING ‚Ä¶` (0 rows = the row is already finalized/deleted ‚Üí no-op, like destroy-S-D3). Additionally the `apply_runs` terminal is append-only (a terminal is not overwritten by a terminal). On a recovery/RunResult race there is no double state commit. **Variant W2** (moving the commit into the Acolyte) is **tied to Phase 3** distributed barrier ‚Äî not now.
  - **Lease invariant (recovery applicability boundary).** ‚öÝÔ∏è *Pre-GATE-1 wording ‚Äî WEAKENED, see amend (e).* Before GATE-1, correct recovery required `acolyte_lease > max(single-host apply time, runTimeout)`, otherwise reclaim would return a **live** Ward (`running`) to `planned` ‚Üí a second SendApply. **After GATE-1 this is moot**: reclaim takes only `claimed` (the render/claim phase, BEFORE delivery), while a live long-running apply sits in `dispatched` (reclaim does not touch it). It suffices that `acolyte_lease > max-RENDER` ‚Äî the current default of **30s is fine**, and revising the lease and Acolyte lease-renew **are not required for the recovery boundary**.
  - **Enabling recovery ‚Äî a gate (an operational step, not a lone `enabled:true`).** ‚öÝÔ∏è *Pre-GATE-1: a 3-condition gate ‚Äî RELAXED, see amend (e).* After GATE-1 the gate is: (a) **fencing-Soul rolled out across all Souls** (all Soul agents understand `ApplyRequest.attempt` ‚Äî the guard on execution + echoing `RunResult.attempt` for the epoch-check on receive) + (b) **slices S2‚ÄìS5 (`dispatched` phase + mark-before-send + reclaim-only-claimed + epoch-on-receive) done** (present). The condition "`acolyte_lease > max-apply` OR Acolyte lease-renew" is **dropped** (the lease invariant is weakened). Before enabling ‚Äî a separate architect re-review of the assembly. Until the gate is satisfied, a lone `enabled:true` is dangerous (the old non-fenced path, see amend (f)).
  - **Three layers of protection against double execution (normative).** Safe recovery rests on three independent layers, each covering its own stage: **fencing** (`attempt`-epoch) ‚Äî protection of **EXECUTION** on the Soul against a *smaller* (stale) attempt; **SID-lease** ([ADR-006](0006-cache-redis.md#adr-006-–∫—ç—à-–∏-–∫–æ–æ—Ä–¥–∏–Ω–∞—Ü–∏—è--redis)) ‚Äî dedup of **DELIVERY** of one and the same epoch (apply to a host goes only through the stream owner); **single-winner** ‚Äî protection of the state **COMMIT** (one handler wins the `incarnation` row). All three are needed simultaneously ‚Äî none covers another's stage: fencing does not dedup delivery, SID-lease does not distinguish attempt, single-winner does not prevent a double apply to a host.
  - **Summons** ‚Äî a Redis pub/sub topic for the planned-tasks signal; the topic is **`apply:summons`** (fixed, propose-and-wait passed).
  - **Names in [naming-rules.md](../naming-rules.md)** ‚Äî Acolyte / Ward / Summons (+ definitions).
  - **Update of [storage.md](../keeper/storage.md)** ‚Äî description of `apply_runs` (statuses + Ward columns + lifecycle; after GATE-1 ‚Äî `planned ‚Üí claimed ‚Üí dispatched ‚Üí terminal`, `running` vestigial, see amend (a)).
  - **Full-roster render on claim (Phase 1 cutover).** On claim the Acolyte renders the **full run roster** (like the run-goroutine: `LoadIncarnationHosts` ‚Üí the whole roster ‚Üí `Render` with `Hosts`=the whole roster), then takes its own host's slice `groupByHost(tasks, plans)[sid]`. This eliminates the per-host‚Üîfull-roster dialect: single-host render (`Hosts`=[host]) silently diverged from the old path ‚Äî `run_once` was not trimmed (`applyRunOnce` with `len(targeted)‚â§1` returns the target as-is ‚Üí the task on ALL hosts instead of one), `soulprint.hosts`/`.where` and `incarnation.host_count` collapsed to a roster of a single host, and cross-host `state_changes.sets` lost their neighbors. The list of full-roster dependencies is **open** ‚Äî that is why the full roster is reproduced rather than pointwise guards. The SID validation "host still in the roster" (disconnected/revoked between dispatch and claim ‚Üí error) is preserved BEFORE the render. Invariant A does not change: input secrets are scenario-level (resolved once, not per-host), so the full roster does NOT increase the secret material in RAM.
  - **KNOWN GAP "stuck-claimed" (Phase 1).** SendApply went through, but `MarkClaimedRunning` (claimed ‚Üí running) failed (PG error) ‚Äî the task sticks in `claimed` forever. It **cannot be fixed locally**: without attempt-fencing (Phase 2), re-sending / resetting to `planned` risks a **double SendApply** (the apply already went to the Soul). **Closed by Phase 2**: the Reaper recovery scan (stale Ward `claimed` ‚Üí `planned`, (i)) + the attempt-epoch cuts off the stale `ApplyRequest` on the Soul ((g)).
  - **KNOWN GAP "`lastSeenAttempt` unbounded".** The Soul-guard keeps `lastSeenAttempt map[apply_id]int32` per-process without eviction ‚Äî the map grows on every new `apply_id` over the lifetime of the Soul process. Acceptable for the MVP (applies are rare relative to the process lifetime, a restart zeroes it ‚Äî and after a restart there is physically no duplicate). Candidate for closure ‚Äî TTL/LRU eviction of entries. Deferred.
  - **KNOWN GAP "Acolyte lease-renew missing".** ‚öÝÔ∏è *Not relevant to the recovery boundary after GATE-1 (amend (e)).* The Acolyte does not extend `claim_expires_at` during a long apply ‚Äî a Ward holds a fixed lease from the moment of claim. Before GATE-1 this was the root of the lease invariant; **after GATE-1** reclaim takes only `claimed` (the render/claim phase, not the whole apply), so a fixed lease **does not affect** recovery correctness (a long apply sits in `dispatched`, which reclaim does not touch). Renew is no longer a recovery blocker; it remains an optional improvement for the predictability of claim timings.

- **Trade-offs.**
  - **Re-render on claim / recovery** ‚Äî the price for the invariant "no plaintext secrets in PG": render is more expensive (repeated on takeover), but the unwrapped material is nowhere persisted. Accepted.
  - **claim-lease in Postgres, not in Redis** ‚Äî task ownership is cold and must survive a Redis restart; the price is that claim operations hit PG (but this is `UPDATE ‚Ä¶ SKIP LOCKED`, cheap) rather than hot Redis.
  - **serial = claimed whole by a single Acolyte in the MVP** ‚Äî we lose parallelism within a serial run; accepted as a deliberate simplification, distributed serial deferred to Phase 3.
  - **Soul-fencing depends on the proto field `attempt`** ‚Äî before Phase 2 fencing degrades to protection by `apply_id`; accepted (SID-lease already cuts off most duplicates, `attempt` closes the stale-attempt).
  - **Full-roster render on claim ‚Äî O(N¬≤) per-host CEL + N√óper-host vault per run.** Each of the N claims renders the **full** roster (N hosts), so the total render cost of a run = O(N¬≤) CEL evaluations and N√óper-host vault resolutions vs O(N) of the old path (the run-goroutine renders the roster once). The price for eliminating the per-host‚Üîfull-roster dialect (see Consequences "Full-roster render on claim"): the list of full-roster dependencies is open, pointwise guards fail-open. Acceptable in **Phase 1** (a handful to a few dozen hosts); **hundreds of hosts** ‚Üí a candidate for optimization (symmetrically to "distributed serial ‚Äî Phase 3"). Accepted.

> **Amend (GATE-1: deliver-once recovery, 2026-05-25).** Redesign of the recovery phase (implemented, passed architect + review). Closes the double-apply window discovered in Phase 2: claimed‚Üírunning was marked **after** SendApply ‚Äî a crash between sending and marking left a `claimed` row that the recovery scan re-claimed ‚Üí a second apply on the same host. Below is the final model; the wordings of (i), (j), the lease invariant, and the recovery-enable gate above are **replaced** by the following (the old text is kept marked as pre-GATE-1 to avoid losing history; on divergence the source of truth is this amend).
>
> - **(a) apply_runs lifecycle updated.** Phase chain: **`planned ‚Üí claimed ‚Üí dispatched ‚Üí terminal`** (`success` / `failed` / `cancelled`). The new phase **`dispatched`** is an intent marker "the task has been handed to the Soul"; it is written **ATOMICALLY** (a commit to PG) **STRICTLY BEFORE SendApply** (`MarkDispatched` before sending, not after). The **`running` status is PRESERVED** in the CHECK (there is no no-backcompat cleanup), but in the Acolyte flow it is **NO LONGER used** (vestigial): only the old synchronous path writes it (`acolytes:0`, `dispatchWave ‚Üí Insert(running)`). Migration **040** extends the CHECK (`status IN (‚Ä¶ , dispatched)`); `down-040` is correct. The name `dispatched` is fixed (propose-and-wait closed by the user 2026-05-25; the motive ‚Äî it coincides with the `dispatch`/`grpc.apply_dispatch` already used in the code, zero cognitive gap) ‚Äî recorded in [naming-rules.md](../naming-rules.md).
> - **(b) Moving the mark before sending (the root of the fix).** Previously claimed‚Üírunning was marked **AFTER** SendApply ‚Üí a race: a crash in the window between send and mark left a `claimed` row, and the recovery scan re-claimed it ‚Üí a **DOUBLE apply**. Now claimed‚Üí`dispatched` **BEFORE** SendApply eliminates this. The death windows of the Acolyte path after the fix: **W-a** (`MarkDispatched` ok, crash **before** SendApply) ‚Üí a `dispatched` row, the task is not delivered, reclaim **does not touch** it ‚Üí a **safe HANG** (NOT a double apply); closure ‚Äî Soul-reconcile, post-MVP (see (g)). **W-b** (the old "mark after send" gap) ‚Äî **ELIMINATED**.
> - **(c) reclaim narrowed to `claimed`.** `reclaim_apply_runs` is now `WHERE status='claimed'` (was `IN ('claimed','running')`). **The nature of the rule has changed**: "finish off an **UNDELIVERED** task" (the owner died **before delivery**, at the render/claim phase), NOT "revive a zombie `running`". The **`dispatched` phase is NOT reclaimed** ‚Äî after delivery the run is owned by the Soul; reclaiming a delivered one = a double apply.
> - **(d) Epoch-check on RESULT RECEIPT (fencing symmetry).** attempt-fencing becomes **two-sided**: (1) protection of **execution** in the Soul (`AcceptAttempt`, was in (g)) + (2) protection of result **RECEIPT** in the Keeper. `RunResult` carries an echo of `attempt` (proto **only-add field 4**, [ADR-012(c)](0012-keeper-soul-grpc.md#adr-012-–∫–æ–Ω—Ç—Ä–∞–∫—Ç-keepersoul-grpc-–æ–¥–∏–Ω-eventstream-—Å-oneof-keeper-side-—Ä–µ–Ω–¥–µ—Ä-forward-compat-only-add) forward-compat). `correlateRunResult` compares against `apply_runs.attempt`: `recvAttempt < rowAttempt` (and `!= 0`) ‚Üí a result of a **stale** attempt ‚Üí **stale-drop** (metric `keeper_runresult_stale_total`), state is **NOT committed**; `==` / `0` (forward-compat: an old Soul without the echo) / `>` (impossible, fail-safe + defensive warn-log) ‚Üí commit. The result is accepted on **ANY** Keeper instance (Soul failback, [ADR-002](0002-transport-grpc-ha.md#adr-002-transport-keeper--souls--grpc-bidirectional-stream-over-mtls-ha-keeper-cluster)) ‚Äî the authority for actuality is in the **shared PG** (`apply_runs.attempt`), not in an instance's memory. **Three orthogonal layers** of protection against double execution (replacing the "Three layers of protection" wording above, augmenting it with receipt): **execution fencing** (the Soul rejects a *smaller* attempt) / **SID-lease delivery dedup** ([ADR-006](0006-cache-redis.md#adr-006-–∫—ç—à-–∏-–∫–æ–æ—Ä–¥–∏–Ω–∞—Ü–∏—è--redis)) / **single-winner commit + receipt epoch-drop** (Keeper). The epoch-check on receipt is orthogonal to single-winner: epoch cuts off a *stale* attempt, single-winner ‚Äî a *repeated terminal of the same* attempt.
> - **(e) Lease invariant WEAKENED.** recovery-enable **NO LONGER requires** `acolyte_lease > max-apply` and **does not require** Acolyte-lease-renew. It suffices that `acolyte_lease > max-**RENDER**` (the current default of **30s** ‚Äî fine), because reclaim takes only `claimed` (the render/claim phase **BEFORE** delivery), not the whole apply: a live long-running apply sits in `dispatched`, which reclaim does not touch. This **removes** the old lease-invariant wording and **relaxes** the recovery-enable gate. The gate is now: **fencing-Soul rollout** (also needed for accepting the RunResult echo) + slices S2‚ÄìS5 done. (The condition "`acolyte_lease > max-apply` OR Acolyte lease-renew" is dropped; the KNOWN GAP "Acolyte lease-renew missing" above becomes irrelevant to the recovery boundary.)
> - **(f) TWO KNOWN-GAPs (recorded explicitly so they are not "fixed" back).**
>   - **The old synchronous path without Reaper recovery.** With `acolytes:0` (the prod default; Acolyte is opt-in) tasks are written directly to `running`, and narrowing reclaim to `claimed` means the Reaper **does NOT recover** these `running` rows. This is **NOT a regression**: recovery of the old path is the in-memory run-goroutine of the owner instance (+`runTimeout`), not Reaper-reclaim; on the owner's death its run was not recovered anyway (a property of the pre-Acolyte model). **`reclaim_apply_runs` and `acolytes > 0` are a coupled pair: enabling reclaim with `acolytes:0` is FORBIDDEN** ‚Äî on the non-fenced (old) path reclaim is itself unsafe (a double apply without epoch protection).
>   - **Cancel visibility of `dispatched`.** The `RequestCancel` guard **does NOT include** `dispatched` (by design in the pilot): to already-delivered Souls the Keeper **does NOT send** `CancelApply` (best-effort cancel of delivered runs is a deferred admin-API layer via `Outbound.SendCancel`). The barrier read (`SelectStatusesByApplyID` by `apply_id`) **does see** `dispatched` ‚Äî it waits for the terminal correctly. Full closure = a separate sub-slice (`guard += 'dispatched'` **TOGETHER** with emitting `CancelApply` to the Soul + receiving `RunStatus.CANCELLED`), to be combined with Soul-reconcile (see (g)).
> - **(g) Soul-also-dead (Keeper and Soul both dead after delivery).** A window that deliver-once itself **does not close**: `dispatched` will hang ‚Äî `RunResult` will not arrive, and reclaim does not touch the row. **SOLUTION:** closure **post-MVP via Soul-reconcile** ‚Äî on reconnect the Soul reports the `apply_id`s it is actually driving, the Keeper reconciles and terminates orphaned `dispatched`. In the **MVP** ‚Äî a documented known-gap (the operator restarts the run manually). **We do NOT do a Reaper dispatched-timeout** (a terminal-by-timeout without confirmation from the Soul is unsafe). ‚Äî **IMPLEMENTED 2026-05-25 (S6 Soul-reconcile), known-gap removed:** on (re)connect the Soul sends `WardRoster` (a ReplaceAll snapshot of driven `apply_id`+attempt; a new `FromSoul` only-add message, oneof field 8 ‚Äî [ADR-012(k)](0012-keeper-soul-grpc.md#adr-012-–∫–æ–Ω—Ç—Ä–∞–∫—Ç-keepersoul-grpc-–æ–¥–∏–Ω-eventstream-—Å-oneof-keeper-side-—Ä–µ–Ω–¥–µ—Ä-forward-compat-only-add)) right after Hello; in `OrphanDispatched` the Keeper, via an epoch-fenced single-winner reconciliation, terminates the `dispatched` rows of this SID that are not in the set ‚Üí a new terminal **`orphaned`** (migration 044; the barrier classifies it as fail ‚Üí incarnation `error_locked`). The Soul's set is in-memory (active ‚à™ recently-finished ring TTL 30s; a Soul restart ‚Üí empty ‚Üí its dispatched are legitimately orphaned). An old Soul without `WardRoster` ‚Üí sweep no-op (fail-safe; under the same fencing-Soul rollout gate as recovery-enable (e)).
> - **(h) Amendment (2026-05-25, multi-keeper guard).** The invariant is made **explicit**: the run-goroutine path (`acolytes:0`) is **single-keeper-only**; **N>1 Keeper instances require `acolytes>0`**. A live multi-keeper test confirmed the footgun: with `acolytes:0` an apply created on keeper-A with a Soul on keeper-B's stream runs on the host (`apply_runs=success`), but the incarnation **hangs in `applying` forever** ‚Äî the in-memory owner on keeper-A does not observe the `RunResult` that went to keeper-B. The guard is **warn, not refuse** (without a new instance-registry entity): (1) a startup advisory with `acolytes:0`; (2) a targeted `WARN` in the run-goroutine dispatch path when the target Soul's SID-lease belongs to a **different KID** (the footgun fires exactly here). A refuse-startup-guard + **a registry of live Keeper instances** were **IMPLEMENTED 2026-05-25** (the registry = **Conclave**, see (k)): with `acolytes:0` + `Conclave.CountLive>1` Keeper refuses to start (explicit opt-out `allow_unsafe_single_path_multi_keeper` / env `KEEPER_ALLOW_UNSAFE_MULTI_KEEPER`); the dispatch-time WARN is kept as a runtime safety-net on top of refuse. The invariant is recorded in [docs/keeper/config.md](../keeper/config.md).
> - **(i) Amendment (2026-05-25, FINDING-01: apply_runs does not over-report non-targets).** On the Acolyte path (`acolytes>0`) `dispatchPlanned` writes an `apply_runs` row for the WHOLE incarnation roster BEFORE resolving `on:`/`where:` (resolution is per-host at claim); the barrier holds `wantHosts = len(roster)`. Previously a host for which `on:`/`where:` filtered out all tasks (0 tasks) was closed as `success` (no-op) ‚Üí `apply_runs` falsely showed "success on all" (found by Phase 2 HA acceptance: a 9-host incarnation, `on:[cluster-a]` ‚Üí actually 3 tasks, but 9√ó success). Fix: a non-target no-op host ‚Üí a new terminal **`no_match`** (migration 045; `classify` counts `no_match` as a benign terminal on the success side ‚Äî the barrier tallies up `wantHosts`, incarnation `ready`; `failed`/`cancelled`/`orphaned` alongside are still fail-closed). `apply_runs` is currently NOT exposed externally via the API (history = `state_history`), so the impact was LOW; `no_match` makes the record honest until a `/runs` endpoint appears. **A residual dialect** (NOT fixed by this slice): the old `acolytes:0` path does not materialize non-target rows at all (target-only), while the Acolyte path writes the whole roster (`success`+`no_match`); full unification of the row count between modes is a separate decision.

> - **Extension 2026-05-27 (Tide-spawn apply_runs ‚Äî back-link to `tides`).** [ADR-040](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override) introduces a **second path** for materializing `apply_runs` rows that does not violate this ADR's invariants.
>
>   **Two paths for spawning apply_runs:**
>
>   - **Classic (single-run, `tide_id IS NULL`):** the scenario-runner, on dispatch `POST /v1/incarnations/.../scenarios/.../run` **without** the `wave` field, writes one `apply_run` per host in `planned` and sends a `Summons` ‚Äî the Acolyte flow works as described in (a)‚Äì(j) above.
>   - **Tide path (`tide_id` + `surge_index` populated):** **TideWorker** (a separate pool, see below) per-Surge spawns N apply_run rows **in one PG transaction** with UPDATE `tides.current_surge_index=N` + `current_apply_id=Y` ‚Äî the same-tx invariant [ADR-040 amendment 2026-05-27](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override). From there the Acolyte flow is indistinguishable from the classic one (Acolyte claim ‚Üí render ‚Üí dispatch ‚Üí terminal): the Tide orchestrator knows about these `apply_run` rows **only** through the `current_apply_id` field and does not manage their lifecycle directly.
>
>   **`apply_runs.tide_id` / `surge_index` ‚Äî nullable back-link:**
>
>   - `tide_id` ‚Äî a nullable FK on `tides(tide_id)`; `surge_index` ‚Äî an int (nullable, always together with `tide_id`).
>   - Classic single-run: both `NULL`.
>   - Tide-spawn: both populated; the composition `(tide_id, surge_index)` identifies a Surge within a Tide.
>   - **Reaper apply_runs cleanup (`purge_apply_runs` / `reclaim_apply_runs`) does NOT depend on `tide_id`** ‚Äî the apply_run cycle is independent of the Tide cycle. A Surge's `apply_run` is cleaned up by its own rules (max_age from `finished_at`); the Tide itself lives on a separate cycle and is cleaned up by `reclaim_tides` (recovery of stale claims). There is no cascading `ON DELETE` ‚Äî `apply_runs.tide_id` has no `ON DELETE` clause (default NO ACTION); deleting a `tides` row post-MVP (retention of `tides`) will require a separate decision on orphan apply_runs.
>
>   **TideWorker pool ‚Äî separate from Acolyte:**
>
>   - **Not a shared pool** ‚Äî deadlock prevention: with `acolytes: 1` + 1 Tide occupying the Acolyte slot, a Surge within the Tide would have nowhere to land. TideWorker and Acolyte compete only for PG connections, not for worker slots.
>   - Configuration ‚Äî a separate block `keeper.yml::tide:` (recommendation ‚Äî `tide_workers ‚â§ acolytes-1` per-instance). See [`docs/keeper/config.md`](../keeper/config.md).
>   - The worker cycle is simple: `claim ‚Üí orchestrate ‚Üí poll/sleep`. Tide operations are rare and large (one Tide = N Surges over minutes-hours), without the complex wake logic of the Acolyte path.
>
>   **ADR-027 invariants do not change:**
>
>   - apply_run lifecycle: `planned ‚Üí claimed ‚Üí dispatched ‚Üí terminal` ‚Äî the same for both paths (Tide-spawn lands in `planned` with `tide_id`/`surge_index` already set).
>   - The fencing epoch (`attempt`-fencing of execution + epoch-check of receipt) ‚Äî works identically; the Tide orchestrator does not interfere with the Acolyte's attempt increment.
>   - Single-winner state-commit ‚Äî `UPDATE incarnation ‚Ä¶ WHERE status='applying'` remains the root of protection; per-Surge state-commit is implemented by a separate SQL guard on the Tide-orchestrator side (commit-only under confirmed lease ownership via `FinalizeWithOwnership`, [ADR-040 amendment 2026-05-27](0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override)).
>   - SID-lease + Soul-fencing for delivery dedup ‚Äî also unchanged.

> **Amend (l): Voyage-orchestrator self-orphan-release of the applying-lock before re-run (2026-06-21).** Implemented (`keeper/internal/voyageorch` + `incarnation.ReleaseApplyingOrphan`), passed review + re-audit. Closes a hole in the Voyage path that `reclaim_apply_runs` structurally **does NOT cover**.
>
>   - **Problem.** The Voyage owner (a Keeper instance) died during a per-incarnation scenario-run: `incarnation.status='applying'` was committed (`lockRun`), but the run did not reach a terminal. After reclaim the Voyage is picked up by a new owner that wants to re-run this incarnation ‚Äî but `lockRun` rejects it ("incarnation already applying"), and the Voyage **hangs forever**. `reclaim_apply_runs` **does not reach** here: it reclaims a stale `claimed` Ward into `planned` (the render phase BEFORE delivery), whereas the `incarnation`-lock is an **ownerless flag without epoch/owner/lease** on a separate `incarnation` row, not `apply_runs`. The `apply_run` itself may not **exist** at all by the time of re-run: it is ephemeral (a crash BEFORE the Insert after `lockRun`) or removed by the `purge_apply_runs` retention-purge (terminal >30d). ‚öÝÔ∏è The former attribution "reclaim_apply_runs cleaned up the orphaned apply_runs row" is **incorrect** ‚Äî reclaim does not delete rows; the source of "the row is gone" is purge-retention OR a pre-Insert crash.
>
>   - **Detection.** Before spawning the re-run, the reclaimed VoyageWorker finds an orphan candidate by the **back-link `voyage_targets.apply_id`** of THIS Voyage from the **previous attempt**: an incarnation row in `running`/`awaiting` with a recorded `apply_id`. `MarkTargetRunning` writes the back-link UNDER the CAS `voyages.attempt=$attempt` AFTER the spawn; we are NOW BEFORE the spawn of our attempt ‚Äî which means the recorded apply_id is from the previous pass. Another Voyage / a direct run with a different apply_id **do not have** a back-link under our voyage_id ‚Üí their lock is not picked up by this seam (the protection rests on the FENCING-2/3 bundle, not on FENCING-1 alone).
>
>   - **Fencing (all mandatory before release).** (1) **no-live-rival** ‚Äî there is no active (non-terminal) `apply_run` with a FOREIGN apply_id for the incarnation (a live direct run / another Voyage holding its own row ‚Üí block). (2) **reclaimed-attempt + VerifyOwnership** ‚Äî we are the current Voyage owner with claim-epoch `run.Attempt`; if we ourselves were reclaimed (`ErrLeaseLost`) ‚Üí we do not touch it. (3) **single-winner CAS** `applying ‚Üí ready` under `FOR UPDATE` + the guard `status='applying'`: an honest RunResult from the previous owner that already took the row out of applying yields a no-op. `state` is NOT touched (last known-good = pre-run state, the orphan did not reach the state-commit).
>
>   - ‚òÖ **Residual double-apply class ‚Äî the same acceptable one as `reclaim_apply_runs`.** A network partition: a live partitioned previous owner (still driving apply to hosts) + a re-run by the new owner ‚Üí a double dispatch to a host. Protection ‚Äî **gate-1 attempt-fencing RunResult** (the Soul rejects a smaller attempt at execution + the epoch-check on receipt) + **module idempotency**. The seam does NOT introduce a new risk class beyond the existing recovery.
>
>   - ‚òÖ **Always enabled together with `reclaim_voyages` (default-ON), NOT behind a separate opt-in.** A deliberate choice: `reclaim_voyages` is already default-ON; without the orphan-release seam Voyage recovery is **functionally broken** (a reclaimed Voyage hangs on the very first applying incarnation). A separate gate here would create a "recovery is enabled but does not work" mode.
>
>   - ‚òÖ **Coexistence window with `UpdateStateFromRun`.** `UpdateStateFromRun` (the run terminal) fences only the **status-CAS** (`WHERE status='applying'`), NOT attempt/apply_id ‚Äî this is an existing property of **(j)**/W1. Consequence: an honest terminal from the old owner that **passed gate-1** can overwrite a re-run (both see `applying`, both are valid by the status-CAS). The seam **widens** their coexistence window (the re-run starts earlier ‚Äî right after orphan-release, not after the natural finish of the old one). This is not a correctness regression: both outcomes are valid terminals of the same logical operation under gate-1; strengthening to attempt-fencing of the state-commit is tied to Phase 3 / W2, not now.

> **Amend (m): standalone-orphan reconcile via `incarnation` epoch columns + the Reaper rule `reconcile_orphan_applying` (2026-06-21).** A backstop foundation (slice S0 ‚Äî schema + contract; the rule insertion and epoch write in `lockRun` ‚Äî slices S1). Closes a hole in the **direct** `incarnation.run` (without Voyage), symmetric to (l), but which the (l) seam structurally does **NOT** cover.
>
>   - **Problem.** A direct `incarnation.run` (standalone, not under Voyage) sets `incarnation.status='applying'` (`lockRun`); if the Keeper owner of the run dies before a terminal ‚Äî the lock hangs **forever**. The (l) seam does not reach here: it finds an orphan candidate by the back-link `voyage_targets.apply_id` of THIS Voyage ‚Äî a direct run has **no** `voyage_targets` row, and the back-link is structurally unreachable. `reclaim_apply_runs` also does not reach (it reclaims a stale `claimed` Ward into `planned`, whereas the `incarnation`-lock is a separate flag on the `incarnation` row, not `apply_runs`). Root cause: the applying flag is an **ownerless bool without epoch / owner / lease**; it is impossible to distinguish "a live run is still going" from "the owner is dead, the lock is orphaned".
>
>   - **Solution ‚Äî an epoch for the applying flag (4 NULLABLE `incarnation` columns).** `applying_apply_id` / `applying_attempt` / `applying_by_kid` / `applying_since` (migration 082, additive, without backfill ‚Äî for existing rows NULL = "epoch unknown", fail-safe for the rule). `lockRun` writes them in the same tx as `status='applying'` (S1), and the run terminal clears them to NULL together with removing the applying status (S1). This turns the bare bool into an **inline-epoch**: the rule sees WHO holds the lock (`applying_by_kid`) and HOW LONG AGO (`applying_since`), without a separate table and without a back-link.
>
>   - **The rule `reconcile_orphan_applying` (name fixed by the user; default-ON + presence-gate; insertion ‚Äî S1).** Two-phase:
>     - **(1) SQL candidates.** `SELECT ‚Ä¶ FROM incarnation WHERE status='applying' AND applying_since < NOW() - <stale_after>` (partial index `incarnation (status, applying_since) WHERE status='applying'`, migration 082 ‚Äî parity with `apply_runs_claim_scan_idx`). `stale_after=90s` (parity with `mark_disconnected`).
>     - **(2) presence check.** For a candidate with a non-empty `applying_by_kid`: whether the owner is alive in Conclave (`InstanceAlive(applying_by_kid)`, ADR-006(e)). Alive ‚Üí skip (the run is really going). Dead ‚Üí orphan.
>     - **(3) fenced single-winner.** `applying ‚Üí ready` under `FOR UPDATE` + the guard `status='applying'` (reusing the idempotent `ReleaseApplyingOrphan` already introduced by (l)); an honest terminal from the real owner that managed to remove applying yields a no-op. `state` is NOT touched (the orphan did not reach the state-commit, last known-good = pre-run state).
>   - **Default ‚Äî ON with a presence-gate (the user's decision).** No-op without a live Conclave (the presence check is fail-safe: on a Redis error / an empty `applying_by_kid` ‚Äî it does not reclaim, so as not to disrupt a live run with an unknown epoch). The release audit-event ‚Äî `reaper.reconcile_orphan_applying.executed` (name fixed; registration/emission ‚Äî S1).
>
>   - **Fencing (all mandatory before release, S1 insertion).** (1) **FENCING-1 no-live-rival** ‚Äî there is no active `apply_run` with a different apply_id for the incarnation (a live direct run / another Voyage holding its own row ‚Üí block), as with (l)/`reclaim_apply_runs`. (2) **presence-death of the owner** ‚Äî `InstanceAlive(applying_by_kid)=false` (phase 2). (3) **single-winner CAS** `applying‚Üíready` (phase 3). The residual double-apply class ‚Äî **the same acceptable one** as with (l) and `reclaim_apply_runs` (a network partition of a live partitioned owner + a re-run); protection ‚Äî gate-1 attempt-fencing `RunResult` + module idempotency. The seam introduces no new risk class.
>
>   - **Coexistence with the Voyage path (l).** Both converge on **one** idempotent CAS `ReleaseApplyingOrphan` (`applying‚Üíready` under `FOR UPDATE` + a status-guard). (l) detects an orphan by the `voyage_targets` back-link (the Voyage path), (m) ‚Äî by the `incarnation` epoch columns (the standalone path); the release is a shared function. There is no conflict: if both rules target one row (theoretically impossible ‚Äî a Voyage incarnation has a `voyage_targets` row, a standalone one does not), the second CAS is a no-op.

The normative wordings below fix the S1 behavior; on divergence from the rule description in (m) above, the source of truth is this block (it details, it does not override (m)).
>
>   - **Atomicity of the epoch write.** `lockRun` writes ALL 4 columns (`applying_apply_id`/`applying_attempt`/`applying_by_kid`/`applying_since`) in the SAME PG-tx and the SAME `UPDATE` as `status='applying'` (via `updateStatus` in `keeper/internal/scenario/state.go`). There is no "applying without epoch" window ‚Äî a crash before commit rolls back the whole `UPDATE`. `applying_by_kid` = the KID of this Keeper instance (the same source as the lease-holder/Voyage-claim); `applying_apply_id` = the `apply_id` of the run; `applying_attempt` = an echo of the current `apply_runs.attempt` (at the moment of `lockRun` there is no `apply_runs` row yet ‚Äî this is the run's initial attempt); `applying_since=NOW()`. The run terminal (`UpdateStateFromRun` success + lock-fail + abort) clears all 4 to `NULL` atomically with removing applying.
>
>   - **FromLocked/rerun-create-start writes the epoch.** FromLocked/rerun-create-start writes the epoch via the same `lockApplyingWithEpoch` (the `spec.FromLocked` branch of `lockRun`) ‚Äî the applying row gets a non-empty `applying_by_kid`/`apply_id`/`since` and falls under `reconcile_orphan_applying`. The residual micro-window between the UnlockForRerun-tx and the lockRun-tx (applied in two different txs of one request) degrades into the existing NULL-epoch known-gap (no worse than the current one) ‚Äî full atomicity for this rare recovery-over-recovery class is not required.
>
>   - **NULL-epoch = known-gap.** The rule's candidate SQL filters `AND applying_by_kid IS NOT NULL` (in addition to `status='applying' AND applying_since < NOW()-stale_after`). A NULL-epoch (legacy/pre-082, or applying not via the S1 `lockRun`) is **NOT** reclaimed by the rule ‚Äî a documented known-gap, fixed by a manual `Unlock` by the operator. A separate path for NULL-epoch is NOT introduced (without `applying_by_kid` there is no presence witness of death ‚Äî a weaker fencing class, rejected, symmetrically to the presence invariant of amend (n)).
>
>   - **Reuse of `ReleaseApplyingOrphan` WITHOUT a signature change.** The rule calls the existing `ReleaseApplyingOrphan(ctx, pool, name, orphanApplyID, historyID)` as-is. `orphanApplyID` = the candidate's `incarnation.applying_apply_id` (non-empty by the filter + the atomic write; the `empty orphan apply_id` guard is unreachable; the Reaper additionally defensive-skips an empty one with a warn). `historyID` = `audit.NewULID()`, generated by the Reaper before the call (identically to the Voyage adapter). The `ReleaseApplyingOrphanByEpoch` wrapper is NOT introduced.
>
>   - **The standalone-path fencing set (normative).** (1) **presence-death** ‚Äî `InstanceAlive(applying_by_kid)=false` (phase 2 of the rule; direct proof of death in Conclave, replacing the Voyage FENCING-3 `VerifyOwnership` ‚Äî a standalone has no Voyage-claim, so do NOT call `voyage.VerifyOwnership`). (2) **FENCING-1 no-live-rival** ‚Äî there is no active `apply_run` with `apply_id ‚âÝ orphanApplyID` (inside `ReleaseApplyingOrphan`). (3) **single-winner CAS** `applying‚Üíready` under `FOR UPDATE` + `status='applying'` (inside `ReleaseApplyingOrphan`). The Voyage analogue reclaimed-attempt (`run.Attempt>orphan`) does NOT apply ‚Äî the Reaper has no re-run moment with an increased attempt; the witness of death = presence.
>
>   - **`applying_attempt` is NOT read for standalone release** (it is redundant: presence proves death, FENCING-1 fences by `apply_id`). The column is STILL written in `lockRun` atomically (parity with `apply_runs.attempt`, groundwork for a post-MVP epoch-check of `RunResult` receipt) ‚Äî it is NOT a dead write.
>
>   - **The rule `reconcile_orphan_applying` ‚Äî default-ON + presence-gate.** Enabled by default (path-defaulting, like `reclaim_voyages`), BUT a no-op without a live Conclave client (graceful, like `mark_disconnected` without Redis). `stale_after=90s` (parity with `mark_disconnected`). Phases: (1) SQL candidates via `incarnation_applying_scan_idx` (`status='applying' AND applying_since < NOW()-stale_after AND applying_by_kid IS NOT NULL`); (2) presence ‚Äî `InstanceAlive(applying_by_kid)`, alive‚Üískip; (3) defensive-skip of an empty `applying_apply_id`/`applying_by_kid` (warn+metric, unreachable with a correct `lockRun`); (4) release ‚Äî `ReleaseApplyingOrphan` as-is. Metrics `keeper_reaper_*{rule="reconcile_orphan_applying"}`; the audit-event `reaper.reconcile_orphan_applying.executed` (area `reaper.*`, `source: keeper_internal`, payload `{incarnation, prev_kid, apply_id}`) is emitted on a successful release (per-row, like `voyage.reclaimed`).
>
>   - **Residual double-apply** ‚Äî the same acceptable class as (l) and `reclaim_apply_runs`: a network partition of a live partitioned owner (presence holds ‚Üí the rule does NOT touch it, fail-safe). Protection ‚Äî gate-1 attempt-fencing `RunResult` + module idempotency.

> **Amend (n): presence-gated force-release of the SID-lease on a detected-crash reconnect (2026-06-21).** A backstop foundation (slice S0 ‚Äî the server-side lease function + the presence primitive; the insertion into `acquireSoulLease` ‚Äî slice S2). Extends the SID lease mechanics ([ADR-006(b)](0006-cache-redis.md#adr-006-–∫—ç—à-–∏-–∫–æ–æ—Ä–¥–∏–Ω–∞—Ü–∏—è--redis)).
>
>   - **Problem.** After a SIGKILL of the Keeper owner, the SID-lease (`soul:<sid>:lock`, [SoulLease](../keeper/storage.md)) lives until TTL (`defaultSoulLeaseTTL=60s`): the graceful `Release` did not get to run (crash). A reconnect of the SAME SID to a new Keeper gets `AcquireSoulLease@ErrLeaseTaken` ‚Üí the stream is closed with `code.AlreadyExists`, and `handleWardRoster` (Soul-reconcile (g)) is **unreachable** for ~60s ‚Äî the Souls are invisible for a window of almost a minute after the crash.
>
>   - **Why NOT a blind `DEL`.** Blindly removing someone else's lease undermines the SID-lease delivery dedup: with a network partition of a **live** holder (it still holds the stream to the Soul, it is just temporarily invisible to the reconnect-Keeper), a `DEL` + re-acquire produce **two** owners of one SID stream ‚Üí double delivery of `ApplyRequest` / splitting of `RunResult`. The lease dedup rests on "one key ‚Äî one owner"; a blind DEL breaks it.
>
>   - **Solution ‚Äî a presence-gated CAS re-acquire.** On `AcquireSoulLease@ErrLeaseTaken`: `SoulLeaseOwner(sid)` ‚Üí `prevKID`; `InstanceAlive(prevKID)` (Conclave, ADR-006(e)):
>     - **dead in Conclave** ‚Üí `ForceAcquireSoulLease(sid, prevKID, myKID, ttl)` ‚Äî a Lua-atomic CAS-**by-prev-holder**: re-acquire ONLY if the key is still `== prevKID`. If the key changed in a race (another Keeper already re-acquired / the Soul reconnected to a third) ‚Äî a normal fail ‚Üí `AlreadyExists`.
>     - **alive** (or a Redis error of the presence check, or a partition in which the Conclave presence of a live holder still holds) ‚Üí we do NOT remove it, `AlreadyExists` as now (fail-safe toward preserving the current owner).
>   - **Boundary of the effect.** The invisibility window is reduced from the SID-lease TTL (`60s`) to the **Conclave-presence TTL of the dead holder** (`DefaultConclaveTTL=30s`) ‚Äî NOT to zero: presence-death is detected only after the Conclave key of the crashed owner expires. This is a deliberate trade-off against split-brain (presence is the only reliable sign that "the owner is really dead, not partitioned").
>
>   - **Boundaries of S0 (this slice).** S0 provides only the server-side `ForceAcquireSoulLease` (Lua CAS-by-prev-holder) + the presence primitive `InstanceAlive` ‚Äî **WITHOUT** the insertion into the `acquireSoulLease` handler (the presence-gated call on `ErrLeaseTaken` ‚Äî slice S2). The release audit-event ‚Äî `eventstream.lease_force_released` (name fixed; emission ‚Äî S2).

> **Amend (o): a pre-flight assert gate BEFORE entering `applying` (2026-06-23, form A). Pilot.** Jointly with the [ADR-009](0009-scenario-dsl.md#adr-009-scenario--–ø–æ–ª–Ω–∞—è-dsl-–∑–∞–¥–∞—á-destiny-–≥—Ä–∞–Ω–∏—Ü–∞-—Å-destiny--—Ä–µ–∫–æ–º–µ–Ω–¥–∞—Ü–∏—è) amendment of the same date. Introduces a new **synchronous run phase BEFORE the apply_runs lifecycle** ‚Äî it wedges into the request path BEFORE any mutation, without changing the apply_runs machine itself (`planned ‚Üí claimed ‚Üí dispatched ‚Üí terminal`).
>
>   - **Where in the lifecycle.** The full `create` run chain: request `POST /v1/incarnations` ‚Üí sync RBAC ‚Üí sync `ValidateInput` (input schema) ‚Üí **‚òÖ pre-flight assert (the new gate)** ‚Üí `incarnation.Create` (commit the row) ‚Üí `Runner.Start` (async) ‚Üí `lockRun` (`status='applying'` + epoch, amend (m-S1)) ‚Üí render ‚Üí dispatch (`planned`‚Üí‚Ä¶). Pre-flight stands BETWEEN `ValidateInput` and `incarnation.Create`: the incarnation does NOT exist yet, the `applying` lock is NOT taken yet, and there is not a single `apply_runs`/`state_history` row yet.
>   - **What it does.** Synchronously evaluates the `create` scenario's `assert:` predicates ([ADR-009](0009-scenario-dsl.md#adr-009-scenario--–ø–æ–ª–Ω–∞—è-dsl-–∑–∞–¥–∞—á-destiny-–≥—Ä–∞–Ω–∏—Ü–∞-—Å-destiny--—Ä–µ–∫–æ–º–µ–Ω–¥–∞—Ü–∏—è)) in the full scenario-CEL context (the roster of connected-souls by the incarnation Coven = `soulprint.hosts`). Any `false` ‚Üí **HTTP 422 `assert-failed`** to the operator (NOT `error_locked`): the incarnation is NOT created, `applying` is NOT set, the run does not start. This is a "fail at the model stage" ‚Äî a rejection BEFORE entering the lifecycle, not a post-factum `error_locked` via an async render-abort.
>   - **Why it does NOT break ADR-027 invariants.** The gate does not touch the work-queue: it works on the request path BEFORE `Runner.Start`, before `lockRun`, before the apply_runs spawn. Single-winner state-commit (j)/(m), epoch-fencing (d)/(g), recovery (i)/(l)/(m) ‚Äî all DOWNSTREAM, pre-flight does not touch them. On a pre-flight failure there remains neither an orphaned `applying` lock (amend (m): there was none) nor a hanging `apply_run` (amend (i): there was none) ‚Äî the rejection is clean by construction (no mutation). Pre-flight is read-only to PG (a roster-query + a service-snapshot Load), without writing.
>   - **The render-assert fail-safe is preserved (TOCTOU).** Pre-flight validates the roster at the moment of CREATION; between it and the start of the scenario-goroutine the roster could change. The render `assert:` branch at run start remains a safeguard: its failure = `error_locked` via the same async-abort described originally in ADR-009. Pre-flight = the main UX (422, frequent), render-assert = the fail-safe (a rare `error_locked`).
>   - **The pilot scope ‚Äî only the `create` path.** `incarnation.run` (on an existing one) / `destroy` / Voyage-spawn ‚Äî follow-up; for them the render-assert fail-safe is so far the only gate. Implementation ‚Äî `keeper/internal/scenario` (`Runner.PreflightAssert`), `keeper/internal/render` (`Pipeline.EvalAsserts`, a shared eval with the render branch), `keeper/internal/api/handlers/incarnation_typed.go` (`CreateTyped`), `keeper/internal/api/problem` (`TypeAssertFailed` 422).
