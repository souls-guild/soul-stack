# core.augur

Read-probe **live** access to external system (Vault / Prometheus / ELK) via
broker [Augur](../../../keeper/augur.md) ([ADR-025](../../../adr/0025-augur.md)).
**Soul-side**, statically built into the `soul` binary. Implementation -
[`soul/internal/coremod/augur/augur.go`](../../../../soul/internal/coremod/augur/augur.go).

Unlike the pre-resolved model (everything external is resolved by Keeper **before**
sending `ApplyRequest`), `core.augur.fetch` requests the value **at the moment
apply on the host** - for cases that pre-resolve cannot resolve in advance:
short-lived dynamic-secret, live-metric as step condition, reading from ELK to
during the run. Border [ADR-012(d)](../../../adr/0012-keeper-soul-grpc.md)
does not move: in MVP-1 (`delegate=false`) data arrives inline **via Keeper**,
external credential does not apply to Soul.

This is a verb module: the only state is `fetch` (without declarative semantics
"lead to state"). Typical use of `register:` - read-only probe
(`changed_when: false` is not needed - the module is already `changed=false`) with reading
`register.<name>.*` in subsequent `where:` / `failed_when:` / `output:`.

## States

| State | Destination | Idempotency (when `changed=true`) |
|---|---|---|
| `fetch` | Request the value from Omen from the Augur broker at the time of apply. | `changed=false` **ALWAYS**, constructive and non-configurable: read-probe does not change the host state (use case - [`core.http.probe`](../http/README.md), [`core.exec.run`](../exec/README.md)). |

## fetch — params

| Param | Type | Required/default | Meaning |
|---|---|---|---|
| `omen` | string | required | The name of Omen (`omens.name`) - the external system to which access is requested. |
| `query` | string | required | Query to Omen: KV-path (vault, including `#field`-projection), promQL (prometheus), index-query (elk). Verified against `Rite.allow` on Keeper. |

`request_id` and `apply_id` the module sets itself (not params): `request_id`
generated by Soul (ULID, unique per-stream, [§5.1 augur.md](../../../keeper/augur.md#51-augurrequest-soul--keeper)),
`apply_id` is taken from the run context.

## Output / register

When `status=ok` the module puts `AugurReply.inline_data`
([`google.protobuf.Struct`](../../../keeper/augur.md))
in register **as is**:

- **scalar** (for example vault-KV `#field`) - object `{ "value": <scalar> }`
(Keeper wraps the scalar in the key `value` - `Struct` does not carry a bare scalar);
- **map** (entire vault KV / Prometheus result / ELK response) - natural
object (the keys of the original map become the register keys).

The `#field` projection is made by **Keeper** when reading Omen - Soul already receives
projected value and does not see the entire secret.

## Error Contract

The step falls (`failed`), the value is not returned, for any of:

- **`denied`** — Keeper authorization rejected (Omen not found / Soul not in Rite
/ query outside allow-list, [§6 augur.md](../../../keeper/augur.md));
- **`error`** — execution failure on Keeper / Omen (external system unavailable);
- **`unspecified`** (default-deny) - the absence of an explicit `ok` is interpreted as
**ban** ([§5.1 augur.md](../../../keeper/augur.md#51-augurrequest-soul--keeper)),
mismatch protection;
- broker is not available in the run (push mode `soul apply` without EventStream session);
- timeout / break stream before response (cancel ctx / close session).

The error message carries the name of Omen for diagnostics, but **not** `query` and **not**
the value itself (query can carry the path to the secret; values/tokens in diagnostics
are not written - [§8 augur.md](../../../keeper/augur.md#8-audit)).

## Example

```yaml
# Live reading of dynamic-secret at apply time (delegate=false: data via Keeper).
- name: Fetch DB credentials at apply time
  module: core.augur.fetch
  register: db_creds
  params:
    omen: vault-prod
    query: database/creds/app-role#password

# Use the value in a subsequent step.
- name: Write app config
  module: core.file.rendered
  params:
    path: /etc/app/config.yml
    template: app-config.tmpl
    vars:
      db_password: ${ register.db_creds.value }
```

## Security

- **Master-credential NEVER hits Soul** (normative invariant
Augur, [§security augur.md](../../../keeper/augur.md)).
In MVP-1 (`delegate=false`) Soul does not receive a credential at all - only a value
inline via Keeper. Delegation (`delegate=true`, scoped-token/static-cred) -
MVP-2, this module is **not** processed (on OK without `inline_data` the module gives
step error).
- **Authorization is on Keeper, not on Soul.** Soul does not decide what he can: he
sends a request, Keeper checks the SID (from mTLS peer cert, not from payload) → covens →
Rite → allow-list. Any failed check → `denied` without value.
- **Default-deny.** `unspecified`-status is interpreted as a ban - absence
explicit `ok` never "continues".
- **Minimize secrets in diagnostics.** No errors are included in error messages
`query`, neither value nor token.
