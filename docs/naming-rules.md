# Словарь имён Soul Stack

Все сущности системы называются по этому словарю — не SaltStack-овскими терминами. Имена обязательны в коде (пакеты, типы), в API (поля, эндпоинты, MCP-инструменты), в конфигурации (флаги, ключи), в логах, метриках и пользовательской документации. Когда параллель с SaltStack помогает читателю, она упоминается один раз в скобках; первичные термины — наши.

Введение нового имени — через правило propose-and-wait в [CLAUDE.md](../CLAUDE.md#новая-сущность--предложить-варианты-и-ждать-propose-and-wait).

## Сущности предметной области

| Имя | Аналог в SaltStack | Смысл |
|---|---|---|
| **Keeper** | master | Хранитель, центральный сервер. Управляющий узел. |
| **Souls** | minions | Управляемые агенты — единичная: «Soul», «душа». |
| **Destiny** | states | Желаемое состояние хоста после прогона. |
| **Soulprint** (Принты) | grains | Факты о системе хоста. |
| **Essence** | pillars | Параметры и секреты, привязанные к душе. |
| **Coven** | — | Метка/тег группы Soul — **только стабильные** логические признаки (кластер, проект, окружение, ЦОД, тип железа). **Роль (master/replica) Coven НЕ является** — она волатильна, получается живым probe (см. [ADR-008](adr/0008-coven-stable-tags.md#adr-008-coven--только-стабильные-логические-теги)). `incarnation.name` — корневая Coven-метка; под-ковенов `{incarnation.name}-{role}` нет. Используется в RBAC, таргетинге и потенциально в маршрутизации. |
| **SoulSeed** | — | Артефакт mTLS-идентичности Soul: пара «сертификат + приватный ключ». Выпускается Keeper-ом, ротируется регулярно, хранится в реестре `soul_seeds` (только fingerprint, без PEM и ключа). |
| **Archon** (Архонт) | (нет; в SaltStack оператор — это просто внешняя identity, не сущность системы) | Оператор Soul Stack, человек или machine-identity. Первый Архонт выпускается командой `keeper init` при инициализации кластера — от него родятся все остальные через обычный OpenAPI/MCP с RBAC-проверкой. Реестр — таблица `operators` в Postgres. Полная фиксация — [ADR-013](adr/0013-bootstrap-archon.md#adr-013-bootstrap-первого-архонта) и [ADR-014](adr/0014-operator-identity.md#adr-014-identity-модель-оператора-archon). |
| **Augur** | (нет прямого аналога; идея близка salt-vault) | Keeper-side подсистема-брокер **живого** (во время рендера / apply) доступа Soul-а к внешним системам (Vault / Prometheus / ELK), которого не покрывает pre-resolved-модель. Две фазы: брокер (`delegate=false`, данные через Keeper) и делегация (`delegate=true`, Soul ходит сам с эфемерным scoped-credential). Полная фиксация — [ADR-025](adr/0025-augur.md#adr-025-augur--keeper-side-брокер-внешнего-доступа-soul), спека — [keeper/augur.md](keeper/augur.md). |
| **Omen** | (приближённо — внешний источник; ср. Provider для облаков) | Внешняя система, к которой Augur посредничает доступ (один Vault-mount / Prometheus / ELK-кластер). Реестр — таблица `omens` в Postgres (managed через OpenAPI/MCP), аналог [Provider](#plugin-инфраструктура-и-cloud). См. [ADR-025](adr/0025-augur.md#adr-025-augur--keeper-side-брокер-внешнего-доступа-soul), [keeper/augur.md](keeper/augur.md). |
| **Rite** | (нет; ср. Vault policy-mapping) | Grant / policy-mapping Augur-а: разрешение «субъект (`coven` XOR `sid`) × Omen → allow-list + `delegate` + ttl/uses». Реестр — таблица `rites` в Postgres. См. [ADR-025](adr/0025-augur.md#adr-025-augur--keeper-side-брокер-внешнего-доступа-soul), [keeper/augur.md](keeper/augur.md). |
| **Sigil** | (нет; в SaltStack/Ansible своего эквивалента нет) | **Печать доверия** Keeper-а к конкретному бинарю плагина: подпись допущенного digest-а. Trust-модель целостности плагинов (`soul-mod-*` / `soul-cloud-*` / `soul-ssh-*`) — Keeper-signed digest-индекс (**Вариант A**), заменяющий TOFU. Keeper — authority над allow-list-ом `(namespace, name, ref) → sha256` (реестр `plugin_sigils` в Postgres); запись допускается **Архонтом явно** через OpenAPI/MCP. При раздаче на host Keeper отдаёт бинарь + подписанный своим ключом фрагмент индекса (digest + подпись = Sigil), host верифицирует до seal/exec. Подпись покрывает manifest с пришитым `binary_sha256` (декларации `side_effects` / `capabilities` не подделываемы). Имя ложится на существующий код seal/`sealDigest` в [`shared/pluginhost`](#go-модули-основного-репо). Полная фиксация — [ADR-026](adr/0026-sigil.md#adr-026-sigil--целостность-плагинов-keeper-signed-digest-индекс), спека — [keeper/plugins.md → Integrity-model](keeper/plugins.md#integrity-model). |
| **Acolyte** | (нет; ср. salt-master worker-thread) | Пул воркеров исполнения apply на каждом Keeper-инстансе (`keeper.acolytes:N`). Заменяет одиночную run-goroutine инстанса-владельца: Acolyte атомарно клеймит per-host задание (`apply_runs`-строку) через `FOR UPDATE SKIP LOCKED`, рендерит его **just-in-time при claim** (резолв секретов в RAM, не в PG), гонит apply и держит барьер своего среза. Любой инстанс через Acolyte подхватывает любое задание → исполнение распределено по кластеру. См. [ADR-027](adr/0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim). |
| **Ward** | (нет; ср. claim/lease задания в очередях) | Claim (под-опека) задания исполнения: колонки `claim_by_kid` / `claim_at` / `claim_expires_at` / `attempt` + статусы `claimed` → `dispatched` на `apply_runs`. «Взять Ward» = атомарно захватить planned-задание (`attempt++` — fencing-epoch). Lifecycle `apply_runs`: `planned → claimed → dispatched → terminal` (статус **`dispatched`** = «отдано Soul-у», отмечается claimed→dispatched **перед** SendApply; `running` — vestigial, только старый синхронный путь `acolytes:0`). recovery-скан Reaper-лидера (`reclaim_apply_runs`) возвращает в `planned` только протухший **`claimed`** (фаза до отдачи) — `dispatched` НЕ реклеймится (после отдачи владеет Soul). Зависший `dispatched` (Keeper и Soul оба мертвы после отдачи) закрывается терминалом **`orphaned`** (Soul-reconcile, S6): Soul на reconnect декларирует ведомые `apply_id` через `WardRoster`, Keeper терминалит `dispatched`-строки SID-а вне набора (`applyrun.OrphanDispatched`); барьер классифицирует `orphaned` как фейл (incarnation → `error_locked`). Нецелевой roster-хост (Acolyte-путь пишет `planned` на КАЖДЫЙ roster-хост ДО per-host резолва `on:`/`where:`; после резолва осталось 0 задач) закрывается терминалом **`no_match`** (FINDING-01 вариант (б); НЕ `success` — apply_runs не over-reports «успех» на нецелевых хостах); барьер классифицирует `no_match` как ТЕРМИНАЛ и НЕ-провал (benign, как `success`): прогон, где целевые `success` + не-целевые `no_match`, ведёт incarnation в `ready`. Полное множество терминалов `apply_runs.status`: `success` / `failed` / `cancelled` / `orphaned` / `no_match`. См. [ADR-027](adr/0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim) (amend «GATE-1: deliver-once recovery» + «S6: Soul-reconcile» + «FINDING-01: no_match нецелевого хоста»). |
| **Summons** | (нет; ср. notify-сигнал) | pub/sub-сигнал «появились planned-задания» (Redis, **best-effort**) + periodic poll fallback на стороне Acolyte. Потеря сигнала не теряет задание — poll всё равно подхватит `planned`. Redis-topic — **`apply:summons`** (зафиксировано, propose-and-wait пройден; стиль `<подсистема>:<событие>` как `rbac:invalidate`). См. Реестр — таблица `vigils` в Postgres (managed через OpenAPI/MCP, toggle + RBAC). Активный набор для хоста едет ему через `VigilSnapshot` (ReplaceAll). См. [ADR-030](adr/0030-vigil-oracle.md#adr-030-vigil--oracle--event-driven-мониторинг-beacons--reactor) + [amendment 2026-05-26](adr/0030-vigil-oracle.md#amendment-2026-05-26-s5-closure). |
| **Portent** | beacon-событие | Событие, которое Vigil поднимает Soul → Keeper при смене состояния хоста (**edge-triggered**). Едет only-add в `EventStream`-oneof (`PortentEvent` в `FromSoul`). Payload — `google.protobuf.Struct` в MVP (typed — отдельным ADR, как Soulprint). Для Oracle — **недоверенный вход** (Soul может быть скомпрометирован). См. [ADR-030](adr/0030-vigil-oracle.md#adr-030-vigil--oracle--event-driven-мониторинг-beacons--reactor). |
| **Oracle** | reactor (роутер) | Keeper-side reactor-подсистема: приём Portent → match по реестру Decree → постановка named-scenario в work-queue ([ADR-027](adr/0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim)). Сам apply не исполняет — только маршрутизирует. См. [ADR-030](adr/0030-vigil-oracle.md#adr-030-vigil--oracle--event-driven-мониторинг-beacons--reactor). |

Опц. `where` — keeper-local CEL-sandbox (корень `event`). **Action = ТОЛЬКО named scenario** (whitelist через work-queue; raw-команда отвергнута как RCE-вектор — команды доступны через `core.exec.run` внутри scenario). Реестр — таблица `decrees` в Postgres (managed через OpenAPI/MCP). См. [ADR-030](adr/0030-vigil-oracle.md#adr-030-vigil--oracle--event-driven-мониторинг-beacons--reactor). |
| **Scry** | (ср. Salt `test=True` / Ansible `--check`) | **Подсистема drift-детекта, read-only.** Считает drift = «dry-run reconcile показал бы `changed=true` хотя бы на одном **декларированном** ресурсе» (declarative drift, НЕ полный дамп хоста). | **Leg** | (нет; «переход» внутри плавания; ср. Surge / Ansible batch / k8s rollout-wave) | Единица батча [Voyage](#сущности-предметной-области)-прогона ([ADR-043](adr/0043-voyage.md#adr-043-voyage--унифицированный-батчевый-прогон)) — один последовательный «отрезок пути». ## Идентификаторы

| Имя | Смысл |
|---|---|
| **SID** | Soul ID. Равен FQDN хоста — стабильный, человекочитаемый, даёт автоматический дедуп при переустановке. |
| **KID** | Keeper ID. Стабильный идентификатор инстанса в Keeper-кластере; используется в lease, аудит-логах и метриках. Человекочитаемая kebab-case строка, regex `^[a-z][a-z0-9-]{0,62}$` (по аналогии с AID, без обязательного префикса). |
| **AID** | Archon ID. Идентификатор оператора. Строчная ASCII-строка, regex `^[a-z0-9][a-z0-9._@-]{1,127}$` (примеры: `archon-alice`, `archon-ops-01`, email-подобный `alice@corp.com`, ldap-uid `uid-4815`). Первый символ — буква/цифра; charset `a-z 0-9 . _ @ -`. Обязательный префикс `archon-` снят (ADR-014 amendment 2026-05-29) ради LDAP/Keycloak auto-provision внешних identity. Charset намеренно безопасный: нет `/`/`\` (path-traversal), только ASCII-lowercase (нет unicode-двойников и регистра), нет управляющих/кавычек (нет инъекций). Используется в RBAC (membership `rbac_role_operators`), FK-полях аудита (`created_by_aid`, `changed_by_aid` и т.п.), API/MCP-call-context. Свободно от конфликтов с известными стандартами (не OID/ASN.1, не DID/W3C, не PID/GID/unix). |

| `soul-legion` | **Test-only** нагрузочный stub-генератор: плодит N фейк-Souls, каждый держит реальный `EventStream` (gRPC bidi-mTLS) к Keeper для load-теста Keeper / PG / Redis. **НЕ поставочный бинарь** — [ADR-004](adr/0004-binaries.md#adr-004-раскладка-бинарей--keeper-soul-soul-lint-push-режим--модуль-внутри-keeper) фиксирует ровно три артефакта (`keeper` / `soul` / `soul-lint`), а это test-артефакт в `tests/load/`, не раздаётся операторам. См. [«Soul Legion»](#soul-legion) ниже. |

### Trial

**Trial** (испытание) — прогон Destiny/Scenario или state_schema-миграции на фиксированных входных данных (fixtures) с проверкой результата. Метафора: испытание судьбы перед воплощением. Инструмент — бинарь `soul-trial`. Метрика покрытия DSL — **`trial coverage`** (какие задачи / CEL-ветки / `enum`-значения / `state_changes`-переходы прогнаны), в отличие от **`code coverage`** (обычное Go-покрытие ядра). Уровни L0 (render-only hermetic) / L1 (migration) — MVP; L2/L3 — post-MVP. Полная фиксация — [ADR-023](adr/0023-trial-test-runner.md#adr-023-тест-раннер-trial-soul-trial-и-dsl-coverage), спека — [docs/destiny/testing.md](destiny/testing.md).

### Soul Legion

**Soul Legion** (`soul-legion`) — **test-only** нагрузочный stub-генератор: плодит N фейк-Souls, каждый из которых держит реальный `EventStream` (gRPC bidi-mTLS) к Keeper, чтобы создать нагрузку на Keeper / PG / Redis под масштабом, недостижимым функциональными уровнями (тысячи-десятки тысяч стримов). Stub **НЕ парсит Destiny и НЕ применяет** — измеряет нагрузку на Keeper, а не реализм apply на хосте. Метафора: «легион» = множество душ («имя мне легион, ибо нас много»). **НЕ поставочный бинарь** — [ADR-004](adr/0004-binaries.md#adr-004-раскладка-бинарей--keeper-soul-soul-lint-push-режим--модуль-внутри-keeper) фиксирует ровно три артефакта (`keeper` / `soul` / `soul-lint`); `soul-legion` — test-артефакт (пакет/бинарь `soul-legion` в `tests/load/`), строится на фундаменте `soulstub` и операторам не раздаётся. План — [docs/testing/load-testing.md](testing/load-testing.md).

Расширение перечня — propose-and-wait. |
| **Charon** (Харон) | **Зарезервированное имя** на случай, если scope Жнеца расширится за рамки cleanup-а (миграции таблиц, перенос архивных данных, GC холодных слоёв). Сейчас не используется. |
| **Conductor** (Дирижёр) | **Leader-elected исполнитель [Cadence](#сущности-предметной-области)-расписаний** ([ADR-048](adr/0048-conductor.md#adr-048-conductor--leader-elected-исполнитель-cadence-расписаний)). Отдельная keeper-side подсистема (не бинарь, как Reaper), вынесенная из Reaper: cleanup-домен Reaper (`reaper.interval` 1h) и scheduling-домен Cadence имеют разный естественный ритм, общий тик не даёт Cadence спавнить часто. Свой Redis-lease **`conductor:leader`** ([ADR-006](adr/0006-cache-redis.md#adr-006-кэш-и-координация--redis) single-executor; **независим** от `reaper:leader` — лидер Conductor и лидер Reaper могут быть на разных инстансах) + свой tick-interval (блок конфига **`cadence_scheduler`** в `keeper.yml`, рекомендация ~15–30s). Сидит на generic-примитиве `keeper/internal/leaderloop` (общий с Reaper). Переносит spawn-логику Cadence из Reaper **без изменений** (due-выборка + три `overlap_policy` + пересчёт `next_run_at` + spawn-tx); Reaper теряет правило `spawn_due_cadence` и `action: spawn`. Audit спавна остаётся `source: background` (новый source `scheduler` НЕ вводится). **Default-ON при наличии Redis** (footgun-guard: Cadence без планировщика молча не спавнит). Метафора — дирижёр, задающий ритм исполнения (музыкально-душевная линия рядом с Choir/Voice/Cadence). **NB:** имя коллизит с Netflix/Temporal Conductor — коллизия принята осознанно (пользователь, 2026-06-02). |
| **Conclave** (Конклав) | Реестр **живых Keeper-инстансов** кластера в Redis: presence keeper-инстанса с TTL + renew (ADR-006 amend, soul-shedding S1). Каждый инстанс на старте регистрирует свой ключ `keeper:instance:<kid>` (имя ключа техническое, как `soul:<sid>:lock`; сущность в словаре = Conclave) с TTL и продлевает его renewal-goroutine-ой; graceful shutdown — `DEL`, crash — TTL-expiry. **НЕ эксклюзивный lock** (в отличие от [Reaper](#модули-и-подсистемы-внутри-keeper)-lease / SoulLease): каждый инстанс держит свой ключ по своему KID, конкуренции за один ключ нет. Авторитет presence — Redis (инвариант presence→Redis; PG-вариант отвергнут). Перечисление живых (`LiveKIDs` / `CountLive`) — `SCAN keeper:instance:*` (единый источник истины по TTL-ключам, без отдельного Redis-Set — как `SoulsStreamAlive`). Питает refuse-guard «я не один» (`CountLive > 1`) и soul-shedding (есть куда уходить). Метрика — `keeper_conclave_instances` (gauge живых инстансов). Коллизия KID (два keeper-процесса с одинаковым `kid` в конфиге) — warn + регистрация поверх (ошибка конфигурации оператора, не блокирующая). |
| **Shepherd** (Пастырь) | **Балансировка нагрузки Souls между Keeper-инстансами** при горизонтальном scale-out (PLANNED, backlog 2026-05-25; имя выбрано пользователем, propose-and-wait пройден). Проблема: новый Keeper за LB простаивает — существующие долгоживущие EventStream-стримы залипают на старых инстансах (LB балансирует только новые соединения; Soul сам не реконнектится, пока стрим не оборвётся). Механизм: инстансы публикуют снимок нагрузки (число стримов; опц. глубина Acolyte-очереди) в [Conclave](#модули-и-подсистемы-внутри-keeper)-записи (renew-тик 10s); инстанс с перекосом выше справедливой доли (`сумма стримов / CountLive`) сбрасывает **излишек** своих стримов частичным `StreamManager.CloseAll` (не все, в отличие от Watchman) с jitter/cap → Souls реконнектятся и раскидываются. **Домен балансировки = priority-группа / один VIP** (разные priority = failover, не баланс). Отличие от Watchman: Watchman = бинарный полный сброс при ИЗОЛЯЦИИ; Shepherd = частичный сброс ради РАВНОМЕРНОСТИ. drain-режим (авто-порог / полу-авто / ручной) — TBD. Метрики (план) — `keeper_shepherd_*` (stream-count gauge, shed-total). **Реализация отложена** (ADR-002/006 amendments). |
| **Watchman** | **Изоляция-детект + soul-shedding** одного Keeper-инстанса (soul-shedding S2, ADR-002 HA-кластер). Фоновая goroutine: периодически (`watchman_interval`, дефолт 5s) пингует PG+Redis — те же зависимости, что `/readyz`. **Debounce/flap-guard**: изоляция объявляется только после `watchman_fail_threshold` (дефолт 3) подряд идущих провалов probe (единичный сетевой spike не сбрасывает весь флот стримов — thundering-herd reconnect). При устойчивой изоляции — **активно закрывает ВСЕ локальные EventStream-стримы** (hard-close, без drain): отменяет per-stream ctx каждого (`StreamManager.CloseAll`) → handler возвращается → gRPC шлёт Soul-у EOF → Soul по reconnect-loop/failback-list уходит на живой Keeper. При восстановлении зависимостей — возобновляет нормальную работу (новые стримы принимаются, lease-renewal/Acolyte оживают сами). **Решение «я изолирован» централизовано в Watchman** — НЕ дублируется в per-stream renewal-loop-ах. Метрики — `keeper_watchman_isolated` (gauge 0/1) и `keeper_watchman_streams_shed_total`. (Имя `Sentinel` отвергнуто — коллизия с Redis Sentinel.) |
| **Toll** | (нет; аналог Health-checks, но cluster-wide rate-of-loss) | Detector массового оттока Soul-ов на cluster-level ([ADR-038](adr/0038-toll.md#adr-038-toll--cluster-wide-detector-массового-оттока-souls)). Sliding 60s window, threshold 20% от baseline connected. Per-instance gRPC-disconnect observer + Redis-leader агрегация. Выставляет cluster:degraded flag → блокирует POST scenarios/push-apply. Hysteresis с 60s grace на clearing. Warmup 60s после старта инстанса (cluster restart immunity). Имя — propose-and-wait (звон колокола / death toll, souls-метафора, не коллизит с soul-shedding=Watchman). |
| **`keeper/internal/soulpurview`** | **Резолвер scoped-видимости списка Souls по [Purview](#purview--scope-резолвер-adr-047)** ([ADR-047](adr/0047-purview.md#adr-047-purview--scoped-rbac-видимость-узлов-role-default_scope--расширенный-селектор), S3b; имя выбрано пользователем, propose-and-wait пройден). Souls-аналог `keeper/internal/statepredicate` (тот резолвит инкарнации по state-CEL; этот — souls по scope-измерениям Purview). Принимает уже-резолвнутый `rbac.Purview` **параметром** (не ходит в enforcer сам — `ResolvePurview` зовёт handler), переводит scope-границу в параметры souls-запроса. Однонаправленная зависимость `soulpurview→rbac`. **NB (ADR-047 S4):** command-путь Voyage (`errand.run`, [ADR-043 §6](adr/0043-voyage.md#adr-043-voyage--унифицированный-батчевый-прогон)) реюзит `soulpurview` для пересечения таргета с Purview — **новых RBAC-селектор-ключей для `errand.run` НЕ добавляется** (ключи RBAC ограничены {service, coven, incarnation, host}; пересечение через тот же Purview-механизм, гибрид-семантика 403/урезание/422 — [ADR-047 §S4](adr/0047-purview.md#s4--target--purview-для-command-пути-voyage-security-fix)). |
| **notification-dispatcher** (tap) | **Диспетчер уведомлений Herald/Tiding** ([ADR-052](adr/0052-herald-notifications.md#adr-052-herald--tiding--уведомления-о-событиях-прогонов)). **tap** — multi-writer декоратор поверх `audit.Writer` (точка [ADR-022(f)](adr/0022-audit-pipeline.md#adr-022-audit-pipeline-storage-schema-retention)): после успешной PG-записи audit-события отдаёт его dispatcher-у (audit — primary, сбой dispatcher-а не откатывает audit-write). **dispatcher** матчит событие против включённых [Tiding](#сущности-предметной-области)-правил (`event_types`-area-glob + `only_failures`/`only_changes` + селекторы `incarnation`/`cadence`/`voyage_id` для ephemeral) и ставит задание на доставку — **инвариант горячего пути: dispatcher делает только «match + enqueue копии»** (поля гибкого тела `annotations`/`projection` он НЕ трогает, [ADR-052 Amendment](adr/0052-herald-notifications.md#amendment-2026-06-11-разовые-уведомления-в-run--гибкое-тело)). **Доставка** — at-least-once через claim-queue worker (parity [VoyageWorker](#сущности-предметной-области)/[ADR-027(d)](adr/0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim): `FOR UPDATE SKIP LOCKED` + lease + `attempt++`), retry+backoff; статусы попыток — Redis (hot→Redis, [ADR-006](adr/0006-cache-redis.md#adr-006-кэш-и-координация--redis)); терминалы — audit `herald.delivered`/`herald.failed`. **Merge `annotations` + применение `projection`** к payload-копии при сборке `webhookPayload` — в worker-е (off-path), не в dispatcher-е. **Scope MVP — события прогонов** (см. [Tiding](#сущности-предметной-области)). |
| **Tempo** | **Per-AID rate-limiter write-API** (Redis token-bucket) ([ADR-050](adr/0050-tempo.md#adr-050-tempo--per-aid-rate-limiting-write-api); имя выбрано пользователем, propose-and-wait пройден). Сквозной ограничитель частоты обращений оператора (по `claims.Subject` = AID) к resolver-тяжёлым write-эндпоинтам — **третий anti-DoS-слой** после body-limit и [Toll](#модули-и-подсистемы-внутри-keeper). Метафора — музыкальная линия рядом с [Conductor](#модули-и-подсистемы-внутри-keeper)/[Cadence](#сущности-предметной-области)/[Choir](#сущности-предметной-области) («допустимый темп обращений к API»). **Граница с Cadence:** Cadence = **когда спавнить** Voyage; Tempo = **как часто оператор дёргает API**. |

## Модули Destiny

Модуль — единица работы Destiny (поставить пакет, развернуть файл, перезапустить сервис, …). См. разделы [«Модель модулей»](architecture.md#модель-модулей) и [«Адресация модулей»](architecture.md#адресация-модулей) в архитектуре.

Модули адресуются трёхуровневой формой `<namespace>.<module>.<state>`:

| Уровень | Роль |
|---|---|
| **namespace** (коллекция) | Префикс коллекции / автора. `core` для встроенных, `official` для первых 10 SoulModule-плагинов первого release-cycle ([ADR-016 amendment 2026-05-27](adr/0016-parity-license.md#adr-016-стратегия-parity-с-saltstackansible-и-лицензия-soul-stack); зарезервирован для soul-stack-issued, Sigil-подписывается keeper-cluster-issuing-key), `wb` / `community` / … для сторонних. Имя сущности в Soul Stack-словаре пока не закреплено (open Q, см. [module-collections.md](module-collections.md)); до решения используем нейтральное «коллекция / namespace». |
| **module** | Объект управления — `pkg`, `file`, `service`, `user`, `exec`, `template`, `http`, `haproxy`, … |
| **state** | Желаемое состояние объекта (declarative): `installed`, `absent`, `latest`, `present`, `running`, `stopped`, `restarted`, `enabled`. Для non-stateful (read-probe / императивных) модулей — verb-форма (`run`, `shell`, `probe`). |

Сопутствующие термины:

| Имя | Роль |
|---|---|
| **Core-модули** | Базовые модули, статически встроенные в `soul`-бинарь (**Soul-side**) либо в `keeper`-бинарь (**Keeper-side**). Живут в namespace `core`. Адресуются как `core.pkg.installed`, `core.file.present`, … Диспетчер Soul-side/Keeper-side — scenario-ключ `on:` ([scenario/orchestration.md §3](scenario/orchestration.md#3-таргет-шага--on)): Soul-side core — `on:` опущен или coven-метки; Keeper-side core — `on: keeper`. Первый Keeper-side core — `core.soul.registered` ([keeper/modules.md](keeper/modules.md)). |
| **Custom-модули** | Внешние модули, поставляемые как отдельные исполняемые файлы `soul-mod-<имя>`. Запускаются `soul`-бинарём как sub-process с gRPC-stdio handshake. Живут в namespace своей коллекции. |
| **`soul-mod-<имя>`** | Стандартное имя файла custom-модуля. Например, `soul-mod-haproxy`, `soul-mod-myapp`. |
| **SoulModule API** | gRPC-контракт между `soul`-бинарём (host) и модулем (sub-process). Сервис `SoulModule` с методами `Validate`, `Plan`, `Apply` ([ADR-020(a)](adr/0020-plugin-infrastructure.md#adr-020-plugin-инфраструктура-формат-manifest-handshake-lifecycle): `Manifest()` RPC в MVP не вводится; манифест — статический `manifest.yaml`). Параметр `state` — обязательная часть вызова. |

Версия самого модуля как артефакта в манифесте **не указывается** — это git ref, под которым он закоммичен (см. [ADR-007](adr/0007-versioning-git-ref.md#adr-007-версионирование-артефактов--через-git-ref-а-не-через-поле-в-манифесте)). Манифест используется `soul-lint`-ом для локальной валидации Destiny. |

### SDK marker-интерфейсы

Optional marker-интерфейсы в `sdk/module/`. Модуль их реализует, чтобы декларировать capability host-у. Host применяет default-deny: без декларации — путь не вызывается. Метод носит то же имя, что и интерфейс; тело пустое.

| Интерфейс | Декларирует | Кто проверяет | Default-deny при отсутствии |
|---|---|---|---|
| **`PlanReadSafe`** | Plan — pure-read (НЕ мутирует хост) | Soul-runner (apply-stream) | dry_run task → `plan.unsupported` |
| **`ErrandReadSafe`** | Модуль безопасен к вызову через Errand (НЕ мутирует incarnation.state) | Soul-side Errand-runner | reject с `errand_module_not_allowed` |

`BaseModule` НЕ реализует ни один marker по умолчанию.

## Service / Incarnation / Scenario

См. разделы [«Service — структура и manifest»](architecture.md#service--структура-и-manifest) и [«Incarnation — runtime-инстанс сервиса»](architecture.md#incarnation--runtime-инстанс-сервиса). Спецификация scenario — [`docs/scenario/`](scenario/README.md).

| Имя | Роль |
|---|---|
| **Service** | Тип сервиса — переиспользуемое определение (Redis HA, PostgreSQL, Vector). Один service = один git-репо. Содержит `service.yml`, `essence/_default.yaml`, `scenario/<name>/main.yml`, `migrations/`, `tests/`. |
| **`lifecycle.auto_create`** / **`lifecycle.auto_destroy`** (манифест `service.yml`) | Опциональный блок `lifecycle:` в `service.yml` — политика жизненного цикла инкарнаций. `auto_create: bool` (default `true`) — `POST /v1/incarnations` сразу запускает scenario `create`; `false` — инкарнация создаётся в `ready` без прогона, оператор запускает `create` вручную. `auto_destroy: bool` (default `true`) — удаление запускает teardown `destroy` по логике `allow_destroy`; `false` — удаление всегда прямое, без teardown (приоритет над `allow_destroy`). Отсутствие блока = оба `true` (backcompat). См. [Service — структура и manifest](architecture.md#service--структура-и-manifest). |
| **`runnable`** (поле каталога сценариев) | bool-поле в ответе `GET /v1/services/{name}/scenarios` — признак «сценарий запускаем оператором из Run-формы». `create` = `true`; operational (вкл. `converge`) = `true`; `destroy` = `false` (спец-флоу удаления через `DELETE /v1/incarnations/{name}`). UI фильтрует Run-форму по `runnable`, не по хардкоду имён ([ADR-042](adr/0042-backend-driven-ui.md#adr-042-backend-driven-dynamic-data-в-ui--ui-не-хардкодит-динамические-каталоги)). |
| **Incarnation** | Конкретный runtime-инстанс сервиса (один кластер). Хранится в Postgres, не в git. Имеет `spec` (что задекларировал оператор), `state` (текущая конфигурация), `status` (running / error_locked / drift / …). |
| **Scenario** | Операция над incarnation: `create`, `add_user`, `update_acl`, `add_replica`, `restart`, и т.п. Имеет typed `input:` и описание `state_changes`. Папка `scenario/<name>/` в service-репо, точка входа `main.yml`. После [ADR-009](adr/0009-scenario-dsl.md#adr-009-scenario--полная-dsl-задач-destiny-граница-с-destiny--рекомендация) наследует полное DSL-ядро задач destiny ([destiny/tasks.md](destiny/tasks.md)) + оркестрационная дельта ([scenario/orchestration.md](scenario/orchestration.md)). |
| **converge** (scenario-kind, operational) | **Operational** scenario-kind, описывающий ЖЕЛАЕМОЕ конечное состояние сервиса ([Scry](#сущности-предметной-области), [ADR-031 amendment 2026-06-10](adr/0031-scry-drift.md#adr-031-scry--drift-detection-declarative-dry-run-reconcile)). **НЕ входит в lifecycle-набор** (`LifecycleScenarioNames` = `create`/`destroy`). Несёт двойную роль: как обычный `run` — Apply-reconcile (реально сводит хосты к декларации, `runnable: true`); как dry-run target `check-drift` — declarative drift-detect. Файл `scenario/converge/main.yml` в service-репо. Распознаётся по имени (auto-discover, симметрично остальным сценариям — никакого нового YAML-поля; drift-функция грузит файл по константе `ConvergeScenarioName`). Сервис, поддерживающий drift-проверку, ОБЯЗАН его положить; отсутствие файла → `keeper.incarnation.check-drift` возвращает «drift-проверка недоступна» (информационно, не ошибка). converge-input резолвится автоматически по конвенции имени из `incarnation.state.<param>` + opt-override оператора. В `dry_run`-режиме Keeper зовёт `mod.Plan` вместо `mod.Apply`, собирает per-task `changed` и формирует [`DriftReport`](#сообщения-proto-keepersoul). |
| **create** / **destroy** (scenario-kind, lifecycle) | Lifecycle-набор (`LifecycleScenarioNames`) — два специализированных scenario-kind-а фаз жизненного цикла. **`create`** — bootstrap инкарнации (`runnable: true`, запускается `POST /v1/incarnations` при `lifecycle.auto_create: true` либо вручную из Run-формы; rerun из `error_locked` — `POST /v1/incarnations/{name}/rerun-create`). **`destroy`** — teardown-операция (S-D2b, [ADR-027](adr/0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim); файл `scenario/destroy/main.yml`, `runnable: false` — спец-флоу удаления через `DELETE /v1/incarnations/{name}` / `keeper.incarnation.destroy`, не из Run-формы). |

### Scenario: ключи оркестрации

Дельта scenario поверх DSL-ядра задач — см. [`docs/scenario/orchestration.md`](scenario/orchestration.md).

| Ключ | Роль |
|---|---|
| **`on:`** | Стабильный таргет шага: где выполняется (резолв по Postgres). Опущен = весь incarnation (корневой coven неявно); `keeper` = keeper-сторона; `[coven, …]` = пересечение covens, всегда ⊆ хостов incarnation. Кросс-incarnation запрещён грамматикой. |
| **`where:`** | Волатильный per-host предикат-строка по `register:` предыдущего probe-шага. Резолв в рантайме, не по Postgres. Ссылка на register в `where:` — канон-форма `register.<name>.*` (как в DSL-ядре, [destiny/tasks.md §10](destiny/tasks.md#10-шаблонный-контекст)); голая форма `<name>.*` запрещена (ошибка валидации). **`where:`-ключ шага** (на каких хостах делать) и **`soulprint.where(...)`-функция** в выражении (откуда взять данные хоста) — разные позиции, не путать. |
| **`apply:`** | Делегирование в изолированную destiny: `apply: { destiny: <name>, input: {…} }`. Альтернатива `module:` в scenario-задаче. |
| **`serial:`** | Волновое (rolling) cross-host исполнение шага: волны по ≤N хостов (целое `1..M` или `"<N>%"`), порядок хостов по `SID`, волны последовательны. Применим к module/apply/`block:`-задаче. Подробности — [`docs/scenario/orchestration.md §2`](scenario/orchestration.md#2-дельта-scenario-относительно-dsl-ядра). |
| **`run_once:`** | Шаг на одном хосте таргета (первый по `SID` из резолва `on:`+`where:`); `bool`, default `false`. Взаимоисключающ с `serial:`. Подробности — [`docs/scenario/orchestration.md §2`](scenario/orchestration.md#2-дельта-scenario-относительно-dsl-ядра). |

> **`serial:`/`run_once:` — ось «хосты таргета».** Ортогональны `parallel:`/`loop:` ([destiny/tasks.md §6–§7](destiny/tasks.md#6-параллелизм-parallel-true)), которые работают по оси данных / тредов на одном хосте.

> **Граница `when:` vs `where:`.** `when:` — **делать ли шаг вообще** (булево, как в [destiny/tasks.md §9](destiny/tasks.md#9-прочность-и-контроль-исполнения)). `where:` — **на каких хостах** выполнить шаг (per-host фильтр таргета по `register:`). Это ортогональные ключи: `when:` решает «шаг идёт / пропущен», `where:` (на хостах, где шаг идёт) сужает множество хостов.

### `state_changes`: CRUD-глаголы и ключи

`state_changes` сценария — **упорядоченный список операций** (YAML-список, не map), каждый элемент = один **CRUD-глагол** (сингуляр). Грамматика — [ADR-057](adr/0057-state-changes-crud-verbs.md), нормативная спека — [`docs/scenario/orchestration.md §7.1`](scenario/orchestration.md#71-грамматика-state_changes--список-crud-операций). Множественность выражается **`match`-предикатом**, не флагами/ручками.

| Глагол | Роль |
|---|---|
| **`set`** | `set: <field>` + `value: "${CEL}"` — перезапись поля `incarnation.state.<field>` целиком (заменяет прежний `sets`-map). |
| **`add`** | `add: <collection>` — добавить элемент. map-коллекция: `key:` + `value:`. list-коллекция: `value:` + опц. `match:` (дедуп). `on_conflict: skip` (default) `\| replace \| error`. |
| **`modify`** | `modify: <collection>` + `match:` + `patch: { <путь-в-элементе>: "${CEL}" }` — патч **всех** подходящих под предикат (all-by-default). |
| **`remove`** | `remove: <collection>` + `match:` — удалить **всех** подходящих под предикат (all-by-default). |
| **`foreach`** | `foreach: "${CEL-список\|map}"` + `as: <имя>` + `do: { <глагол…> }` — bulk fan-out N операций. Форма буквально из migration-DSL ([ADR-019](adr/0019-state-migration-dsl.md#adr-019-state_schema-migration-dsl)). |

| Ключ | Роль |
|---|---|
| **`value:`** | Значение для `set`/`add`: CEL-выражение `${ … }` либо YAML-литерал (map/list/scalar). |
| **`key:`** | Ключ записи при `add` в map-коллекцию: `key: "${CEL}"`. |
| **`match:`** | CEL-предикат над элементом коллекции — выбор подходящих в `modify`/`remove`, дедуп-предикат в `add`(list). all-by-default; empty-match → no-op. |
| **`patch:`** | Map `<путь-в-элементе>: "${CEL}"` для `modify` — что переписать в каждом зацепленном элементе. |
| **`on_conflict:`** | Поведение `add` при коллизии (ключ map занят / list-`match` уже находит): `skip` (default, идемпотентно) `\| replace \| error`. |
| **`expect:`** | Опц. runtime-ассерт кратности для `modify`/`remove`: `one \| at_most_one \| any` (default `any`). Зацепил ≠ ожидаемого → `error_locked` до коммита. |

| Биндинг CEL | Роль |
|---|---|
| **`elem`** | Текущий элемент list-коллекции (или скаляр) в `match`/`patch`/`value`. Имя `elem` (не `self`) — во избежание коллизии с per-host `soulprint.self`. |
| **`key`** / **`value`** | Ключ / значение текущей записи map-коллекции в `match`/`patch`/`value`. |

> Поверх биндингов доступен полный sets-контекст (`input.*` / `incarnation.*` / `soulprint.self.*` / `register.*` / `vars.*` / `essence.*`) — тот же CEL-env, что у `params:` задач ([ADR-010](adr/0010-templating.md#adr-010-шаблонизатор-cel-для-yaml-выражений-go-texttemplate-для-файлов)).

> **`remove` (state_changes) ≠ `delete` (migration-DSL) — намеренно разные имена.** `remove` ([ADR-057](adr/0057-state-changes-crud-verbs.md)) = «вынуть элемент из коллекции» по `match`-предикату над элементами (runtime-сценарий). `delete` ([ADR-019](adr/0019-state-migration-dsl.md#adr-019-state_schema-migration-dsl), [migrations.md](migrations.md)) = «снести путь в state-структуре» по `path:` (миграция формы state при смене схемы). Не путать «удалить элемент коллекции» и «удалить поле/путь схемы».

> **Не вводятся** ([ADR-057](adr/0057-state-changes-crud-verbs.md) отвергнутые альтернативы): `clear` (= `set` пустым), `rename`/`move` (это migration-DSL), `upsert` (= `add` + `on_conflict: replace`), позиционный `remove` (первый/последний — невоспроизводимо в JSONB), парные глаголы `*_one`/`*_all`, флаг `all:` (множественность = свойство `match`, не имени/флага).

> **Transit (breaking).** `state_changes` сменил форму map → list. Старая map-форма (`sets:`/`appends:`/`modifies:`) парсится **один релиз** как DEPRECATED (dual-parse + soul-lint warn), затем удаляется. `appends`/`modifies` были no-op-плейсхолдерами без источника значения (state не рос) — баг закрыт глаголами `add`/`modify`.

### Поле задачи `id:` — адрес для алертов

| Ключ | Роль |
|---|---|
| **`id:`** | Опциональный стабильный адрес задачи (формат snake-case `^[a-z][a-z0-9_]*$`) для подписки на per-task-changed-алерты «таска X изменила» ([ADR-052 §j](adr/0052-herald-notifications.md), [ADR-009 amendment](adr/0009-scenario-dsl.md#adr-009-scenario--полная-dsl-задач-destiny-граница-с-destiny--рекомендация)). Поле DSL-ядра задачи (destiny + scenario), не scenario-only. Взаимоисключим с `register:` (с register адрес уже есть → `id:` для задач **без** register); один формат — register и id в **одном адресном пространстве** подписки. Pilot: допустим только на module-задаче. Грамматика — [destiny/tasks.md §3](destiny/tasks.md#3-полный-список-блоков-задачи). |

> **Не путать `id:` с другими идентификаторами.** `id:` — адрес задачи **для алертов** (snake-case, как `register:`). Это **не** `register:` (тот захватывает output задачи **и** служит адресом для requisites; `id:` — ТОЛЬКО адрес, output не несёт) и **не** идентификаторы сущностей `sid` (FQDN хоста) / `aid` (Archon ID, `archon-…`) / `apply_id` (идентификатор apply-задания). Адресное пространство подписки на per-task-changed — `register ∪ id`.

### Voyage: invocation- и config-ключи

Параметры батчевого прогона [Voyage](#сущности-предметной-области) ([ADR-043](adr/0043-voyage.md#adr-043-voyage--унифицированный-батчевый-прогон) + [amendment 2026-06-01](adr/0043-voyage.md#adr-043-voyage--унифицированный-батчевый-прогон) + [amendment 2026-06-09](adr/0043-voyage.md#adr-043-voyage--унифицированный-батчевый-прогон)). Уровень **request** — опт-поля invocation-payload `/v1/voyages`; уровень **config** — оператор-инварианты в `keeper.yml::voyage`. Все имена — Soul Stack-словарь, без SaltStack-дословщины (`max_failures` вместо `failhard`, `require_alive` вместо `minion`-присутствия).

| Ключ | Уровень | Роль |
|---|---|---|
| **`batch`** | request | Размер пачки строкой: `N` (абсолютное число хостов) \| `N%` (процент 1–100 от резолвнутого scope, `N%` → `ceil(scope·pct/100)`). Парсит Keeper (грамматика `^(\d+)%?$` после trim, fail-closed → `422 validation-failed`). Единое поле + удобный CLI (`--batch 20%`). Только `batch_mode: barrier`. На резолве распадается на `batch_size`/`batch_percent` (additive). |
| **`max_failures`** | request | Порог провалов строкой: `N` (абсолют) \| `N%` (процент, `N%` → `ceil(scope·pct/100)`). Считается по **единицам прогона** (инкарнации для `kind=scenario` / хосты для `kind=command`); `cancelled`/`no_match` НЕ считаются провалами. `on_failure: abort` ≡ `max_failures: 1`; `continue` ≡ порог не задан. Работает в обоих `batch_mode`. Замещает `fail_threshold`. |
| **`batch_mode`** | request | Режим батчинга: `barrier` (default, реализован — Leg = пачка `batch` + барьер между Leg) \| `window` (скользящее окно — пул держит `concurrency` активных единиц без барьеров; окно по хостам для `kind=command`, по инкарнациям для `kind=scenario` с сохранением внутри-инкарнационного barrier+commit, [ADR-009 §7](adr/0009-scenario-dsl.md#adr-009-scenario--полная-dsl-задач-destiny-граница-с-destiny--рекомендация)). |
| ~~**`batch_size`**~~ | request | **Deprecated** (amendment 2026-06-09), принимается. Размер пачки как абсолют (int). Замещён `batch`. Смешение с `batch`/`batch_percent` → `422` `voyage_batch_spec_conflict`. |
| ~~**`batch_percent`**~~ | request | **Deprecated** (amendment 2026-06-09), принимается. Размер пачки как % от scope (int 1..100, parity Salt `-b 25%`), XOR с `batch_size`. Замещён `batch`. Смешение с `batch` → `422` `voyage_batch_spec_conflict`. Только `batch_mode: barrier`. |
| ~~**`fail_threshold`**~~ | request | **Deprecated** (amendment 2026-06-09), принимается. Абсолютный порог провалов (int). Замещён `max_failures` (который умеет и `N%`). Смешение с `max_failures` → `422` `voyage_batch_spec_conflict`. Работает в обоих `batch_mode`. |
| **`inter_unit_interval`** | request | Per-unit пауза в `batch_mode: window` (parity к `inter_batch_interval`, который между Leg-ами в `barrier`). Тип [`duration`](#форматы-значений). Только window. |
| **`require_alive`** | request | bool (default `false`): presence-фильтр живых на резолве scope — `SoulLeaseChecker` отсекает Soul-ы без presence-lease ([ADR-006 Redis](adr/0006-cache-redis.md#adr-006-кэш-и-координация--redis)). Snapshot-scope не ослаблен (фильтр на резолве). |
| **`max_batch_size`** | config | Оператор-инвариант в `keeper.yml::voyage` рядом с `max_scope`: потолок размера батча/окна (`batch_size` для `barrier`, `concurrency` для `window`). Fail-closed → `422` `voyage_batch_size_too_large` (parity `voyage_scope_too_large`). Дефолт + `0` = без предела; отрицательное → `value_out_of_range`. |

> **Семантика `voyage_targets.batch_index` в двух режимах.** В `batch_mode: barrier` — индекс Leg-а единицы (группировка two-level drill). В `batch_mode: window` — **`0` для всех единиц** (пачек нет, прогон плоский; UI drill показывает плоский список).

> **Эндпоинт `POST /v1/voyages/preview`** ([ADR-043 amendment 2026-06-09](adr/0043-voyage.md#adr-043-voyage--унифицированный-батчевый-прогон)). Dry-resolve scope **без создания** Voyage — принимает тот же таргет-фрагмент + batch-настройки, что `POST /v1/voyages`, возвращает `{scope_size, total_batches, effective_batch_size}`. **Не раскрывает SID-список** (только числа). Реюзит резолверы scope в read-only, потолок `voyage.max_scope`, RBAC-by-`kind` (`incarnation.run` для scenario / `errand.run` для command — guard как у create) и тот же rate-limit. Назначение — предпоказ числа батчей при **late-binding** таргете (`coven`/`require_alive`); для snapshot-таргета число считает клиент сам.

> **Error-code `voyage_batch_spec_conflict`** ([ADR-043 amendment 2026-06-09](adr/0043-voyage.md#adr-043-voyage--унифицированный-батчевый-прогон)). `422` при смешении нового строкового batch-формата со старым int-форматом в одном request-е: `batch` вместе с `batch_size`/`batch_percent`, либо `max_failures` вместе с `fail_threshold`.

### State и essence

| Имя | Роль |
|---|---|
| **State_schema** | Схема того, что хранится в `incarnation.state` (jsonb в БД). Объявляется в `service.yml`. Версионируется через `state_schema_version`. |
| **Migration** (state_schema migration) | Преобразование `incarnation.state` с версии N на N+1 при upgrade сервиса. Файл `migrations/<NNN>_to_<MMM>.yml` в service-репо. Грамматика — плоский (`rename`/`set`/`delete`/`move`) + CEL-выражения в `set.value` + структурный `foreach` ([ADR-019](adr/0019-state-migration-dsl.md#adr-019-state_schema-migration-dsl), полная спека — [migrations.md](migrations.md)). Forward-only в MVP. Atomic: одна PG-транзакция на цепочку. **Не путать с runtime-`state_changes`** ([ADR-057](adr/0057-state-changes-crud-verbs.md)): migration-`delete` адресует `path:` (снести поле/путь схемы), state_changes-`remove` адресует `match:` (вынуть элемент коллекции в сценарии). |
| **Default essence** | Дефолтные значения параметров. Хранятся в иерархии `service-repo/essence/` (см. ниже про pipeline). Подложка под `incarnation.spec` оператора. |
| **Essence pipeline** | Декларативная сборка essence в духе Salt `top.sls` / PillarStack: `_default.yaml` → `os/<soulprint.os.family>.yaml` → `coven/<X>.yaml` → `incarnation.spec` (default order) либо явный `essence/_stack.yaml` с операторами `file/inline/when/foreach/optional`. На каждом шаге доступны накопленные переменные. essence **role-agnostic**: ступени `role/<Y>.yaml` в pipeline-е **нет** ([ADR-008](adr/0008-coven-stable-tags.md#adr-008-coven--только-стабильные-логические-теги)); полный порядок и обоснование — [architecture.md → Essence: pipeline сборки](architecture.md#essence-pipeline-сборки). |
| **`error_locked`** | Статус incarnation, когда сценарий упал частично. Любые дальнейшие сценарии отвергаются до явного `keeper.incarnation.unlock`. |
| **`incarnation_archive`** | PG-таблица — архив снесённых incarnation (S-D3, каскад V3). Снимок compliance-минимума (`name` / `service` / `service_version` / `spec` / `state` / `status` + временные метки) + `archived_at`. **БЕЗ FK на live `incarnation`** — переживает физический `DELETE` + каскад. Заполняется в той же транзакции, что и DELETE, ДО него. |
| **`state_history_archive`** | PG-таблица — архив журнала `state_history` снесённых incarnation (S-D3). Снимок change-полей (`history_id` / `incarnation_name` / `scenario` / `state_before` / `state_after` / `apply_id` / `at`) + `archived_at`. **БЕЗ FK на live `incarnation`** — переживает каскадный снос live `state_history`. |

Поле в `manifest.yaml` и в handshake-строке (должны совпадать). | **`protocol_version`** | int. Версия plugin-протокола (`proto/plugin/vN/`); дублируется в manifest и handshake; жёсткое соответствие `protocol_version: N` ↔ `proto/plugin/vN/`. ### Plugin manifest: regex имён

Нормативные regex-ы для kebab-case идентификаторов в `manifest.yaml` и в адресации модуля `<namespace>.<name>.<state>` (используются парсером manifest-а и `soul-lint`-ом). Поля `Handshake` — см. соответствующую запись выше.

| **`name`** | `^[a-z][a-z0-9-]{0,62}$` | Имя плагина внутри коллекции. По аналогии с AID/KID без обязательного префикса. |
| **`<state-name>`** | `^[a-z][a-z0-9-]{0,30}$` | Третий уровень адресации `<namespace>.<name>.<state>` (`installed` / `running` / `restarted` / …). |
| **binary-name** | `soul-mod-<namespace>-<name>` / `soul-cloud-<provider_kind>` / `soul-ssh-<short>` / `soul-beacon-<name>` | Имя итогового бинаря плагина по `kind` ([ADR-016 amendment 2026-05-27](adr/0016-parity-license.md#adr-016-стратегия-parity-с-saltstackansible-и-лицензия-soul-stack) — namespace в имени SoulModule-бинаря; примеры: `soul-mod-official-postgres-user`, `soul-mod-wb-haproxy`). ### Поведение host-а после handshake: `plugin_init_failed` / `plugin_crash`

Имена причин (значения в открытом каталоге `TaskError.reason`), которыми host фиксирует non-graceful exit плагина после успешного handshake. Полная таблица сценариев — [`docs/keeper/plugins.md → Поведение host-а после handshake`](keeper/plugins.md#поведение-host-а-после-handshake-crash-плагина).

| Имя | Когда фиксируется |
|---|---|
| **`plugin_init_failed`** | Плагин завершился с exit code ≠ 0 после handshake, но **до начала Apply** (до первого RPC). |
| **`plugin_crash`** | Плагин завершился с exit code ≠ 0 **в середине Apply-stream-а** (panic / OOM / SIGSEGV / любой non-graceful exit). |

Нормирование полного каталога `TaskError.reason` — отдельная backlog-задача вместе с закрытием `proto/plugin/v1/` и audit-pipeline.

### Sigil: реестр и поля целостности

Имена trust-модели целостности плагинов **Sigil** ([ADR-026](adr/0026-sigil.md#adr-026-sigil--целостность-плагинов-keeper-signed-digest-индекс), спека — [keeper/plugins.md → Integrity-model](keeper/plugins.md#integrity-model)).

| Имя | Роль |
|---|---|
| **`plugin_sigils`** | PG-таблица — Keeper-side allow-list плагинов: ключ `(namespace, name, ref)`, значение `sha256` (hex, lowercase) + audit-поля допуска (кто/когда допустил). Authoritative-источник «какой бинарь допущен». Managed через OpenAPI/MCP (явный допуск Архонтом). Колонка **`commit_sha`** — git-verified происхождение бинаря ([ADR-026(g)](adr/0026-sigil.md#adr-026-sigil--целостность-плагинов-keeper-signed-digest-индекс), audit-метка ВНЕ подписываемого блока) — **вводится при impl A1-S3**. |
| **`sigil_signing_keys`** | PG-таблица — реестр trust-anchor-**ключей подписи** Sigil ([ADR-026(h)](adr/0026-sigil.md#adr-026-sigil--целостность-плагинов-keeper-signed-digest-индекс), миграция 037, R3 multi-anchor). Ключ `key_id` (SHA-256 SPKI, hex) → `pubkey_pem` (SPKI, публичная часть) + `vault_ref` (приватник в Vault KV) + `is_primary` / `status` (active\|retired) + audit-FK на `operators`. **Приватник НИКОГДА не в Postgres.** Инварианты: ровно один primary среди active (partial unique index `sigil_signing_keys_one_primary`), ≥1 active при Retire. Managed через OpenAPI/MCP (operator-facing ротация, R3-S7). НЕ путать с `plugin_sigils` (тот — допуски бинарей, этот — ключи их подписи). |
| **`binary_sha256`** | Новое **опциональное** поле в `proto/plugin` manifest (only-add, [ADR-026(c)/(e)](adr/0026-sigil.md#adr-026-sigil--целостность-плагинов-keeper-signed-digest-индекс)). SHA-256 бинаря, **пришитый** в подписываемый блок Sigil → manifest перестаёт быть подделываемым. |

Транспорт раздачи Sigil Soul-у — only-add proto-message в `oneof payload` существующего `EventStream` ([ADR-012(c)](adr/0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add), без нового RPC); конкретные message-/поля-имена закрепляются отдельным propose-and-wait при имплементации (как `augur.proto` относительно [ADR-025](adr/0025-augur.md#adr-025-augur--keeper-side-брокер-внешнего-доступа-soul)).

> **NB (роут OpenAPI-вьювера `/docs` — осознанное исключение из словаря, [ADR-054 §OpenAPI-вьювер](adr/0054-openapi-code-first.md#adr-054-operator-api--разворот-на-code-first-go-типы--openapi-через-huma-v2)).** Визуальный OpenAPI-вьювер Keeper-а смонтирован на роуте **`GET /docs`** (+ статика `GET /docs/assets/*`). Имя `/docs` — **индустрия-стандарт**, выбран пользователем над словарными вариантами (`/scriptorium`, `/codex`) сознательно: вьювер первичен для оператора, который ищет документацию по узнаваемому конвенциональному пути, а не по «душевному» имени. Это единственное **санкционированное исключение** из правила «имена — только из словаря Soul Stack» для meta-роута; на доменные сущности (`/v1/*`, реестры, модули) исключение НЕ распространяется. Вьювер — meta-роут вне `/v1` (рядом с `/healthz`/`/readyz`/`/openapi.yaml`), не несёт собственной предметной сущности словаря.

> **NB (роут embed-UI `/ui` — осознанное исключение из словаря, [ADR-055](adr/0055-embed-ui-bundle.md)).** Опционально вшитый в keeper UI (`soul-stack-web`) раздаётся на роуте **`GET /ui`** (+ `GET /ui/*` с SPA-fallback на `index.html`). Имя `/ui` — **индустрия-стандарт** (как `/docs`): оператор ищет веб-интерфейс по узнаваемому конвенциональному пути. Санкционированное исключение из «имена — только из словаря» для meta-роута, наравне с `/docs`; на доменные сущности (`/v1/*`) НЕ распространяется. `/ui`-дерево — публичная статика вне `/v1` (parity `/docs`; защита — на API-границе `/v1`), делит listener Operator API, новых портов нет. Тоггл — config-ключ `web_ui_enabled` (см. ниже).

> **NB (пакет `keeper/internal/webui/` + config-ключ `web_ui_enabled`, [ADR-055](adr/0055-embed-ui-bundle.md)).** Пакет **`keeper/internal/webui/`** несёт go:embed завендоренного build-снапшота UI (`soul-stack-web`) и раздаёт его на `/ui`; статический артефакт лежит в подпапке **`keeper/internal/webui/assets/`** (имя `assets/`, НЕ `dist/` — gitignore-правило `dist/` молча съело бы embed-дерево). Технический пакет раздачи статики (как `keeper/internal/api/docsassets/` для `/docs`), не сущность словаря; новых имён словаря не вводится. Синк build-снапшота из companion — `scripts/sync-webui.sh`, drift-guard — `make check-webui` (skip без companion; калька plugin-template `sync-template.sh`/`check-template`). Нормативная типизация config-ключа — [`docs/keeper/config.md`](keeper/config.md) (docs-writer при имплементации).

## Сообщения proto Keeper↔Soul

Имена message-типов в `proto/keeper/v1/*.proto`. Зафиксированы [ADR-012](adr/0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add).

| Имя | Направление | Роль |
|---|---|---|
| **`BootstrapRequest`** / **`BootstrapReply`** | Soul → Keeper unary | Онбординг: Soul предъявляет `bootstrap_token` + CSR (public key в PEM), получает выпущенный SoulSeed-сертификат и CA chain. Передаётся **до** mTLS-стрима, на отдельном listener-е (server-only TLS). `BootstrapReply` дополнительно несёт опц. `sigil_pubkey_pem` (field 5, [ADR-026](architecture.md), S2b) — PEM/SPKI публичного ed25519-ключа подписи Sigil; Soul персистит его trust-anchor-ом рядом с SoulSeed для verify допусков плагинов (S6) — и multi-anchor `sigil_pubkey_pem_set` (field 6, [ADR-026(h)](architecture.md), R3) — полный набор SPKI PEM ключей подписи для безразрывной ротации. **Приоритет set > single**: при непустом `sigil_pubkey_pem_set` одиночное `sigil_pubkey_pem` игнорируется; **оба пустых = Sigil выключен** на этом хосте. |
| **`Hello`** / **`HelloReply`** | оба, первое сообщение EventStream-а | Soul после mTLS-handshake шлёт `Hello` с версией бинаря и опц. snapshot Soulprint; Keeper отвечает `HelloReply` с `session_id` (= KID + server time + ULID), серверным временем и подтверждением. SID **не** в payload как identity-claim — берётся из mTLS peer cert. |
| **`ApplyRequest`** | Keeper → Soul | Команда «применить план». Содержит `apply_id`, `repeated RenderedTask tasks` (план **после** Keeper-side рендера, см. ADR-012(d)), `attempt` (fencing-epoch, [ADR-027](adr/0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim)), `dry_run` (drift-режим: при `true` Soul вызывает `Plan` вместо `Apply`, read-only-проход; **only-add** [ADR-031](adr/0031-scry-drift.md#adr-031-scry--drift-detection-declarative-dry-run-reconcile)) и `passage` (индекс [Passage](#сущности-предметной-области) этапа staged-render, 0-based; **only-add** [ADR-056](adr/0056-staged-render-passage.md); один хост получает N `ApplyRequest` — по одному на Passage). |
| **`TaskEvent`** | Soul → Keeper | Финальный результат одной задачи (агрегация на Soul-е после завершения SoulModule sub-process-а): `task_idx`, `apply_id`, `passage` (индекс [Passage](#сущности-предметной-области), **only-add** [ADR-056](adr/0056-staged-render-passage.md)), `register` (DSL-ядро: `.changed`/`.failed`/`.timed_out`) + декларированные в destiny `output:` поля. Прогресс long-running шагов в MVP не передаётся. |
| **`RunResult`** | Soul → Keeper | Финальный отчёт о прогоне apply: `apply_id`, `passage` (индекс [Passage](#сущности-предметной-области), **only-add** [ADR-056](adr/0056-staged-render-passage.md); per-Passage отчёт — N `RunResult` на `(apply_id, sid, passage)`), статус (`success` / `failed` / `cancelled` / `error_locked`), агрегированные `state_changes` для последующего commit-а `incarnation.state` Keeper-ом. Заменяет ранее предлагавшееся `StateReport` (конфликт с зарезервированным «State» = `incarnation.state`). |
| **`CancelApply`** | Keeper → Soul | Команда отмены текущего apply (`apply_id`). Soul пытается остановить in-flight SoulModule sub-process, шлёт `RunResult` со статусом `cancelled`. |
| **`SoulprintReport`** | Soul → Keeper | Периодический фоновый push фактов о хосте (`refresh_interval`). Payload — `google.protobuf.Struct` до закрытия open Q №6 (тогда будет typed message). |
| **`SeedRotationRequest`** / **`SeedRotationReply`** | Soul → Keeper, потом обратно | Ротация SoulSeed по живому стриму: Soul за `expires_at - 24h` шлёт новый CSR в EventStream-е, Keeper отвечает новым сертификатом, Soul атомарно swap-ает. Отдельным RPC не выделено — это часть EventStream-а oneof. |
| **`AugurRequest`** / **`AugurReply`** | Soul → Keeper, потом обратно | Augur-запрос живого доступа к внешней системе ([ADR-025](adr/0025-augur.md#adr-025-augur--keeper-side-брокер-внешнего-доступа-soul)). `AugurRequest{request_id, apply_id, omen_name, query}` (SID — из mTLS peer cert, не в payload, [ADR-012(i)](adr/0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add)); `AugurReply{request_id, status, oneof result, error}`. **Only-add** в EventStream oneof, новый файл `proto/keeper/v1/augur.proto`; нового RPC нет. Спека — [keeper/augur.md](keeper/augur.md). |
| **`PluginSigil`** | Keeper → Soul | Транспорт одной печати доверия Sigil ([ADR-026](architecture.md), S2b): `PluginSigil{namespace, name, ref, binary_sha256 (hex lowercase), signature (сырые 64B ed25519), manifest (сырые байты manifest.yaml)}`. Самодостаточен для Soul-side verify (S6): re-собирает подписываемый блок и проверяет подпись trust-anchor-ом `BootstrapReply.sigil_pubkey_pem`. **Роль — broadcast-уведомление о новом допуске, НЕ мутация active-набора** ([ADR-026(h)](architecture.md)): Soul по нему upsert не делает, авторитет набора — только `SigilSnapshot`. **Only-add** в `FromKeeper.oneof payload` (field 5), новый файл `proto/keeper/v1/sigil.proto`; нового RPC нет. S2b вводит только контракт — раздача сообщений Keeper-ом и verify на Soul-е — S6. |
| **`SigilSnapshot`** | Keeper → Soul | **Полный active-набор допусков** плагинов: `SigilSnapshot{repeated PluginSigil sigils}` ([ADR-026(h)](architecture.md), field 7). **ЕДИНСТВЕННЫЙ источник истины** active-набора на Soul-стороне; применяется **ReplaceAll** (Soul заменяет весь набор, не upsert) — отсутствующий в snapshot допуск Soul забывает (near-instant revoke/retire без перезапуска). Пустой `sigils[]` = ни один плагин не допущен. **Only-add** в `FromKeeper.oneof payload`. |
| **`SigilTrustAnchors`** | Keeper → Soul | **Полный набор trust-anchor-ов** подписи Sigil: `SigilTrustAnchors{repeated string pubkey_pem}` (SPKI PEM ed25519-ключей, [ADR-026(h)](architecture.md), field 8). Применяется **ReplaceAll**: runtime-доставка **полностью замещает** bootstrap-anchors (replace, не мёрж), multi-anchor для безразрывной ротации ключа подписи (любой anchor из набора верифицирует подпись). **Only-add** в `FromKeeper.oneof payload`. |
| **`PortentEvent`** | Soul → Keeper | Событие beacon ([Vigil](#сущности-предметной-области), [ADR-030](adr/0030-vigil-oracle.md#adr-030-vigil--oracle--event-driven-мониторинг-beacons--reactor)): `PortentEvent{beacon_name, data (google.protobuf.Struct, [deprecated=true] с V5-1 — legacy-payload, заполняется параллельно с typed во время 1-release hand-off-периода), payload (oneof из 7 typed-message: FileChangedPortent / ServiceDownPortent / PortClosedPortent / DiskFullPortent / ProcessAbsentPortent / HttpUnhealthyPortent / InotifyPortent + custom Struct для plugin-beacon, V5-1+V5-3, [ADR-030 amendment 2026-05-26](adr/0030-vigil-oracle.md#amendment-2026-05-26-s5-closure)), collected_at (Soul-side), sid (ECHO; авторитет — mTLS peer cert, [ADR-012(i)](adr/0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add)), опц. dedup_key, опц. severity}`. **Edge-triggered** (на смену состояния). Для Oracle — недоверенный вход. **Only-add** в `FromSoul.oneof payload` (field 7), файл `proto/keeper/v1/beacon.proto`; нового RPC нет. Hard-cut `data` — S5-final (post-1-release, parity с push S7). |
| **`VigilSnapshot`** | Keeper → Soul | **Полный active-набор Vigil** для этого SID ([ADR-030](adr/0030-vigil-oracle.md#adr-030-vigil--oracle--event-driven-мониторинг-beacons--reactor), field 9): `VigilSnapshot{repeated VigilDef vigils}`, где `VigilDef{name, interval (duration-конвенция), check (адрес core-beacon, напр. "core.beacon.file_changed"), params (google.protobuf.Struct)}`. Применяется **ReplaceAll** (как `SigilSnapshot` / `SigilTrustAnchors`): Soul-scheduler заменяет весь локальный набор, отсутствующий Vigil останавливает. Пустой `vigils[]` = ни одной активной проверки. **Only-add** в `FromKeeper.oneof payload`, новый файл `proto/keeper/v1/beacon.proto`; нового RPC нет. |
| **`WardRoster`** | Soul → Keeper | **Снимок ведомых Soul-ом apply-прогонов** (Soul-reconcile, [ADR-027(g)](adr/0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim), S6, field 8): `WardRoster{repeated ActiveApply active}`, `ActiveApply{apply_id, attempt, (status зарезервирован, в MVP не читается)}`. Ложится на [Ward](#сущности-предметной-области): декларирует, какие Ward-задания ещё в полёте. Шлётся **сразу после `Hello` и ДО первого app-сообщения** на каждом (re)connect-е, применяется **ReplaceAll** (как `SigilSnapshot`/`VigilSnapshot`). Keeper по набору терминалит осиротевшие `dispatched`-строки SID-а в статус `orphaned` (закрывает дыру «Keeper и Soul оба мертвы после отдачи»: reclaim сужен до `claimed`, Reaper dispatched-timeout не делаем). Пустой `active[]` = «ничего не ведётся» (рестарт Soul-процесса) → все `dispatched`-строки SID-а осиротляются. Старый Soul без `WardRoster` его не шлёт → sweep не запускается (fail-safe висяк). **Only-add** в `FromSoul.oneof payload`. |
| **`DriftReport`** | Soul → Keeper | **Отчёт о расхождении** (drift-detect, [Scry](#сущности-предметной-области), [ADR-031](adr/0031-scry-drift.md#adr-031-scry--drift-detection-declarative-dry-run-reconcile)): агрегированный результат dry-run-прогона (`Plan` вместо `Apply` по `ApplyRequest.dry_run`) — какие декларированные ресурсы показали бы `changed=true`. Имя в ряд с [`RunResult`](#сообщения-proto-keepersoul) / [`SoulprintReport`](#сообщения-proto-keepersoul) (финальный отчёт прогона). |

`FromSoul` и `FromKeeper` — wrapper-message с `oneof payload`, перечисляющим все сообщения по направлению (см. ADR-012(a)).

### Soul-capabilities

Канонические строковые значения `Hello.capabilities` ([repeated string], [ADR-056](adr/0056-staged-render-passage.md) §S5 forward-compat): Soul анонсирует поддерживаемые фичи протокола Keeper↔Soul при connect-е, Keeper персистит набор рядом с presence (Redis, жизненный цикл = SID-lease) и сверяет ДО dispatch-а фич-зависимых прогонов. Пустой набор = старый Soul без анонса (forward-compat [ADR-012(c)](adr/0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add) only-add) → не поддерживает ни одной фичи (fail-closed). Константы — `shared/config/soul_capability.go` (одна строка на keeper- и soul-стороне).

| Значение | Производное от | Смысл |
|---|---|---|
| **`passage`** | [Passage](#сущности-предметной-области) | Soul эхает `ApplyRequest.passage` в `TaskEvent`/`RunResult`, то есть умеет участвовать в staged-render (N > 1 Passage, [ADR-056](adr/0056-staged-render-passage.md)). Soul без этого признака под staged-сценарием отвергается keeper-ом ДО dispatch (`soul_passage_unsupported`, fail-closed): иначе barrier следующего Passage ждал бы терминал, которого старый бинарь не пришлёт. |

Расширение набора — отдельным capability-строкой (производное от уже зафиксированного имени) + строка в этой таблице; новое самостоятельное имя — propose-and-wait.

### Permissions `role.*`

| Таблица | Роль |
|---|---|
| **`synods`** | Каталог групп. `name` PK (kebab-case, CHECK на формат — как `rbac_roles.name`), `description`, `builtin` BOOL (защита от `synod.delete`), `created_at`, `created_by_aid` FK→`operators(aid)` NULL-able. Симметрия `rbac_roles`. |
| **`synod_operators`** | **Membership** «Synod ↔ архон». `synod_name` FK→`synods(name)` `ON DELETE CASCADE`, `aid` FK→`operators(aid)`, `added_at`, `added_by_aid` FK→`operators(aid)` NULL-able, PK `(synod_name, aid)`. Симметрия `rbac_role_operators`. |
| **`synod_roles`** | **Bundle** «Synod ↔ роль». `synod_name` FK→`synods(name)` `ON DELETE CASCADE`, `role_name` FK→`rbac_roles(name)` `ON DELETE CASCADE`, `granted_at`, `granted_by_aid` FK→`operators(aid)` NULL-able, PK `(synod_name, role_name)`. CASCADE с обеих сторон. |

**Permission-семейство `synod.*`** (NoSelector — управление группами кластер-уровневое, не scoped; как `role.*` / `operator.*`): `synod.create` / `synod.update` (паттерн `*.update` — меняет ТОЛЬКО `description`; `name` PK immutable, ADR-049 amend) / `synod.delete` / `synod.list` + membership-пара `synod.add-operator` / `synod.remove-operator` + bundle-пара `synod.grant-role` / `synod.revoke-role`. **Эндпоинты:** `/v1/synods` (`synod.create` POST / `synod.list` GET), `/v1/synods/{name}` (`synod.update` PATCH / `synod.delete` DELETE), `/v1/synods/{name}/operators` (`add-operator` POST / `remove-operator` DELETE `{aid}`), `/v1/synods/{name}/roles` (`grant-role` POST / `revoke-role` DELETE `{role_name}`). `synod.update` — без subset-check/self-lockout, builtin РАЗРЕШЁН (description прав не выдаёт/не отнимает). **Security-инвариант** ([ADR-049 §f](adr/0049-synod.md#adr-049-synod--группа-архонов)): `synod.grant-role` / `synod.add-operator` — под least-privilege subset (эффективные права инициатора = прямые ∪ через Synod); self-lockout-инвариант распространён на `synod.delete` / `synod.revoke-role` / `synod.remove-operator` (эффективный `*` может приходить через Synod). Резолв эффективных ролей — дополнение snapshot-сборки enforcer-а ([ADR-028(d)](adr/0028-rbac-storage.md#adr-028-rbac-storage--postgres)); матчинг-слой Purview не меняется.

### Permission `incarnation.create-rerun`

Permission для атомарного rerun-а сценария `create` из `error_locked` — **`incarnation.create-rerun`** (REST `POST /v1/incarnations/{name}/rerun-create`, MCP-tool `keeper.incarnation.rerun-create`). Audit-событие — `incarnation.create_rerun` (НЕ переиспользует `incarnation.unlocked`). См. [Атомарность и `error_locked`](architecture.md#атомарность-и-error_locked).

### Purview — scope-резолвер ([ADR-047](adr/0047-purview.md#adr-047-purview--scoped-rbac-видимость-узлов-role-default_scope--расширенный-селектор))

| Имя | Тип | Смысл |
|---|---|---|
| **Purview** | Сущность (тип-результат) | Резолвер разрешённого scope оператора: `ResolveScope(aid, resource, action) → Purview`. Обобщает `Enforcer.CovenScope` (`(covens, unrestricted)`) на измерения `covens`/`regexes`/`soulprint`/`state` + флаги `unrestricted`/`deny`. Верхняя граница видимости/таргетинга оператора (souls-list, incarnations-list, target Cadence/Vigil/Voyage). НЕ путать с `Ward` (claim apply-задания, [ADR-027](adr/0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim)) — другая подсистема. |
| **`Purview.Deny`** | Терминальный флаг `Purview` | «Доступ запрещён по scope» — терминальный, отрезает раньше сбора измерений. Первое реальное использование (ADR-047 §г G1): `ResolvePurview` для **revoked-оператора** возвращает `Purview{Deny:true}` (зеркало revoked-ветки `Enforcer.Check`). Единая точка revoked-aware-резолва для всех read-souls-путей: `HoldsAction`→`false`→403, `soulpurview.Resolve`→Empty→404, `InScope`→false. До G1 поле было заготовкой (всегда false). НЕ путать с `default_policy: deny` (общая базовая политика enforcer-а) — `Deny` это per-(aid,resource,action)-результат. |
| **`default_scope`** | Поле `rbac_roles` | Scope-селектор, наследуемый всеми permission-ами роли. Per-permission селектор (`on coven=X`) переопределяет. Пустое явно-введённое измерение → default-deny; `*`-permission и bare-permission (без default_scope и без per-perm-селектора) — исключения (allow), см. [ADR-047(б)](adr/0047-purview.md#adr-047-purview--scoped-rbac-видимость-узлов-role-default_scope--расширенный-селектор). |

**Селектор-ключи Purview.** К exact-ключам `coven`/`host`/`incarnation`/`service` (грамматика permission, [rbac.md → Формат permissions](keeper/rbac.md#формат-permissions)) добавляются:

| Ключ | Измерение | Семантика |
|---|---|---|
| **`regex`** | regexes | Совпадение SID/имени хоста с RE2-паттерном (Go `regexp`, без backtracking; длина ≤256 против ReDoS). Quoted-форма `regex='^web-.*'` (кавычки отделяют regex от `,`-разделителя value-list). Реализован в S2a ([rbac.md → Грамматика селектора](keeper/rbac.md#грамматика-селектора)): парсинг + `Purview.Regexes` + `Check`-матчинг (host/sid-context) + least-privilege subset (string-equality fail-closed). Применение к list-видимости/target — S3/S4. |
| **`soulprint`** | soulprint | CEL-предикат над `soulprint.self.*` ([ADR-018](adr/0018-soulprint-typed.md#adr-018-soulprint-typed-схема-mvp)). Quoted-форма `soulprint='soulprint.self.os.family == "debian"'` (одинарные кавычки отделяют CEL от `,`-разделителя value-list; параллель `regex='…'`; длина ≤512). Реализован в S2b ([rbac.md → Грамматика селектора](keeper/rbac.md#грамматика-селектора)): парсинг + компиляция-валидация на load через `shared/cel` (sandbox `NewFlowControl`) + `Purview.SoulprintExprs` + least-privilege subset (string-equality fail-closed). `Check`-матчинг S2b — fail-closed (context не несёт nested-факты); реальный CEL-eval против фактов хоста — слайсы S3/S4. |
| **`state`** | state | CEL-предикат над `incarnation.state`. **Слайс S2c, грамматика ключа — propose-and-wait при реализации.** |

### Enforcer-методы и middleware

Поверхность авторизации в коде: enforcer-методы (`keeper/internal/rbac`) и middleware-обёртки (`keeper/internal/api/middleware`). `Check` / `ResolvePurview` / `RequirePermission` существовали ранее; `HoldsAction` / `RequireAction` введены ADR-047 §г G1 для read-видимости.

| Имя | Слой | Смысл |
|---|---|---|
| **`Check`** | enforcer-метод | Scope-aware-проверка `Check(aid, resource, action, context) error`: «применима ли permission в данном scope-контексте». Контекст приходит из path/body. Для scoped-permission с пустым контекстом даёт ложный deny (селектор-ключа нет в nil-контексте) — поэтому НЕ годится как gate read-эндпоинта. Остаётся гейтом **мутаций** (scope известен из path/body). |
| **`ResolvePurview`** | enforcer-метод | Резолвит [Purview](#purview--scope-резолвер-adr-047) оператора для (resource, action). С G1 **revoked-aware**: revoked-AID → `Purview{Deny:true}` (единая точка для всех read-souls-путей). |
| **`HoldsAction`** | enforcer-метод | Existence-check `HoldsAction(aid, resource, action) bool`: «держит ли оператор действие в ПРИНЦИПЕ, в любом scope, игнорируя селектор-контекст». Реализован поверх `ResolvePurview`: `true`, если `Unrestricted` ИЛИ заполнено любое измерение (coven/regex/soulprint/state) И НЕ `Deny`. Существует, чтобы пускать scoped-оператора к собственному списку без ложного deny scope-aware `Check`-а; сужение по scope — дальше в handler-е. |
| **`RequirePermission`** | middleware-фабрика | Scope-aware-gate поверх `Check`: извлекает scope-контекст (`SelectorExtractor` из path/body) и режет оператора, чей контекст не сматчил селектор. Гейт **мутаций**. |
| **`RequireAction`** | middleware-фабрика | Existence-gate поверх `HoldsAction` (ADR-047 §г G1): пускает любого держателя действия, scope-сужение делает handler. Гейт **read-видимости** souls (`GET /v1/souls`, `/{sid}`, `/{sid}/soulprint`, `/{sid}/history`), где scope-контекст на gate-этапе ещё неизвестен (резолвится из строк БД). |

### Redis pub/sub-каналы Keeper↔Keeper

Внутренние wire-идентификаторы cross-instance координации HA-кластера ([ADR-006](adr/0006-cache-redis.md#adr-006-кэш-и-координация--redis)) — НЕ пользовательская поверхность. Все несут JSON-envelope с `origin_kid` (KID публикующего инстанса) и **self-filter по KID** (нода игнорирует собственное эхо публикации). Operator-видимая операционная сводка тех же ключей/каналов — [operations/infra.md → Что лежит в Redis](operations/infra.md#что-лежит-в-redis).

| Канал | Роль |
|---|---|
| **`events:shard:<n>`** | **Cluster-routing apply/run-событий между Keeper-инстансами** (роль (c) [ADR-006](adr/0006-cache-redis.md#adr-006-кэш-и-координация--redis), `applybus`). Несёт `TaskEvent` / `RunResult` / `ErrandResult` прогонов: SSE-подписчик `GET /mcp/events?apply_id=X` на Keeper-A получает события прогона, чей Soul подключён к Keeper-B. Канал **шардирован**: фиксированное множество `events:shard:<n>`, где `n = fnv32a(apply_id) % 256` (K=256 шардов), Redis-bridge поднимается per-shard, forward-loop фильтрует по `envelope.apply_id`. Заменяет прежний per-applyID канал `apply:<apply_id>` (упирался в `maxclients` на больших флотах — отдельная подписка на каждый id). Префикс `events` (не `apply`) — канал несёт и `apply.*`, и `errand.*`. Envelope: `origin_kid` / `kind` / `apply_id` / `at` / `payload`. Реализация — `keeper/internal/redis/applybus.go` (`ApplyBusChannel`/`ApplyBusShardIndex`/`ApplyBusShardCount`) + `keeper/internal/applybus/bus.go` ([ADR-006(c.1)](adr/0006-cache-redis.md#adr-006-кэш-и-координация--redis)). |
| **`rbac:invalidate`** | Сигнал «снимок enforcer-а устарел». Мутация роли / permissions / membership на любой ноде публикует envelope `{origin_kid, at}` → все ноды перечитывают снимок из БД. Best-effort + TTL-poll fallback ([rbac.md → Как enforcer резолвит](keeper/rbac.md#как-enforcer-резолвит), [ADR-028(d)](adr/0028-rbac-storage.md#adr-028-rbac-storage--postgres)). |
| **`herald:invalidate`** | Сигнал «снимок включённых [Tiding](#сущности-предметной-области)-правил notification-dispatcher-а устарел» ([ADR-052](adr/0052-herald-notifications.md#adr-052-herald--tiding--уведомления-о-событиях-прогонов), S4). CRUD [Herald](#сущности-предметной-области)/[Tiding](#сущности-предметной-области) на любой ноде публикует envelope `{name, at}` → каждая нода дёргает свой `Dispatcher.InvalidateRules` (следующий матч перечитывает enabled-снимок из PG). Best-effort + TTL-poll fallback (`DefaultRuleCacheTTL=15s`). Стиль `<подсистема>:<событие>` как `rbac:invalidate`. |

### Термин «FK» — разведение

С [ADR-028](adr/0028-rbac-storage.md#adr-028-rbac-storage--postgres) термин «FK» в RBAC означает **настоящий PG foreign key**: `created_by_aid` / `changed_by_aid`, `rbac_role_operators.aid`, `granted_by_aid` — все ссылаются на `operators(aid)`. Прежний метафорический «FK» YAML-списка `roles[].operators` ([ADR-014](adr/0014-operator-identity.md#adr-014-identity-модель-оператора-archon), до ADR-028) — это **membership** (привязка «роль ↔ оператор»), теперь материализованная строкой `rbac_role_operators`, а не ссылка в файле. В новых текстах: «membership / привязка» = строка `rbac_role_operators`; «FK» = настоящий PG-foreign-key.

## Audit-events

Audit-event — структурированная запись действия в audit-pipeline Keeper-а. Общая нормировка pipeline-а (storage = Postgres-таблица `audit_log`, schema, write-path, retention через Reaper-правило `purge_audit_old`, OTel dual-write) — **[ADR-022](adr/0022-audit-pipeline.md#adr-022-audit-pipeline-storage-schema-retention)**, [keeper/storage.md → Таблица `audit_log`](keeper/storage.md#таблица-audit_log), блок [`keeper.yml → audit`](keeper/config.md#audit).

**Convention имени — `<area>.<action>`** (lowercase, dot-separated ASCII; `<area>` — `[a-z][a-z0-9_]*`; `<action>` — `[a-z][a-z0-9-]*`, допускаются `_` для составных глаголов и `-` для kebab-составных: `reload_succeeded`, `access_denied`, `permissions-updated`, `operator-granted`, `kv-read`). По аналогии с RBAC permissions ([keeper/rbac.md → Формат permissions](keeper/rbac.md#формат-permissions)) — но раздельные пространства имён (permission `incarnation.read` и audit-event `incarnation.created` не пересекаются).

**Каталог открытый**: новые имена добавляются обычным PR в этот раздел при нормировании соответствующих write-path подсистем (см. write-path таблицу в [ADR-022(g)](adr/0022-audit-pipeline.md#adr-022-audit-pipeline-storage-schema-retention)). Полный перечень имён в этом документе **не нормируется** — наполняется по факту имплементации. После первого упоминания имени в нормативной формулировке оно стабильно — переименование требует backward-compat-плана.

### Примеры event-types по областям (для имплементации)

Имена ниже — **примеры convention-а**, иллюстрирующие, как раскладываются event-types по областям при нормировании соответствующих pipelines. Они **ещё не зафиксированы** ни одним ADR (кроме `config.*` выше); каждый будет добавлен PR-ом в таблицу «Зарегистрированные event-types» в момент имплементации write-path-инициатора с явной ссылкой на ADR.

**Область `config.*`** — изменения `keeper.yml` ([ADR-021](adr/0021-hot-reload-config.md#adr-021-hot-reload-конфига-с-write-back-yaml), уже зарегистрированы выше): `config.reload_succeeded`, `config.reload_failed`.

## Статусы push_runs

Перечисление значений `push_runs.status` (внутренний контракт push-orchestrator, [ADR-032](adr/0032-push-orchestrator.md#adr-032-push-orchestrator-variant-c--multi-host-destiny-push-без-incarnationscenario)). Не путать с `apply_runs.status` (`success`/`failed`/`cancelled`/`orphaned`/`no_match`) — push-orchestrator не пересекается с Acolyte work-queue.

`push_runs.status: pending | running | succeeded | partial_failed | failed`

- **`pending`** — insert-row, executeAsync ещё не стартовал.
- **`running`** — executeAsync взял prog, идёт.
- **`succeeded`** — все хосты ok.
- **`partial_failed`** — часть хостов прошла, часть упала.
- **`failed`** — все хосты упали ИЛИ render-error/lookup-error до старта.

## Статусы tides

Перечисление значений `tides.status` (внутренний контракт Tide-orchestrator, [ADR-040](adr/0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override)). Не путать с `apply_runs.status` (`success`/`failed`/`cancelled`/`orphaned`/`no_match`) — Tide живёт на уровень выше: один Tide → N `apply_runs` (по одному на Surge, FK `apply_runs.tide_id`).

`tides.status: pending | running | succeeded | failed | partial_failed | cancelled`

- **`pending`** — Tide создан, никакой Keeper-инстанс ещё не сделал claim. После `reclaim_tides` Reaper-правила ([ADR-040 amendment 2026-05-27](adr/0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override)) — также сюда возвращается Tide, у которого `claim_expires_at < NOW()` (`attempt++`, parity [ADR-027](adr/0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim) `claimed → planned`).
- **`running`** — Tide claimed Keeper-инстансом (`claimed_by_kid` NOT NULL, `claim_expires_at` NOT NULL — CHECK-инвариант), surges выполняются sequentially.
- **`succeeded`** — все surges завершились success.
- **`failed`** — `on_surge_failure: abort` + один Surge провален (остальные не стартовали).
- **`partial_failed`** — `on_surge_failure: continue` + один или более surges провалены, остальные прошли.
- **`cancelled`** — оператор отменил Tide (`DELETE /v1/tides/{tide_id}`, отложено пост-MVP).

Отдельного терминала `orphaned` НЕТ (в отличие от `apply_runs.status`): Reaper-правило `reclaim_tides` возвращает Tide в `pending` для пере-claim другим Keeper-инстансом, а не закрывает его терминалом. Симметрия с [ADR-027](adr/0027-apply-work-queue.md#adr-027-модель-исполнения-apply--work-queue--claim-acolyte-пул-ward-claim) — Ward-claim reclaim тоже возвращает `claimed → planned`, не вводит отдельного `orphaned` для протухших claim-ов.

|---|---|---|
| `target.coven[]` | `list<string>` | Override scenario `on:` — **AND-merge** со списком coven (только сужение, не расширение). Security invariant [ADR-009 §3](adr/0009-scenario-dsl.md#adr-009-scenario--полная-dsl-задач-destiny-граница-с-destiny--рекомендация). |
| `target.where` | CEL-строка | Override scenario `where:` — **AND-merge** с CEL-predicate (∧ существующего scenario `where:`). |
| `wave.size` | int | Wave-size (souls per Surge). Если total scope ≤ `wave_size` — один Surge (Tide вырождается в classic single-run по execution, но запись `tides` создаётся). |
| `wave.on_failure` | enum (`abort` \| `continue`) | Failure-policy между surges: `abort` (default, fail-fast) или `continue` (выкатить до конца, terminal = `partial_failed`). `stop-and-wait` — отложено пост-MVP. |
| `concurrency` | int (nullable) | **REPLACE** scenario `serial:` (runtime-knob). Без override — scenario `serial:` сохраняется. |

## Error codes

Error code — стабильное имя ошибки в формализованном канале: HTTP-response под RFC 7807, поле `TaskError.reason` в `proto/keeper/v1/`, MCP-tool error `code`, диагностический вывод парсеров YAML / manifest / destiny, поле `validation_errors[]` audit-event-а `config.reload_failed`. От audit-event names (`<area>.<action>`) отличается ролью: audit-event описывает **что произошло**, error code — **почему отвергнуто** или **чем упало**.

Семейство `<ключ>_on_block_invalid` (Категория 1) — fail-closed отказ module-специфичного ключа на `block`-уровне: `block` — группирующая конструкция, не module-задача, поэтому module-специфичные ключи на самой block-задаче отвергаются парсером. Коды: `register_on_block_invalid`, `changed_when_on_block_invalid`, `failed_when_on_block_invalid`, `retry_on_block_invalid`, `timeout_on_block_invalid`, `output_on_block_invalid`, `no_log_on_block_invalid`, `params_on_block_invalid`, `parallel_on_block_invalid` — каждый формулируется как «`<ключ>` не допускается на block-уровне (block — группирующая конструкция, не module-задача)». Block-наследуемые ключи (`when:` / `where:` / `serial:` / requisites / `vars:`) сюда **не** входят — они валидны на block-уровне. Поведение `block:` — [`docs/destiny/tasks.md §6.5`](destiny/tasks.md).

Группа stratify/passage (Категория 1, [staged-render](#сущности-предметной-области) [ADR-056](adr/0056-staged-render-passage.md)) — fail-closed диагностики стратификации задач по register-зависимости (probe→where; задача-потребитель попадает в Passage строго ПОСЛЕ probe-эмиттера):

- `register_dependency_cycle` — register-зависимость по кругу (probe A читает `register.B`, probe B читает `register.A`): топологического порядка по Passage нет, прогон отвергается.
- `within_block_register_dependency` — потомок `block:` читает register, эмитнутый соседним потомком **того же** блока: невозможно на render. `block` атомарен по Passage (весь fan-out — один Passage), peer-register доступен только Soul-side ПОСЛЕ probe, а `where`/`params`/`vars`/`apply.input` резолвятся Keeper-side ДО dispatch → отбор хостов по устаревшему/внешнему register молча (silent-wrong-target). Flow-control `when` (Soul-side per-task gating после FC-5 narrow-fix) сюда **не** входит — within-block `when: register.peer` валиден (peer-probe исполняется тем же ApplyRequest ДО потребителя, Soul видит peer-register в накопленном срезе блока). Probe и потребитель должны быть разными top-level задачами (тогда Stratify штатно разведёт их по Passage). Детектится офлайн (`soul-lint validate-scenario`) и runtime-страховкой. Симметрия с семейством `_on_block_invalid` (отказ block-неверного на block-уровне).
- `cross_passage_requisite_unsupported` — задача ссылается через requisite (`onchanges:`/`onfail:`/…) на register, чей источник в другом Passage, при недоступном журнале аудита (`AuditReader`): cross-passage gating требует читать CHANGED/FAILED-факты из аудита, без него keeper не может определить факт молча → fail-closed reject (runtime, [ADR-056 R3](adr/0056-staged-render-passage.md)).
- `cross_passage_when_unsupported` — задача гейтит `when:`/`changed_when:`/`failed_when:` по register, эмитнутому в **более раннем** Passage: flow-control = Soul-side per-task gating ([ADR-012(d)](adr/0012-keeper-soul-grpc.md)), видит только register **своего** Passage → cross-passage register недоступен (другой `ApplyRequest`) → `no such key` молча, задача FAILED. `where:` (Keeper-side targeting) cross-passage умеет, `when:` — нет (легитимная асимметрия). Лечится `where:` для cross-task register-таргетинга или `register.self` для same-task gating. Детектится офлайн (`soul-lint validate-scenario`) и runtime-страховкой ([ADR-056 amend 2026-06-21](adr/0056-staged-render-passage.md), FC-5).

- Категория 1 (parser / validation) — [`docs/keeper/config.md`](keeper/config.md), [`docs/keeper/plugins.md`](keeper/plugins.md), [`docs/keeper/rbac.md`](keeper/rbac.md).
- Категория 2 (HTTP / RFC 7807) — [`docs/keeper/operator-api.md → Error format`](keeper/operator-api.md).
- Категория 3 (runtime task/plugin) — `proto/keeper/v1/` (поле `TaskError.reason` в `apply.proto`), [`docs/keeper/plugins.md → Поведение host-а после handshake`](keeper/plugins.md#поведение-host-а-после-handshake-crash-плагина).

## DSL-синтаксис

### Маркеры выражений

| Маркер | Назначение |
|---|---|
| **`${ expr }`** | CEL-выражение в строковом контексте YAML (`params:`, `vars:`, `apply: input:`, `on:`-литералы, миграционные `set:` и т.п.). Парсер ищет ровно последовательность `${`; одиночный `$` маркером не является. Top-level expression-ключи (`where:`, `when:`, `changed_when:`, `failed_when:`, `until:`) — вся строка трактуется как CEL-выражение **без обёртки** `${ … }`. Спецификация — [templating.md](templating.md), фиксация выбора — [ADR-010](adr/0010-templating.md#adr-010-шаблонизатор-cel-для-yaml-выражений-go-texttemplate-для-файлов). |

### Форматы значений

| Имя | Формат |
|---|---|
| **`duration`** | Go-`time.ParseDuration` (`1s` / `500ms` / `1h30m`) + расширение `<N>d` для дней (`30d` = 720h, удобство long-lived retention). Композитная форма `1d2h` **не** поддерживается. Используется для всех duration-полей в конфигах (`keeper.yml`, `soul.yml`). Каноническая фиксация — [`docs/keeper/config.md → Конвенции типов`](keeper/config.md). |

## Метрики и OTel resource-attributes

Нормировано [ADR-024](adr/0024-observability.md#adr-024-observability-prometheus-primary--otel-bridge); полная спека конвенций — [`docs/observability.md`](observability.md).

### Namespace-префиксы метрик

Метрики различаются по компоненту **префиксом имени** (не label `component=`). Имена — Prometheus-convention: `snake_case`, единица в суффиксе (`_seconds` / `_bytes`), `_total` для counters.

| Префикс | Кто экспонирует | Пример |
|---|---|---|
| **`keeper_*`** | Keeper-side метрики. | `keeper_http_requests_total`, `keeper_grpc_streams_active` |
| **`soul_*`** | Soul-side метрики. | `soul_apply_tasks_total` |

Конкретный каталог метрик не нормируется в словаре — наполняется по факту имплементации подсистем (как [Audit-events](#audit-events)), каждая метрика следует конвенциям [observability.md §2](observability.md#2-namespace-метрик-soul_--keeper_).

### Config-сущность `metrics.auth.basic` (keeper)

Опц. HTTP Basic-auth на keeper-`/metrics` ([ADR-024](adr/0024-observability.md#adr-024-observability-prometheus-primary--otel-bridge), [keeper/config.md → metrics](keeper/config.md#metrics)).

| Ключ | Тип | Смысл |
|---|---|---|
| `metrics.auth.basic.enabled` | `bool` | Включить basic-auth. |
| `metrics.auth.basic.username` | `string` | Имя пользователя (обязательно при `enabled`). |
| `metrics.auth.basic.password_ref` | `vault-ref` | `vault:<mount>/<path>` на секрет с полем `password`. Plaintext-пароль запрещён (только vault-ref). |

**На Soul `metrics.auth` нет** — у Soul-агента нет vault-клиента ([ADR-012](adr/0012-keeper-soul-grpc.md#adr-012-контракт-keepersoul-grpc-один-eventstream-с-oneof-keeper-side-рендер-forward-compat-only-add)) для резолва `password_ref`; Soul-`/metrics` защищён loopback-bind (`metrics.listen` = `127.0.0.1`). Auth для Soul — отложено.

### Кастомные OTel resource-attributes

Доменная идентичность инстанса в трейсах/метрик-экспорте. Префикс `soulstack.` — namespace кастомных атрибутов проекта (не конфликтует с OTel semconv).

| Attribute | Значение | Применяется на |
|---|---|---|
| **`soulstack.kid`** | [KID](#идентификаторы) (Keeper ID). | `service.name="keeper"` |
| **`soulstack.sid`** | [SID](#идентификаторы) (Soul ID = FQDN). | `service.name="soul"` |

Стандартный `service.name` = `"keeper"` \| `"soul"` (имя бинаря по словарю). Высокая кардинальность KID/SID — причина класть их в OTel resource-attrs, **не** в Prometheus metric-labels ([observability.md §2.2](observability.md#21-конвенция-именования), [§3](observability.md#3-otel-resource-attributes)).
