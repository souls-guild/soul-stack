# Soul Stack Name Dictionary

All system entities are named according to this dictionary - not SaltStack terms. Names are required in the code (packages, types), in the API (fields, endpoints, MCP tools), in the configuration (flags, keys), in logs, metrics and user documentation. When the SaltStack parallel is helpful to the reader, it is mentioned once in parentheses; the primary terms are ours.

Introduction of a new name - through the propose-and-wait rule in [CLAUDE.md](../CLAUDE.md).

### Criterion "large â†’ Soul Stack, small â†’ familiar"

Clarifies **when** the name is taken from the "soulful" metaphor of Soul Stack, and when from the usual DevOps/Developer vocabulary (the propose-and-wait rule itself for any new name remains in force - this criterion only sets the direction of choice):

- **Large blocks** - entities, subsystems, run phases, artifacts, protocols - are named in the spirit of the "spiritual" metaphor: `Keeper`, `Souls`, `Destiny`, `Soulprint`, `Essence`, `Archon`, `Reaper`, `Voyage`, `Choir`, `Covenant`, `Augur`, `Sigil`, `Herald`, etc.
- **Small units** - config keys, flags, fields, enum values, internal helpers - are called by familiar DevOps/Developer terms: `transport`, `retry`, `timeout`, `provider`, `profile`, `extends` etc.
- **Why so:** the product is great. In small things, familiar names reduce cognitive load and confusion; metaphor, on the contrary, consolidates large concepts into a recognizable vocabulary.
- **Example of application (2026-06-29):** config-key for selecting the bootstrap-token delivery transport is selected `transport: direct|teleport` - a familiar word, and **not** the "soulful" `passage`, which was first proposed by architect (this is a small unit - a config-key, not a new entity).
- **Use example (2026-07-01):** log-shipping data plane agent is named **`vector`** - upstream product name (Vector.dev), like `node-exporter`/`redis-exporter`: this is an external tool that Soul Stack installs and configures (destiny [`examples/destiny/vector/`](../examples/destiny/vector/destiny.yml), rolls with service scripts next to exporters - the PUSH log plane, which complements the pull metrics plane, does not replace it). Thematic name rejected - vector is not a Soul Stack entity, but a managed product; inventing your name on top of the upstream name would only confuse the operator.

### `state_changes`: CRUD verbs and keys

`state_changes` script - **ordered list of operations** (YAML list, not map), each element = one **CRUD verb** (singular). Grammar - [ADR-057](adr/0057-state-changes-crud-verbs.md), normative spec - [`docs/scenario/orchestration.md Â§7.1`](scenario/orchestration.md). Multiplicity is expressed by **`match`-predicate**, not by flags/handles.

| Verb | Role |
|---|---|
| **`set`** | `set: <field>` + `value: "${CEL}"` - rewrites the entire `incarnation.state.<field>` field (replaces the previous `sets`-map). |
| **`add`** | `add: <collection>` - add an element. map-collection: `key:` + `value:`. list-collection: `value:` + opt. `match:` (grandfather). `on_conflict: skip` (default) `\| replace \| error`. |
| **`modify`** | `modify: <collection>` + `match:` + `patch: { <path-in-element>: "${CEL}" }` â€” patch **all** matching the predicate (all-by-default). |
| **`remove`** | `remove: <collection>` + `match:` â€” remove **all** matching the predicate (all-by-default). |
| **`foreach`** | `foreach: "${CEL-list|map}"` + `as: <name>` + `do: { <verb...> }` â€” bulk fan-out N operations. The form is literally from migration-DSL ([ADR-019](adr/0019-state-migration-dsl.md#adr-019-state_schema-migration-dsl)). |

| Key | Role |
|---|---|
| **`value:`** | Value for `set`/`add`: CEL expression `${ â€¦ }` or YAML literal (map/list/scalar). |
| **`key:`** | The entry key for `add` in the map collection is: `key: "${CEL}"`. |
| **`match:`** | CEL predicate over a collection element - selecting suitable ones in `modify`/`remove`, dedup predicate in `add`(list). all-by-default; empty-match â†’ no-op. |
| **`patch:`** | Map `<path-in-element>: "${CEL}"` for `modify` - what to rewrite in each linked element. |
| **`on_conflict:`** | Behavior of `add` in case of collision (the map key is busy / list-`match` already finds): `skip` (default, idempotent) `\| replace \| error`. |
| **`expect:`** | Opt. runtime multiplicity assertion for `modify`/`remove`: `one \| at_most_one \| any` (default `any`). Hooked â‰  expected â†’ `error_locked` before commit. |

| Binding CEL | Role |
|---|---|
| **`elem`** | The current list collection element (or scalar) in `match`/`patch`/`value`. Name `elem` (not `self`) - to avoid conflicts with per-host `soulprint.self`. |
| **`key`** / **`value`** | The key/value of the current map collection entry in `match`/`patch`/`value`. |

> The full sets context is available on top of the bindings (`input.*` / `incarnation.*` / `soulprint.self.*` / `register.*` / `vars.*` / `essence.*`) - the same CEL-env as for `params:` tasks ([ADR-010](adr/0010-templating.md)).

> **`remove` (state_changes) â‰  `delete` (migration-DSL) - intentionally different names.** `remove` ([ADR-057](adr/0057-state-changes-crud-verbs.md)) = "remove an element from the collection" by `match` predicate over elements (runtime script). `delete` ([ADR-019](adr/0019-state-migration-dsl.md#adr-019-state_schema-migration-dsl), [migrations.md](migrations.md)) = "demolish the path in the state structure" by `path:` (migration of the state form when changing the scheme). > **Not entered** ([ADR-057](adr/0057-state-changes-crud-verbs.md) rejected alternatives): `clear` (= `set` empty), `rename`/`move` (this is migration-DSL), `upsert` (= `add` + `on_conflict: replace`), positional `remove` (first/last - not reproducible in JSONB), paired verbs `*_one`/`*_all`, flag `all:` (plurality = property `match`, not name/flag).

### The term "FK" is breeding

## push_runs statuses

Enumeration of `push_runs.status` values (push-orchestrator internal contract, [ADR-032](adr/0032-push-orchestrator.md)). Not to be confused with `apply_runs.status` (`success`/`failed`/`cancelled`/`orphaned`/`no_match`) - push-orchestrator does not overlap with Acolyte work-queue.

`push_runs.status: pending | running | succeeded | partial_failed | failed`

- **`pending`** â€” insert-row, executeAsync has not started yet.
- **`running`** - executeAsync took the prog, it goes.
- **`succeeded`** - all hosts are ok.
- **`partial_failed`** - some hosts passed, some fell.

## Tides statuses

Enumeration of `tides.status` values (Tide-orchestrator internal contract, [ADR-040](adr/0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override)). Not to be confused with `apply_runs.status` (`success`/`failed`/`cancelled`/`orphaned`/`no_match`) - Tide lives one level higher: one Tide â†’ N `apply_runs` (one each on Surge, FK `apply_runs.tide_id`).

`tides.status: pending | running | succeeded | failed | partial_failed | cancelled`

- **`pending`** â€” Tide has been created, no Keeper instance has made a claim yet. After `reclaim_tides` Reaper rule ([ADR-040 amendment 2026-05-27](adr/0040-tide.md#adr-040-tide--invocation-time-scope-chunking--target-override)) - Tide also returns here, which has `claim_expires_at < NOW()` (`attempt++`, parity [ADR-027](adr/0027-apply-work-queue.md) `claimed â†’ planned`).
- **`running`** â€” Tide claimed Keeper instance (`claimed_by_kid` NOT NULL, `claim_expires_at` NOT NULL â€” CHECK-invariant), surges are executed sequentially.
- **`succeeded`** â€” all surges ended in success.
- **`failed`** â€” `on_surge_failure: abort` + one Surge failed (the rest did not start).
- **`partial_failed`** â€” `on_surge_failure: continue` + one or more surges failed, the rest passed.
- **`cancelled`** - Operator canceled Tide (`DELETE /v1/tides/{tide_id}`, deferred post-MVP).

There is NO separate terminal `orphaned` (unlike `apply_runs.status`): Reaper rule `reclaim_tides` returns Tide to `pending` for re-claim by another Keeper instance, rather than closing it with a terminal. Symmetry with [ADR-027](adr/0027-apply-work-queue.md) - Ward-claim reclaim also returns `claimed â†’ planned`, does not introduce a separate `orphaned` for rotten claims.
